# 04. 변경: 체험 우주 UX 정돈

> [plan/19.engram-experience](../plan/19.engram-experience.md)의 이미 구현된 동작을 바꾸는 **변경 제안**(WHAT 델타).

## 현재 (as-is)

`/`는 인증 사용자의 메인 우주 셸이고, 랜딩(`/landing`)의 히어로 CTA와 이론 카드 CTA는 `enterDemoMode()` 후 같은 `/` 셸에 demo 데이터로 진입한다. 그러나 문서·카피는 "데모", "체험", "우주", 오래된 `/universe` 표현이 섞여 있어 사용자-facing 의미가 흐린다.

체험 우주 안의 `DemoSimPanel`은 이미 시간 머신, 별 띄우기, 다감정 하루, 페르소나 전환, 겹쳐보기, 이론 모달을 제공한다. 하지만 UI가 기능 설명 패널처럼 길고, z-index/겹침이 회상 패널·목록 오버레이·드롭다운과 충돌할 수 있으며, 처음 온 사용자가 "게임처럼 눌러 우주가 커지는" 흐름으로 읽기 어렵다.

시간 이동은 `runTimeSkip(queryClient, days)`가 가상 시계를 한 번에 `days`만큼 트윈하고, 트윈이 자연 완료될 때 `demoConsolidate()`를 한 번만 호출한다. 그래서 `한 달 지나기`가 실제 24시간 배치 30회를 근사하지 못하고, Live force-sim은 이동 중 반지름/표상 부동 re-kick을 계속 받아 별이 스프링처럼 요동쳐 보일 수 있다.

## 목표 (to-be)

- 용어를 고정한다: **메인 우주** = `/`의 일반 `HomePage` 우주 셸, **체험 우주** = 랜딩에서 CTA/카드로 들어가는 같은 `/` 우주 셸의 demo 데이터 버전, **기억 실험실** = 체험 우주 안의 컨트롤 패널. `demo mode`/`데모 모드`는 코드와 내부 상태명으로만 쓴다.
- 랜딩 CTA와 문서는 체험 우주 기준으로 정돈한다. 히어로 1차 CTA는 "체험 우주 시작하기", 이론 카드 CTA는 "체험 우주에서 해보기"로 통일하고, 진입 경로는 실제 라우트인 `/` + 선택적 `?sim=<id>`로 기술한다.
- 기억 실험실은 체험 놀이터처럼 읽히게 재구성한다. 주요 섹션은 **우주 키우기**(별/다감정 하루 띄우기), **시간 보내기**(하루/한 달), **다른 삶 보기**(페르소나/겹쳐보기)로 묶고 설명문은 짧게 줄인다.
- 체험 HUD의 계층을 명확히 한다. 회상 패널·일기/잠든 별 목록·겹쳐보기 같은 더 강한 오버레이가 열리면 기억 실험실은 닫히거나 숨겨져 서로 덮지 않는다. 이론 모달은 명시적 모달 계층에 있고, 드롭다운 목록은 그 위에 표시된다.
- 별 추가는 별도 설명 카드 없이 시각 반영을 우선한다. 별/다감정 하루를 띄우면 새 별과 새 연결이 즉시 우주에 반영되고, 짧은 강조로 "우주가 커졌다"는 감각을 준다.
- 시간 이동은 실제 야간 배치 근사로 바꾼다. `한 달 지나기`는 내부적으로 `하루 전진 → demoConsolidate()`를 30회 적용하며, 별·시냅스 개수는 변하지 않는다.
- 시간 이동 중 스프링식 요동은 보이지 않게 한다. 체험 시간 이동은 데이터 배치와 클라이언트 force-sim 재안정화를 조용히 끝낸 뒤 settled 좌표를 보여준다. 좌표는 여전히 클라이언트 force-sim에서 창발하며 서버/proto 좌표나 stable coord 노출은 만들지 않는다.

## 범위 / 비목표 (회귀 경계)

- 범위: FE 전용. 랜딩 CTA/카피, `widgets/demo-sim`, 체험 시간 이동 오케스트레이션, 체험 전용 레이아웃 settling, 관련 plan/policy 문서를 수정한다.
- 범위: 체험 우주에서만 보이는 UX·시뮬레이션 정돈. 메인 우주의 인증/멤버십/기록/회상/강화 서버 동작은 바꾸지 않는다.
- 범위: `plan/19`, `plan/15`, `policy/ux/landing`, `policy/ux/interaction`의 as-built 용어와 동작 설명을 갱신한다.
- 비목표: 랜딩 전체 비주얼 리디자인, 실서비스 야간 배치 BE 로직 변경, proto/RPC/sqlc/DB 변경, 좌표 영속화, 별도 요약 카드/튜토리얼 플로우 추가.
- 회귀 경계: 원본 일기 불변, 별·시냅스 삭제 금지, 좌표는 클라 창발, HUD는 캔버스 밖 2D DOM, FSD import 방향을 유지한다.

## 수용 기준 (인수 조건)

1. 문서와 사용자-facing 카피에서 `/` 일반 셸은 "메인 우주", 랜딩에서 진입한 demo 데이터 셸은 "체험 우주", 체험 컨트롤 패널은 "기억 실험실"로 구분된다.
2. 랜딩 히어로 1차 CTA는 "체험 우주 시작하기"이고, 이론 카드 CTA 기본 라벨은 "체험 우주에서 해보기"이며, 둘 다 `enterDemoMode()` 후 `/`로 이동한다(카드는 `?sim=<id>` 유지).
3. `DemoSimPanel`의 컨트롤러는 "우주 키우기"·"시간 보내기"·"다른 삶 보기" 중심으로 재구성되어, 별 추가와 시간 이동이 주요 액션으로 보이고 긴 설명문은 축약된다.
4. 회상 패널이 열려 있거나 일기/잠든 별 목록 오버레이가 펼쳐진 상태에서는 기억 실험실 컨트롤러가 자동으로 닫히거나 숨겨져 서로 덮지 않는다. 겹쳐보기 화면에서도 단일 우주용 체험 HUD가 함께 뜨지 않는다.
5. 감정 별 또는 다감정 하루를 띄우면 새 별과 새 시냅스가 우주에 즉시 반영되고, 별·연결이 생겼음을 시각적으로 짧게 강조한다. 별도 설명/요약 카드가 없어도 된다.
6. `하루 지나기`는 하루 전진과 `demoConsolidate()` 1회를 적용하고, `한 달 지나기`는 같은 1일 배치 단위를 30회 적용한다.
7. 시간 이동 전후 체험 우주의 별 개수와 시냅스 개수는 변하지 않는다(새 별 추가 액션 제외). 가지치기는 선 삭제가 아니라 weight/밝기 하향만 한다.
8. 시간 이동 중에는 사용자가 스프링처럼 요동치는 중간 force-sim 상태를 보지 않고, 조용히 재안정화된 최종 우주를 본다.
9. 시간 이동 구현은 서버/proto/RPC/DB를 변경하지 않고, 좌표는 여전히 클라이언트 force-sim 내부에서만 산출된다.
10. 관련 단위 테스트와 `pnpm --filter ./frontend build`, `pnpm --filter ./frontend lint`가 통과한다.
