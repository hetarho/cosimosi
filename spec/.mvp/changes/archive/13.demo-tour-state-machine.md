# 13. 변경: 데모 둘러보기 진행을 xstate 머신으로

> [plan/48.demo-guided-tour](../plan/48.demo-guided-tour.md)의 이미 구현된 동작을 바꾸는 **변경 제안**(WHAT 델타).
> 투어의 step·phase 진행을 명령형 컴포넌트 상태에서 XState v5 순수 머신으로 옮긴다. STEPS는 /create-change-job이 만드는 job에.

## 현재 (as-is)

투어 진행이 `widgets/demo-tour/ui/DemoGuidedTour.tsx` 한 컴포넌트의 명령형 상태에 흩어져 있다.

- **진행 상태**: `useState(phaseIndex)` + step remount(`key={stepIndex}`)로 phase·baseline을 리셋. step index는 `shared/lib/demo/flag.ts`의 sessionStorage(`tourStep`)에 따로 저장.
- **전이 로직**: `goNext`/`goPrev`가 "phase 단위로 움직이되 마지막 phase에서만 다음 step" 규칙을 컴포넌트 안에서 분기로 구현.
- **await 충족 판정**: `isAwaitMet(await, obs, baseline)` 스위치 + DOM 관찰 `useEffect`(popover/persona/clockDay/sidebar/explorer 변화를 rAF로 setState).
- **항해 실습 자동진행**: 별도 `useEffect`가 `sampleNav()`를 rAF로 폴링해 `navigation-input` 누적 카운터·항해 FSM 모드의 임계 도달 시 `setPhaseIndex`.
- **부수효과**: `onPhaseMode`(카메라 모드 구동)·Esc(`onExit`)·포커스 이동이 각각 또 다른 `useEffect`.
- **페이지 결합**: `HomePage`가 관찰값(popover/persona/clockDay/sidebar/explorer)·`sampleNav`·표면 open/close·모드 구동 콜백을 props로 내려, "진행 결정"과 "페이지 부수효과"가 컴포넌트와 페이지에 양분돼 있다.

> 이 변경 직전, 같은 작업 세션에서 plan 48 동작을 세 군데 직접 수정했다(아직 SSOT 미반영): ① 행동 phase(`await≠null`)의 `다음` 버튼 제거, ② 메뉴(햄버거) step 제거, ③ 시점 전환 항해 실습 구간 오버레이 투명화(딤·카드). 이 변경은 그 세 동작을 **머신 구조로 정식화·보존**하며, 구현 job이 ①②③과 머신화를 함께 plan 48/policy로 반영한다.

같은 "유한 상태"가 두 변수(컴포넌트 `phaseIndex` ↔ flag `tourStep`)와 여러 동기화 `useEffect`에 표현돼 있어, `다음`으로 행동 phase를 건너뛰면 다음 phase가 깨진 맥락(열리지 않은 팝오버)을 가리키는 류의 엣지버그가 구조적으로 가능하다 — tech/state-machines가 경고하는 "같은 논리 상태를 두 곳이 다른 변수로 표현"의 사례.

## 목표 (to-be)

투어 진행을 `widgets/demo-tour/model/tour.machine.ts`의 **XState v5 순수 머신** 하나로 단일화한다. 머신 정의는 React/DOM/three를 import하지 않는다(헌법4). 컴포넌트는 스냅샷에서 파생한 값으로 렌더하고 사용자 입력을 이벤트로 보내기만 한다.

- **step·phase를 상태로**: 각 step의 phase가 머신 상태(중첩/병렬 아님, 단순 시퀀스로 충분)다. 전이는 이벤트로만(`NEXT`/`PREV`/`EXIT` + 아래 행동 이벤트). "마지막 phase에서만 다음 step"이 분기가 아니라 상태 위상으로 표현된다.
- **게이팅이 구조가 된다(point 1)**: 행동 phase 상태는 `NEXT`를 **처리하지 않는다**. 따라서 `snapshot.can({type:'NEXT'})===false`이고 UI는 이 값만 보고 `다음`을 숨긴다. 정보 phase는 `NEXT` 전이를 가져 `다음`이 보인다. "행동 phase=행동으로만, 정보 phase=다음으로"가 가드/전이 자체로 표현된다.
- **DOM 관찰 await=이산 이벤트**: 컴포넌트/페이지의 얇은 effect가 관찰 변화를 의미 있는 이산 이벤트로 변환해 보낸다(`UI_HIDDEN`/`UI_SHOWN`, `PERSONA_OPENED`, `PERSONA_SELECTED(id)`, `TIME_OPENED`, `TIME_MOVED`, `EXPLORER_OPENED` 등). 해당 phase 상태가 그 이벤트를 받을 때만 진행한다. 변화 기준 await(persona/clock)는 진입 시점 값을 context에 담고 가드로 비교.
- **항해 실습=invoked actor**: 시점 전환 실습 phase는 머신이 `invoke`하는 `fromCallback` 액터(`navSampler`)가 rAF로 `navigation-input` 누적 카운터·항해 FSM 모드를 **내부에서** 샘플링하고, 진입 baseline 대비 임계(`spec/values.yaml` `demo_tour.*`) 도달 시 **단 한 번의 이산 이벤트**(`PRACTICE_MET`)를 보낸다. 매 프레임 React state도, 매 프레임 머신 이벤트도 만들지 않는다(헌법4 / 60fps 금지). 액터 구현은 비순수 싱글턴을 만지므로 UI 레이어가 `provide()`로 주입 — 머신 정의는 순수.
- **오버레이 투명화도 머신 메타로(point 3)**: 항해 실습 phase는 상태 태그(예: `nav-practice`)를 단다. UI는 `snapshot.hasTag('nav-practice')`로 딤(거의 투명)·코치 카드(반투명·상단 중앙)를 적용 — ad-hoc `phase.mode != null` 분기 대신 머신이 출처.
- **페이지 부수효과=액션/emit**: 카메라 모드 구동(`onPhaseMode`)·표면 open/close·UI 숨김 복구는 머신의 (provided) 액션 또는 `emit`에서 파생해 호출한다. 산발적 동기화 effect가 제거된다.
- **지속성**: flag.ts를 지속성 경계로 유지한다 — 머신은 진입 시 `getTutorialStep()`로 시작 step을 hydrate하고, step 전환 구독에서 `setTutorialStep()`로 저장(phase는 저장 안 함 — 현행과 동일). `다시 보기`=첫 step 재시작, `exitDemoMode()`=투어 상태 정리.
- **액터 배치**: 모듈 싱글턴 `tourActor`(navigationActor·focusActor와 동형). `HomePage`와 `DemoGuidedTour`가 `@xstate/react` selector·send로 같은 액터에 붙는다. demo 종료 시 stop/clear.

`tech/state-machines`의 triage에 8번째 머신으로 등재한다(Lifecycle/demo, 단계 시퀀스의 명시적 수명주기 — 승격 가치: 두 변수 불일치 제거 + 게이팅의 구조화 + 단위 테스트).

## 범위 / 비목표 (회귀 경계)

**범위**

- 투어 step·phase 진행·게이팅·항해 실습 자동진행·표면 오케스트레이션을 `tour.machine`으로 이전.
- 직전에 직접 반영한 ①행동 phase `다음` 제거 ②메뉴 step 제거 ③시점 전환 오버레이 투명화를 머신 구조(전이/태그)로 정식화·보존.
- `DemoGuidedTour`를 스냅샷-구동 렌더 + 이벤트 송신으로 슬림화(진행 결정 로직 제거).
- `tour.machine.test.ts` 순수 단위 테스트 추가.
- 구현 job이 머신화 + ①②③을 plan 48 / policy·tech로 반영.

**비목표 (안 바꾼다)**

- **새 단계·분기 콘텐츠를 추가하지 않는다.** 구조는 분기·조건부 단계가 쉽도록 두되, 이 변경에서 step 목록·문구·순서는 (메뉴 제거 외) 그대로다.
- 항해 **조작·물리·제스처 규칙 불변** — 실습은 신호를 관찰만 한다(`navigation-input`·항해 FSM 비변경).
- 스포트라이트/glow/딤의 **시각 디자인을 더 바꾸지 않는다**(이미 반영한 항해 실습 투명화 외).
- 투어 **데이터를 머신 context로 복제하지 않는다** — step/phase index·진입 baseline 비교값 같은 제어 값만(tech/state-machines §0).
- **매 프레임 React state·매 프레임 머신 이벤트를 만들지 않는다**(헌법4 / 60fps 금지).
- backend/proto/RPC/DB schema·서버 저장 없음(FE-only). 실계정 우주에는 투어·머신이 나타나지 않는다.
- plan 48의 다른 수용 기준(안내 순서·하이라이트·crash-free fallback·접근성·reduced-motion·캔버스 밖 DOM)을 약화하지 않는다.

## 수용 기준 (인수 조건)

1. 투어 step·phase 전이가 `widgets/demo-tour/model/tour.machine.ts`의 XState v5 순수 머신으로 표현되고, 머신 정의는 React/DOM/three를 import하지 않는다(헌법4).
2. 행동 phase 상태는 `NEXT`를 처리하지 않아 `snapshot.can({type:'NEXT'})===false`이고 UI가 그 값으로 `다음`을 숨긴다. 정보 phase는 `can('NEXT')===true`라 `다음`이 보인다(point 1을 구조로 고정).
3. DOM 관찰 await는 머신이 받는 이산 이벤트(`UI_HIDDEN`/`UI_SHOWN`·`PERSONA_OPENED`·`PERSONA_SELECTED`·`TIME_OPENED`·`TIME_MOVED`·`EXPLORER_OPENED` 등)로만 진행하고, 변화 기준(persona/clock)은 진입 baseline을 context에 담아 가드로 비교한다.
4. 시점 전환 항해 실습은 머신의 invoked `fromCallback` 액터가 rAF로 내부 샘플링 후 임계 도달 시 단 한 번의 이산 이벤트(`PRACTICE_MET`)를 보낸다 — 매 프레임 React state도, 매 프레임 머신 이벤트도 없다(헌법4/60fps). 항해 조작·물리·제스처는 불변(관찰만).
5. 항해 실습 phase는 상태 태그(`nav-practice`)를 노출하고, UI가 그 태그로 딤(거의 투명)·코치 카드(반투명·상단 중앙)를 적용한다(point 3을 머신 메타로 구동).
6. 메뉴(햄버거) 단계는 투어 step 목록·progress dots에 없다(point 2).
7. 카메라 모드 구동·표면 open/close·UI 숨김 복구가 머신의 액션/`emit`/노출 상태에서 파생돼 호출되며, 컴포넌트의 산발적 진행-동기화 `useEffect`가 제거된다.
8. 새로고침 시 같은 데모 세션이면 저장된 step에서 이어지고(머신이 `getTutorialStep()`로 hydrate), step 전환 시 `setTutorialStep()`로 저장된다. `다시 보기`=첫 step 재시작, `exitDemoMode()`=투어 flow·step·머신 상태 정리(plan 48 계약 불변).
9. `DemoGuidedTour`는 스냅샷 파생값으로 렌더하고 입력을 이벤트로 보내기만 하며, "어느 phase로·언제 진행"하는 결정 로직을 갖지 않는다. 머신 context에 투어 데이터를 복제하지 않는다.
10. `tour.machine.test.ts`가 `createActor`+`send`+`getSnapshot`만으로(React 없이) 진행 시퀀스와 게이팅을 단언한다: 행동 phase에서 `NEXT` 무시, 정보 phase에서 `NEXT` 진행, 각 step의 마지막 phase에서만 다음 step, 행동 이벤트 수신 시 진행, `EXIT`/마지막 단계 종료가 `free`로 수렴.
11. plan 47 자유모드 진입·페르소나 전환·시간 이동·망원경·새 별 생성·일반 실계정 우주에 회귀가 없고, plan 48의 기존 수용 기준(안내 순서·하이라이트·crash-free 중앙 fallback·접근성 `Tab`/`Enter`/`Esc`·reduced-motion·FE-only·3D 씬 안 `<Html>` 금지)이 그대로 성립한다.
12. build·lint·typecheck·steiger·vitest를 통과한다.
