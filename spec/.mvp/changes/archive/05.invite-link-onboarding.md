# 05. 변경: 초대 URL 온보딩

> [plan/41.invite-membership](../plan/41.invite-membership.md)의 이미 구현된 동작을 바꾸는 **변경 제안**(WHAT 델타).

## 현재 (as-is)

plan 41의 초대 게이트는 이미 구현되어 있다. 관리자는 `/admin?tab=invite`에서 초대 코드를 발행하고, 발행 목록에서 **코드 문자열**을 복사할 수 있다. 코드는 직교 모델(`max_uses` x `expires_at`)로 저장되고, `InviteAdminService`는 admin allowlist 뒤에 있다.

가입자는 인증 후에야 `/invite`에 도달한다. `MembershipGate`가 인증됐지만 비멤버인 사용자를 `/invite?redirect=<원래경로>`로 보내고, 사용자는 초대 코드를 직접 입력한 뒤 `RedeemInviteCode`를 호출한다. redeem이 성공하면 멤버십 쿼리 캐시를 제거하고 환영 연출 후 redirect로 입장한다.

현재 `/invite` 라우트는 `SessionGate` 안에 있다. 따라서 미로그인 사용자가 초대 코드를 포함한 URL을 열어도 초대장 표면을 먼저 보지 못하고, 곧장 `/sign-in?redirect=<원래경로>`로 리다이렉트된다. 또한 `/invite`는 `code` search param을 계약으로 다루지 않으므로, 로그인 뒤에도 코드는 자동 redeem되지 않고 수동 입력이 필요하다.

`/sign-in`은 기존 OTP·Google 로그인 화면이다. 새 계정 가입은 별도 가입 폼이 아니라 Supabase sign-in 흐름이 처리한다. 현재는 "초대장을 받았다"는 문맥이나 초대 코드 보존 상태를 표시하지 않는다. 특히 plan 01 기준으로 Google OAuth는 풀페이지 라운드트립 후 `/`로 복귀해 `?redirect`가 소실되므로, 초대 URL에서 시작한 사용자는 현재 모델로는 초대 코드를 잃을 수 있다.

## 목표 (to-be)

- 관리자는 초대 코드 목록에서 코드 문자열뿐 아니라 **초대 URL**을 복사하거나 OS/브라우저 공유 시트를 열 수 있다. 초대 URL 형식은 `/invite?code=<초대코드>`이며, 절대 URL은 현재 origin을 붙여 만든다. 예: `https://cosimosi.app/invite?code=ABCD1234`.
- "초대장 발송"의 이번 범위는 **URL 복사/공유 버튼**까지다. 서버가 이메일·문자·카카오톡 등을 실제 발송하지 않는다. 관리자가 복사한 URL을 원하는 채널에 붙여 보내는 모델이다.
- `/invite?code=<code>`는 초대장 진입 표면이 된다. 미로그인 사용자가 열면 기존처럼 바로 사인인으로 튕기지 않고, 우주 백드롭 위에 "초대장을 받았어요. 나만의 우주를 만들어볼까요?" 계열의 초대장 카피와 `회원가입하기` CTA를 먼저 본다.
- `회원가입하기`를 누르면 기존 `/sign-in` 화면으로 이동한다. redirect에는 원래 초대 URL(`/invite?code=<code>&redirect=<optional>`)을 보존해, OTP 또는 Google 로그인 후 같은 초대 코드 흐름으로 돌아온다. 풀페이지 OAuth에서도 초대 코드가 소실되지 않아야 한다. 별도 회원가입 폼이나 새로운 auth API는 만들지 않는다.
- 인증된 비멤버가 `/invite?code=<code>`에 도착하면 수동 입력 없이 `RedeemInviteCode(code)`를 자동 호출한다. 성공하면 기존 `/invite`의 환영 연출과 동일한 리듬으로 멤버십 캐시를 제거하고 redirect(없으면 `/`)로 입장한다.
- 자동 redeem이 실패하면 기존 수동 입력 UX로 내려앉는다. 실패 사유(`NOT_FOUND`·`EXPIRED`·`EXHAUSTED`·`REVOKED`)는 기존 reason copy 규칙을 재사용하고, 사용자는 코드를 수정하거나 다른 코드로 다시 시도할 수 있다.
- 이미 멤버인 사용자가 초대 URL을 열면 코드를 추가 소비하지 않고 원래 redirect(없으면 `/`)로 이동한다. 현재 `RedeemInviteCode`의 "이미 멤버면 무소비 OK" 멱등 계약을 유지하거나, 멤버십 상태 확인으로 redeem 자체를 생략해도 된다.
- `/invite`에 `code`가 없으면 기존 수동 코드 입력 페이지로 동작한다. MembershipGate가 보내는 `/invite?redirect=<원래경로>` 흐름은 유지된다.

## 범위 / 비목표 (회귀 경계)

- 범위: FE 중심 변경. `/admin` 초대 코드 탭의 URL 복사/공유 액션, `/invite` search param 계약(`code`), 초대장 공개 진입 상태, 로그인 redirect 보존, 인증 후 자동 redeem을 수정한다.
- 범위: 기존 `InviteService.RedeemInviteCode`·`GetMembershipStatus`·멤버십 캐시 제거·환영 연출을 재사용한다. 서버 DB 스키마, 초대 코드 발행 모델, 사용 횟수/만료/취소 판정은 바꾸지 않는다.
- 범위: `/sign-in`은 기존 로그인 화면과 Supabase 인증 수단을 그대로 쓴다. 초대장 문맥을 표시할 필요가 있으면 search/redirect 기반의 얕은 카피 보강까지만 허용한다. Google OAuth의 풀페이지 복귀에서도 초대 코드가 보존되도록 redirect/session storage 등 클라이언트 보존 방식을 job에서 선택할 수 있다.
- 범위: `policy/domain/access.md`, `policy/domain/admin.md`, 필요 시 `policy/ux/voice-tone.md`에 초대 URL 표면과 카피 톤을 반영한다.
- 비목표: 서버 이메일 발송, 주소록/수신자 관리, 초대장 발송 이력, 발송 성공/오픈 추적, referral/reward 그래프, 일반 사용자 초대 발행, 공개 익명 validate RPC 신설.
- 비목표: Supabase Auth 공급자나 OTP·Google 로그인 수단 자체 변경, 멤버십 인터셉터 체인 변경, invite DB 마이그레이션 변경, 초대 코드 길이·알파벳·프리셋 변경.
- 회귀 경계: 비멤버가 코어 우주 RPC를 멤버십 없이 호출할 수 없어야 한다. `/admin`은 여전히 admin allowlist 뒤에 있고, `/invite`는 redeem 표면이라 멤버십 게이트 밖이어야 한다. `redirect`는 내부 경로만 허용해 오픈 리다이렉트를 만들지 않는다.
- values.yaml 변경 없음. 새 튜닝 숫자를 도입하지 않고, 성공 후 이동·환영 연출은 기존 `invite.celebrate_ms` 등 plan 41의 값 SSOT를 재사용한다.

## 수용 기준 (인수 조건)

1. `/admin?tab=invite`의 각 발행 코드 행에는 `https?://<현재 origin>/invite?code=<code>` 형식의 초대 URL을 복사하는 액션이 있다. 기존 코드 문자열 복사·revoke·목록 표시는 회귀하지 않는다.
2. 브라우저가 Web Share API를 지원하면 초대 URL 공유 액션이 OS/브라우저 공유 시트를 연다. 지원하지 않거나 실패하면 URL 복사로 폴백한다. 서버 발송 RPC나 이메일 전송은 생기지 않는다.
3. 미로그인 사용자가 `/invite?code=<code>`에 진입하면 `/sign-in`으로 즉시 리다이렉트되지 않고, 초대장 화면을 먼저 본다. 화면에는 "초대장을 받았어요. 나만의 우주를 만들어볼까요?" 계열의 절제된 카피와 `회원가입하기` CTA가 보인다.
4. 초대장 화면의 `회원가입하기`는 기존 `/sign-in`으로 이동하며, 인증 후 돌아올 redirect에 `/invite?code=<code>`를 보존한다. redirect는 내부 경로만 통과하고 `//`·`/\`·자기 재귀 루프를 허용하지 않는다.
5. OTP 인증 또는 Google OAuth 후 사용자는 `/invite?code=<code>`로 돌아온다. 풀페이지 OAuth에서도 초대 코드가 소실되지 않는다. 인증된 비멤버라면 별도 코드 입력 없이 자동으로 `RedeemInviteCode(code)`가 호출된다.
6. 자동 redeem 성공 시 멤버십 캐시는 stale false가 남지 않도록 제거되고, 기존 환영 연출 후 redirect(없으면 `/`)로 이동한다.
7. 자동 redeem 실패 시 코드는 소비되지 않고 기존 사유별 오류 카피가 표시된다. 사용자는 같은 `/invite` 화면에서 코드를 수정하거나 다른 코드를 입력해 수동 redeem할 수 있다.
8. 이미 멤버인 사용자가 `/invite?code=<code>`를 열면 초대 코드는 추가 소비되지 않고 redirect(없으면 `/`)로 이동한다.
9. `/invite`에 `code`가 없으면 기존 수동 초대 코드 입력 페이지가 그대로 동작한다. `MembershipGate`가 보내는 `/invite?redirect=<원래경로>` 흐름은 회귀하지 않는다.
10. `/admin` 비관리자 NotFound 위장, `InviteAdminService` admin allowlist, `InviteService` auth-only, 코어 RPC membership 강제는 plan 41 수용 기준 그대로 유지된다.
11. 사용자-facing 카피는 `policy/ux/voice-tone.md`의 규칙을 따른다. 장식 이모지는 쓰지 않고, reduced-motion 환경에서도 초대장·redeem 상태의 의미가 전달된다.
12. FE build/lint와 관련 라우트·초대 흐름 테스트가 통과한다.
