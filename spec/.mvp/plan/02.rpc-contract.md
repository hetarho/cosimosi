# 02. RPC 계약 - proto + Connect 서버/전송/인증 (as-built)

> 현재 구현 기준 문서. 초기 MemoryService 5개 RPC 스텁 단계가 아니라, 이후 spec들이 누적된 실제 Connect 계약과 서버 장착 상태를 설명한다. | 범위: FE/BE contract | 상태: 구현 완료

## 목적

웹과 Flutter는 같은 protobuf 계약을 생성해 사용해야 한다. 서버는 Connect unary RPC만 제공하고, 대부분의 RPC는 Supabase JWT로 인증된다. 공개 우주 방문만 별도 `VisitService`로 분리해 무인증 표면을 구조적으로 제한한다.

## Proto 계약

- proto 소스:
  - `proto/cosimosi/v1/memory.proto`
  - `proto/cosimosi/v1/admin.proto`
  - `proto/cosimosi/v1/share.proto`
  - `proto/cosimosi/v1/gift.proto`
- Go 생성물:
  - `backend/internal/gen/cosimosi/v1/*.pb.go`
  - `backend/internal/gen/cosimosi/v1/cosimosiv1connect/*.connect.go`
- TypeScript 생성물:
  - `frontend/src/shared/api/gen/cosimosi/v1/*_pb.ts`
- `buf.yaml`은 리포 루트에 있고, 생성 템플릿은 `backend/buf.gen.yaml`이다.
- 실제 생성 명령은 루트 스크립트 `pnpm gen:proto` 또는 `pnpm buf:gen`이며, 내부적으로 Docker `bufbuild/buf:latest generate --template backend/buf.gen.yaml proto`를 사용한다.

### `MemoryService`

`MemoryService`는 현재 다음 unary RPC를 가진다. 스트리밍 RPC는 정의하지 않는다.

- `SegmentMemory`: 저장 없는 AI 조각 preview.
- `RecordMemory`: 원본 record 저장과 조각 별 fan-out 진입점.
- `GetUniverse`: 개인 우주 전체 그래프 조회. `NO_SIDE_EFFECTS`.
- `ReinforceLinks`: 공동 회상 강화 배치.
- `RecallMemory`: 별 회상과 원본/조각 텍스트 조회.
- `ListDormant`: 오래 회상되지 않은 별 목록. `NO_SIDE_EFFECTS`.
- `GetEvolutionHistory`: 별 변천사 조회. `NO_SIDE_EFFECTS`.
- `ListRecords`: 원본 일기 목록/길찾기 조회. `NO_SIDE_EFFECTS`.

핵심 메시지의 현재 형태:

- `Mood`: `MOOD_UNSPECIFIED` + 13개 값. 기존 `JOY=1`부터 `NEUTRAL=7`은 유지되고, `EXCITEMENT=8`부터 `EMPTINESS=13`이 추가되어 있다.
- `RecordMemoryRequest`: `body`, optional whole-diary hints인 `mood/intensity/valence`, `entry_date`, `idempotency_key`, user-confirmed `segments`.
- `RecordMemoryResponse`: `record_id`, `memory_ids[]`.
- `SegmentDraft`: fragment text와 mood/intensity/valence.
- `Star`: `memory_id`, fragment mood/intensity/valence, `last_recalled_at`, reshaping state, `record_id`, `fragment_index`, `resonant`, `relevance`. 좌표 필드는 없다.
- `Synapse`: `a_id`, `b_id`, `weight`, `link_type`, `last_activated_at`, `co_activation_count`.
- `GetUniverseResponse`: `stars[]`, `synapses[]`, `ambient`.
- `AmbientMood`: 최근 감정 요약 `hue/sat/arousal/valence`.

### Other Services

- `SettingsService`: 인증된 사용자별 visual override 조회/수정. `GetSettings`는 `NO_SIDE_EFFECTS`.
- `AdminService`: 관리자 콘솔. 인증 뒤 allowlist gate를 추가로 통과해야 한다.
- `ShareService`: 소유자의 공유 설정과 인증 기반 공명 bridge 조회.
- `VisitService`: 공개 우주 방문 전용 무인증 서비스. 공개 가능한 RPC만 여기에 둔다.
- `GiftService`: 별 보내기/조회/수락/거절/취소/목록/공명 정보. 전부 인증 필요.

## 서버 장착

- `backend/internal/platform/rpcserver/server.go`가 Connect 서버 플럼빙을 소유한다.
- `cmd/api/main.go`가 실제 handler들을 조립해 `rpcserver.New(...)`에 주입한다.
- 공통 보호 체인:
  - `connect.WithReadMaxBytes(256 KiB)`
  - logging interceptor
  - Supabase auth interceptor
  - recover handler
- `AdminService`는 공통 보호 체인 뒤에 admin allowlist interceptor를 추가한다.
- `VisitService`는 별도 체인으로 장착한다. read cap, logging, recover만 있고 auth interceptor가 없다.
- `/health`는 mux에 직접 등록되어 interceptor를 타지 않는다.
- 서버는 `net/http`, h2c, CORS를 사용한다. CORS 허용 헤더에는 `Authorization`이 포함된다.
- 모든 서비스는 unary RPC다. 클라이언트가 streaming을 가정하면 안 된다.

## 인증

- 인증 interceptor는 `Authorization: Bearer <jwt>`를 요구하고, 성공 시 JWT `sub`를 context의 user id로 넣는다.
- 지원 검증 방식:
  - `SUPABASE_PROJECT_URL`: Supabase JWKS로 ES256/RS256 access token 검증.
  - `SUPABASE_JWT_SECRET`: legacy HS256 shared secret 검증.
- 둘 다 없으면 protected RPC는 fail-closed로 `Unauthenticated`를 반환한다.
- `rpcserver.UserIDFromContext(ctx)`가 feature handler의 user scope 단일 진입점이다.
- `rpcserver.UserEmailFromContext(ctx)`는 verified email일 때만 admin allowlist 보조 정보로 쓴다.

## FE 전송

- `frontend/src/shared/api/transport.ts`가 단일 Connect transport를 만든다.
- 개발 모드:
  - `baseUrl = '/api'`
  - Vite proxy가 `/api` prefix를 제거해 backend root로 전달한다.
- production/preview:
  - `baseUrl = import.meta.env.VITE_API_URL`
- auth interceptor는 `getAccessToken()`을 읽어 Bearer 헤더를 붙인다.
- `transport`는 `useHttpGet: true`를 켠다. proto의 `NO_SIDE_EFFECTS` read RPC는 HTTP GET으로 나갈 수 있다.
- `memoryClient = createClient(MemoryService, transport)`를 공개한다.
- `giftTransport`는 상태 변화/만료를 최신으로 읽기 위해 `useHttpGet`을 끈 별도 transport다.
- `frontend/src/shared/api/index.ts`는 명시 export만 사용하며 generated proto 타입/서비스를 공개 API로 다시 export한다.

## Flutter 동등성 기준

- Flutter도 같은 proto 파일에서 Dart client를 생성해야 한다.
- `NO_SIDE_EFFECTS` read는 GET 가능성이 있지만, 캐시 정책은 서비스별로 분리해야 한다. 특히 gift류 상태 조회는 stale cache를 피해야 한다.
- 공개 방문은 `VisitService`만 무인증으로 호출한다. 다른 서비스에 client-side allowlist 구멍을 만들지 않는다.
- 모든 보호 RPC는 access token을 Bearer 헤더로 첨부한다.

## 수용 기준

1. `pnpm gen:proto`가 Go/TS 생성물을 현재 경로에 생성한다.
2. proto에 streaming RPC가 없다.
3. 무인증으로 protected RPC를 호출하면 `Unauthenticated`가 나온다.
4. `VisitService`는 무인증으로 호출 가능하지만, 공개 DTO에는 원본 일기/조각 텍스트/개인 id가 없다.
5. Admin RPC는 인증 후에도 allowlist 불일치 시 `PermissionDenied`다.
6. FE transport는 개발에서 `/api`, 배포에서 `VITE_API_URL`을 사용한다.
7. `GetUniverse`, `ListDormant`, `ListRecords`, `GetEvolutionHistory`, `GetSettings` 같은 read RPC는 proto idempotency와 transport 설정이 맞물린다.
