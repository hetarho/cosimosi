# 10. record-memory-ui - as-built

> 현재 작성 흐름은 "본문을 바로 저장하고 단일 별을 낙관 추가"가 아니다. 사용자는 먼저 `SegmentMemory`로 조각 미리보기를 받고, 조각/감정을 검토한 뒤 `RecordMemory`에 확정 조각을 보내며, 성공 이벤트가 query invalidate를 일으켜 동기 fan-out된 별들이 `GetUniverse`로 들어온다.

## RPC 계약
Generated `cosimosi.v1.MemoryService`의 현재 작성 관련 RPC:

```ts
SegmentMemory(SegmentMemoryRequest) -> SegmentMemoryResponse
RecordMemory(RecordMemoryRequest) -> RecordMemoryResponse
GetUniverse(GetUniverseRequest) -> GetUniverseResponse
ListRecords(ListRecordsRequest) -> ListRecordsResponse
```

중요 메시지:

```ts
interface SegmentDraft {
  text: string
  mood: Mood
  intensity: number // 0..1
  valence: number   // -1..1
}

interface SegmentMemoryRequest {
  body: string
}

interface SegmentMemoryResponse {
  segments: SegmentDraft[]
}

interface RecordMemoryRequest {
  body: string
  mood: Mood          // optional whole-diary hint
  intensity: number   // optional whole-diary hint
  entryDate: string   // YYYY-MM-DD
  idempotencyKey: string
  valence: number     // optional whole-diary hint
  segments: SegmentDraft[]
}

interface RecordMemoryResponse {
  recordId: string
  memoryIds: string[]
}
```

- review flow에서는 `RecordMemoryRequest.segments`를 비워 보내지 않는다. 사용자가 확인한 조각을 그대로 보낸다.
- `segments`가 non-empty이면 backend가 record와 fragment stars를 같은 transaction에서 만든다. 응답의 `memoryIds`는 최종 별 id 목록이다.
- `segments`가 empty인 legacy path는 async extract path다. 현재 `MemoryForm` UI의 일반 사용자 경로가 아니다.
- `RecordMemoryResponse.recordId`는 불변 원본 record id다.
- `RecordMemoryResponse.memoryIds`는 생성된 fragment star ids다. 단일 `memoryId` 응답으로 취급하지 않는다.

## API wrapper
- `segmentMemory(body)`:
  - `memoryClient.segmentMemory({ body }, { timeoutMs: 30_000 })`를 호출한다.
  - 반환 `SegmentDraft[]`에 local React key용 `id: crypto.randomUUID()`를 붙인다.
  - 서버에 아무것도 저장하지 않는다.
- `recordMemory(input)`:
  - `body`, `entryDate`, `idempotencyKey`, `segments`를 보낸다.
  - 각 fragment는 `{ text, mood, intensity, valence }`만 서버로 보낸다. local fragment `id`는 보내지 않는다.
  - 반환은 `{ recordId: res.recordId, memoryIds: res.memoryIds }`다.
- `scheduleSynapseSync(queryClient)`:
  - 기존 timer를 clear한다.
  - 8초 뒤 `universeInvalidateKey()`를 invalidate한다.
  - 목적은 별 자체가 아니라 async embedding/link job 이후 시냅스가 생겼을 시간을 반영하는 지연 refetch다.

## 클라이언트 검증과 에러 카피
- `MAX_BODY_CHARS = 4000`
  - `[...body].length`로 Go rune 수와 맞춘다.
- `MAX_FRAGMENTS = 10`
  - AI 제안은 보통 1..5개이고, 수동 추가까지 포함해 10개를 넘지 못한다.
- 빈 본문:
  - compose 단계 `SEGMENT` 전에 차단하고 `일기 본문을 입력하세요`를 표시한다.
- 너무 긴 본문:
  - `일기가 너무 길어요 — 4,000자 이내로 줄여 주세요.`
- 빈 fragment:
  - `비어 있는 조각이 있어요 — 내용을 채우거나 지워 주세요.`
- `ConnectError(Code.InvalidArgument)`는 backend sentinel text를 보고 입력 안내로 바꾼다.
- `ConnectError(Code.ResourceExhausted)`는 256KB read cap 같은 상위 제한 방어로 보고 본문 길이 안내로 바꾼다.
- 기타 segment 실패:
  - `기억을 조각내지 못했어요. 잠시 후 다시 시도해 주세요.`
- 기타 record 실패:
  - `별을 띄우지 못했어요. 잠시 후 다시 시도해 주세요.`

## compose state machine
`frontend/src/features/record-memory/model/compose.machine.ts`가 작성 흐름의 단일 상태기다. 이전 draft store와 submit hook을 대체한다.

Context:
```ts
{
  body: string
  entryDate: string // local YYYY-MM-DD
  fragments: DraftFragment[]
  submitNonce: string
  errorText: string
}
```

Events:
- `SET_BODY`
- `SET_DATE`
- `SEGMENT`
- `UPDATE_FRAGMENT`
- `ADD_FRAGMENT`
- `REMOVE_FRAGMENT`
- `BACK_TO_COMPOSE`
- `SUBMIT`

States:
- `composing`
  - body/date 입력.
  - `SEGMENT`가 body 검증을 통과하면 `segmenting`.
- `segmenting`
  - 이전 error를 지우고 `segmentMemory(body)` invoke.
  - success: `fragments = output`, `reviewing`.
  - error: `composing`으로 돌아가 `segmentErrorMessage()` 표시.
- `reviewing`
  - fragment text/mood/intensity/valence 편집.
  - `ADD_FRAGMENT`는 `MAX_FRAGMENTS` 미만에서만 동작.
  - fragment 편집/추가/삭제는 `submitNonce`를 새로 만든다. 수정된 재제출이 이전 idempotency commit으로 dedup되는 것을 막기 위해서다.
  - `BACK_TO_COMPOSE`는 fragments를 버리고 compose로 돌아간다.
  - `SUBMIT`이 body/fragments 검증을 통과하면 `submitting`.
- `submitting`
  - `recordMemory({ body, entryDate, fragments, idempotencyKey: "rec-" + submitNonce })` invoke.
  - success: analytics capture, draft reset, `emit({ type: 'submitted' })`, `composing`.
  - error: `reviewing`으로 돌아가 `recordErrorMessage()` 표시, failure analytics capture.

Selectors:
- `selectPhase`: `compose` 또는 `review`
- `selectBody`
- `selectEntryDate`
- `selectFragments`
- `selectErrorText`
- `selectIsSegmenting`
- `selectIsSubmitting`

## MemoryForm UI
- `MemoryForm`은 `composeActor` 모듈 싱글턴을 `useSelector`로 구독한다.
- 같은 actor를 desktop panel과 mobile sheet가 공유한다.
- compose phase:
  - body textarea
  - date input, 기본값은 local today `YYYY-MM-DD`
  - submit button label: `별로 분해`
  - segmenting 중 body/date를 잠근다. 클릭 시점 본문과 preview 결과가 어긋나는 것을 막는다.
- review phase:
  - fragment list
  - 각 fragment는 mood dropdown, intensity range, text textarea, delete button을 가진다.
  - mood 변경 시 `MOOD_AFFECT[mood].valence`도 같이 갱신한다.
  - `조각 추가` 버튼은 10개 cap을 지킨다.
  - submit button label: `별 띄우기 (N)`
  - submitting 중 fragment 입력을 잠근다.
- mood option은 13개 proto `Mood`를 quadrant order로 보여준다.
- body/fragment textarea에는 `ph-no-capture` class가 붙는다. analytics autocapture가 원문을 건드리지 않게 하기 위한 이중 가드다.
- 데모 모드에서는 실서버 작성 `MemoryForm`을 숨긴다. 자유모드의 하단 `새 별 띄우기` 버튼은 서버 작성 폼 대신 `demoAddRandomStars`로 체험 별을 즉시 만든다.

### 작성 진입점 (change 09)
- `MemoryForm`은 우주 셸의 **하단 중앙 floating 버튼 `새 별 띄우기`**(lucide `Plus`)가 여는 비차단 `Surface`로 뜬다(`composeOpen`). 구 햄버거 사이드바/메뉴에는 작성 항목이 없다 — 작성은 이 floating 버튼이 단일 진입점이다.
- 데모 모드에서는 같은 버튼이 서버 작성 폼을 열지 않고 체험 별을 즉시 만든다(서버 쓰기 없음).

## 성공 후 동기화
- `composeMachine`은 queryClient를 직접 import하지 않는다. feature model은 React context/app layer를 모른다.
- `frontend/src/app/App.tsx`가 root에서 `composeActor.on('submitted')`를 구독한다.
- submitted 수신 시:
  - `queryClient.invalidateQueries({ queryKey: universeInvalidateKey() })`
  - `queryClient.invalidateQueries({ queryKey: recordsInvalidateKey() })`
  - `scheduleSynapseSync(queryClient)`
- `universeQueryOptions()` refetch가 성공하면 `applyUniverse(res)`가 store에 병합한다.
- 이 흐름에서는 temp star를 local store에 직접 추가하지 않는다.
  - 사용자 확정 조각은 backend에서 동기 fan-out된다.
  - 별은 성공 직후 invalidate/refetch로 들어온다.
  - 시냅스는 embedding/link job이 끝난 뒤 지연 invalidate에서 들어온다.

## GetUniverse 적용
- `universeQueryOptions()`는 connect-query key와 `transport`를 사용한다.
- demo mode에서는 backend 대신 `demoStars()`/`demoSynapses()`를 같은 query path로 태운다.
- cache:
  - `staleTime = 5분`
  - `gcTime = 30분`
  - stale 상태에서 window focus refetch 허용.
- `applyUniverse(res)`:
  - `res.stars.map(mapStar)`를 `mergeStars()`로 병합한다.
  - 비어 있는 우주가 성공적으로 로드되면 `loadedEmpty = true`로 표시한다.
  - `res.ambient`가 있으면 server summary를 쓴다. 없으면 loaded stars에서 `deriveAmbient()`를 수행한다.
  - `res.synapses.map(toSynapseEdge)` 후 `lastActivatedAt`/`brightness`를 보강하고 `mergeEdges()`로 병합한다.
- query failure:
  - `HomePage`가 첫 로드 실패에 한해 error card와 retry button을 보여준다.
  - 기존 data가 있는 background refetch 실패는 마지막 우주를 유지한다.

## ListRecords 적용
- `recordsQueryOptions()`는 `MemoryService.method.listRecords`를 connect-query로 읽는다.
- 성공한 record write 이후 `recordsInvalidateKey()`를 invalidate해 일기 목록에 새 record가 뜬다.
- records는 immutable original list다. `RecordSummary`는 excerpt/date/star_count만 들고, 전체 원문은 recall path에서 가져온다.

## transport/auth
- `memoryClient`는 `frontend/src/shared/api/transport.ts`의 `transport`를 사용한다.
- dev baseUrl은 `/api`이고 Vite proxy가 backend root로 rewrite한다.
- prod/preview baseUrl은 `VITE_API_URL`이다.
- auth interceptor는 `getAccessToken()`을 읽어 있으면 `Authorization: Bearer <token>`을 붙인다.
- `transport`는 `useHttpGet: true`다. proto `NO_SIDE_EFFECTS`가 붙은 read RPC는 GET으로 나간다.
- `keepaliveFetch`는 recall reinforcement flush 같은 tiny unary request가 page teardown 이후에도 도달하게 한다.
- gift 전용 `giftTransport`는 상태 변화 캐시 문제 때문에 `useHttpGet`을 끈 별도 transport다.

## 현재 사용자 흐름
1. 사용자가 `/universe`에 들어간다.
2. `HomePage`가 `GetUniverse`와 settings를 query로 불러온다.
3. 사용자가 `MemoryForm`에 본문과 날짜를 입력한다.
4. `별로 분해`를 누르면 `SegmentMemory`가 preview 조각을 반환한다. 아직 저장되지 않는다.
5. 사용자가 조각 text/mood/intensity를 수정하거나 추가/삭제한다.
6. `별 띄우기 (N)`을 누르면 `RecordMemory`가 확정 조각을 저장한다.
7. 성공하면 form이 reset되고 `submitted` event가 app root로 올라간다.
8. app root가 universe/list records를 invalidate한다.
9. refetch된 `GetUniverse`가 `recordId`와 `memoryIds[]`에 해당하는 fragment stars를 렌더 store에 병합한다.
10. 8초 뒤 추가 universe invalidate가 실행되어 async embedding/link 결과인 synapse를 반영한다.

## 테스트로 고정된 동작
- `compose.machine.test.ts`
  - 초기 상태는 composing + 빈 본문.
  - 유효 본문 `SEGMENT`는 segmenting 후 reviewing으로 간다.
  - 빈 본문 `SEGMENT`는 composing 유지 + error.
  - `SUBMIT` 성공은 composing reset + `submitted` emit.
  - analytics capture는 reset 전 context로 실행되어 `fragment_count`를 잃지 않는다.
  - 빈 fragment submit은 reviewing 유지 + error.
  - submit 실패는 reviewing 유지 + server error copy.
  - fragment edit은 text를 갱신하고 nonce를 새로 만든다.

## Flutter 동기화 포인트
- Flutter도 two-step compose/review/submit 흐름이어야 한다. AI segment preview를 보지 않고 바로 저장하면 web과 다른 제품 경험이 된다.
- `RecordMemoryResponse.recordId`와 `memoryIds[]`를 기준으로 처리한다. 단일 memory id 응답으로 모델링하지 않는다.
- UI 일반 경로는 confirmed `segments`를 non-empty로 보낸다.
- 성공 후 local temp star를 직접 넣는 것이 아니라, universe/list records refetch로 서버 확정 상태를 받아 병합한다.
- 같은 draft를 desktop/mobile equivalent surfaces가 공유해야 한다. Flutter에서도 sheet/panel 전환이 draft를 잃으면 안 된다.
- idempotency key는 draft lifecycle과 fragment edit에 맞춰 갱신해야 한다.
- body/fragment 원문은 analytics/autocapture 대상에서 제외한다.
