# 21. 기억 분할 - 1일기 → N조각 별 fan-out (as-built)

> 현재 구현 기준 문서. 일기 한 편을 사건 경계로 나눠 각자의 감정을 가진 N개의 조각 별로 띄우는 fan-out 경로의 SSOT다. 원본 일기는 한 글자도 안 바뀌는 단 1개의 불변 record로 남는다. | 범위: FS | 상태: 구현 완료, 반영: 28 wayfinding · 36 공명 · 39 작성 머신

## 목적

하루는 여러 감정을 품는다 — 아침의 평온, 낮의 분노, 밤의 안도. 사람의 기억도 사건 경계에서 이산적 흔적으로 끊겨 부호화된다(event segmentation). 21 이전 우주는 "1 일기 = 1 record = 1 memory = 1 별"이었다. 이 스펙은 그 단일 별을 깨서, 일기 1편을 20번 스펙의 `ai.Extractor`로 N개의 조각으로 나누고 각 조각을 자기 감정·강도·정동가를 가진 별로 fan-out한다. 같은 일기에서 나온 조각끼리는 강한 일내 시냅스(w=0.8)로 묶는다(within-event binding). 감정은 사용자가 고르는 것이 아니라 AI가 조각마다 감지하고, 수동 입력은 선택적 fallback 힌트로만 남는다.

조각별 감정·텍스트는 **가변 별(memories) 레이어**에 둔다 — 불변 record에 조각 데이터를 두면 헌법1 위반이기 때문이다. record는 사용자가 쓴 1개의 원본 그대로 보존된다.

## 현재 구현

### 두 fan-out 경로, 하나의 코어

조각 fan-out에는 두 진입점이 있고, 둘 다 `backend/internal/db/fragment/fragment.go`의 `FanOutTx`를 공유한다 — 그래프 토폴로지가 두 경로에서 절대 어긋날 수 없다.

- **동기 경로(현재 작성 UI의 1급 경로).** 사용자가 `SegmentMemory`로 조각 preview를 받고 검토·수정해 확정하면, `RecordMemory`가 그 확정 조각을 record와 **같은 트랜잭션**에서 fan-out한다. 응답의 `memory_ids`가 즉시 채워진다. (preview/review/submit RPC 계약은 04·10 소관; 21은 그 확정 목록을 record와 한 tx로 fan-out하는 부분을 소유한다.)
- **비동기 extract 경로(legacy/backup).** `RecordMemory`에 `segments`가 비면 record 1행 + `extract` job 1개만 적재하고 별은 비동기로 도착한다. 현재 `MemoryForm` 일반 사용자 경로는 아니지만(별은 즉시 도착하는 동기 경로를 쓴다) 계약·워커는 유효하게 살아 있다.

`FanOutTx(ctx, q, recordID, userID, segs)`는 caller의 트랜잭션 핸들 위에서: N개 `InsertMemory` + 조각당 `EnqueueEmbedJob` 1개 + 2개 이상이면 모든 조각 쌍의 일내 링크(`BatchUpsertIntraEntryLinks`, w=0.8)를 만들고 조각 별 id 목록을 반환한다. caller가 트랜잭션과 멱등 펜스를 소유한다.

### extract 워커 (`backend/internal/job/worker.go`)

비동기 경로의 워커 파이프라인. `processOne`은 `extract → embed → consolidate` 순으로 claim한다 — 조각이 있어야 embed가 생기므로 extract를 먼저 비운다.

`handleExtract(j)`:
1. `FragmentIDs(record_id)`로 이미 fan-out됐는지 **먼저** 확인 — crash-before-Complete로 재클레임된 job이 (유료) LLM 추출을 다시 돌리지 않게 short-circuit.
2. `GetRecordForExtract(record_id)`로 불변 body·user_id·entry_date·수동 힌트를 읽는다.
3. `extractor.Extract(ctx, rec.Body)` → `[]ai.Segment`.
4. `applyManualHint(toSegments(...), rec)`로 보정 후 `FanOutFragments(record_id, user_id, segs)` 호출.
5. `Complete(j)`.

`handle(j)`(embed 단계, 05 소관이지만 21이 입력을 조각 텍스트로 바꿈): `GetMemoryForEmbed`가 돌려준 조각 텍스트(`fragment_text`, NULL이면 `r.body`)로 **조각마다 따로** 임베딩 → `UpsertEmbedding` → `KnnNearest` → 의미 링크. 의미 링크 weight는 일내 결속 아래로 캡된다(`semanticWeightCap = 0.79`, 아래 "가중치" 절).

`applyManualHint`는 추출이 **단일-중립-조각 폴백 형태로 강등됐을 때만** 수동 힌트를 적용한다(조각 1개 ∧ valence 0 ∧ mood가 빈 값/neutral). 실제 다조각·정동 감지 결과는 힌트가 덮어쓰지 않는다.

### 수동 힌트 fallback

`RecordMemoryRequest.mood/intensity/valence`는 선택적 전체-일기 힌트다. 비면 무시되고 AI 감지가 권위다. `records`에 nullable로 저장되고(전체-일기 prior), `GetRecordForExtract`가 `RecordForExtract.HintMood/HintIntensity/HintValence`로 읽어 `applyManualHint`에 넘긴다. `valence=0`은 "미설정"으로 NULL 저장(`valenceToDB`), 추출/확정된 조각의 0은 진짜 값으로 저장(`fragment.f32ToDB`)된다.

### 멱등 · 안전

- record 멱등: `records (user_id, idempotency_key)` 부분 UNIQUE. 재호출 시 기존 record_id + `ListMemoryIDsByRecord`로 이미 fan-out된 조각 id들을 반환, 새 행 미생성.
- fan-out 멱등: `FanOutFragments`는 tx 안에서 `ListMemoryIDsByRecord`로 기존 조각을 먼저 보고 있으면 no-op으로 기존 id 반환. 동시 이중 실행(lease 만료 후 두 워커)은 `memories` UNIQUE 인덱스가 23505로 거부 → 패자는 retry storm 대신 멱등 경로(기존 id)로 전환된다(`isUniqueViolation`).

### 입력 검증 (service, 17)

records는 append-only라 쓰기 전 검증이 유일한 방어선이다. `Service.RecordMemory`가 본문(빈 값 `ErrEmptyBody`·4000 runes `ErrBodyTooLong`)·intensity `[0,1]`·valence `[-1,1]`(NaN 거부)을 검증하고, 확정 조각도 각각 빈 값(`ErrEmptySegment`)·길이(`ErrSegmentTooLong`)·범위·개수(`MaxSegments=10` 초과 시 `ErrTooManySegments`)를 검증한다. 핸들러가 전부 `InvalidArgument`로 매핑한다.

## 마이그레이션 (00004_memory_fragmentation.sql)

`backend/internal/db/migrations/00004_memory_fragmentation.sql` — 00001은 수정 금지(append-only DDL)라 신규 파일. (00002=30 personalization, 00003=34 admin-console.)

- `memories`에 조각 컬럼 추가: `mood TEXT`(AI 감지, nullable) · `intensity REAL`(0~1, nullable) · `fragment_index INT NOT NULL DEFAULT 0`(0-based) · `fragment_text TEXT`(임베딩용 조각 텍스트, NULL이면 `r.body` fallback) · `valence REAL DEFAULT 0`(−1..1, 26이 λ_eff에 소비).
- `records`에 `valence REAL` 추가(수동 힌트 prior — `mood`/`intensity`와 같은 결).
- **백필:** 21 이전 1:1 별의 `mood`/`intensity`를 `records`에서 `memories`로 옮긴다(`m.mood IS NULL`인 행만) — GetUniverse가 `m.mood`만 읽어도 색이 비지 않게.
- **jobs 키잉 재설계:** `jobs.memory_id`를 nullable로(`DROP NOT NULL`) + `record_id TEXT REFERENCES records(id)`(extract job 키) + `user_id TEXT`(27 consolidate job 키) 추가. 이제 embed job=`memory_id`, extract job=`record_id`, consolidate job=`user_id`로 키잉된다.
- **이중 fan-out 펜스:** `CREATE UNIQUE INDEX memories_record_fragment_idx ON memories (record_id, fragment_index)`. 같은 일기의 같은 조각은 정확히 1행. 기존 1:1 행은 전부 `fragment_index=0`이라 위반 없음.

## proto 계약 (cosimosi.v1, proto/cosimosi/v1/memory.proto)

- `RecordMemoryResponse{string record_id=1; repeated string memory_ids=2;}` — 21 이전의 단일 `memory_id` 필드를 제거하고 두 필드로 교체. 동기(확정 조각) 경로면 `memory_ids` 채워짐, 비동기 경로면 빈 배열(조각은 다음 `GetUniverse` refetch로 도착, 헌법6) — 멱등 재호출에서만 이미 fan-out된 id들로 채워진다.
- `RecordMemoryRequest`: `mood`(2)·`intensity`(3)·`valence`(6)는 선택적 전체-일기 힌트(주석으로 강등). `segments`(7, `repeated SegmentDraft`)는 확정 조각 목록(04·10 review flow). (`SegmentDraft`·`SegmentMemoryRequest/Response`·`SegmentMemory` RPC 자체는 04 소관.)
- `Star`에 `valence`(필드 12, double) 추가 — 렌더·물리 입력. (mood enum 13종 정의는 29 소관; 21은 그 값을 memories에 영속만.) Star 필드 번호 배치: 1–4 base · 5–8 재성형(23) · 9–10 wayfinding(28) · 11 resonant(36) · 12 valence(21) · 13 relevance(26).

## 가중치: 일내 vs 일간 (policy/domain/synapse.md)

- **일내(intra_entry):** 같은 `record_id`의 모든 조각 쌍을 `weight=0.8`, `link_type='intra_entry'`로 결속(`BatchUpsertIntraEntryLinks`, `link.sql`). a<b는 DB 콜레이션(`LEAST`/`GREATEST`)에서 정규화. `ON CONFLICT … GREATEST`로 재실행이 약화시키지 않는다.
- **일간(cross-entry semantic):** 05의 `w0 = clamp(α·cos + temporal, 0, 1)`를 쓰되 워커가 `min(w0, semanticWeightCap=0.79)`로 캡한다(`worker.go`). 같은-사건 결속이 항상 가장 굵게 읽히도록. KNN 후보 풀은 `candidateK = knnK*2 = 16`, 흥분성 재랭크(22) 뒤 최종 `biasedK = 5`개 링크가 남는다 — 일간 링크 수는 8이 아니라 5다.

값 SSOT는 spec/policy/domain/synapse.md(가중치)·memory.md(조각 모델·fan-out·키잉)에 있다. 위 수치는 코드에서 검증한 현재값이다.

## FE 작성 흐름 (features/record-memory, compose 머신 — tech/state-machines.md)

현재 흐름은 XState 작성 머신 `model/compose.machine.ts`가 소유한다(tech/state-machines.md) — 구 `draft-store.ts`(phase×status)와 `use-record-memory.ts`(async 오케스트레이션)를 한 FSM으로 합쳤다.

- 상태: `composing`(본문 입력) → `SEGMENT`("별로 분해") → `segmenting`(invoke `segment` 액터) → `reviewing`(조각·감정 확인/수정) → `SUBMIT`("별 띄우기") → `submitting`(invoke `submit`) → `composing`(리셋 + `emit('submitted')`).
- `reviewing`에서 조각 추가/삭제/수정 가능(`MAX_FRAGMENTS=10`). 조각 편집은 매번 `submitNonce`를 새로 굴려, 내용 바뀐 재제출이 옛 커밋으로 dedup되지 않게 한다.
- 별은 **즉시** 태어난다(동기 fan-out). `submitted` 이벤트에 페이지가 우주/일기목록 쿼리를 무효화하고, `scheduleSynapseSync`가 8초 뒤 한 번 더 refetch해 임베딩 비동기로 생긴 시냅스를 받는다.
- `api/record-memory.ts`: `segmentMemory(body)`(저장 없는 preview, 30s 타임아웃) · `recordMemory(input)`(확정 조각 fan-out) · 에러 카피 매핑(서버 sentinel ↔ 한국어). 데모 분기는 여기 없다(체험 모드는 자유모드 컨트롤이 `shared/lib/demo` 경로로 별을 만든다).
- `entities/memory/model/store.ts`: 단건 `addStar`/`replaceStar`/`removeStar` + 묶음 `addStars(nodes[])`/`replaceMany(tempIds[], nodes[])`. 서버 별은 절대 삭제하지 않는다(헌법2). `entities/memory/api/map-star.ts`의 `mapStar`가 `valence`·`recordId`·`fragmentIndex`를 스레딩하고, 조각 seed는 `seedFromId(memory_id)`로 결정론 파생(새로고침 후에도 같은 모양).

## Flutter 동등성 기준

- fan-out은 **서버 권위**다. Flutter는 자체 분할을 하지 않고 `SegmentMemory` preview → review/edit → `RecordMemory(segments)` 흐름과 응답의 `record_id`/`memory_ids` 분리(원본 일기 vs 조각 별)만 재현하면 된다.
- 동기 경로면 `memory_ids`가 즉시 채워지고, segments 없는 비동기 경로면 빈 배열 + 다음 `GetUniverse`로 도착한다 — 두 도착 모델을 모두 다뤄야 한다.
- 조각 별의 `mood`/`intensity`/`valence`는 record가 아니라 별(memories)에서 온다. `Star.valence`(필드 12)를 읽어 렌더·물리에 쓴다.
- 같은 `record_id`를 공유하는 별들을 일기 단위로 그룹할 수 있어야 하고(헌법3 — 좌표는 서버에 없으니 force-sim은 클라 몫), 조각 별 seed는 `memory_id`로 결정론 파생해야 새로고침 후 같은 모양을 유지한다.
- 일내(0.8)/일간(≤0.79) 가중치 차이는 서버가 부여하므로 Flutter는 `Synapse.weight`·`link_type`을 그대로 시각에 반영하면 된다.

## 수용 기준

1. 여러 장면·감정의 일기로 record를 만들면 불변 `records` 1행만 쓰고, fan-out이 조각마다 `memories` 1행(공유 `record_id`, 0-based `fragment_index`, 조각 `fragment_text`·`mood`·`intensity`·`valence`)을 생성한다.
2. 조각마다 `kind='embed'` job이 적재되고, embed 워커는 `COALESCE(fragment_text, body)`로 조각별로 따로 임베딩한다.
3. 같은 일기에서 2개 이상 조각이 나오면 모든 조각 쌍이 `link_type='intra_entry'`, `weight=0.8`로 결속되고, 일간 의미 링크는 항상 0.79 이하로 캡된다.
4. 조각마다 감정이 다르면 각 별이 자기 `mood`/`intensity`/`valence`를 가지며, `GetUniverse`의 `Star.mood`/`intensity`/`valence`는 `memories`에서 읽힌다(records JOIN 아님).
5. 한 일기에 대해 `records`는 정확히 1행이고 그 행을 UPDATE/DELETE하지 않는다(조각화는 가변 별 레이어에서만).
6. 사용자가 감정을 직접 안 골라도 AI가 조각 감정을 채운다. `RecordMemoryRequest`의 mood/intensity/valence는 선택적 힌트이며, 추출이 단일-중립-조각으로 강등됐을 때만 fallback으로 적용된다.
7. 확정 조각(review flow)을 보내면 record와 같은 트랜잭션에서 fan-out돼 `memory_ids`가 즉시 채워진다. segments 없는 비동기 경로면 extract job만 적재되고 `memory_ids`는 비며 조각은 다음 `GetUniverse` refetch로 도착한다.
8. 같은 `idempotency_key` 재호출은 새 record를 만들지 않고 기존 record_id + 이미 fan-out된 조각 id들을 반환한다.
9. fan-out은 멱등하다 — 이미 조각이 있는 record의 extract job은 LLM을 다시 부르지 않고 기존 id로 short-circuit하며, 동시 이중 실행은 `(record_id, fragment_index)` UNIQUE가 막는다.
10. 조각 별의 seed는 `memory_id`로 재현 가능하게 파생되어 새로고침 후에도 같은 별 모양을 보인다.
11. 좌표는 서버에서 만들지 않으며 조각 별 위치는 클라 force-sim에서 창발한다(서버는 가중치 그래프만 권위).
12. 도메인/proto/sqlc 경계가 유지된다 — `db/fragment`·`job`·`memory` 도메인 타입에 db/proto 태그가 없고, fan-out 코어는 `entities/memory/model`·`features/record-memory/model`이 three/React/DOM을 import하지 않는다.
13. 체험 모드에서 다감정(빈 줄 문단) 일기를 작성하면 색이 다른 N개 별로 갈라지고 조각끼리 0.8 일내 선으로 묶인다. 랜딩 `FragmentationCard`가 활성 시 1편→3별 분할 모션을 보인다.
