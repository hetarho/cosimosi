# 03. 데이터 스키마 - pgvector + goose + sqlc (as-built)

> 현재 구현 기준 문서. `00001`은 초기 기반 마이그레이션이고, 현재 sqlc/Flutter/API 판단의 스키마 스냅샷은 누적 `backend/internal/db/schema.sql`이다. | 범위: BE data | 상태: 구현 완료

## 목적

영속 계층은 원본 일기, 조각 별, 임베딩, 시냅스, job queue, 설정, 관리자 LLM 구성, 공유/선물/공명 데이터를 Postgres + pgvector에 저장한다. 원본 `records`는 불변이고, 별/시냅스는 삭제 대신 상태/가중치/밝기 입력값만 변화한다.

## SSOT 규칙

- goose 마이그레이션이 적용 순서의 단일 출처다.
- `backend/internal/db/schema.sql`은 sqlc 입력용 누적 스냅샷이다. 마이그레이션 추가 시 이 파일도 반드시 동기화한다.
- `backend/sqlc.yaml`은 migration directory가 아니라 `internal/db/schema.sql`을 schema 입력으로 사용한다. goose Down의 DROP문을 sqlc가 누적 DDL로 해석하는 문제를 피하기 위해서다.
- `00001_engram_schema.sql`은 기반 테이블을 만든다. 이후 `00002`부터 `00008`까지가 현재 런타임 스키마를 확장한다.
- 로컬 Postgres 이미지는 `pgvector/pgvector:pg16`.
- `docker compose up`만으로 마이그레이션은 적용되지 않는다. 표준 실행은 `pnpm db:migrate`다.
- `platform/postgres/postgres.go`는 pgx pool `AfterConnect`에서 pgvector 타입을 등록한다.

## 현재 테이블 요약

### Core Memory

- `records`
  - 원본 일기 1행. `id`, `user_id`, `body`, `entry_date`, `created_at`.
  - `mood`, `intensity`, `valence`는 전체 일기 단위의 선택적 hint다. 조각의 실제 감정은 `memories`에 있다.
  - `idempotency_key`는 `(user_id, idempotency_key)` partial unique로 보호된다.
  - 원칙: records에 대한 UPDATE/DELETE 쿼리를 만들지 않는다.
- `memories`
  - 조각 별 1행. `record_id`는 non-unique FK라 1 record가 N memories를 가질 수 있다.
  - 조각별 `mood`, `intensity`, `valence`, `fragment_index`, `fragment_text`를 가진다.
  - `last_recalled_at`은 활성도/밝기 계산의 raw 입력이다.
  - 재공고화 상태: `brightness_offset`, `hue_shift`, `form_seed_delta`, `version`.
  - `stable_x/y/z`는 야간 재안정화의 재진입 cache다. proto로 보내는 좌표 권위가 아니다.
  - `(record_id, fragment_index)` unique index가 fan-out 멱등성을 돕는다.
- `embeddings`
  - `memory_id` PK, `user_id`, `embedding vector(1536)`, `model`.
  - HNSW `vector_cosine_ops` 인덱스와 `user_id` 인덱스가 있다.
- `memory_links`
  - 무방향 시냅스. PK `(a_id, b_id)`, `CHECK (a_id < b_id)`.
  - `user_id`, `weight`, `link_type`, `co_activation_count`, `last_activated_at`.
  - semantic, intra-entry, co-recall, nightly pruning 모두 행을 삭제하지 않고 값만 조정한다.
- `processed_batches`
  - `ReinforceLinks` batch id 멱등성. batch id PK.

### Jobs

- `jobs`
  - `kind`는 현재 `extract`, `embed`, `consolidate`를 사용한다.
  - `status`: `pending`, `running`, `done`, `failed`.
  - `next_run_at`, `attempts`, `error`, `updated_at`으로 retry/backoff를 관리한다.
  - keying:
    - extract job: `record_id`, `user_id`, `memory_id NULL`.
    - embed job: `memory_id`, `record_id/user_id`는 비어 있을 수 있다.
    - consolidate job: `user_id`, `memory_id/record_id NULL`.
  - `jobs_claim_idx(status, next_run_at)`가 claim을 돕는다.
  - 사용자별 active consolidate job은 partial unique index로 중복 적재를 막는다.

### Evolution / Settings / Admin / Sharing

- `evolution_history`
  - 별 재성형 스냅샷 append-only log.
  - `memory_id`, `user_id`, `version`, `brightness`, `hue_shift`, `form_seed_delta`, `trigger`, `pe`, `dir`, `created_at`.
- `user_settings`, `user_emotion_colors`
  - 사용자별 visual override만 저장한다. 기본값은 클라이언트가 소유한다.
- `llm_provider_configs`, `llm_selection`, `llm_usage_daily`
  - 관리자 콘솔의 LLM provider/model/key override와 usage 집계.
  - API key는 암호문과 last4만 저장한다.
- `universe_shares`
  - 공개 우주 slug, enabled flag, display name.
  - 원본 일기/조각 텍스트는 저장하지 않는다.
- `star_gifts`
  - 별 보내기 token, sender/recipient, 상태, 만료, 메시지.
- `resonances`
  - 보낸 별과 수락으로 태어난 별의 공명 관계. 삭제하지 않는다.

## 마이그레이션 흐름

- `00001_engram_schema.sql`: records, memories, embeddings, memory_links, jobs, processed_batches, vector extension.
- `00002_user_settings.sql`: visual settings override tables.
- `00003_admin_llm.sql`: admin LLM provider/selection/usage tables.
- `00004_memory_fragmentation.sql`: memories fragment columns, records valence hint, nullable job keys.
- `00005_reconsolidation.sql`: evolution history and reshaping columns.
- `00006_nightly_consolidation.sql`: stable coordinate cache.
- `00007_universe_shares.sql`: public universe sharing.
- `00008_star_gifts.sql`: star gifts and resonances.

## Public Data Contracts

- `records.body`는 원본 전체 일기이며, private RPC에서도 필요할 때만 반환한다.
- `memories.fragment_text`는 조각 별의 텍스트다. 공개 우주 DTO에는 포함하지 않는다.
- `Star.record_id`/`fragment_index`는 개인 우주 길찾기용이다. 공개 DTO에는 개인 식별자가 없다.
- 좌표는 DB 권위가 아니다. `stable_x/y/z`는 서버 내부 재안정화 cache이며, 클라이언트 렌더 좌표는 force-sim에서 창발한다.
- `mood` 문자열은 proto enum name과 도메인/FE palette key에 맞춰 해석한다. 현재 mood 체계는 13개 값이다.

## Flutter 동등성 기준

- Flutter는 서버 proto 응답을 그대로 렌더 입력으로 사용하고, 좌표/밝기/두께는 클라이언트 모델에서 계산해야 한다.
- Flutter는 원본 일기와 조각 별을 구분해야 한다. 일기 목록은 record 단위, 우주 렌더는 memory 단위다.
- 공개 방문 화면은 private schema 개념을 노출하지 말고, `VisitService` DTO만 사용해야 한다.

## 수용 기준

1. `pnpm db:migrate` 후 schema에는 위 테이블과 주요 인덱스가 존재한다.
2. `backend/internal/db/schema.sql`과 최신 goose Up DDL이 동기화되어 sqlc가 현재 쿼리를 생성할 수 있다.
3. `records`를 수정/삭제하는 쿼리가 없다.
4. `memory_links`는 DB collation 기준 `a_id < b_id`로 정규화된다.
5. `jobs`는 extract/embed/consolidate 각각의 nullable keying을 지원한다.
6. `embeddings.embedding`은 `vector(1536)`이고 HNSW cosine 인덱스를 가진다.
7. 공개 공유/방문 관련 테이블에는 원본 일기 본문이나 조각 텍스트 공개용 컬럼이 없다.
