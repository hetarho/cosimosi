# 17. error-handling - FE 에러 바운더리·렌더러 폴백 + BE 입력 검증·요청 한도·패닉 복구 (as-built)

> 현재 구현 기준 문서. 어떤 경로로 깨져도 흰 화면을 보여주지 않고, 쓰기 경로를 빈 일기·거대 페이로드·패닉으로부터 지키는 실패-처리 경계의 SSOT다. | 범위: FS | 상태: 구현 완료, 반영: FE 에러 표면화는 compose.machine(tech/state-machines.md)·18 Sentry 보강

## 목적

records는 불변·영구(헌법 §1)라 잘못 들어온 입력을 사후에 정리할 경로가 없다 — 쓰기 전 검증이 유일한 방어선이다. 동시에 베타 사용자에게 R3F 렌더 크래시·렌더러 불가·없는 경로가 흰 화면으로 끝나면 안 된다. 그래서 이 스펙은 두 가지를 구현한다: ① FE에서 바깥→안 3겹 에러 바운더리와 렌더러 init 실패의 명시적 안내, ② BE에서 service 레이어 입력 검증·요청 body 상한·HTTP 타임아웃·Connect/워커 패닉 복구. 검증 정책은 전송 무관하게 service에 두고(테스트 가능), 매핑만 handler가, 한도/복구는 rpcserver가 맡는다(§4.8 #7 핸들러 얇게).

UX 값(폴백 카피·바운더리 계층표·검증 메시지 매핑)의 SSOT는 `spec/policy/ux/error-feedback.md`다.

## 현재 구현 — FE 바운더리 3겹

`policy/ux/error-feedback.md`의 계층표가 값 SSOT다. 바깥일수록 넓게, 안쪽일수록 좁게 복구한다.

1. **전역 `Sentry.ErrorBoundary`** — `frontend/src/app/App.tsx`. `RouterProvider`를 포함한 트리 전체를 감싼다. fallback은 `GlobalErrorScreen`("문제가 생겼어요" + 새로고침 = `window.location.reload()`). 라우터 폴백조차 못 그린 크래시의 마지막 그물.
2. **라우터 폴백** — `frontend/src/app/router.tsx`의 `createRouter` 옵션:
   - `defaultErrorComponent: RouteErrorScreen` — 라우트 로드/렌더 실패("이 화면을 불러오지 못했어요" + 다시 시도(`reset()`) + 처음으로).
   - `defaultNotFoundComponent: NotFoundScreen` — 없는 경로("이 좌표에는 아무것도 없어요" + 내 우주로/처음으로).
3. **캔버스 전용 `Sentry.ErrorBoundary`** — `frontend/src/pages/home/ui/HomePage.tsx`. `<UniverseCanvas/>`만 감싼다(겹쳐보기 분기에선 `<UniverseOverlay/>`도 동일 fallback으로 감쌈). fallback은 `CanvasErrorFallback`. 바운더리가 Canvas에만 걸려 있어 우주가 죽어도 형제인 HUD(작성 폼·회상 패널·내비)는 살아 있다 — 일기를 쓰는 코어 가치는 렌더러 없이도 동작한다.

- Sentry init이 안 된 환경(로컬 DSN 없음)에서도 폴백 렌더는 동일하게 동작한다 — 캡처만 no-op이다. DSN 유무로 분기하지 않는다.
- fallback은 모듈 레벨 컴포넌트로 전달한다(`GlobalErrorScreen`/`CanvasErrorFallback`). 인라인 화살표를 주면 Sentry가 함수를 element type으로 써서 부모 리렌더마다 폴백이 리마운트되고 재시도 버튼 포커스를 잃는다.
- 에러 후 라우터 상태를 신뢰할 수 없으므로 폴백 화면의 이동은 `<Link>`가 아니라 평범한 `<a>`(전체 리로드)다.

### 폴백 화면 컴포넌트

- `frontend/src/app/ui/ErrorScreens.tsx` — `GlobalErrorScreen`, `RouteErrorScreen`(둘 다 `Shell`(GlassCard) 위에 `errorMessage(error)` 표시). `NotFoundScreen`은 여기서 재export만 한다.
- `frontend/src/shared/ui/NotFoundScreen.tsx` — 라우터 notFound와 34의 비관리자 `/admin` 위장이 함께 쓰므로 shared/ui 소유.

## 현재 구현 — 렌더러 불가 폴백

- `frontend/src/shared/lib/r3f/renderer.ts`
  - `createRenderer(props)`는 `three/webgpu`의 `WebGPURenderer`를 만들고 `await renderer.init()` 한다. WebGPU가 없으면(`'gpu' in navigator` 거짓) `forceWebGL: true`로 WebGL2 경로를 바로 탄다(불필요한 콘솔 경고 회피). three r184의 `init()`이 WebGPU 백엔드 실패 시 WebGL2로 자체 폴백하므로, `catch`는 WebGL2까지 실패한 진짜 불가 환경에서만 도달한다 — 그때 `RendererUnavailableError`를 던진다(원인은 `cause`에 보존).
  - `rendererBackend(r)`는 실제 선택된 백엔드(`'webgpu' | 'webgl2'`)를 반환한다(18 analytics와 DEV 로그가 사용).
- `frontend/src/widgets/universe-canvas/ui/UniverseCanvas.tsx` — **렌더러 init 실패 표면화.** R3F는 async `gl` 팩토리의 reject를 fire-and-forget으로 삼켜 바운더리에 전달하지 않는다. 그래서 `glFactory`가 `createRenderer(props).catch`로 실패를 `initError` state에 담고 다시 throw하며(R3F 내부 진행도 멈춤), 렌더 본문 상단의 `if (initError != null) throw initError`가 이를 **렌더 중 throw**로 바꿔 페이지의 캔버스 바운더리에 넘긴다.
- `CanvasErrorFallback`(`pages/home/ui/HomePage.tsx`)이 원인을 구분한다:
  - `error instanceof RendererUnavailableError` → "이 브라우저/기기에서는 우주를 그릴 수 없어요" + 브라우저/그래픽가속 안내 + "일기 작성은 그대로". 재시도 버튼 없음(재시도 무의미).
  - 그 외 → "우주를 불러오지 못했어요" + `errorMessage(error)` + 다시 시도(`resetError` → 캔버스 리마운트).
- 쿼리 실패(connect-query 데이터 계층)와 캔버스 크래시(17)는 `UniverseErrorCard` chrome을 공유한다(같은 파일).

## 현재 구현 — BE 입력 검증

검증은 `backend/internal/memory/service.go`의 `RecordMemory`·`SegmentMemory`에 있다(전송 무관·단위 테스트 가능). sentinel은 `backend/internal/memory/memory.go`에 선언한다.

- sentinel 7종: `ErrEmptyBody`, `ErrBodyTooLong`, `ErrIntensityRange`, `ErrValenceRange`, `ErrEmptySegment`, `ErrSegmentTooLong`, `ErrTooManySegments`.
- 상수: `MaxBodyRunes = 4000`(본문·조각 텍스트 공통 캡), `MaxSegments = 10`(확정 조각 목록 상한; AI는 최대 5개 제안, 검토 단계의 수동 추가까지 합친 상한).
- `RecordMemory` 검증 순서: 빈 body(`strings.TrimSpace == ""`) → body rune 길이(`utf8.RuneCountInString > MaxBodyRunes`) → intensity NaN/범위 `[0,1]` → valence NaN/범위 `[-1,1]` → segments 개수(`> MaxSegments`) → 각 segment 텍스트 trim·빈값·길이·intensity·valence. NaN은 두 비교에 모두 false라 명시적으로 거부한다(바이너리 protobuf double은 NaN을 실어 보낼 수 있음). 통과하면 `entry_date` 기본값(빈값 → UTC today) 후 `repo.RecordMemory`로 위임.
- `SegmentMemory`도 같은 body 캡을 적용한다 — 미리보기가 record가 거부할 일기를 받아선 안 된다.

### 상수 정렬(거울)

`MaxBodyRunes`는 임베더 입력 캡과 한 잣대다 — 더 길면 임베딩 전에 잘려 별의 의미 좌표가 꼬리를 무시하므로, 자르기 전에 거부한다.

- `backend/internal/ai/openai.go` `maxInputRunes = 4000`(주석에 17 거울 명시).
- FE `frontend/src/features/record-memory/api/record-memory.ts` `MAX_BODY_CHARS = 4000`, `MAX_FRAGMENTS = 10`.

## Error Mapping

handler(`backend/internal/memory/handler.go`)는 sentinel을 Connect 코드로만 매핑한다(정책은 service).

- `RecordMemory`: 7종 sentinel 전부 → `CodeInvalidArgument`. `parseEntryDate` 실패 → `CodeInvalidArgument`. 그 외 → `CodeInternal`. 인증 컨텍스트 없음 → `CodeUnauthenticated`.
- `SegmentMemory`: `ErrEmptyBody`·`ErrBodyTooLong` → `CodeInvalidArgument`; extractor 실패 → `CodeUnavailable`(저장 없는 재시도 안전 미리보기); 인증 없음 → `CodeUnauthenticated`.
- `RecallMemory`: `ErrNotFound` → `CodeNotFound`.
- 요청 메시지가 `maxRequestBytes`(256KB)를 넘으면 핸들러 검증 전에 Connect가 `CodeResourceExhausted`로 거부한다.

⚠️ sentinel의 **메시지 문구**는 FE가 substring 매칭해 한국어 카피를 고르는 계약이다(`record-memory.ts`). 문구를 바꾸면 그 매핑이 조용히 깨진다 — `service_test.go`의 `TestValidationSentinelTextPinned`가 `"body is empty"`·`"exceeds max length"`·`"intensity"` 문구를 핀으로 고정한다.

## 현재 구현 — 요청 한도·HTTP 타임아웃

`backend/internal/platform/rpcserver/server.go`.

- `maxRequestBytes = 256 << 10`(256 KiB) — 모든 서비스 핸들러에 `connect.WithReadMaxBytes`로 적용. 최대 정당 페이로드(RecordMemory body ≤4000 runes ≈16KB UTF-8)에 넉넉하고, 1GB Lightsail에서 크고 느린 POST DoS·임베딩 비용 증폭을 차단한다(클라 64KB keepalive flush 캡과도 정합).
- `http.Server` 타임아웃: `ReadHeaderTimeout = 10s`, `ReadTimeout = 30s`, `WriteTimeout = 30s`, `IdleTimeout = 120s`. unary 핸들러는 ms 단위로 끝나므로 큰 값은 트래픽이 아니라 남용만 묶는다.
- h2c 래퍼는 `&http2.Server{IdleTimeout: idleTimeout}`을 따로 둔다 — Read/WriteTimeout은 x/net이 BaseConfig에서 per-stream deadline으로 가져가 h2c 스트림에도 닿지만, IdleTimeout은 전파되지 않아 stream-less h2c 연결을 위해 미러한다.
- 공개 `VisitService`(35)는 auth 인터셉터 없는 별도 체인이지만 `WithReadMaxBytes(maxRequestBytes)`는 동일하게 받는다 — 무인증 표면도 256KB로 묶인다.

## 현재 구현 — 패닉 복구(Connect 인터셉터 + 워커)

### Connect recover 인터셉터 + PanicCapture 훅

`backend/internal/platform/rpcserver/recover.go`.

- `type PanicCapture func(ctx context.Context, procedure string, p any)` — 복구된 RPC 패닉을 에러 트래커로 전달하는 훅. 컴포지션 루트(cmd/api)가 Sentry 구현을 주입하고, nil이면 로그 전용. 훅 주입 덕에 rpcserver는 인프라 전용으로 남는다(Sentry를 직접 import하지 않음).
- `newRecoverHandler(logger, capture)`는 `connect.WithRecover` 콜백을 만든다: 패닉 **값과 스택**을 slog로 기록(`debug.Stack()`), capture 훅이 있으면 호출, 클라에는 절대 값을 누출하지 않고 `connect.NewError(connect.CodeInternal, errors.New("internal error"))`만 반환.
- `server.New`의 모든 서비스 핸들러 opts에 `connect.WithRecover(newRecoverHandler(...))`가 들어 있다(공개 VisitService 포함). Sentry DSN 유무와 무관하게 패닉이 프로세스를 죽이지 않는다.
- 배선(`backend/cmd/api/main.go`): `sentryEnabled`일 때만 `panicCapture` 클로저를 만들어 request-scoped hub(`sentry.GetHubFromContext`)로 `hub.RecoverWithContext`를 호출하고 `rpcserver.New(..., panicCapture)`에 넘긴다. 패닉은 이제 connect 안에서 복구돼 sentryhttp wrap까지 전파되지 않으므로(RPC 경로), 이 훅이 Sentry로 가는 유일한 길이다. sentryhttp wrap은 /health 같은 비-RPC mux 핸들러 패닉과 request-scoped hub 부착을 위해 남는다.

### 워커 잡 처리 recover

`backend/internal/job/worker.go`.

- `safeHandle(ctx, j)`가 `handle`/`handleExtract`/`handleConsolidate`를 `defer recover()`로 감싼다. 패닉 시 잡 id/kind/memory/record/패닉값/스택을 slog로 기록하고 `err = fmt.Errorf("job panic recovered: %v", p)`로 평범한 에러로 환원한다 → `processOne`이 이를 `failWithBackoff`로 넘겨 백오프 재시도/실패 처리한다. 단일 바이너리(API+워커 공유 프로세스)가 죽지 않는다.
- 스택은 `safeHandle`에서만 로깅한다(패닉 프레임이 존재하는 유일한 지점) — 영속 잡 에러에는 값만 남고, shutdown-interruption 분기가 증거를 삼키지 못한다.
- `failWithBackoff`는 attempts를 올리고 `base·2^(attempts-1)`(max 5분) 백오프로 재스케줄하며, `defaultMaxAttempts = 5` 도달 시 `StatusFailed`로 보존(삭제 금지 — §1/§2).

## 현재 구현 — FE 폼 에러 표면화

- `frontend/src/features/record-memory/api/record-memory.ts`
  - `recordErrorMessage(e)` — `ConnectError` + `Code.InvalidArgument`일 때 `e.rawMessage` substring으로 한국어 카피를 고른다: `'segment text is empty'`→`EMPTY_FRAGMENT_MSG`, `'segment text exceeds'`, `'too many segments'`, `'body is empty'`→"일기 내용을 입력해 주세요.", `'exceeds max length'`→`BODY_TOO_LONG_MSG`, `'intensity'`, `'entry_date'`, 그 외→"입력값을 확인해 주세요.". `Code.ResourceExhausted`(256KB 초과)→`BODY_TOO_LONG_MSG`. 그 외→"별을 띄우지 못했어요. 잠시 후 다시 시도해 주세요.".
  - `segmentErrorMessage(e)` — 분해 미리보기 실패 카피(InvalidArgument: body 빈값/길이, 그 외 재시도 안내).
  - 클라 사전 차단: `MAX_BODY_CHARS`/`MAX_FRAGMENTS` + `BODY_TOO_LONG_MSG`/`EMPTY_FRAGMENT_MSG`로 서버 왕복 전에 거른다.
- `frontend/src/features/record-memory/model/compose.machine.ts`(39 XState 머신, 옛 `use-record-memory` 훅을 대체) — `submitting` 상태의 `submit` invoke `onError`가 `recordErrorMessage(event.error)`를 `errorText`에 담고 `reviewing`으로 되돌아간다. nonce를 유지해 순수 재시도가 같은 idempotency key로 서버에서 dedup되게 한다. `segmenting` 상태도 `onError`로 `segmentErrorMessage`를 담는다.
- 현재 흐름엔 **낙관적 임시 별이 없다** — 별은 제출 성공 후 `emit('submitted')` → `App.tsx`가 universe/records 쿼리를 무효화해 refetch로만 도착한다(10의 단일 낙관 별은 21이 조각 fan-out으로 대체). 따라서 실패 시 롤백할 별이 없고, 머신은 그대로 `reviewing`에 남아 메시지만 보인다.
- `MemoryForm.tsx`는 `selectErrorText`로 `errorText`를 구독해 표시한다.

## Flutter 동등성 기준

에러 바운더리·렌더러 폴백은 웹 R3F 전용이라 직접 포팅 대상이 아니다. Flutter는 자체 렌더(Impeller/Skia) 실패와 위젯 트리 예외를 자기 방식(`ErrorWidget`/`runZonedGuarded`)으로 처리하되, "흰 화면 금지 + 다음 행동 제시" 원칙과 "코어 가치(일기 작성) 생존"은 동일하게 지켜야 한다.

포팅 대상은 BE 계약과 공유 카피다:

- RecordMemory/SegmentMemory 호출 시 `InvalidArgument` sentinel **문구**를 같은 substring 규칙으로 매핑해 같은 한국어 카피를 보여야 한다(`'body is empty'`, `'exceeds max length'`, `'intensity'`, `'entry_date'`, `'segment text is empty'`, `'too many segments'`).
- 본문 길이 사전 차단은 4000 코드포인트, 조각 수 상한은 10으로 같다(서버 거울).
- `ResourceExhausted`(256KB 메시지 초과)는 길이 안내로 매핑한다(재시도 아님).
- 검증 실패는 영구 조건으로 안내하고("무엇을 고치면 되는지"), 서버에는 아무것도 생성되지 않으므로 클라 상태도 제출 전으로 되돌린다.

## 수용 기준

1. R3F 트리 내부에서 에러가 throw되면 흰 화면 대신 캔버스 전용 폴백(`CanvasErrorFallback`)이 뜨고, 형제인 HUD(작성 폼·패널)는 그대로 동작한다.
2. WebGPU·WebGL2가 모두 불가한 환경이면 `RendererUnavailableError`가 던져지고 폴백은 "이 브라우저/기기에서는 우주를 그릴 수 없어요" 안내를 보인다(빈 캔버스 금지, 재시도 버튼 없음).
3. 일반 렌더 크래시는 "우주를 불러오지 못했어요" + 다시 시도(캔버스 리마운트)를 보인다.
4. 라우트 로드/렌더 실패는 `RouteErrorScreen`, 없는 경로는 `NotFoundScreen`을 렌더한다.
5. 라우터 폴백조차 못 그린 크래시는 전역 `GlobalErrorScreen`이 잡는다. Sentry DSN이 없어도 모든 폴백 렌더는 동일하게 동작한다.
6. 빈(공백뿐) body로 RecordMemory를 호출하면 `InvalidArgument`(`ErrEmptyBody`)를 반환하고 record/memory/job을 만들지 않는다.
7. intensity가 `[0,1]` 밖(또는 NaN), valence가 `[-1,1]` 밖(또는 NaN), body가 4000 runes 초과, 조각이 비었거나 4000 runes 초과거나 10개 초과면 각각 `InvalidArgument`를 반환한다.
8. 요청 메시지가 256KB를 넘으면 핸들러 검증 전에 거부된다(`ResourceExhausted`).
9. 핸들러 패닉은 프로세스를 죽이지 않고 값+스택을 slog로 기록한 뒤 `CodeInternal`만 반환하며, 패닉 값은 응답에 누출되지 않는다.
10. 워커 잡 처리 중 패닉은 프로세스를 죽이지 않고 해당 잡을 백오프 실패 처리(`failWithBackoff`)한다.
11. FE는 서버 `InvalidArgument`의 sentinel 문구를 substring 매칭해 해당 한국어 메시지를 폼에 표시하고, `reviewing`에 머문다(제출 전 상태 유지).
12. `MaxBodyRunes`(4000)는 `ai.maxInputRunes`와 FE `MAX_BODY_CHARS`, `MaxSegments`(10)는 FE `MAX_FRAGMENTS`와 같은 값을 유지한다.
