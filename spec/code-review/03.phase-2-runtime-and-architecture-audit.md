## Architecture Baseline
- `spec/plan/00.overview.md` is the operational roadmap: Foundation plans 01-13 are built; Epic A plans 16-24 and 28 are
  built; plans 25-27 remain forward work.
- Concept invariants still govern Phase 2: positions are derived, never persisted; memories do not directly synapse to
  memories; emotion affects appearance/field behavior rather than becoming a coordinate; graph identity remains a
  product fact, while coordinates are an ephemeral projection.
- Frontend follows FSD plus promote-on-reuse: web and mobile share product logic through `packages/*`, with app-specific
  forks only where the host primitive genuinely differs.
- The universe canvas contract from plan 23: web runs the force sim through a Worker, React Native uses the same
  `UniverseSimBridge` seam inline until a native worker primitive exists, and both platforms should share graph builder,
  machine, widget, and entity logic.
- Plan 24 adds three rendering entities (`star`, `cell-star`, `filament`) as one-way domain-to-visual projections:
  domain mirrors may be read only through `@x`, concrete bodies come through `@cosimosi/3d-renderer`'s asset-source
  port, star emotion drives color only, and filaments connect neuron slots only.
- Backend remains domain-first Go: SQL/proto shapes stay at the boundary, user scope comes from the platform transport
  layer, and dev auth must fail closed unless the pinned local user id matches.

### R001 Worker coordinate carry-over preserves array slots, not node identities
- Priority: P2
- Area: Universe sim bridge / graph identity
- Evidence:
  - `packages/universe/src/sim-bridge.ts:51-54` stores only `displayedNeurons` / `displayedMemories`, not the ids that
    owned the previous coordinate slots.
  - `packages/universe/src/sim-bridge.ts:86-96` copies the previous coordinate buffer by contiguous neuron and memory
    slices (`subarray(...)`), then repositions the copied memory slice at the new neuron offset.
  - `packages/force-sim/src/graph.ts:88-103` assigns node indices from the incoming array order.
  - The backend orders neurons by random hex id (`apps/api/db/queries/memory/universe.sql:25-44`), and ids are random
    bytes (`apps/api/internal/platform/id.go:9-17`). New neurons are created through that random id seam during launch
    (`apps/api/internal/memory/launch.go:200-218`).
  - Plan 24 now binds real semantic bodies to those slots: memories draw through `StarLayer` with
    `firstNodeIndex={neuronCount}` (`apps/web/src/widgets/universe-canvas/ui/UniverseCanvasWidget.tsx:120-127`) and
    filaments resolve endpoint slots through `nodeIndex.neurons` (`apps/web/src/widgets/universe-canvas/ui/UniverseCanvasWidget.tsx:128-132`).
- Why it matters:
  - A `GetUniverse` refetch after `LaunchStars` can insert a new neuron before existing neurons in `ORDER BY n.id`.
    The worker branch then copies slot 0 to the new slot 0 even if slot 0 now belongs to a different node. For at least
    the bridge swap window before the worker's first returned tick, existing stars/edges/click targets can visually
    belong to the wrong ids. The recent no-flicker fix avoids origin flashes, but it does so by making identity
    stability depend on array order.
- Recommendation:
  - Carry previous coordinates by stable `forceSimNodeKey(kind, id)` instead of by slot. Either keep the previous
    `ForceSimNodeIndex` beside the buffer or pass id-keyed `previousPosition` / seed hints into the new graph before
    constructing the next sim. Add a regression test where a new random-looking neuron id sorts before an existing id.
- Suggested job split: `Phase 2 graph identity and refetch stability hardening` (with R002).

### R002 Web and mobile sim bridges now diverge on refetch stability
- Priority: P2
- Area: Web/mobile parity / React Native inline sim
- Evidence:
  - The shared factory chooses worker vs inline at `packages/universe/src/sim-bridge.ts:41-43`.
  - The worker branch keeps a resized previous buffer on screen during `start(graph)` (`packages/universe/src/sim-bridge.ts:71-99`).
  - The inline branch simply rebuilds the simulation and replaces `coordinates.current` with the new simulation buffer
    (`packages/universe/src/sim-bridge.ts:143-148`).
  - Mobile always selects the inline branch because `apps/mobile/src/widgets/universe-canvas/lib/sim-worker-spawner.ts:3-9`
    returns `null`.
  - Plan 23 requires the same bridge seam and shared graph/widget logic across web and mobile, with forks only for host
    primitives (`spec/plan/23.universe-canvas.md:183-188`, `spec/plan/23.universe-canvas.md:230-241`).
- Why it matters:
  - Web tries to preserve visual continuity on a refetch; mobile reseeds/snaps to a fresh simulation. That may be an
    acceptable MVP polish reduction, but the current code does not name it as intentional, and it means the shared
    Phase 2 graph behavior differs exactly where launch/refetch interactions are most visible.
- Recommendation:
  - Move refetch carry-over into a shared, id-keyed seed/previous-position path used by both worker and inline modes.
    If mobile intentionally accepts snap-on-refetch until an RN worker lands, document that as a scoped parity exception
    in the plan/job and add a test that pins the two behaviors.
- Suggested job split: `Phase 2 graph identity and refetch stability hardening` (with R001).

### R003 Mobile dev auth uses a hardcoded user id that must be hand-synced with web/API env
- Priority: P2
- Area: Local auth/runtime config
- Evidence:
  - `apps/mobile/src/shared/config/runtime-config.ts:35-45` hardcodes `mobileDevUserId` to `'dev-user'` outside
    production and states that it does not read an env var.
  - `apps/mobile/src/app/providers/auth-provider.tsx:45-54` turns that id into a `FakeAuthAdapter` session.
  - The API verifier accepts only `fake-token-<COSIMOSI_DEV_USER_ID>` when `COSIMOSI_DEV_AUTH` is truthy
    (`apps/api/cmd/api/dev_auth.go:11-39`).
  - `.env.example:40-48` and `README.md:44-48` document the web/API ids as env-driven and required to match; mobile is
    not part of that same source of truth.
- Why it matters:
  - The user-visible failure mode is an `Unauthenticated` universe fetch in local mobile dev whenever a developer
    changes `COSIMOSI_DEV_USER_ID` / `VITE_DEV_USER_ID` away from `dev-user`. The code currently relies on a comment
    and manual sync in the exact dev-auth seam Phase 2 uses to test the real graph.
- Recommendation:
  - Add one shared dev-user source for all local clients. Options: a Metro-visible env reader, a generated local dev
    config file, or a startup diagnostic that refuses a non-default API/web override unless mobile has the same value.
    Keep the production value `undefined`.
- Suggested job split: `Dev auth and production-env guard hardening` (with R004).

### R004 `VITE_DEV_USER_ID` is guarded at runtime, despite docs promising a build-time failure
- Priority: P2
- Area: Web auth / release safety
- Evidence:
  - `.env.example:40-48` says the web build fails fast if `VITE_DEV_USER_ID` is set outside a dev build.
  - `README.md:44-48` repeats that a production build immediately errors when `VITE_DEV_USER_ID` exists.
  - The actual check is inside `createDefaultWebAuthFacade()` (`apps/web/src/app/providers/auth-provider.tsx:29-35`),
    which only runs when the provider renders.
  - `apps/web/vite.config.ts:8-16` sets plugins/envDir/ports but has no `loadEnv` or build-time guard.
  - `VITE_DEV_USER_ID=probe pnpm build:web` exited 0 during this audit.
- Why it matters:
  - CI/deploy can produce a production bundle with a dev-auth value present, and the failure becomes a runtime app
    crash instead of a release-blocking build error. The current implementation is safer than silently authenticating
    as a fake user, but it does not enforce the documented deployment invariant.
- Recommendation:
  - Move or duplicate the guard into Vite config/build setup with `loadEnv(mode, envDir, '')`, throwing during
    production builds when `VITE_DEV_USER_ID` is set. Keep the runtime assert as defense in depth.
- Suggested job split: `Dev auth and production-env guard hardening` (with R003).

### R005 The dev API image installs Air with `@latest` and has no CI build coverage
- Priority: P3
- Area: Dev infrastructure reproducibility
- Evidence:
  - `apps/api/Dockerfile.dev:5-9` builds from `golang:1.26` and runs `go install github.com/air-verse/air@latest`.
  - `apps/api/.air.toml:7-17` is the new hot-reload build loop for `pnpm dev:api`.
  - The root gate and CI run `pnpm check` (`package.json:84`, `.github/workflows/ci.yml:15-20`), but they do not build
    `apps/api/Dockerfile.dev`.
- Why it matters:
  - Local Phase 2 verification depends on a reliable hot-reload API, especially after auth/CORS/env changes. An upstream
    Air release can change behavior or minimum Go requirements and break `pnpm dev:api` without any repo diff or CI
    signal.
- Recommendation:
  - Pin Air to an explicit version in `Dockerfile.dev`, and consider adding a lightweight CI or local-script check that
    at least builds the dev image when that Dockerfile changes.
- Suggested job split: `Dev infrastructure reproducibility hardening` or bundle with the dev-auth hardening if the goal
  is one local-dev reliability job.

### R006 The new `@x` boundary exception is not covered by the boundary probes
- Priority: P3
- Area: FSD boundary enforcement / plan 24 rendering entities
- Evidence:
  - Plan 24 added an `entities-x` element and rules that permit `entities -> entities-x` while constraining `@x` files
    to their owner slice in web (`apps/web/eslint.config.js:21-31`, `apps/web/eslint.config.js:44-52`) and mobile
    (`apps/mobile/.eslintrc.js:16-26`, `apps/mobile/.eslintrc.js:33-40`).
  - The web and mobile probes still create only an `entities -> pages` violation
    (`scripts/probe-web-boundary.mjs:28-46`, `scripts/probe-mobile-boundary.mjs:28-46`).
  - The root gate runs those probes through `pnpm lint`, and they pass, but they do not prove the newly important
    `entities -> entities/@x` exception or the forbidden direct `entities -> entities/<other-slice>/model` path.
- Why it matters:
  - Plan 24 relies on `@x` as the anti-corruption seam that lets rendering entities read domain mirrors without
    importing mirror internals. If the new `entities-x` matching or dependency rules drift, the existing probes can
    still pass while rendering entities silently gain access to private domain slice internals.
- Recommendation:
  - Extend both boundary probes with two plan-24 cases: an allowed rendering-entity import through another entity's
    `@x` file, and a forbidden direct same-layer import into another entity's private `model` file. Add a case that
    proves an `@x` file can reach only its owner slice.
- Suggested job split: `Plan 24 boundary and SSOT hardening` (with R007/R008).

### R007 Plan 24 still has normative `Line2` language after the as-built switch to a custom ribbon
- Priority: P3
- Area: SSOT / rendering docs
- Evidence:

    `spec/plan/24.star-neuron-rendering.md:169-176`) and acceptance criterion 9 still says `Line2`

## Cross-Cutting Themes
- Stability fixes are currently landing at array/literal seams. Phase 2 graph behavior should prefer stable product
  identities (`kind:id`) over positions in transport or render arrays.
- Web/mobile parity is structurally strong, but runtime polish can still drift when the platform fork sits below a
  shared interface. The review surfaced this in the sim bridge rather than in FSD placement.
- Local dev is now a product-critical workflow. It should be reproducible through pinned tools and one config source,
  not comments that ask the developer to keep multiple ids in sync.
- Plan 24 made the domain-to-visual anti-corruption seam real. That seam needs both accurate normative docs and probes
  that exercise the new exception, not just a passing lint configuration.

## Questions / Tradeoffs
- Should mobile intentionally accept snap-on-refetch until an RN worker primitive exists? If yes, document it as a
  parity exception; if no, the fix should be shared id-keyed seed carry-over rather than a mobile-only patch.
- Should `Dockerfile.dev` be part of CI for every PR, only when the file changes, or only as an explicit local
  `check:dev-infra` script? The current gap is reproducibility, not the main merge gate.
- Should plan 24 docs be cleaned up inside a refactor job, or is a small docs-only follow-up enough? The runtime code
  already follows the custom-ribbon decision.
- When should the project mark mobile rendering as truly verified: source-gate parity, simulator render, or physical
  device render? The current docs use more than one standard.
