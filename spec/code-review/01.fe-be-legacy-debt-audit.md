## Architecture Baseline
- FE follows FSD import direction: `app -> pages -> widgets -> features -> entities -> shared`.
- Slices expose public APIs via `index.ts`; wildcard barrels are forbidden by the architecture doc and existing inline comments.
- Same-layer direct imports are forbidden except approved `entities/*/@x` cross-entity APIs.
- Pure FE layers (`entities/*/model`, `shared/lib`, `shared/api`, `shared/config`) must not import React, DOM, Three, or R3F. `shared/lib/r3f` is a renderer-port exception.
- BE follows package-by-feature boundaries. Cross-feature dependencies should go through explicit ports/adapters in composition roots rather than direct package cycles.
- Comments should preserve current invariants and constraints. Old change/job history, stale phase notes, and explanations that no longer match code should move back to specs/jobs or be corrected.

## Findings

### R001 Wildcard barrels remain in the shared R3F shader public API
- Priority: P2
- Area: FE / FSD public API / `shared/lib/r3f`
- Evidence:
  - `frontend/src/shared/api/index.ts:1` states that public API barrels use explicit exports and forbid `export *`.
  - `frontend/src/shared/lib/r3f/index.ts:21-22` exports the shader-art toolkit with `export * from './shader-art'`.
  - `frontend/src/shared/lib/r3f/shader-art/index.ts:3-8` re-exports six modules via wildcard: `noise`, `field`, `pattern`, `finish`, `sdf`, `geometry`.
  - `rg -n "export \\*" frontend/src --glob '!frontend/src/shared/api/gen/**'` found only those R3F shader-art wildcard exports plus the shared API comment.
- Why it matters:
  - Wildcard barrels hide public surface growth and make dead-symbol audits noisy because every new shader helper becomes public by default.
  - This directly conflicts with the repo's FSD public API rule and the local `shared/api` comment that has already encoded the no-wildcard convention.
- Recommendation:
  - Replace `shared/lib/r3f` and `shared/lib/r3f/shader-art` wildcard exports with named exports.
  - After conversion, add a lightweight grep/check to the FSD or lint gate so future `export *` usage outside generated code is caught intentionally.
- Suggested job split:
  - One FE-only refactor job: enumerate the shader-art exported symbols, convert to named exports, update imports only where needed, and add a no-wildcard verification command.

### R002 Several FE public API barrels expose more symbols than current consumers import
- Priority: P2
- Area: FE / FSD public API / dead-export candidates
- Evidence:
  - `frontend/src/entities/appearance/index.ts:3-65` exports 52 named symbols. The import trace for `@/entities/appearance` shows current consumers use a narrower set such as `useAppearance`, `backgroundMeta`, `themeAccent`, `paletteForBackground`, `buildSelfForm`, `decodeSelfSelection`, settings helpers, purchase helpers, emotion-color helpers, and a few types.
  - A TypeScript AST heuristic over `index.ts` barrels found these `@/entities/appearance` exports not imported through that public alias: `Background`, `BackgroundMeta`, `SelfObject`, `SelfObjectMeta`, `SelfSkinMeta`, `Theme`, `isBackground`, `SELF_OBJECTS`, `isSelfObject`, `parseSelfObject`, `SelfForm`, `SelfSurface`, `SelfSelection`, `SELF_PRESETS`, `DEFAULT_SELF_FORM`, `DEFAULT_SELF_SURFACE`, `DEFAULT_SELF_SELECTION`, `normalizeSelfSelection`, `SelfFormBuild`, `SELF_FORM_BUILDERS`, `SELF_SURFACE_BUILDERS`, `BackgroundForm`, `ServerInventory`.
  - `frontend/src/entities/synapse/index.ts:1-39` exports 33 named symbols. The same heuristic found unused-by-barrel-import candidates including `SynapseFilamentsProps`, `SynapseDustProps`, `VizSynapseProps`, legacy style catalog symbols, `SynapseForm`, `SynapseSurface`, `SynapseSelection`, `SynapseSkinMeta`, `SYNAPSE_PRESETS`, and default form/surface values.
  - `frontend/src/widgets/demo-tour/index.ts:1-21` explicitly says it exposes only what pages need, but the heuristic found type exports such as `DemoGuidedTourProps`, `TourStep`, `TourPhase`, `TourTargetId`, `TourSurface`, and `TourAwait` are not imported through `@/widgets/demo-tour`.
  - `frontend/src/entities/star/index.ts:17` contains an explicit transitional comment for `change 29 Phase 1`, while the same public API exports internal render builders and types at `frontend/src/entities/star/index.ts:20-31`.
- Why it matters:
  - Oversized barrels keep legacy types and props alive after callers stop using them. That makes "unused prop" and "unused helper" review depend on manual tracing instead of the public API boundary doing part of the work.
  - Type-only exports can still be legitimate API, so these are refactor candidates, not automatic deletions.
- Recommendation:
  - Prune public barrels first, slice by slice, before deleting implementation code.
  - Classify each candidate as public contract, same-slice internal helper, `@x` cross-entity contract, test-only support, or dead export.
  - Keep transitional comments only while the transition is actively tracked by a current plan/job; otherwise move the history into the spec artifact.
- Suggested job split:
  - One FE public API cleanup job for appearance/synapse/star/demo-tour barrels. Do not remove underlying model/render code until all route, dynamic, test, and `@x` uses are checked.

### R003 Focus state-machine SSOT is stale after the pair-focus extension
- Priority: P2
- Area: FE / state machines / spec-to-code drift
- Evidence:
  - Actual focus machine includes pair focus: `frontend/src/entities/memory/model/focus.machine.ts:21-25` stores `pairMyId` and `pairTheirId`.
  - The event union includes `SELECT_PAIR` at `frontend/src/entities/memory/model/focus.machine.ts:32`.
  - The root transition targets `.pair` at `frontend/src/entities/memory/model/focus.machine.ts:71`.
  - The machine states include `pair` at `frontend/src/entities/memory/model/focus.machine.ts:74`.
  - `selectPairFocus` is exported at `frontend/src/entities/memory/model/focus.machine.ts:92-95`.
  - `spec/plan/28.diary-wayfinding.md:73-74` and `spec/plan/37.universe-overlay.md:76-77` describe `idle | star | diary | pair`.
  - `spec/tech/state-machines.md:52` and `spec/tech/state-machines.md:119` still say focus only handles `idle | star | diary`.
- Why it matters:
  - `spec/tech/state-machines.md` is used as architecture baseline. A stale "only three states" invariant can cause future refactors to remove or bypass pair-focus behavior that is already implemented and planned.
  - This is exactly the kind of old or incorrect explanatory text that makes legacy cleanup risky.
- Recommendation:
  - Update `spec/tech/state-machines.md` to include `pair`, `SELECT_PAIR`, and `selectPairFocus`.
  - Keep the orthogonality distinction with shell/evolution state, but make clear that pair focus is part of focus, not shell.
- Suggested job split:
  - One documentation/SSOT refactor job. No product behavior change required.

### R004 Quality-gates SSOT still references old migration range and old BloomPass location
- Priority: P2
- Area: BE/FE verification docs / stale audit baseline
- Evidence:
  - `spec/tech/quality-gates.md:45` says current migrations are `00001_engram_schema` through `00008_star_gifts`.
  - `spec/tech/quality-gates.md:136` says goose applies migrations `00001~00008`.
  - Current migration files extend through `backend/internal/db/migrations/00014_reconsolidation_rewrite.sql`.
  - `spec/tech/quality-gates.md:66` says Bloom is at `widgets/universe-canvas/ui/BloomPass.tsx`.
  - Current consumers import Bloom from `@/shared/ui`, and the implementation is `frontend/src/shared/ui/BloomPass.tsx:20`.
- Why it matters:
  - The quality gate is supposed to be the source of truth for verification. If its migration range is stale, DB-contract review can miss newer migrations.
  - A wrong file path makes renderer/postprocessing checks harder to verify and encourages future comments to copy the stale location.
- Recommendation:
  - Update the migration range to the current highest migration or phrase it as "all files in `backend/internal/db/migrations`" to avoid repeated drift.
  - Update the BloomPass path to `shared/ui/BloomPass.tsx` and keep renderer-port wording aligned with the current import graph.
- Suggested job split:
  - One docs-only quality gate refresh job. It can be bundled with R003 if the team wants a single SSOT cleanup.

### R005 Backend share/gift composition uses a mutable post-construction cycle with no guard
- Priority: P2
- Area: BE / composition root / cross-feature ports
- Evidence:
  - `backend/cmd/api/main.go:106-112` explains that `shareResonanceAdapter` has a settable `inner` because share needs gift resonances while gift needs share display names.
  - `backend/cmd/api/main.go:113-118` stores `inner *gift.Service` and immediately dereferences `a.inner.ResonancesBetween(...)`.
  - `backend/cmd/api/main.go:211-220` constructs `shareSvc` with an empty adapter, constructs `giftSvc`, then assigns `resonanceAdapter.inner = giftSvc`.
- Why it matters:
  - The current order works, and Docker `go test`, `go vet`, and `go build` passed. The risk is maintainability: any future composition-root reordering, early request hook, or test that calls the adapter before binding can panic on a nil pointer.
  - The comments are current, but they document a fragile wiring cycle rather than removing it.
- Recommendation:
  - Prefer breaking the cycle with a smaller read-model/repository port or a composite service that owns the cross-feature query.
  - If the cycle must stay temporarily, add an explicit binder/constructor that cannot expose an unbound adapter, plus a nil guard returning an internal error and a focused wiring test.
- Suggested job split:
  - One BE refactor job for share/gift service wiring. Keep RPC contracts unchanged.

### R006 `data-theme` is live, but it only knows the old theme ids
- Priority: P2
- Area: FE / appearance naming / CSS token drift
- Evidence:
  - `frontend/src/app/RootLayout.tsx:13-16` reads `useAppearance((s) => s.theme)` and writes it to `<html data-theme=...>`.
  - The current appearance domain says the correct concept is Background, not Theme: `frontend/src/entities/appearance/model/types.ts:1-13`.
  - Current background ids are `galaxy`, `vortex`, `crystal`, and `mandala` at `frontend/src/entities/appearance/model/backgrounds.ts:25-106`.
  - `frontend/src/app/styles/color.css:12` still documents `<html data-theme="vast | lively | calm">`.
  - `frontend/src/app/styles/color.css:107-127` only has `[data-theme='vast']`, `[data-theme='lively']`, and `[data-theme='calm']` overrides. None of the current Background ids have matching CSS rules.
  - Landing repeats the same legacy axis: `frontend/src/pages/landing/ui/LandingPage.tsx:27-38` sets `data-landing-theme={theme}`, while `frontend/src/app/styles/index.css:49-103` only has specific overrides for `vast`, `lively`, and `calm`.
  - The store intentionally keeps the field name `theme` for wire/storage compatibility: `frontend/src/entities/appearance/model/store.ts:41-65`, `frontend/src/entities/appearance/model/store.ts:191-195`.
- Why it matters:
  - This was missed in the first pass because the report grouped `Theme` as a public API/deprecated alias candidate, but did not audit semantic data attributes and CSS token selectors against the current Background catalog.
  - At runtime, `data-theme="galaxy"` (and the paid background ids) will not match the old `vast/lively/calm` selectors. The default `:root` values still apply, so this can look harmless, but the dynamic theme axis is effectively stale for current product ids.
  - It also keeps the forbidden product concept "theme" visible in app-level CSS and comments, even though the domain has moved to Background.
- Recommendation:
  - Decide whether app-level CSS should still vary by background. If not, remove `data-theme`, `data-landing-theme`, and the `--theme-*` naming in favor of static neutral tokens or `--background-*` names.
  - If variation is still desired, rename the runtime attribute to `data-background` and add rules for current ids (`galaxy`, `vortex`, `crystal`, `mandala`) or derive tokens from the Background catalog instead of hard-coded legacy ids.
  - Keep BE/proto/DB `theme` wire names only where compatibility requires them, and isolate that naming at API/storage boundaries.
- Suggested job split:
  - One FE appearance-token cleanup job: remove or rename live `data-theme`/`data-landing-theme` paths, align CSS selectors with current Background ids, and update comments from Theme to Background.

## Cross-Cutting Themes
- The strongest FE debt is not obviously dead files; it is broad public API exposure that keeps old symbols reachable and makes dead-code proofs expensive.
- Some legacy concepts remain live through naming and CSS selectors even when the domain type has been renamed. `theme` needs a semantic audit, not only an unused-symbol audit.
- Comments that explain active invariants are valuable. Comments that encode old phase history or stale file locations now need the same maintenance discipline as code.
- BE compiler checks eliminated most simple unused-function concerns. The remaining backend issue is architectural wiring, not compiler-detectable dead code.

## Questions / Tradeoffs
- Public API pruning should be conservative. Some type exports may be intentionally reserved for app, tests, dynamic imports, or future platform clients; each should be classified before removal.
- Generated API exports under `shared/api/gen` and proto-derived response types were not treated as dead code even when a named import was not observed.
- The current dirty worktree includes active FE customization/star/synapse changes. Re-run the FE barrel trace after those land before deleting any implementation.
- No DB schema or proto field removal is recommended in this report. Any such cleanup should become a change proposal, not a refactor job.
