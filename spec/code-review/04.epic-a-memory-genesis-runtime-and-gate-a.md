## Architecture Baseline

The rules judgment was measured against:

- **Backend = domain-first Go.** Domain types free of proto/sqlc/pgx/JSON tags; dependencies inward
  (`rpc`/`pg` → context behavior → pure domain); thin handlers; consumer-declared ports; external services behind
  ports with a keyless-mock fallback; async work via a `jobs` table claimed `FOR UPDATE SKIP LOCKED` with
  exponential backoff (§2.4, §2.8, §2.9).
- **Frontend = FSD.** One-way imports; `entities↔entities` only via `@x`; domain-mirror vs rendering entities split
  at one seam (§3.4); control-state in XState, data in Zustand/Query; **pure cross-app code promotes _down_ to
  `packages/`, never copied sideways** (§3.1, §3.5); a `*.native` sibling only where a primitive genuinely differs.
- **Cross-cutting.** Per-user isolation on every row/query; **derived-not-stored** (positions emergent from
  force-sim, never persisted — [I5]); config is build-time (`values.yaml` → `pnpm gen:values`, never hardcoded).
- **Domain invariants.** [I2] Diary never mutated by recall; [I3] emotion → color (arousal also → size/forgetting),
  never position/synapse; [I4]/[I6] synapses only between neurons; [I5] positions emergent; [I9] LTD ≠ downscaling;
  single 1.0 strength cap; [I10] universe time monotonic.
- **Comments.** `code-comments.md`: explain current code only — no ticket/plan numbers, no process/history.

**What is solid (verified, not assumed).** The gate baseline is green and the Go suites genuinely exercise a DB.
Schema honors the invariants: no coordinate column anywhere ([I5]); `synapses` reference `neuron_a_id`/`neuron_b_id`
with `CHECK (neuron_a_id < neuron_b_id)` + `UNIQUE(user_id, …)` so a memory↔memory edge is unrepresentable
([I4]/[I6]); every product table and query is user-scoped. Golden-parity is real and _guarded_: TS reads the shared
Go fixture directly (`packages/memory-logic/.../synapse-plasticity-golden.json`) and `emotion/strength.test.ts`
asserts the two arousal-strength copies stay byte-identical, so TS↔Go drift fails the build. The AI provider seam is
well-factored (vendor errors normalized at the port, schema-forced structured output against the real SDK, idempotent
terminal writes, cost meter that actually trips). The findings below are the exceptions.

## Findings

### R001 Job-queue lease has no fencing — poison-job crash loop, lost updates, and double-charged AI calls

- **Priority:** P1
- **Area:** backend / async engine (`internal/memory` + `internal/platform/jobqueue`)
- **Evidence:**
  - `apps/api/db/queries/memory/jobs.sql:11-14` — `ClaimDueJob` sets `status='running', next_run_at=lease_until`
    but **never** `attempts = attempts + 1`; it claims `WHERE next_run_at <= now_at AND status IN ('pending',
'running')`.
  - `apps/api/db/queries/memory/jobs.sql:17-39` — `CompleteJob`/`RetryJob`/`FailJob` key only on
    `WHERE user_id = $1 AND id = $2`; no `AND status='running'` / lease-token guard.
  - `apps/api/internal/memory/pg/store.go:427-429` — lease = `AiJobBackoffBaseMs * AiJobMaxAttempts` (5 min, fixed).
  - `apps/api/db/migrations/00002_memory_aggregate_schema.sql:71-80` — `jobs` has no `locked_by`/`lease_token`/
    `version` column.
- **Why it matters:** three failure modes share one root cause (a time-only lease with no fence, no attempts-on-claim):
  1. **Poison-job infinite loop.** A handler that kills the worker (panic/OOM/SIGKILL) or overruns the 5-min lease is
     re-claimed with `attempts` unchanged, so it can _never_ reach `MaxAttempts`/`failed` — it is retried forever and
     never quarantined. Attempts advance only on a _gracefully returned_ error.
  2. **Lost update.** A lease-expired worker A can still `CompleteJob`/`FailJob` a row that worker B has re-claimed,
     clobbering B's state with no fence.
  3. **Double-charged external calls.** Two workers running the same job concurrently each spend a metered LLM/
     embedding call — defeating the cost cap. (Data corruption is avoided only because the terminal writes
     `UpsertEmbeddings`/`SaveSemanticStages` happen to be idempotent.)

  This is the async substrate every future epic (B consolidation, C recall workers) builds on. Impact is latent
  pre-deployment (mock default is free; a handler crash is rare), but the fix is cheap now and expensive to retrofit
  onto a live queue.

- **Recommendation:** add a fencing column (`locked_by`/`lease_generation`) set at claim; guard the terminal
  transitions with `AND status='running' AND lease_generation = $n`; bump `attempts` (or a separate `claims`
  counter) _on claim_ so lease-lost/crash reclaims still converge to `failed`. Reconcile the lease duration with the
  backoff schedule (see R009 note).
- **Suggested job split:** one backend job — "job-queue lease fencing + crash-safe attempts."

### R002 Epic A's write vertical is duplicated across web and mobile instead of promoted to `packages/`; the mobile half is 100% untested

- **Priority:** P1
- **Area:** frontend / FSD promote-on-reuse (§3.1, §3.5)
- **Evidence:** byte-identical (`diff` → 0 lines) in `apps/web/src` **and** `apps/mobile/src`:
  - `widgets/writing-flow/model/writing-flow.machine.ts` (pure XState, 92 lines)
  - `widgets/writing-flow/model/proposal.ts` (merge/split/dedupe algebra)

    `entities/latent-star/model/latent-field.ts` (seeded PRNG), `entities/nebula/lib/contributors.ts`,
    `entities/star/model/star-channels.ts` (+ cell-star/filament channels), and the _non-forking_ R3F UI
    `entities/nebula/ui/NebulaField.tsx`, `entities/star/ui/StarLayer.tsx`, `entities/{cell-star,filament,
latent-star}/ui/*.tsx`, `features/awaken-neuron/ui/AwakenNeuron.tsx` (three imported only via
    `@cosimosi/3d-renderer` — no DOM/RN divergence).
  - Test asymmetry: `find apps/web/src/{features,widgets,entities} -name '*.test.*'` → **10**;
    `apps/mobile/src` → **0**.
  - Precedent it ignored: `packages/universe/src/universe-navigation.machine.ts` (+ test) — the same class of pure
    control-state, correctly packaged and consumed by both apps.
- **Why it matters:** §3.5 is explicit — machines and `model`/`api` are "imported from `packages/` and shared

- **Recommendation:** promote the pure `model`/`api`/`lib` modules and the writing-flow machine to `packages/`

  `*.native`. Thin app-local Zustand stores (`draft-store.ts`, `launched-neurons-store.ts`) may stay per-app.
- **Suggested job split:** one FE job — "promote the writing-flow/awaken/star-channel pure code to `packages/` and
  wire both apps to it" (the tests move with the code).

### R003 Provider `Retry-After` hint is captured but never honored

- **Priority:** P2
- **Area:** backend / AI retry
- **Evidence:** `apps/api/internal/ai/errors.go:20-24` — `RateLimitedError.RetryAfter time.Duration` (populated by
  `anthropic/client.go:115` and `voyage/client.go:178`). The runner keys retry timing off a
  `retryAtError{ RetryAt() time.Time }` interface (`internal/platform/jobqueue/runner.go:31-32,178-188`), and the
  **only** implementer of `RetryAt()` is `CostLimitError` (`internal/ai/metering.go:27`). `RateLimitedError` does not
  implement it, so `scheduledRetryAt` never matches it.
- **Why it matters:** on a `429 Retry-After: 3600`, the worker ignores the server's hint and reschedules on its own
  ~60–120 s exponential backoff, re-hammering a throttled provider well inside the window it was told to wait — the
  opposite of the cost/limit posture the meter enforces elsewhere.
- **Recommendation:** have `RateLimitedError` implement `RetryAt()` (returning `now + RetryAfter` when the hint is
  non-zero, else fall through to backoff), or teach `scheduledRetryAt` to read `RetryAfter`.
- **Suggested job split:** fold into the R001 job (both are job-queue/AI retry correctness).

### R004 A named provider with an empty API key silently falls back to the keyless mock

- **Priority:** P2
- **Area:** backend / AI degraded-config
- **Evidence:** `apps/api/internal/ai/factory.go:119` (`if llmClient == nil && opts.LLM.APIKey != ""`) and `:130`
  (embedding) — the real-vs-mock decision gates **only** on a non-empty API key and ignores whether a provider was
  explicitly named. An unknown/empty provider _with_ a key present hard-errors (`factory_test.go:74-84`), so the
  asymmetry is the gap: key-without-provider errors, provider-without-key silently mocks. The only signal is the
  `ai_mode=…` log line (`cmd/api/memory.go:63`, `cmd/worker/main.go:55`).
- **Why it matters:** the classic "one env var of a pair unset" footgun. `COSIMOSI_LLM_PROVIDER=anthropic` with an
  unmounted/typo'd key boots happily and serves deterministic mock extractions/embeddings to real users, with no
  fatal and no warning that a named provider was downgraded.
- **Recommendation:** when a provider is explicitly named but its key is empty, fail fast (or emit a loud `warn`)
  rather than silently selecting the mock.
- **Suggested job split:** small backend job (may fold into R001/R003 as "AI seam hardening").

### R005 `InstancedNodeLayer` orphans per-instance GPU buffers on every node-count change

- **Priority:** P2
- **Area:** rendering / lifecycle (`packages/3d-renderer`)
- **Evidence:** `packages/3d-renderer/src/layers/InstancedNodeLayer.tsx:116-123` — on a count change the else-branch
  does `mesh.geometry.setAttribute(name, new THREE.InstancedBufferAttribute(...))` **without** disposing the previous
  attribute; the per-instance attributes ride the _long-lived, memoized_ `body.geometry` (disposed only on body
  change/unmount at `:104`). The sibling `ColorField.tsx:89-93` performs the identical operation and **does**
  `existing?.dispose()` first, with a comment naming exactly this leak.
- **Why it matters:** `StarLayer`/`CellStarLayer` feed `count = ids.length`; launching a diary adds 2–5 stars, so
  count changes on the **core loop**. Each change orphans the `tint`/`brightness`/`seed` buffers (3 per change),
  accumulating GPU memory over a session. Two layers doing the same thing chose opposite behaviors — one correct.
- **Recommendation:** mirror `ColorField` — `existing?.dispose()` before `setAttribute` in the count-change branch.
- **Suggested job split:** rendering job — "dispose orphaned instance buffers in `InstancedNodeLayer`" (see also
  R008/R013 rendering items).

### R008 Renderer default props duplicate — and drift from — generated config values

- **Priority:** P3

### R011 Optimistic launch flickers: the eager refetch can overwrite the just-inserted star

- **Priority:** P3
- **Area:** frontend / async (both apps)
- **Evidence:** `widgets/writing-flow/ui/WritingFlowSheet.tsx` (web ~110-118 / mobile ~105-113) calls
  `insertLaunchedMemories(...)` then immediately `queryClient.invalidateQueries({ queryKey: GetUniverse })`;
  `use-universe.ts` unconditionally `setMemories(universe?.memories ?? [])` on the refetch.
- **Why it matters:** if the encode worker hasn't indexed the new memory when the eager refetch resolves, the
  returned set excludes the optimistic star and overwrites the store — the star appears, vanishes, then reappears on
  a later read. The "surfaces on next read" pattern is intended, but firing `invalidateQueries` _immediately_ after
  the insert makes the flicker likely rather than deferred. (Related symptom of the R002 hand-copied forks: web uses
  `void queryClient.invalidateQueries(...)` while mobile uses `.catch(() => undefined)` for the same call.)
- **Recommendation:** merge optimistic memories into the refetched set (don't clobber), or defer/omit the eager
  invalidation and let the next natural read reconcile.
- **Suggested job split:** fold into the R002 FE job (same files).

### R014 `.node-version` pins a Node the gate never ran under, and only the engines floor is enforced

- **Priority:** P3
- **Area:** readiness / toolchain parity
- **Evidence:** `.node-version` = `22.22.1`; the local runtime that ran every gate here is `v25.2.1`.
  `scripts/check-node-version.mjs` enforces only the `engines.node` floor (`>=22.13`), which both satisfy — nothing
  enforces the pin. CI uses `.node-version` via `actions/setup-node`, so CI runs under 22.x while local dev can run
  under 25.x.
- **Why it matters:** a Node-major gap between local and CI means a suite can pass locally under 25 and behave
  differently under 22.22.1 (or vice-versa). Low risk here (both are green), but it is real toolchain skew and the pin
  is not enforced.
- **Recommendation:** either enforce the `.node-version` pin locally (a `check:node` that compares against the pin,

## Cross-Cutting Themes

1. **The write vertical was copied sideways, not promoted down (R002, R011, R012, R013).** _Root cause:_ the
   writing-flow/awaken/star-channel work skipped the §3.5 promote-on-reuse step that its own precedent
   (`packages/universe/universe-navigation.machine.ts`) established. Consequences cascade: forks drift (import source,
   `void`/`.catch`), mobile ships untested, and the refetch-clobber bug lives in two places.
2. **The async substrate lacks fencing (R001, R003).** _Root cause:_ a time-only lease with no ownership token and no
   attempts-on-claim, plus a provider retry hint that never reaches the scheduler. Latent pre-deployment, structural
   for every worker-driven epic to come.
3. **Config-seam and gate guards drifted from where the code actually lives (R006, R007, R008, R009, R014).** _Root
   cause:_ guards/umbrellas written for the Foundation weren't extended as Epic A added a Go backend, a renderer
   package, and new tuning — so a package goes untypechecked, Go comments go unlinted, and layout tuning + renderer
   defaults live as literals that can silently contradict `values.yaml`.
4. **Forward seams shipped ahead of consumers (R010, R013).** _Root cause:_ Epic A laid scaffolding for Epics C/D
   (effective values, shared-neuron/temporal initials, `past_dated`, awaken ceiling) that currently governs nothing
   but reads as live behavior.
