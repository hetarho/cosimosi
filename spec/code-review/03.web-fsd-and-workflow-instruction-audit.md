## Architecture Baseline
- Frontend is FSD for both web and mobile: one-way imports `app -> pages -> widgets -> features -> entities -> shared`,
  with same-layer cross-imports forbidden except `entities/@x`.
- `app` is the composition layer: entry, providers, routing, global styles. `pages` own route-level screens, `shared`
  owns domain-agnostic seams, and pure cross-app code belongs in `packages/`.
- The executable web gates currently enforce layer direction through Steiger, `eslint-plugin-boundaries`, and a negative
  `entities -> pages` probe. They do not currently enforce a required substructure inside `app/`.
- Plans are the WHAT, jobs are the buildable checklist, and `implement-job` is expected to verify acceptance criteria,
  review the diff, reflect the SSOT, mark status accurately, and archive completed jobs.
- Comments should explain current code only, not process history, tickets, phases, or decision logs.

## Findings

### R001 Web FSD is present, but the app-layer substructure is undocumented and unenforced
- Priority: P2
- Area: Frontend architecture / FSD enforcement
- Evidence:
  - `spec/ARCHITECTURE.md:272` says the frontend layers are `app -> pages -> widgets -> features -> entities -> shared`,
    and `spec/ARCHITECTURE.md:274` says this is enforced by Steiger plus `eslint-plugin-boundaries`.
  - `spec/ARCHITECTURE.md:280` to `spec/ARCHITECTURE.md:292` gives `app/` a role ("entry · providers ... router ·
    global styles") but no concrete app-layer folders.
  - The current web app composes flat app-layer files directly:
    `apps/web/src/app/App.tsx:12` to `apps/web/src/app/App.tsx:15` imports `./auth-provider.tsx`,
    `./i18n-provider.tsx`, `./observability-provider.tsx`, and `./query-provider.tsx`.
  - The same layer also keeps config flat: `apps/web/src/app/query-config.ts:6` and
    `apps/web/src/app/i18n-config.ts:18`.
  - The full gate passed with that layout: `pnpm check` ran `steiger ./src --fail-on-warnings`,
    `eslint "src/**/*.{ts,tsx}" --max-warnings=0`, and `probe-web-boundary.mjs`, all green.
  - Mobile was later specified more concretely: `spec/plan/13.mobile-app-shell.md:121` to
    `spec/plan/13.mobile-app-shell.md:124` names `app/App.tsx`, `app/providers/`, and `app/navigation/`.
- Why it matters:
  - This is not evidence that FSD is absent. Top-level `app`, `pages`, and `shared` are present and layer imports are
    guarded. The gap is narrower and more dangerous: the repo says "FSD enforced", but the enforcement only asks about
    layer direction. It never asks whether app-wide providers/config/routing are grouped consistently.
  - Agents followed the actual instructions they were given. Since no document or probe required `app/providers`,
    `app/config`, or `app/router`, flat files were a valid local minimum.
- Recommendation:
  - Decide and document the web app-layer contract explicitly, for example:
    `app/App.tsx`, `app/providers/*`, `app/config/*`, `app/router/*`, `app/styles/*` (or a deliberate smaller version).
  - Move the current flat provider/config files into that structure.
  - Add an executable guard or probe that fails on new app-layer provider/config files placed directly under `app/`
    unless they are on a small allowlist (`App.tsx`, `index.css`, maybe route root).
  - Update `ARCHITECTURE.md`, `spec/tech/test-harness.md`, and the relevant as-built plans after the move.
- Suggested job split: `web-app-layer-fsd-structure` refactor job.

### R002 The plan/job workflow loses architecture placement unless a plan already spells out exact paths
- Priority: P2
- Area: Workflow instructions / architecture transmission
- Evidence:
  - `/create-plan` asks for "Design" (`.codex/skills/create-plan/SKILL.md:27`) and acceptance criteria
    (`.codex/skills/create-plan/SKILL.md:29`), but it does not require an architecture placement table for FSD layer,
    slice, segment, package, and executable guard impact.
  - `/create-plan-job` derives tasks from the plan's Design and fills affected files from "the plan + a code grep"
    (`.codex/skills/create-plan-job/SKILL.md:25` to `.codex/skills/create-plan-job/SKILL.md:29`).
  - `/implement-job` then expects the source spec and job blast radius to match (`.codex/skills/implement-job/SKILL.md:33`
    to `.codex/skills/implement-job/SKILL.md:37`).
  - Earlier jobs and plans encoded flat web paths: `spec/jobs/archive/01.platform-foundation.md:43` says
    `src/main.tsx -> an app/ entry`, while `spec/jobs/archive/04.auth-session.md:48` lists
    `apps/web/src/app/**`; `spec/plan/04.auth-session.md:68` names `apps/web/src/app/auth-provider.tsx`.
- Why it matters:
  - The workflow is faithful but shallow. Once a plan or job names a flat path, later implementers preserve that path
    because the skills prioritize scoped blast radius and existing docs. There is no required "does this path still
    satisfy ARCHITECTURE §3.1?" checkpoint before code is written.
  - This explains why the architecture was not recognized as a placement decision: it existed as a broad SSOT rule, but
    the job checklist never translated it into concrete app-layer moves or new gates.
- Recommendation:
  - Update `/create-plan` and `/create-plan-job` instructions/templates to require an "Architecture Placement" section
    for every planned FE/BE file group:
    layer, slice, segment, package/shared boundary, owning tech doc, and guard/probe change.
  - Require jobs to include an "Architecture guard delta" task whenever the plan relies on an architectural rule that is
    not already executable.
  - Require `implement-job` to re-check generated paths against the placement table before coding.
- Suggested job split: `workflow-architecture-placement-contract` refactor job.

    follow-up job/change with revised acceptance criteria.

  - Teach `/implement-job` to run that consistency check before final report.

### R006 Comment/process-history rules are manual and already leaking into source comments
- Priority: P3
- Area: Code hygiene / instruction enforcement
- Evidence:
  - The comment principle says comments must explain current code and "Never record process or history"
    (`spec/principle/code-comments.md:7` to `spec/principle/code-comments.md:10`).
  - `apps/web/src/pages/universe/ui/UniverseHomePage.tsx:17` to

- Why it matters:
  - The codebase has good comment principles, but the enforcement is reviewer memory. As implementation volume grows,

- Recommendation:

    explicit plan owners are intentional.
  - Clean up existing source comments by rewriting process notes into timeless runtime constraints or deleting them.
- Suggested job split: small hygiene task inside `workflow-skill-template-grounding-refresh`, or a separate

## Cross-Cutting Themes
- **Architecture intent stayed prose-only.** FSD layer direction is executable, but app-layer placement and workflow
  placement review are not.
- **Path specificity beat architecture specificity.** Plans and jobs named concrete flat paths, so implementers
  preserved those paths.
- **SSOT correction happened in prose, not acceptance bars.** Plan 14 and job 17 both contain "as-built" notes while
  leaving stale or unchecked criteria in the machine-readable/checklist parts.
- **The strongest gates are negative probes.** Where a deliberate violation exists (web/mobile forbidden imports,
  raw strings), the guard is trustworthy. Where no probe exists (app-layer structure, job status consistency, stale
  skill links), drift persists.

## Questions / Tradeoffs
- Should `app/` be allowed to stay flat until it reaches a certain size, or should it be structured now for symmetry
  with mobile? The current web size is already enough to justify `providers/` and `config/`, but this is a team
  preference that should be written once.
- Should pending hardware/device verification block `status: done`, or should the repo represent it as a separate
  explicit follow-up job? The current mixed state is the part that is unsafe.
- Should comment-history lint scan only source comments, or also Markdown job/plan prose for phrases such as
  "mid-flight", "original journey", and stale phase labels?
