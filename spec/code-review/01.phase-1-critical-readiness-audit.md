## Architecture Baseline
- `spec/` is the SSOT: concept owns product laws and neuroscience guardrails, ubiquitous language owns names, and
  architecture owns placement/dependency rules (`spec/plan/00.overview.md`).
- Frontend is FSD for both web and mobile: six one-way layers, same-layer cross-imports forbidden except the `@x`
  entity public API, and the rule is expected to be enforced by Steiger plus `eslint-plugin-boundaries`
  (`spec/ARCHITECTURE.md:264-273`, `spec/ARCHITECTURE.md:375-386`).
- Shared `packages/*` are the pure cross-app core. Future domain mirrors, deterministic domain logic, cache helpers,
  state machines, i18n, config values, and transport clients will be consumed by both apps.
- Backend is domain-first Go: transport/proto and sqlc rows stay at the edge, behavior owns ports, persistence adapters
  sit under context-specific `pg` packages, and RPC handlers stay thin.
- Per-user isolation is uniform, not feature-by-feature: every persisted row carries user id and every query is scoped
  through an interceptor-injected context value (`spec/ARCHITECTURE.md:406-408`).
- State-machine context carries ids/control metadata only; data stays in Query/Zustand/refs and per-frame rendering
  never flows through React state (`spec/tech/state-machine.md`).
- UI copy goes through the i18n seam and raw user-facing strings are linted (`spec/tech/i18n.md`).
- Observability vendor SDK imports are allowed only at app/platform boundaries and guarded by
  `scripts/check-observability-boundaries.mjs` (`spec/tech/observability-and-flags.md:20-29`).

- Evidence:

### R002 CI does not run the same logical gate surface as the root Phase 1 check
- Priority: P1
- Area: CI / quality gates / shared packages
- Status: RESOLVED by job 15
- Evidence:
  - The root full gate composes lint, all package/app typechecks, all package/app tests, web/blog/api builds, API vet,

  - Root tests include values, `api-client`, `auth`, `client-cache`, `observability`, `state-machine`, `i18n`, `ui`,
    web, mobile, and API (`package.json:52-63`).
  - CI runs web lint/typecheck/test/build (`.github/workflows/ci.yml:20-23`), mobile lint/typecheck only
    (`.github/workflows/ci.yml:35-36`), blog typecheck/build (`.github/workflows/ci.yml:48-49`), API checks
    (`.github/workflows/ci.yml:62-65`), codegen (`.github/workflows/ci.yml:77`), and language lint only

  - CI does not run the mobile Jest smoke test even though `apps/mobile/package.json:8-10` exposes it and plan 13 wires
    the smoke test into `pnpm test:mobile` (`spec/plan/13.mobile-app-shell.md:164-166`,
    `spec/plan/13.mobile-app-shell.md:181-182`).
  - CI does not run root-only `lint:observability` or `lint:raw-strings` (`package.json:36-39`), nor the shared package
    test/typecheck matrix from the root scripts.
- Why it matters:
  - Phase 2 pure domain logic is expected to land in `packages/*`. A PR can currently pass CI while breaking package
    tests that are part of `pnpm check` locally.
  - Mobile provider regressions can also pass CI because `test:mobile` is absent from the mobile job.
  - The repo has two definitions of readiness: local `pnpm check` and the narrower GitHub workflow.
- Recommendation:
  - Make CI mirror the root logical gate. Either run `pnpm check` as the authoritative job, or split jobs while still
    covering every root script: shared package typechecks/tests, mobile test, raw-string lint, observability lint,
    language lint, generated freshness, API vet/build/test, and app builds.
- Suggested job split:
  - One CI hardening job: add a reusable gate matrix or explicit CI jobs so a clean CI run means the same thing as
    `pnpm check`.

### R004 The per-user persistence contract is not automated before first product tables
- Priority: P1
- Area: Backend persistence / privacy boundary
- Status: RESOLVED by job 15
- Evidence:
  - Architecture requires every persisted row to carry user id and every query to be scoped by it
    (`spec/ARCHITECTURE.md:406-408`).
  - The data-persistence plan says product tables need `user_id`, persistence methods require `platform.UserScope`,
    and cross-user queries are disallowed by default (`spec/plan/05.data-persistence.md:113-123`).
  - The helper exists and fails without an authenticated request context (`apps/api/internal/platform/user_scope.go:19-24`).
  - The probe service demonstrates a consumer-owned store interface that requires `platform.UserScope`
    (`apps/api/internal/persistenceprobe/service.go:12-14`), and the pg probe rejects an empty scope
    (`apps/api/internal/persistenceprobe/pg/store.go:25-28`).
  - The current database footprint is intentionally only the platform extension and a `SELECT 1` probe
    (`apps/api/db/migrations/00001_platform_extensions.sql:1-5`,
    `apps/api/db/queries/platform/persistence_probe.sql:1-2`); there is no SQL/migration gate that will reject a future
    product table without `user_id` or a future query that omits a user predicate.
- Why it matters:
  - This is the first real privacy/security boundary the product will need. The current helper is good API shape, but
    the first business migration can still add unscoped tables or queries unless review catches it manually.
  - RLS is explicitly deferred, so the app-level convention needs a stronger guard before product data exists.
- Recommendation:
  - Before the first product table, add an executable persistence isolation guard or test harness: product migrations
    must declare `user_id` unless explicitly platform-owned, sqlc query fixtures should prove user-scoped predicates,
    and repository adapter tests should require `platform.UserScope`.
- Suggested job split:
  - One backend privacy-gate job: add SQL/repository isolation checks and a template test for the first product
    context.

### R006 Sign-out during bootstrap is dropped, re-authenticating a session the user signed out of
- Priority: P1 (security / data isolation)
- Area: Auth session lifecycle
- Status: RESOLVED by job 14
- Evidence:
  - `packages/auth/src/facade.ts:33` — the adapter `onChange` handler begins with
    `if (disposed || !bootstrapped) return`, so every adapter event during the bootstrap window is discarded before
    the `signedOut` case (`facade.ts:36-37`) can run `epoch += 1`.
  - `bootstrapEpoch` is captured at `facade.ts:55` and `shouldApply()` (`facade.ts:57`) is
    `!disposed && bootstrapEpoch === epoch`. A dropped `SIGNED_OUT` never bumps `epoch`, so `shouldApply()` stays
    true when `adapter.bootstrap()` (`facade.ts:144`) resolves with the stored authenticated session.
  - `BOOTSTRAP_DONE` (`facade.ts:145-150`) then drives the machine to `authenticated` (`hasUser` guard,
    `session-machine.ts:72,83`). `locallySignedOut` is also never set, so the `getAccessToken` guard
    (`facade.ts:113`) does not compensate.
- Why it matters:
  - A `SIGNED_OUT` (cross-tab sign-out, server-side revocation, or expiry) that lands while the very first
    `bootstrap()` is in flight is silently lost, and the user is returned to an authenticated state they explicitly
    left. This is the product's first real privacy/isolation boundary.
- Recommendation:
  - During bootstrap, do not blanket-drop adapter events: at minimum advance `epoch` (and set `locallySignedOut`) for
    `signedOut`/`expired` so the resolving bootstrap fails `shouldApply()`, or queue the terminal event and apply it
    after `BOOTSTRAP_DONE`.
- Suggested job split:
  - One auth-correctness job covering R006–R008 in `packages/auth` with regression tests for the bootstrap/sign-out
    race, sign-out failure, and stale-token windows.

### R007 signOut() optimistically clears local state before awaiting the adapter, with no rollback on failure
- Priority: P1
- Area: Auth session lifecycle
- Status: RESOLVED by job 14
- Evidence:
  - `packages/auth/src/facade.ts:88-94` — `epoch += 1`, `locallySignedOut = true`, `suppressAdapterRefresh = true`,
    and `actor.send({ type: 'SIGN_OUT' })` (machine → `signedOut`, `getAccessToken` → null) all run before
    `await adapter.signOut()` (`facade.ts:93`). There is no `try/catch` and no rollback.
- Why it matters:
  - If `adapter.signOut()` rejects (network/Supabase error), the awaited call throws to the caller while the local UI
    already reports signed-out and the Supabase session may still be live — local state and adapter state diverge with
    no recovery path until the next bootstrap.
- Recommendation:
  - Wrap the adapter call so a rejection restores the prior session state (or retries), and surface the error without
    leaving the machine in a falsely-terminal state.

### R008 getAccessToken() returns null on expiry without forcing a refresh, sending unauthenticated RPCs
- Priority: P1
- Area: Auth adapter / transport
- Status: RESOLVED by job 14
- Evidence:
  - `packages/auth/src/supabase-adapter.ts:58-62` — `getAccessToken()` calls `client.auth.getSession()` only (never
    `refreshSession()`), then returns null when `toAuthSession()` rejects the session because
    `expiresAt <= Date.now()` (`supabase-adapter.ts:97-101`).
  - A live refresh path exists but is unused here: `refresh()` calls `refreshSession()` (`supabase-adapter.ts:53-57`).
  - The port contract promises the opposite: `auth-adapter.ts:52` documents "refreshing first if the adapter deems it
    stale".
- Why it matters:
  - In the window after `expires_at` but before supabase-js auto-refresh fires, the Connect interceptor receives null
    and the RPC goes out unauthenticated → spurious `Unauthenticated` for a call that should have refreshed first.
- Recommendation:
  - Attempt a refresh when the cached session is expired/near-expiry before returning null, honoring the documented
    adapter contract.

### R009 Partial Supabase configuration crashes the API at boot instead of degrading to auth-disabled
- Priority: P1
- Area: Backend startup / auth configuration
- Status: RESOLVED by job 14
- Evidence:
  - `apps/api/internal/platform/supabase_auth.go:108` — the degraded-mode early return only fires when
    `supabaseURL == "" && jwtSecret == ""`. With `SUPABASE_JWT_SECRET` set but `SUPABASE_PROJECT_URL`/`SUPABASE_URL`
    unset, the `&&` is false, so it proceeds to `NewSupabaseJWTVerifier` with an empty URL, which returns
    `ErrSupabaseAuthNotConfigured` (`supabase_auth.go:63-64`).
  - `apps/api/cmd/api/main.go:77-78` — `authHandlerOptions` calls `logger.Fatalf` on that error (`os.Exit(1)`),
    rather than taking the intended `!ok` auth-disabled path.
- Why it matters:
  - A common misconfiguration (secret without URL, e.g. local HS256 dev) takes the whole API down at boot instead of
    starting with auth disabled and logging "not configured".
- Recommendation:
  - Treat "any required Supabase field missing" as the unconfigured/degraded case (return `ok=false`, no error), or
    validate the trio together and emit a single clear non-fatal log.

### R011 Feature-flag memory adapter stores under a different key than it reads, silently dropping overrides
- Priority: P2
- Area: Observability / feature-flag test seam
- Status: RESOLVED by job 14
- Evidence:
  - `packages/observability/src/memory-adapter.ts:68` reads via `featureFlags.get(definition.remoteKey ?? definition.key)`,
    but `memory-adapter.ts:71` writes via `featureFlags.set(key, value)` using the raw argument.
  - The public API takes the natural key (`facade.ts:43`). Setting `'platform.diagnosticsSurface'` stores under that
    string, while reads resolve via the `remoteKey` (`'platform-diagnostics-surface'`) → miss → registry default.
  - The only passing test (`observability.test.ts:89`) happens to pass the `remoteKey` literal, masking the asymmetry.
- Why it matters:
  - Any test/dev tooling that sets a flag by its natural key gets a silent no-op, which will hide real wiring bugs and
    produce misleading green tests as the flag set grows.
- Recommendation:
  - Make set/get use the same key resolution (resolve through the definition on both sides), and update the test to
    set by the natural key.

### R012 Mobile vendor effect re-initializes Sentry on prop change without closing the prior client

### R013 Web auth and observability providers permanently dispose under StrictMode remount (dev), diverging from mobile
- Priority: P2 (dev-only behavior; signals a lifecycle anti-pattern)
- Area: Web provider lifecycle
- Status: RESOLVED by job 15
- Evidence:
  - `apps/web/src/app/auth-provider.tsx:19-28` — owned facade is `useMemo([facade])` and disposed in an unmount
    effect. `main.tsx:7` wraps `<App/>` in `<StrictMode>` and the entrypoint passes no `facade`, so the owned branch
    runs. StrictMode's dev mount→cleanup→mount disposes the facade, then reuses the same (now-disposed) memo →
    `getAccessToken()` null, `subscribe()` no-op.
  - `apps/web/src/app/observability-provider.tsx:28,48-52` — the runtime is held in `useRef` (survives remount) and the
    unmount effect disposes `runtimeRef.current.facade`; it is never recreated, so `capture*`/`track` early-return for
    the rest of the dev session.
  - Mobile avoids both by using `useState` and not disposing on unmount.
- Why it matters:
  - In dev, auth and telemetry silently die after the first StrictMode cycle, which is both a DX hazard and evidence
    that the web providers' ownership/dispose model is wrong rather than a one-off bug.
- Recommendation:
  - Adopt the mobile pattern on web (create-once via `useState`, drop dispose-on-unmount for app-lifetime singletons),
    or make `dispose()` idempotent-and-revivable. Fix both providers together as one lifecycle change (altitude).

### R015 Consent gating returns the registry default for all remote flags, disabling future kill-switches for consent-denied users

    `a.b`, `a_b`, and `a-b` all map to the same `COSIMOSI_FLAG_A_B`. No collision check exists in

- Why it matters:
  - As the flag set grows, two keys can share an env override and toggle together — including a future kill-switch.
- Recommendation:

## Cross-Cutting Themes
- The Phase 1 code is mostly seam-oriented and intentionally light on business logic, but several seams are only
  documented or locally wired, not enforced uniformly across local and CI gates.
- The highest-risk drift is between SSOT docs/scripts and as-built paths (`observability-provider`), then between local
  `pnpm check` and CI coverage.
- The backend auth and API tests are stronger than the SQL isolation automation. The `UserScope` shape is good; the
  first product table/query needs an executable guard (R004), and the probe/`UserScope` it would build on is currently
  unwired (R018).
- **Auth/session lifecycle is the densest cluster of real correctness risk** (R006–R009): a sign-out lost during
  bootstrap, no rollback on sign-out failure, and an expired-token window that skips refresh — all in the seam every
  later feature depends on.
- **Web and mobile providers diverge where they should be one implementation** (R013, R017): the same glue is copied,
  the copies already differ, and the two platforms have made opposite (and partly wrong) lifecycle choices. Promoting
  the platform-pure glue into `packages/*` fixes the duplication and the StrictMode lifecycle bug at the right altitude.
- **Feature-flag plumbing has three latent footguns** (R010, R011, R015, R016): a key-resolution asymmetry, a
  consent gate that will swallow kill-switches, an env-name collision, and a mobile surface with no enable path — none
  user-visible yet, all cheap to fix before the flag set grows.

## Questions / Tradeoffs
- Should CI run a single root `pnpm check`, or should it stay split for speed with an explicit matrix that exactly
  mirrors the root scripts?
- Should mobile adopt the same Steiger setup as web immediately, or start with `eslint-plugin-boundaries` only until
  product slices exist?
- For persistence isolation, should the first guard be SQL text lint, migration metadata, repository test templates, or
  a stricter app-layer interface pattern? A small combination is likely better than one broad regex.
- Should Node be pinned to the exact minimum (`22.13.x`) or the current active Node 22 LTS patch in a tool-version file?
- For provider lifetime objects (auth facade, observability runtime), is the intended model "create once, never
  dispose for app lifetime" (the mobile choice) or "owned + disposed"? The two apps currently disagree (R013); one
  contract should win and move into the shared package.
- Should consent gate *all* remote flags, or only analytics/experiment flags while safety (kill-switch/operational)
  flags bypass it (R015)? This decision also shapes the mobile override seam (R010).
- Is the persistence probe / `UserScope` (R018) meant to stay as a pinned convention until the first product table, or
  should it be exercised by a smoke path now so R004's isolation guard has something to build on?
