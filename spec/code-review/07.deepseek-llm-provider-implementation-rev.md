## Architecture Baseline

- **Backend placement/dependency direction:** external AI clients are outer-ring subpackages under `internal/ai`; they
  import the provider-neutral `ai` seam, while composition roots blank-import concrete providers
  (`ARCHITECTURE.md` §2.2-§2.4).
- **Structured-output contract:** `LLMRequest.OutputSchema` is first-class. A provider must return a
  schema-conforming value or typed `MalformedStructuredOutputError`; a provider that cannot honor that contract is not
  a valid slot (`plan/28` lines 47-54, 123-128, 189-191; PRD [W4a]).
- **Error/cancellation contract:** provider failures normalize to the shared typed set, but standard context
  cancellation remains observable through Go's error chain; transient provider failures feed worker backoff.
- **Defense in depth:** the AI suggests; memory use-cases execute. Let-go references are intersected with the offered
  set and approved ids are revalidated server-side ([X6]).
- **Timeless source comments and accurate as-built docs:** dated migration/history notes belong in specs or vendor
  docs, and `spec/tech` must describe the registry that actually ships.

The provider placement, endpoint ownership, authentication header, current default model, token cap propagation, and
composition-root registration are sound. The release blocker is that the adapter is registered as implemented while
not yet satisfying the seam's schema-conformance contract.

## Findings

### R001 DeepSeek is marked implemented without enforcing the requested JSON Schema

- Priority: **P1**
- Area: `internal/ai/deepseek` / `LLMClient` contract
- Evidence:
  - `apps/api/internal/ai/deepseek/client.go:113` requests only
    `response_format.type = "json_object"`.
  - `client.go:181-190` serializes the requested schema into a system-prompt sentence; it is not sent through a native
    schema field.
  - `client.go:170-178` validates only that the content unmarshals into a non-nil JSON object, not that it conforms to
    `req.OutputSchema`.
  - DeepSeek's JSON Output documentation promises valid JSON, while its actual JSON-Schema adherence is a separate
    strict-tool beta requiring the `/beta` base URL. The strict mode also does not support `minItems`/`maxItems`, which
    the semantic-stage schema uses.
  - The repo contract explicitly requires schema conformance (`spec/plan/28.ai-provider-abstraction.md:47-54`,
    `:123-128`, `:189-191`) and says a provider unable to honor it is not a valid slot (`:73-74`).
  - `client_test.go:44-68` proves only that the schema text is present and a valid object is accepted; it never sends a
    schema-violating object and expects `MalformedStructuredOutputError`.
- Why it matters: selecting DeepSeek changes `LLMClient` semantics. Current port adapters all pass a `Validate`
  callback, which catches many unusable values, and the let-go use-case independently filters foreign ids; those
  defenses reduce immediate domain risk but are not JSON Schema enforcement. They do not generally enforce
  `additionalProperties`, every `required` field, or every enum/array constraint, and future consumers are entitled to
  rely on the capability contract. A prompt claiming “match exactly” is probabilistic, not structural.
- Recommendation: do not advertise DeepSeek as implemented until one explicit strategy is chosen and tested:
  (a) use DeepSeek strict-tool mode and normalize unsupported schema keywords, accepting the beta endpoint tradeoff; or
  (b) keep stable JSON Output but validate the returned object locally against `req.OutputSchema` before returning,
  wrapping every violation in `MalformedStructuredOutputError`, and update the SSOT to describe native JSON mode plus
  local schema enforcement. Add adversarial tests for required, enum, additional-properties, and array constraints.
- Suggested job split: Job A (DeepSeek structured-output contract and completion semantics).

### R002 Non-success finish reasons can be accepted and cached as successful output

- Priority: **P2**
- Area: DeepSeek response lifecycle / metered cache
- Evidence:
  - `client.go:163-169` special-cases only `insufficient_system_resource`; `length`, `content_filter`, and
    `tool_calls` fall through to the normal JSON-object path at `:170-178`.
  - DeepSeek documents `length` as token/context exhaustion and warns JSON content may be partially cut.
  - `apps/api/internal/ai/metering.go:189-197` caches any response accepted by the consumer validator.
  - `apps/api/internal/ai/sealsuggester.go:60-74` accepts `{}` as an empty suggestion because a missing
    `candidates` field decodes as nil and is not rejected. Therefore a syntactically valid `{}` returned with
    `finish_reason=length` or `content_filter` passes validation and is cached for identical input.
  - `client_test.go:72-115` covers malformed JSON and `insufficient_system_resource`, but no other non-stop finish
    reason.
- Why it matters: a truncated or filtered generation can become a silent, stable “no suggestions” result instead of a
  retryable/typed failure. More generally, accepting a non-stop completion breaks the assumption that returned JSON is
  the provider's completed answer.
- Recommendation: require `finish_reason == "stop"` before accepting content. Keep
  `insufficient_system_resource` retryable; map every other non-stop reason deliberately (normally malformed
  structured output unless policy requires a distinct terminal classification). Add table tests proving no non-stop
  response reaches the cache, including a syntactically valid `{}` body.
- Suggested job split: Job A.

### R003 Transport wrapping erases context cancellation

- Priority: **P2**
- Area: async cancellation / worker shutdown / RPC cancellation
- Evidence:
  - `client.go:132-137` maps every `http.Client.Do` error to `RateLimitedError` and formats the cause with `%v`, not
    `%w`; the returned chain no longer satisfies `errors.Is(err, context.Canceled)` or
    `errors.Is(err, context.DeadlineExceeded)`.
  - `apps/api/internal/ai/errors.go:30` exposes the stored cause through `Unwrap`, but the stored cause is already the
    non-wrapping formatted error.
  - `apps/api/internal/platform/jobqueue/runner.go:224-231` explicitly uses context identity as a stop signal.
  - There is no DeepSeek test that cancels a request while the fake server is in flight.
- Why it matters: caller cancellation is not a provider throttle. Reclassifying it pollutes retry/error telemetry and
  can make synchronous cancellation look like a rate limit. The worker's root `ctx.Err()` is a partial backstop during
  shutdown, but it does not repair the adapter's error contract or per-call deadline behavior.
- Recommendation: return `ctx.Err()` unchanged when the request context is done; for genuine transport failures,
  preserve the cause with `%w` inside the typed error. Add cancellation and deadline tests. The same pattern exists in
  the older Anthropic/Voyage clients, so fix and test all provider transports together rather than leaving divergent
  semantics.
- Suggested job split: Job B (AI provider HTTP/cancellation lifecycle).

### R004 Error responses are closed without being consumed

- Priority: **P3**
- Area: HTTP connection lifecycle under provider faults
- Evidence: `client.go:139-143` defers `Body.Close` and returns from `mapStatus` before reading the error body. Go's
  HTTP/1 transport requires observing EOF before it can reuse that connection; closing the unread
  `bodyEOFSignal` takes its early-close path.
- Why it matters: exactly during 429/5xx bursts—when the adapter will retry—the client can discard reusable
  connections and add TCP/TLS churn. HTTP/2 mitigates this, but the adapter should remain correct across negotiated
  transports and proxies.
- Recommendation: consume a small bounded amount of the error body to EOF before closing (without logging secrets or
  unbounded vendor text), or centralize status handling in a helper that guarantees bounded drain + close. Test with a
  non-empty error body.
- Suggested job split: Job B.

### R005 The tests prove implementation-local init, not production registration or schema rejection

- Priority: **P3**
- Area: masking tests / composition seam
- Evidence:
  - `client_test.go:177-183` calls `ai.ImplementedLLM("deepseek")` from package `deepseek`; importing the package under
    test necessarily ran its own `init`, so this assertion cannot catch a missing blank import in either production
    binary.
  - The actual registrations are separate lines in `cmd/api/worker.go:15-17` and
    `cmd/worker/main.go:20-22`.
  - The success test asserts that a schema string was embedded in a prompt, which stays green even though R001's
    contract is not enforced.
- Why it matters: both load-bearing claims—“the shipped binaries expose DeepSeek” and “the provider enforces the
  schema”—currently have tests that can remain green when the respective claim breaks.
- Recommendation: add a composition-root/catalog test that observes DeepSeek as implemented in the API binary and a
  worker wiring test (or a shared explicit provider-import package consumed by both roots). Replace the prompt-presence
  proxy with behavioral schema-rejection tests.
- Suggested job split: Job A for schema tests; Job C for registration/catalog coverage.

### R006 The new comments/docs contain immediately drifting provider history and slot names

- Priority: **P3**
- Area: timeless comments / as-built SSOT accuracy
- Evidence:
  - `client.go:23-25` records a dated legacy-alias retirement event. `spec/principle/code-comments.md:9-10` explicitly
    forbids dated/history notes in source comments; vendor lifecycle belongs in docs, not beside the constant.
  - `spec/tech/ai-worker.md:47` records LLM slots as
    `anthropic · openai · deepseek · zai · gemini`, while the actual registry is
    `anthropic · openai · deepseek · glm · gemini · kimi`
    (`apps/api/internal/ai/factory.go:39`).
  - `pnpm lint:comments` passed, so the executable guard does not catch this dated vendor-note form.
- Why it matters: the model comment becomes stale at a known wall-clock instant, while the as-built document omits two
  operator-visible identifiers and names another differently from the runtime/admin catalog.
- Recommendation: keep the source comment timeless (what the default represents now and how to override it), move
  deprecation history to an operational/vendor note if it must be retained, and make `spec/tech/ai-worker.md` use exact
  registry identifiers with a parenthetical that `glm` is the Z.ai/Zhipu slot.
- Suggested job split: Job C (provider registration, docs, and guard coverage).

## Cross-Cutting Themes

- **Prompted structure is being treated as enforced structure.** R001/R002/R005 all arise because tests and comments
  validate the request's intent, not the provider response contract.
- **Fault paths receive less lifecycle discipline than the happy path.** Cancellation identity, finish reasons, and
  non-200 response bodies need explicit handling (R002-R004).
- **Executable guards are narrower than the prose rules.** The full gate is green while the dated comment escapes the
  comment-history lint and a mobile worker force-exit warning remains non-fatal.

## Questions / Tradeoffs

- DeepSeek strict schema mode is currently beta-only and does not support `minItems`/`maxItems`. Is the product willing
  to depend on the beta endpoint, or should plan 28 explicitly permit stable native JSON mode plus local JSON Schema
  validation?
- DeepSeek's optional `user_id` accepts only `[a-zA-Z0-9_-]` up to 512 characters and must not carry private
  information. Production Supabase subject UUIDs fit, but local dev ids are unconstrained. Should the adapter omit the
  field, validate-and-omit unsupported ids, or send a stable pseudonymous hash? This was not promoted to a finding
  because current production ids are compatible.
- Should bounded error-body draining and cancellation handling become one provider-neutral HTTP helper, or stay
  vendor-local to preserve the existing “vendor knowledge per subpackage” boundary?
