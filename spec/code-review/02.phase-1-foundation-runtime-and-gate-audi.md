## Architecture Baseline

- `spec/` is the SSOT: concept owns the laws + neuroscience guardrails, ubiquitous-language owns the names,
  ARCHITECTURE owns placement/dependency rules, `00.overview` owns the roadmap.
- **Frontend FSD, identical for web and mobile** (`ARCHITECTURE.md` §3.1): six one-way layers
  `app → pages → widgets → features → entities → shared`; same-layer cross-import forbidden except the `entities` `@x`
  public API; one `index.ts` per slice; enforced by `steiger` + `eslint-plugin-boundaries`.
- **Promote-on-reuse** (§3.5): platform-pure glue lives in `packages/*` (auth/observability/client-cache/state-machine/
  i18n/config/api-client); each app keeps only its genuinely platform-specific wiring. `*.native.tsx` siblings exist
  only where a primitive truly differs.
- **Control-state in XState, data in stores** (§3.2): machine context carries ids/control metadata only; collections/
  buffers live in Query/Zustand/refs; per-frame reads via `getSnapshot()`, never React state.
- **Backend domain-first Go** (§2): sqlc rows and proto types are infra, mapped at the edge; interfaces declared
  consumer-side; thin RPC handlers; **per-user isolation uniform** — every persisted row carries `user_id` and every
  query is scoped via an interceptor-injected context value (§4); the root lint gate scans migrations/queries for it.
- **Config is build-time and generated** (§4): tuning scalars flow `values.yaml` → `pnpm gen:values` → TS/Go consts;
  never hardcoded.
- **Seams are linted**: observability vendor imports only at the platform boundary; raw user-facing strings go through
  the i18n seam; comments and docs are timeless (no process/history/ticket numbers).

### R001 The owned auth facade is created in a render-phase initializer and never disposed (StrictMode orphan)

- Priority: P2
- Area: Auth provider lifecycle (web)
- Evidence:
  - `packages/auth/src/react.ts:15` — `const [ownedFacade] = useState<AuthFacade | null>(() => (facade ? null : (createFacade?.() ?? null)))`.
  - `createAuthFacade` has body side effects: it starts an XState actor (`facade.ts:30`) and registers a live Supabase
    `onAuthStateChange` subscription + autorefresh client; the only teardown is `dispose()` / `adapterUnsubscribe`.
  - `AuthProvider` has **no `useEffect` cleanup** calling `facade.dispose()`; a repo-wide grep finds `dispose()` invoked
    only from tests/fakes, never from a production component. Web mounts under `<StrictMode>` (`apps/web/src/main.tsx:7`)
    and passes no facade, so the owned branch runs.
- Why it matters:
  - Creating a stateful resource inside a `useState` lazy initializer is the impure-initializer anti-pattern: React's
    dev StrictMode double-invokes the initializer, so a **second facade is created and discarded with a live, orphaned
    actor + subscription + refresh timer** — and on unmount the retained facade is never torn down either. This is
    code-review 01's R013 lifecycle class resurfacing in the promoted seam, in the security-critical auth path.
- Recommendation:
  - Create-and-dispose the owned facade in a `useEffect` (or hold it in a ref and dispose it in the effect cleanup), so
    the StrictMode-orphaned facade and the unmounted facade are both torn down. Make `dispose()` idempotent.
- Suggested job split: part of a provider-lifecycle job (with R006, R007, R008).

### R002 The persistence-isolation migration guard fails open on tables with trailing options

- Priority: P2
- Area: Backend per-user isolation guard (the R004 deliverable)
- Evidence:
  - `scripts/check-persistence-isolation.mjs:17` — the migration scan is
    `/\bCREATE\s+TABLE\s+…\(([\s\S]*?)\)\s*;/gi`, anchoring the table body's closing paren on `)\s*;`.
  - A statement such as `CREATE TABLE x ( id uuid PRIMARY KEY ) TABLESPACE fast;` (closing paren followed by
    ` TABLESPACE fast;`, not `\s*;`) **matches nothing**, so the table is silently skipped and its missing `user_id`
    is never flagged. Trailing options (`TABLESPACE`, `INHERITS`, `USING …`) evade the guard.
- Why it matters:
  - Per-user isolation is the product's first privacy boundary, and this is the executable guard that is supposed to
    enforce it. It fails _open_: a future product table can omit `user_id` and still pass. Latent today (migrations
    contain only `CREATE EXTENSION`), but it must be airtight before the first product table lands.
- Recommendation:
  - Match the full statement rather than anchoring the body on `)\s*;` (e.g. allow `[^;]*;` after the closing paren),
    or split into statements and test each `CREATE TABLE`. Add a probe asserting an unscoped table with trailing
    options fails.
- Suggested job split: part of a guard-hardening job (with R003, R004, R012, R013).

### R004 `gen:values` emits gofmt-dirty Go for any group mixing a scalar with an array/map (latent gate deadlock)

- Priority: P2
- Area: Codegen / quality gates
- Evidence:
  - `scripts/gen-values.mjs:235` — `return parts.join('\n')` joins the generated Go `const (…)` block and `var (…)`
    block with a single newline.
  - `gofmt` inserts a blank line on a `const`→`var` declaration-kind change, so a group that produces both blocks is
    gofmt-dirty: `pnpm check:gen` (regenerate + byte-diff) demands the blank-line-free output, while `pnpm lint:api`
    (`gofmt -l`) demands the blank line — **the two gates become mutually unsatisfiable**.
  - Latent: the live `values.yaml` has no group mixing a scalar with an array/map. `scripts/gen-values.test.mjs:25-54`
    exercises a mixed shape but never runs gofmt, giving false confidence.
- Why it matters:
  - The first time anyone adds a numeric array or scalar map alongside a scalar in an existing group, two green gates
    deadlock and the build cannot pass — a confusing failure far from its cause.
- Recommendation:
  - Separate Go declaration blocks with a blank line (join with `\n\n` on a declaration-kind change, or run the
    rendered Go through `gofmt` before writing), and add a gofmt assertion to `gen-values.test.mjs`.
- Suggested job split: part of the guard-hardening job (with R002, R003, R012, R013).

### R005 Native Toast hardcodes `accessibilityRole="alert"` for every variant, diverging from web's per-variant roles

- Priority: P2
- Area: Design-system accessibility / cross-platform parity
- Evidence:
  - `packages/ui/src/primitives/toast.native.tsx:30` — `accessibilityRole="alert"` (static, all variants).
  - Web `packages/ui/src/primitives/toast.tsx:8-13` —
    `const ROLE: Record<ToastVariant, 'status' | 'alert'> = { info: 'status', success: 'status', warning: 'alert', danger: 'alert' }`,

- Recommendation:

- Priority: P3

## Cross-Cutting Themes

- **Phase 1 is genuinely healthy post-resolution.** The full `pnpm check` gate is green, CI now runs `pnpm check`
  directly (review 01's R002 resolved), and all 23 review-01 findings independently re-verified as resolved. There is
  **no P1 and no live correctness or security bug** — the auth/session cluster (R006–R009 in review 01) is correctly
  fixed and was the densest risk last time. The remaining findings are latent, dev-only, or hygiene.
- **Lifecycle/dispose of app-lifetime singletons is the one place the promote-to-packages refactor left ragged edges**
  (R001, R006, R007, R008): the owned auth facade is created in a render-phase initializer and never disposed (a
  StrictMode dev orphan — review 01's R013 family resurfacing in the promoted seam); mobile observability doesn't
  dispose on unmount; and web vs mobile observability disagree on lifecycle. The fix is altitude, not patches: settle
  **one** "create-once + dispose-on-unmount (idempotent)" contract and apply it across the shared provider seams.
- **Guard integrity is the highest-leverage cluster** (R002, R003, R004, R012, R013). Several executable guards are
  green but not actually load-bearing: the web FSD boundary probe never runs (R003), the persistence migration guard
  fails _open_ on trailing table options (R002), the `gen:values` output can deadlock `check:gen` against `gofmt`
  (R004), the persistence query guard false-fails on dollar-quoted bodies (R012), and the observability guard is
  cwd-brittle (R013). **R002 is the one to fix before Phase 2 persistence** — it is the per-user isolation boundary and
  it currently fails open.
- **Small cross-platform divergences persist after promote-on-reuse** (R005, R007, R011, R021, R025): the platform-pure
  logic is shared, but thin app-layer surfaces — toast a11y role, auth re-export barrel, skeleton timing, observability
  lifecycle, i18n precedence — have each drifted slightly. None is a bug today; each is a seam where the two platforms
  will drift further unless converged or consciously documented.
- **Comment / naming hygiene residue** (R009, R010, R014, R017, R019, R020, R024, R026): review 01's R023 cleaned
  source comments but missed test files; plus a false "mirror" comment, an overstated guard comment, dead operands/
  branches, a Go shadow, an unguarded `/test` route, and a phase-named symbol. A cheap single sweep.

## Questions / Tradeoffs

- **Provider lifecycle contract.** Should app-lifetime singletons (auth facade, observability runtime, query client)
  uniformly follow "create-once + dispose-on-unmount (idempotent)", or is "create-once, never dispose" acceptable for
  some? The seams currently disagree (R001 vs R006/R007). Pick one and apply it; decide whether `dispose()` must be
  idempotent for StrictMode.
- **Persistence guard hardening (R002/R012).** Regex patch, a small SQL-aware tokenizer, or migration metadata? A
  statement-/paren-aware approach is more robust than anchoring on `)\s*;`. Worth doing before the first product table —
  and possibly alongside wiring `UserScope` into a live smoke path (R016) so the guard has a real consumer.
- **Web observability lifecycle (R007).** Document the intentional asymmetry, or lift a shared vendor-binding contract
  into `packages/observability`? Given web config is build-time-static and the vendor SDKs are non-re-runnable
  singletons, documenting is likely the cheaper correct call.
- **`/test` route (R024).** Gate behind `import.meta.env.DEV` and move fakes out of the prod barrel — or accept shipping
  the diagnostic page since the fakes already tree-shake out?
- **Mobile token-bridge reach (R023).** Re-export `space`/`fontSize` from `@cosimosi/ui` and migrate all mobile screens,
  or accept the existing app-layer numeric convention? This is a broad consistency call, not a `DiagnosticsScreen`
  one-off.
- **Local Node vs the pin.** `check:node` enforces only the `engines` floor (`>=22.13`); the `.node-version` pin
  (`22.22.1`, honored by CI) is advisory locally, so a developer can run a different major (this review ran Node 25).
  Gates passed on Node 25, so this is informational — but worth deciding whether local runs should match CI's pin.
