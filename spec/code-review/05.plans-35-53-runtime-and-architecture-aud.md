## Architecture Baseline

- **Backend:** domain-first Go. Composition belongs in `cmd/api`; context use-cases and pure domain stay free
  of Connect/proto/sqlc/pgx; adapters point inward; every row/query is user-scoped; related graph writes use
  one per-user serialization boundary; external trust is behind fail-closed ports.
- **Frontend:** Feature-Sliced Design. Control state lives in XState, user/read data in Zustand or Query, and
  app composition remains thin. Pure code used by web and mobile is promoted to `packages/`, never copied
  sideways. Mobile mirrors the `pages` layer; `app/navigation` registers screens rather than owning product pages.
- **State and time:** server facts are authoritative, continuously changing presentation is derived at read,
  and universe time is monotonic. Sync-to-today uses the UTC-normalized server date and must be consented to
  before the mutating call.
- **Economy and deletion:** a paid action and its debit are atomic and retry-safe; external earns are verified;
  user-originated full deletion becomes a real hard delete after the configured grace period; append-only history
  must never be repaired by mutation.
- **Read policy:** every `NO_SIDE_EFFECTS` RPC has an explicit user/public cache policy, and enforcement is
  exhaustive rather than dependent on individual feature authors remembering a second registry.

The Go package placement, normal transaction atomicity, generated freshness, and standard CI aggregation are solid.
The findings below are the exceptions.

- **Evidence:** `packages/client-cache/src/provider.ts:42-48` installs

### R002 Production Twinkle earn endpoints trust caller-created payments and nonexistent inviters

- **Classification:** CONFIRMED
- **Priority:** P1
- **Area:** backend / external trust boundary / Twinkle economy
- **Evidence:** the production composition root unconditionally binds `KeylessPaymentVerifier{}` and
  `DistinctSignup` at `apps/api/cmd/api/twinkle.go:29-40`. The verifier at
  `apps/api/internal/twinkle/defaults.go:17-36` accepts any non-empty caller receipt for the known pack and mints the
  configured amount without contacting a store; changing the string produces a new digest. `DistinctSignup` at
  `defaults.go:39-43` checks only non-empty, unequal IDs. `apps/api/internal/twinkle/service.go:248-289` treats the
  invite code as the inviter user ID and credits both scopes, while the ledger uniqueness is only
  `(user_id, dedup_key)`. The authenticated RPCs expose both operations at
  `internal/twinkle/rpc/server.go:66-91`. Plan 44 requires a real signup and says production grants no value without
  verified store receipt (`spec/plan/44.earn-spend-usecase.md:57-62,227-234,328-331`). Nuance: plan 44:229-231 does
  ship `DistinctSignup` as the sanctioned permissive Epic F invite default, so the invite half under-implements a
  documented tradeoff (even the promised "a real, distinct signup exists" is not checked), while the keyless payment
  binding contradicts the plan's production requirement outright.
- **Why it matters:** an authenticated caller can mint unlimited additional Twinkle by submitting arbitrary receipt
  strings, and can claim an invite against an arbitrary nonexistent ID to receive the invitee grant while creating a
  phantom inviter balance. The payment port also carries no verified purchaser/account binding, and receipt uniqueness
  is not global across users.
- **Recommendation:** disable/fail-close Charge until a configured production verifier exists; have verification bind
  the provider transaction, app/store account, and authenticated beneficiary; enforce global provider-transaction
  uniqueness. Replace raw user-ID invite codes with a trusted account-backed resolver and verify inviter existence plus
  invitee eligibility before the ledger transaction.
- **Suggested job split:** one security-focused Twinkle external-trust job.

### R003 The 30-day hard-delete promise has no inactive-user trigger and leaves source text in job payloads forever

- **Classification:** CONFIRMED
- **Priority:** P1
- **Area:** backend / retention / privacy / async jobs
- **Evidence:** the only production call to `Service.Sweep` is the next Release invocation at
  `apps/api/internal/memory/release.go:53-69`; expired Restore returns `ErrRestoreWindowExpired` without sweeping at
  `release.go:145-156`. Plan 49 requires an operational maintenance-job or per-user-read trigger
  (`spec/plan/49.release-usecase.md:224-235`), and archived job 60 marked that choice complete
  (`spec/jobs/archive/60.release-usecase.md:94-103`). Framing correction (post-report verification): plan 49's
  as-built note (`:295-299`) and `spec/policy/domain/deletion-rules.md:56-58` later recorded the Release-start
  trigger and explicitly accepted "a user who never returns is not swept in v1" — so the retention gap is a
  documented v1 tradeoff rather than a silent violation; the implementing job must treat the fix as a deliberate
  policy change and retire both passages. Separately, `internal/memory/jobs.go:18-45` persists neuron text,
  memory name/current text, and kept stage text in JSON. The `jobs` table has no target FK or expiry
  (`db/migrations/00002_memory_aggregate_schema.sql:71-80`), `db/queries/memory/jobs.sql:1-54` has no delete, and the
  sweep list at `release.go:318-340` never touches jobs. A soft-deleted memory can still be sent to the LLM before
  `SetSemanticStages` finally refuses the write.
- **Why it matters:** a user who deletes once and then becomes inactive retains Diary bodies, memories, and provenance
  indefinitely. Even when a later action happens to sweep them, completed/failed/pending jobs keep copies of the text
  indefinitely and a pending worker may continue externally processing deleted content.
- **Recommendation:** enqueue a per-release retention job at `deleted_at + retentionWindow` so inactive users converge;
  make async payloads identity/revision-only and re-read live source at execution; cancel/redact target jobs on Release,
  purge them during Sweep, and apply a bounded terminal-job retention. Migrate or scrub existing payloads and test the
  no-further-activity and expired-Restore paths end to end.
- **Suggested job split:** one retention-execution and deletion-safe-payload job.

### R004 Release/restore sealing is non-compositional and races every other graph writer

- **Classification:** CONFIRMED
- **Priority:** P1
- **Area:** backend / deletion graph integrity
- **Evidence:** `ClassifyNeurons` ignores every soft-deleted activation at
  `apps/api/internal/memory/deletion.go:58-84` (`if fact.MemoryDeleted ... continue`). With memories A and B sharing N:
  Release A keeps N; Release B now sees A as absent, seals N, and records N only in B's ledger; Restore A unseals only
  A's recorded neurons (`release.go:159-178`, `db/queries/memory/release.sql:46-89`). B's later sweep preserves N because
  A is live, but deletes B's ownership ledger, leaving A permanently linked to sealed N. No overlap/restore test exists.
  In addition, Launch and recall acquire the per-user graph lock (`launch.go:61-75`, `sync.go:49-55`) while
  `pg/release.go:15-37` opens an ordinary transaction and `ReleaseTx` has no lock seam
  (`release_ports.go:122-173`). Contribution weakening is an unlocked read followed by an absolute write
  (`db/queries/memory/deletion.sql:61-84`, `pg/release.go:345-384`).
- **Why it matters:** overlapping releases can make a restored live memory semantically incomplete forever. Concurrent
  launch may attach a new live memory to a neuron that Release then seals, and concurrent recall/consolidation strength
  changes can be silently overwritten by Release's stale absolute value.
- **Recommendation:** serialize every per-user graph mutation on the same advisory lock; use row locks or atomic delta
  SQL for strength changes. Model seal ownership/refcounts across live and restorable activations, or conservatively
  count retained soft-deleted memories and reclassify on Restore. Add A/B overlap plus release-vs-launch/recall/
  consolidation integration tests.
- **Suggested job split:** one release/restore graph-composition and serialization job.

### R005 Consolidation can publish empty history, overwrite newer meaning, and double-downscale an interval

- **Classification:** CONFIRMED
- **Priority:** P1 (empty/stale semantic content); P2 facet for direct interval re-run
- **Area:** backend / semanticization / async convergence
- **Evidence:** `apps/api/internal/memory/consolidate.go:204-234` initializes `text := ""`, appends that immutable
  provenance row, and advances the stage even when the ladder text is absent; repair is enqueued only afterward at
  `consolidate.go:147-153`. `consolidate_test.go:648-649` documents the empty
  row as intended (the assertion at `:650-652` pins the stage advance, not the empty text), contradicting plan 41's
  “rather than materializing an empty stage” rule (`spec/plan/41.consolidation-usecase.md:268-271`) and plan 40's
  required pregenerated provenance text. Reconsolidation and repair jobs snapshot `CurrentText`/`KeptStages`
  (`recall.go:293-325`, `consolidate.go:252-277`), but the worker writes by `(user_id, memory_id)` only
  (`jobs.go:92-129`, `db/queries/memory/jobs.sql:48-54`). An older retried job can therefore overwrite a newer recall's
  stages; lease fencing protects only the terminal job-row update, not the side effect. Finally every invocation with
  `to > from` calls `downscaleSynapses` (`consolidate.go:127-171`) with no processed-interval marker, while the rerun test
  has no synapse fixture. Plan 41 AC10 explicitly says a rerun downscales nothing further (`:300-301`).
- **Why it matters:** append-only provenance can contain permanently blank variants, delayed old LLM work can revert a
  user's latest representation, and a replayed handler can repeatedly weaken the graph while tests remain green.
- **Recommendation:** do not materialize a stage/provenance row until its real text exists; finalize the pending rise once,
  in the text-completion transaction. Add a representation revision to job input and conditional writes; validate live
  source at execution and fence side effects, not only job completion. Persist an interval/version marker so downscale is
  exactly-once, and add delayed/out-of-order/lease-loss and repeated-interval tests.
- **Suggested job split:** one consolidation/semanticization convergence and version-fencing job.

### R007 Dismissible paid requests and ambiguous retry paths can lose results and charge twice

- **Classification:** CONFIRMED
- **Priority:** P1
- **Area:** web/mobile paid recall and gist flows / state machines / economy idempotency
- **Evidence:** `packages/universe/src/recall-flow.machine.ts:52-59` accepts `CLOSE` while reconsolidating, contrary to
  the plan 36 state table (`spec/plan/36.recall-flow-ui.md:198-204`). The UI then clears target and draft
  (`RecallFlowSheet.tsx:79-84`) while the old async continuation still applies data and sends DONE
  (`:93-123`). Gist view starts its paid request at `GistViewSheet.tsx:36-52` but leaves the raw dialog close callback
  active at `:54-55`; shared Dialog always closes on Escape, backdrop, and X
  (`packages/ui/src/primitives/dialog.tsx:22,28,47-51`). Recall and gist catches route every error back to a payable retry,
  even when the server committed and only the response was lost. The paid proto requests contain no operation key
  (`proto/cosimosi/memory/v1/memory.proto:203-238`), and spend ledger rows set no dedup key
  (`apps/api/internal/twinkle/service.go:200-208`).
- **Why it matters:** a close/reopen or rapid submit can let an old completion mutate a new flow; a timeout after commit
  can be deliberately retried and debit/reinforce twice; a successful gist charge can yield no visible result.
- **Recommendation:** make in-flight paid states non-dismissible or durably recoverable, guard continuations with a
  session/attempt epoch, and distinguish known pre-commit refusals from ambiguous failures. Add client operation IDs and
  server action receipts/idempotency so a retry returns the original result without re-spending or reapplying recall.
- **Suggested job split:** one end-to-end paid recall/gist lifecycle job with R006-R008.

### R008 Client-local “today” can bypass or spuriously require the UTC sync-consent gate

- **Classification:** CONFIRMED
- **Priority:** P1
- **Area:** web/mobile time consent / server-client contract
- **Evidence:** the authoritative recall sync advances to `utcDate(s.now())` at
  `apps/api/internal/memory/sync.go:49-74`, matching plan 30's UTC contract
  (`spec/plan/30.time-advance-usecase.md:47-50,151-155`). Web recall instead constructs local calendar today with
  `getFullYear/getMonth/getDate` and compares it at `RecallFlowSheet.tsx:28-33,67-69`; mobile does the same at

  `DiaryReaderBlock.tsx:30-35,75-79` and mobile `DiaryReaderBlock.tsx:31-36,76-78`.
- **Why it matters:** west of UTC just after UTC midnight, the client may decide no consent is needed while the server
  advances a day. East of UTC before UTC midnight, the client may ask for consent when the server would not advance.

- **Recommendation:** return server-authoritative `today`/`needs_sync` with the read model (or a dedicated free read) and
  drive both flows from it. At minimum share a UTC helper, but do not make a consent decision from an untrusted local
  wall clock when the mutation uses server time. Test UTC-12, UTC+14, and skew boundaries.
- **Suggested job split:** fold into the paid recall lifecycle job.

### R010 Palette persistence is cross-session, rolls back to unsaved state, and misses the first-frame contract

- **Classification:** CONFIRMED
- **Priority:** P1 for cross-user write; P2 for rollback/bootstrap facets
- **Area:** web/mobile palette preference lifecycle
- **Evidence:** both apps use a module-global promise queue (`apps/web/src/features/change-palette/api/change-palette.ts:11-33`;
  mobile is byte-identical) with no user/session epoch. The queued callback starts the RPC only after the preceding
  request settles, while the auth interceptor reads the live token at dispatch time
  (`packages/api-client/src/platform.ts:43-54`). Thus A's queued selection can execute under B's token. Two failed rapid

- **Recommendation:** scope/cancel queues by session epoch, ignore stale completions, and track the last server-confirmed

  rise-state mutation is deferred until commit.
- **Why it matters:** the gap cue can vanish during an allowed below-to-above fly-through. Under concurrent/aborted React
  rendering, the frame loop can observe new instance ordering while the committed mesh/channels still have the old count,
  producing transient position or selection mismatches.
- **Recommendation:** render fog double-sided (or paired opposite faces) and verify both camera directions. Publish one
  atomic committed instance snapshot in a layout/effect boundary and make channels/count/frame lookup consume it together;
  add renderer-level regression tests or deterministic scene assertions.
- **Suggested job split:** one gist depth-layer renderer correctness job.

### R014 The palette validator assigns red warmth to grayscale and can warn on a color with no hue

- **Classification:** CONFIRMED
- **Priority:** P3
- **Area:** emotion palette guardrail
- **Evidence:** `packages/emotion/src/axis-consistency.ts:49-61` maps `delta === 0` to hue `0`, while
  `hueWarmth` at `:41-47` interprets 0 degrees as strongly warm. Its comment claims a desaturated color “never produces
  a warning on its own.” With FEAR valence `-0.82` and threshold `0.6`
  (`packages/config/src/values.gen.ts:58,64-66`), gray yields severity about `0.628` and is warned. Tests cover saturated
  inversion and neutral mood only (`axis-consistency.test.ts:17-49`).
- **Why it matters:** future valid grayscale/desaturated palettes can be rejected by a warn-only semantic guard for a hue
  they do not have.
- **Recommendation:** return neutral warmth for zero saturation or weight warmth by saturation, and add grayscale plus
  near-gray threshold tests.
- **Suggested job split:** a small facet of the palette/client-state job rather than a standalone job.

### R015 The persistence isolation lint accepts a user_id reference or projection as if it were a predicate

- **Classification:** CONFIRMED
- **Priority:** P2
- **Area:** quality gate / per-user isolation enforcement
- **Evidence:** architecture says every product row and query is scoped and the root lint enforces it
  (`spec/ARCHITECTURE.md:424-426`). Yet `scripts/check-persistence-isolation.mjs:97-110` accepts any table body containing
  the token `user_id`, and `:114-125` accepts any whole statement containing the token while reporting that it checked a
  predicate. Tests at `scripts/check-persistence-isolation.test.mjs:42-67` cover only total omission. A safe temp probe
  containing `owner_id REFERENCES users(user_id)` and `SELECT user_id, body FROM leaked_records` returned
  `violations: []`. Archived job 16 explicitly deferred this heuristic until product tables landed
  (`spec/jobs/archive/16.phase-1-foundation-hardening-and-hygiene.md:193-198`); this range adds three migrations and 10
  new/changed product query files (13 files in total).
- **Why it matters:** current SQL passed a manual scope audit, so no live cross-user query was proven. The enforcement
  contract is nevertheless fail-open at the exact point where the product persistence surface expanded; a projection,
  FK reference, comment-stripped token, or unrelated join can mask a missing ownership predicate.
- **Recommendation:** make the checker statement-kind/alias aware, preferably with a SQL parser; verify a declared
  `user_id` column and a real mutation/read predicate tied to the owned table. Add negative controls for FK references,
  projections, unrelated aliases, CTEs, and multi-statement files.
- **Suggested job split:** one quality-gate hardening job with R016.

### R016 Mobile tests and primary lints report lifecycle failures/warnings but still return success

- **Classification:** CONFIRMED
- **Priority:** P2 for mobile lifecycle gate; P3 for warning policy
- **Area:** readiness / Jest / lint signal
- **Evidence:** `CI=true pnpm test:mobile` returned 0 with 20 suites/59 tests but printed 32 unwrapped `act(...)`
  warnings and “A worker process has failed to exit gracefully and has been force exited.” The changed
  `apps/mobile/src/app/App.test.tsx:88-100` signs out outside `act`, and `:135-138` resets a global palette while mounted
  subscribers are still registered; `packages/emotion/src/palette.ts:73-83` synchronously notifies them. Diagnostic
  `jest --runInBand --detectOpenHandles` reported one failed suite and one open Timeout created downstream of the
  synchronous palette notification (stack `App.test.tsx:137 -> palette.ts:83`; `palette.ts` itself schedules no
  timer), then remained alive past 120 s. Separately, web oxlint and mobile ESLint run without zero-warning enforcement and the
  standard gate accepted six changed React export warnings plus one mobile `no-void` warning.
- **Why it matters:** `pnpm check` and CI are green while the mobile shell has leaked work, unowned external-store
  updates, and a flaky Diary wait. These are precisely the lifecycle problems auth/palette changes can introduce, so the
  smoke gate currently masks regressions instead of containing them.
- **Recommendation:** unmount/cleanup renders before resetting facades/stores, wrap external mutations in `act`, eliminate
  the open handle and flaky wait, and fail tests on unexpected console errors or leaked handles. Clean or explicitly
  waive existing lint warnings, then enforce zero warnings in the primary web/mobile lint commands.
- **Suggested job split:** one mobile-test lifecycle and enforcement-signal job with R015.

## Cross-Cutting Themes

- **Second registries and global singletons are not session- or contract-complete.** Cache-policy entries, Zustand
  mirrors, and palette queues rely on feature authors remembering a separate lifetime/registry update.
- **Retry semantics stop at transaction atomicity.** Individual backend transactions are sound, but client operation
  idempotency and worker side-effect versioning are missing, so “commit succeeded, response/worker completion arrived
  late” is unsafe.
- **Deletion and restore need a graph-wide lifecycle model.** The row-level writes are scoped and transactional, but
  overlapping release ownership, inactive-user scheduling, and async payload retention were considered independently.
- **Green gates do not exercise the production composition.** Router transports bypass the cache interceptor, direct
  use-case tests bypass external trust configuration, and normal Jest hides open handles.
- **Web/mobile parity was implemented as copying.** That gives immediate visual parity but recreates the maintenance
  problem the package-sharing rule exists to prevent.

## Questions / Tradeoffs

1. Should Charge remain entirely disabled until a real store adapter is configured, or should production startup fail?
   Either is safe; accepting keyless receipts on the public RPC is not.
2. Should sync consent consume a `server_today` field on GetUniverse/GetDiaries, or a dedicated free clock-status read?
   A client-local UTC helper is only a partial fix because device clock skew remains.
3. Should palette bootstrap block the whole authenticated tree or show a neutral shell while preference resolves? The
   plan fixes “chosen palette before universe first render,” not which loading treatment achieves it.
4. For deletion-safe jobs, can existing terminal payloads be redacted in place, or is a migration/purge required by the
   product's retention promise? The implementation job should make that decision explicit before schema work.
