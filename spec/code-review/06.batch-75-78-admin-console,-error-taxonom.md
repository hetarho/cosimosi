## Architecture Baseline

Rules used for judgment:

- **Backend domain-first Go / CC8** (`ARCHITECTURE.md` §2): each bounded context imports no other context's
  internals; only `<ctx>/pg` imports `dbgen`; cross-context access is through consumer-owned ports bound at
  `cmd/api`; the proto/sql layer is an anti-corruption boundary.
- **Error contract** (`policy/platform/errors.md`): expected domain refusals keep a safe message + canonical Connect
  code + one `ErrorInfo{reason,domain,request_id,metadata}`; unexpected failures are rebuilt at the outer boundary as
  masked `Internal`; every failed RPC carries a non-empty `request_id`; `debug_detail` is the only diagnostic
  exception, gated by `COSIMOSI_ERROR_DETAIL=verbose`, fail-closed, never rendered by the FE.
- **Admin policy** (`policy/ops/admin.md`): env-seed ∪ DB-promoted authorization, seed admins undemotable;
  metadata-only privacy [I2]; capped/append-only/audited grants [I1][I3]; per-provider keys write-only across the RPC
  boundary; DB→env→keyless-mock resolution; web-only surface (parity waived).
- **FSD on the FE** (`ARCHITECTURE.md` §3): shared logic in `packages/*`, app-layer wiring in `apps/web` with
  `apps/mobile` parity (waived for `/admin`); promote-on-reuse; all user-visible copy through the i18n seam.
- **SSOT seams**: numeric tuning flows through `spec/values.yaml` → `pnpm gen:values`; user-facing strings through
  `packages/i18n`; the constitution [I1]–[I11] via `00.overview` §3.

Overall the batch is well-built and the gate is green. The masking/write-only key boundary, the CC8 boundaries of
`internal/admin`, the unmapped-cause→`Internal` fall-through, `request_id` coverage, and the reason registry↔`errors.md`
table all hold up under adversarial checks (see the REFUTED list). The findings below are the exceptions.

## Findings

### R001 `ClearProviderKey` can hard-fail the entire AI pipeline until an operator intervenes

- Priority: **P2**
- Area: `internal/admin` runtime config / `internal/ai` factory
- Evidence: `apps/api/internal/admin/service.go:392` `ClearProviderKey` deletes a provider key with **no** check for a
  capability still selecting it. `apps/api/internal/ai/runtime.go:60-77` — when a DB selection exists
  (`found && provider != ""`) but `ReadProviderKey` returns not-found, `effective()` returns the DB `cfg` with an
  empty `APIKey` (it does **not** fall through to env). `apps/api/internal/ai/factory.go:167-171` then hard-errors:
  `return Adapters{}, fmt.Errorf("ai: LLM provider %q configured without an API key (%s)", …)`.
- Why it matters: an operator clearing a key that is still selected (a plausible console action, no guard) makes the
  next `current()` build fail for **every** memory port — Extractor/Semanticizer/Embedder/PredictionError/SealSuggester
  — in **both** `cmd/api` and `cmd/worker`. Because `current()` builds LLM before embedding, a cleared _LLM_ key also
  takes down embedding. Encode and embedding are down until the key is re-added or the selection cleared — a
  self-inflicted prod outage reachable without a redeploy, exactly the path the runtime seam was meant to make safe.
- Recommendation: either refuse `ClearProviderKey` (and a selection change) while a capability selects that provider,
  or have `effective()` treat "selected provider with missing key" as fall-through to env → keyless mock rather than
  returning a DB cfg with an empty key.
- Suggested job split: Job A (admin AI-config runtime safety).

### R002 Admin persistence + runtime-config seam have zero automated coverage that runs in CI

- Priority: **P2**
- Area: test coverage / CI parity — `internal/admin/pg`, `internal/ai/runtime.go`
- Evidence: `internal/admin/pg/` contains only `store.go` — no `_test.go` (the go test run reports

- Why it matters: the batch's two most behavior-defining pieces are unverified where it counts. The money/trust path
  (`admin_stardust_grants` + `admin_audit_log` written in one pgx tx, append-only, `ON CONFLICT DO NOTHING`) has no

  cleared-key resolution). Decide explicitly whether CI should run a Postgres service (so these run in CI) or document
  that DB integration coverage is local-only.
- Suggested job split: Job B (admin integrity + coverage).

### R003 `GrantStardust` money-path idempotency is enforced only by client convention; `ErrGrantIDRequired` is dead code

- Priority: **P2**
- Area: `internal/admin` domain / contract drift
- Evidence: `apps/api/internal/admin/service.go:276-279` — `grantID = strings.TrimSpace(grantID); if grantID == "" {
grantID = s.newID() }`. `EarnAdminGrant` dedups solely on the grant id, so an empty client id mints a fresh id and
  both the twinkle earn and the `admin_stardust_grants` insert happen again on retry → double credit. Meanwhile
  `ErrGrantIDRequired` is fully wired but never returned by the service: defined at
  `internal/admin/errors.go:24`, mapped at `internal/admin/rpc/reasons.go:8` (`ADMIN_GRANT_ID_REQUIRED`), handled at
  `internal/admin/rpc/server.go:314-315`, and asserted at `internal/admin/rpc/server_test.go:24`.
- Why it matters: AC7 and the proto comment promise end-to-end idempotency, but the server guarantee is upheld only
  because the web client always sends `crypto.randomUUID()` and disables the button while pending
  (`apps/web/src/features/admin-users/ui/UsersSection.tsx:126`, `:173`). Any non-web/direct-RPC caller (a script, a
  future mobile surface, a test) that omits the id double-credits on retry — on the one path that mints real balance.
  The wired-but-unreachable `ErrGrantIDRequired` is misleading dead code.
- Recommendation: return `ErrGrantIDRequired` on an empty id (server-side enforcement of the money-path contract);
  the FE already supplies one, so this only closes the direct-RPC hole and revives the dead error path.
- Suggested job split: Job B (admin integrity + coverage).

### R004 `RevokeAdmin` does not refuse a seed admin identified by email

- Priority: **P2**
- Area: `internal/admin` authorization / contract
- Evidence: `apps/api/internal/admin/service.go:194` refuses only id-seeds: `if _, isSeed := s.seedIDs[target];
isSeed { return true, ErrSeedAdminUndemotable }`. But `ADMIN_USER_IDS` supports **email** seeds and `isSeedIdentity`
  (`service.go:252`) checks both `seedIDs` and `seedEmails`; `RevokeAdmin` consults only `seedIDs`.
- Why it matters: revoking a seed-by-email admin (by their UUID) is not refused — `store.Revoke` removes no row and the
  method returns `(false, nil)`, reporting `isAdmin:false` even though `IsAdmin` still resolves them as admin via the
  seed email. Policy rule 1 / AC3 ("seed admins are undemotable → refused") is silently unenforced for email seeds and
  the RPC response is misleading. Not privilege escalation (seed status is env-derived, so the user keeps admin), but a
  broken invariant + a lying response.
- Recommendation: in `RevokeAdmin`, resolve the target's email via the directory and refuse with
  `ErrSeedAdminUndemotable` when it matches a seed email, mirroring `isSeedIdentity`.
- Suggested job split: Job B (admin integrity + coverage).

### R007 Web Google-OAuth denial/failure surfaces no error on the login page (AC5 gap)

- Priority: **P2**
- Area: `apps/web` auth (redirect round-trip, degraded path)
- Evidence: `apps/web/src/app/providers/auth-provider.tsx:66` hard-codes the Google `redirectTo` to
  `window.location.origin + '/'`. A denied consent redirects back to `/` with `?error=…`;
  `detectSessionInUrl` establishes no session, `bootstrap()` → `getSession()` returns null, and the machine settles
  `signedOut` (not `failed`, no `error`). The gate routes to `/login`, which mounts fresh with `error === null` and
  `method` defaulting to `'password'` (`apps/web/src/pages/login/ui/LoginPage.tsx:19,73-77`). Mobile _does_ surface it

- Recommendation: on the web return, detect `error`/`error_description` in the URL and drive the machine to `failed`
  (or seed the login page error) so `login_google_failed()` shows.

  given rare rotations. The key-drift case fails loudly (job retries), not silently, but is a real ops footgun for a
  two-process deploy.
- Recommendation: no disposal change strictly required; document the shared `LLM_KEY_ENCRYPTION_KEY` requirement across
  `api`/`worker` (partly captured in memory already). Optionally close the prior adapter on rebuild.
- Suggested job split: Job A (admin AI-config runtime safety).

## Cross-Cutting Themes

- **The novel seams are the least tested.** R001/R002/R003/R015 all trace to one root cause: the batch's two genuinely
  new mechanisms — the runtime DB→env→mock provider-config seam and the admin persistence/grant path — carry the
  thinnest automated coverage (zero for `runtime.go` and `admin/pg`), and the tests that exist lean on fakes that
  can't fail (R015) or client convention instead of server enforcement (R003). CI's missing Postgres (R002) means even
  the DB tests that exist don't run there.
- **A new shared FE error seam that its newest surface doesn't use.** Plan 59 built parse → i18n → toast; the plan-58
  admin console (landed in the same batch) renders raw `cause.message` (R006), swallows mutation errors (R009), shows
  zeros on failure (R005), and hardcodes copy (R012). The seam and its first non-adopter shipped together.
- **Guards narrower than the rules they're claimed to enforce.** `lint:raw-strings` passes over `{'unset'}` (R012),
  `lint:comments` passes over `T010`/`Ax` (R011), and the boundary audit checks string-presence not reachability
  (R008) / `COSIMOSI_ERROR_DETAIL` relies on operator discipline (R013). Documented-as-enforced, actually partial.
- **Web↔mobile drift on the shared auth surface.** Opposite `method` defaults (R016), duplicated error-toast glue
  (R010), and asymmetric OAuth failure surfacing (R007) — the two apps made independent choices on code that is meant
  to be parity-mirrored or promoted.

## Questions / Tradeoffs

- **R001 fix direction** — refuse the destructive admin action (safer, but a valid "clear then reselect" workflow
  needs ordering) vs. fall through to env/mock on a missing key (more forgiving, but silently serves mock extractions
  under a provider the operator thinks is configured). The factory already fails-fast on this exact case by design
  (`factory.go:168-170`), so refusing at the admin boundary is more consistent with that stance.
- **R002 / R017 CI DB + Go** — is running a Postgres service (and pinned Go) in CI worth the runtime, or is DB
  integration coverage deliberately local-only? If local-only, that should be stated in the tech doc so a green CI is
  not mistaken for full coverage.
- **R003** — is the money-path idempotency contract meant to be server-enforced or client-supplied? The proto comment
  and AC7 read as server-guaranteed; the code trusts the client.
- **R007 / R019 return-path** — is "Google always returns to the universe root" intended (change02 AC1) or an
  oversight relative to plan-53's `from` contract? Decide and document either way.
