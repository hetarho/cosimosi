# cosimosi 정책 (policy) — 최종 결과의 정의

> **policy = "무엇이 참인가"(최종 결과의 정의).** plan = "어떻게 만들었나"(구현 과정).
> 이 디렉터리는 cosimosi가 만족해야 하는 **정전(canonical) 규칙·조건·튜닝된 파라미터 값**을 도메인/경험별로 못 박는다.
>
> ⚠️ **핵심 규칙 — policy는 *지금 구현된 상태*만 담는다.** 계획·미구현·"예정"인 내용은 **절대 넣지 않는다**(그건 plan/ 소관). 정책의 모든 문장은 **현재 코드의 실제 동작과 일치**해야 한다 — 정책은 코드의 거울이다. plan이 구현 완료된 *그 시점에* 비로소 해당 규칙이 정책에 들어온다. 그래서 정책에는 상태 배지(✅/🚧)가 없다 — 적혀 있으면 곧 구현돼 있다는 뜻이다.

---

## 왜 분리하나 — 문서의 역할 분담

cosimosi의 스펙 문서는 **역할**로 나뉜다. 충돌하면 위에서 아래 순으로 우선한다.

| 문서 | 역할 | 질문 |
|---|---|---|
| [concept.md](../concept.md) | **비전(헌법)** | *왜* 만드는가 (엔그램 우주) |
| [architecture.md](../tech/architecture.md) | **구조** | *어떻게* 쌓는가 (FSD·WebGPU/TSL·Connect·sqlc) |
| [plan/](../plan/) | **구현 과정** | *언제·어떤 단계로* 만들었나 (번호별 태스크·체크박스) |
| **policy/ (이 디렉터리)** | **최종 결과 정의** | *무엇이 참이어야 하는가* (도메인 규칙·UX 규칙·정전 파라미터) |

**핵심 원칙:** plan은 *과정*에 초점을 맞추고, **최종 동작·규칙·수치의 정의는 policy가 단일 출처(SSOT)** 다. 같은 값(예: 반감기 30일, τ=0.75, A_MIN=0.05)이 여러 plan에 흩어지면 표류한다 — policy가 그 값의 권위다. plan은 policy를 *구현*하고, plan의 "완료 정의(DoD)"는 해당 policy 규칙을 충족했는지로 검증한다.

## 구조

```
spec/policy/
├── README.md            ← 이 문서 (철학·색인·유지 규칙)
├── domain/              ← 도메인별 규칙 (무엇이 그 도메인을 그 도메인이게 하는가; 현재 구현 기준)
│   ├── star.md          ← 별: 별이 되는 조건(1일기=1별), 형태·색·크기·밝기·시간 감쇠·잠듦
│   ├── synapse.md       ← 시냅스: 연결 생성(의미 KNN), weight·헵 강화, 시각·삭제 없음
│   ├── memory.md        ← 기억·엔그램: 2겹(불변 원본/가변 별), 감정 입력, 멱등·격리
│   ├── universe.md      ← 우주 전체: 좌표(정적 fibonacci)·권위, 배경, 삭제 없음
│   ├── navigation.md    ← 항행·카메라: 모드, fly-to 속도·감쇠, 비행 물리·줌
│   └── data-sync.md     ← 데이터 동기화: 쿼리 캐시·이벤트 무효화·병합 규칙·출처 경계 리셋
└── ux/                  ← 사용자 경험 규칙
    ├── interaction.md   ← 회상·기록·공동회상·체험(demo) 상호작용
    ├── landing.md       ← 랜딩 비전 showcase·이론 카드·내러티브
    ├── motion-accessibility.md  ← 모션·reduced-motion·접근성·glow
    ├── error-feedback.md ← 실패 상태 UX: 바운더리 3겹·폴백 카피·검증 메시지(흰 화면 금지)
    └── analytics-privacy.md ← 분석·관측: 이벤트 설계표·본문 금지·마스킹·Sentry·큐 요약 로그
```

- **domain/** 은 도메인별로 다시 나눈다. 새 도메인이 생기면 파일을 더한다(예: 소셜 레이어 도입 시 `domain/sharing.md`).
- 각 정책 파일은 **정의 → 규칙·파라미터(정전 값) → 불변식 → 구현 근거(이 규칙을 구현한 plan NN + 코드 경로)** 순으로 쓴다.
- **상태 배지·"계획됨"·"예정" 금지.** 정책에 적힌 모든 규칙은 *이미 구현돼 현재 참*이다. 미래 기능은 규칙으로 적지 않고, 필요하면 `## 정의`에서 "X는 plan NN에서 다룬다(아직 정책 아님)" 정도의 포인터만 둔다.

## 유지 규칙 (plan ↔ policy 동기화)

plan 스펙이 **구현 완료될 때**, 그 작업이 정한/바꾼 최종 동작·규칙·수치를 해당 policy 파일에 반영한다(없으면 신설). 정책은 *현재 구현 상태*만 담으므로, **아직 구현되지 않은 plan의 규칙을 미리 policy에 적지 않는다** — 구현이 끝난 그 시점에 들어온다. 이것이 plan과 policy를 갈라놓지 않는 유일한 규칙이다.

1. plan의 "## 정책 영향(policy)" 절에 **이 작업이 구현 완료 시 policy에 추가·반영할 규칙**을 미리 적어둔다(이건 plan 문서이지 policy가 아니다).
2. 구현 중 정전 값을 확정하면(예: fly-to 감쇠 계수) plan에 적고, **구현이 끝나면 policy에 옮긴다** — plan은 과정 기록, policy는 그 결정의 영구 거처.
3. 구현 완료 후 정책이 코드와 **여전히 일치하는지** 확인한다(코드가 바뀌면 policy도 바뀐다). 정책에 상태 배지는 두지 않는다.
4. 값 충돌 시 **policy가 권위**다(단, policy는 코드 현재 상태와 일치해야 하므로, 불일치는 둘 중 잘못된 쪽을 고친다). plan 문서의 옛 수치는 policy를 가리키게 좁힌다.

> 헌법(불변 원칙)은 [concept.md](../concept.md)·[plan/00.overview.md](../plan/00.overview.md)에 있고, policy의 모든 규칙은 그 헌법을 깨지 않는다.
