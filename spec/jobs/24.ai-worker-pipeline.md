## Acceptance Criteria (from plan/22.ai-worker-pipeline)
<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->
- [ ] A1 The `Extractor`, `Embedder`, and `Semanticizer` **ports are declared in `internal/memory`** (consumer-owned), and
  their **concrete adapters live in `internal/ai`**; `internal/ai` imports no memory domain behavior and contains no
  split/dedup/synapse/gist logic [ARCHITECTURE §2.2/§2.4].
- [ ] A2 Each port has two adapters — a real LLM/embedding SDK adapter and a **keyless deterministic mock** — and the
  composition root (`cmd/api` / `cmd/worker`) selects between them by config (API-key presence), with no change to
  `internal/memory` when switching [§2.4/§2.8].
- [ ] A3 The keyless mock runs with **no API key and no network**, and is **deterministic**: the same input yields the same
  split/name/neurons, the same `ai.embedding_dim`-length embedding vector, and the same 4 gist stage texts — so tests
  and golden-parity fixtures pass offline.
- [ ] A4 Both the real and mock `Extractor` return only the **schema-forced** shape `{memories:[{name, mood, neurons:[{name,
  type}]}]}` — no position/color/strength/time/delete field is present, so a prompt-injected reply cannot structurally
  yield an invariant violation [W4a].
- [ ] A5 The worker claims jobs with `SELECT … FOR UPDATE SKIP LOCKED`, moves them `pending→running`, and on failure retries
  with **exponential backoff via `next_run_at`** (base `ai.job_backoff_base_ms`) up to `ai.job_max_attempts`, after
  which the job is marked `failed` — and no failed/exhausted job deletes any neuron, memory, or synapse [I1][§2.8].
- [ ] A6 Concurrent worker goroutines/processes claim **disjoint** jobs (no two claim the same row) via `SKIP LOCKED`.
- [ ] A7 `cmd/worker` drains the `jobs` table and **dispatches each kind to the owning memory use-case**: `embed` (embed
  neurons → `embeddings`) and `semanticize` (write the 4 gist texts to `episodic_memories.semantic_stages` [C7]).
  The worker runs as a standalone process **and** can run as a goroutine in dev.
- [ ] A8 `extract`, `link`, and `consolidate` are valid `jobs.kind` values (plan 16) but have **no dispatch branch** in
  Epic A (`extract` is synchronous via `SplitDiary`; `link` is synchronous at launch, plan 21; `consolidate` is
  Epic E); an unhandled kind is marked `failed`, never panics.
- [ ] A9 The real adapters enforce cost metering: a call whose output would exceed `ai.per_call_token_cap` is bounded, and a
  user over `ai.daily_call_cap` gets a typed cost-limit error (job retries later, sync split returns a limit error);
  identical inputs are cache-eligible so retries/re-runs do not re-bill. The mock is unmetered.
- [ ] A10 Neither the `Extractor` nor the `Semanticizer` (real or mock) writes or reshapes a `Diary`; the `semanticize` job
  writes only `episodic_memories.semantic_stages` [I2][C7].
- [ ] A11 Slow results are **optimistic**: after `LaunchStars` returns ids, embeddings and `semantic_stages` appear on the
  **next `GetUniverse` read/refetch** with **no polling and no server-streaming**; a not-yet-filled `semantic_stages`
  reads as `NULL` [§2.8][§2.9 #9].
- [ ] A12 The `jobs`-queue enqueue happens in the **same transaction** as the launch persist (plan 20), so a committed launch
  always has its `embed`/`semanticize` jobs queued — no lost work.

## Implementation Checklist
<!-- How to build it, top to bottom. [P] = parallel (different files, no dependency). Flag (gen)/(migrate)/(gen:values).
     Config/tuning numbers are never hardcoded — add them to spec/values.yaml, then import the (gen:values) constant. -->
- [ ] T001 **(gen:values)** Add the four `ai.*` keys this unit introduces to `spec/values.yaml`: `ai.per_call_token_cap`
  (max output tokens per single LLM call), `ai.daily_call_cap` (max billable LLM/embedding calls per user per day),
  `ai.job_max_attempts` (retry cap → `failed`), `ai.job_backoff_base_ms` (exponential-backoff base for `next_run_at`).
  The plan defers the numbers to implement — **choose and record them now**, plus the two deferred tuning decisions:
  the daily accounting window (universe-day vs calendar-day) and the exact backoff curve (e.g. `base * 2^attempts`,
  jitter or not). Run `pnpm gen:values`; every task below imports the generated constants
  (`internal/platform/values` / `packages/config` — never hardcoded literals). `ai.embedding_dim` is **consumed**, not
  added — it is introduced by plan 16 (job 20).
- [ ] T002 [P] Declare the three **consumer-owned AI ports in `internal/memory`** (§2.3 `ports.go` shape, §2.4 — never in
  `internal/ai`), extending the context package job 20 created: `Extractor.Split(ctx, body, date, existingNeurons) →
  (memories, error)` plus the `ReviseSplit` variant (prior result + natural-language instruction, same return shape);
  `Embedder.Embed(ctx, texts) → ([]vector, error)` (dimension `ai.embedding_dim`);
  `Semanticizer.GenerateSemanticStages(ctx, memory) → ([4]string, error)`. Define the **schema-forced** result DTO as a
  memory-owned type — `{memories:[{name, mood, neurons:[{name, type}]}]}` with **no** position/color/strength/time/
  delete fields ([W4a] is structural). No SDK, proto, or sqlc type appears in any signature.
- [ ] T003 [P] Declare the consumer-side `LLMClient` / `EmbeddingClient` capability interfaces in `internal/ai`, next to
  the port adapters that consume them (plan 28 says they live beside their plan-22 consumers): a completion call whose
  request carries an **output JSON schema**, and `Embed(ctx, texts) → vectors` of `ai.embedding_dim`. **Minimal
  surface only** — the provider set, native schema-forcing per vendor, and the concrete Anthropic/Voyage clients are
  plan 28's (job 25); add **no vendor SDK and no provider subpackage** here.
- [ ] T004 Build the **keyless deterministic mock** adapters in `internal/ai`, one per port: mock `Extractor` (split by a
  stable rule, name derived from the body, fixed neuron set — returning the T002 schema-forced DTO), mock `Embedder`
  (stable hash of the text → an `ai.embedding_dim`-length vector, via the generated constant), mock `Semanticizer`
  (4 gist stage texts derived from the memory's fields). Fully offline — no API key, no network; same input → same
  output; honors the same invariants as the real adapters (never mutates a `Diary` [I2]).
- [ ] T005 Build the **real adapters** in `internal/ai` on the T003 seam: the `Extractor` adapter owns the
  split/name/emotion/neuron-extract prompt, the `ReviseSplit` prompt, and the schema-forced output schema; the
  `Semanticizer` adapter owns the 4-stage gist prompt; the `Embedder` adapter maps texts → vectors via
  `EmbeddingClient`. Each maps seam request/response ↔ memory DTOs at its own boundary (the anti-corruption role,
  §2.4) — no memory domain *behavior* imported, no split/dedup/synapse/gist logic in `internal/ai`.
- [ ] T006 Wrap every **real** call in cost metering (the mock is free and unmetered): bound single-call output by
  `ai.per_call_token_cap` (the `max_tokens`-style ceiling on the seam request); enforce `ai.daily_call_cap` per user
  per T001's recorded accounting window, returning a **typed cost-limit error** the caller can distinguish (a job
  retries later per backoff; a synchronous split surfaces a limit error to the RPC); cache identical
  embedding/extraction inputs keyed by a request hash (adapter-internal) so retries/re-runs do not re-bill.
- [ ] T007 [P] **(gen)** Add the worker-side `jobs` queries under `db/queries/memory/` (the table and enqueue baseline are
  plan 16's; per-context query ownership, §2.3): claim the oldest due `pending` row (`next_run_at <= now()`) with
  `SELECT … FOR UPDATE SKIP LOCKED` and mark it `running` in one transaction; mark `done`; retry (`attempts + 1`, back
  to `pending` with a caller-computed `next_run_at`); mark `failed`. Return `user_id` from the claim so the dispatched
  use-case runs user-scoped and `pnpm lint:persistence` passes. Add the `semanticize` result query: update
  `episodic_memories.semantic_stages` (JSONB) for one memory scoped by `user_id` — touching no `diaries` column [I2].
  Run `pnpm gen:sql` (sqlc → `db/gen`, never hand-edited).
- [ ] T008 Implement the worker-side **`JobQueue` port** consumer-owned in `internal/memory` (plan 16 declares
  repository/`JobQueue` ports consumer-owned in plans 20–22) and its `memory/pg` adapter over the T007 queries,
  mapping sqlc rows ↔ a small job value (id, user, kind, payload, attempts) — no sqlc row escapes `memory/pg`.
  Backoff math uses `ai.job_backoff_base_ms` / `ai.job_max_attempts` per T001's recorded curve; an exhausted job is
  marked `failed` and kept — it never deletes any neuron, memory, or synapse row [I1].
- [ ] T009 [P] Add the **generic claim/backoff loop helper in `internal/platform`** (e.g. `internal/platform/jobqueue` —
  the plan's decision that the mechanism MAY live in platform): given a queue and a `kind→handler` dispatch map, poll →
  claim → run → complete on success | retry with exponential backoff up to the attempts cap → `failed`. It declares
  the tiny generic interfaces it consumes (consumer-owned, §2.4) and knows only generic job rows — no neuron, gist, or
  memory type. An unknown/unhandled `kind` is a defensive `failed`, **never** a panic.
- [ ] T010 Implement the two Epic A **job-kind handlers as memory use-cases** and register them in the dispatch map:
  `embed` → `Embedder.Embed` over the payload's neurons → upsert `embeddings` rows (plan 16's baseline insert query);
  `semanticize` → `Semanticizer.GenerateSemanticStages` → write the 4 gist texts to
  `episodic_memories.semantic_stages` via T007's query — never touching the `Diary` [I2][C7]. `extract` / `link` /
  `consolidate`: **no task — reserved seams** (valid `jobs.kind` values per plan 16, no dispatch branch; they fall
  through to the defensive `failed`).
- [ ] T011 Compose it: expose the **adapter-selection factory** from `internal/ai` (API key present → real adapter;
  absent → keyless mock — runtime env/secret wiring, no feature flag, no values key) and build `cmd/worker/main.go`
  (**NEW**, `main()` → `run(...)` style like `cmd/api`): wire the pg pool (`platform/db`), the `memory/pg` queue +
  use-cases, the AI adapters via the factory, and run the T009 loop as a standalone process. Wire the **same code** as
  an optional dev goroutine inside `cmd/api` so one `docker compose up` gives the full loop. Enqueue stays plan 20's
  (`LaunchStars` enqueues `embed`/`semanticize` in the launch transaction — this job only provides the queue surface
  it calls, A12's transactional shape).
- [ ] T012 Adapter tests: mock determinism **offline with no API key set** — same input → same split/name/neurons, same
  `ai.embedding_dim`-length vector, same 4 gist texts (golden fixtures pass with no network); schema-force — the
  `Extractor` result type used by both adapters structurally lacks position/color/strength/time/delete fields [W4a];
  metering — a request is bounded by `ai.per_call_token_cap`, an over-`ai.daily_call_cap` user gets the typed
  cost-limit error, identical inputs hit the cache (seam faked in tests; no billable call).
- [ ] T013 Queue-mechanism tests against the dockerized Postgres: concurrent claimants get **disjoint** jobs
  (`SKIP LOCKED`); `pending→running→done` happy path; failure → `attempts + 1` + exponential `next_run_at`;
  `attempts >= ai.job_max_attempts` → `failed` with the row kept and every domain row intact [I1]; unknown kind →
  `failed`, no panic; end-to-end — enqueued `embed`/`semanticize` jobs fill `embeddings` + `semantic_stages` while
  `diaries` is untouched [I2] and an unfilled `semantic_stages` reads `NULL` (the mechanism half of A11; the
  `GetUniverse` refetch surface is plan 23's).
- [ ] T014 Gates (BE scope): `go vet ./... && go build ./... && go test ./...` in Docker (`pnpm lint:api` /
  `pnpm test:api`), `pnpm lint:persistence` (every new query carries `user_id`), `pnpm gen:values` / `pnpm gen:sql`
  outputs committed and clean, no migrations added (plan 16 owns the schema), no FE surface touched.
