## Acceptance Criteria (from plan/32.reconsolidation-rules)
<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->
- [ ] A1 The prediction-error gate is a **consumer-owned LLM port** (`PredictionError.Differs(currentText, rewrite) → bool`)
  declared in `internal/memory`, reusing the plan-22 keyless-mock fallback + cost metering; the domain depends on the
  interface, and no LLM SDK is imported into the domain ([R6], ARCHITECTURE §2.4).
- [ ] A2 On **prediction error (Differs = true)**, the reconsolidation rule requires `current_text ← rewrite`, `seed ←
  Reshape(...)` ([V5]), regeneration of **only the not-yet-created remaining** stage texts with already-created gist
  stages kept ([C7]), and a `reconsolidated/user` provenance row ([R6][R8a]). (Executing it is plan 33 / job 44.)
- [ ] A3 On **no prediction error (Differs = false)**, the rule requires **reinforce only** — brightness reset ([R2]),
  recall LTP + strength bump ([R3]), gist-timer reset ([C6a]) — with `current_text`, `seed`, and stage texts
  **unchanged** and **no** `reconsolidated` provenance row ([R4][I8]).
- [ ] A4 `Reshape(currentSeed, newSeed)` returns a seed that **differs** from `currentSeed`, is pure/deterministic given
  the supplied `newSeed`, and is invoked **only** on reconsolidation — plain recall never changes `seed` ([V5][R4][I8]).
- [ ] A5 `EffectiveStrength(base, recallCount)` is **monotone non-decreasing** in `recallCount`, exhibits **diminishing
  returns** asymptotic to `synapse.strength_cap` (never exceeding it), and satisfies `EffectiveStrength(base, 0) = base`
  (no regression to the Epic-A stub) ([R3][V3], plan 18).
- [ ] A6 `NeighborForgettingDelta(sharedSemanticCount)` returns `0` at `0`, a **negative** delta at exactly `1` (slow
  forgetting), and a **positive** delta at `≥ reconsolidation.neighbor_speed_threshold (= 2)` (speed forgetting); the
  shared count is over **semantic neurons only** (spatial/entity excluded, emotion never counted — [I3]) ([R5], CC4).
- [ ] A7 A goose migration adds `episodic_memories.forgetting_offset_days REAL NOT NULL DEFAULT 0`; `pnpm db:migrate`
  applies cleanly and `pnpm db:down` reverses cleanly; existing rows read `0`; the read-time contract
  `effectiveElapsed = max(0, (now − last_recalled_universe_time) + forgetting_offset_days)` is fixed for plan 37 to
  consume (CC4).
- [ ] A8 A goose migration creates append-only-while-retained `memory_provenance (id, user_id, episodic_memory_id, kind,
  source, text, universe_time, created_at)` with `episodic_memory_id → episodic_memories(id) ON DELETE CASCADE`; it is
  **never** UPDATE-d, and is DELETE-d only by Epic H's post-window user full-delete sweep; the **created/original
  baseline is synthesized at read**, not backfilled (a memory with zero reconsolidations still yields a one-entry
  변천사) (CC5, [R8a][D1]).
- [ ] A9 **The `Diary` is never mutated** by any rule here: no `UPDATE diaries`, and `memory_provenance` carries no
  diary-as-mutable row — the objective original stays the `Diary`/export ([I2][R7]).
- [ ] A10 On recall, the ± offset applies to **neighbors only** (`forgetting_offset_days += delta`); the **recalled star
  itself** takes no self-offset and recovers wholly ([F5][R5]).
- [ ] A11 The neighbor nudge reuses `Depress`/LTD (associative/local) as its ≥2 mechanism and is **distinct** from
  `Downscale` (SHY, homeostatic/sleep, Epic E) — no `Downscale` is implemented here ([I9]).
- [ ] A12 **Golden-parity stance holds:** the numeric rules (`EffectiveStrength` recall term, `Reshape`,
  `NeighborForgettingDelta`) are pure and pinned by the plan-18 TS↔Go golden-fixture harness reading the same generated
  constants; the **prediction-error gate is a port, not golden-parity**, covered by contract tests instead.
- [ ] A13 Every `memory_provenance` and `forgetting_offset_days` read/write is **scoped to the authenticated user** and
  `pnpm lint:persistence` passes ([U1], §4).
- [ ] A14 No formula, curve shape, clamp, or enum content is stored in `spec/values.yaml`; only the coefficients/threshold
  listed in the plan's values section are values.
