## Acceptance Criteria (from code-review/05.plans-35-53-runtime-and-architecture-aud)

<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->

- [x] A1 Recall and whole-Diary recall obtain `needs_sync` from server-authoritative UTC state, never a local `Date`.
      The mutation validates explicit consent at execution, so UTC boundaries/client skew cannot advance time without
      consent or spuriously require it.
- [x] A2 `Recall`, `RecallDiaryStars`, and `ViewSemantic` require a client operation ID. Same ID + same canonical input
      returns the original committed response; concurrent duplicates and response-loss retries never spend or apply
      effects twice. Same ID + different input is rejected.
- [x] A3 Operation receipt, Twinkle debit, sync/effects, and result commit in one transaction. Spend rows use an
      operation-derived dedup key, and a duplicate append never applies a second balance delta, including each member of
      a whole-Diary recall.
- [x] A4 In-flight paid actions cannot be dismissed or repeated by Escape/backdrop/X/back/re-submit. Async completions
      are fenced by active operation/session epoch and cannot mutate a closed, reopened, retargeted, or signed-out flow.
- [x] A5 Known pre-commit refusals return safely to quote/consent. Ambiguous failures retain the operation ID and recover
      the receipt on retry rather than initiating a fresh blind spend.
- [x] A6 Recall resolves the selected memory and renders `currentDecayText(memory, universeTime)` as its real faded
      prompt; it never passes literal null or exposes the Diary.
- [x] A7 Successful/receipt-recovered Recall applies returned `currentText` and all existing authoritative fields, then
      refreshes GetUniverse, target provenance, and balance as appropriate. Server alone decides the outcome.
- [x] A8 Web/mobile parity tests cover UTC-12/UTC+14/skew, double-submit, close/back during loading, late completion,
      committed-response-loss retry, and prompt/current-text rendering. Apart from R006-R008, no unrelated behavior or
      plan 30/33/34/36/44/47 regression occurs.

## Review

- [x] `/code-review` (user-invocable only in this harness) covered by a careful acceptance-criteria and architecture
      self-review. The local `codex review` executable could not run because its Linux binary package is absent, so the
      same diff was reviewed manually and verified through focused plus full-repository gates. Findings:
  - **Applied — foreign/empty whole-diary recall (P1):** a `RecallDiaryStars` on a diary with zero live memories for the
    caller (an all-let-go diary, or another user's diary whose id references `diaries()` globally) used to advance the
    clock for free and write a cross-user receipt. Now refused with `ErrRecallNoLiveMemories` before sync/spend/receipt,
    so the whole transaction rolls back ([U1], the reader UI already disables the action at zero live stars).
  - **Applied — consent-loop misroute (P2):** an insufficient-balance (or other non-consent) known refusal while the
    clock was behind used to re-show the consent modal (loop). The FE now returns to the consent modal only for a
    genuine un-consented sync race, and re-quotes/charges otherwise (recall gates on `consented`; diary gates on the
    call's `consent`).
  - **Applied — ambiguous-retry recovery blocked by the quote gate (P1, A5):** after an ambiguous failure that had
    actually committed, the retry used to re-quote and a now-depleted balance blocked the free receipt replay. Recall
    keeps the passed cost gate and re-confirms; gist's retry re-issues the same operation id directly; diary re-issues
    once with the same id — all bypassing the quote so the committed receipt is recovered without a second spend.
  - **Applied — gist quote/spend stage mismatch (P1):** `QuoteSpendRequest.semantic_stage` now carries the selected
    risen stage. The server validates it against `ViewableGistStage` and prices that exact stage, matching the later
    `ViewSemantic` debit; web/mobile pending-spend inputs and query keys preserve the stage end-to-end.
  - **Applied — target/session late completion (P1, A4):** all three paid widgets use the shared synchronous session
    controller. Retarget, target clear/sign-out, close, and unmount invalidate the captured attempt; stale responses and
    stale `finally` blocks cannot mutate stores, invalidate current caches, animate, navigate, or clear a newer busy state.
  - **Applied — receipt tenant ownership (P1, U1):** receipt target FKs are composite `(user_id, target_id)` references,
    so PostgreSQL itself enforces that a receipt and its retained memory/Diary target have the same owner.
