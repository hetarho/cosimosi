## Acceptance Criteria (from plan/49.release-usecase)

<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->

- [x] A1 `Release` **soft-deletes** every `EpisodicMemory` linked to one `Diary` (`deleted_at = now()`, real-clock UTC) in one transaction, seals their activations, and applies [48.deletion-rules](../plan/48.deletion-rules.md)'s removal-set neuron rules (orphan → `sealed_at`; shared → kept + LTD on each removed memory's synaptic contribution); it performs **no hard delete** and never updates the `Diary` body in place. ([X1][X2][I1][I2])
- [x] A2 A `Release`d diary's memories are **excluded from all dynamics** — absent from `GetUniverse`, force-sim, decay, and consolidation — and their sealed shared neurons are preserved, not deleted. ([X3][I1], via [48.deletion-rules](../plan/48.deletion-rules.md))
- [x] A3 `Release` persists a retention-scoped release-effect ledger sufficient for exact restore: affected memory ids, orphan neurons sealed by the release, and synapse strengths/deltas changed by the release; ledger rows are swept with the release after the retention window. ([X2])
- [x] A4 `Restore` reverses a diary `Release` **within `release.soft_delete_retention_days`**: it clears `deleted_at` for all released memories, unseals the orphan neurons that Release sealed, and reverses the shared-neuron LTD from the ledger, returning the memories to full participation; it **refuses** once the release has been swept or if the diary release was not soft-deleted. ([X2])
- [x] A5 `SuggestLetGo` calls the `SealSuggester` over a **pre-filtered this-memory-only `semantic` neuron** candidate set and returns candidates for review; it **persists nothing** and can never surface a shared, spatial, entity, emotion, color, or star reference. ([X4][X6])
- [x] A6 `LetGo` seals **only the user-approved** neurons, after **server-side re-validation** that each approved id is a this-memory-only semantic neuron (a shared/foreign/non-semantic id is rejected); the star, emotion, color, and spatial/entity/time neurons are untouched. ([X4][X6][I11])
- [x] A7 Letting-go is **permanent** — `LetGo` writes no `deleted_at`, starts no timer, and has **no `Restore` path**; only full delete (`Release`) is restorable. ([X5])
- [x] A8 After `LetGo`, the star survives as a **silent engram**: its position/connections recalculate as the sealed semantic neurons drop out of the force-sim, while its color/emotion/size remain. ([X4][I5])
- [x] A9 The **only** hard delete is `RetentionSweeper`, which removes solely soft-deleted diary release groups (their original `Diary` row/body, `episodic_memories`, retained `memory_provenance`, exclusive sealed dependents, and release-effect rows) whose `deleted_at` is older than `release.soft_delete_retention_days`; it never deletes `deleted_at IS NULL` data, never deletes shared neurons, and runs with **no cron** (an operational trigger chosen at implement). ([X2][I1][T4])
- [x] A10 `SuggestLetGo` (and `Release`/`Restore`/`LetGo`) are mutations — **not** `NO_SIDE_EFFECTS`; `SuggestLetGo` invokes the cost-metered LLM but writes nothing. ([X6], §2.8)
- [x] A11 RPC handlers are **thin** (proto↔domain map + call); the soft-delete/seal orchestration, restore-window guard, approved-id re-validation, and retention rule live in the use-case/domain; no proto/sqlc type leaks into the use-case or pure domain; the `SealSuggester` and repositories are consumer-owned interfaces; orphan-ness / approved-id validity are server-side decisions. (§2.4/§2.7/§2.9#4/#6/#7/#8)
- [x] A12 Every read, seal, unseal, delete, suggestion, and sweep is **scoped to the authenticated user** and `pnpm lint:persistence` passes. (§4)
