## Acceptance Criteria (from plan/44.earn-spend-usecase)

<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->

- [x] A1 The concrete `internal/twinkle` **`SpendGate`** replaces the Epic-C no-op at `cmd/api`; a recall/gist-view now
      **derives balance → checks → deducts** before proceeding — the priced core loop is live. ([CC2][G1])
- [x] A2 `CheckAndSpend(scope, spend)` **prices the `SpendIntent`** via [43]'s `RecallCost` / `GistViewCost`, plans the
      draw with [43]'s pure `PlanSpend` and, on `ok`, **writes the spend rows basic-first then additional** ([G2]
      order); on `!ok` it returns the canonical `insufficient-twinkle`, writes nothing, and the recall/view is
      **refused** (nothing deleted [I1]). ([G1][G2])
- [x] A3 The spend is **atomic with its caller's transaction** — the recall/reconsolidation and its debit commit or
      roll back together; there is no charge without the recall and no recall without the charge. (§2.8, [CC2])
- [x] A4 The **price is derived by the gate** from [43.stardust-ledger](../plan/43.stardust-ledger.md)'s cost curve over
      the intent's depth signal (recall-by-decay-depth, gist-by-gist-depth); the **caller passes only the
      `SpendIntent`** and **never hardcodes or computes** a price. ([CC3][G4])
- [x] A5 `QuoteSpend` returns a server-derived `{cost, covered, shortfall}` for recall, gist-view, and diary-recall
      batch spends, is `NO_SIDE_EFFECTS`, and writes no ledger row; the real `CheckAndSpend` remains authoritative at
      action time. ([G4])
- [x] A6 **Earn-on-write** grants `twinkle.earn_write` to **additional** balance, fired from memory's launch via the
      consumer-owned `EarnPort` **inside the launch transaction**, once **per launched diary** (not per star); a
      **past-dated diary that launches no star earns nothing**. ([G3][I10])
- [x] A7 **Earn-on-invite** credits the inviter `twinkle.earn_invite_inviter` and the new friend
      `twinkle.earn_invite_invitee` on a **valid signup**, **idempotent per (inviter, invitee)** so a signup credits
      each side **exactly once**. ([G3])
- [x] A8 **Earn-on-payment** credits additional balance only after `Charge` verifies a store receipt through
      `StorePaymentVerifier`; repeated receipts are idempotent, and production grants no value without verification.
      ([G3])
- [x] A9 There is **no login/attendance earn** — the daily-reset basic balance ([43][G2]) plays that role. ([G3])
- [x] A10 **Core-loop protection** ([G5]) holds as a relationship: the daily **basic** reset covers expected everyday
      rumination ([M5]) at the cheap end of the recall curve — everyday recall stays free within basic; the gate bites
      only _excess_. Verified as a `basic_daily_amount ≥ expected_daily_ruminations × cheap_recall_cost` property, not
      a magic constant. ([G5])
- [x] A11 **`GetBalance`** returns the derived `{ basic, additional, total }`, is `NO_SIDE_EFFECTS`, and never earns or
      spends on read. ([G2])
- [x] A12 The **`valid-signup` seam** ([G6]) is a consumer-owned predicate with a permissive default (real distinct
      signup, not self-invite); the concrete anti-abuse criteria are a reserved slot bound later with no change to
      `Earn(invite)`. ([G6])
- [x] A13 Handlers are **thin** (proto↔domain map + call); balance-check, spend-order, earn-reason, idempotency, and
      valid-signup policy live in the use-case/domain; no proto/sqlc type leaks inward; `SpendGate`/`EarnPort`
      (implemented here), the ledger store, `SpendSignalReader`, the store verifier, and `ValidSignup` are
      consumer-owned interfaces; **twinkle never imports memory**. (§2.2/§2.4/§2.7/§2.9#4/#6/#7, [CC8])
- [x] A14 Every balance derivation, quote, earn, and spend is **scoped to the authenticated user**, and
      `pnpm lint:persistence` passes (§4).
