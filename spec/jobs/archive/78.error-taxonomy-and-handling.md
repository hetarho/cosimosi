## Acceptance Criteria (from plan/59.error-taxonomy-and-handling)

<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->

- [x] A1 **Domain errors carry a precise reason to the client.** A denied spend returns Connect `ResourceExhausted` **and** an `ErrorInfo` detail with `reason == "TWINKLE_INSUFFICIENT"`; a recall needing consent returns `FailedPrecondition` with `reason == "MEMORY_SYNC_CONSENT_REQUIRED"` — distinguishable from the other `FailedPrecondition` reasons.
- [x] A2 **Every failed RPC — domain or internal — includes a non-empty `request_id`** in its `ErrorInfo`, equal to the value the server logged for that call.
- [x] A3 **Internal errors are masked in production**: an unexpected error (e.g. a DB failure) returns `{code:"internal"}`, `reason == "INTERNAL"`, **empty `debug_detail`**, and generic message — the raw error string never appears in the client-visible payload.
- [x] A4 **Off-prod exposes the detail**: with the error-detail env gate on (dev/staging), the same internal error's `ErrorInfo.debug_detail` contains the raw error string, so a developer diagnoses from the browser without SSHing to the container.
- [x] A5 **No raw-error `default` remains**: every context `rpc` `domainError` routes unmapped errors through `apperr.Internal` (verified by test), and every enumerated domain error has a mapped reason.
- [x] A6 **FE decodes uniformly**: `toAppError` returns the correct `reason`/`requestId`/`retriable` for a domain error, an internal error, and a non-Connect error; a settled non-retriable failure surfaces the localized message, and an internal failure surfaces the generic message **with the correlation id**.
- [x] A7 **The current prod symptom is explainable end-to-end**: once shipped, the `GetBalance`/`SyncStatus` failure shows the operator a correlation id in the UI that resolves to the exact root cause in logs/Sentry (this plan makes the failure _diagnosable_; the underlying infra/schema fix is separate).
- [x] A8 No regression: existing green paths (successful RPCs, existing Connect-code behavior, retry policy) are unchanged; `pnpm gen`, build, lint, vet pass.
