## Acceptance Criteria (from code-review/05.plans-35-53-runtime-and-architecture-aud)

- [x] A1 **Production Charge is fail-closed.** With no configured store-verification adapter, `Charge` returns a
      canonical unavailable/not-verified error and appends no ledger entry or balance delta; arbitrary non-empty caller
      strings can never mint Twinkle. Deterministic/keyless payment verifiers are test-only or behind an explicit local
      development binding and are never the production default.
- [x] A2 A successful payment grant comes only from a verifier-authenticated claim that binds the provider transaction
      id, provider/store, known pack, authoritative amount, and authenticated beneficiary. A beneficiary mismatch,
      unknown pack, missing transaction identity, non-positive amount, or verifier failure credits nothing.
- [x] A3 A provider transaction is globally single-use across all users. A same-user retry returns the already-applied
      result without another earn; a different user cannot replay the transaction. The append-only Twinkle ledger is
      not rewritten or deleted to enforce this invariant.
- [x] A4 `ClaimInvite` resolves an opaque invite code through a trusted account/signup resolver. The resolver proves
      that the inviter exists, is distinct from the authenticated invitee, and that this signup is eligible; a raw user
      id, fabricated/nonexistent inviter, self-invite, or unavailable resolver credits neither side.
- [x] A5 A verified invite credits inviter and invitee atomically and exactly once for the trusted signup identity. A
      resolver, validation, deduplication, or persistence failure cannot leave a one-sided or partial grant.
- [x] A6 Thin RPC handlers preserve authentication and map the new fail-closed domain errors to stable Connect codes
      without exposing receipt bodies, provider secrets, or account-directory details in responses or telemetry.
- [x] A7 **No unrelated behavior change / no regression:** write-diary earns, balance derivation, daily basic reset,
      basic→additional spend order, quotes, and recall/gist spend atomicity retain their plan-43/44 behavior and tests.

## Implementation Checklist

      normalized provider transaction identity, store/provider, verified pack and amount, and beneficiary user id.

      only after existence/distinctness/eligibility checks. Keep provider SDK, auth-directory, proto, sqlc, and pgx

      verified beneficiary to `scope.UserID()`, derive dedup keys from normalized provider/signup identities rather than

      for unavailable trust adapters, beneficiary mismatch, invalid transactions, and ineligible invites.
