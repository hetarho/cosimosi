## Acceptance Criteria (from code-review/05.plans-35-53-runtime-and-architecture-aud)

- [x] A1 Every successful `Release` atomically schedules retention work for
      `deleted_at + release.soft_delete_retention_days`; the Diary and its released aggregate are hard-deleted even
      when that user never calls another RPC. The worker remains the no-cron operational trigger.
- [x] A2 Restore before the deadline prevents the scheduled work from deleting restored data. Restore at/after the
      deadline never resurrects data, and the expired group still converges to the same scoped sweep result.
- [x] A3 Pending, running, completed, and failed job rows contain target identities/revisions only—never Diary bodies,
      memory names/current text, neuron names, semantic stage text, or other user-authored source copies.
- [x] A4 Embed/semanticize handlers re-read the current user-scoped, live source immediately before external processing
      and validate the expected source revision. A missing, soft-deleted, sealed, or superseded target performs no
      external call and no derived write; a deletion/revision race before persistence is fenced to a no-op.
- [x] A5 `Release` cancels and redacts queued work for its target memories in the same transaction; `Sweep` removes
      remaining target jobs/metadata in FK-safe order. Previously claimed work cannot write derived results after the
      target is released or swept.
- [x] A6 Existing text-bearing job rows are safely migrated/scrubbed, and terminal jobs are automatically purged after
      one generated, documented retention window. Migration and cleanup never mutate a Diary body, provenance text, or
      append-only memory history.
- [x] A7 **No unrelated behavior change / no regression:** live jobs still retry/back off/lease-fence as before; live
      embed, semanticize, reconsolidation, and consolidation outputs remain equivalent; the 30-day Restore promise and
      the rule that only a user-originated release may cause hard deletion remain intact.
