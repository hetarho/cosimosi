## Acceptance Criteria (from plan/30.time-advance-usecase)
<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->
- [ ] A1 On **launch of a launchable diary** (`diary_date ≥ clock`, or first-ever launch), the star(s) rise **and** the clock
  advances to the diary date via `AdvanceClock`/the `GREATEST` upsert; there is **no** warning ([T2] case 1).
- [ ] A2 On **launch of a past-dated diary** (`diary_date < clock`), the `Diary` is saved, **no** star launches, and the clock
  is **not** advanced ([T1][I10]); this replaces Epic A's `max(created_universe_time)` guard with the stored clock.
- [ ] A3 The whole launch — Diary + memories + neurons + synapses + **clock advance** + progression hook — is **one atomic
  transaction**; a failure rolls all of it back together.
- [ ] A4 `SyncToToday` advances the clock to `utcDate(now())`, is **idempotent** the same day, and returns the interval
  `{ previous, current }`; it is a **capability** with no standalone RPC — recall (Epic C) composes it behind the
  sync-consent gate ([T2] case 2 / [R1a]).
- [ ] A5 `LaunchStarsResponse` carries `previous_universe_time` and `universe_time` (clock before/after the advance) so the UI
  can play the acceleration; for a past-dated launch both equal the unmoved clock (no acceleration).
- [ ] A6 `GetUniverse.universe_time` returns the stored `universe_state.current_universe_time` (contract shape unchanged),
  with a **one-release fallback** to `max(created_universe_time)` when the clock row is absent, and **nil** for an empty
  universe; the read never advances the clock ([T3], `NO_SIDE_EFFECTS`).
- [ ] A7 Every clock advance (launch **and** sync) fires `AdvanceProgression.OnAdvance(scope, from, to)` inside the advance
  transaction; Epic B ships it as a **no-op** default (forgetting is read-time; [C]/[E] writes bind here later), with
  **no cron** anywhere ([T4]).
- [ ] A8 The pre-existing **future-date rejection** (`diary_date > today + 1 day`) still runs before the transaction, so a
  launch can never advance the clock past real time ([I10]).
- [ ] A9 Handlers are **thin** (proto↔domain map + call); the read→constraint→advance→hook policy lives in the use-case/domain;
  no proto/sqlc type leaks into the use-case or clock domain; `UniverseClockStore` and `AdvanceProgression` are
  consumer-owned interfaces (§2.4/§2.7/§2.9#4/#6/#7).
- [ ] A10 Every clock read, advance, and progression call is **scoped to the authenticated user** and `pnpm lint:persistence`
  passes (§4).
