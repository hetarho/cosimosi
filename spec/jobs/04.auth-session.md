## Acceptance Criteria (from plan/04.auth-session)
- [ ] A1 Web and mobile have an auth provider/facade that exposes session status, user id, token access for transport, and auth actions without leaking Supabase objects into feature slices.
- [ ] A2 The session lifecycle is represented by an XState v5 machine whose context contains only serializable control data and ids.
- [ ] A3 RPC clients attach bearer tokens for protected calls and can be tested with an in-memory auth adapter.
- [ ] A4 The API validates Supabase JWTs for protected RPC methods and places the canonical user id in request context.
- [ ] A5 Public RPC methods remain callable without a token only when explicitly allowlisted.
- [ ] A6 Missing, expired, or invalid tokens return the canonical unauthenticated error shape.
- [ ] A7 A repository/use-case isolation helper or interface exists so later data units cannot accidentally query without a user id.
- [ ] A8 No onboarding flow, profile domain, invite flow, or product feature UI is introduced.

## Implementation Checklist
- [ ] T001 Before changing dependencies, check current official Supabase Auth, React Native token-storage, Connect auth interceptor, and JWT validation docs.
- [ ] T002 Add a platform-pure auth facade/session model shared by web and mobile, with no Supabase object leakage into feature slices.
- [ ] T003 Add the XState v5 session machine with bootstrapping, signed-out, signing-in, authenticated, refreshing, expired, and failed states; keep context serializable.
- [ ] T004 [P] Wire the web auth provider/adapter with browser-compatible Supabase session storage.

- [ ] T009 Add repository/use-case isolation helper types so future data code cannot run without authenticated user identity.
