## Acceptance Criteria (from plan/05.data-persistence)
- [ ] A1 `docker compose up` starts a local Postgres service with pgvector available.
- [ ] A2 `pnpm db:migrate`, `pnpm db:status`, `pnpm db:down`, and `pnpm db:reset` target the clean `apps/api` database path.
- [ ] A3 The initial migration enables required platform extensions and creates no product domain tables.
- [ ] A4 `pnpm gen:sql` runs sqlc when schema/query inputs exist and skips cleanly before they do.
- [ ] A5 Generated sqlc code is placed under an infra-only package path and is not imported by domain model/context behavior.
- [ ] A6 The API platform DB helper can open a pool, ping the database, and close cleanly in tests.
- [ ] A7 Persistence port/adapter examples or tests show consumer-owned interfaces declared where used and implemented by `<context>/pg` packages when an interface is actually needed.
- [ ] A8 The persistence convention requires future product tables and queries to be scoped by authenticated user id.
- [ ] A9 No ORM, server-authoritative client-derived layout state, or direct cross-context table access is introduced.
