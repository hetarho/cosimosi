## Acceptance Criteria (from code-review/06 · R007, R016, R019)

<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->

- [ ] A1 (R007) A failed/denied Google OAuth on **web** is surfaced on the login page (change02 AC5): on the return to
      the app with an `error`/`error_description` in the URL, the session machine is driven to `failed` (or the login
      page's error is seeded) so `login_google_failed()` renders. The flow stays retryable (already true).
- [ ] A2 (R016) The login `method` default is aligned web ↔ mobile: web adopts mobile's convention
      (`'password' | 'google' | null`, unknown reads as Google) so an ambient/bootstrap error shows consistent copy on
      both platforms.
- [ ] A3 (R019) `externalOAuthPending` is set **synchronously before** `await adapter.signInWithGoogle()` so a fast
      foreground (`AppState 'active'`) cannot leave the mobile machine stuck `signingIn`.
- [ ] A4 (R019) The fake adapter's `completeOAuthSignIn` mirrors the real adapter's `error=` parse (rejects on an error
      callback) **or** is documented as success-only.
- [ ] A5 (R019) `facade.dispose()` is either wired in an `AuthProvider` cleanup **or** plan 04 is updated to state the
      root facade is deliberately never disposed (resolve the spec-vs-impl gap either way); the Google `redirectTo`
      return-path behavior (always `/` vs. honoring `from`) is decided and documented in plan 04/53.
- [ ] A6 No behavior change to the success path: a successful Google or password sign-in navigates exactly as today;
      the session machine, adapter, refresh, and token storage contracts are untouched.
