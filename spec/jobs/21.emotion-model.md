## Acceptance Criteria (from plan/17.emotion-model)
<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->
- [ ] A1 `Emotion` is a value object on `EpisodicMemory` with fields `{ mood, valence, arousal, intensity }`, defined in the
  Go core domain context `internal/memory` and mirrored as a pure TS type in `packages/emotion`; it is a field of the
  aggregate root, never an independent aggregate ([M1]).
- [ ] A2 Each memory has exactly one primary `Emotion`; a diary splitting into N memories yields N independent emotions ([M1]).
- [ ] A3 The mood vocabulary is a fixed 13-value ENUM — `JOY, CALM, SAD, ANGER, FEAR, LOVE, NEUTRAL, EXCITEMENT, GRATITUDE,
  RELIEF, STRESS, TIRED, EMPTINESS` — used as the single naming authority across Go, DB, proto, and the FE mirror ([M2]).
- [ ] A4 The four V±×A± quadrants + neutral are represented as domain code (mood → quadrant map), matching the [M2] table;
  the per-mood `(valence, arousal)` scalars are read from `values.yaml`, not hardcoded.
- [ ] A5 A single pure function `moodColor(mood) → Color` in `packages/emotion` is the **only** source of a mood's color;
  no other slice, package, or shader binding computes mood color ([M3], [M6]).
- [ ] A6 The color table is code content in `packages/emotion` and is **not** present in `values.yaml`.
- [ ] A7 The palette is a named default behind one seam; a different palette can be supplied through the seam without editing
  any rendering consumer (`star`/`nebula` import only `moodColor`) ([M6]).
- [ ] A8 Star and nebula color derive solely from a memory's stored `mood` via `moodColor`; there is no code path that colors
  a body from anything other than its emotion ([M3], [V4], [I3]).
- [ ] A9 `arousalToInitialStrength(arousal) → base_strength` is a pure function bounded by `emotion.arousal_strength_min` and
  `emotion.arousal_strength_max`; it takes **only** arousal (the mood's arousal coordinate — no extra data) ([E8a]).
- [ ] A10 `valence` is never passed to any strength, position, or connection function; only color reads valence. No function
  signature accepts valence for a layout/strength/synapse computation ([I3], [M7]).
- [ ] A11 Swapping the palette cannot override a memory's stored `mood`/`valence`/`arousal`, cannot decouple "color =
  emotion", and cannot feed position/strength/connection; the seam consumes a `Mood` and returns a `Color` only
  ([I11], [P2]).
- [ ] A12 The arousal → strength math is mirrored TS↔Go and pinned by a shared golden-parity fixture; the Go and TS
  implementations produce identical `base_strength` for the same arousal inputs and the same `values.yaml` bounds.
- [ ] A13 Adding/removing a mood requires a synchronized change to the ENUM and both `values.yaml` number-maps
  (`emotion.mood_valence`, `emotion.mood_arousal`); a mismatch fails at `gen:values` / build.

## Implementation Checklist
<!-- How to build it, top to bottom. [P] = parallel (different files, no dependency). Flag (gen)/(migrate)/(gen:values).
     Config/tuning numbers are never hardcoded — add them to spec/values.yaml, then import the (gen:values) constant. -->
- [ ] T001 **(gen:values)** Add the `emotion` group to `spec/values.yaml`: `emotion.mood_valence` and
  `emotion.mood_arousal` (two one-level number-maps, one entry per mood — 13 keys each; choose + record the concrete
  coordinates now, with signs matching the quadrant table: V+A+ `JOY`·`EXCITEMENT`·`LOVE`, V+A− `CALM`·`GRATITUDE`·
  `RELIEF`, V−A+ `ANGER`·`FEAR`·`STRESS`, V−A− `SAD`·`TIRED`·`EMPTINESS`, `NEUTRAL` at valence 0; valence ∈ −1..1,
  arousal ∈ 0..1), `emotion.arousal_strength_min` / `emotion.arousal_strength_max` (choose + record the two bound
  scalars now), and `emotion.default_intensity` (choose + record, 0..1). Run `pnpm gen:values` — regenerates
  `packages/config/src/values.gen.ts` (TS) and `apps/api/internal/platform/values/values_gen.go` (Go). The color
  palette, quadrant structure, mood enum, and the strength formula stay **out** of `values.yaml` (code/content, plan
  §Policy/Values).
- [ ] T002 Go domain types in `internal/memory` (the core domain context owned by plan 16 / job 20 — that job must have
  created the context first; this job only adds emotion behavior, no tables). Invoke `/be-architecture`. Define (or
  complete, if job 20 stubbed it): the `Mood` enum with exactly the 13 constants (`JOY · CALM · SAD · ANGER · FEAR ·
  LOVE · NEUTRAL · EXCITEMENT · GRATITUDE · RELIEF · STRESS · TIRED · EMPTINESS`), the mood → quadrant map (four
  V±×A± quadrants + neutral, Russell 1980) as domain code, the pure coordinate helper `MoodCoordinate(mood) →
  (valence, arousal)` reading the two generated number-maps from `internal/platform/values` (the **only** way emotion
  axes enter downstream math), and the `Emotion` value object `{ Mood, Valence, Arousal, Intensity }` — a field of the
  `EpisodicMemory` aggregate root, not a root; pure Go, free of proto/sqlc/pgx/JSON/DB tags (§2.4).
- [ ] T003 Go arousal math in `internal/memory`: pure `ArousalToInitialStrength(arousal float64) float64` — linear
  interpolation of `arousal ∈ 0..1` across the generated `emotion.arousal_strength_min .. emotion.arousal_strength_max`
  bounds (decided 2026-07: linear; a non-linear curve would be a later formula change, not a values change). The
  signature takes a **single arousal scalar** — never an `Emotion`, never valence ([I3], [M7], [E8a]).
- [ ] T004 [P] Scaffold the **NEW** pure cross-app package `packages/emotion` (`@cosimosi/emotion`, §3.5): `package.json`
  (typecheck + vitest scripts, dependency on `@cosimosi/config`), `tsconfig.json`, `src/index.ts` — no `three`, no DOM,
  no native deps. Register `typecheck:emotion` and `test:emotion` in the root `package.json` chains (mirroring the
  other packages); `pnpm install` updates `pnpm-lock.yaml`.
- [ ] T005 TS domain mirror in `packages/emotion`: the `Mood` string union (13 canonical names mirroring the Go enum),
  the mood → quadrant map, `moodCoordinate(mood) → { valence, arousal }` reading the generated `@cosimosi/config`
  number-maps, and the pure `Emotion` type `{ mood, valence, arousal, intensity }`.
- [ ] T006 The palette in `packages/emotion`: exactly one pure entry point `moodColor(mood) → Color` — the sole source
  of a mood's color ([M3], [M6]); the default 13-entry mood → color table as **code content** (never `values.yaml`);
  and the swap seam (`resolvePalette()`-style, mirroring plan 14's skin seam) so a later palette-customization unit
  ([P1], Epic I) can supply a different table without editing any consumer. The seam's type shape enforces the
  presentation-only guard ([I11], [P2]): it consumes a `Mood` and returns a `Color` — no memory identity in, nothing a
  layout/strength function reads out. Follow V+ warm / V− cool ([P3]) in the default palette as a recommendation.
- [ ] T007 TS strength mirror in `packages/emotion`: pure `arousalToInitialStrength(arousal) → base_strength` — the
  same linear interpolation over the same generated bounds as T003, signature-restricted to a single arousal scalar.
- [ ] T008 Golden-parity fixture: the **canonical** fixture file lives at
  `packages/emotion/fixtures/arousal-strength.golden.json` (arousal inputs → expected `base_strength`, computed from
  the same `values.yaml` bounds), read by the vitest test in `packages/emotion`. Because the BE gate runs Go in Docker
  with only `apps/api` mounted (`scripts/go-check.mjs` mounts `apps/api` → `/app`), the Go test **cannot** reach
  `packages/emotion` — so check in a byte-identical **mirror** at
  `apps/api/internal/memory/testdata/arousal-strength.golden.json`, read by the Go test in `internal/memory`
  (standard `testdata/` access — works on host and in Docker). Add a **sync guard** on the TS side: a vitest test in
  `packages/emotion` asserts the mirror is byte-for-byte equal to the canonical fixture (content equality — the
  workspace root is reachable from vitest), so any drift fails `pnpm test`. Both implementations must agree with the
  fixture within a fixed floating-point tolerance (A12). Do **not** edit `scripts/go-check.mjs`.
- [ ] T009 Enum ↔ values sync guard (A13): a TS type-level assertion that the generated `emotion` number-maps' keys are
  exactly the `Mood` union (a mismatch fails `pnpm typecheck`/build) plus a Go test asserting the 13 `Mood` constants
  and the generated map keys match 1:1; also test that each mood's recorded coordinates carry the signs of its quadrant
  (A4). Add unit tests for `moodCoordinate`, `moodColor` (13 moods → 13 colors through the seam), and the palette-swap
  seam (a substitute palette re-colors without consumer edits, A7).
- [ ] T010 Guard sweep: grep that no package/slice other than `packages/emotion` defines a mood → color mapping (A5);
  confirm no function signature accepts valence for strength/position/connection (A10); confirm the palette seam
  cannot write back into domain state (A11). Reserved seams stay unbuilt — **no task**: LLM mood assignment (plan 20),
  writing-flow mood editing (plan 27), star/nebula rendering consumers (plans 24/26), forgetting-speed arousal
  modulation ([F6], Epic D), palette-customization UI ([P1], Epic I), the `episodic_memories` DDL (plan 16), and the
  proto mood field (arrives with plan 20's write contract, job 26; plan 23's read model, job 28, mirrors it).
- [ ] T011 Run the gates — FS scope, both sides: FE `pnpm typecheck`, `pnpm lint` (incl. `lint:fsd:layout`,
  `lint:language`, `lint:persistence`), `pnpm test` (incl. the new `test:emotion` + golden fixture), `pnpm build:web`;
  BE `pnpm vet:api`, `pnpm build:api`, `pnpm test:api` (go vet/build/test in Docker). No `apps/web/src` or
  `apps/mobile/src` diffs.
