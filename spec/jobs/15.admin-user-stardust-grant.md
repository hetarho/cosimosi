# Job 15: 관리자 사용자 목록 + 별가루 지급  (new)

> 구현 작업 문서. 출처 스펙: [plan/46.admin-user-stardust-grant](../plan/46.admin-user-stardust-grant.md).
> `/implement-job 15`이 아래 두 체크리스트로 구현한다. 끝나면 SSOT(plan/policy/values) 반영, status: done.

## 인수 조건 (acceptance — 기획 관점, plan/46.admin-user-stardust-grant에서)
<!-- 출처 스펙의 수용 기준. 구현 후 각 항목이 코드에서 참인지 검증한다. -->
- [ ] A1 `/admin?tab=users`에 진입하면 admin allowlist 사용자는 사용자 탭을 볼 수 있고, 비관리자는 기존 `/admin`과 동일하게 NotFound 위장 화면을 본다.
- [ ] A2 production DB에서 사용자 목록은 `auth.users`의 모든 사용자 id를 포함한다. 아직 일기·지갑·설정 행이 없는 사용자도 목록에 나타난다.
- [ ] A3 `auth.users`가 없는 로컬 DB에서도 목록 조회는 에러 없이 앱 도메인 테이블의 `user_id` 합집합으로 동작한다.
- [ ] A4 목록 행은 `user_id`, 별가루 유효 잔액, 지갑 시드 여부를 보여준다. 지갑 행이 없는 사용자는 DB를 쓰지 않고 `customization.starting_stardust`를 유효 잔액으로 표시한다.
- [ ] A5 검색어를 입력하면 서버가 `user_id` 부분 일치(대소문자 무시)로 필터링해 결과를 반환한다.
- [ ] A6 목록은 안정적인 `user_id` 오름차순 keyset pagination을 사용하고, `next_page_token`으로 다음 페이지를 이어서 조회할 수 있다.
- [ ] A7 관리자가 양의 정수 amount를 지급하면 target 지갑이 없을 경우 `starting_stardust`로 먼저 시드되고, 지급 후 잔액은 기존 유효 잔액 + amount가 된다.
- [ ] A8 지급은 한 트랜잭션이다. 잔액 증가와 감사 행 insert 중 하나라도 실패하면 둘 다 적용되지 않는다.
- [ ] A9 지급 성공 시 `admin_stardust_grants`에 관리자 id, 대상 user_id, amount, 이전/이후 잔액, 시각이 남는다.
- [ ] A10 0 이하 amount, overflow를 일으키는 amount, production에서 존재하지 않는 target user_id는 거부되고 지갑/감사 행을 바꾸지 않는다.
- [ ] A11 사용자 직접 충전·결제·환불·차감 경로는 생기지 않는다. 별가루 증가 경로는 시작 잔액 시드와 관리자 지급뿐이다.
- [ ] A12 신규 RPC는 모두 `AdminService` admin gate 뒤에 있고 unary다. build·lint·vet·기존 admin/커스터마이즈 테스트가 회귀 없이 통과한다.

### 값·계약·DB (먼저 — BE/FE가 공유할 바닥)
- [ ] T001 (gen:values) `spec/values.yaml`에 `admin.user_list_default_page_size: 50`, `admin.user_list_max_page_size: 100` 추가. FE/BE는 생성 상수만 import하고 page size 숫자를 하드코딩하지 않는다. `pnpm gen:values`.
- [ ] T002 (gen) `proto/cosimosi/v1/admin.proto`: `AdminUser`, `ListAdminUsersRequest/Response`, `GrantUserStardustRequest/Response` 메시지와 `ListAdminUsers`(`NO_SIDE_EFFECTS`)·`GrantUserStardust` unary RPC 추가. `pnpm gen`.
- [ ] T003 (migrate) `backend/internal/db/migrations/00011_admin_stardust_grants.sql` 신규: `admin_stardust_grants(id, admin_user_id, target_user_id, amount, balance_before, balance_after, created_at)` 감사 테이블 up/down. 잔액 컬럼은 지갑과 같은 정수 범위를 전제로 하고, 조회용 인덱스는 `target_user_id, created_at DESC`와 `admin_user_id, created_at DESC`까지만 둔다. `pnpm db:migrate`.
- [ ] T004 (gen) `backend/internal/db/queries/admin.sql`: 사용자 목록/검색 keyset query, target 존재 확인 query, 지갑 seed/update/audit insert/post-grant read query를 추가하고 sqlc 생성. `auth.users`는 `to_regclass('auth.users')` 가드로 접근하고, 없을 때는 앱 도메인 테이블 `user_id` 합집합 fallback으로 에러 없이 동작시킨다.

### BE — admin bounded context
- [ ] T005 `backend/internal/admin/admin.go`: `AdminUser`, `ListUsersInput`, `ListUsersResult`, `GrantStardustInput` 도메인 모델과 repository port 확장. 검증 sentinel은 `ErrInvalidPageSize`, `ErrInvalidGrantAmount`, `ErrUserNotFound`, `ErrStardustOverflow` 등으로 분리해 handler가 Connect code에 명확히 매핑하게 한다.
- [ ] T006 `backend/internal/admin/repository_pg.go`: `ListUsers` 구현. 검색어 trim 후 case-insensitive contains filter, `user_id > page_token` keyset, `ORDER BY user_id ASC`, `LIMIT page_size + 1`로 `next_page_token` 산출. 목록 조회는 `user_wallet`을 쓰지 않고 `COALESCE(wallet.stardust, values.CustomizationStartingStardust)`와 `wallet_seeded`만 반환한다.
- [ ] T007 `backend/internal/admin/repository_pg.go`: `GrantUserStardust` transaction 구현. target 존재 확인 -> 지갑 멱등 seed -> overflow guard 포함 잔액 증가 -> `admin_stardust_grants` insert -> post-grant `AdminUser` 반환을 한 트랜잭션으로 묶고, 중간 실패 시 지갑/감사 행이 모두 롤백되게 한다.
- [ ] T008 `backend/internal/admin/service.go`: page size default/cap, page token/search normalization, amount 양수 검증, production target 존재 규칙, overflow 에러 정리. 시작 잔액은 `values.CustomizationStartingStardust`, page size는 신규 admin values를 사용한다.
- [ ] T009 `backend/internal/admin/handler.go`: proto↔domain 매핑과 에러 매핑 추가. `ListAdminUsers`는 조회 전용, `GrantUserStardust`는 auth context의 관리자 `sub`를 `admin_user_id`로 전달한다. 신규 RPC는 기존 `AdminService` gate 뒤에만 노출된다.
- [ ] T010 BE 테스트: `backend/internal/admin/service_test.go`에 page size 기본/상한, 검색/토큰 validation, 0 이하 amount, unknown target, overflow 매핑을 추가. repository/통합 테스트 패턴이 있으면 unseeded wallet 지급(`starting_stardust + amount`), seeded wallet 지급, 감사 행 insert, transaction rollback을 검증한다.

### FE — `/admin` 사용자 탭
- [ ] T011 `frontend/src/pages/admin/api/admin-queries.ts`: `adminUsersQueryOptions({ userIdQuery, pageToken })`와 `useGrantUserStardust` 추가. 지급 성공 시 해당 admin users query를 invalidate하거나 응답 row로 캐시를 갱신해 잔액이 즉시 바뀐다.
- [ ] T012 `frontend/src/pages/admin/ui/UsersTab.tsx` 신규: 검색 input + 검색 버튼, 사용자 table/list, `user_id`·별가루 유효 잔액·지갑 시드 여부, 행별 positive integer amount input + 지급 버튼, `next_page_token` 기반 다음 페이지 버튼, loading/empty/error 상태. 이 범위 밖 기능(삭제·정지·이력 조회·이메일 검색)은 넣지 않는다.
- [ ] T013 `frontend/src/pages/admin/ui/AdminPage.tsx`: `TABS`에 `users` 탭 추가, `?tab=users` 딥링크 지원, `UsersTab` lazy/normal import 배선. 비관리자 NotFound 위장 흐름은 기존 admin shell 동작 그대로 유지한다.
- [ ] T014 FE 테스트/점검: 기존 테스트 인프라가 있는 범위에서 `isPermissionDenied` 회귀, users query key, grant mutation invalidate, `UsersTab` 검색/다음 페이지/지급 disabled 상태를 검증한다. 테스트 인프라가 빈약하면 최소한 타입체크와 수동 admin 화면 점검을 DoD에 남긴다.
