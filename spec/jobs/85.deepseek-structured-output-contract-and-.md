- [ ] A1 (R001) DeepSeek remains on the stable Chat Completions JSON Output endpoint and validates every non-empty
      `LLMRequest.OutputSchema` locally before returning an `LLMResponse`. An invalid requested schema or a response
      that violates it returns `MalformedStructuredOutputError`; a schema-violating value never crosses the
      `LLMClient` seam.
- [ ] A2 (R001/R005) Behavioral tests reject schema violations for `required`, `enum`, `additionalProperties`,
      `minItems`, and `maxItems`, while accepting a representative conforming object. The tests assert the returned
      type/behavior, not merely that schema text appeared in the prompt.
- [ ] A3 (R002) DeepSeek accepts content only when `finish_reason == "stop"`.
      `insufficient_system_resource` remains a retryable `RateLimitedError`; `length`, `content_filter`, `tool_calls`,
      an empty reason, and any unknown non-stop reason return `MalformedStructuredOutputError` even when the content
      is syntactically valid JSON.
- [ ] A4 (R002) A provider error is never inserted into the identical-input cache: the metering-seam regression test
      proves an identical retry invokes the provider again, while the first subsequent success is cached normally.
- [ ] A5 No valid-path behavior regresses: the stable DeepSeek endpoint, authentication, model override/default,
      JSON-only request mode, non-thinking mode, user id, token cap, consumer-owned `Validate` callback, and successful
      response bytes remain unchanged. No beta endpoint or provider fallback is introduced.
- [ ] A6 Plan 28 and the AI-worker as-built SSOT state the actual contract: a provider must enforce the whole requested
      schema before returning; DeepSeek does so with stable native JSON Output plus local JSON Schema validation.
