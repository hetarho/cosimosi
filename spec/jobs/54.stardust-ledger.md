## Acceptance Criteria (from plan/43.stardust-ledger)

<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->

- [ ] A1 A goose migration creates `twinkle_balances (user_id TEXT PRIMARY KEY, additional INT NOT NULL DEFAULT 0,
basic_spent_this_window INT NOT NULL DEFAULT 0, basic_reset_window DATE NOT NULL, updated_at TIMESTAMPTZ ...,
CHECK (additional >= 0), CHECK (basic_spent_this_window >= 0))` and an append-only `twinkle_ledger_entries` log
      with `UNIQUE (user_id, dedup_key)`; `pnpm db:migrate` applies cleanly and `pnpm db:down` reverses cleanly
      ([G2][I1]).
- [ ] A2 Both tables carry `user_id`, every query scopes by it, and `pnpm lint:persistence` passes ([U1], §4).
- [ ] A3 The balance is **two-tier**: `Balance = { basic, additional }`; `additional` is a **stored, carrying** counter,
      `basic` is a **derivation** that equals `twinkle.basic_daily_amount` at the start of a reset window and does
      **not** carry unspent amount into the next window ([G2]).
- [ ] A4 The basic reset cadence is a **real UTC calendar day**, explicitly distinct from universe time, and the reset
      is **lazy** (no scheduled/cron job — the window rolls forward on the first read/spend in a new day) ([G2][T4]).
- [ ] A5 `PlanSpend(basicRemaining, additional, cost)` draws **basic first, additional for the overflow**, returns the
      per-tier draw and an `ok` flag, and **never** produces a negative balance in either tier ([G2][G5]).
- [ ] A6 `RecallCost(accessibilityCost)` is **non-decreasing** in decay-depth (a more-decayed memory costs at least as
      much to recall), consumes the accessibility signal from [38.forgetting-cost](../plan/38.forgetting-cost.md) (no
      decay math here), and is bounded by `twinkle.recall_max_cost` ([F4][G4], CC3).
- [ ] A7 `GistViewCost(semanticStage)` is **non-increasing** in gist-depth (a deeper gist costs no more than a
      shallower one), consumes `semantic_stage` from [40.semanticization](../plan/40.semanticization.md), and is floored
      at `twinkle.gist_min_cost` (cheap but never free) ([R8][G4]).
- [ ] A8 The pure functions (`RecallCost`, `GistViewCost`, `PlanSpend`, `basicRemaining`) produce **identical outputs**
      Go (`internal/twinkle`) ↔ TS (`packages/twinkle-logic`) for a shared golden-fixture table within a fixed
      tolerance, and both read the same generated `twinkle.*` constants (golden-parity, per
      [18.synapse-plasticity](../plan/18.synapse-plasticity.md)).
- [ ] A9 The **balance is server-authoritative**: it is written only server-side, the FE reads and displays it (never
      advances it), and a spend is committed inside a single-writer transaction so concurrent spends cannot oversell
      ([I5]-consistent stored state, per [29.universe-clock](../plan/29.universe-clock.md)).
- [ ] A10 `twinkle_ledger_entries` is **append-only** (no system `UPDATE`/`DELETE`), and a retried earn/spend carrying
      the same `dedup_key` does **not** double-apply (idempotent) ([I1]).
- [ ] A11 `twinkle` is a **standalone context** that does **not** import `internal/memory`; the `SpendGate` and ledger
      repository ports are **consumer-owned** (declared by memory / the twinkle use-case, not pre-declared here), and
      only `twinkle/pg` imports `dbgen` (CC2/CC8, ARCHITECTURE §2.4).
- [ ] A12 No formula, curve shape, clamp, or the reset-window rule is stored in `spec/values.yaml`; only the
      coefficients, caps, and the daily grant listed below are values (formulas stay code).
