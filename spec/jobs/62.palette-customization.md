## Acceptance Criteria (from plan/51.palette-customization)

<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->

- [ ] A1 A **palette registry** in `packages/emotion` exposes **≥2** named `MoodPalette` values (the plan-17
      `defaultMoodPalette` plus at least one alternative) keyed by a stable **palette id**, with `paletteById(id)`,
      `listPalettes()`, and `DEFAULT_PALETTE_ID`; every entry passes `assertCompletePalette` (all 13 moods present)
      ([P1][M6]).
- [ ] A2 Palette **tables are code/content** in `packages/emotion` and appear **nowhere** in `spec/values.yaml`; the
      count of palettes is a derived array length, not a declared number ([M6]).
- [ ] A3 A per-user **palette preference** stores a single **palette id** (default `cosimosi-default` when unset),
      read/written through `GetPalettePreference` / `SetPalettePreference` on a thin RPC, scoped to the authenticated
      user; the read is `NO_SIDE_EFFECTS` ([P1], §2.7/§2.9).
- [ ] A4 `SetPalettePreference`'s **only** writable field is `palette_id`; the contract carries no mood, color,
      coordinate, memory id, position, or strength field — so a preference write cannot structurally reach the meaning
      layer ([I11][P2]).
- [ ] A5 `SetPalettePreference` accepts **only a registry-known id** (unknown id rejected on write / coerced to
      default on read); a client cannot inject an arbitrary color table through the preference ([I11][P2]).
- [ ] A6 At app init the stored preference is read and applied via **`setMoodPalette(chosen)` before the first
      render**, so the universe is colored by the user's palette from the first frame through plan 17's single
      `moodColor` seam; an unset / unknown id or a failed/unauthenticated read falls back to `defaultMoodPalette`
      ([M6]).
- [ ] A7 Setting the preference **persists the id and re-applies live** through `setMoodPalette`, re-coloring the
      running universe with **no** rendering-code change and **no** `GetUniverse` refetch (only `activePalette`
      changed) ([M6]).
- [ ] A8 `entities/star` and `entities/nebula` are unchanged and import only `moodColor`; a palette swap re-colors the
      whole universe through the one function with zero rendering edits ([M6][V4]).
- [ ] A9 A memory's color is **always** `moodColor(memory.mood)` under every palette; swapping the palette never
      overrides a memory's stored `mood`/`valence`/`arousal`, never decouples "color = emotion", and never feeds
      position, strength, or connection ([I3][I11][P2]).
- [ ] A10 A pure `checkPaletteAxisConsistency(palette)` validator flags warm/cool ↔ valence mismatches as
      **warnings** (not hard blocks), reading the mood valence from plan 17's `emotion.mood_valence` map; **every
      shipped registry palette returns no warnings** ([P3]).
- [ ] A11 The palette registry and guardrail are **frontend-only** (not golden-parity); the backend stores only the id
      scalar and computes no color, consistent with plan 17's projection stance.
- [ ] A12 **web + mobile parity:** both apps share the `packages/emotion` registry and the preference `api` segment
      verbatim; only the app-layer init host may fork (§3.5).

## Grounding

- Constitution ([I1]–[I11]): [00.overview](../plan/00.overview.md) §3 (_The constitution_). This job is bound above
  all by **[I11]** — customization cannot touch the meaning layer: the palette (emotion↔color) is swappable, but
  "color = emotion" and a star's emotion/position/strength are inviolable — enforced **structurally** (the contract's
  only writable field is `palette_id`; the seam consumes a `Mood`, returns a `Color`). It honors **[I3]** (emotion
  drives color only; the palette is a color projection and feeds nothing into position/connection).
- Architecture (placement): [ARCHITECTURE.md](../ARCHITECTURE.md) BE **§2.2** (a thin **supporting** context —
  `internal/account`; integrated through its published get/set behavior, never its table), **§2.4** (domain pure of
  proto/sqlc; consumer-owned store interface), **§2.7** (Connect unary; proto = DTO; the read is `NO_SIDE_EFFECTS`
  GET-eligible but user-scoped — never shared-CDN), **§2.9#8** (a _stored preference_ is legitimately server-owned;
  the color table it selects stays client-side) · FE **§3.1** (the init apply is an app-layer segmented step, not a
  feature; `features/change-palette` carries `model`/`api`), **§3.4** (the mood→color palette is the one-way
  domain→visual projection for emotion — a swap changes the projection, never the domain fact), **§3.5** (registry +
  `api` shared verbatim; only the init host forks) · **§4** (per-user isolation; values generated, never hardcoded).
  Invoke `/be-architecture` for every file under `apps/api`, `/fe-architecture` + `/mobile-architecture` for the FE
  slices.
- tech / policy / values touched: **values** — new `palette` group (1 key, T001, confirmed at implement); excluded:
  palette tables/ids/formula/schema/proto/count (code/content/contract per plan). **policy/ux** — the rule "a user
  may choose the emotion palette from a named registry; the palette is swappable presentation, but 'color = emotion'
  and a star's emotion/position/strength are immutable ([I11][P2]); axis-consistency is warn-only ([P3])" extends
  [policy/ux/emotion-palette.md](../policy/ux/emotion-palette.md) at completion (**not pre-created**). **tech/** —
  the as-built registry + guardrail + preference context extend [tech/emotion-model.md](../tech/emotion-model.md)
  (or a new account tech note) at completion. **Depends on** [job 21](archive/21.emotion-model.md) (the plan-17
  `moodColor`/`setMoodPalette` seam + `assertCompletePalette` + `emotion.mood_valence` — done),
  [job 04](archive/04.auth-session.md) / [job 05](archive/05.data-persistence.md) (per-user scope + persistence),
  [11.config-values](../plan/11.config-values.md) (the values pipeline). **Coordinates with**
  [job 39](39.3d-assets-and-background-design.md) (owns the default palette _values_). **Consumed by**
  [job 63](63.settings-page.md) (hosts the picker over this engine); the init applies on the entry
  [job 64](64.auth-universe-gate.md) formalizes. Rendering consumers
  ([24.star-neuron-rendering](../plan/24.star-neuron-rendering.md) /
  [26.nebula-color-field](../plan/26.nebula-color-field.md)) consume the swapped `moodColor` **unchanged**.
