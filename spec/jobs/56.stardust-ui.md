## Acceptance Criteria (from plan/45.stardust-ui)

<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->

- [ ] A1 A persistent **balance HUD** on the universe page shows **basic** and **additional** Twinkle distinctly
      (basic = daily-reset allowance, additional = permanent carry-over), read from `twinkle.v1 GetBalance` via
      `entities/twinkle`; it renders a figure from first login (basic is always granted) and updates whenever
      `GetBalance` refetches after a spend or earn ([G2][G5]).
- [ ] A2 `entities/twinkle` is a **pure** mirror (no three) with a `model` (the `TwinkleBalance` type + store, `total`
      derived as `basic + additional`) and an `api` that maps `GetBalance` + the earn/charge responses; it is shared
      verbatim web + mobile ([G2]).
- [ ] A3 A reusable **cost display** shows, for a pending spend, **the price** the spend will cost — a **recall**
      priced by the star's decay depth (deeper decay → costlier), a **gist-view** priced by gist depth (deeper gist →
      cheaper) — sourced from the ledger's server quote; the FE **never computes a price** ([G4], CC3).
- [ ] A4 The cost display states whether the balance **covers** the spend and, on **shortfall**, shows the shortfall
      and offers a **charge affordance** rather than dead-ending; it returns a proceed/cancel decision only and
      **never itself calls spend** ([G3][G4]).
- [ ] A5 The cost display is **composed by** the spend flows: [36.recall-flow-ui](../plan/36.recall-flow-ui.md)
      surfaces it before 회고하기 and [35.star-detail-panel](../plan/35.star-detail-panel.md) surfaces it before 요지
      보기; "이 일기로 태어난 별 보기" ([D3]) shows the **sum** of that diary's recall costs via the same display
      ([G4]).
- [ ] A6 **Free** surfaces show **no** cost: meta info (shape / emotion / 작성일 / 강도 / forgetting state), the free
      forgotten current text, and reading the original diary are never priced — the cost display appears only on 회상
      (recall) and 요지 보기 (gist-view) ([G1][G4]).
- [ ] A7 A **charge sheet** exposes the three earn paths: **payment** (tiers shown, platform store receipt sent to
      verified `Charge`, balance updated only after backend verification), **invite** (inviter bonus + both-sides
      grant on valid signup), and **write-earn feedback** (a restrained reward confirmation when a diary launch earns
      Twinkle); each credited amount lands in `entities/twinkle` ([G3]).
- [ ] A8 There is **no login bonus** anywhere — the charge sheet never offers one; the daily basic grant plays that
      role ([G3]).
- [ ] A9 **Everyday remembering is never blocked** by the cost gate: a recall affordable from the daily basic grant
      reaches the proceed path without ever hitting the shortfall / charge path ([G5][M5]).
- [ ] A10 The charge sheet's **control-state** is one XState machine (`idle → charging → (paying | inviting) → idle` +
      error paths); the balance figures, pending-spend cost, and charge results live in **Zustand/Query**, not machine
      context (§3.2).
- [ ] A11 The balance HUD, cost display, and charge sheet expose **no** meaning-layer or placement control — they show
      and transact Twinkle only, and never mutate a `Diary` ([I2][I11]).
- [ ] A12 **All** user-visible copy resolves through the i18n seam (no raw strings; `lint:raw-strings` passes), in a
      literary / restrained Korean voice, honest and unpressured about price.
- [ ] A13 **web + mobile parity:** the same widget + three features + the entity mirror run on both apps, sharing
      `model`/`api` verbatim; `*.native.tsx` siblings exist **only** for the overlay / sheet / HUD primitives (§3.5).

## Implementation Checklist

- [ ] T001 Build the `entities/twinkle` domain mirror (web + mobile — pure, **no three**, per plan 45 §Placement):
      `model` — the `TwinkleBalance` type (`basic`, `additional`, derived `total = basic + additional`) + the small
      Zustand store, shared verbatim per the plan-31 `entities/universe-clock` as-built precedent (the store seats in
      `@cosimosi/universe` beside `universe-clock-store.ts` — `packages/universe/src/twinkle-balance-store.ts`, exact
      seat confirmed at implement; the app entities re-export); `api` — the `twinkle.v1 GetBalance` call over the
      generated client mapped into the store, plus applying the earn/charge responses back into it; `index.ts` public
      API. Refetch on spend/earn resolution only — no polling (§2.7) (A1/A2).
- [ ] T002 [P] Add the charge-sheet machine per the as-built shared-machine precedent
      (`packages/universe/src/stardust.machine.ts`, the `writing-flow.machine.ts`/`recall-flow.machine.ts` home —
      §3.2 + §3.5 verbatim sharing): states **exactly per plan 45 §Charge-sheet control-state** — `idle` (HUD showing,
      sheet closed, `OPEN_CHARGE` → `charging`), `charging` (`PAY` → `paying`, `INVITE` → `inviting`, `CLOSE` →
      `idle`), `paying` / `inviting` (`done` → `idle` with the balance refetched; `error` → `charging`, retriable).
      Context holds phase + refs only — the balance figures, pending-spend cost, and charge results live in
      Zustand/Query (A10). Machine tests (`stardust.machine.test.ts`) covering the phase table + the retriable error
      paths; export from `index.ts`.
- [ ] T003 Build `features/twinkle-balance-hud` (web + mobile; `ui` reading `entities/twinkle`): the persistent,
      restrained overlay showing **basic** and **additional** distinctly (today's allowance vs. the permanent
      reserve); renders a figure from first login (basic is always granted — never an empty state); updates on every
      `GetBalance` refetch; figures only — no meaning-layer or placement control (A1, [I11]).
- [ ] T004 Build `features/spend-cost-display` (web + mobile): `api` — the `QuoteSpend` call carrying only the kind +
      target id (`episodic_memory_id` for recall/gist-view, `diary_id` for the diary-recall batch); `model` — the
      pending-spend + the proceed/cancel decision and its tiny control (idle / shown / insufficient), driven by the
      composing flow; `ui` — the price (recall: deeper decay → costlier; gist-view: deeper gist → cheaper — the
      server quote rendered verbatim, the FE never computes a price), covers/shortfall against `entities/twinkle`,
      and on shortfall the shortfall amount + a charge affordance that opens the charge sheet instead of dead-ending.
      It returns a decision only — **never itself calls spend** (A3/A4). The diary-recall batch quote renders the
      summed price through this same display; its composing page is [job 58](58.diary-reader-page.md)'s (A5).
- [ ] T005 Build `features/charge-twinkle` (web + mobile; `ui` + `api`): the three earn paths — **payment** (the tier
      figures from Epic-F generated config (`twinkle.charge_pack`), start the platform store purchase, send
      `{pack_id, platform, receipt}` to job 55's verified `Charge`; credit lands only after the backend returns the
      verified result), **invite** (surface the invite link/share, redeem via `ClaimInvite`, show the inviter bonus +
      both-sides grant), and **write-earn feedback** (the restrained reward confirmation when the launch's earn
      resolves — the write itself stays the writing flow's). Each credited amount lands in `entities/twinkle`
      (response apply or `GetBalance` refetch). **No login-bonus surface anywhere** (A7/A8).
- [ ] T006 Build `widgets/stardust` (web + mobile: `ui` — the `StardustOverlay` HUD host + charge-sheet host, with
      `*.native.tsx` siblings **only** for the overlay/sheet/HUD primitives; `model` — wiring the T002 machine + the
      balance/cost/charge stores; `index.ts`): composes the three features over the running `widgets/universe-canvas`

- [ ] T007 Compose the priced surfaces + app wiring (web + mobile): mount the overlay in the universe page's app-layer

- [ ] T008 i18n: the balance labels (기본 별가루 / 추가 별가루), the price / covers / shortfall copy, and the three

## Grounding

- Constitution ([I1]–[I11]): [00.overview](../plan/00.overview.md) §3 (_The constitution_). This job is bound by
  **[I2]** (presentation over immutable data — pricing a recall never mutates a `Diary`; the composed flows already
  keep the Diary untouched, and this unit adds no write surface) and **[I11]** (the HUD, cost display, and charge
  sheet expose no meaning-layer or placement control — the economy gates _access_ to a recall, never what the memory
  _is_; no emotion/position/strength word crosses these surfaces).

  `features/*`; `entities/twinkle` is a pure domain mirror with `model`/`api`; app-layer wiring stays segmented),

  **§3.5** (web + mobile peers; `model`/`api` + machine shared verbatim via `packages/`; `*.native.tsx` only for
  overlay/sheet/HUD primitives), **§2.7** (Connect unary — refetch on spend/earn, no polling/streaming). Invoke
