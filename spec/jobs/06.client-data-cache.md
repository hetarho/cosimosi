## Acceptance Criteria (from plan/06.client-data-cache)
- [ ] A1 Web and mobile app roots have QueryClient providers and can use the same generated/connect-query client facade.
- [ ] A2 QueryClient test helpers create isolated clients and fake transports for unit tests and `/test` panels.
- [ ] A3 Cache key helpers are generated or exported through API segments; feature code does not rely on ad-hoc string keys.
- [ ] A4 Optimistic write helpers support snapshot, rollback, and settle invalidation without embedding product-specific logic.
- [ ] A5 Idempotent unary reads can use HTTP GET, while writes remain POST.
- [ ] A6 User-scoped data is never configured for public/shared CDN caching.
- [ ] A7 The cache layer does not store per-frame coordinates, continuously-derived brightness, or other render-loop data.
- [ ] A8 Cache timing defaults are read from generated values, not hardcoded in feature code.
