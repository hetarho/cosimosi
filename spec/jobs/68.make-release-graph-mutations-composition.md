## Acceptance Criteria (from code-review/05.plans-35-53-runtime-and-architecture-aud)

- [ ] A1 With memories A and B sharing neuron N, any Release/Restore/Sweep ordering preserves N for every live or
      retention-restorable owner. Restoring A can never leave its N sealed because B's release recorded the seal, and
      the last expired owner still cleans up an actually exclusive orphan.
- [ ] A2 Release classification and letting-go eligibility count retained soft-deleted activations conservatively (or
      use an equivalent explicit seal-ownership model). Restore reclassifies release-origin seals; it never unseals a
      neuron permanently sealed by LetGo or one still legitimately owned by another release effect.
- [ ] A3 Launch, sync/recall/consolidation, Release, Restore, LetGo, and Sweep acquire one per-user graph-mutation lock
      before graph reads/writes, in a single documented lock order. Disjoint users remain concurrent and no lock depends
      on an existing universe-state row.
- [ ] A4 Contribution weakening cannot overwrite a concurrent Potentiate/Downscale result: affected synapses are
      row-locked or changed by an atomic delta, and the release ledger records the actual applied delta that Restore
      later adds to the then-current strength.
- [ ] A5 Existing invalid release-seal ownership is repaired non-destructively: any live activation made incomplete by
      the old overlap behavior is restored, release-effect metadata is reconciled, and no LetGo seal, live memory,
      provenance row, or append-only history is deleted or rewritten.
- [ ] A6 Concurrent release-vs-launch, release-vs-recall, release-vs-consolidation, release-vs-restore, and overlapping
      release tests converge to a valid serial outcome: no live activation references a release-sealed neuron and no
      synapse update is lost.
- [ ] A7 **No unrelated behavior change / no regression:** single-diary Release/Restore, 30-day retention, LetGo
      permanence, the canonical alive predicate, pure plasticity formulas, monotonic universe time, and per-user
      isolation retain their plan-48/49 behavior.

## Implementation Checklist

- [ ] T001 Choose and document the compositional ownership rule before coding. Preferred minimal model: an activation
      owned by any retained memory row (live or soft-deleted inside its Restore window) prevents a new release/LetGo
      seal; Restore reclassifies release-origin seal effects; Sweep determines exclusivity from all remaining retained
      activations. Keep this pure decision in `internal/memory/deletion.go`, not SQL or handlers.
- [ ] T002 Replace the clock-specific lock vocabulary with one consumer-visible per-user graph-mutation lock backed by
      the existing transaction-scoped advisory key (or add an exact shared alias). Embed it in `LaunchTx`, `RecallTx`/
      progression, and `ReleaseTx`; acquire it first in Release, Restore, LetGo, and Sweep as launch/sync already do.
      Establish one lock order (graph advisory lock → release-group/row locks → graph rows) to avoid deadlocks.
- [ ] T003 Update classification facts and the this-memory-only semantic queries so soft-deleted-but-retained memories
      still count as owners. Update Sweep exclusivity to inspect the release memories' activated neurons, not only the
      subset stored in `release_sealed_neurons`, so a conservatively preserved neuron is removed when its last retained
      owner is truly swept.
- [ ] T004 Make Restore reclassify every neuron activated by the restored memories against current retained/live owners.
      Unseal and retire conflicting **release-origin** seal effects when the restored memory needs the neuron, while
      preserving permanent LetGo seals. Keep group-row locking and the restore reversal in one transaction.
- [ ] T005 **(migrate)** Add a data-repair migration `apps/api/db/migrations/000NN_release_seal_ownership_repair.sql`
      if the chosen model needs ownership metadata; otherwise use a guarded migration to reconcile existing
      `release_sealed_neurons` rows whose neuron has a live activation. The migration must identify release-origin
      effects, preserve LetGo seals, avoid product-history mutation, and include a reversible schema Down path.
- [ ] T006 **(gen)** Add/modify the scoped queries in `clock.sql`, `deletion.sql`, and `release.sql`: shared graph lock;
      retained-owner activation facts; restore reclassification/effect retirement; last-owner sweep eligibility; and
      `FOR UPDATE` selection (or atomic delta update) for contribution synapses. Regenerate sqlc.
- [ ] T007 Update `memory/pg` adapters so weakening computes the canonical pure `Depress` result from locked rows (or
      applies an exactly equivalent atomic delta) and returns the actual pre/post delta. Preserve Restore's existing
      atomic `current strength + recorded delta` clamp and keep sqlc/pgx types at the adapter edge.
- [ ] T008 Extend pure tests with live, retained-soft-deleted, swept, and mixed-owner truth tables. Add use-case tests for
      Release A → Release B → Restore A, both restore orders, both sweep orders, and LetGo + Release interactions;
      assert seal ownership, memory liveness, exact LTD reversal, and final orphan cleanup.
- [ ] T009 Add deterministic Docker concurrency tests using barriers: release vs launch sharing a neuron; release vs
      recall Potentiate; release vs consolidation Downscale; release vs Restore/Sweep at the boundary; and two releases
      of overlapping graphs. Compare final rows to one of the permitted serial executions and assert user B never
      blocks or changes user A's graph.
- [ ] T010 [P] Update plan 48/49 as-built notes, `spec/tech/memory-persistence.md`, and
      `spec/policy/domain/deletion-rules.md` with retained ownership, restore reclassification, the shared graph lock,
      lock order, and lost-update-safe contribution weakening.
- [ ] T011 Run migration up/down/up, `pnpm gen:sql`, `pnpm check:gen`, persistence negative controls,
      `pnpm lint:api`, `pnpm vet:api`, `pnpm build:api`, `pnpm test:api` (including race/concurrency cases), and full
      `pnpm check`.
