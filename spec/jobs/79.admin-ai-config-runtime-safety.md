## Acceptance Criteria (from code-review/06 · R001, R020)

<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->

- [ ] A1 (R001) Clearing a provider key (`ClearProviderKey`), or changing a capability's selection, **cannot** leave
      the AI pipeline hard-failing: after any such operator action, `RuntimeConfigSource.effective()` for both the LLM
      and embedding capabilities resolves to a **buildable** config — either the selected provider _with_ a key, or a
      fall-through to env → keyless mock. A capability that selected a now-keyless provider never yields a DB cfg with
      an empty `APIKey` that `NewAdapters` then rejects.
- [ ] A2 (R001) The chosen mechanism is explicit and consistent with the factory's existing fail-fast stance
      (`factory.go:168-170`): **either** `ClearProviderKey`/selection-change is refused while a capability selects that
      provider (returning a wired domain error), **or** `effective()` treats "selected provider, missing key" as
      fall-through to env then mock. Behavior is covered by a `runtime.go` test (see job 80 for the broader test
      coverage; this job adds at least the cleared-key resolution test).
- [ ] A3 (R020) The shared `LLM_KEY_ENCRYPTION_KEY` requirement across `cmd/api` and `cmd/worker` is documented in
      `spec/tech/admin-console.md` §5 (a mismatch/unset on the worker while keys were stored by the API makes every
      resolve fail): the operational contract is stated, not just implied.
- [ ] A4 (R020, optional) On a fingerprint-triggered rebuild, the prior `Adapters`' disposable client is released (or
      a code comment records the deliberate no-dispose rationale for the app-lifetime client).
- [ ] A5 No behavior change to the happy path: a keyed, selected, implemented provider resolves and builds exactly as
      today; metering / error taxonomy / mock fallback below the seam are unchanged.

## Implementation Checklist

<!-- How to build it, top to bottom. [P] = parallel (different files, no dependency). Flag (gen)/(migrate)/(gen:values). -->

- [ ] T001 Decide the R001 mechanism (refuse-destructive vs. fall-through). Default per review: **fall-through** in
      `effective()` so a missing key for a selected provider degrades to env → mock rather than a hard error — this
      keeps the pipeline alive without constraining the operator's clear/reselect ordering. (If refuse is chosen
      instead, wire a new `ErrProviderKeyInUse` domain error + reason in `internal/admin` and gate `ClearProviderKey`.)
- [ ] T002 Implement the fix in `apps/api/internal/ai/runtime.go` `effective()` (lines ~60-77): when
      `found && provider != ""` but `ReadProviderKey` returns not-found (or empty), do **not** return the DB cfg with
      an empty key — fall through to `envCfg` (and its fingerprint) so `NewAdapters` gets either a keyed real client or
      the keyless mock.
- [ ] T003 If the refuse-path was chosen in T001 instead: guard `apps/api/internal/admin/service.go:392`
      `ClearProviderKey` (and the selection-set path) against a capability that selects the provider, using the
      consumer-owned catalog/config reader already available to the service.
- [ ] T004 Add a focused `apps/api/internal/ai/runtime_test.go` case proving the cleared/deselected-key path resolves
      to a buildable config (fake `ConfigReader` returning a selection with no key → `current()` succeeds, not errors).
- [ ] T005 (R020) Document the shared `LLM_KEY_ENCRYPTION_KEY` requirement in `spec/tech/admin-console.md` §5.
- [ ] T006 (R020, optional) In `runtime.go` `current()` rebuild branch (~line 116-121), release the prior adapter's
      client on swap, or add a one-line rationale comment if the app-lifetime client is deliberately not disposed.
