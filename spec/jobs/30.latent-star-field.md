## Acceptance Criteria (from plan/25.latent-star-field)
<!-- The source spec's acceptance criteria. After building, verify each is true in the running code. -->
- [ ] A1 **Rendering-only, no DB.** The latent-star field is generated and drawn entirely on the client; it produces **no**
  database rows, no RPC calls, and no server state. A `neurons` row exists only for an *activated* neuron, and it is
  written by Encode ([20.encode-usecase](../plan/20.encode-usecase.md)), not by this unit ([E7a]).
- [ ] A2 **Field density & placement.** On scene init the universe shows roughly `rendering.latent_star_count` gray points,
  each with **no brightness, no color, and no identity**, positioned within the hippocampus z-band
  `[force_sim.hippocampus_z_min, force_sim.hippocampus_z_max]` (from [19.force-sim-layout](../plan/19.force-sim-layout.md))
  and filling the visible hippocampus volume in x,y.
- [ ] A3 **Background layer.** The field renders as a background layer inside [23.universe-canvas](../plan/23.universe-canvas.md),
  *behind* the real stars/neurons/synapses of [24.star-neuron-rendering](../plan/24.star-neuron-rendering.md); real bodies
  always render on top.
- [ ] A4 **Awaken on new-neuron creation.** When a launch creates a new neuron
  ([20.encode-usecase](../plan/20.encode-usecase.md)/[27.writing-flow-ui](../plan/27.writing-flow-ui.md)), exactly one gray
  latent star plays an **awaken** animation and hands off to the real `cell-star`. Multiple new neurons in one launch each
  awaken a distinct latent star.
- [ ] A5 **Awaken anchor = near a recent neuron, else random.** The gray star chosen to awaken is the one nearest a
  *recently-active* neuron (the [L4] temporal window used conceptually); when there is no recent cue the pick is
  **random**. This selection is a client presentation decision — nothing about it is persisted or sent to the server.
- [ ] A6 **Final position emergent, not stored.** The awaken start point is only the animation's anchor; the real neuron's
  **final position is emergent** from the force-sim ([19.force-sim-layout](../plan/19.force-sim-layout.md)) and is **never
  stored** ([I5]). No "awaken position" is written anywhere.
- [ ] A7 **Instanced + TSL.** The field is drawn with `InstancedMesh` bucket(s) and a material authored in **TSL** (compiles
  to WGSL + GLSL; no raw-GLSL `onBeforeCompile`), mounted through the [14.rendering-foundation](../plan/14.rendering-foundation.md)
  renderer boundary — no slice imports `three` directly ([ARCHITECTURE §3.3](../ARCHITECTURE.md)).
- [ ] A8 **Count tunable; mobile lowers it.** `rendering.latent_star_count` is a single build-time generated scalar
  (`values.yaml` → `pnpm gen:values`); changing it changes the field density with no code edits, and the mobile MVP
  renders a reduced count ([ARCHITECTURE §3.5](../ARCHITECTURE.md)).
- [ ] A9 **Emergent stays rendered-not-modeled.** The unit adds **no** `latent-star` type to the Go domain, DB, proto, or the
  FE domain-mirror; visual words do not travel upward into the domain slices ([ARCHITECTURE §3.4](../ARCHITECTURE.md)).

## Implementation Checklist

- [ ] T001 Add `rendering.latent_star_count` to `spec/values.yaml` (under the existing `rendering:` group — the
  approximate number of gray latent-neuron points, [E7a], [V7]). The plan enumerates the key but leaves the number
  "roughly" — **choose + record the value now**. Run `pnpm gen:values` so `@cosimosi/config` exposes the camelCase
  constant; all code imports the generated constant, never a literal. The mobile MVP's **reduced count** must also come
  through generated values — if the reduction proves to need its own scalar (not a code-level composition choice), add
  it under `rendering.*` here too and record it; never a hardcoded fraction. **(gen:values)**
- [ ] T002 Create `entities/latent-star` `model` (a *visual* entity, §3.1/§3.4 — imports **no** domain-mirror entity):
  a **deterministic, seeded field generator** — a pure function producing instance transforms for
  `rendering.latent_star_count` points, PRNG seeded by the generated `force_sim.seed` (plan 19) so web↔mobile produce
  the identical, reproducible field; each point's z ∈ `[force_sim.hippocampus_z_min, force_sim.hippocampus_z_max]`,
  x/y filling the visible hippocampus volume; points carry **no brightness, no color, no name, no identity**. Also the
  slice field state the awaken feature reads (positions + "consumed" marks — a consumed point stops being drawn as
  latent). Build-order note: the `force_sim.*` generated constants land with job 23 (plan 19) — build after them.
- [ ] T003 Create `entities/latent-star` `ui`: the R3F background-layer component — one (or a few) `InstancedMesh`
  bucket(s), transforms written **once** into `InstancedMesh.matrix` at scene init; material authored in **TSL**
  through `@cosimosi/3d-renderer`'s seams (never `three` directly): a single uniform gray tint, small size, optional
  subtle **shader-time** ambient drift (meaning-free); **no per-frame React state and no force-sim participation** —
  latent stars are not sim nodes and never attract/repel real nodes.
- [ ] T004 Add `entities/latent-star` `config` (slice-local constants that are **not** tuning numbers — tuning stays in
  generated values) and the slice public `index.ts` (single public API, no wildcard barrel).
- [ ] T005 Mount the field inside `widgets/universe-canvas` (job 28's scene composition) as a **background layer
  behind** the plan-24 real bodies — arrange render order/depth so real stars/neurons/synapses always draw on top
  (AC A3); the widget imports the slice only via its `index.ts`.
- [ ] T006 [P] Create `features/awaken-neuron` `model`: the **seed-pick logic** ([E7a]) — (1) anchor = a
  *recently-active* neuron over the visible graph (read from plan 23's `entities/neuron` mirror; the [L4] window used
  **conceptually** — a client-side presentation heuristic, not a server fact); (2) seed = the latent star **nearest**
  the anchor; (3) a **random** latent star when there is no recent cue (e.g. the first-ever neuron); N simultaneous
  births consume N **distinct** latent stars (picks must not collide). The precise window + nearest-star metric are
  code-level presentation constants **chosen + recorded at implement**; if one proves genuinely tunable, promote it
  under `rendering.*` then via `pnpm gen:values`. Nothing here is persisted or sent to the server.
- [ ] T007 Model the awaken lifecycle (idle → awakening → done) as slice `model` control-state — an XState machine
  (`model/*.machine.ts`, §3.2) **only if** it needs discrete phases; per-frame position reads via `getSnapshot()`/refs,
  never 60 fps React state; several concurrent awakens (one per new neuron) must coexist.
- [ ] T008 Create `features/awaken-neuron` `ui`: the awaken animation — the chosen gray point brightens/scales/hands
  off while the real `cell-star` (plan 24) fades in at its **emergent** target position (arriving from the sim on the
  next `GetUniverse` read); the latent point is "consumed" (stops being drawn as latent; the remaining field count is
  unaffected). **Choose + record the animation vocabulary** (brighten vs. scale vs. dissolve, duration, easing) at
  implement — the motion curve / shader graph is **code**, never a `values.yaml` entry. No "awaken position" is
  written anywhere ([I5]).
- [ ] T009 Expose the feature's public `index.ts` reacting to **`new_neuron_ids`** (the `LaunchStarsResponse` field
  owned by plan 20 — only genuinely-*created* ids awaken; dedup onto an existing neuron does not) and mount the
  feature in the universe scene. The launch-flow invocation itself is plan 27 (job 32) — this job provides the seam +
  scene mount; until job 32 lands, drive the trigger from tests / the dev harness.
- [ ] T010 Mobile parity (§3.5): mirror `entities/latent-star` + `features/awaken-neuron` in `apps/mobile/src` and
  mount them in the mobile universe-canvas mirror — the field generator + TSL material are platform-pure and reused
  **verbatim**; the MVP renders the **reduced count** through the generated constant (T001) and may simplify the
  ambient drift; a `*.native.tsx` sibling exists **only if** the R3F host wiring genuinely differs — the
  field/animation logic does not fork.
- [ ] T011 Unit tests: field-generator **determinism** (same seed → identical field; count = the generated constant;
  every z inside the hippocampus band; no per-point identity/brightness/color) — AC A2; seed-pick logic (nearest to a
  recent anchor; random fallback when no cue; N births → N distinct picks) — AC A4/A5.
- [ ] T012 Boundary greps: no slice imports `three` directly (only `@cosimosi/3d-renderer`) — AC A7; the two new slices
  carry **no** `api` segment, no RPC call, no transport import (rendering-only, AC A1); no `latent` vocabulary appears
  in the domain-mirror entities, `packages/*` domain code, `proto/`, or `apps/api` — AC A9 (there is no BE diff at all).
- [ ] T013 Run the FE gates: `pnpm typecheck`, `pnpm lint` (incl. `pnpm lint:fsd:layout` — the new slices are
  segmented; `lint:raw-strings`; boundary probes), `pnpm test:web` + `pnpm test:mobile`, `pnpm build:web`. No
  `apps/api`, `proto/`, or `db/` diffs.
- *(No task — reserved / non-goals stay unbuilt: no DB schema / server / RPC surface ([E7a]); no real-neuron logic —
  extraction/dedup/activation/synapses are plans 20/21; no `cell-star`/`star`/`filament` bodies (plan 24); no renderer
  bootstrap (plan 14); no stored awaken position. A representational-drift *meaning* for the ambient wobble is a later
