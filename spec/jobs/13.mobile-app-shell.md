## Acceptance Criteria (from plan/13.mobile-app-shell)
- [ ] A1 `apps/mobile/src/app/App.tsx` is a provider composition root rather than a hello-world component.
- [ ] A2 The mobile app has a typed navigation skeleton with `Boot`, `ShellHome`, and `Diagnostics` or equivalent non-product routes.
- [ ] A3 Session, transport, QueryClient, state-machine, i18n, theme/design-system, observability, error-boundary, and native shell providers are composed in a documented order.
- [ ] A4 The shell can render in host tests with fake auth, transport, storage, locale, and observability adapters.
- [ ] A5 Mobile consumes the same generated/shared API, values, i18n, state-machine, and token contracts as web; no parallel mobile-only architecture is introduced.
- [ ] A6 Native-only concerns such as secure token storage, device locale, safe area/status bar, and link handling stay at app/shared boundaries and do not leak into feature/domain slices.
- [ ] A7 The diagnostics surface exposes provider health without secrets or product/private memory data.
- [ ] A8 Quality gates include mobile TypeScript/lint and a shell smoke test that does not require an emulator.
- [ ] A9 No product feature screen, universe renderer, RN WebGPU integration, new RPC, DB schema, or domain model is added.
