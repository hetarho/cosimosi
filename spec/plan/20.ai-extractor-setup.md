# 20. AI 추출기 포트 배선 (ai-extractor-setup)

> 죽어 있던 `ai.Extractor` 포트에 실제 LLM 어댑터를 배선해 일기를 사건 경계 기준 1~N개 조각으로 분절하고, 조각마다 감정·강도·개체를 구조화 출력으로 뽑는다. | 범위: BE | 선행: 05 | 차단: 21

## 목적 (Why)
05는 `Extractor` 포트와 빈 `Extraction`·`NoopExtractor`만 두고 실제 추출은 v1로 미뤘다. 하지만 사용자 의도의 핵심("하루의 일기에는 여러 감정이 섞여 있다 → AI가 여러 조각으로 나눠 여러 별이 된다")을 구현하려면 **먼저 추출기를 살려야** 한다. 이 스펙은 AI 토대 작업이다: `ai.Extractor`를 실제 LLM 어댑터(구조화 출력)로 채우고, 일기를 **사건 경계(event boundary)** 기준 1~N개 조각으로 분절하며, 각 조각의 감정·강도·개체를 뽑는다. **별 fan-out(1 일기 → N 별)은 하지 않는다 — 그건 21의 일이다.** 20은 포트·어댑터·팩토리·job kind·단위 테스트까지만이며, 21이 워커에서 이 추출기를 호출해 별을 분기한다.

## 범위
- **포함:**
  - `internal/ai`: 빈 `Extraction`을 채운다(`Extraction{ Segments []Segment }`, `Segment{ Index, Text, Mood, Intensity, Entities }`). 순수 도메인 — proto/db/json 태그 금지(헌법5).
  - `ai.Extractor` 뒤 **실제 어댑터**(`OpenAIExtractor` — chat/responses + JSON schema 구조화 출력) + **`MockExtractor`**(키 불필요·결정론적: 빈 줄/문장 단위 분절, NEUTRAL 감정 — 기본값).
  - `NewExtractor` 팩토리(`NewEmbedder` 미러링: `AI_EXTRACTOR=mock|openai`, `OPENAI_API_KEY` 재사용).
  - LLM 프롬프트 = **사건 분절 루브릭**: "장소·사람·활동/목표·주제·감정 톤 중 하나라도 전환되면 새 조각을 시작하라; 그런 전환이 ≥1개일 때만 분리, 아니면 1조각."
  - 견고성(concept §4.6 "LLM 출력은 깨질 수 있다"): 구조화 출력 강제·스키마 검증·범위 클램프·파싱 실패 시 **단일 조각 폴백**.
  - `job.Kind`에 `KindExtract` 추가. 워커는 21을 대비한 **no-op-safe `KindExtract` 분기 스텁**만 추가(실제 fan-out은 21).
  - config(`AI_EXTRACTOR`)·`.env.example`.
  - `internal/ai` 단위 테스트(Mock 결정성·다장면 ≥2 분절·폴백·클램프).
- **비목표(Out of scope):**
  - **별 fan-out**(1 일기 → N memories, `InsertMemory`, fragment 당 embed job) → **21(memory-fragmentation)**. 20은 `RecordMemory`가 `extract` job을 적재하지 **않는다**(그건 21).
  - **DB 마이그레이션**(`memories`에 `mood`/`intensity`/`fragment_index`/`fragment_text` 추가) → **21**.
  - **proto 계약 변경**(`RecordMemoryResponse.memory_ids` 등) → **21**.
  - **FE·랜딩 카드 분절 시각화 로직 자체** → **21**. (단, 본 스펙은 헌법 외 규약상 랜딩 카피 점검 + 체험 토대 표기 task를 둔다 — 아래 작업 참조.)
  - 흥분성·재공고화·관련성 감쇠 등 다른 이론 → **22~27**.

## 설계 요점
**`Extraction`·`Segment`(순수 도메인 — proto/db/json 태그 금지, 헌법5).** `ai` 패키지는 `memory.Mood`를 import하지 않는다(결합 회피 — 추출기가 별 도메인에 의존하면 모바일/타 어댑터 재사용이 깨진다). 대신 `ai`가 자기 소유의 작은 `Mood` 문자열 enum을 가지며 값은 `memory.Mood`와 1:1 매핑되도록 정렬한다(매핑은 호출부/21이 수행). `Intensity`는 `[0,1]`.
```go
// internal/ai/embedder.go — Extraction 채우기(05의 빈 struct 교체)
type Mood string // ai 소유 enum, memory.Mood와 값 정렬: "", joy, calm, sad, anger, fear, love, neutral
const (
    MoodUnspecified Mood = ""
    MoodNeutral     Mood = "neutral"
    // joy/calm/sad/anger/fear/love …(memory.Mood 7값과 동일 문자열)
)
type Entities struct{ People, Places, Topics []string }
type Segment struct {
    Index     int      // 0-based, 조각 순서(21이 fragment_index로 영속)
    Text      string   // 조각 요약/본문(임베딩 대상 — 원본 record 수정이 아님, 헌법1)
    Mood      Mood     // 조각 감정(검증·클램프된 enum 값)
    Intensity float64  // 0..1
    Entities  Entities // people/places/topics
}
type Extraction struct{ Segments []Segment } // 항상 ≥1개(폴백 보장)
```

**실제 어댑터 `OpenAIExtractor`(`ai.Extractor` 뒤, openai.go 패턴 재사용).** chat/responses API + **JSON schema 구조화 출력**(`response_format`/`json_schema`, `strict: true`)으로 `{segments:[{index,text,mood,intensity,entities:{people,places,topics}}]}`를 강제한다. `OPENAI_API_KEY` 재사용, 토큰 상한·텍스트 해시 캐시·`Metrics`(LLM 계측은 임베딩과 **분리** — §4.7 비용 관리)를 openai.go와 같은 방식으로 둔다. 프롬프트 = 사건 분절 루브릭:
```
시스템: 너는 일기를 '사건 경계'로 나눈다. 장소·사람·활동(목표)·주제·감정 톤 중
하나라도 바뀌면 새 조각을 시작한다. 그런 전환이 하나도 없으면 1조각으로 둔다.
각 조각에 감정(joy|calm|sad|anger|fear|love|neutral)과 강도(0~1), 인물/장소/주제를 붙인다.
대략 80~120단어당 1조각, 단 의미 전환이 우선. JSON schema만 출력.
```

강제할 JSON schema(요지):
```json
{ "type": "object", "required": ["segments"], "additionalProperties": false,
  "properties": { "segments": { "type": "array", "minItems": 1, "maxItems": 8,
    "items": { "type": "object",
      "required": ["index","text","mood","intensity","entities"],
      "additionalProperties": false,
      "properties": {
        "index":     { "type": "integer", "minimum": 0 },
        "text":      { "type": "string" },
        "mood":      { "type": "string", "enum": ["joy","calm","sad","anger","fear","love","neutral"] },
        "intensity": { "type": "number", "minimum": 0, "maximum": 1 },
        "entities":  { "type": "object", "additionalProperties": false,
          "properties": { "people": {"type":"array","items":{"type":"string"}},
                          "places": {"type":"array","items":{"type":"string"}},
                          "topics": {"type":"array","items":{"type":"string"}} } } } } } } }
```
schema의 `maxItems:8`은 1차 방어선이고, 응답 후에도 코드에서 `[1,5]` 클램프·하드캡 8을 재적용한다(LLM이 schema를 어겨도 안전하도록 — concept §4.6).

**`MockExtractor`(키 불필요·결정론적 — 기본).** API 키 없이 동작: 빈 줄(문단) → 없으면 문장(`. ! ?` + 개행) 단위로 분절, 각 조각 `Mood=MoodNeutral`·`Intensity=0`·빈 `Entities`. 같은 입력은 항상 같은 분절(테스트·키리스 E2E 보장, mock.go의 결정성 철학과 동일). 단어 수 규칙(아래 params)을 적용해 `[1,5]` 클램프·하드캡 8.

**견고성·검증·폴백(concept §4.6).** LLM 응답은 (a) JSON 파싱 → (b) 스키마 검증(필수 필드·타입) → (c) 정규화(`Mood`를 enum 화이트리스트로 매핑, 모르는 값은 `MoodNeutral`; `Intensity`를 `[0,1]` 클램프; `Text` trim·공백 정리; `Index` 0-based 재부여) 순으로 거른다. **어느 단계든 실패하면 단일 조각 폴백** = `Segment{Index:0, Text: 전체 원문, Mood: 사용자 힌트 또는 MoodNeutral, Intensity: 힌트 또는 0}`. 폴백은 에러가 아니라 정상 경로다(별이 0개가 되지 않도록 `Extraction.Segments`는 항상 ≥1).

**분절 파라미터(여기서 정의 — 21이 소비).**
- 대략 1조각 / 80~120단어, 결과 조각 수 `[1,5]` 클램프, 하드캡 8.
- <60단어 또는 단일 연속 장면(전환 0) → 1조각.
- 조각 시드(21이 별 form/positions에 사용): `seed = stableHash(diary_id ":" fragment_index ":" sha1(normalized_text))`. `normalized_text` = trim + 연속 공백 1칸 + 소문자. 결정론적·재현 가능(이 스펙에서 정의, 21이 호출).

**job.Kind + 워커 스텁.** `job.go`에 `KindExtract Kind = "extract"` 추가(`KindEmbed`와 나란히). `worker.go`의 `processOne`은 현재 `w.jobs.Claim(ctx, KindEmbed)`로 임베딩 job만 claim한다 — 21이 fan-out을 붙일 자리로 **no-op-safe `KindExtract` 분기 스텁**을 둔다. 20에서는 워커가 `KindExtract`를 적극적으로 폴링하지 않거나(임베딩 루프 불변 유지), 폴링하더라도 추출기 호출 후 별 생성 없이 곧장 `Complete`하는 안전한 빈 분기여야 한다. **20에서 `RecordMemory`는 `extract` job을 적재하지 않는다**(04의 적재 경로는 그대로 `embed`만, 21이 `extract` 적재를 연결) — 따라서 스텁이 실제로 도는 일은 21 전까지 없다.
```go
// worker.go — 21이 채울 자리(20은 빈 스텁만)
// case KindExtract:
//   ext, err := w.extractor.Extract(ctx, m.Body)  // 21: Worker에 ai.Extractor 주입
//   // 21: ext.Segments → InsertMemory fan-out + 조각별 embed job 적재
//   return w.jobs.Complete(ctx, j.ID)              // 20: 별 생성 없이 안전 완료
```
> ⚠️ 20에서 `Worker` 구조체에 `ai.Extractor`를 **주입하지 않아도 된다**(스텁은 미실행). 주입을 미리 해 두면 빌드 시 미사용 필드 경고가 날 수 있으므로, 주입·`extract` 적재·실제 fan-out은 모두 21에서 한 번에 배선한다. 20의 워커 변경은 `KindExtract` 상수 인지 + 안전 분기 주석/빈 case에 한정한다.

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 추출기가 의미 전환(장소·사람·활동·주제·감정 톤 중 ≥1)이 있는 다장면 일기를 받으면 THE SYSTEM SHALL `Segments`를 **2개 이상** 반환하고 각 조각에 `Index`(0-based 연속)·`Text`·검증된 `Mood`·`Intensity∈[0,1]`를 채운다. _(헌법7)_
1.2 WHEN 추출기가 <60단어이거나 전환이 전혀 없는 단일 연속 장면을 받으면 THE SYSTEM SHALL `Segments`를 **정확히 1개** 반환한다. _(분절 루브릭)_
1.3 WHEN 추출 결과 조각 수가 규칙상 한계를 넘으면 THE SYSTEM SHALL 조각 수를 `[1,5]`로 클램프하고 하드캡 8을 절대 넘기지 않는다. _(분절 params)_
1.4 IF LLM 응답이 JSON 파싱 실패·스키마 위반·빈 결과 중 하나이면 THEN THE SYSTEM SHALL 에러를 올리지 않고 **단일 조각 폴백**(`Text`=전체 원문, `Mood`=사용자 힌트 또는 `MoodNeutral`, `Intensity`=힌트 또는 0)을 반환한다. _(concept §4.6, 헌법7)_
1.5 WHEN 각 조각의 `Mood`를 정규화할 때 THE SYSTEM SHALL 화이트리스트(`""`,joy,calm,sad,anger,fear,love,neutral) 밖 값은 `MoodNeutral`로, `Intensity`는 `[0,1]`로 클램프한다. _(견고성)_
1.6 WHEN `AI_EXTRACTOR=mock`이면 THE SYSTEM SHALL `OPENAI_API_KEY` 없이 결정론적으로 분절하고, 동일 입력에 항상 동일한 `Segments`를 반환한다. _(헌법7, 키리스 E2E)_
1.7 WHEN `AI_EXTRACTOR=openai`인데 `OPENAI_API_KEY`가 비어 있으면 THE SYSTEM SHALL `NewExtractor`에서 명확한 에러로 실패한다(fail-fast). _(설정 선택)_
1.8 WHEN `AI_EXTRACTOR`가 알 수 없는 값이면 THE SYSTEM SHALL `NewExtractor`에서 `unknown AI_EXTRACTOR` 에러를 반환한다. _(NewEmbedder 미러)_
1.9 WHEN `ai` 패키지가 빌드되면 THE SYSTEM SHALL proto/db/json 태그 없는 순수 도메인 타입(`Extraction`/`Segment`/`Entities`)만 노출하고 `internal/memory`를 import하지 않는다. _(헌법5·7)_
1.10 WHEN 조각 시드를 계산하면 THE SYSTEM SHALL `stableHash(diary_id:fragment_index:sha1(normalized_text))`로 결정론적·재현 가능한 값을 반환한다(같은 입력→같은 시드). _(21 소비)_
1.11 WHEN `KindExtract`가 정의되면 THE SYSTEM SHALL 워커에 no-op-safe 분기 스텁만 존재하고 20에서는 `extract` job이 적재되지 않아 실제 fan-out이 일어나지 않는다(21이 연결). _(범위 경계)_

## 완료 정의 (DoD)
- `ai.Extraction`이 더 이상 빈 struct가 아니다: `Segments []Segment`(각 `Index`/`Text`/`Mood`/`Intensity`/`Entities`)를 노출하고, `ai` 패키지에 proto/db/json 태그·`internal/memory` import가 없다(헌법5·7, grep 확인).
- `AI_EXTRACTOR=mock`에서 **`OPENAI_API_KEY` 없이** 다장면 문단(예: "아침엔 카페에서 회의했다. 점심엔 친구와 싸워서 화가 났다. 저녁엔 집에서 평온했다.")을 추출하면 `Segments` ≥ 2가 나오고, 같은 입력은 항상 같은 분절(결정론적)을 낸다.
- 짧거나 전환 없는 단일 장면 입력은 정확히 1조각을 낸다. 조각 수는 항상 `[1,5]`·하드캡 8.
- 깨진/스키마 위반 LLM 응답을 주입해도 에러 없이 단일 조각 폴백(전체 원문, NEUTRAL 또는 힌트)으로 복구된다(`Segments` 항상 ≥1).
- `AI_EXTRACTOR=openai`는 `OPENAI_API_KEY`가 있을 때만 어댑터를 만들고(키 없으면 `NewExtractor` fail-fast), 알 수 없는 값은 unknown 에러(`NewEmbedder`와 동일 동작).
- `job.KindExtract`가 정의되고 워커에 no-op-safe 스텁만 존재한다(20에선 `extract` job 미적재 → 실제 fan-out 없음; 21이 연결).
- Docker/WSL: `docker run --rm -v ${PWD}/backend:/src -w /src golang:1.26 go build ./...`·`go test ./internal/ai/...` 통과(`.exe` 호스트 직접 실행 안 함).

## 과학적 근거
사건 경계 분절은 인지심리·신경과학의 **Event Segmentation Theory**에 근거한다: 사람은 연속 경험을 장소·인물·목표·인과 등이 바뀌는 "사건 경계"에서 자동으로 분절해 부호화하며, 이 경계가 기억의 단위가 된다. 하루의 일기를 단일 덩어리가 아니라 여러 사건 조각으로 나누는 것은 이 이론의 직접 구현이다. 한 사건이 별개로 부호화되는 것(분리 부호화)은 해마의 **패턴 분리**와도 일치한다. 또한 하나의 시간 단위 안에 **여러(섞인) 감정**이 공존한다는 점(감정 입자도/혼합 감정)이 조각별 감정·강도 부여의 근거다.

- 사건 분절 이론(연속 경험의 자동 분절·부호화 단위): Zacks JM et al., *Psychological Bulletin* (2007). [DOI](https://doi.org/10.1037/0033-2909.133.2.273)
- 사건 분절의 인지·신경 기제: Kurby CA, Zacks JM, *Trends in Cognitive Sciences* (2008). [DOI](https://doi.org/10.1016/j.tics.2007.11.004)
- 사건 경계가 기억 부호화·인출을 조직: Zacks JM et al., *Journal of Cognitive Neuroscience* (2011). [DOI](https://doi.org/10.1162/jocn_a_00078)
- 패턴 분리(유사 경험을 분리된 표상으로 부호화): Yassa MA, Stark CEL, *Trends in Neurosciences* (2011). [DOI](https://doi.org/10.1016/j.tins.2011.06.006)
- 감정 입자도·혼합 감정(한 순간에 여러 감정 공존): Erbas Y et al., *Cognition & Emotion* (2018). [DOI](https://doi.org/10.1080/02699931.2018.1465894)
- 일상 경험의 혼합 감정 동시성: Schneider S, Stone AA, *Psychology and Aging* (2015). [DOI](https://doi.org/10.1037/pag0000018)
