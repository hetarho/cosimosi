# 20. AI 추출기 포트 배선 - ai.Extractor + internal/llm 공급자 추상화 (as-built)

> 현재 구현 기준 문서. 일기를 사건 경계 기준 1~N 조각으로 분절하고 조각마다 감정·강도·정서가를 뽑는 추출 계층, 그리고 그 LLM 호출을 5개 공급자 뒤로 숨기는 `internal/llm` 추상화의 SSOT다. | 범위: BE | 상태: 구현 완료, 반영: 21 워커 fan-out · 29 감정 13종 · 34 admin-controlled 공급자 선택

## 목적

하루의 일기에는 여러 감정이 섞여 있다 → AI가 사건 경계로 여러 조각으로 나누고, 21이 그 조각들을 여러 별로 fan-out한다. 그 토대인 추출 계층이 이 스펙이다. 두 축으로 구성된다.

- `internal/ai`의 `Extractor` 포트와 실제 추출기: 일기를 1~N 조각으로 분절하고, 조각마다 감정(13종 — 29)·강도(arousal)·정서가(valence ∈ [-1,1])·개체를 구조화 출력으로 뽑는다.
- `internal/llm` 공급자 추상화: 단일 `llm.Client` 포트 뒤에 openai·gemini·claude·deepseek·grok 5개 공급자를 둔다. 도메인(ai)은 어떤 LLM인지 모른다(헌법7).

감정 모델은 삼분(三分)이다. **mood**(13종, 4사분면 — 색/UX 레이어, Lindquist 2012)·**intensity**(= arousal, 기억의 무게 — 편도체 BLA→해마 공고화, McGaugh 2004)·**valence**(부호 있는 둘째 경로 — 부정 기억의 내구·디테일, Kensinger & Corkin 2004). LLM이 가장 신뢰성 높게 뽑는 정동 차원이 valence라(최약은 arousal) valence만 연속 출력으로 받고, arousal은 별도 점수를 요구하지 않는다. 사건 경계 분절 자체는 Event Segmentation Theory(Zacks 2007)의 직접 구현이다.

## 현재 구현

추출 계층은 두 패키지로 나뉜다. `internal/ai`가 추출 품질 로직(프롬프트·스키마·검증·폴백·시드·계측)을 소유하고, `internal/llm`이 공급자 HTTP를 소유한다. ai는 `llm.Client` 포트만 본다.

### 어댑터 선택 — env 노브 없음 (34가 재배선)

원래 20의 `AI_EXTRACTOR=mock|llm` env 노브는 34(admin-console)에서 제거되었다. 현재 추출기 선택은 **admin 콘솔의 활성 LLM 선택을 따른다** — env 변경도 재시작도 아니다.

- `ai.NewExtractor(cfg, client, src)` (`backend/internal/ai/factory.go`):
  - `src == nil`(admin 배선 없는 단독 도구·테스트) → 키리스 `MockExtractor` 고정.
  - `src != nil` → `SwitchingExtractor`(실 LLM 추출기 + mock). `client`가 nil이면 `llm.NewResolver(src, cfg, nil)`로 직접 만든다.
  - `NewEmbedder`(`AI_EMBEDDER=mock|openai`)와 달리 `NewExtractor`에는 env 분기가 없다.
- `SwitchingExtractor` (`backend/internal/ai/switching_extractor.go`):
  - 매 `Extract` 호출마다 `src.ActiveLLM(ctx)`로 활성 여부를 확인하고, 활성이면 `LLMExtractor`로·아니면 `MockExtractor`로 라우팅한다.
  - 결과는 `switchTTL = 30s`로 캐시한다(34의 ≤30s 반영 보장과 동일). source 실패 시 마지막 경로를 유지한다(일시 장애에 mock으로 플랩하면 실 추출이 조용히 평탄화된다).
- 조립 루트(`backend/cmd/api/main.go`, `backend/cmd/worker/main.go`):
  - `extractor := ai.NewExtractor(cfg, llm.NewResolver(adminSvc, cfg, adminSvc), adminSvc)`. `adminSvc`(internal/admin)가 `llm.ConfigSource`·`llm.UsageSink`를 구현한다.
  - 같은 extractor 인스턴스를 워커(비동기 extract job)와 memory 서비스(동기 `SegmentMemory` 프리뷰)가 공유한다.

### `Extractor` 포트와 도메인 타입

- 포트 (`backend/internal/ai/extractor.go`): `Extractor.Extract(ctx, text) (Extraction, error)`. 모든 어댑터가 이 뒤에 끼워진다(헌법7).
- 도메인 타입 (`backend/internal/ai/embedder.go`): proto/db/json 태그 없는 순수 도메인. ai는 `internal/memory`를 import하지 않는다(결합 회피 — 추출기가 별 도메인에 묶이면 모바일·타 어댑터 재사용이 깨진다).
  - `Mood` — ai 소유 문자열 enum, 13값. `memory.Mood`(29의 13종)와 1:1 문자열 정렬되며 매핑은 호출부 책임. 값: `""`(MoodUnspecified)·joy·calm·sad·anger·fear·love·neutral·excitement·gratitude·relief·stress·tired·emptiness.
  - `Segment{Index int; Text string; Mood Mood; Intensity float64; Valence float64; Entities Entities}`. Index는 0-based 연속, Text는 조각 본문(임베딩 대상 — 원본 record 수정 아님, 헌법1), Intensity는 [0,1]=arousal, Valence는 [-1,1](0=중립/미상).
  - `Entities{People, Places, Topics []string}`.
  - `Extraction{Segments []Segment}` — 어댑터가 항상 ≥1 보장(폴백 형태).

### `LLMExtractor` — 실제 추출기

`backend/internal/ai/llm_extractor.go`. 프롬프트·canonical 스키마·검증/폴백·텍스트 해시 캐시·계측을 모두 ai가 소유하고, `llm.Client`로만 호출한다 — 공급자를 바꿔도 추출 품질 로직은 그대로다.

- 프롬프트 = 사건 분절 루브릭(`extractSystemPrompt`): "장소·사람·활동(목표)·주제·감정 톤 중 하나라도 바뀌면 새 조각, 전환 0이면 1조각." 감정 13종은 **사분면 먼저(HAP/LAP/HAN/LAN) → 사분면 내 선택** 순으로 고른다. 사분면 간 tie-break(calm vs tired = valence 부호, neutral vs tired = 신호 유무, anger vs stress = 지향)와 한국어 정동 레시피(한→sad·답답/억울→stress 또는 anger·화병→anger·지침/소진→tired·권태→tired/neutral·공허→emptiness·혐오→anger·뿌듯→joy/excitement·죄책감→sad/fear)는 29의 추출 가이드라인을 그대로 옮긴 것이다. 혼동 쌍(calm/tired, neutral/tired, anger/stress, excitement/fear)에 대한 few-shot 한국어 예시 6개를 동봉한다.
- canonical JSON schema(`extractSchema`, name `diary_extraction`): 전 공급자 공용. 모든 object에 `additionalProperties:false` + 전 필드 required. **수치 범위(`minimum`/`maximum`)·배열 제약(`minItems`/`maxItems`)은 스키마에 두지 않는다** — 공급자 strict 모드 지원이 갈리기 때문. 조각 수 클램프·범위 클램프는 항상 코드가 재적용한다.
- 입력 상한: `maxInputRunes`(= 4000, 임베더 `openai.go`와 공유)로 토큰 캡. `memory.MaxBodyRunes`가 이미 더 짧은 body를 거르므로 최후 가드다.
- `MaxTokens`: `extractMaxTokens = 8192`(≤4000 rune 일기의 5조각 + 개체가 여유 있게 들어감).
- 캐시: 텍스트 해시 키(`cacheKey`)로 `Extraction`을 메모리 캐시. 캐시 반환·저장 모두 `cloneExtraction`으로 깊은 복사해 호출자 변이가 캐시를 오염시키지 않는다.
- 계측: `ExtractMetrics`(`metrics.go` — ExtractCalls·ApproxTokens·CacheHits·TokenCapHits·Fallbacks). 임베딩 `Metrics`와 **분리**(LLM은 비싸므로 독립 계측, Architecture §4.7).
- 에러 분리: **전송 에러(HTTP/네트워크)는 에러로 올린다**(워커가 백오프·재시도 — 일시 장애로 일기를 영구히 1조각으로 평탄화하지 않기 위해). **내용 깨짐(JSON 파싱 실패·빈 결과)만 단일 조각 폴백**한다.

### 검증·정규화·폴백 (공유 헬퍼)

`backend/internal/ai/extractor.go`의 헬퍼가 모든 어댑터의 raw 조각을 보장된 형태로 거른다(concept §4.6).

- `decodeSegments(raw)`: 모델의 JSON을 raw 조각으로 파싱. 파싱 실패·빈 segments면 에러(호출부가 폴백 결정).
- `normalizeExtraction(raw, original)`: 빈 텍스트 조각 제거 → mood 화이트리스트(`moodWhitelist`, 13값; 밖이면 `MoodNeutral`) → intensity `[0,1]`·valence `[-1,1]` 클램프(NaN→0, `clampRange`) → 텍스트 공백 정리(`collapseSpace`) → Index 0-based 재부여. 살아남은 조각이 없으면 `fallbackExtraction`.
- 조각 수 클램프: `const maxSegments = 5`. 초과분은 **마지막 조각에 병합**(텍스트를 붙임 — 절대 버리지 않음). 그래서 일기의 모든 단어가 정확히 한 조각에 남고(21의 임베딩 fan-out이 텍스트를 잃지 않음) 결과 수는 항상 ≤5다.
- `fallbackExtraction(text, hintMood, hintIntensity)`: 전체 원문을 단일 조각(Index 0)으로. mood는 힌트 또는 neutral, intensity는 힌트 클램프, **valence는 항상 0**. 에러가 아니라 정상 경로 — 일기가 별 0개가 되지 않게 한다.
- `NoopExtractor`: 전체 텍스트를 단일 neutral 조각으로 반환(`fallbackExtraction` 형태 그 자체). 워커에 extractor가 nil일 때 기본값.

### `MockExtractor` — 키리스·결정론적

`backend/internal/ai/mock_extractor.go`. API 키 없이 동작하는 기본 경로(MockEmbedder의 철학과 동일 — 의미가 아니라 안정성을 모델링).

- 빈 줄(문단, `splitParagraphs`)로 분절 → 조각이 2개 미만이면 문장(`. ! ?` + 개행, `splitSentences`)으로 분절.
- 모든 조각은 `Mood: MoodNeutral`·Intensity 0·Valence 0·빈 Entities. 실제 감정 추출은 LLM 어댑터의 일.
- `normalizeExtraction`을 통과시켜 `[1,5]` 클램프·0-based index·빈 입력 폴백을 동일하게 적용.
- 같은 입력은 항상 같은 분절(키리스 E2E·테스트 재현성).

### 조각 시드 (21 소비)

조각별 형태/방향 시드는 현재 저장된 별 id와 조각 index에서 파생한다. 과거 AI seed helper는 prod 호출 경로가 없어 제거됐다.

### `internal/llm` — 공급자 추상화 (헌법7)

`backend/internal/llm/llm.go`가 순수 포트·타입을 정의한다. 모든 공급자 HTTP 세부는 이 패키지 안에만 산다.

- `Client` 포트: `Complete(ctx, Request) (Response, error)` + `Model() string`. unary 텍스트 완성, 선택적 JSON schema 구조화 출력.
- `Request{System, User string; Schema *Schema; MaxTokens int}` · `Response{Text string; Usage Usage}` · `Schema{Name string; Raw json.RawMessage}`.
- `Usage{InputTokens, OutputTokens int}` — 34의 단위경제 입력(공급자 usage 메타에서 best-effort 매핑).
- `ConfigSource`(34) — 런타임 활성 선택의 출처 포트: `ActiveLLM(ctx) (provider, model, apiKey string, ok bool, err error)`. internal/admin이 구현(DB 선택 + 키 복호화). llm은 DB도 admin도 모른다.
- `UsageSink`(34) — `Complete` 성공 후 토큰 usage 수신. best-effort(실패는 로그만, 추출을 실패시키지 않음).

#### 공급자 매트릭스 (3 어댑터로 5 공급자 커버)

매트릭스는 `backend/internal/llm/providers.go`에 데이터화돼 있다(20에서 표로 정의, 34에서 코드 SSOT로). `factory.New`(env 경로)·`Resolver`(DB 경로)·admin 콘솔의 키 테스트가 공유한다.

| provider | 어댑터 family | 엔드포인트 | 인증 | 구조화 출력 | 기본 모델 | 키 env |
|---|---|---|---|---|---|---|
| `openai` | openai-compat | `api.openai.com/v1/chat/completions` | `Authorization: Bearer` | `response_format: json_schema (strict)`, `max_completion_tokens` | `gpt-5.4-mini` | `OPENAI_API_KEY` |
| `deepseek` | openai-compat | `api.deepseek.com/chat/completions` | Bearer | `response_format: json_object` + 프롬프트 동봉 스키마, `max_tokens` | `deepseek-v4-flash` | `DEEPSEEK_API_KEY` |
| `grok` | openai-compat | `api.x.ai/v1/chat/completions` | Bearer | `response_format: json_schema (strict)`, `max_tokens` | `grok-4.3` | `XAI_API_KEY` |
| `claude` | anthropic | `api.anthropic.com/v1/messages` | `x-api-key` + `anthropic-version: 2023-06-01` | `output_config.format: json_schema` | `claude-opus-4-8` | `ANTHROPIC_API_KEY` |
| `gemini` | gemini | `generativelanguage.googleapis.com/v1beta/models/{m}:generateContent` | `x-goog-api-key` | `responseMimeType: application/json` + `responseJsonSchema` | `gemini-3.5-flash` | `GEMINI_API_KEY` |

- `openai_compat.go`: 제네릭 OpenAI-호환 chat completions(base URL·모델·키·`jsonSchema` 지원여부·`maxTokensField` 파라미터화). `jsonSchema=false`(deepseek)면 `json_object` 모드로 강등하고 스키마를 시스템 프롬프트에 동봉(`"JSON"` 단어 포함이 json_object 요구).
- `anthropic.go`: Messages API. `max_tokens` 필수(없으면 `anthropicDefaultMaxTokens=4096`). content 블록에서 첫 `type=="text"` 추출, 거부 시 빈 텍스트(호출부가 폴백).
- `gemini.go`: generateContent. `systemInstruction`+`contents`, `generationConfig`에 스키마·`maxOutputTokens`. `candidates[0].content.parts` 텍스트 연결.
- 공통 HTTP: `postJSON`(전 어댑터 공유 — marshal·POST·non-2xx를 잘린 body와 함께 에러). 1MiB 응답 상한.

#### 팩토리와 리졸버

- `llm.New(cfg)` (`factory.go`): `LLM_PROVIDER`(기본 `DefaultProvider="openai"`) → 매트릭스 조회 → 키 없으면 어느 env가 필요한지 명시하며 fail-fast, unknown이면 `unknown LLM_PROVIDER` 에러. `LLM_MODEL`이 공급자 기본 모델을 오버라이드. `llmTimeout=120s`. `NewForProvider(provider, model, key, hc)`가 매트릭스 family로 어댑터를 만드는 공유 생성자.
- `llm.NewResolver(src, cfg, sink)` (`resolver.go`, 34): 동적 `Client`. 매 `Complete`마다 `ConfigSource.ActiveLLM`을 `resolverTTL=30s` 캐시로 읽어 활성 선택의 어댑터로 위임한다. DB에 아무것도 없으면(`ok=false`) `llm.New(cfg)`로 만든 env 폴백 클라이언트로 강등(= 20의 env 동작 그대로). source/빌드 실패는 로그 후 폴백, 절대 추출을 실패시키지 않는다. 성공 후 `UsageSink.RecordUsage`로 토큰 계측(best-effort).

### memory 서비스의 동기 프리뷰 경로

`SegmentMemory` 프리뷰(저장 없이 조각 후보 반환)는 같은 추출기를 쓰되, memory가 ai를 import하지 않도록 자기 소유 포트를 둔다.

- `internal/memory`가 소비자 포트 `memory.Extractor.Extract(ctx, body) ([]SegmentInput, error)`를 정의(`backend/internal/memory/memory.go`). nil이면 단일 whole-body neutral 조각으로 강등(ai.NoopExtractor와 같은 동작, ai import 없이).
- 조립 루트의 `segmenterAdapter{inner: ai.Extractor}`(`cmd/api/main.go`)가 ai 추출 결과를 `memory.SegmentInput`(Text·Mood·Intensity·Valence)으로 매핑. memory가 ai를 import하면 import cycle이 되므로 어댑터가 루트에 산다.

### job kind와 워커 (21이 실행 연결)

- `job.go`: `KindExtract Kind = "extract"`(`KindEmbed`·`KindConsolidate`와 나란히). extract job은 `RecordID`로 키잉(조각이 아직 없음); embed job은 `MemoryID`, consolidate job은 `UserID`.
- 워커(`worker.go`)는 20의 "no-op 스텁"을 넘어 **21이 실제 fan-out을 배선했다**: `processOne`이 extract → embed → consolidate 순으로 claim하고, `handleExtract`가 원본 일기 로드 → `Extractor.Extract` → `FanOutFragments`(한 tx: N memories + N embed job + 일내 시냅스)를 실행한다. 멱등(이미 fan-out된 record는 LLM 재호출 없이 no-op). `applyManualHint`는 추출이 단일-중립-조각 폴백 형태일 때만 사용자 힌트를 적용(다조각·정동 결과는 덮어쓰지 않음). 20에서 정의한 도메인·시드·계측을 21의 워커가 그대로 소비한다.

## Flutter 동등성 기준

이 스펙은 순수 BE 추출 계층이라 Flutter가 재현할 클라이언트 UI는 없다. Flutter 팀이 같은 BE를 쓰면 추출 결과는 RPC 경계 뒤에서 그대로 흘러온다(`SegmentMemory` 프리뷰의 `Segments[]`, `RecordMemory` 후 `GetUniverse`로 도착하는 N개 별). Flutter는 다음을 가정해 동작해야 한다.

- 일기 하나가 1~N 조각으로 나뉘고, 조각마다 mood(13종)·intensity·valence가 붙는다.
- 추출기는 admin 콘솔 활성 LLM 선택을 따른다 — 콘솔에 선택이 없으면 키리스 mock(neutral·결정론적 분절)이 동작한다. Flutter 클라이언트는 mock 환경에서도 동일한 흐름이 깨지지 않아야 한다.
- 추출 실패는 별 0개가 아니라 단일 조각 폴백으로 항상 ≥1 별을 보장한다.
- 별 색/UX는 13종 mood로, 별의 무게·내구는 intensity·valence로 결정된다(소비는 21·26).

만약 BE 없이 추출을 직접 포팅한다면 `Extractor` 포트·검증/폴백 헬퍼·`MockExtractor`·`internal/llm` 5공급자 어댑터·공급자 매트릭스를 옮겨야 한다.

## 수용 기준

1. 의미 전환(장소·사람·활동·주제·감정 톤 중 ≥1)이 있는 다장면 일기는 `Segments`를 2개 이상으로 반환하고, 각 조각이 0-based 연속 `Index`·`Text`·검증된 `Mood`(13종)·`Intensity∈[0,1]`·`Valence∈[-1,1]`를 채운다.
2. <60단어이거나 전환이 없는 단일 연속 장면은 정확히 1조각을 반환한다.
3. 조각 수는 항상 `[1,5]`로 클램프되고, 초과분은 마지막 조각에 병합돼 텍스트가 유실되지 않는다.
4. LLM 응답이 JSON 파싱 실패·빈 결과면 에러를 올리지 않고 단일 조각 폴백(전체 원문·mood 힌트 또는 neutral·intensity 힌트 또는 0·valence 0)을 반환한다.
5. 정규화 시 화이트리스트 밖 mood는 `MoodNeutral`로, intensity는 `[0,1]`로, valence는 `[-1,1]`로(NaN→0) 클램프된다. 빈 텍스트 조각은 버려진다.
6. 전송 에러(HTTP/네트워크)는 폴백이 아니라 에러로 올라가 워커가 백오프·재시도한다.
7. mock 추출기는 어떤 API 키도 없이 문단→문장 단위로 결정론적으로 분절하고, 같은 입력에 항상 같은 `Segments`(모두 neutral·valence 0)를 낸다.
8. 조각 fan-out은 재시도/재조회 후에도 같은 memory id 기반 seed 파생 경로를 사용한다.
9. `LLM_PROVIDER`가 openai·gemini·claude·deepseek·grok 중 하나이고 그 키가 있으면 `llm.New`가 해당 어댑터를 만들고, 코드 변경 없이 env만으로 공급자가 교체된다.
10. 선택된 공급자의 키가 비면 `llm.New`는 어느 env가 필요한지 명시하며 fail-fast하고, 알 수 없는 `LLM_PROVIDER`는 `unknown LLM_PROVIDER` 에러를 낸다. `LLM_MODEL`은 공급자 기본 모델을 오버라이드한다.
11. `ai` 패키지는 `llm.Client` 포트만 사용하고 공급자 HTTP 세부를 모르며, `internal/llm`은 `internal/ai`를, `ai`는 `internal/memory`를 import하지 않는다. ai 도메인 타입에 proto/db/json 태그가 없다.
12. 추출기 선택에 env 노브가 없다. `src`가 nil이면 mock, 있으면 admin 콘솔 활성 선택을 따르는 `SwitchingExtractor`(활성→실 LLM, 비활성→mock)가 만들어진다.
13. extract job은 `RecordID`로 키잉되고, 워커가 `KindExtract`를 embed보다 먼저 claim해 조각 fan-out을 실행한다(실제 fan-out·DB 영속·proto 노출은 21).
