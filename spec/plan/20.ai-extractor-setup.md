# 20. AI 추출기 포트 배선 (ai-extractor-setup)

> 죽어 있던 `ai.Extractor` 포트에 실제 LLM 추출기를 배선해 일기를 사건 경계 기준 1~N개 조각으로 분절하고, 조각마다 **감정(13종 enum — 29)·강도(arousal)·정서가(valence ∈ [-1,1])·개체**를 구조화 출력으로 뽑는다. LLM 호출 자체는 신설 **`internal/llm` 공급자 추상화**(단일 `llm.Client` 포트) 뒤에 두어, **openai·gemini·claude·deepseek·grok 5개 공급자를 env만으로 갈아끼운다**(헌법7의 완전한 구현). | 범위: BE | 선행: 05, 29 | 차단: 21

## 목적 (Why)
05는 `Extractor` 포트와 빈 `Extraction`·`NoopExtractor`만 두고 실제 추출은 v1로 미뤘다. 하지만 사용자 의도의 핵심("하루의 일기에는 여러 감정이 섞여 있다 → AI가 여러 조각으로 나눠 여러 별이 된다")을 구현하려면 **먼저 추출기를 살려야** 한다. 이 스펙은 AI 토대 작업이다: `ai.Extractor`를 실제 LLM 어댑터(구조화 출력)로 채우고, 일기를 **사건 경계(event boundary)** 기준 1~N개 조각으로 분절하며, 각 조각의 감정·강도·개체를 뽑는다. **별 fan-out(1 일기 → N 별)은 하지 않는다 — 그건 21의 일이다.** 20은 포트·어댑터·팩토리·job kind·단위 테스트까지만이며, 21이 워커에서 이 추출기를 호출해 별을 분기한다.

## 범위
- **포함:**
  - `internal/ai`: 빈 `Extraction`을 채운다(`Extraction{ Segments []Segment }`, `Segment{ Index, Text, Mood, Intensity, Valence, Entities }`). 순수 도메인 — proto/db/json 태그 금지(헌법5).
  - **`internal/llm` 공급자 추상화 패키지(신설)**: 단일 `llm.Client` 포트(`Complete(ctx, llm.Request) (llm.Response, error)`, JSON schema 구조화 출력 지원) + 공급자 어댑터 — `openai`/`deepseek`/`grok`는 OpenAI-호환 chat completions 제네릭 어댑터 1개(base URL·모델·키 파라미터화), `claude`는 Anthropic Messages API 어댑터, `gemini`는 generateContent 어댑터. 도메인(ai 패키지)은 `llm.Client`만 보고 공급자 HTTP를 모른다.
  - `ai.Extractor` 뒤 **실제 추출기**(`LLMExtractor` — 프롬프트·JSON schema·검증·폴백·캐시·계측을 소유, `llm.Client`로 호출) + **`MockExtractor`**(키 불필요·결정론적: 빈 줄/문장 단위 분절, NEUTRAL 감정·`Valence=0` — 기본값).
  - `NewExtractor` 팩토리(`NewEmbedder` 미러링: `AI_EXTRACTOR=mock|llm`) + `llm.New` 팩토리(`LLM_PROVIDER=openai|gemini|claude|deepseek|grok`, `LLM_MODEL` 모델 오버라이드, 공급자별 API 키 env).
  - LLM 프롬프트 = **사건 분절 루브릭**: "장소·사람·활동/목표·주제·감정 톤 중 하나라도 전환되면 새 조각을 시작하라; 그런 전환이 ≥1개일 때만 분리, 아니면 1조각." + **조각마다 감정(13종 — 29)·강도(arousal 0~1)·정서가(valence -1~1)** 부여. 사분면-우선 선택·tie-break·한국어 정동 레시피는 **29(emotion-taxonomy)의 추출 가이드라인이 단일 출처**(아래 프롬프트).
  - 견고성(concept §4.6 "LLM 출력은 깨질 수 있다"): 구조화 출력 강제·스키마 검증·범위 클램프·파싱 실패 시 **단일 조각 폴백**.
  - `job.Kind`에 `KindExtract` 추가. 워커는 21을 대비한 **no-op-safe `KindExtract` 분기 스텁**만 추가(실제 fan-out은 21).
  - config(`AI_EXTRACTOR`·`LLM_PROVIDER`·`LLM_MODEL`·공급자 키 5종)·`.env.example`.
  - `internal/ai`·`internal/llm` 단위 테스트(Mock 결정성·다장면 ≥2 분절·폴백·클램프·공급자 선택/키 fail-fast).
- **비목표(Out of scope):**
  - **별 fan-out**(1 일기 → N memories, `InsertMemory`, fragment 당 embed job) → **21(memory-fragmentation)**. 20은 `RecordMemory`가 `extract` job을 적재하지 **않는다**(그건 21).
  - **DB 마이그레이션**(`memories`에 `mood`/`intensity`/`fragment_index`/`fragment_text` 추가) → **21**.
  - **proto 계약 변경**(`RecordMemoryResponse.memory_ids`, `Star.valence` 등) → **21**.
  - **valence 영속·물리 소비** → 20은 valence를 **생산(추출·검증)만** 한다. `memories.valence` 컬럼·`Star.valence` proto·FE `Memory.valence` 매핑은 **21**(memories 마이그레이션에 함께 실음), valence가 **감쇠저항·연결강도(λ_eff)** 를 구동하는 물리는 **26**(현 `R_emo`=intensity에 valence 항을 더함 — "물리까지 v1" 결정). 20은 `ai.Segment.Valence` 도메인 필드까지만.
  - **FE·랜딩 카드 분절 시각화 로직 자체** → **21**. (단, 본 스펙은 헌법 외 규약상 랜딩 카피 점검 + 체험 토대 표기 task를 둔다 — 아래 작업 참조.)
  - 흥분성·재공고화·관련성 감쇠 등 다른 이론 → **22~27**.

## 설계 요점

**감정 모델 = 13종 mood(색·UX, 4사분면 — 29) + intensity(arousal, 기억 무게) + valence(부호 있는 내구·연결 축).** mood 분류는 **29(emotion-taxonomy)가 단일 출처**다: 7종에서 13종(Russell 원형모형 4사분면×3+중립)으로 확장해 비어 있던 저각성-부정 사분면(힘듦·공허함)을 채웠다. 범주는 여전히 색/UX 레이어이고(Lindquist 2012), closed-enum+`strict:true`로 받되 13에서의 신뢰성은 **사분면-우선 추출 가이드(29)** 가 지킨다(평면 13지선다가 아니라 사분면→사분면 내 선택). 기억의 "무게"를 실제로 정하는 건 **각성(arousal)** 이고(편도체 BLA→해마 공고화, McGaugh 2004), 이는 이미 있는 `Intensity[0,1]`다 — 새 필드를 추가하지 않고 **의미만 arousal/salience로 재정의**한다. 진짜 빠진 축은 분리 가능한 둘째 경로인 **정서가(valence)** 다(부정 기억의 내구·디테일, Kensinger & Corkin 2004). 그래서 `Segment`에 `Valence float64 [-1,1]` 하나만 가산한다. valence는 **LLM이 텍스트에서 가장 신뢰성 높게 뽑는 정동 차원**(GPT-4 valence r≈0.79 vs arousal r≈0.55)이라, LLM에 별도 arousal 점수를 요구하지 않고(최약 차원) **valence만 연속 출력**으로 받는다. 고통/혐오 같은 강한 부정 경험은 별도 범주 없이 `sad/fear/anger + 강한 음 valence + 높은 intensity`로 표현된다(프롬프트 레시피).

**`Extraction`·`Segment`(순수 도메인 — proto/db/json 태그 금지, 헌법5).** `ai` 패키지는 `memory.Mood`를 import하지 않는다(결합 회피 — 추출기가 별 도메인에 의존하면 모바일/타 어댑터 재사용이 깨진다). 대신 `ai`가 자기 소유의 작은 `Mood` 문자열 enum을 가지며 값은 `memory.Mood`와 1:1 매핑되도록 정렬한다(매핑은 호출부/21이 수행). `Intensity`는 `[0,1]`(= arousal/salience), `Valence`는 `[-1,1]`(0 = 중립/미상).
```go
// internal/ai/embedder.go — Extraction 채우기(05의 빈 struct 교체)
type Mood string // ai 소유 enum, memory.Mood(29의 13종)와 값 정렬: "", joy, calm, sad, anger, fear, love, neutral, excitement, gratitude, relief, stress, tired, emptiness
const (
    MoodUnspecified Mood = ""
    MoodNeutral     Mood = "neutral"
    // joy/calm/sad/anger/fear/love/excitement/gratitude/relief/stress/tired/emptiness …(memory.Mood 13값과 동일 문자열 — 29)
)
type Entities struct{ People, Places, Topics []string }
type Segment struct {
    Index     int      // 0-based, 조각 순서(21이 fragment_index로 영속)
    Text      string   // 조각 요약/본문(임베딩 대상 — 원본 record 수정이 아님, 헌법1)
    Mood      Mood     // 조각 감정(검증·클램프된 13종 enum 값 — 29) — 색/UX 레이어
    Intensity float64  // 0..1, = arousal/salience(기억 무게 — 공고화 이득; 21+가 밝기·연결·감쇠에 배선)
    Valence   float64  // -1..1, 부호 있는 정서가(내구·연결 축; 0=중립). 21이 영속, 26이 λ_eff에 소비
    Entities  Entities // people/places/topics
}
type Extraction struct{ Segments []Segment } // 항상 ≥1개(폴백 보장)
```

**`internal/llm` — 공급자 추상화 패키지(신설, 헌법7).** 도메인 로직이 "어떤 LLM"인지 모르고 `llm.Client` 포트로만 요청하게 한다. 포트는 unary 텍스트 완성 + 선택적 JSON schema 구조화 출력:
```go
// internal/llm/llm.go — 순수 포트(공급자/전송 비의존)
type Client interface {
    Complete(ctx context.Context, req Request) (Response, error)
    Model() string // 계측·로그용
}
type Schema struct{ Name string; Raw json.RawMessage } // JSON Schema 문서
type Request struct {
    System    string
    User      string
    Schema    *Schema // nil이면 자유 텍스트
    MaxTokens int
}
type Response struct{ Text string } // Schema 설정 시 JSON 텍스트
```
- **어댑터 3개가 5개 공급자를 커버한다**: `openai_compat.go`(제네릭 OpenAI-호환 chat completions — base URL·모델·키·json_schema 지원여부 파라미터화)가 `openai`/`deepseek`/`grok`를, `anthropic.go`(Messages API, `output_config.format` json_schema)가 `claude`를, `gemini.go`(generateContent, `generationConfig.responseJsonSchema`)가 `gemini`를 담당.
- **공급자 매트릭스(기본 모델은 `LLM_MODEL`로 오버라이드):**

| provider | 엔드포인트 | 인증 | 구조화 출력 | 기본 모델(2026-06 공식 문서 검증) | 키 env |
|---|---|---|---|---|---|
| `openai` | `api.openai.com/v1/chat/completions` | `Authorization: Bearer` | `response_format: json_schema (strict)` | `gpt-5.4-mini` | `OPENAI_API_KEY` |
| `deepseek` | `api.deepseek.com/chat/completions` | Bearer | `response_format: json_object` + 프롬프트 내 스키마(json_schema 미지원) | `deepseek-v4-flash` (`deepseek-chat`은 2026-07 퇴역) | `DEEPSEEK_API_KEY` |
| `grok` | `api.x.ai/v1/chat/completions` | Bearer | `response_format: json_schema (strict)` | `grok-4.3` (`grok-4-1-fast-*` 2026-05 퇴역) | `XAI_API_KEY` |
| `claude` | `api.anthropic.com/v1/messages` | `x-api-key` + `anthropic-version: 2023-06-01` | `output_config.format: json_schema` | `claude-opus-4-8` | `ANTHROPIC_API_KEY` |
| `gemini` | `generativelanguage.googleapis.com/v1beta/models/{m}:generateContent` | `x-goog-api-key` | `responseMimeType: application/json` + `responseJsonSchema` | `gemini-3.5-flash` | `GEMINI_API_KEY` |

- **휴대 가능한 canonical 스키마**: `minItems`/`maxItems` 같은 배열 제약은 공급자별 strict 모드에서 지원이 갈리므로 **스키마에 넣지 않는다** — 조각 수 `[1,5]`·하드캡 8은 항상 코드 클램프(아래 견고성)가 보장한다. 모든 object에 `additionalProperties:false` + 전 필드 `required`(OpenAI strict·Anthropic 공통 요구).
- 공급자가 json_schema를 미지원하면(deepseek) 어댑터가 **스키마를 프롬프트에 동봉 + json mode**로 강등한다 — 어차피 응답은 ai 쪽 검증·폴백을 거치므로 안전(concept §4.6).

**실제 추출기 `LLMExtractor`(`ai.Extractor` 뒤, `llm.Client` 소비).** 프롬프트(분절 루브릭)·canonical JSON schema·응답 파싱·검증·폴백·**텍스트 해시 캐시**·토큰 상한(`maxInputRunes` 미러)·`ExtractMetrics`(LLM 계측은 임베딩과 **분리** — §4.7 비용 관리)를 전부 ai 패키지가 소유한다 — 공급자를 바꿔도 추출 품질 로직은 그대로다. 전송 에러(HTTP/네트워크)는 에러로 올려 워커 백오프·재시도에 태우고, **내용 깨짐(파싱·스키마 위반·빈 결과)만 단일 조각 폴백**한다. 프롬프트 = 사건 분절 루브릭:
```
시스템: 너는 일기를 '사건 경계'로 나눈다. 장소·사람·활동(목표)·주제·감정 톤 중
하나라도 바뀌면 새 조각을 시작한다. 그런 전환이 하나도 없으면 1조각으로 둔다.
각 조각에 감정(13종 — 아래) 하나와 강도(intensity 0~1, 감정의 '세기'=각성),
정서가(valence -1~1, 음=부정·0=중립·양=긍정), 인물/장소/주제를 붙인다.
- 감정 13종 = joy|excitement|love|calm|gratitude|relief|anger|fear|stress|sad|tired|emptiness|neutral.
  선택은 **사분면 먼저(HAP 고각성긍정 / LAP 저각성긍정 / HAN 고각성부정 / LAN 저각성부정)
  → 사분면 내 선택** 순서로 한다(29의 추출 가이드라인).
- 사분면 내·간 tie-break와 한국어 정동 레시피(한→sad, 답답/화병→stress 또는 anger,
  지침/소진→tired, 권태→tired/neutral, 공허→emptiness, 혐오→anger, 죄책감→sad/fear,
  뿌듯→joy/excitement)는 **29(추출 가이드라인) 단일 출처**를 그대로 옮긴다.
- intensity는 감정의 세기(차분=낮음, 격렬=높음). valence와 별개다(고강도 슬픔: intensity 높고 valence 음).
- 대략 80~120단어당 1조각, 단 의미 전환이 우선. JSON schema만 출력.
(few-shot 1~2개를 confusable 쌍(calm/tired, neutral/tired, anger/stress, sad/emptiness, joy/excitement)에
대해 한국어로 동봉 — 29의 가이드라인 기준.)
```

강제할 canonical JSON schema(요지 — 전 공급자 공용, 배열 제약 없음):
```json
{ "type": "object", "required": ["segments"], "additionalProperties": false,
  "properties": { "segments": { "type": "array",
    "items": { "type": "object",
      "required": ["index","text","mood","intensity","valence","entities"],
      "additionalProperties": false,
      "properties": {
        "index":     { "type": "integer" },
        "text":      { "type": "string" },
        "mood":      { "type": "string", "enum": ["joy","excitement","love","calm","gratitude","relief","anger","fear","stress","sad","tired","emptiness","neutral"] },
        "intensity": { "type": "number" },
        "valence":   { "type": "number" },
        "entities":  { "type": "object", "additionalProperties": false,
          "required": ["people","places","topics"],
          "properties": { "people": {"type":"array","items":{"type":"string"}},
                          "places": {"type":"array","items":{"type":"string"}},
                          "topics": {"type":"array","items":{"type":"string"}} } } } } } } }
```
수치 범위(`minimum`/`maximum`)·배열 제약(`minItems`/`maxItems`)은 공급자 strict 모드에서 지원이 갈려 **스키마에 두지 않는다** — 프롬프트가 1차로 지시하고, 응답 후 코드에서 `[1,5]` 클램프·하드캡 8·범위 클램프를 항상 재적용한다(LLM이 schema를 어겨도 안전하도록 — concept §4.6).

**`MockExtractor`(키 불필요·결정론적 — 기본).** API 키 없이 동작: 빈 줄(문단) → 없으면 문장(`. ! ?` + 개행) 단위로 분절, 각 조각 `Mood=MoodNeutral`·`Intensity=0`·`Valence=0`·빈 `Entities`. 같은 입력은 항상 같은 분절(테스트·키리스 E2E 보장, mock.go의 결정성 철학과 동일). 단어 수 규칙(아래 params)을 적용해 `[1,5]` 클램프·하드캡 8.

**견고성·검증·폴백(concept §4.6).** LLM 응답은 (a) JSON 파싱 → (b) 스키마 검증(필수 필드·타입) → (c) 정규화(`Mood`를 13종(29) enum 화이트리스트로 매핑, 모르는 값은 `MoodNeutral`; `Intensity`를 `[0,1]` 클램프; **`Valence`를 `[-1,1]` 클램프(NaN→0)**; `Text` trim·공백 정리; `Index` 0-based 재부여) 순으로 거른다. **어느 단계든 실패하면 단일 조각 폴백** = `Segment{Index:0, Text: 전체 원문, Mood: 사용자 힌트 또는 MoodNeutral, Intensity: 힌트 또는 0, Valence: 0}`. 폴백은 에러가 아니라 정상 경로다(별이 0개가 되지 않도록 `Extraction.Segments`는 항상 ≥1).

**분절 파라미터(여기서 정의 — 21이 소비).**
- 대략 1조각 / 80~120단어, 결과 조각 수 `[1,5]` 클램프, 하드캡 8.
- <60단어 또는 단일 연속 장면(전환 0) → 1조각.
- 조각 시드(21이 별 form/positions에 사용): `seed = stableHash(diary_id ":" fragment_index ":" sha1(normalized_text))`. `normalized_text` = trim + 연속 공백 1칸 + 소문자. 결정론적·재현 가능(이 스펙에서 정의, 21이 호출).

**job.Kind + 워커 스텁.** `job.go`에 `KindExtract Kind = "extract"` 추가(`KindEmbed`와 나란히). `worker.go`의 `processOne`은 현재 `w.jobs.Claim(ctx, KindEmbed)`로 임베딩 job만 claim한다 — 21이 fan-out을 붙일 자리로 **no-op-safe `KindExtract` 분기 스텁**을 둔다. 20에서는 워커가 `KindExtract`를 적극적으로 폴링하지 않거나(임베딩 루프 불변 유지), 폴링하더라도 추출기 호출 후 별 생성 없이 곧장 `Complete`하는 안전한 빈 분기여야 한다. **20에서 `RecordMemory`는 `extract` job을 적재하지 않는다**(04의 적재 경로는 그대로 `embed`만, 21이 `extract` 적재를 연결) — 따라서 스텁이 실제로 도는 일은 21 전까지 없다.
```go
// worker.go — 21이 채울 자리(20은 빈 스텁만)
// case KindExtract:
//   ext, err := w.extractor.Extract(ctx, m.Body)  // 21: Worker에 ai.Extractor 주입
//   // 21: ext.Segments → InsertMemory fan-out + 조각별 embed job 적재
//   return w.jobs.Complete(ctx, j.ID)              // 20: 별 생성 없이 안전 완료
```
> ⚠️ 20에서 `Worker` 구조체에 `ai.Extractor`를 **주입하지 않아도 된다**(스텁은 미실행). 주입을 미리 해 두면 빌드 시 미사용 필드 경고가 날 수 있으므로, 주입·`extract` 적재·실제 fan-out은 모두 21에서 한 번에 배선한다. 20의 워커 변경은 `KindExtract` 상수 인지 + 안전 분기 주석/빈 case에 한정한다.

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 추출기가 의미 전환(장소·사람·활동·주제·감정 톤 중 ≥1)이 있는 다장면 일기를 받으면 THE SYSTEM SHALL `Segments`를 **2개 이상** 반환하고 각 조각에 `Index`(0-based 연속)·`Text`·검증된 `Mood`(13종 — 29)·`Intensity∈[0,1]`·`Valence∈[-1,1]`를 채운다. _(헌법7)_
1.2 WHEN 추출기가 <60단어이거나 전환이 전혀 없는 단일 연속 장면을 받으면 THE SYSTEM SHALL `Segments`를 **정확히 1개** 반환한다. _(분절 루브릭)_
1.3 WHEN 추출 결과 조각 수가 규칙상 한계를 넘으면 THE SYSTEM SHALL 조각 수를 `[1,5]`로 클램프하고 하드캡 8을 절대 넘기지 않는다. _(분절 params)_
1.4 IF LLM 응답이 JSON 파싱 실패·스키마 위반·빈 결과 중 하나이면 THEN THE SYSTEM SHALL 에러를 올리지 않고 **단일 조각 폴백**(`Text`=전체 원문, `Mood`=사용자 힌트 또는 `MoodNeutral`, `Intensity`=힌트 또는 0, `Valence`=0)을 반환한다. _(concept §4.6, 헌법7)_
1.5 WHEN 각 조각의 `Mood`를 정규화할 때 THE SYSTEM SHALL 화이트리스트(29의 13종: `""`,joy,calm,sad,anger,fear,love,neutral,excitement,gratitude,relief,stress,tired,emptiness) 밖 값은 `MoodNeutral`로, `Intensity`는 `[0,1]`로, `Valence`는 `[-1,1]`로(NaN→0) 클램프한다. _(견고성)_
1.6 WHEN `AI_EXTRACTOR=mock`이면 THE SYSTEM SHALL 어떤 API 키도 없이 결정론적으로 분절하고, 동일 입력에 항상 동일한 `Segments`를 반환한다. _(헌법7, 키리스 E2E)_
1.7 WHEN `AI_EXTRACTOR=llm`인데 선택된 `LLM_PROVIDER`의 API 키가 비어 있으면 THE SYSTEM SHALL `NewExtractor`(→`llm.New`)에서 어느 키가 필요한지 명시하는 에러로 실패한다(fail-fast). _(설정 선택)_
1.8 WHEN `AI_EXTRACTOR` 또는 `LLM_PROVIDER`가 알 수 없는 값이면 THE SYSTEM SHALL 팩토리에서 `unknown …` 에러를 반환한다. _(NewEmbedder 미러)_
1.12 WHEN `LLM_PROVIDER`가 `openai|gemini|claude|deepseek|grok` 중 하나이고 해당 키가 있으면 THE SYSTEM SHALL 그 공급자 어댑터로 동일한 추출 동작(분절·검증·폴백)을 수행한다 — 코드 변경 없이 env만으로 공급자가 교체된다. _(헌법7)_
1.13 WHEN `ai` 패키지가 LLM을 호출하면 THE SYSTEM SHALL `llm.Client` 포트만 사용하고 공급자 HTTP 세부(엔드포인트·인증·요청 모양)를 모르며, `llm` 패키지는 `internal/ai`를 import하지 않는다. _(레이어 격리)_
1.9 WHEN `ai` 패키지가 빌드되면 THE SYSTEM SHALL proto/db/json 태그 없는 순수 도메인 타입(`Extraction`/`Segment`/`Entities`)만 노출하고 `internal/memory`를 import하지 않는다. _(헌법5·7)_
1.10 WHEN 조각 시드를 계산하면 THE SYSTEM SHALL `stableHash(diary_id:fragment_index:sha1(normalized_text))`로 결정론적·재현 가능한 값을 반환한다(같은 입력→같은 시드). _(21 소비)_
1.11 WHEN `KindExtract`가 정의되면 THE SYSTEM SHALL 워커에 no-op-safe 분기 스텁만 존재하고 20에서는 `extract` job이 적재되지 않아 실제 fan-out이 일어나지 않는다(21이 연결). _(범위 경계)_

## 과학적 근거
사건 경계 분절은 인지심리·신경과학의 **Event Segmentation Theory**에 근거한다: 사람은 연속 경험을 장소·인물·목표·인과 등이 바뀌는 "사건 경계"에서 자동으로 분절해 부호화하며, 이 경계가 기억의 단위가 된다. 하루의 일기를 단일 덩어리가 아니라 여러 사건 조각으로 나누는 것은 이 이론의 직접 구현이다. 한 사건이 별개로 부호화되는 것(분리 부호화)은 해마의 **패턴 분리**와도 일치한다. 또한 하나의 시간 단위 안에 **여러(섞인) 감정**이 공존한다는 점(감정 입자도/혼합 감정)이 조각별 감정·강도 부여의 근거다. 단, 혼합 감정은 **조각 내부 블렌딩이 아니라 조각 간(여러 별)** 으로 표현한다 — 사건 분절이 이미 "섞인 하루 = 여러 색 별"을 구현하고, 일상 혼합 감정의 지배적 원천도 사건 간이며(Schneider & Stone 2015), "동아시아권이 혼합 감정을 더 느낀다"는 주장은 순간 수준에선 약해(일본 한정·부분 비재현) 조각 단위 단일 라벨을 정당화한다.

**감정 모델의 삼분(三分) 근거 — mood / intensity(arousal) / valence.** 이산 감정 범주는 뇌에 깔끔히 국재화되지 않으므로(Lindquist 2012), mood(29의 13종, 4사분면)는 **색/UX 레이어**로 두고 기억의 무게는 차원 변수가 진다. (1) **각성(arousal)** = 편도체 기저외측핵(BLA)을 거쳐 해마 공고화를 강화하는 핵심 변수(McGaugh 2004; Cahill & McGaugh 1998)로, valence와 독립적으로 기억을 높인다(Hamann 1999) — 이것이 이미 있는 `Intensity[0,1]`다(별 필드 추가 없이 의미를 arousal로 재정의). 단 각성은 *균일 배수*가 아니라 *우선순위 가중*이다(Mather & Sutherland 2011) — 21+가 연결·감쇠에 배선할 때 이미 현저한 별 사이의 경쟁을 편향시키는 식으로 적용한다. (2) **정서가(valence)** = 분리 가능한 둘째 경로로 부정 기억의 내구·디테일을 좌우한다(Kensinger & Corkin 2004; Hurlemann 2006). 부호 있는 `Valence[-1,1]`로 가산하며, 26이 λ_eff(감쇠 저항)에 음 valence 항으로 소비한다("물리까지 v1"). (3) **LLM 추출 신뢰도**: 세분화 분류는 붕괴하지만(GoEmotions 28라벨 zero-shot 저조) coarse closed-enum은 견고하므로 13종(29) enum + `strict:true` + 사분면-우선 가이드(평면 13지선다를 사분면→내부로 분해해 신뢰성 유지)를 쓰고, 연속 출력은 LLM의 최강 정동 차원인 valence만 받는다(arousal 자유 점수는 최약이라 요구하지 않음).
