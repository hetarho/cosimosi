# 17. error-handling — FE 바운더리·폴백 + BE 입력 검증·요청 한도·패닉 복구

> 베타 사용자에게 흰 화면을 보여주지 않고, 서버를 빈 일기·거대 페이로드·패닉으로부터 지킨다. | 범위: FS (베타 하드닝) | 선행: 16 | 차단: 18

## 목적 (Why)

2026-06 아키텍처 검토에서 확인된 사실들:

**FE — 에러 상태 설계가 사실상 없다.**
- `Sentry.init`은 DSN만 넘기고(`main.tsx`) **integration 0개**, 그리고 **어떤 것도 에러 바운더리로 감싸지 않는다**. R3F 트리 안의 throw 한 번(이상한 WebGL2 드라이버의 TSL 컴파일 실패, NaN 지오메트리, 컨트롤러의 null)에 전체 앱이 **흰 화면**이 되고, 캡처도 복구 UI도 없다. `@sentry/react`는 이미 의존성에 있다.
- `createRenderer`(`shared/lib/r3f/renderer.ts`)는 WebGPU와 WebGL2 폴백이 **둘 다** 실패하면 reject하는데, 이를 받는 UI가 없다 — 구형 GPU/잠긴 브라우저 사용자에게 `/universe`는 복구 불가능한 빈 화면이다.
- 라우터에 `defaultErrorComponent`/`defaultNotFoundComponent`가 없다(`app/router.tsx`) — 잘못된 URL·라우트 로드 실패에 설계된 화면이 없다.

**BE — 쓰기 경로가 입력을 그대로 믿는다.**
- `RecordMemory` 핸들러는 `entry_date`만 검증하고 `Body`/`Intensity`를 그대로 통과시킨다(`handler.go` 40–53행). **빈 일기도** record+memory+임베딩 잡(유료 OpenAI 호출)을 만들고, records는 append-only(헌법 §1)라 **정리 경로가 없다**. `intensity`는 0~1 클램프 없이 저장돼 별 크기 셰이더(§6 `f(intensity)`) 계약을 깬다.
- `rpcserver.New`는 `ReadHeaderTimeout`만 설정 — `WriteTimeout`/`IdleTimeout`/요청 body 상한이 없다. 1GB Lightsail에서 크고 느린 POST는 값싼 DoS이자 임베딩 비용 증폭 벡터다.
- 패닉 복구: Sentry DSN이 있을 때만 `sentryhttp`가 핸들러 패닉을 잡는다. **워커 고루틴에는 recover가 없어** `handle()` 패닉이 단일 바이너리(API+워커) 전체를 죽인다.

## 범위

**포함**
- FE: 전역 + 우주 캔버스 전용 에러 바운더리(폴백 UI), 렌더러 init 실패 안내, 라우터 error/notFound 컴포넌트

- FE 폼: 서버 검증 에러를 사용자 메시지로 표면화(빈 일기 등)

**비목표 (Out of scope)**
- Sentry integration 강화(browserTracing·replay·release 태깅)·지표 수집 → **18**
- 레이트리밋·비용 상한(별도 후속 — 18 이후 백로그)
- 데이터 로딩 에러 UI(GetUniverse/ListDormant) → **16에서 완료** (이 스펙은 *렌더/라우트/입력* 에러)

## 설계 요점

**FE — 바운더리 2겹 + 라우터 폴백:**
```
main.tsx / App.tsx
  Sentry.ErrorBoundary(fallback=전역 에러 화면: "문제가 생겼어요" + 새로고침 버튼)
    └─ RouterProvider
         ├─ defaultErrorComponent / defaultNotFoundComponent (router.tsx)
         └─ /universe
              └─ ErrorBoundary(fallback="우주를 그릴 수 없어요" 카드)  ← UniverseCanvas 전용
                   └─ <UniverseCanvas/>
```
- `Sentry.ErrorBoundary`는 init이 안 됐어도(로컬 DSN 없음) 폴백 렌더는 정상 동작 — 캡처만 no-op. DSN 유무로 분기하지 않는다.
- 캔버스 전용 바운더리의 폴백에는 **원인 구분 메시지**: 렌더러 init 실패(WebGPU·WebGL2 모두 불가 — "이 브라우저/기기에서는 우주를 그릴 수 없어요")와 일반 렌더 크래시("우주를 불러오지 못했어요 — 다시 시도"). `createRenderer`가 실패 시 식별 가능한 에러(예: `RendererUnavailableError`)를 던지게 한다.
- 우주가 죽어도 **HUD(일기 작성 폼)는 살린다** — 바운더리를 Canvas에만 감싼다. 일기를 쓰는 코어 가치는 렌더러 없이도 동작해야 한다.

**BE — 검증은 service, 매핑은 handler, 한도는 rpcserver:**
- `memory.Service.RecordMemory`에 정책 추가(전송 무관·테스트 가능): ① `strings.TrimSpace(body) == ""` → `ErrEmptyBody` ② `intensity < 0 || > 1` → `ErrIntensityRange` ③ `utf8.RuneCountInString(body) > maxBodyRunes`(임베더 입력 캡 `openai.go maxInputRunes`와 한 상수로 정렬) → `ErrBodyTooLong`. sentinel 에러로 선언.
- `handler.go`: sentinel → `connect.CodeInvalidArgument` 매핑(현재 blanket `CodeInternal`에서 분리).
- `rpcserver.New`: 핸들러에 `connect.WithReadMaxBytes`(256KB — unary 페이로드 최대치 대비 넉넉, keepalive 64KB 캡과도 정합), `http.Server`에 `WriteTimeout`/`IdleTimeout` 추가.
- recover: ① Connect recover 인터셉터 — 패닉을 slog로 기록하고 `CodeInternal` 반환(Sentry 유무 무관) ② `worker.go`의 잡 처리 본문을 recover로 감싸 패닉을 **백오프 실패 처리**(`failWithBackoff`)로 전환 — 프로세스 생존.

**FE 폼 연동:** `MemoryForm`/`use-record-memory`가 `InvalidArgument`를 받으면 낙관적 별을 롤백(기존 경로)하고 사용자 메시지("일기 내용을 입력해 주세요" 등)를 표시.

## 수용 기준 (EARS/GWT)

- 2.1 WHEN R3F 트리 내부에서 에러가 throw되면 THE SYSTEM SHALL 흰 화면 대신 폴백 UI를 표시하고(HUD·작성 폼은 유지), DSN이 있으면 Sentry에 캡처한다.
- 2.2 WHEN WebGPU·WebGL2가 모두 불가한 환경이면 THE SYSTEM SHALL "이 브라우저/기기에서는 우주를 그릴 수 없다"는 안내를 표시한다(빈 캔버스 금지).
- 2.3 WHEN 알 수 없는 경로로 접근하면 THE SYSTEM SHALL notFound 화면을 렌더한다.
- 2.4 WHEN 빈(공백뿐인) body로 RecordMemory를 호출하면 THE SYSTEM SHALL `InvalidArgument`를 반환하고 record/memory/job을 **생성하지 않는다**.
- 2.5 WHEN intensity가 0~1 밖이거나 body가 최대 길이를 넘으면 THE SYSTEM SHALL `InvalidArgument`를 반환한다.
- 2.6 WHEN 요청 body가 256KB를 넘으면 THE SYSTEM SHALL 요청을 거부한다(연결·메모리 점유 없이).
- 2.7 WHEN 핸들러 또는 워커 잡 처리 중 패닉이 나면 THE SYSTEM SHALL 프로세스를 죽이지 않고 — 핸들러는 `CodeInternal` 응답 + slog 기록, 워커는 해당 잡을 백오프 실패 처리한다.
- 2.8 WHEN 서버가 `InvalidArgument`를 반환하면 THE FE SHALL 낙관적 별을 롤백하고 사용자 메시지를 표시한다.
