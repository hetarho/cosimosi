# 51. palette customization

> Palette customization — a named **palette registry** (≥2 palettes), a **stored per-user palette preference** (backend)
> and the **app-init wiring** that applies it through plan 17's single swap seam, plus **axis-consistency guardrails**
> ([P3]) and a structurally-enforced **meaning-layer immutability** boundary ([I11][P2]). The change UI is plan 52.
> Scope: FS | Status: built ([job 62](../jobs/archive/62.palette-customization.md))
>
> Layer U/P. Home: a **backend preference** (a small supporting slice — `internal/account` or `platform` — with a
> get/set RPC) + a **FE apply** (`packages/emotion` registry + an app-layer init that calls `setMoodPalette`). It fills
> the "no user preference yet" seam plan 17 left open ([M6]); it leaves the _change UI_ (settings) to
> [52.settings-page](52.settings-page.md), and rendering (star/nebula) stays palette-agnostic exactly as plan 17 built it.

## Purpose

Color perception is personal and cultural — the mapping that reads as "warm joy" to one person reads differently to
another ([P1]). cosimosi's honest promise is "the universe's color is the emotion you re-read" ([M5]); to keep that
promise _for every user_ the product must let a user choose the `mood → color` palette that reflects **their** emotional
color sense, without ever letting that choice corrupt the meaning underneath it ([I11][P2]).

Plan 17 already built the entire swap mechanism: `moodColor(mood)` is the single entry point over a runtime-swappable
`activePalette`, with `defaultMoodPalette` + `setMoodPalette()`/`resolvePalette()`/`resetMoodPalette()` in
`packages/emotion/src/palette.ts`. What it deliberately did **not** build is _user preference_: it ships exactly one
default palette and no way to store or choose another (plan 17 §"single entry-point, swappable"). This unit closes that
gap. It adds (1) a **registry of ≥2 named palettes** the user can pick from, (2) a **stored per-user palette preference**
on the backend so the choice survives across sessions and devices, (3) the **app-init wiring** that reads the stored
preference and calls `setMoodPalette(chosen)` before the first render, and (4) **axis-consistency guardrails** ([P3])
that warn when a palette breaks the warm=positive / cool=negative reading. It is headless of the settings screen: it
builds the preference + the apply, and [52.settings-page](52.settings-page.md) hosts the picker.

## Scope / Non-goals

**In scope**

- **Palette registry** — a named set of **≥2** `MoodPalette` values in `packages/emotion` (the existing
  `defaultMoodPalette` plus at least one alternative), exposed by a stable **palette id** (a short string key) so the
  backend can store an id and the FE can resolve id → palette. The palette _tables_ (the 13 `mood → color` entries per
  palette) are **code/content**, not `values.yaml` ([M6], plan 14 exclusion rule). The registry is the id ↔ palette
  authority: `paletteById(id)`, `listPalettes()`, and a `DEFAULT_PALETTE_ID`.
- **Stored per-user palette preference (backend)** — a small supporting slice that persists a single scalar per user:
  the chosen **palette id**. `GetPalettePreference` / `SetPalettePreference` on a thin RPC; the value is a per-user
  string with the default applied when unset. **No** mood-color table is ever sent from the server — only the id ([M6]).
- **App-init apply (FE)** — an app-layer (composition-root) step that, on load, reads the stored preference, resolves it
  through the registry, and calls `setMoodPalette(chosen)` **once, before the first render**, so the whole universe is
  colored by the user's palette through plan 17's single seam. Unset / unknown id → `defaultMoodPalette` (fail-safe).
- **Set-and-apply path** — setting the preference persists the id (backend) **and** re-applies via `setMoodPalette` so
  the running universe re-colors immediately through the one seam, with no rendering-code change (the [52] picker calls
  this path).
- **Axis-consistency guardrails ([P3])** — a pure validator over a candidate palette that checks warm=positive /
  cool=negative axis consistency (positive-valence moods trend warm, negative-valence moods trend cool) and returns a
  **warning**, not a hard block. It gates _custom_ palettes only; the shipped registry palettes are authored to pass.
- **Meaning-layer immutability boundary ([I11][P2])** — the preference stores **only** a palette id; there is
  structurally no field by which it can override a memory's stored `mood`/`valence`/`arousal`, decouple "color =
  emotion", or touch position/strength/connection. A memory's color stays `moodColor(memory.mood)` under every palette.

**Non-goals**

- **No settings/picker UI.** The screen that lists palettes and lets the user choose — plus account and the
  staging-layer customization placeholder — is [52.settings-page](52.settings-page.md). This unit builds the preference
  - apply the picker _calls_, not the picker.
- **No free-form palette editor.** v1 is "a good default + choose from a small named set" ([P1]); a user authoring an
  arbitrary 13-color palette is out of v1 scope (PRD §8.2). The guardrail validator is shaped so an editor could reuse
  it later, but no editor ships here.
- **No new swap mechanism.** `moodColor` / `setMoodPalette` / `resolvePalette` are plan 17's ([17.emotion-model](17.emotion-model.md));
  this unit _supplies_ palettes and a stored choice to that seam. It does not re-invent the seam.
- **No rendering change.** `entities/star` and `entities/nebula` import only `moodColor` and stay palette-agnostic
  ([M6], plan 17); swapping a palette re-colors through the one function with **zero** rendering edits.
- **No staging-layer (non-meaning) customization** — background / theme / effect / camera-mood ([P4]) is a reserved
  placeholder in [52.settings-page](52.settings-page.md), not built here.
- **No color-space / accessibility engine** (color-blind remaps, contrast tuning). The registry may _include_ such a
  palette as content, but no generative color-transform ships.
- No mood-color scalars in `values.yaml` — palette tables are content (excluded per plan 14 / plan 17).

## Design

### Placement (domain-first BE + app-layer FE)

**Decided (2026-07):** the palette preference is a **supporting-context** concern, placed in **`internal/account`** — a
new, minimal supporting context for user preferences (per the cross-cutting brief CC8: "`internal/account` (or reuse
`platform`): user preferences (the palette choice, Epic I). Keep minimal."). It is _not_ core-domain: the palette choice
is presentation preference, not memory behavior, so it must not live in `internal/memory` (that would let a presentation
choice ride the memory aggregate). It is a supporting context, not `platform`, because it carries a (small) business
meaning ("this user prefers palette X") rather than pure infrastructure — but it stays a **single-scalar** slice; if it
never grows beyond one field, a `platform` preference table is an acceptable equivalent (decided at implement per the
be-architecture procedure). The RPC handler is thin (proto↔domain map + call); per-user scope from the auth context is
applied on every read/write (§2.9).

The FE apply is an **app-layer** step (`app/` composition root, §3.1) — reading the stored preference and calling
`setMoodPalette` is a wiring concern that runs once before the first render, not a `feature`. The **registry** lives in
`packages/emotion` beside the palette it extends. The set-and-apply path (used by [52]) is an `api` segment that both
persists the id (RPC) and re-applies via `setMoodPalette`.

### Palette registry (`packages/emotion`, code/content)

The registry extends plan 17's `MoodPalette` (`{ name, colors: Record<Mood, Color> }`) with a stable **id** and a lookup:

```
// packages/emotion — extends palette.ts (plan 17)
export const DEFAULT_PALETTE_ID = 'cosimosi-default'
export const PALETTES: Readonly<Record<string, MoodPalette>>   // ≥2 entries, keyed by id
export function paletteById(id: string): MoodPalette            // unknown id → defaultMoodPalette (fail-safe)
export function listPalettes(): ReadonlyArray<{ id: string; name: string }>   // for the [52] picker
```

- **≥2 named palettes.** The `defaultMoodPalette` (id `cosimosi-default`, the plan-17 table) plus **at least one**
  alternative authored for a different color sense (e.g. a warmer or higher-contrast reading). The count is a derived
  array length, not a declared number; adding a palette is a content change in `packages/emotion`.
- **Tables are content.** Each palette is a 13-entry `mood → color` object — array/table _content_, exactly like a theme
  CSS table (plan 14 / plan 17 exclusion). It stays in `packages/emotion`, **never** `values.yaml` ([M6]).
- **Completeness is enforced** by plan 17's existing `assertCompletePalette` (every `MoodPalette` must cover all 13
  moods); every registry entry passes it, so no palette can silently drop a mood's color.
- **id, not table, is the stored/transported unit.** The backend stores an id; the client resolves id → palette locally.
  The server never ships a color table, keeping the projection frontend-only (plan 17 golden-parity stance: colors are
  presentation, not server-authoritative facts).

### Stored preference + RPC (`internal/account`, Connect unary — §2.7)

A single per-user scalar: the chosen **palette id** (a short string; default `cosimosi-default` when unset). Persisted
per user (schema owned at implement by the account/platform slice — DB schema is code, excluded from values). Two thin
unary RPCs:

```proto
// account.v1.AccountService (or a platform preference service — decided at implement)
rpc GetPalettePreference(GetPalettePreferenceRequest) returns (PalettePreference);       // read (NO_SIDE_EFFECTS)
rpc SetPalettePreference(SetPalettePreferenceRequest) returns (PalettePreference);       // write

message GetPalettePreferenceRequest {}                         // caller = the authenticated user (scope from auth ctx)
message SetPalettePreferenceRequest { string palette_id = 1; } // the chosen registry id — the ONLY writable field
message PalettePreference          { string palette_id = 1; }  // resolved id (default when unset)
```

- **The `palette_id` is the entire writable surface.** There is no `mood`, no color, no coordinate, no memory id, no
  position/strength field anywhere in the contract — so a preference write **cannot** structurally reach the meaning
  layer ([I11][P2]). This mirrors plan 20's schema-forced defense: the boundary's shape, not a runtime check, is what
  makes the invariant unbreakable.
- **Validation.** `SetPalettePreference` accepts only an id **known to the registry** (or a first-party allow-list the
  registry publishes) — an unknown id is rejected (or, on read, coerced to default). Because the id maps to a first-party
  content table, a client cannot inject arbitrary colors through it. **Decided (2026-07):** v1 stores an **id from the
  first-party registry only**; an arbitrary user-authored color table is out of scope (§Non-goals), so the server never
  needs to accept or store a color table.
- **Read is `NO_SIDE_EFFECTS`;** per-user scope applies to both RPCs (§2.9). The get RPC is what app-init reads.

### App-init apply (FE app-layer, §3.1)

The authenticated product apply is a commit gate: it resolves before palette-dependent children mount, so the whole
universe is colored by the user's palette from the first committed frame — through plan 17's single seam, with no
rendering-code involvement:

```
1. app boot → GetPalettePreference() → palette_id      (default 'cosimosi-default' if unset / unauthenticated)
2. resolvePaletteById(palette_id) → { canonical id, MoodPalette }
                                                       (unknown id → canonical default, fail-safe)
3. setMoodPalette(chosen)                               (plan 17 seam; activePalette now = the user's palette)
4. release authenticated children → moodColor(mood) reads it everywhere (star, nebula)
```

- **Order matters:** a neutral/loading host remains mounted until apply completes, so the canvas cannot commit with a
  previous/default palette and visibly flash. Failed, unset, or unknown reads resolve to the canonical default. Login
  and boot surfaces sit outside this authenticated product gate.
- **Set-and-apply (used by [52]'s picker):** `SetPalettePreference(id)` persists the id **and** immediately calls
  `paletteById(id)` → `setMoodPalette(...)`, re-coloring the running universe through the one seam. **Decided (2026-07):**
  the picker's optimistic apply is `setMoodPalette` first (instant visual), then the RPC persists; a persist failure
  reverts to the separately tracked last canonical server-confirmed id and re-applies. Rapid writes serialize within
  the initiating auth epoch; queued writes are cancelled before dispatch after a scope change and late old-user
  completions cannot update the new user's state. The _live_ re-color needs no `GetUniverse` refetch — only
  `activePalette` changed, and rendering re-reads `moodColor` on the next frame.
- **web + mobile parity (§3.5):** both apps share `packages/emotion` (the registry) and the `api` segment verbatim; only
  the app-layer host that fires the init may differ per platform. `setMoodPalette` is process-global module state (plan
  17), so a single call re-colors the whole app.

### Axis-consistency guardrails ([P3])

A **pure validator** in `packages/emotion` inspects a candidate palette against the emotion axes so a palette cannot
silently strip emotion information (warm should read positive, cool should read negative):

```
export type PaletteAxisWarning = { mood: Mood; issue: 'valence_hue_mismatch' | ... }
export function checkPaletteAxisConsistency(palette: MoodPalette): PaletteAxisWarning[]   // [] = consistent
```

- **Rule.** For each mood, compare its color's saturation-attenuated HSL hue (warm ↔ cool, derived from the color — a
  formula, code) against the
  mood's **valence** (from plan 17's `moodCoordinate` / the `emotion.mood_valence` map): positive-valence moods should
  trend warm, negative-valence cool. Grayscale has exactly zero hue evidence; near-gray evidence approaches zero. A
  mismatch beyond a tolerance yields a warning ([P3]).
- **Warn, never block ([P3] is SHOULD).** The validator returns warnings for the [52] picker to surface; it does **not**
  hard-reject. The shipped registry palettes are authored to return **no** warnings (an acceptance criterion). Only
  _custom_ palettes could warn — and since v1 stores only first-party ids, the validator's live role in v1 is to keep
  the registry honest (a build/test check) with the seam ready for a future editor.
- **Hue math is a formula (code);** only the warm/cool **valence threshold** scalar is a tuning number → `values.yaml`
  (below). The valence coordinates it reads are plan 17's existing `emotion.mood_valence` map (not re-declared here).
- **This is not the meaning guard.** [I11]/[P2] immutability (the hard boundary — the id is the only writable field) is
  the _structural_ guarantee; the axis guardrail is a _quality_ recommendation ([P3]) layered on top. Plan 17 explicitly
  deferred "whether to _enforce_ axis-consistency on customized palettes" to this unit — resolved here as **warn-only**.

### Golden-parity stance

**Not golden-parity.** Consistent with plan 17: the `mood → color` palette (and thus the registry + the hue-based
guardrail) is a **presentation** concern that lives only on the frontend (`packages/emotion`). The Go side owns emotion
_facts_ (mood, coordinates) and now the _stored id_ (a scalar), but never a color or the hue math — so there is no
TS↔Go color computation to pin. The preference is a stored scalar, not derived state, so it needs no parity fixture; the
only server logic is store/fetch/validate-against-known-ids.

## Acceptance Criteria

1. A **palette registry** in `packages/emotion` exposes **≥2** named `MoodPalette` values (the plan-17 `defaultMoodPalette`
   plus at least one alternative) keyed by a stable **palette id**, with `paletteById(id)`, `listPalettes()`, and
   `DEFAULT_PALETTE_ID`; every entry passes `assertCompletePalette` (all 13 moods present) ([P1][M6]).
2. Palette **tables are code/content** in `packages/emotion` and appear **nowhere** in `spec/values.yaml`; the count of
   palettes is a derived array length, not a declared number ([M6]).
3. A per-user **palette preference** stores a single **palette id** (default `cosimosi-default` when unset), read/written
   through `GetPalettePreference` / `SetPalettePreference` on a thin RPC, scoped to the authenticated user; the read is
   `NO_SIDE_EFFECTS` ([P1], §2.7/§2.9).
4. `SetPalettePreference`'s **only** writable field is `palette_id`; the contract carries no mood, color, coordinate,
   memory id, position, or strength field — so a preference write cannot structurally reach the meaning layer ([I11][P2]).
5. `SetPalettePreference` accepts **only a registry-known id** (unknown id rejected on write / coerced to default on
   read); a client cannot inject an arbitrary color table through the preference ([I11][P2]).
6. At app init the stored preference is read and applied via **`setMoodPalette(chosen)` before the first render**, so the
   universe is colored by the user's palette from the first frame through plan 17's single `moodColor` seam; an unset /
   unknown id or a failed/unauthenticated read falls back to `defaultMoodPalette` ([M6]).
7. Setting the preference **persists the id and re-applies live** through `setMoodPalette`, re-coloring the running
   universe with **no** rendering-code change and **no** `GetUniverse` refetch (only `activePalette` changed) ([M6]).
8. `entities/star` and `entities/nebula` are unchanged and import only `moodColor`; a palette swap re-colors the whole
   universe through the one function with zero rendering edits ([M6][V4]).
9. A memory's color is **always** `moodColor(memory.mood)` under every palette; swapping the palette never overrides a
   memory's stored `mood`/`valence`/`arousal`, never decouples "color = emotion", and never feeds position, strength, or
   connection ([I3][I11][P2]).
10. A pure `checkPaletteAxisConsistency(palette)` validator flags warm/cool ↔ valence mismatches as **warnings** (not
    hard blocks), reading the mood valence from plan 17's `emotion.mood_valence` map; **every shipped registry palette
    returns no warnings** ([P3]).
11. The palette registry and guardrail are **frontend-only** (not golden-parity); the backend stores only the id scalar
    and computes no color, consistent with plan 17's projection stance.
12. **web + mobile parity:** both apps share the `packages/emotion` registry and the preference `api` segment verbatim;
    only the app-layer init host may fork (§3.5).
