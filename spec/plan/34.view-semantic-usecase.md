# 34. view semantic usecase

> View-semantic use-case — the backend read-only viewing of a gist star's already-pregenerated compressed text: read
> `semantic_stages[stage]` for a memory, spend Twinkle through the consumer-owned gate (no-op until Epic F), and pass the
> gist-depth signal to the (Epic F) cost curve — cheaper the deeper the gist ([R8][G4]). It never rewrites anything.
> Scope: BE | Status: built (as-built read-path: [tech/memory-recall.md](../tech/memory-recall.md) §8)
>
> Layer U (use-case) — the third unit of Epic C (Sprint 2). Home: `internal/memory` use-case `ViewSemantic` + the
> consumer-owned `SpendGate` port. It reads [16.memory-aggregate-schema](16.memory-aggregate-schema.md)'s
> `semantic_stages`/`semantic_stage` (pregenerated at launch by [20.encode-usecase](20.encode-usecase.md) /
> [22.ai-worker-pipeline](22.ai-worker-pipeline.md)), shares the `SpendGate` seam with
> [33.recall-usecase](33.recall-usecase.md), and is consumed by Epic E's [42.gist-star-rendering](42.gist-star-rendering.md).

## Purpose

A gist star (신피질) is what a memory _becomes_ once time has compressed it — the specifics fall away and only the
meaning remains. cosimosi lets you **look at** that compressed version, but it is strictly a viewing act, not a
remembering act: **the gist star is read-only** ([R8]). You do not re-write it, you do not reconsolidate it, you do not
change its shape — you simply read the abstracted text and pay a little Twinkle for the privilege ([R8][G1]). And because
a gist is _easier to reach_ the more abstracted it has become, viewing it is **cheaper the deeper the gist** ([R8][G4]) —
the opposite of episodic recall, which gets _costlier_ the more a memory has faded ([F4]).

This unit owns that read: the `ViewSemantic` use-case that, given a memory and a gist stage, returns the pregenerated
compressed text for that stage and charges the view through the economy gate. It is the sharp boundary between the
**episodic** and **semantic** halves of the product. Recall/reconsolidation ([33.recall-usecase](33.recall-usecase.md),
[32.reconsolidation-rules](32.reconsolidation-rules.md)) act on the _episodic_ memory (해마) — they reset brightness,
recover decay, and may rewrite `current_text` on prediction error ([R1]–[R7]). `ViewSemantic` does **none** of that: it
reads a stored gist string and returns it. Nothing in this unit writes to `episodic_memories`, `diaries`, or
`memory_provenance`.

The gist-stage _texts already exist_ — they were pregenerated at launch into `semantic_stages` (the semanticize job,
[20.encode-usecase](20.encode-usecase.md) / [22.ai-worker-pipeline](22.ai-worker-pipeline.md), [C7]). So in Sprint 2
this unit is fully buildable as a headless read: it looks up `semantic_stages[stage]`, spends, and returns. What it does
_not_ own — the machinery by which a gist star actually **rises** to a viewable stage over universe-time — is Epic E's
([40.semanticization](40.semanticization.md) / [41.consolidation-usecase](41.consolidation-usecase.md)); this unit reads
the stage's text, it does not advance the stage.

## Scope / Non-goals

**In scope**

- **`ViewSemantic`** — a read-only use-case in `internal/memory`. Given `(userScope, episodicMemoryId, stage)`:
  1. load the memory (per-user scoped); require it to have a **populated `semantic_stages`** and to have **risen to at
     least `stage`** (`stage ≤ semantic_stage`, [C6]) — viewing a not-yet-risen stage is refused, not fabricated;
  2. derive the **gist-depth signal** from the requested `stage` (deeper = cheaper, [R8][G4]) and **spend Twinkle**
     through the consumer-owned `SpendGate` port (no-op allow-all until Epic F — see below);
  3. return the pregenerated **`semantic_stages[stage]` text** (read-only) plus the meta the viewer needs.
- The consumer-owned **`SpendGate` port** (shared seam with [33.recall-usecase](33.recall-usecase.md)) —
  check-and-spend, given a **spend intent** carrying the _gist-depth signal_ (not a price). Epics C–E bind a **no-op
  allow-all default** so the loop works without cost; Epic F ([44.earn-spend-usecase](44.earn-spend-usecase.md)) binds
  the real balance-check-and-deduct.
- The **`ViewSemantic` RPC** on `memory.v1.MemoryService` (Connect, unary) — a thin handler mapping proto↔domain and
  calling the use-case; **not** GET/`NO_SIDE_EFFECTS` (it spends).
- **Per-user scoping** on the memory load and the spend (§4).

**Non-goals**

- **No rewrite, no reconsolidation, no episodic mutation.** `ViewSemantic` never touches `current_text`, `seed`,
  `recall_count`, brightness, decay, the gist timer, or `memory_provenance`. Recall/reconsolidation (episodic-only, [R1])
  are [33.recall-usecase](33.recall-usecase.md) / [32.reconsolidation-rules](32.reconsolidation-rules.md). Viewing a gist
  is a pure read.
- **No gist-text generation.** The 4 gist-stage texts are pregenerated at launch by the semanticize job
  ([20.encode-usecase](20.encode-usecase.md) / [22.ai-worker-pipeline](22.ai-worker-pipeline.md), [C7]); this unit only
  _reads_ `semantic_stages[stage]`. It never calls an LLM.
- **No stage advancement (rising).** `semantic_stage`++ / the gist timer / the z-axis 2-store coords — how a gist rises
  to a viewable stage over universe-time — are Epic E ([40.semanticization](40.semanticization.md),
  [41.consolidation-usecase](41.consolidation-usecase.md), [C5][C6][C6a]). This unit reads whatever stage the memory has
  already reached.
- **No Twinkle pricing, balance, or curve values.** The gist-view **cost curve** (gist-depth → Twinkle price) and the
  two-tier balance / spend order / earn amounts are all Epic F ([43.stardust-ledger](43.stardust-ledger.md) /
  [44.earn-spend-usecase](44.earn-spend-usecase.md)). This unit passes the _gist-depth signal_; it **never** hardcodes a
  price. No cost constant appears here (CC2/CC3).
- **No universe-clock advance.** Reading a gist does **not** advance the clock ([T3]) — viewing is not one of the two
  advance triggers ([T2]). No `SyncToToday`, no launch.
- **No UI.** The gist-star rendering (신피질 stars rising, the z-axis layer, the view interaction) is Epic E
  ([42.gist-star-rendering](42.gist-star-rendering.md)); the star-detail entry point that offers "변천사"/gist viewing is
  [35.star-detail-panel](35.star-detail-panel.md). This unit is headless.
- **No schema/table changes.** `semantic_stages`, `semantic_stage`, and `semanticize_timer_reset_at` already exist on
  `episodic_memories` ([16.memory-aggregate-schema](16.memory-aggregate-schema.md)); this unit only reads them.

## Design

### Placement (domain-first Go)

`ViewSemantic` is an application use-case in the core-domain `internal/memory` context, alongside `Recall`/`Reconsolidate`
(Epic C) and `PersistEncoded` (Epic A). By ARCHITECTURE §2.3 the context has already earned its `{app,pg,rpc}` split
(Epic A). The **`SpendGate`** is the shared spend seam — **declared once with [33.recall-usecase](33.recall-usecase.md)**
and referenced here (§2.4, consumer-owned ports; both recall and view consume it); its
concrete is injected at `cmd/api` — a no-op allow-all in Epics C–E, the real ledger gate in Epic F
([44.earn-spend-usecase](44.earn-spend-usecase.md)). Dependency direction:
`memory/rpc → ViewSemantic (use-case) → pure domain (the gist-depth signal derivation)`, with the memory repository and
the `SpendGate` reached only through consumer-owned interfaces. No proto/sqlc type crosses inward (§2.9#4); the handler
stays thin (§2.7/§2.9#7).

**Decided (2026-07):** `ViewSemantic` lives in `internal/memory`, **not** in a separate "semantic/gist" context. The
gist stages are _columns on the `episodic_memories` aggregate root_ ([16.memory-aggregate-schema](16.memory-aggregate-schema.md)) —
reading them is the memory aggregate's own behavior (CC8). The economy is the only cross-context concern, and it is
reached through the `SpendGate` port, not by importing `internal/twinkle`.

### `ViewSemantic` — the read

Given `(userScope, episodicMemoryId, stage)`:

1. **Load the memory** (per-user scoped) through the view's own consumer-owned **`GistReader`** port —
   `EpisodicMemoryGist(scope, memoryID) → MemoryGist{semantic_stage, semantic_stages}`, a standalone read (the recall
   surface reads the same columns but only inside its transaction). It must reference the authenticated user; a
   missing, other-user, or soft-deleted row is the canonical not-found (no cross-user read, §4; full-delete hides the
   whole row, further release semantics are [48.deletion-rules](48.deletion-rules.md)'s).
2. **Validate the stage against what the memory has actually reached** ([C6], §2.9#8 — server-authoritative, the client
   cannot claim a stage it has not risen to):
   - `semantic_stages` must be **populated** (non-NULL — the semanticize job has run; until then a gist is not viewable);
   - the requested `stage` must be in the valid stage range **and** `stage ≤ semantic_stage` (the memory has _risen_ to
     at least this stage in the 신피질, [C6]/[V9]). A stage above the reached stage is **refused** with a canonical
     error (`ErrViewSemanticStageNotRisen`) — the unit never fabricates a text for a stage that has not risen. Stages
     are the **1-based ladder** ([C6a]: `semantic_stage` 0 = concrete, nothing viewable; stage _k_'s text is array
     index _k−1_); a `stage < 1` is invalid input, not a refusal.
   - **Decided (2026-07):** the number of gist stages is a **derived array length** over `semantic_stages` (the [C6a]
     4-stage model is owned by [40.semanticization](40.semanticization.md)); this unit reads whatever stages exist and
     does not declare a stage count in values (config rule — array lengths are derived).
3. **Derive the gist-depth signal** — the _deeper_ the requested stage, the cheaper the view ([R8][G4]). This unit
   produces a **gist-depth signal** (the reached/requested `stage`, a monotone "how abstracted" measure), **not** a
   Twinkle price. The pure relationship _gist-depth → depth signal_ is a formula in this unit's domain (code, not values);
   the **price** it maps to is Epic F's cost curve (CC3).
4. **Spend through the `SpendGate`** — call `SpendGate.CheckAndSpend(ctx, scope, GistViewSpendIntent(memoryId, stage))`,
   a `SpendIntent` carrying the _view kind_ (`view_gist`) and the _gist-depth signal_ (the requested `stage` itself, a
   monotone "how abstracted" measure — never a price). In Epics C–E the bound gate is
   the **no-op allow-all** (always permits, deducts nothing), so viewing works without an economy. In Epic F the real
   gate resolves the signal against the cost curve, checks the two-tier balance, and deducts (basic→additional, [G2]) —
   or refuses with the canonical `insufficient-twinkle` error. `ViewSemantic` **never** computes or hardcodes the amount;
   it hands the signal to the gate. **The spend is a precondition of the read** — on a gate refusal the text is not returned.
5. **Return** the pregenerated `semantic_stages[stage]` text (read-only) and the meta the viewer needs (the memory id,
   the viewed stage, the reached `semantic_stage`, so the UI can show "stage k of the reached depth"). No write of any
   kind occurs — no `recall_count`, no timer, no provenance, no clock advance.

**The recall/view asymmetry (CC3), stated once here so the two units stay coherent:**

|                      | trigger                | acts on                       | mutation                                                                      | cost direction                             | owner                                     |
| -------------------- | ---------------------- | ----------------------------- | ----------------------------------------------------------------------------- | ------------------------------------------ | ----------------------------------------- |
| **회고 (recall)**    | 회고하기 button ([R1]) | _episodic_ memory (해마)      | brightness reset, decay recovery, timer reset, LTP, maybe rewrite ([R2]–[R7]) | **costlier the deeper the decay** ([F4])   | [33.recall-usecase](33.recall-usecase.md) |
| **요지 열람 (view)** | gist-star view ([R8])  | _semantic_ gist text (신피질) | **none** (read-only)                                                          | **cheaper the deeper the gist** ([R8][G4]) | this unit                                 |

### `SpendGate` port (consumer-owned, shared with recall — declared once with [33])

The shared spend seam (§2.4). It is **declared once** in `internal/memory` (with
[33.recall-usecase](33.recall-usecase.md), the primary consumer); this unit **references the same interface**.
Domain-shaped inputs/outputs only — no proto, no sqlc, no twinkle row-types cross this boundary:

- `CheckAndSpend(ctx, scope, spend SpendIntent) → error` — check-and-spend. `SpendIntent` is a **domain value** carrying
  the spend _kind_ (`view_gist` here; `recall` for [33.recall-usecase](33.recall-usecase.md)), the target memory, and
  its cost _signal_ (the **gist-depth `stage`** for a view; the **decay-depth/accessibility signal** for a recall, [F4]).
  The gate — not the use-case — turns the signal into a price and checks the balance; a refusal returns the canonical
  `insufficient-twinkle` error (nil = permitted). The amount charged / remaining balance for the HUD is a separate
  `GetBalance` read ([44.earn-spend-usecase](44.earn-spend-usecase.md)), not carried on the spend.

The **no-op allow-all default** (Epics C–E) always returns "permitted, nothing charged". Epic F's
[44.earn-spend-usecase](44.earn-spend-usecase.md) binds a concrete that consults
[43.stardust-ledger](43.stardust-ledger.md)'s cost curve + two-tier balance. Recall and view share this **one** port so
the economy has a single check-and-spend seam across Epic C, exactly as [20.encode-usecase](20.encode-usecase.md)'s
`Extractor` / [30.time-advance-usecase](30.time-advance-usecase.md)'s `AdvanceProgression` reserve their seams behind
consumer-owned ports.

**Decided (2026-07):** the `SpendGate` port is **declared once with [33.recall-usecase](33.recall-usecase.md)** (shared
by recall and view — canonical `CheckAndSpend(ctx, scope, spend SpendIntent) → error`) and its **no-op allow-all default
is bound in Epic C**; both plans reference the same interface so Epic F rebinds a single concrete.
The cost _signal_ is a plain domain scalar/enum this unit computes from `stage`; the **mapping to Twinkle** is entirely
the gate's (CC2/CC3) — no price crosses this boundary in either direction until Epic F.

### RPC contract (`memory.v1.MemoryService`, Connect, unary — §2.7)

proto is a DTO layer mapped at `memory/rpc`; the handler is thin (map proto↔domain + call the use-case, §2.9#7). The
request names the memory and the stage; the response returns the gist text and the meta needed to render/label it.

```proto
// read-only gist-stage view; spends Twinkle (via the SpendGate) — NOT NO_SIDE_EFFECTS.
rpc ViewSemantic(ViewSemanticRequest) returns (ViewSemanticResponse);

// stage the user chose to view [C6] + the client operation id (idempotency, A2)
message ViewSemanticRequest  { string episodic_memory_id = 1; int32 stage = 2; string operation_id = 3; }
message ViewSemanticResponse { string text = 1;               // pregenerated semantic_stages[stage] (read-only) [R8][R8a]
                               int32  stage = 2;               // the stage that was viewed
                               int32  reached_stage = 3; }     // memory.semantic_stage — how far it has risen [C6][V9]
// NOTE: no current_text, no seed, no provenance, no recall/timer field — this RPC reads a gist, it never rewrites [R8][I2].
```

The paid view runs in a **memory-owned transaction** (`InViewSemanticTx`: graph lock + gist read + receipt +
spend), so the target read, the Twinkle debit, and the idempotency receipt commit together (A3). It requires
`operation_id`: a response-loss retry with the same id + same (memory, stage) replays the committed gist text with **no
second debit**, and a same-id/different-input reuse is `ErrOperationConflict`. The transaction still writes no
clock/LLM/reconsolidation/provenance/representation field — the view remains read-only in every sense but its own
debit + receipt.

`ViewSemantic` is **not** GET-eligible / `NO_SIDE_EFFECTS`: it spends Twinkle (a side effect on the ledger in Epic F, a
no-op today), so it is a unary POST like recall — distinct from the free reads (`GetUniverse` meta, the forgotten
episodic text [R1], and the star-detail meta [35.star-detail-panel](35.star-detail-panel.md)) which are `NO_SIDE_EFFECTS`.
**Decided (2026-07):** it is a **distinct RPC** (not folded into `GetUniverse`) because it spends and is a deliberate
per-star action, mirroring recall's own RPC.

### Golden-parity stance

**Not golden-parity.** Unlike the shared FE↔BE math of [18.synapse-plasticity](18.synapse-plasticity.md) /
[37.forgetting-decay](37.forgetting-decay.md), `ViewSemantic` is a server-only read+spend: the client renders whatever
text the server returns and never recomputes it (§2.9#8 — the reached stage and the spend are server-authoritative). The
one domain formula it carries — the _gist-depth signal_ — is a monotone function of `stage` used only to feed the gate;
the client never mirrors it (the price is the gate's, Epic F). So there is no dual-runtime function to keep in lockstep.

### Thin handler / dependency rule

The `memory/rpc` `ViewSemantic` handler maps `episodic_memory_id`/`stage` to the use-case input and maps the domain
result back to the response; **all** policy — the stage-reached validation, the gist-depth signal derivation, the
spend-before-return ordering — lives in the use-case/domain (§2.9#7). No proto/sqlc type reaches the use-case or the pure
domain (§2.9#4); the memory repository and `SpendGate` are consumer-owned interfaces (§2.9#6).

## Acceptance Criteria

1. `ViewSemantic(userScope, episodicMemoryId, stage)` returns the **pregenerated `semantic_stages[stage]` text**
   read-only; it writes **nothing** — no `current_text`, `seed`, `recall_count`, brightness, gist timer,
   `memory_provenance`, or clock advance is touched. ([R8][I2][I8])
2. The gist text is **never generated** in this unit — it is read from `semantic_stages`, pregenerated at launch by the
   semanticize job; `ViewSemantic` calls **no** LLM. ([C7][R8a])
3. A view is **refused** (canonical error) when `semantic_stages` is NULL (not yet pregenerated) or when
   `stage > semantic_stage` (the memory has not _risen_ to that stage) — the unit never fabricates a text for an unreached
   stage; the reached stage is **server-authoritative**. ([C6][V9], §2.9#8)
4. `ViewSemantic` derives a **gist-depth signal** (deeper stage ⇒ cheaper) and **spends via the consumer-owned
   `SpendGate`** carrying that signal — it **never** hardcodes or computes a Twinkle price; the spend is a **precondition**
   of the read (a gate refusal returns no text). ([R8][G4][G1], CC2/CC3)
5. In Epics C–E the bound `SpendGate` is the **no-op allow-all** default (viewing works with no economy, nothing charged);
   Epic F ([44.earn-spend-usecase](44.earn-spend-usecase.md)) rebinds the real balance-check-and-deduct against the gist
   cost curve — with **no change** to `ViewSemantic`. ([G1][G2])
6. `ViewSemantic` performs **no reconsolidation and no episodic mutation** — it neither invokes the reconsolidation
   predicate ([32.reconsolidation-rules](32.reconsolidation-rules.md)) nor resets brightness/decay/timer/LTP
   ([33.recall-usecase](33.recall-usecase.md)); recall/reconsolidation are episodic-only and separate. ([R1][R8])
7. Reading a gist **does not advance the universe clock** — it is not one of the two advance triggers; the clock is
   unchanged. ([T3][T2][I10])
8. The `ViewSemantic` RPC is unary and **not** `NO_SIDE_EFFECTS` (it spends), the handler is **thin** (proto↔domain map +
   call), and no proto/sqlc type leaks into the use-case or pure domain; the `SpendGate` and the memory repository are
   consumer-owned interfaces. (§2.4/§2.7/§2.9#4/#6/#7)
9. The memory load and the spend are **scoped to the authenticated user**; a memory belonging to another user is
   not-found, and `pnpm lint:persistence` passes. (§4)
