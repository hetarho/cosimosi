# 34. 관리자 콘솔 - AdminService + LLM 런타임 운영 (as-built)

> 현재 구현 기준 문서. `/admin` 콘솔이 LLM 공급자·키·활성 선택을 재배포 없이 운영하고(키는 AES-256-GCM 봉투 암호화·평문 비반환), 외부 도구가 못 잡는 비즈니스/AI 비용 대시보드를 담는 SSOT다. | 범위: FS | 상태: 구현 완료

## 목적

20이 5사 LLM 공급자 추상화를 깔았지만 선택·키가 전부 env라, 교체 = VPS `.env` 수정 + 컨테이너 재생성이었다. 21부터 실 추출이 일기마다 돌면 키 로테이션·모델 변경·공급자 전환이 잦아지므로, 그 마찰을 웹 콘솔로 옮긴다. 키는 탈취돼도 무의미해야 하므로 DB에는 봉투 암호문만 두고 마스터키는 서버 env에만 둔다. 대시보드는 Sentry(에러)·PostHog(행동)·호스팅 콘솔(CPU)이 이미 담당하는 지표를 복제하지 않고, 빈 곳인 비즈니스 합계·잡 큐·LLM 토큰 사용량(사용자당 AI 원가 = 수익화 단위경제)만 담는다.

값 SSOT는 `spec/policy/domain/admin.md`다(접근 규칙·키 보안·런타임 우선순위·계측·대시보드 범위).

## 현재 구현

- 바운디드 컨텍스트는 `backend/internal/admin/`다(30 settings 미러: handler/service/repository/crypto 분리). 키 암복호화·공급자 설정 병합·대시보드 집계를 소유하고, `llm.ConfigSource`/`llm.UsageSink`를 구현해 `llm` Resolver에 주입한다 — `llm`은 DB·admin을 모른다(의존 역전, 헌법7).
- `internal/llm`은 공급자 매트릭스 데이터화(`providers.go`)·동적 클라이언트(`resolver.go`)·토큰 usage 매핑(`llm.go` + 어댑터 3종)으로 확장됐다.
- 추출 on/off는 콘솔 액션이다. **extractor에 env 노브가 없다**(`config.go` 주석으로 명시). `ai.NewExtractor`가 `SwitchingExtractor`를 반환해, 콘솔에 활성 selection이 있으면 실 LLM 추출, 없으면 키리스 mock으로 라우팅한다.
- FE `/admin`은 lazy 코드 스플릿 라우트로 메인 번들에 영향이 0이며, 관리자 판정은 전적으로 서버 게이트가 한다.

## 접근 게이트 (allowlist · fail-closed)

- `AdminService`의 모든 RPC는 `rpcserver.NewAdminGateInterceptor(cfg.AdminUserIDs)` 뒤에 있다(`backend/internal/platform/rpcserver/admin_gate.go`).
- 마운트는 `server.go`의 `New`에서 한다: 공유 옵션 체인(logging→auth)을 **복제**한 뒤 `connect.WithInterceptors(adminGate)`를 누적해 `NewAdminServiceHandler`에만 장착한다 — 공유 `opts` 슬라이스를 앨리어싱하지 않는다.
- allowlist 항목은 쉼표 구분 **Supabase user UUID 또는 계정 이메일**이고, 게이트는 `UserIDFromContext`(JWT `sub`)와 `UserEmailFromContext`(검증된 `email`)를 **대소문자 무시**로 대조한다. 둘 중 하나라도 맞으면 통과.
- 이메일 매칭은 토큰의 `user_metadata.email_verified=true`일 때만 신뢰한다(`auth.go`의 `emailVerified`). 미검증 가입으로 allowlist 이메일을 선점하는 우회를 막는다 — 플래그 없는 토큰은 UUID 항목으로만 통과한다.
- **빈 allowlist = 전원 거부(fail-closed).** 거부는 항상 `PermissionDenied` + 불투명 메시지("permission denied") 한 종류다. 미인증은 게이트에 닿기 전 auth 인터셉터의 `Unauthenticated`가 먼저 끊는다.

## 키 암호화 (AES-256-GCM 봉투 · write-only)

- `backend/internal/admin/crypto.go`의 `Cipher`가 봉투 암호화를 소유한다.
- 마스터키 `LLM_KEY_ENCRYPTION_KEY` = base64 32바이트(`openssl rand -base64 32`), 서버 env에만 존재. `NewCipher("")`는 `(nil, nil)`을 반환하는 정상 degraded 상태(읽기는 동작, 키 저장은 거부). 잘못된 base64·길이는 에러를 반환하고, 조립 루트(`cmd/api`/`cmd/worker`)가 그 에러로 **부팅 실패**시킨다 — 오타가 조용히 암호화를 끄는 것보다 낫다.
- 저장 포맷(BYTEA): `0x01(버전) ‖ nonce(12B) ‖ ciphertext+tag`. **AAD = provider명** — 다른 행으로 복사한 암호문은 복호화가 실패한다(행 간 스왑 방지). 버전 바이트는 키 회전 대비 포맷 진화용(회전 스크립트는 비목표).
- 평문 키의 생애: `SetProviderKey`/`TestProviderKey` 요청 본문 → 메모리에서 암호화/1회 핑 → 끝. 어떤 RPC 응답·로그·Sentry에도 평문 미포함(응답은 `key_set`+`key_last4`+`key_updated_at`만). `TestProviderKey` 실패 메시지는 service의 `scrub`로 키 원문을 `***`로 치환한다.
- 마스터키 미설정 시 `Seal`/`Open`이 `ErrEncryptionKeyMissing`을 반환하고, handler가 `FailedPrecondition`(env 이름 명시)으로 매핑한다.

## 런타임 LLM 설정 (DB > env · overrides-only)

- 공급자 5종의 목록·기본 모델·엔드포인트 SSOT는 **코드 매트릭스** `backend/internal/llm/providers.go`다(30의 overrides-only 철학). DB(`llm_provider_configs`)는 관리자가 바꾼 것만 담는다 — 시드 행 없음. `GetConfig`가 매트릭스 위에 DB 행을 병합해 항상 5 카드를 채운다.
- 매트릭스 키와 기본 모델(2026-06 기준): `openai`→`gpt-5.4-mini`, `gemini`→`gemini-3.5-flash`, `claude`→`claude-opus-4-8`, `deepseek`→`deepseek-v4-flash`, `grok`→`grok-4.3`. 각 공급자는 어댑터 family(openai-compat·anthropic·gemini)·키 env명·엔드포인트·`JSONSchema` 지원·`MaxTokensField`를 데이터로 가진다. 매트릭스의 키 집합이 곧 `AdminService`가 허용하는 provider 도메인이다.
- `NewForProvider(provider, model, apiKey, hc)`가 `factory.New`(env 경로)·`Resolver`(DB 경로)·`TestProviderKey`(임시 키 경로) 셋이 공유하는 단일 어댑터 생성자다. `factory.New`의 switch 중복은 매트릭스 참조로 정리됐다.
- 활성 추출 LLM은 `llm_selection` 단일행(`id=1 CHECK`). `model=''`은 공급자 기본 모델. `SetActive`는 `model ∈ models ∪ {default_model, ''}` 밖이면 `ErrInvalidModel`(→ `InvalidArgument`), provider가 매트릭스 밖이면 `ErrUnknownProvider`.
- 우선순위 DB > env는 `llm.NewResolver(src, cfg, sink)`가 구현한다(`resolver.go`):
  - `Complete` 시 `ConfigSource.ActiveLLM`을 **TTL 30s 캐시**로 읽어 `NewForProvider`로 위임한다 → `SetActiveLLM`·키 교체가 재시작 없이 ≤30s 내 다음 `Complete`부터 반영.
  - selection 비어 있음(`ok=false`)이면 env `factory.New` 클라이언트로 폴백. 소스 에러(DB 다운·복호화 실패)는 로그 후 마지막 라우트(없으면 폴백) 유지 — 계측·해결(SetActiveLLM)이 다음 TTL에서 재해결한다. 어떤 경우에도 추출을 실패시키지 않는다.
  - selection identity(provider/model/key)가 바뀔 때만 어댑터를 재빌드한다.
- **mock 키리스 경로는 resolver를 거치지 않는다.** `SwitchingExtractor`(`ai/switching_extractor.go`)가 resolver 호출 전에 `ActiveLLM`을 별도 30s TTL로 probe해, 활성 selection이 없으면 키리스 mock으로 라우팅한다. resolver 자체의 env `factory.New` 폴백은 두 TTL이 어긋난 창에서만 닿는 엣지 경로다.

## 사용량 계측 (usage)

- Resolver 경유 `Complete` 성공 1회 = `llm_usage_daily`(**UTC day** × provider × model × kind) 행에 `calls`/`input_tokens`/`output_tokens` upsert 누적. service의 `RecordUsage`가 `UsageSink` 구현이며 kind는 `'extract'` 고정(임베딩 계측은 비목표 — `kind` 컬럼만 확장 대비).
- 계측은 베스트에포트: resolver가 싱크 실패를 `slog.Warn`만 남기고 호출을 실패시키지 않는다.
- 토큰 수는 공급자 응답 usage 필드 매핑(부재 시 0). `Response.Usage{InputTokens, OutputTokens}`를 openai-compat(`usage.prompt_tokens/completion_tokens`)·anthropic(`usage.input_tokens/output_tokens`)·gemini(`usageMetadata.*`) 어댑터가 채운다.
- 비용 환산은 FE 정적 단가표(`pages/admin/lib/pricing.ts`, USD/1M tokens) × 토큰 — 단가는 자주 바뀌므로 서버·DB에 두지 않고 "추정" 라벨을 단다. 단가표에 없는 모델은 비용을 표시하지 않는다.

## 대시보드 집계 (비중복 · on-demand)

- `GetAdminOverview`가 관리자 클릭 시 1회 집계한다(폴링 없음). 합계 4종(users·records·memories·synapses)·잡 큐(pending/processing/failed/done_24h)·최근 30일 일기 시리즈·최근 30일 usage.
- 합계는 인덱스/시퀀셜 카운트, 시리즈는 `records`의 30일 `GROUP BY day`(UTC), usage는 `llm_usage_daily` 범위 조회. 베타 규모(~50명)에서 ms 단위.
- 가입자 수 기본값은 `COUNT(DISTINCT user_id) FROM records`(스키마에 사용자 테이블 없음). `repository_pg.go`가 raw SQL `to_regclass('auth.users')` 가드로 Supabase DB에선 `count(*) FROM auth.users`로 대체하고, auth 스키마 없는 로컬 도커 pg에선 에러 없이 기본값을 쓴다.

## AdminService RPC 표면

`proto/cosimosi/v1/admin.proto`(같은 패키지, DTO 전용·unary 전용). 9 RPC.

- `GetLLMConfig` (NO_SIDE_EFFECTS): 항상 5 공급자 카드(매트릭스 ∪ DB) + 활성 선택 + `encryption_ready`. 활성 선택이 없으면 env 폴백 값을 비춘다(UI가 void를 안 보이게).
- `SetProviderKey`: write-only. provider 도메인·공백·최소 8자 검증 후 `Seal` → `UpsertLLMProviderKey`. 응답은 갱신된 `ProviderConfig`(키 상태만).
- `DeleteProviderKey`: `api_key_enc`/`api_key_last4`를 NULL로 비움(행의 모델 리스트는 보존).
- `UpdateProviderModels`: 모델 리스트 통째 교체(trim·중복 제거·빈 값 드롭·순서 보존). `updated_at`은 건드리지 않는다(그 컬럼은 "키 변경 시각"으로 소비 — 로테이션 감사 보존).
- `SetActiveLLM`: provider/model 도메인 검증 후 `UpsertLLMSelection`. 응답은 `{active_provider, active_model}`.
- `TestProviderKey`: `MaxTokens 16`·30s timeout 완성 1회로 키 검증. `api_key`가 비면 저장된 키를 복호화해 사용. 공급자 에러는 RPC 에러가 아니라 `{ok:false, message, latency_ms}`로 인라인 반환(메시지에서 키 스크럽).
- `GetAdminOverview` (NO_SIDE_EFFECTS): 위 대시보드 페이로드.
- `ListAdminUsers` (NO_SIDE_EFFECTS): 사용자 목록·`user_id` 검색·keyset 페이지(spec 46). production은 `auth.users`, 로컬은 앱 도메인 테이블 합집합. 행 = `user_id`·유효 잔액·지갑 시드 여부.
- `GrantUserStardust`: 별가루 보정 지급(spec 46) — target 지갑 멱등 시드 후 amount 증가 + `admin_stardust_grants` 감사, 한 트랜잭션. 자세한 규칙은 [46.admin-user-stardust-grant](46.admin-user-stardust-grant.md)·`policy/domain/admin.md`.

ProviderConfig 메시지: `provider`·`default_model`(읽기 전용)·`models`·`key_set`·`key_last4`·`key_updated_at`(RFC3339). **평문 키 필드는 어떤 메시지에도 없다.**

## 도메인 모델

`backend/internal/admin/admin.go` (태그 없는 순수 값, 헌법5):

## Flutter 동등성 기준

- AdminService는 BE 계약이다. Flutter 관리자 화면을 만든다면 9 RPC를 그대로 호출하고, 평문 키는 `SetProviderKey`/`TestProviderKey` 요청으로만 보내며 응답에서 키를 기대하지 않아야 한다(상태는 `key_set`+`key_last4`+`key_updated_at`만).
- 비관리자 응답(`PermissionDenied`)은 admin 표면 비노출을 위해 NotFound로 위장 렌더해야 한다(존재 광고 금지).
- 활성 모델 후보 = 그 공급자의 `default_model` ∪ `models`, `""` = 기본. `SetActiveLLM` 저장 후 ≤30s 내 추출이 무재시작으로 전환된다는 모델을 공유해야 한다.
- 비용은 클라이언트 정적 단가표 × 토큰의 "추정"이다 — 서버는 토큰만 돌려준다.
- 봉투 암호화·게이트·계측은 전부 서버 책임이며 클라이언트가 재현할 표면이 아니다.

## 수용 기준

1. `SetProviderKey`는 DB `api_key_enc`에 `0x01‖nonce‖ct`(AAD=provider) 암호문만 기록하고 평문 컬럼·평문 로그를 남기지 않는다.
2. 어떤 AdminService RPC 응답도 평문 API 키를 포함하지 않는다(`key_set`/`key_last4`/`key_updated_at`만).
3. `LLM_KEY_ENCRYPTION_KEY` 미설정이면 `SetProviderKey`는 `FailedPrecondition`(env 이름 명시)으로 거부하고, `GetLLMConfig.encryption_ready=false`로 FE가 배너를 띄운다. 마스터키 형식이 잘못되면 서버 부팅이 실패한다.
4. 한 provider의 암호문을 다른 provider로 복사하면 복호화에 실패한다(AAD 불일치).
5. `DeleteProviderKey`는 `api_key_enc`/`api_key_last4`를 NULL로 비우고 모델 리스트는 보존한다.
6. `SetActiveLLM` 후 프로세스 재시작 없이 ≤30s 내 다음 `Complete`부터 새 공급자·모델을 사용한다.
7. `llm_selection`이 비면 Resolver는 env로 폴백하고, selection이 없으면 `SwitchingExtractor`가 키리스 mock으로 라우팅한다(회귀 0).
8. `SetActiveLLM.model`이 `models ∪ {default_model, ''}` 밖이거나 provider가 매트릭스 밖이면 `InvalidArgument`를 반환한다.
9. `TestProviderKey`는 MaxTokens 16 완성 1회로 키를 검증해 `{ok, message, latency_ms}`를 반환하고, 실패 메시지에 키 원문을 포함하지 않는다. `api_key`가 비면 저장된 키를 사용한다.
10. 호출자 `sub` 또는 검증된 `email`이 `ADMIN_USER_IDS`에 없으면 모든 AdminService RPC는 `PermissionDenied`로 거부한다(미인증은 기존대로 `Unauthenticated`).
11. `ADMIN_USER_IDS`가 비면 검증된 관리자 포함 전원을 거부한다(fail-closed).
12. 이메일 매칭은 토큰의 `email_verified=true`일 때만 성립한다(미검증 토큰은 UUID 항목으로만 통과).
13. 비관리자가 `/admin`에 진입하면 NotFound 화면을 렌더한다(콘솔 무에러).
14. `GetAdminOverview`는 합계·잡 카운트·최근 30일 일기 시리즈·LLM usage를 반환한다(NO_SIDE_EFFECTS).
15. Resolver 경유 `Complete` 성공은 `llm_usage_daily`에 (UTC day, provider, model, kind='extract') 행을 upsert 누적하고, 싱크 실패는 호출을 실패시키지 않는다(로그만).
16. `auth.users`가 없는 DB(로컬 도커 pg)에서도 집계가 무에러로 동작한다(`to_regclass` 가드, 기본 users = `COUNT(DISTINCT user_id) FROM records`).
17. `GetLLMConfig`는 DB 행 유무와 무관하게 항상 매트릭스 5 공급자 카드를 반환한다(overrides-only 병합).
18. `UpdateProviderModels`는 모델 리스트를 통째 교체하되 `updated_at`(키 변경 시각)을 덮어쓰지 않는다.
