# 08. i18n seam

> i18n seam — Paraglide message generation and copy boundaries from the first real
> UI. As-built.
> Scope: FE | Status: built

## Purpose

The product's concept is bilingual, but app code and specs are English. This unit
adds the UI string seam before real screens multiply: messages are generated,
components call typed message functions, and raw user-facing strings stop appearing
inside feature UI. The goal is not full localization; it is making localization the
default path.

## Scope / Non-goals

**In scope**
- Paraglide/inlang setup compiled to platform-pure TypeScript usable by web and mobile.
- Message source structure for English and Korean.
- Generated message functions committed and reproducible from the repo root.
- App-level locale provider for **both web and mobile**, wired in Phase 1: web
  stored/route/navigator negotiation; mobile device-locale resolution composed into
  the mobile app shell (final order owned by unit 13). Localized product *copy* is
  later; the seam and the providers are now.
- A lint that catches raw user-facing strings in UI code, with allowlists for tests,
  logs, structural attributes, generated files, and domain model/proto/SQL terms.
- Initial messages for platform UI only: hello-world remnants and common primitive labels.

**Non-goals**
- No copywriting for final product flows.
- No marketing/blog localization; the Astro blog keeps its own content pipeline.
- No user preference settings screen.
- No server-side translation, AI translation, or database-stored localized content.
- No changes to canonical domain terms; [ubiquitous-language.md](../ubiquitous-language.md)
  remains the vocabulary authority.

## Design

### Package and app seam

The catalogue is a pure cross-app package; each app re-exports it through a thin
`shared/i18n` seam and negotiates locale in an app-layer provider:

```text
packages/i18n/
├── project.inlang/settings.json     baseLocale en · locales [en, ko] · message-format plugin
├── messages/{en,ko}.json            canonical message sources
└── src/
    ├── gen/**                        Paraglide output (committed; pnpm gen:messages)
    ├── locale.ts                     locale store + resolveLocale (drives Paraglide; platform-pure)
    └── index.ts                      facade: m, locale API, types

apps/{web,mobile}/src/
├── shared/i18n/index.ts             re-export m + useActiveLocale hook (useSyncExternalStore)
└── app/i18n-provider.tsx            platform locale negotiation + apply
```

`@cosimosi/i18n` carries no DOM/native deps, so `m.*` and the locale store run
verbatim on web, React Native, and in Node tests. Feature code imports message
functions and the reactive hook from its app's `shared/i18n`; the package is the
single source of message functions both apps share.

### Generation

`pnpm gen:messages` (part of `pnpm gen`) compiles `messages/*.json` →
`packages/i18n/src/gen/` with the `globalVariable` locale strategy (no DOM globals
in the runtime). Output is committed and freshness-checked by `pnpm check:gen`
(a second run is byte-identical). Consumers read the generated typed-JS via
`allowJs`. The inlang plugin cache is gitignored.

### Locale behavior

`src/locale.ts` owns an in-memory active locale and drives Paraglide through
`overwriteGetLocale`/`overwriteSetLocale`, so message functions read whatever locale
each app set. `defaultLocale` = `en`. `resolveLocale(candidates)` picks the first
candidate that matches a supported locale (exact → primary subtag, case-insensitive,
`-`/`_` separators) else the default; it is pure and tested without browser globals.

- Web precedence: stored choice (`localStorage`) → `navigator.languages` → default;
  negotiation runs in an effect, and the provider syncs `<html lang>`.
- Mobile precedence: device locale (RN `NativeModules`, best-effort) → default.

### Raw string rule

`scripts/lint-raw-strings.mjs` (`pnpm lint:raw-strings`) AST-scans `.tsx`/`.jsx`
under `apps/{web,mobile}/src` and fails on JSX text with letters and string literals
on user-facing attributes (`title`/`placeholder`/`alt`/`label`/`aria-*`/
`accessibility*`). Logs, thrown errors, route ids, structural attributes, `{m.x()}`,
and domain names in model/proto/SQL are out of scope; tests, generated code, and
stories are skipped by path; an `i18n-ignore` line opts a node out.
`pnpm lint:raw-strings:probe` self-tests catch-and-ignore.

### Interaction with design system

Design-system primitives accept labels, aria-labels, descriptions, and validation
messages as **props** (message-function results); they own no product copy. The
primitives arrive with unit 09; this rule binds them.

## Acceptance Criteria

1. ✅ Paraglide/inlang message generation is configured and reproducible from the repo
   root (`pnpm gen:messages` / `pnpm gen`; `pnpm check:gen` clean on re-run).
2. ✅ English and Korean message sources exist for platform-level UI copy
   (`packages/i18n/messages/{en,ko}.json`).
3. ✅ Web consumes typed message functions through a stable i18n facade
   (`apps/web/src/shared/i18n` → `@cosimosi/i18n`).
4. ✅ Mobile consumes the same typed message functions through its app-shell locale
   provider (device-locale resolution + default fallback), wired in Phase 1.
5. ✅ Locale resolution has a documented default (`en`) and is tested without browser
   globals (Node-env suites in `packages/i18n` and `apps/web`).
6. ✅ A raw-string check fails for obvious user-facing strings and ignores documented
   non-UI cases (`pnpm lint:raw-strings`, `:probe`).
7. ✅ Design-system primitives receive copy through props/message calls rather than
   embedding product strings (rule in [tech/i18n.md](../tech/i18n.md) §5; enforced by
   the raw-string lint).
8. ✅ No product content, glossary term changes, or server translation system is
   introduced.
