# 08. i18n seam

> i18n seam - Paraglide message generation and copy boundaries from the first real UI.
> Scope: FE | Status: planning

## Purpose

The product's concept is bilingual, but app code and specs are English. This unit adds the UI string seam before real
screens multiply: messages are generated, components call typed message functions, and raw user-facing strings stop
appearing inside feature UI. The goal is not full localization work; it is making localization the default path.

## Scope / Non-goals

**In scope**
- Paraglide/inlang setup for web and mobile-compatible TypeScript usage.
- Message source structure for at least English and Korean.
- Generated message functions committed or generated deterministically as the chosen tool requires.
- App-level locale provider for **both web and mobile**, wired in Phase 1: web route/storage negotiation; mobile
  device-locale resolution composed into the mobile app shell (unit 13). Localized product *copy* is later; the seam
  and the providers are now.
- Lint/check that catches raw user-facing strings in UI code, with allowlists for tests, logs, non-user-facing ARIA
  internals when appropriate, and generated files.
- Initial messages for platform UI only: hello-world remnants, auth/session proofing, test harness labels, and common
  design-system primitives.

**Non-goals**
- No copywriting for final product flows.
- No marketing/blog localization; the Astro blog can keep its own content pipeline.
- No user preference settings screen.
- No server-side translation, AI translation, or database-stored localized content.
- No changes to canonical domain terms; [ubiquitous-language.md](../ubiquitous-language.md) remains the vocabulary
  authority.

## Design

### Message ownership

Messages live in one canonical source tree, with generated TypeScript consumed through a facade:

```text
messages/
├── en.json
└── ko.json

apps/web/src/shared/i18n/
├── messages.gen.ts
├── locale.ts
└── index.ts
```

If the implementation chooses a package-level output for mobile reuse, the facade path may move to `packages/i18n`.
Feature code imports only the facade, not generator internals.

### Locale behavior

Web locale resolution:

1. explicit route or stored user choice when available;
2. browser language;
3. default locale.

Mobile locale resolution reads the device locale through an RN adapter, falling back to the default locale, and is
wired into the mobile app shell in Phase 1. The shared message facade stays platform-pure (no DOM/RN deps) so both
providers consume the same typed message functions verbatim.

### Raw string rule

User-facing copy in UI components must call message functions. Allowlisted raw strings include:

- test names and assertions;
- developer logs;
- route ids;
- non-user-facing constants;
- generated code;
- domain canonical names in model/proto/SQL contexts.

The check should bias toward catching drift in UI, not blocking legitimate domain code.

### Interaction with design system

Design-system primitives accept labels, aria labels, descriptions, and validation messages as props; they do not own
product copy. Shared primitive examples may use message calls or test-only placeholders.

## Acceptance Criteria

1. Paraglide/inlang message generation is configured and reproducible from the repo root.
2. English and Korean message sources exist for platform-level UI copy introduced by Phase 1.
3. Web consumes typed message functions through a stable i18n facade.
4. Mobile consumes the same typed message functions through its app-shell locale provider (device-locale resolution
   with a default fallback), wired in Phase 1.
5. Locale resolution has a documented default and can be tested without browser globals.
6. A raw-string check fails for obvious user-facing strings in UI components and ignores documented non-UI cases.
7. Design-system primitives receive copy through props/message calls rather than embedding product strings.
8. No product content, glossary term changes, or server translation system is introduced.
