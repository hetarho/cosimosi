# 41. consolidation usecase

> Consolidation use-case (우주의 잠 / sleep) — the backend orchestration that runs _during a clock advance_ and, over the
> advance interval `[from, to]`, does the sleeping brain's three jobs: advance the gist stages whose semanticize timer
> crossed ([C1][C6a]→[40]), **replay-reorganize** related memories ([C2]), and **Downscale** weak synapses (homeostatic
> SHY, [C4]). It binds the no-op `AdvanceProgression` hook [30] left open with a _real_ handler. No cron ([T4]).
> Scope: BE | Status: planning
>
> Layer U (use-case) — the second unit of Epic E, built after [40.semanticization](40.semanticization.md) (D) and
> before [42.gist-star-rendering](42.gist-star-rendering.md) (P). Home: `internal/memory` use-case `Consolidate` + a NEW
> pure domain fn `Downscale` (seated in the golden-parity `packages/memory-logic ↔ internal/memory` home next to
> `Depress` [18]). It **binds** [30.time-advance-usecase](30.time-advance-usecase.md)'s `AdvanceProgression` port, calls
> [40.semanticization](40.semanticization.md)'s `Semanticize`/timer model, and **activates** the reserved `consolidate`
> job kind on [22.ai-worker-pipeline](22.ai-worker-pipeline.md)'s worker. The visual choreography is
> [42.gist-star-rendering](42.gist-star-rendering.md) ([V8], placeholder).

## Purpose

Time in cosimosi does not tick — it _jumps_. Launch a star and the universe accelerates to that diary's date; recall a
star and it syncs to today ([T2]). Each jump is the product's "우주의 잠": the interval the clock skips is the sleep the
brain would have spent consolidating. This unit is what actually _happens_ during that sleep.

Three things must happen on an advance, and they must happen **because the clock moved, not on a schedule** ([T4]).
[30.time-advance-usecase](30.time-advance-usecase.md) already fires an `AdvanceProgression.OnAdvance(scope, from, to)`
hook inside every advance transaction, shipped wired to a **no-op** default precisely so Epic E can slot the real work
in without re-plumbing. This unit is that real work:

1. **Gist stages rise** — for every memory whose semanticize timer ([C6a]) crossed a stage boundary somewhere in
   `[from, to]`, advance `semantic_stage` (via [40.semanticization](40.semanticization.md)'s pure `Semanticize` model)
   so the neocortical gist version rises one layer ([C6]).
2. **Replay reorganizes constellations** — the sleeping brain replays waking activity, re-activating _related_ memories
   together (Wilson & McNaughton 1994). The large, companion re-layout that recall alone does not do is this stage's job
   ([C2]).
3. **Weak synapses Downscale** — homeostatic synaptic downscaling (SHY; Tononi & Cirelli 2014): a **global,
   proportional renormalization** that weakens weak edges to keep the graph from saturating. It is emphatically **not**
   LTD and **not** deletion ([I9][C4]) — the invariant the whole feature is built around.

The "no cron" contract ([T4]) is only honest if _every_ advance carries its own consolidation. This plan makes the
sleep real.

## Scope / Non-goals

**In scope**

- **`Consolidate(scope, from, to)`** — the use-case bound to [30.time-advance-usecase](30.time-advance-usecase.md)'s
  `AdvanceProgression` port; it replaces the Epic B no-op with the real handler and runs **inside the same advance
  transaction** (launch **and** sync-to-today both fire it). It orchestrates the state-producing stages below over the interval.
- **Stage advance ([C1][C6])** — for each of the user's memories, ask [40.semanticization](40.semanticization.md)'s
  timer model how many stage boundaries the interval `[from, to]` crosses (units/stage from `values.yaml`, anchored at
  `semanticize_timer_reset_at`), and advance `semantic_stage` by that many (clamped to the 4-stage ceiling; z-axis is
  one-way [C7]). The **stage texts already exist** (pregenerated at launch [C7]/[22]) — this stage flips which layer is
  _visible/materialized_, it does not call the LLM. The neocortical z-coords are [40]'s (copy x,y, raise z).
- **Forgetting-stage text persistence ([F1][R8a])** — over the same interval, detect newly crossed forgetting stages using
  [37.forgetting-decay](37.forgetting-decay.md)'s `DecayStage` function, call its deterministic `decayStageText` algorithm
  for each newly reached missing stage, and persist those strings into `episodic_memories.decay_stages`. This satisfies the
  PRD's "stored per-stage forgetting text" contract without making brightness/stage math a stored tick.
- **Replay reorganization ([C2])** — mark the interval's re-activated related memories so the force-sim performs the
  larger companion re-layout on the next read. Replay is scoped to **constellations touched in the interval** (memories
  sharing neurons with the ones whose stage advanced), not the whole universe.
- **`Downscale` — a NEW pure domain fn ([C4][I9])** — global/homeostatic proportional weakening of the user's synapses,
  seated in `packages/memory-logic ↔ internal/memory` (golden-parity, next to `Depress`). Distinct primitive from
  `Depress`/LTD: `Downscale(strength, factor) → strength'` scales _toward a floor_ by a global factor; weak edges lose
  proportionally more headroom, strong edges are preserved. It **never** removes an edge ([I1][C4]).
- **Enqueue `consolidate` / `semanticize` worker jobs ([22])** — the reserved `consolidate` job kind
  ([16.memory-aggregate-schema](16.memory-aggregate-schema.md)) is **activated here**. Heavy or LLM-touching work the
  interval implies (e.g. re-embedding after replay, generating a _not-yet-created_ remaining stage text if a
  reconsolidation left one pending) is enqueued, not run inline; the transaction stays fast. `semanticize` (already
  dispatched by [20]/[22]) is re-enqueued only when a remaining stage text is genuinely missing.
- **Idempotency over the interval** — `Consolidate` is a pure function of `[from, to]` + current state: re-running it for
  an already-consolidated interval advances no stage and downscales nothing new (the timer anchor + `semantic_stage`
  ceiling make it convergent). A `from == to` advance is a total no-op.
- **Per-user scoping** on every read, stage write, downscale, and enqueue (§4).

**Non-goals**

- **No cron, scheduler, or background sweep** ([T4]) — the _only_ trigger is the advance hook. This unit owns no timer
  thread; it computes what the interval `[from, to]` implies and writes it in the advance transaction.
- **No semanticize model, gist-stage texts, gist-timer definition, or z-coords** — those are
  [40.semanticization](40.semanticization.md) (`Semanticize`, the 4 stage texts, `semantic.gist_units_per_stage`, the
  neocortex z-band). This unit _consumes_ them to decide _when_ a stage rises; it does not define the abstraction model.
- **No LTD / associative depression** — `Depress` ([18.synapse-plasticity](18.synapse-plasticity.md)) is the local,
  associative primitive used by recall competition ([R]) and deletion ([H]). `Downscale` is a _different_ mechanism
  ([I9][C4]); this unit adds `Downscale` and does not touch `Depress`.
- **No forgetting decay / brightness dimming / word-loss math ownership** — the [F] axis is
  [37.forgetting-decay](37.forgetting-decay.md) (read-time), independent of consolidation ([C]/[F] are distinct axes,
  PRD §5.3.5). Consolidation compresses meaning; forgetting erases words. This unit only invokes [37]'s already-owned
  functions to persist newly reached `decay_stages` during the advance; it does not define the forgetting formulas.
- **No recall / reconsolidation** — `Recall`/`Reconsolidate` ([33.recall-usecase](33.recall-usecase.md)) reset the
  gist timer ([C6a]) and rewrite `current_text`; consolidation _reads_ the resulting anchor. This unit does not recall.
- **No consolidation choreography / rendering** — the "sequential twinkle + rise into the neocortex" animation ([V8][C8])
  is a **placeholder** in [42.gist-star-rendering](42.gist-star-rendering.md); this unit only produces the state
  (advanced stages, re-layout markers, downscaled synapses) it will one day play.
- **No clock model, `AdvanceClock`, or the hook's plumbing** — those are [29.universe-clock](29.universe-clock.md) /
  [30.time-advance-usecase](30.time-advance-usecase.md). This unit _binds_ the already-fired hook.
- **No schema/table definition** — it reuses [16.memory-aggregate-schema](16.memory-aggregate-schema.md)'s existing
  columns (`semantic_stage`, `semanticize_timer_reset_at`, `synapses.strength`) and the reserved `consolidate` job kind;
  it adds no column and no table.

## Design

### Placement (domain-first Go)

`Consolidate` is an application use-case in the core-domain `internal/memory` context, alongside `AdvanceOnLaunch` /
`SyncToToday` ([30]). It is **the concrete `AdvanceProgression` handler**: Epic B declared and fired the
`AdvanceProgression` port with a no-op default; this unit provides the real implementation and it is **bound at
`cmd/api`** replacing the no-op (ARCHITECTURE §2.4 — the handler is a concrete injected at the composition root, the port
stays consumer-owned by [30]). `Downscale` is a **pure IO-free domain function** — it lives in the golden-parity home
(`packages/memory-logic` TS ↔ `internal/memory` Go), next to `Potentiate`/`Depress`/`EffectiveSynapseStrength` ([18]).
Dependency direction:
`memory/rpc (nothing new) → advance tx → Consolidate (use-case) → pure domain (Downscale, [40]'s timer model)`, with the
memory/synapse repositories and the `JobQueue` reached only through consumer-owned interfaces.

**Decided (2026-07):** this unit adds **no RPC**. Consolidation is a _consequence_ of an advance, never a user action —
there is no "consolidate" button, exactly as [30]'s `SyncToToday` is a capability with no standalone RPC. It runs purely
inside the launch/sync transactions that already exist. (The visible result surfaces on the next `GetUniverse` read.)

### `Consolidate(scope, from, to)` — the advance handler ([T4])

Bound as `AdvanceProgression.OnAdvance`, it runs inside the same `InTx` as the advance ([30]). Given the interval:

1. **Guard the interval.** If `from == to` (no time passed — e.g. an idempotent same-day sync, or an equal-date launch),
   return immediately: no stage advances, nothing downscales. Consolidation only does work when the clock actually moved.
2. **Advance gist stages ([C1][C6]).** For each of the user's non-deleted memories, recompute the target gist stage at
   `now = to` using [40.semanticization](40.semanticization.md)'s gist-timer + `Semanticize` (this unit invents no new
   timer API): `unitsElapsed = floor(effectiveDays / semantic.gist_units_per_stage)` with the [F6][F7] arousal/
   connection-strength modulation (coefficients owned by [40]/[37]), then
   `semantic_stage = Semanticize(semantic_stage, unitsElapsed)` — **monotone** and **clamped to `SEMANTIC_MAX_STAGE = 4`**
   (z-axis is one-way [C7] — a stage never decrements here). The stage's **text already exists** in `semantic_stages` (pregenerated
   at launch [C7]/[22]); advancing the stage only changes which gist layer is materialized/visible, so **no LLM call**
   is made inline. Memories at the ceiling, or whose timer did not cross, are untouched.
3. **Persist newly reached forgetting-stage texts ([F1][R8a]).** For each non-deleted memory, compute the previous and
   target forgetting stages over `[from, to]` with [37.forgetting-decay](37.forgetting-decay.md)'s `DecayStage` function
   (using the same anchors and generated `forgetting.*` constants the read path uses). For every newly crossed stage whose
   `decay_stages[stage]` string is missing, call [37]'s deterministic `decayStageText(current_text, stage, seed)` and write
   the result into `episodic_memories.decay_stages`. This is a content write, not a tick: brightness and current stage stay
   read-time derived, and the handler never stores a "current brightness" or a pre-rendered current text.
4. **Replay-reorganize ([C2]).** Collect the **constellations touched** by the interval: the memories whose stage
   advanced in step 2, plus the memories sharing neurons with them. These are the "replayed" set. Mark them for the
   force-sim's larger companion re-layout on the next read (a read-time re-layout signal, **not** a stored coordinate
   [I5]) — this is the big co-reorganization recall alone does not perform. Replay is bounded to the touched
   constellations, never the whole universe, so a long interval does not re-layout everything at once.
   **Decided (2026-07):** replay is expressed as a **re-layout marker consumed at read** (the force-sim reseeds the
   touched cluster), not a persisted position, keeping [I5] intact. If the marker needs any heavy recompute (e.g.
   re-embedding a neuron after reorg), that is enqueued as a `consolidate` job (step 6), not run inline.
5. **Downscale weak synapses ([C4]).** Apply the pure `Downscale(strength, factor)` to the user's synapses as a
   **global, homeostatic renormalization** (SHY): weak edges lose proportionally more of their remaining strength, strong
   edges are preserved, and no edge is removed ([I1][C4]). The `factor` is a `values.yaml` coefficient; the write is a
   single scoped batch UPDATE in `memory/pg` (the sole sqlc/pgx seam, §2.6). This is **distinct from `Depress`/LTD**
   ([I9]) — `Depress` is associative and local (a specific pair weakened by competition); `Downscale` is global and
   proportional (every edge renormalized once per sleep). Both keep `strength` in `0..1`.
6. **Enqueue interval-implied heavy work ([22]).** The reserved **`consolidate`** job kind is activated: enqueue a
   `consolidate` job for any interval-implied work that must not block the advance transaction (re-embedding after
   replay reorg; a `semanticize` re-enqueue when a _remaining_ stage text is genuinely missing — e.g. a prior
   reconsolidation left a not-yet-generated stage per [C7]). Steps 2, 3, 4, and 5 are cheap and stay inline; step 6 keeps
   the transaction fast and lets the worker do the rest.

The whole handler stays inside the advance `InTx` (semantic-stage writes + decay-stage text writes + replay markers +
downscale batch + enqueue are atomic with the clock advance — a launch/sync lands wholly or not at all, [30] step 3).
Every read and write is user-scoped.

### `Downscale` — the new pure fn ([C4][I9], golden-parity)

`Downscale(strength, factor) → strength'` — homeostatic synaptic downscaling. Signed intent: scale each synapse's
stored base strength _down_ by a global `factor`, biased so **weak edges lose proportionally more headroom** while
strong edges are preserved (the "smart" renormalization SHY describes — sleep keeps the signal-carrying edges and prunes
the noise). It returns a value in `0..1` and **never returns 0-by-removal** — the floor is a small positive residual, so
a weak edge dims toward silence but is never deleted ([I1][F2] silent-engram spirit, at the synapse level).

It is a sibling of `Depress` but a **different mechanism** ([I9]): `Depress(strength, amount)` is associative/local
(subtract a competition-driven amount from _one_ pair); `Downscale(strength, factor)` is homeostatic/global (renormalize
_all_ edges once per sleep). Keeping them as two functions is the invariant [I9] demands, and this plan states the seam
[18] named ("`Depress` … the primitive that later epics call" — this is the later epic, adding the _other_ primitive).

**Golden-parity stance.** Like `Potentiate`/`Depress`/`EffectiveSynapseStrength` ([18]), `Downscale` is a pure function
the client renders against and the server computes, so it lives **once in Go (`internal/memory`) and once in TS
(`packages/memory-logic`)**, pinned equal by golden-parity fixtures. Its coefficients are the only part that leaves code,
into `values.yaml`. The _orchestration_ around it (`Consolidate`: which memories, which interval, the batch write, the
enqueue) is server-only use-case logic and is **not** golden-parity — only the `Downscale` math is mirrored. (Contrast
[40]'s timer/z math, partly golden-parity; [37]'s decay, fully golden-parity like [18].)

**Decided (2026-07):** the exact `Downscale` curve (the weak-edge bias exponent, the residual floor) is a **formula**,
so it stays in code/content and is **excluded from `values.yaml`**; only its tuning _scalars_ (the global factor, the
floor value, the weak-bias exponent) are enumerated as values keys — consistent with [18]'s `potentiation_rate` /
`strength_cap` treatment.

### Binding the `AdvanceProgression` hook ([30] → real handler)

[30.time-advance-usecase](30.time-advance-usecase.md) ships `AdvanceProgression.OnAdvance(scope, from, to)` with a no-op
default and documents Epic E as the binder. This unit provides the concrete handler (`Consolidate`) and swaps it in at
`cmd/api`. **Nothing in [30]'s contract changes** — the port signature, the call site (inside the advance tx on launch
**and** sync), and the `from == to ⇒ no-op` guarantee are all reused as-is. The only change is which implementation the
composition root binds. This is the "reserve the slot, fill it later without re-plumbing" discipline [30] set up.

Because the hook fires on **both** launch and sync-to-today, consolidation runs whenever the universe sleeps: a
forward-dated launch (advance to the diary date) and a recall's sync (advance to today) both consolidate over their
respective intervals. A past-dated launch (no advance, `from == to`) consolidates nothing, correctly.

### Interaction with the [F] axis and reconsolidation

[C] (this unit) and [F] ([37.forgetting-decay](37.forgetting-decay.md)) are **independent axes** (PRD §5.3.5): forgetting
is read-time (it just reads a later "now"), while consolidation is write-producing and must fire on the advance ([T4]) —
exactly the two-kind split [30] anticipated. Reconsolidation ([33.recall-usecase](33.recall-usecase.md)) resets
`semanticize_timer_reset_at` ([C6a]) and may leave a _remaining_ stage text ungenerated ([C7]); `Consolidate` reads the
reset anchor (so a recently-recalled memory's next stage is correctly _delayed_) and, if a remaining stage becomes due
while its text is missing, re-enqueues `semanticize` (step 6) rather than materializing an empty stage. Already-created
gist stars are never un-risen ([C7], z-axis one-way).

## Acceptance Criteria

1. `Consolidate(scope, from, to)` is bound as the concrete `AdvanceProgression.OnAdvance` handler and runs **inside the
   same advance transaction** as launch and sync-to-today; [30]'s port signature, call sites, and no-op-on-`from==to`
   guarantee are unchanged — only the composition-root binding changes from no-op to this handler. ([T4])
2. There is **no cron, scheduler, or background sweep** and **no new RPC** — the advance hook is the sole trigger, and
   consolidation is never a user action. ([T4])
3. For each memory whose semanticize timer crosses one or more stage boundaries in `[from, to]` (per [40]'s timer model
   anchored at `semanticize_timer_reset_at`), `semantic_stage` advances by the number of boundaries crossed, clamped to
   the 4-stage ceiling; a stage **never decrements** (z-axis one-way). No LLM call is made inline — the stage texts are
   already pregenerated ([C7]). ([C1][C6][C7])
4. Newly reached forgetting-stage texts are persisted during the same advance transaction: for every crossed stage whose
   `decay_stages[stage]` is missing, the handler writes [37]'s deterministic `decayStageText` output, while brightness and
   current stage remain read-time derived and no current brightness/text tick is stored. ([F1][R8a][T4])
5. Replay reorganization marks the **constellations touched in the interval** (stage-advanced memories + their
   neuron-sharing neighbors) for the force-sim's larger companion re-layout at read time; it stores **no coordinate**
   ([I5]) and is bounded to touched constellations, never the whole universe. ([C2])
6. `Downscale(strength, factor)` is a **new pure domain function** applied as a global, homeostatic, proportional
   renormalization of the user's synapses (weak edges lose proportionally more, strong edges preserved), returning a
   value in `0..1` with a **positive residual floor** — it **never removes** a synapse. ([C4][I1])
7. `Downscale` is **distinct from `Depress`/LTD** ([I9]): it is a separate pure function (global/homeostatic vs.
   associative/local), and this unit does not modify `Depress`. Both keep `strength` in `0..1`.
8. `Downscale` lives **once in Go and once in TS** (golden-parity, next to [18]'s synapse fns) and is pinned equal by
   golden-parity fixtures; the surrounding `Consolidate` orchestration is server-only and is not mirrored.
9. The reserved **`consolidate`** job kind is **activated**: interval-implied heavy or LLM-touching work (re-embedding
   after reorg; a `semanticize` re-enqueue when a _remaining_ stage text is genuinely missing per [C7]) is enqueued on
   the worker, not run inline; the advance transaction stays fast. ([C7], §2.8)
10. `Consolidate` is **idempotent / convergent over the interval**: re-running it for an already-consolidated interval
    advances no stage and downscales nothing further (timer anchor + stage ceiling), and `from == to` is a total no-op.
11. The Diary is never touched, no synapse/neuron/memory is deleted, and no position is stored — consolidation changes
    only representation (`semantic_stage`, `decay_stages`, synapse `strength`) plus a read-time re-layout marker. ([I1][I2][I5])
12. Every read, stage write, decay-stage text write, downscale batch, and enqueue is **scoped to the authenticated user**, and
    `pnpm lint:persistence` passes (§4).
