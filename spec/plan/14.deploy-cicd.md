# 14. 배포 CI/CD (develop→스테이징 · main→프로덕션 자동 배포)

> 머지 → 3개 배포 대상(Cloudflare Pages · Hetzner VPS · Supabase)에 자동 배포되는 파이프라인 구축. | 범위: MVP 이후(실배포) | 선행: 13(로컬 한 바퀴 완성), 03(goose 채택) | 차단: —

## 목적 (Why)
13까지는 `pnpm dev`로 **로컬에서** 한 바퀴가 도는 상태다. 이 스펙은 그 산출물을 **실제 인터넷에 올리고**, `develop`/`main` 브랜치 머지만으로 손 안 대고 배포되게 만든다. Architecture §7은 인프라 **선택**(Cloudflare/Hetzner/Supabase)만 기록했고 배포 배관은 전무하다 — 이 스펙이 그 공백(프로덕션 Dockerfile·prod compose·Actions 워크플로·마이그레이션 게이트·시크릿·롤백)을 메운다.

## 범위
- **포함:**
  - 백엔드 **프로덕션 Dockerfile**(멀티스테이지 정적 빌드, dev의 air 핫리로드와 분리).
  - **`docker-compose.prod.yml`**(로컬 postgres·MinIO 제거 → Supabase 외부 사용, **Caddy 리버스 프록시**로 자동 HTTPS).
  - **GitHub Actions 워크플로**: ① PR CI(lint/build 게이트) ② 백엔드 배포(이미지 빌드→GHCR push→SSH로 `docker compose pull && docker compose up -d`) ③ goose 마이그레이션 스텝(백엔드 롤아웃 **직전** 실행).
  - **Cloudflare Pages**: 네이티브 Git 연동 + **Pages 프로젝트 2개**(staging=develop, prod=main)로 환경 분리.
  - **브랜치→환경 매핑**(develop→스테이징, main→프로덕션)과 **경로 필터**(프론트만 바뀌면 백엔드 미배포).
  - 시크릿 인벤토리, **롤백** 절차, 1회성 VPS 부트스트랩 체크리스트.
- **비목표(Out of scope):**
  - Hetzner VPS 프로비저닝·Supabase 프로젝트 생성·Cloudflare 도메인 연결 **실제 클릭 작업** — 본 스펙은 **수행 가능한 절차/명령**까지 제공하되 계정 생성·결제는 사용자가 직접(체크리스트로 안내).
  - 무중단/블루-그린·오토스케일·다중 리전 — MVP는 `compose up -d` 재생성(수초 다운타임 허용).
  - 시크릿 매니저(Vault/Doppler) — GitHub Actions Secrets + VPS의 비추적 `.env`로 충분.
  - 프론트 **GitHub Actions + wrangler** 경로 — 본 스펙은 CF 네이티브 Git 연동을 채택(§설계 6). 통합 게이팅이 필요해지면 그때 전환.
  - CI 내 자동 E2E(Playwright 등) — 13의 수동 체크리스트로 갈음.

## 설계 요점

**(1) 브랜치 → 환경 매핑 (단일 원천).**
| 브랜치 | 환경 | 프론트(CF Pages) | 백엔드(Hetzner) | DB(Supabase) |
|---|---|---|---|---|
| `develop` | **staging** | 프로젝트 `cosimosi-staging` | compose 스택 `/srv/cosimosi-staging` (api.staging.<도메인>) | staging 프로젝트/DB |
| `main` | **production** | 프로젝트 `cosimosi-prod` | compose 스택 `/srv/cosimosi-prod` (api.<도메인>) | prod 프로젝트/DB |

- 한 Hetzner VPS에 **두 compose 스택**을 디렉터리·포트·서브도메인으로 분리(별도 VPS 불필요). Caddy가 두 서브도메인의 TLS를 자동 발급.
- staging/prod는 **서로 다른 Supabase DB**(또는 최소한 별도 스키마)를 본다 → 프로덕션 데이터에 staging 마이그레이션이 닿지 않게.

**(2) 3개 배포 다리(leg)와 순서.**
```
머지(develop|main)
 ├─[프론트] CF가 자체 감지 → 빌드 → 해당 Pages 프로젝트 배포   (GitHub Actions 무관, 병렬)
 └─[백엔드] GitHub Actions:
       lint/build 게이트 → 이미지 빌드 → GHCR push
         → SSH: 마이그레이션 파일·.env 동기화(부트스트랩, 아래 6·8)
         → SSH: goose up (마이그레이션, 실패 시 중단)   ← 백엔드 교체 "전"
         → SSH: IMAGE_TAG=<sha>를 스택 .env에 기록 → docker compose pull && docker compose up -d (api 교체)
         → SSH: docker image prune
```
- **마이그레이션은 항상 백엔드 롤아웃 직전**(되돌리기 어려운 가장 위험한 다리 — 먼저·게이트). 실패하면 api 컨테이너를 바꾸지 않는다.
- 프론트와 백엔드는 **독립**(경로 필터). 프론트만 바뀐 PR이 백엔드 배포를 트리거하지 않게 `paths:` / `paths-ignore:`.

**(3) 백엔드 프로덕션 Dockerfile (멀티스테이지·정적).**
- builder: `golang:1.26-alpine`에서 `CGO_ENABLED=0 GOOS=linux go build -o /api ./cmd/api`(connect-go는 Go 1.25+ 요구 — 충족).
- runtime: `gcr.io/distroless/static` 또는 `alpine`(ca-certificates 포함)에 정적 바이너리만 복사. 비루트 유저. `air`/소스 미포함(dev와 완전 분리).
- MVP는 워커가 **같은 바이너리 고루틴**(§4.6)이라 단일 이미지·단일 서비스. `cmd/worker`가 분리되면 같은 Dockerfile에 두 번째 타깃 추가.

**(4) `docker-compose.prod.yml` (dev compose와 별개 파일).**
- 서비스: `api`(GHCR 이미지, `env_file: .env`, `restart: unless-stopped`) + `caddy`(80/443, 자동 HTTPS, api로 리버스 프록시).
- **로컬 postgres·MinIO 없음** — `DATABASE_URL`은 Supabase를 가리킴(§7). MinIO는 §7대로 MVP 미사용.
- Connect 서버는 `http.Handler`이므로 Caddy → api(:8080, h2c) 프록시. CORS는 api가 `connectrpc.com/cors`로 처리하되 허용 오리진은 `CORS_ORIGIN`(= 해당 환경 CF Pages 도메인).
- 이미지 태그는 `${IMAGE_TAG}`로 주입(기본 커밋 SHA) → `docker compose pull && docker compose up -d`가 정확한 버전을 받음. 롤백 = `IMAGE_TAG`를 이전 SHA로 두고 재실행.
- ⚠️ **IMAGE_TAG 전파**: `IMAGE_TAG=$SHA docker compose pull && docker compose up -d` 같은 **인라인 변수 할당은 첫 명령(pull)에만 적용되고 `up`까지 가지 않는다**. 그러므로 IMAGE_TAG는 스택 디렉터리의 **`.env`에 기록**(compose가 `${IMAGE_TAG}` 보간 시 `.env` 자동 로드)하거나 두 명령 모두에 export한 환경으로 전파한다. 표준: 배포 스텝이 `.env`의 `IMAGE_TAG=<sha>` 줄을 갱신한 뒤 `docker compose pull`·`docker compose up -d`를 각각 실행.

**(5) GitHub Actions 워크플로.**
- `.github/workflows/ci.yml` — `on: pull_request`. 프론트 `pnpm install --frozen-lockfile` → `pnpm --filter ./frontend lint`·`build`. 백엔드 `go vet`·`go build ./...`(Docker 불필요, 러너는 Linux라 §8 Windows .exe 제약 무관). **머지 전 게이트.**
- `.github/workflows/deploy-backend.yml` — `on: push: branches: [develop, main]` + `paths: ['backend/**','docker-compose.prod.yml','.github/workflows/deploy-backend.yml']`.
  - `docker/build-push-action`으로 `backend/Dockerfile` 빌드 → `ghcr.io/hetarho/cosimosi-api:<sha>` 및 `:<env>` 태그 push(GHCR 인증은 `GITHUB_TOKEN` 자동).
  - 환경 결정: `main`→prod, `develop`→staging(`github.ref_name`으로 분기, GitHub **Environments**로 prod에 수동 승인 게이트 옵션).
  - `appleboy/ssh-action`으로 VPS 접속 → 해당 스택 디렉터리에서:
    `docker login ghcr.io` → **마이그레이션 파일 동기화**(아래 6a) → `goose up`(아래 6) → **`.env`에 `IMAGE_TAG=<sha>` 기록**(아래 6b) → `docker compose -f docker-compose.prod.yml pull && docker compose -f docker-compose.prod.yml up -d` → `docker image prune -f`.
  - ⚠️ `IMAGE_TAG=<sha> docker compose ... pull && up -d`처럼 **인라인 할당하지 않는다**(첫 명령에만 붙고 `up`엔 안 감 — 위 (4)). IMAGE_TAG는 스택 `.env`에 기록해 compose가 `${IMAGE_TAG}`를 양쪽 명령에서 동일하게 보간하게 한다.

**(6) 마이그레이션 게이트 (goose, Supabase).**
- **(6a) 마이그레이션 파일 부트스트랩(VPS 배치).** goose `docker run`은 호스트의 `migrations/`를 `-v .../migrations:/m`로 마운트하므로 **그 파일들이 먼저 VPS에 있어야 한다**. 03이 산출한 `backend/migrations/*.sql`을 매 배포 시 스택 디렉터리(`/srv/cosimosi-<env>/migrations`)로 동기화한다 — `rsync`/`scp`(예: `appleboy/scp-action`으로 `backend/migrations/* → /srv/cosimosi-<env>/migrations/`) 또는 VPS에서 `git pull`. 동기화를 goose 실행 **이전**에 배치(파일이 곧 적용될 버전과 일치하도록).
- 03이 채택한 goose로, 배포 스텝에서 **백엔드 교체 직전** 실행:
  `docker run --rm -v /srv/cosimosi-<env>/migrations:/m -e GOOSE_DRIVER=postgres -e GOOSE_DBSTRING="$DIRECT_DATABASE_URL" ghcr.io/pressly/goose:3 -dir /m up`
- ⚠️ **Supabase 풀러 함정**: DDL/마이그레이션은 **트랜잭션 풀러(6543)가 아니라 직접 연결(5432)** 로 실행해야 한다 → 마이그레이션용 `DIRECT_DATABASE_URL`(5432)과 앱 런타임용 `DATABASE_URL`(6543 풀러)를 **분리**한다.
- 실패 시 `set -e`로 즉시 중단 → api 교체 안 함(이전 버전 유지).
- **(6b) IMAGE_TAG `.env` 기록.** goose 성공 후 compose 전에, 배포 스텝이 스택 `.env`의 `IMAGE_TAG` 줄을 이번 SHA로 갱신한다(예: `sed`로 치환 또는 append). compose가 `.env`를 자동 로드해 `pull`/`up` 양쪽에서 `${IMAGE_TAG}`를 동일하게 보간 — 인라인 할당의 `up` 미전파 문제(위 4·5)를 회피.

**(7) 프론트 = Cloudflare Pages 네이티브 Git 연동(프로젝트 2개).**
- 대시보드에서 리포 연결. **프로젝트별 설정**: 빌드 명령 `pnpm --filter ./frontend build`, 출력 `frontend/dist`, root는 리포 루트, 모노레포 **빌드 watch path** = `frontend/`(백엔드만 바뀌면 빌드 스킵).
- `cosimosi-staging`: 프로덕션 브랜치=`develop`, env `VITE_API_URL=https://api.staging.<도메인>`.
- `cosimosi-prod`: 프로덕션 브랜치=`main`, env `VITE_API_URL=https://api.<도메인>`.
- ⚠️ `VITE_*`는 **빌드 타임 주입** → 환경마다 별도 빌드(프로젝트 2개로 자연 분리). `VITE_SUPABASE_*`도 환경별 값.
- 롤백: CF 대시보드에서 이전 배포로 즉시 롤백.

**(8) 시크릿 인벤토리.**
- **GitHub Actions Secrets**: `SSH_HOST`/`SSH_USER`/`SSH_KEY`(VPS 배포 키), (prod/staging별) `DIRECT_DATABASE_URL`(마이그레이션용 5432). GHCR는 `GITHUB_TOKEN` 자동이라 불필요.
- **VPS의 비추적 `.env`**(스택 디렉터리, git 밖): `DATABASE_URL`(6543 풀러), `CORS_ORIGIN`, `AI_EMBEDDER`/`OPENAI_API_KEY`, `PORT`, `IMAGE_TAG`(배포 스텝이 SHA로 갱신 — (6b)), `SENTRY_DSN`/`SENTRY_ENVIRONMENT`((9), 비어 있으면 비활성).
- **Cloudflare Pages env**(대시보드): `VITE_API_URL`, `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, `VITE_SENTRY_DSN`((9), 비어 있으면 비활성).
- 원칙: 시크릿은 **리포에 절대 커밋 금지**(`.env`는 이미 .gitignore). 예시는 `.env.production.example`로만.

**(9) 관측가능성 — Sentry init (BE/FE).**
- 02 rpcserver의 slog 구조화 로깅(인터셉터)에 더해, **프로덕션 에러 트래킹**을 위해 BE/FE 각각 Sentry를 초기화한다(MVP는 에러 캡처만, 트레이싱/세션 리플레이는 v1+).
- **BE**: `cmd/api` 기동 시 `sentry-go`로 `sentry.Init(ClientOptions{ Dsn, Environment })`(`SENTRY_DSN`·`SENTRY_ENVIRONMENT` env). connect 인터셉터/`defer sentry.Recover()`로 패닉·핸들러 에러 캡처. DSN 미설정이면 no-op(로컬/13 영향 없음).
- **FE**: 엔트리(`frontend/src/app`)에서 `@sentry/react` `Sentry.init({ dsn: VITE_SENTRY_DSN, environment })`. DSN은 빌드 타임 `VITE_SENTRY_DSN`(CF Pages env, 환경별). 미설정이면 init 스킵.
- env 추가: BE `.env`에 `SENTRY_DSN`·`SENTRY_ENVIRONMENT`(staging/production), CF Pages env에 `VITE_SENTRY_DSN`. 둘 다 비어 있으면 비활성(시크릿 부재 시 빌드/기동 깨지지 않음).

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 개발자가 `develop`에 머지하면 THE SYSTEM SHALL 프론트를 `cosimosi-staging`에, 백엔드를 staging compose 스택에 자동 배포한다.
1.2 WHEN 개발자가 `main`에 머지하면 THE SYSTEM SHALL 프론트를 `cosimosi-prod`에, 백엔드를 prod compose 스택에 자동 배포한다.
2.1 WHEN PR이 열리면 THE SYSTEM SHALL 프론트 `lint`+`build`와 백엔드 `go build`를 실행하고, 실패 시 머지 게이트를 막는다.
2.2 WHEN 백엔드 변경 없이 프론트 파일만 바뀐 PR이 머지되면 THE SYSTEM SHALL 백엔드 배포 워크플로를 트리거하지 않는다(경로 필터).
3.1 WHEN 백엔드 배포가 실행되면 THE SYSTEM SHALL 멀티스테이지 정적 이미지를 빌드해 `ghcr.io/hetarho/cosimosi-api:<sha>`로 push한다.
3.2 WHEN 배포 스텝이 VPS에서 실행되면 THE SYSTEM SHALL `goose up`을 백엔드 컨테이너 교체 **이전**에 실행하고, 마이그레이션 실패 시 api 컨테이너를 교체하지 않는다.
3.3 IF 마이그레이션이 Supabase에 적용되면 THEN THE SYSTEM SHALL 트랜잭션 풀러(6543)가 아닌 직접 연결(5432, `DIRECT_DATABASE_URL`)을 사용한다.
4.1 WHEN 프로덕션 백엔드가 기동하면 THE SYSTEM SHALL Caddy를 통해 유효한 HTTPS로 `api.<도메인>`을 서비스하고, `CORS_ORIGIN`에 해당 환경 CF Pages 도메인을 허용한다.
5.1 WHEN 프로덕션 프론트가 빌드되면 THE SYSTEM SHALL prod `VITE_API_URL`이 박힌 번들을 `cosimosi-prod`에 배포한다(staging URL이 prod 번들에 섞이지 않는다).
6.1 THE SYSTEM SHALL 어떤 시크릿(SSH 키·DB URL·API 키)도 리포에 커밋하지 않는다(`.env.production.example`은 값 없는 키만).
6.2 WHEN 직전 배포가 문제가 되면 THE SYSTEM SHALL `IMAGE_TAG`를 이전 SHA로 재배포(백엔드)하거나 CF 대시보드 롤백(프론트)으로 되돌릴 수 있다.

## 완료 정의 (DoD)
- `docker build -f backend/Dockerfile ./backend` 로컬 성공(정적 바이너리 이미지).
- PR 생성 시 `ci.yml`이 프론트 lint/build + 백엔드 build를 실행하고, 실패가 머지를 막는다(수용 2.1).
- `develop` 머지 → CF `cosimosi-staging` 배포 + staging compose 스택 갱신이 **사람 개입 없이** 완료(수용 1.1). `main` 동일(prod)(수용 1.2).
- 백엔드 배포 로그에서 마이그레이션 파일이 VPS `/srv/cosimosi-<env>/migrations`로 동기화된 뒤 `goose up`이 api 교체 **이전**에 실행되고, 마이그레이션 강제 실패 시 api가 이전 버전으로 유지됨을 확인(수용 3.2).
- 배포 후 VPS 스택 `.env`의 `IMAGE_TAG`가 이번 SHA로 갱신되고 `docker compose pull && docker compose up -d`(인라인 할당 아님)로 실행된 api 컨테이너 이미지 태그가 그 SHA와 일치함을 확인(수용 6.2).
- BE/FE Sentry가 환경별 DSN으로 초기화되어 프로덕션 에러가 Sentry에 도달하고, DSN 미설정 환경에서는 빌드/기동이 깨지지 않음을 확인((9)).
- `https://api.<도메인>/`(헬스 또는 RPC)가 유효 TLS로 응답하고, prod CF 도메인에서의 브라우저 요청이 CORS 통과(수용 4.1).
- prod 번들에 staging API URL이 없음(빌드 산출물/네트워크 탭 확인)(수용 5.1).
- 리포에 시크릿 0건: `git grep`으로 SSH 키·DB URL·`sk-` 0건, `.env.production.example`은 값 없는 키만(수용 6.1).
- 롤백 리허설: 백엔드 `IMAGE_TAG`를 직전 SHA로 재배포 1회, CF 대시보드 롤백 1회 성공(수용 6.2).
- `spec/plan/00.overview.md`에 14가 등재되고 진행도에 반영됨.
