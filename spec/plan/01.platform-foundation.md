## Purpose

cosimosi needs a stable product platform before the engram-domain roadmap can be rebuilt safely. The platform must make the repository shape, development loop, generated contracts, Dockerized backend, and deployment gates obvious from the filesystem and root commands, so future domain plans can change product behavior without re-litigating infrastructure.

The foundation also preserves the MVP's working code while the roadmap is re-authored from [concept.md](../concept.md). This plan captures the current monorepo and toolchain as the baseline that later plans depend on: web, blog, API, proto, generated values, local Postgres, migrations, CI, and backend deploy all have stable entry points.

## Scope / Non-goals

In scope:

- Root monorepo layout and ownership boundaries: `apps/`, `packages/`, `proto/`, `scripts/`, `spec/`, Docker/Compose, and GitHub workflows.
- pnpm workspace configuration for JavaScript apps and future shared packages.
- Vite/React/TypeScript/Tailwind web application foundation, including the root `.env` contract and local proxy.
- Astro blog workspace package and static build entry points.
- Go API application layout, local hot-reload container, production image, and build entry point.
- Local development bootstrap: `.env`, dependencies, Postgres, migrations, codegen, and combined dev server.
- Code generation pipeline: `spec/values.yaml` to TS/Go constants, proto to Go/TS Connect code, and sqlc to Go DB accessors.
- Local DB migration runner and local Docker Compose services.
- CI and backend deploy workflow wiring at the application-boundary level.
- Placeholder boundaries for future `apps/mobile` and `packages/*`.

Non-goals:

- No product behavior, UI flow, RPC method, proto field, DB schema, migration, auth policy, AI provider behavior, or engram-domain rule changes.
- No mobile app implementation. `apps/mobile` is only a placeholder.
- No new shared packages. `packages/*` stays promote-on-reuse.
- No migration from build-time values to runtime/admin-managed configuration.
- No production VPS, Cloudflare, Supabase, or secret provisioning. The deploy workflow can run when its external variables and secrets are configured.
- No implementation job is required for this capture unit unless a later audit finds drift.

## Design

### Repository Boundaries

The repository is a monorepo with explicit app and shared-code boundaries:

```text
cosimosi/
  apps/
    web/      # Vite + React product web app, FSD source tree
    api/      # Go Connect API app, not a pnpm package
    blog/     # Astro static blog package
    mobile/   # future mobile placeholder only
  packages/   # future shared packages, promote-on-reuse
  proto/      # root Protobuf/Connect transport contract
  scripts/    # Node DX wrappers for setup, codegen, DB, spec scaffolds
  spec/       # concept, roadmap plans, values, archived MVP specs
```

`pnpm-workspace.yaml` includes `apps/web`, `apps/blog`, and `packages/*`. `apps/api` remains a Go module (`github.com/cosimosi/backend`) and is intentionally outside the pnpm workspace package list. App code must not import another app's `src` tree directly; shared code is promoted to `packages/*` only after a second real consumer exists, or shared through the root `proto/` transport contract.

`apps/mobile` is a tracked placeholder documenting the future React Native track, renderer decision point, and app-sharing rules. `packages/README.md` defines the promote-on-reuse policy and candidate package families (`core`, `rendering`, `ui`), but no real shared package exists yet.

### Root Commands and Local Development

The root `package.json` is the operational entry point:

- `pnpm setup` runs the one-shot bootstrap: copy `.env.example` to `.env` if missing, install workspace dependencies, start local Postgres, wait for health, run migrations, then run codegen.
- `pnpm dev` runs `@cosimosi/web` Vite dev and the Docker Compose API service concurrently.
- `pnpm dev:web`, `pnpm dev:api`, and `pnpm dev:blog` run individual apps.
- `pnpm infra:up` / `pnpm infra:down` control local Compose infrastructure.
- `pnpm db:migrate`, `pnpm db:status`, `pnpm db:down`, and `pnpm db:reset` call the Dockerized goose wrapper.
- `pnpm gen`, `pnpm gen:values`, `pnpm gen:proto`, and `pnpm gen:sql` are the canonical generation entry points.
- `pnpm build:web`, `pnpm build:blog`, and `pnpm build:api` build the three shipped apps.

The Node scripts are cross-shell wrappers. They avoid depending on host-installed `buf`, `sqlc`, `goose`, or shell-specific quoting. Docker mounts are resolved relative to the repo root, and generated/migration steps skip cleanly if their source config is not present.

### Web App Foundation

`apps/web` is the browser product app:

- Package: `@cosimosi/web`.
- Stack: Vite, React, TypeScript, Tailwind, Vitest, ESLint, Steiger/FSD, R3F/three.
- Dev server: strict port `1214`.
- Env source: Vite `envDir` is the repo root, so `.env` is shared by FE and BE.
- Proxy: `/api` strips the prefix and forwards to `VITE_API_URL` (default `http://localhost:8080`); `/health` proxies to the same backend.
- Path alias: `@` resolves to `apps/web/src`.
- FSD source tree: `app`, `pages`, `widgets`, `features`, `entities`, `shared`.
- Generated TS transport types live under `apps/web/src/shared/api/gen` and are ignored by hand-authored lint/FSD checks where appropriate.

Web build is `tsc -b && vite build`. Linting is split between ESLint (`pnpm --filter @cosimosi/web lint`) and FSD structure (`pnpm --filter @cosimosi/web lint:fsd`).

### Blog Foundation

`apps/blog` is the static content app:

- Package: `@cosimosi/blog`.
- Stack: Astro.
- Commands: `dev`, `build`, and `preview`.
- Astro telemetry is disabled in dev/build/preview scripts, and dev uses `ASTRO_DEV_BACKGROUND=1`.

The blog is part of the pnpm workspace and CI build gate. It is structurally separate from the product web app.

### API Foundation

`apps/api` is the product API app:

- Module: `github.com/cosimosi/backend`.
- Stack: Go 1.26, Connect, pgx/v5, sqlc, pgvector, Sentry, Supabase JWT verification.
- Layout: `cmd/api` composition root plus `internal/*` packages organized by feature and platform.
- Generated proto code lives in `apps/api/internal/gen`.
- Generated sqlc code lives in `apps/api/internal/db/gen`.
- Generated values live in `apps/api/internal/values/values_gen.go`.

Local API dev runs in Docker via `apps/api/Dockerfile.dev`, installs Air, mounts the source, exposes `8080`, and hot-reloads with polling enabled for bind-mounted filesystems. Production API uses `apps/api/Dockerfile`, a multi-stage static build, and a distroless non-root runtime image.

### Local Docker Compose

`docker-compose.yml` defines the local dev stack:

- `postgres`: `pgvector/pgvector:pg16`, port `5432`, persistent `postgres_data`, and a `pg_isready` healthcheck.
- `backend`: dev profile service built from `apps/api/Dockerfile.dev`, depends on healthy Postgres, exposes `8080`, mounts `apps/api`, and receives backend, auth, AI, admin, and invite-gate env from Compose/root `.env`.

The frontend is not containerized in local dev; it runs through pnpm/Vite and proxies API calls to the backend.

### Code Generation

`pnpm gen` is ordered as:

1. `scripts/gen-values.mjs`
2. `scripts/gen.mjs`

`gen-values` reads `spec/values.yaml` as the build-time, git-versioned source for tuning/config values and writes:

- `apps/web/src/shared/config/values.gen.ts`
- `apps/api/internal/values/values_gen.go`

The current generator supports finite numeric scalars, numeric arrays, strings, and one-level scalar maps used by existing config groups. Generated files are committed and marked "DO NOT EDIT".

`gen:proto` uses Dockerized `bufbuild/buf:latest` with root `buf.yaml` and `proto/buf.gen.yaml`. Remote plugins emit:

- Go protobuf messages to `apps/api/internal/gen`.
- Go Connect handlers/clients to `apps/api/internal/gen`.
- TypeScript protobuf/Connect service descriptors to `apps/web/src/shared/api/gen`.

`gen:sql` uses Dockerized `sqlc/sqlc:latest` with `apps/api/sqlc.yaml`. sqlc reads `apps/api/internal/db/schema.sql` and `apps/api/internal/db/queries`, then writes Go pgx/v5 accessors under `apps/api/internal/db/gen`. The schema input is a flattened snapshot rather than the goose migration directory, so sqlc does not parse down migrations.

### Migrations

DB migrations are executed by `scripts/db.mjs`, which runs a one-shot Docker container (`ghcr.io/kukymbr/goose-docker:3.27.1`) on the stable `cosimosi_default` Compose network. Migrations are mounted from `apps/api/internal/db/migrations`, use `postgres` as the network host, and support `up`, `status`, `down`, and `reset`.

The runner skips when `apps/api/internal/db/schema.sql` is absent. In the current repo, the schema and migrations exist.

### CI and Deploy Foundation

`.github/workflows/ci.yml` runs on PRs to `develop`/`main` and pushes to `develop`/`main`:

- `web`: install pnpm workspace dependencies, run web lint, run web build.
- `blog`: install pnpm workspace dependencies, run blog build.
- `api`: in `apps/api`, run `go vet ./...` and `go build ./...` using the Go version from `apps/api/go.mod`.

`.github/workflows/deploy-backend.yml` runs on backend-relevant pushes and manual dispatch:

- Builds the production API image from `apps/api/Dockerfile`.
- Pushes to GHCR only when `DEPLOY_ENABLED=true`.
- Rolls out only when deploy is enabled, with per-branch environment selection (`develop` to staging, `main` to production).
- Syncs migration files from `apps/api/internal/db/migrations`, runs goose against the direct DB URL, records `IMAGE_TAG` in the stack `.env`, and updates the Compose stack.

`docker-compose.prod.yml` runs the GHCR API image on the external `edge` network, and `deploy/edge/docker-compose.yml` defines the shared Caddy edge service that owns ports `80`/`443`.

### Environment Contract

The root `.env.example` is the template for local `.env`. It includes backend values (`PORT`, `DATABASE_URL`, Supabase JWT verification, AI embedder, admin keys, invite gate) and browser-exposed Vite values (`VITE_API_URL`, Supabase URL/key, Sentry/PostHog values, app version).

Secrets stay out of the repo. Vite reads the root `.env` by configuration, while Docker Compose interpolates the same file for the backend service.

## Acceptance Criteria

1. The repo has explicit app boundaries under `apps/web`, `apps/api`, `apps/blog`, and `apps/mobile`, with `proto/`, `packages/`, `scripts/`, and `spec/` at the root.
2. `pnpm-workspace.yaml` includes `apps/web`, `apps/blog`, and `packages/*`; `apps/api` remains a separate Go module and not a pnpm package.
3. `packages/README.md` documents promote-on-reuse and no real shared package is created before a second consumer exists.
4. Root scripts expose setup, dev, infra, DB, codegen, and per-app build entry points without requiring callers to know app-local paths.
5. `pnpm setup` is idempotent: it creates `.env` when missing, installs dependencies, starts Postgres, waits for health, runs migrations, and runs codegen.
6. Local dev uses Vite on strict port `1214`, the Dockerized API on `8080`, and local Postgres on `5432`.
7. `apps/web/vite.config.ts` reads the root `.env`, defines the `@` alias, and proxies `/api` and `/health` to `VITE_API_URL`.
8. `apps/api/Dockerfile.dev` runs the API with Air hot reload, while `apps/api/Dockerfile` builds a static production binary and runs it in a distroless non-root image.
9. `pnpm gen` emits generated values to both web and API before running proto/sql generation.
10. `gen:proto` uses root `buf.yaml` plus `proto/buf.gen.yaml` and outputs Go/Connect/TS generated code to the app-specific generated directories.
11. `gen:sql` uses `apps/api/sqlc.yaml`, `internal/db/schema.sql`, and `internal/db/queries` to generate pgx/v5 sqlc code.
12. DB migration commands run goose in Docker against the local Compose network and use migrations from `apps/api/internal/db/migrations`.
13. CI gates web lint/build, blog build, and API vet/build on PRs and pushes to `develop`/`main`.
14. Backend deploy builds the API image from `apps/api`, syncs migrations from `apps/api/internal/db/migrations`, runs migrations before rollout, and updates the Compose stack by image SHA when deploy is enabled.
15. This platform foundation does not change product behavior, transport fields, DB schema, auth semantics, or engram-domain rules.

## Policy / Values Impact

- No new product policy or UX rule is introduced by this capture plan.
- No new tuning/config value is introduced. `spec/values.yaml` already remains the build-time source for generated FE/BE constants.
- Platform rule captured for future plans: app-to-app source imports are forbidden; shared code must move through `packages/*` after real reuse or through the root `proto/` contract.
- Platform rule captured for future plans: generated transport, DB, and values files are edited only by their source files and generators.
