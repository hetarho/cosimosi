# cosimosi 작업 스펙 — 색인 (00.overview)

> 이 디렉터리(`spec/plan/`)는 **번호별 작업 스펙**이다. 각 파일 = AI 에이전트가 **맥락 없이 받아 한 세션에 끝낼 수 있는** 하나의 작업 단위이고, **마크다운 체크박스(`- [ ]`→`- [x]`)** 로 완료를 추적한다.
>
> 근거: 2026년 spec-driven development 합의(GitHub Spec Kit · AWS Kiro · OpenSpec) — 피처 단위 분할, 체크박스 태스크, 요구사항↔태스크 추적성, 명시적 비목표·영향 파일.

- **무엇을/왜**: [concept.md](../concept.md) (엔그램 우주 비전) — 이 스펙들의 **헌법(constitution)**. 충돌 시 여기로 돌아간다.
- **어떻게**: [Architecture.md](../Architecture.md) (FSD + package-by-feature + WebGPU/TSL + Connect RPC + sqlc/pgvector).
- **무엇이 참인가(최종 결과)**: [policy/](../policy/) — 도메인·UX별 **정전(canonical) 규칙·조건·파라미터 값**의 단일 출처(SSOT). plan이 *과정*이라면 policy는 *최종 결과의 정의*다. ([policy/README.md](../policy/README.md))
- **정합 기준**: 데이터 모델(원본=`records` 불변 / 별=`memories` 가변 분리)과 RPC 계약(단일 `MemoryService`)으로 통일 — 02~05·11·12 스펙 수정의 단일 기준. 충돌 시 헌법(concept.md)으로 우선.

## 이 디렉터리 사용법

1. **선행(Depends on)이 모두 `done`인 스펙**을 하나 고른다(아래 의존 그래프 참고).
2. 그 파일을 연다 → 목적·범위·설계·**영향 파일**·태스크를 읽는다(cold-start로 충분하도록 작성됨).
3. 태스크를 위→아래로 구현하고, 끝낼 때마다 **체크박스를 `- [x]`로 갱신 + 커밋**한다.
4. 파일 최하단 **완료 정의(DoD)** 를 모두 만족하면 그 스펙은 done. 이 색인의 상태표도 갱신한다.

> ⚠️ 체크박스는 **수동 갱신**이다(Spec Kit도 자동 추적 안 함). 미갱신 시 다음 세션이 완료된 일을 재실행할 수 있다.

## 구현 프로세스 (각 스펙 공통 — 3단계)

모든 스펙 구현은 아래 3단계로 진행한다. 한 단계를 건너뛰지 않는다.

1. **구현** — 스펙의 태스크를 위→아래로 구현하고, 끝낼 때마다 체크박스를 `- [x]`로 갱신한다.
2. **코드리뷰 → 리팩토링** — 구현 직후 **codex 리뷰를 1회** 띄우되, 위임 에이전트에게 **동기 실행**(codex 종료까지 대기 후 결과 반환)을 명시한다. ⚠️ 에이전트가 codex를 다시 자기 백그라운드로 detach하면(**이중 백그라운드**) 결과가 고아가 되어 영영 합산되지 않는다 — 2026-06-10 spec 16에서 실제 발생(45분 헛돈 고아 작업). codex가 도는 동안(~25분) `/code-review`를 실행해 그 결과부터 반영하고 문서화를 진행한다(말미 유휴 제거). codex 결과가 도착하면 **합산**(중복 제거 + 심각도 우선순위화)해 반영한다 — codex는 다른 엔진이라 레이스/수명주기류와 *리팩토링 수정분의 2차 버그*를 잘 잡는다(spec 16에서 high 2건). 기각한 지적은 이유를 남긴다. (둘 다 diff 범위 리뷰 — 전체 감사가 필요하면 별도 워크플로) ⚠️ **턴을 끝내고 codex 완료 노티를 기다리지 말 것** — 2026-06-11 spec 18에서 메인 에이전트가 "결과 오면 이어서"라며 턴을 끝냈더니 완료 노티가 전달되지 않아 사용자가 `/codex:status`를 칠 때까지 10분+ 유휴 대기가 발생했다. /code-review 반영이 끝났는데 codex가 아직 돌고 있으면 **`TaskOutput(block=true, timeout=600000)`을 반복 호출해 같은 턴 안에서 블로킹 대기**한다(codex ~25분 ≈ 3회) — 그래야 도착 즉시 합산으로 이어진다.
3. **리포트(한글)** — 구현 결과를 **한글로 리포트**한다: ① 무엇을 바꿨나(파일·핵심 설계 결정) ② DoD 검증 결과(실행한 명령 + 결과) ③ 리뷰 합산에서 반영/기각한 항목과 이유 ④ 남은 일·후속 제안.

> **정책 동기화(지침 — 필수).** ⚠️ **[policy/](../policy/)는 *현재 구현된 상태*만 담는다 — 계획·미구현 규칙은 절대 넣지 않는다.** plan은 *구현 과정*을, policy는 *지금 참인 최종 결과의 정의*를 담는다(코드의 거울). 이 작업이 최종 동작·규칙·수치를 정하거나 바꾸면, **구현이 끝난 시점에** 해당 policy 파일에 반영한다(없으면 신설) — 같은 값(반감기·τ·A_MIN·fly-to 감쇠 등)이 plan 곳곳에 흩어져 표류하지 않도록 **값의 단일 출처는 policy**다. 각 스펙의 `## 정책 영향(policy)` 절에는 *이 작업이 구현 완료 시 policy에 추가·반영할 규칙*을 미리 적어둔다(그건 plan 문서이지 policy가 아니다). 정책엔 상태 배지(✅/🚧)를 두지 않는다 — 적혀 있으면 이미 구현돼 있다는 뜻.

## 정책 (policy) — 최종 결과의 정의

plan이 *어떻게 만들었나(과정)* 라면, [spec/policy/](../policy/)는 *무엇이 참이어야 하나(최종 결과)* 를 도메인·UX별로 못 박는다 — 정전(canonical) 규칙·조건·튜닝 값의 **단일 출처(SSOT)**. plan의 수치가 흩어져 표류하지 않도록, 최종 정의는 policy에 모은다.

- **domain/** — 도메인별 규칙: [star](../policy/domain/star.md)(별이 되는 조건·형태·밝기·감쇠·재성형), [synapse](../policy/domain/synapse.md)(연결 생성·weight·가지치기), [memory](../policy/domain/memory.md)(3겹·조각 분할·감정), [universe](../policy/domain/universe.md)(좌표 창발·요즘 상태·흥분성·야간), [navigation](../policy/domain/navigation.md)(카메라·fly-to 속도·감쇠), [data-sync](../policy/domain/data-sync.md)(쿼리 캐시·이벤트 무효화·병합·출처 경계).
- **ux/** — 경험 규칙: [interaction](../policy/ux/interaction.md)(회상·기록·체험), [landing](../policy/ux/landing.md)(비전 showcase·이론 카드), [motion-accessibility](../policy/ux/motion-accessibility.md).
- 새 도메인이 생기면 `policy/domain/<name>.md`를 더한다. ⚠️ **policy는 *현재 구현된 상태*만 담는다 — 계획·미구현은 넣지 않는다**(상태 배지 없음; 적혀 있으면 곧 구현돼 있다는 뜻). 각 규칙은 구현한 plan NN + 코드 경로를 근거로 단다. 유지 규칙은 [policy/README.md](../policy/README.md)와 아래 **정책 동기화 지침**.

## 브랜치 전략

베타 출시부터 main 단일 브랜치를 버리고 **main / develop 2단 + 스펙 작업 브랜치**로 운영한다(14 deploy-cicd의 develop→스테이징 · main→프로덕션 자동 배포 전제와 정합).

| 브랜치 | 역할 | 규칙 |
|---|---|---|
| `main` | **프로덕션** — 베타 사용자(~50명)가 보는 것 | 직접 push 금지. `develop`→`main` PR(릴리즈) 또는 `hotfix/*` 머지로만 변경. 항상 배포 가능 상태 유지. |
| `develop` | **통합/스테이징** — 스펙 구현 합류 지점 | 스펙 브랜치의 머지 대상. push 시 스테이징 자동 배포(14). |
| `spec/NN-slug` | **스펙 작업** — 스펙 1개 = 브랜치 1개 | `develop`에서 분기 (예: `spec/16-connect-query`). 위 구현 프로세스 3단계 완료 + DoD 충족 후 `develop`으로 머지. 커밋은 Conventional Commits(영문 제목/한글 본문), 의미 단위로 작게. |
| `hotfix/slug` | **프로덕션 긴급 수정** | `main`에서 분기 → `main` 머지 → `develop`에 백머지(둘이 갈라지지 않게). |

```
spec/16-connect-query ──PR──► develop ──PR(릴리즈)──► main (프로덕션)
spec/17-error-handling ─PR──►    ▲                      │
                                 └──── 백머지 ◄── hotfix/* ┘
```

- **부트스트랩**: 완료 — `develop`이 `main`에서 생성되어 origin에 존재한다(2026-06-10).
- **1인 개발 모드(2026-06-10~):** 스펙 브랜치 → `develop` 머지는 PR 없이 **로컬 머지**로 진행한다(혼자라 PR 게이트가 과함 — 위 다이어그램의 spec→develop PR은 협업자가 생기면 복귀). `develop`→`main` 릴리즈 PR은 유지.
- **릴리즈 단위**: 스펙 1~N개가 develop에서 검증되면 `develop`→`main` PR로 베타에 릴리즈. 릴리즈 PR 본문에 포함 스펙 번호를 명시한다.
- **충돌 최소화**: 같은 파일을 건드리는 스펙(영향 파일 겹침)은 병렬 브랜치로 띄우지 말고 선행 관계로 직렬화한다(아래 색인의 `선행` 열).

## 스펙 파일 템플릿 (각 NN.*.md 공통)

```
# NN. {제목}

> 한 줄 목적. | 범위: MVP | 선행: NN, NN | 차단: NN

## 목적 (Why)              — 이 작업이 푸는 문제
## 범위                     — 포함 / 비목표(Out of scope, 과잉구현 방지)
## 참고                     — concept.md §, Architecture.md § (읽고 시작)
## 설계 요점                — 핵심 구조·인터페이스·데이터/계약(필요 시)
## 영향 파일                — 건드릴 정확한 경로(병렬·충돌 판단 근거)
## 수용 기준 (EARS/GWT)     — 테스트 가능한 문장. 1.1 WHEN … THE SYSTEM SHALL …
## 작업 (Tasks)             — - [ ] T001 [P] {정확한 경로+동작}  _참조: 수용 1.1_
## 정책 영향(policy)        — 이 작업이 만드는/바꾸는 policy 규칙(domain/ux). 최종 정의는 policy로(없으면 신설). 없으면 "없음"
## 완료 정의 (DoD)          — 관찰 가능·검증 명령(예: curl, build, E2E)
```

**규약**: 태스크는 *정확한 파일 경로 + 구체 동작*. `[P]`=서로 다른 파일·무의존이라 병렬 가능. `_참조: …_`로 수용 기준 역참조. 커밋은 Conventional Commits(영문 제목/한글 본문). ⚠️ Windows에선 `go`/`sqlc`/`buf`/`protoc`를 **Docker/WSL에서** 실행.

### 공유 설계 결정 (19–27 횡단 — 충돌 방지 단일 기준)

- **조각 fan-out(21):** `memories.record_id`는 non-unique FK라 **1 record→N memories에 FK 마이그레이션 불필요**. 원본 일기는 1행 불변(헌법1); 조각별 감정은 **가변 `memories`** 에 둔다(`records` 아님).
- **마이그레이션 번호(단일 기준 — 빌드/배포 순):** `00002`=30(`user_settings`·`user_emotion_colors`), `00003`=34(`llm_provider_configs`·`llm_selection`·`llm_usage_daily` — 34가 21보다 먼저 빌드), `00004`=21(memories에 mood/intensity/fragment_index/fragment_text/valence + jobs 키잉), `00005`=23(`evolution_history` 테이블 + memories 재성형 컬럼 brightness_offset/hue_shift/form_seed_delta/version), `00006`=27(안정 좌표 캐시 — 필요 시). 29·25·26은 마이그레이션 없음(29=proto/도메인/팔레트, 25=집계 쿼리, 26=클라 모델+proto 필드). **00001은 수정 금지**(append-only DDL).
- **job 키잉:** `jobs.memory_id`가 NOT NULL FK→memories라 record/사용자 단위 job을 못 건다. **00004(21)에서 `jobs.memory_id`를 nullable로 + `jobs.record_id`(extract용)·`jobs.user_id`(consolidate용) 추가.** extract job=record_id(memory_id NULL), embed job=memory_id, consolidate job(27)=user_id. `KindExtract`(20)·`KindConsolidate`(27)를 `KindEmbed`와 나란히.
- **proto(누적):** 21 `RecordMemoryResponse{record_id, repeated memory_ids}`·Request mood/intensity optional; 23 `Star{brightness_offset,hue_shift,form_seed_delta,version}` + `EvolutionSnapshot` 메시지 + `evolution_history` 소유; 24 `GetEvolutionHistory` rpc; 25 `GetUniverseResponse.ambient{hue,sat,arousal}`; 26 `Synapse.co_activation_count`; 28 `Star{record_id,fragment_index}` + `RecallMemoryResponse.fragment_text` + `ListRecords` rpc/`RecordSummary`. **`EvolutionSnapshot`·Star 재성형 필드는 23이 소유**(24는 `GetEvolutionHistory` 읽기 rpc만). 좌표 필드는 영원히 없음(헌법3).
  - **`Star` 필드 번호 할당(충돌 금지):** `1–4` 기존(memory_id/mood/intensity/last_recalled_at) · `5–8` = 23(brightness_offset/hue_shift/form_seed_delta/version) · `9–10` = 28(record_id/fragment_index). `RecallMemoryResponse`: `1` record(11) · `2` fragment_text(28).
- **밝기 바닥 단일화:** 헌법2의 단일 바닥은 `A_MIN=0.05`. 26의 변조 모델도 이 바닥을 재사용한다(랜딩 SilentEngramCard의 `0.12`는 시연용 값 — 실제 floor는 0.05로 통일).
- **흥분성(22):** 클러스터 흥분성 `e(c,t)`는 워커 `buildLinks`에서 기존 타임스탬프로 파생(새 컬럼 없이). 27의 야간 리셋은 워커 인메모리 캐시/파생 재계산이지 영속 컬럼 UPDATE가 아니다.
- **spec 10 대체:** 21이 단일 낙관 별(spec 10 §1.1/1.2)을 "조각내는 중 → refetch로 N 별"로 **supersede**한다.
- **구 번호 → 신 번호 매핑(이전 백로그 #20–#27 재배치):** 구 #24 llm-extraction→**20**, (신규)기억 분할→**21**, 구 #23 competitive-allocation→**22**, 구 #20 reconsolidation→**23**, 구 #21 evolution-history-ui→**24**, 구 #25 ambient-mood→**25**, 구 #23 relevance-decay→**26**, 구 #22 nightly-consolidation→**27**, (신규)원본·조각·별 길찾기→**28**, 구 #26 universe-sharing→**35**, 구 #27 shared-memory-resonance→**36**(29·30은 emotion-taxonomy·personalization이, **34는 admin-console이** 점유). ⚠️ concept.md·Architecture.md·specs 04·05·07·10·11·12의 "v1 #20–#25" 표기는 이 매핑으로 읽는다(추후 일괄 정리 대상).

## 불변 원칙 (constitution — 모든 스펙이 지킨다)

concept.md·Architecture.md에서 추출한, 작업 중 절대 깨면 안 되는 것들:

1. **원본 일기는 불변·영구.** 무엇이 변해도 사용자가 쓴 텍스트는 수정·삭제하지 않는다. (별/연결과 분리 저장)
2. **별·시냅스는 우주에서 삭제하지 않는다.** 감쇠는 *밝기*만(최소 밝기 바닥 `a_min`). 잠든 별로 잔존.
3. **별 좌표는 서버가 아니라 클라 force-sim에서 창발.** 서버는 가중치 그래프만 권위. 기존 별은 pin.
4. **FSD 임포트 방향 단방향.** 슬라이스는 아래 레이어만 import. **같은 레이어 직접 import 금지 — `entities` 간 교차만 `@x` 공개 API로**(`@/entities/A/@x/B`, Architecture §2.3). `model`/`shared/lib`은 `three`/React/DOM 미의존(모바일 재사용). 플랫폼 분기는 `ui`에서만.
5. **proto=전송 DTO, 도메인은 순수, sqlc=영속.** 도메인 타입에 json/db/proto 태그 금지.
6. **Connect는 unary 전용**(RN 스트리밍 미지원). 회상 강화는 클라 로컬 + unary 배치.
7. **AI는 공급자 추상화 뒤.** 어떤 LLM/임베딩도 교체 가능.
8. **셰이더는 TSL**, 별은 **InstancedMesh**, 시냅스는 **Line2** — 수천 노드 전제.
