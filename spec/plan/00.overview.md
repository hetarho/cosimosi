# cosimosi 작업 스펙 — 색인 (00.overview)

> 이 디렉터리(`spec/plan/`)는 **번호별 작업 스펙**이다. 각 파일 = AI 에이전트가 **맥락 없이 받아 한 세션에 끝낼 수 있는** 하나의 작업 단위이고, **마크다운 체크박스(`- [ ]`→`- [x]`)** 로 완료를 추적한다.
>
> 근거: 2026년 spec-driven development 합의(GitHub Spec Kit · AWS Kiro · OpenSpec) — 피처 단위 분할, 체크박스 태스크, 요구사항↔태스크 추적성, 명시적 비목표·영향 파일.

- **무엇을/왜**: [concept.md](../concept.md) (엔그램 우주 비전) — 이 스펙들의 **헌법(constitution)**. 충돌 시 여기로 돌아간다.
- **어떻게**: [Architecture.md](../Architecture.md) (FSD + package-by-feature + WebGPU/TSL + Connect RPC + sqlc/pgvector).
- **정합 기준**: 데이터 모델(원본=`records` 불변 / 별=`memories` 가변 분리)과 RPC 계약(단일 `MemoryService`)으로 통일 — 02~05·11·12 스펙 수정의 단일 기준. 충돌 시 헌법(concept.md)으로 우선.

---

## 이 디렉터리 사용법

1. **선행(Depends on)이 모두 `done`인 스펙**을 하나 고른다(아래 의존 그래프 참고).
2. 그 파일을 연다 → 목적·범위·설계·**영향 파일**·태스크를 읽는다(cold-start로 충분하도록 작성됨).
3. 태스크를 위→아래로 구현하고, 끝낼 때마다 **체크박스를 `- [x]`로 갱신 + 커밋**한다.
4. 파일 최하단 **완료 정의(DoD)** 를 모두 만족하면 그 스펙은 done. 이 색인의 상태표도 갱신한다.

> ⚠️ 체크박스는 **수동 갱신**이다(Spec Kit도 자동 추적 안 함). 미갱신 시 다음 세션이 완료된 일을 재실행할 수 있다.

## 스펙 프리뷰 규약 (모든 스펙이 구현 시 함께)

각 스펙을 구현하면 **그 스펙이 만든 기능을 브라우저에서 확인할 프리뷰**를 프론트의 `/spec-preview` 라우트(`frontend/src/pages/spec-preview/ui/SpecPreviewPage.tsx`)에 추가/갱신한다. 사이드바에서 스펙 번호를 고르면 해당 패널이 뜬다. **이 프리뷰 추가는 각 스펙 완료 정의(DoD)의 암묵적 공통 항목**이다(스펙 본문에 안 적혀 있어도 적용).

- **FE/FS 스펙**(06·08·09·10·11·12·13 등): 실제 동작하는 **인터랙티브 프리뷰**(폼·버튼·결과 렌더). 예: 04는 RecordMemory 폼 + GetUniverse 별 목록.
- **BE/Infra 스펙**(02·03·05·14 등): 인터랙티브 UI가 마땅찮으면 **대략적 아키텍처 설명**(자료구조·흐름·계약 요약)으로 대체한다. (04는 BE지만 RPC를 직접 호출하는 인터랙티브 프리뷰를 둔다 — 호출 가능한 BE는 인터랙티브 우선.)
- `/spec-preview`는 **dev 전용 도구**(프로덕션 번들 제외). 인증이 필요한 호출은 패널의 dev 토큰 입력 또는 Supabase 로그인으로 처리한다.

## 스펙 파일 템플릿 (각 NN.*.md 공통)

```
# NN. {제목}

> 한 줄 목적. | 범위: MVP | 선행: NN, NN | 차단: NN

## 목적 (Why)              — 이 작업이 푸는 문제
## 범위                     — 포함 / 비목표(Out of scope, 과잉구현 방지)
## 참고                     — concept.md §, Architecture.md § (읽고 시작)
## 설계 요점                — 핵심 구조·인터페이스·데이터/계약(필요 시)
## 영향 파일                — 건드릴 정확한 경로(병렬·충돌 판단 근거)
## 수용 기준 (EARS/GWT)     — 테스트 가능한 문장. 1.1 WHEN … THE SYSTEM SHALL …
## 작업 (Tasks)             — - [ ] T001 [P] {정확한 경로+동작}  _참조: 수용 1.1_
## 완료 정의 (DoD)          — 관찰 가능·검증 명령(예: curl, build, E2E)
```

**규약**: 태스크는 *정확한 파일 경로 + 구체 동작*. `[P]`=서로 다른 파일·무의존이라 병렬 가능. `_참조: …_`로 수용 기준 역참조. 커밋은 Conventional Commits(영문 제목/한글 본문). ⚠️ Windows에선 `go`/`sqlc`/`buf`/`protoc`를 **Docker/WSL에서** 실행.

## 불변 원칙 (constitution — 모든 스펙이 지킨다)

concept.md·Architecture.md에서 추출한, 작업 중 절대 깨면 안 되는 것들:

1. **원본 일기는 불변·영구.** 무엇이 변해도 사용자가 쓴 텍스트는 수정·삭제하지 않는다. (별/연결과 분리 저장)
2. **별·시냅스는 우주에서 삭제하지 않는다.** 감쇠는 *밝기*만(최소 밝기 바닥 `a_min`). 잠든 별로 잔존.
3. **별 좌표는 서버가 아니라 클라 force-sim에서 창발.** 서버는 가중치 그래프만 권위. 기존 별은 pin.
4. **FSD 임포트 방향 단방향.** `model`/`shared/lib`은 `three`/React/DOM 미의존(모바일 재사용). 플랫폼 분기는 `ui`에서만.
5. **proto=전송 DTO, 도메인은 순수, sqlc=영속.** 도메인 타입에 json/db/proto 태그 금지.
6. **Connect는 unary 전용**(RN 스트리밍 미지원). 회상 강화는 클라 로컬 + unary 배치.
7. **AI는 공급자 추상화 뒤.** 어떤 LLM/임베딩도 교체 가능.
8. **셰이더는 TSL**, 별은 **InstancedMesh**, 시냅스는 **Line2** — 수천 노드 전제.
