## 1. How we build вҖ” epics, delivered as sprints, layered within

### 1.1 The roadmap is organized by **product epic**

Forward work is grouped into **epics** вҖ” product capability areas drawn from the PRD requirement prefixes
([W] writing, [R] recall, [F] forgetting, вҖҰ). Epics are sequenced into **sprints** (delivery increments, В§2.2). This
is the top-level structure: we ship a capability, not a layer.

### 1.2 Within an epic, units flow through the clean-architecture layers

Each epic's backlog units are ordered by the **product's clean-architecture layers** вҖ” **domain вҶ’ use-case вҶ’
presentation** вҖ” which is *orthogonal* to the code architecture (FSD on the frontend, domain-first Go on the backend):

- A unit's **layer** decides *when within its epic* it is built; the **architecture** decides *where its code lives*
  (which context / slice / segment). A domain unit is the pure model wherever ARCHITECTURE puts it (the `entities/<noun>/model`
  segment, the Go `<context>` package) вҖ” **even while its rendering segment stays empty**. *How a thing is shown* is a
  presentation concern.
- **Domain & use-case units are headless.** Pure domain (types + deterministic functions, FEвҶ”BE golden-parity) and
  use-cases (orchestration behind ports, mock fallback) carry no pixels вҖ” verified by golden-parity tests and the
  temporary **`/test` page** (Foundation В· plan 12). They parallelize in dependency waves.
- **Presentation units are vertical and sequential.** Each is one user-facing flow taken end-to-end through rendering
  + UI and confirmed on screen, built one slice at a time вҖ” **each slice ships web (open beta) + mobile (MVP) together**.
  The rendering layer owns the visual vocabulary and is the one place domain and visual words meet (ARCHITECTURE В§3.4).

### 1.3 Shared substrate

Some domain units (the memory aggregate & schema, emotion model, synapse plasticity, the force-sim layout, the
universe clock) are the **engine substrate** that many epics build on. They are first *needed* by вҖ” and therefore built
*inside* вҖ” the earliest epic, then reused. Reuse promotes **down** a layer (to `packages/`), never copied sideways.
On the presentation side, the **rendering substrate** (plan [14](14.rendering-foundation.md), `rendering-foundation`) вҖ”
the renderer boundary, the TSL shader toolkit, and the skinnable background вҖ” is the seam every Epic-A render unit
mounts.

### 1.4 Numbering

Plans are a **fresh, stable, contiguous series** вҖ” never reused or renumbered once authored. The Foundation is
`spec/plan/01`вҖ“`13`; the next plan `/create-plan` scaffolds is `14`, and so on. **Backlog units below are listed by
title in dependency order, without a number** вҖ” the number is assigned when `/create-plan` scaffolds the unit.

## 4. Cross-cutting decisions

- **App roots:** `apps/api` В· `apps/web` В· `apps/mobile`, plus `apps/blog`. Shared pure code lives in `packages/`.
- **Mobile = React Native вҖ” first-class MVP target, built alongside web (not deferred).** Release targets: **web =
  open beta, mobile = MVP**. RN reuses the one shared stack (no parallel state/i18n lib); the RN WebGPU renderer is in
  scope (pre-1.0 `react-native-webgpu`, early-adopter caveats). Rationale: [ARCHITECTURE.md](../ARCHITECTURE.md) В§3.5.
- **No real deployment yet.** The foundation is deployable, but production hosting (Cloudflare / Lightsail / Supabase)
  is out of scope for now.
- **Config is build-time.** Every tuning number a plan introduces goes in `spec/values.yaml` вҶ’ `pnpm gen:values` вҶ’
  FE(TS)/BE(Go) constants; never hardcoded.
- **Numbering.** Fresh contiguous series (01+); stable IDs вҖ” never reused or renumbered once authored.
- **Docs.** The PRD, concept stay Korean (the vision source); the glossary is bilingual; plan / job / tech /
  ARCHITECTURE / principle docs are English.
- **tech/ & policy/** docs are added incrementally as units land вҖ” the as-built rules each one needs.
