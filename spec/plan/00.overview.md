## 0. How we build — the phase model

### 0.1 The build axis is the product's clean-architecture layers — not the code architecture

Phases are sequenced by the **clean-architecture layers of the product**: **environment → domain → use-case →
presentation**. This axis is **orthogonal to the code architecture** (FSD on the frontend, domain-first Go on the
backend):

- A unit's **clean-architecture layer** decides *when* it is built (which phase).
- The **architecture** decides *where its code lives* (which context / layer / slice / segment).

So a domain concept is placed wherever ARCHITECTURE dictates — e.g. the frontend `entities/<noun>/model` segment, the
Go `<context>` package while small, or `<context>/domain` after the context earns a split — **even while its sibling
`ui` / rendering segment stays empty**. *How a thing is shown* is a presentation concern, deferred to the presentation
phase. "Domain work" does not mean "an `entities/` folder with a renderer"; it means the pure model, wherever the
architecture puts it.

### 0.2 Two cadences

- **Phases 2–3 are headless and parallel.** Pure domain and use-cases carry no pixels. They are verified through
  golden-parity tests and a temporary **`/test` page** (Phase 1 · unit 12) — so many units build concurrently, in
  dependency waves, without any real UI.
- **Phase 4 is presentation — vertical and sequential.** This is where the frontend is *actually* built. Each plan is
  **one vertical slice**: a single user-facing flow carried all the way through its rendering and UI and confirmed on
  screen. Slices are built **one at a time, not in parallel**, because each needs visual verification.

### 0.3 Scope comes from the PRD

The features and boundaries of Phases 2–4 are **PRD decisions**, authored later. This roadmap lists those phases'
units only after the PRD exists; until then they are described by their *rules*, not their contents. **Phase 1 is the
exception** — it is planning-independent platform setup, so its units are enumerated in advance (§1).

## 1. The roadmap

Numbers are a **fresh, stable series** — never reused or renumbered once a unit is authored. Each unit gets a
`spec/plan/NN.*.md`, then a job doc, then implementation into the clean `apps/{api,web,mobile}` roots.

### Phase 1 — Environment / platform *(planning-independent — enumerated now)*
> The providers and systems we install up front so later work plugs into a seam instead of retrofitting one. No
> business logic. Built regardless of the PRD; units parallelize within the phase.
>
> **Both app shells, set up together.** The FE units here wire the **web *and* mobile** app shells — providers
> (data · i18n · theme · session), routing (web router · RN navigation), and design-system primitives (DOM ·
> NativeWind/RN) — and the shared logic (XState machines, Query, Paraglide, domain) lives in `packages/`
> cross-platform **from day one** (no web-first promote-later for these). RN reuses that one shared stack; it gets
> **no parallel state/i18n library**. Deferred to later phases (built with their web counterparts): **mobile feature
> UI** and the **RN WebGPU renderer**.

### Phase 2 — Core domain (pure) *(units pending PRD)*
> The pure domain core — types, deterministic functions, schema, and the frontend domain **mirror** (whose `ui` stays
> empty). **Headless**, verified by golden-parity tests (FE ↔ BE) and the `/test` page. Parallelizes in dependency
> waves. The connection rule and the must-not-implement list ([concept.md](../concept.md) §9–§10) bind every unit here.

### Phase 3 — Use-cases (application) *(units pending PRD)*
> Application services that orchestrate the domain, with external dependencies behind ports (mock fallback). **Headless**
> — exercised through the `/test` page (write → inspect the resulting objects; act → inspect the state change), with no
> rendering. Parallelizes in dependency waves.

### Phase 4 — Presentation (the frontend) *(units pending PRD)*
> The visible product, on top of the settled domain and use-cases. **Vertical and sequential**: each plan is one
> user-facing flow taken end-to-end through rendering + UI and verified on screen, built one slice at a time. The
> rendering layer owns the visual vocabulary and is the one place domain and visual words meet (ARCHITECTURE §3.4).

### Phase 5+ — Depth, personalization, social, ops *(pending PRD)*
> Built after the core loop is alive. Enumerated as the PRD calls for them.

## 2. The constitution (where the rules live — not restated here)

Every unit honors the project's constitution. It is **owned by the docs below**, not duplicated into this index:

- **Behavioral laws + neuroscience guardrails** (forgetting = dimming, recall = partial restore, sleep
  consolidation, emotion ≠ position, the must-not-implement list, …) → **[concept.md](../concept.md) §9–§10**.
- **Architecture rules** (one ubiquitous language owned by the domain model; domain-first Go with adapters at the edge
  and consumer-owned ports; proto/domain/sqlc separation; unary-only transport; FSD one-way imports; instancing/TSL
  rendering; config-as-build-time; …) →
  **[ARCHITECTURE.md](../ARCHITECTURE.md)**.
- **The canonical terms** themselves → **[ubiquitous-language.md](../ubiquitous-language.md)** (normative).

A plan that would break one of these resolves it against the owning doc *before* it is written.

## 3. Cross-cutting decisions

- **App roots:** `apps/api` · `apps/web` · `apps/mobile`, plus `apps/blog`. Shared pure code lives in `packages/`.
- **Mobile = React Native; the app *shell* ships in Phase 1, feature UI + renderer later.** RN reuses the one shared
  stack (no parallel state/i18n lib). Rationale and structure: [ARCHITECTURE.md](../ARCHITECTURE.md) §3.5.
- **No real deployment yet.** The foundation is deployable, but production hosting (Cloudflare / Lightsail / Supabase)
  is out of scope for now.
- **Numbering.** Fresh contiguous series (01+); stable IDs — never reused or renumbered once authored.
- **Docs.** Spec docs are English; the glossary is bilingual; `concept.md` stays Korean as the vision source.
- **tech/ & policy/** docs are added incrementally as units land — the as-built rules each one needs.

### 4.2 Phase 1 dependency graph

- 01 is the root dependency for every Phase 1 unit.
- 02 can be implemented early and should guard the rest of the phase.
- 03 precedes 04, 06, 10, and 12 where those units need RPC transport.
- 04 and 05 can build in parallel after 03, then converge on the per-user isolation contract.
- 11 precedes 06 because cache timing defaults are generated values.
- 08 precedes 09 and 12 for UI copy boundaries.
- 09 precedes the final shape of 12 so the harness can use shared primitives.
- 12 (web test harness) is the browser verification shell; it can degrade gracefully when optional panels are absent.
- 13 (mobile shell) composes 04 · 06 · 07 · 08 · 09; it builds once those providers/primitives exist, in parallel with 12. Mobile feature UI and the RN WebGPU renderer are out of Phase 1 (later phases).
