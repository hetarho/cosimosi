## 1. How we build — epics, delivered as sprints, layered within

### 1.1 The roadmap is organized by **product epic**

Forward work is grouped into **epics** — product capability areas drawn from the PRD requirement prefixes
([W] writing, [R] recall, [F] forgetting, …). Epics are sequenced into **sprints** (delivery increments, §2.2). This
is the top-level structure: we ship a capability, not a layer.

### 1.2 Within an epic, units flow through the clean-architecture layers

Each epic's backlog units are ordered by the **product's clean-architecture layers** — **domain → use-case →
presentation** — which is _orthogonal_ to the code architecture (FSD on the frontend, domain-first Go on the backend):

- A unit's **layer** decides _when within its epic_ it is built; the **architecture** decides _where its code lives_
  (which context / slice / segment). A domain unit is the pure model wherever ARCHITECTURE puts it (the `entities/<noun>/model`
  segment, the Go `<context>` package) — **even while its rendering segment stays empty**. _How a thing is shown_ is a
  presentation concern.
- **Domain & use-case units are headless.** Pure domain (types + deterministic functions, FE↔BE golden-parity) and
  use-cases (orchestration behind ports, mock fallback) carry no pixels — verified by golden-parity tests and the
  temporary **`/test` page** (Foundation · plan 12). They parallelize in dependency waves.
- **Presentation units are vertical and sequential.** Each is one user-facing flow taken end-to-end through rendering
  - UI and confirmed on screen, built one slice at a time — **each slice ships web (open beta) + mobile (MVP) together**.
    The rendering layer owns the visual vocabulary and is the one place domain and visual words meet (ARCHITECTURE §3.4).

### 1.3 Shared substrate

Some domain units (the memory aggregate & schema, emotion model, synapse plasticity, the force-sim layout, the
universe clock) are the **engine substrate** that many epics build on. They are first _needed_ by — and therefore built
_inside_ — the earliest epic, then reused. Reuse promotes **down** a layer (to `packages/`), never copied sideways.
On the presentation side, the **rendering substrate** (plan [14](14.rendering-foundation.md), `rendering-foundation`) —
the renderer boundary, the TSL shader toolkit, and the skinnable background — is the seam every Epic-A render unit
mounts.

### 1.4 Numbering

Plans are a **fresh, stable, contiguous series** — never reused or renumbered once authored. The Foundation is
`spec/plan/01`–`13`, plus the pre-Epic-A substrates `14` (rendering) and `15` (web routing); the next plan
`/create-plan` scaffolds is `16`, and so on. **Backlog units below are listed by
title in dependency order, without a number** — the number is assigned when `/create-plan` scaffolds the unit.

## 3. The constitution (where the rules live — not restated here)

Every unit honors the project's constitution. It is **owned by the docs below**, not duplicated into this index:

- **The invariants [I1]–[I11] + behavioral laws + neuroscience guardrails** (forgetting = dimming, recall = partial
  restore, sleep consolidation, emotion ≠ position, the must-not-implement list, …) → **[PRD.md](../PRD.md) §7, §9**
  and their readable form in **[concept.md](../concept.md)**.
- **Architecture rules** (one ubiquitous language owned by the domain model; domain-first Go with adapters at the edge
  and consumer-owned ports; proto/domain/sqlc separation; unary-only transport; FSD one-way imports; instancing/TSL
  rendering; config-as-build-time; …) → **[ARCHITECTURE.md](../ARCHITECTURE.md)**.
- **The canonical terms** themselves → **[ubiquitous-language.md](../ubiquitous-language.md)** (= PRD §3, normative).

A plan that would break one of these resolves it against the owning doc _before_ it is written. `/create-plan` checks
every new plan against the invariants ([I1]–[I11]).

## 4. Cross-cutting decisions

- **App roots:** `apps/api` · `apps/web` · `apps/mobile`, plus `apps/blog`. Shared pure code lives in `packages/`.
- **Mobile = React Native — first-class MVP target, built alongside web (not deferred).** Release targets: **web =
  open beta, mobile = MVP**. RN reuses the one shared stack (no parallel state/i18n lib); the RN WebGPU renderer is in
  scope (pre-1.0 `react-native-webgpu`, early-adopter caveats). Rationale: [ARCHITECTURE.md](../ARCHITECTURE.md) §3.5.
- **No real deployment yet.** The foundation is deployable, but production hosting (Cloudflare / Lightsail / Supabase)
  is out of scope for now.
- **Config is build-time.** Every tuning number a plan introduces goes in `spec/values.yaml` → `pnpm gen:values` →
  FE(TS)/BE(Go) constants; never hardcoded.
- **Numbering.** Fresh contiguous series (01+); stable IDs — never reused or renumbered once authored.
- **Docs.** The PRD, concept stay Korean (the vision source); the glossary is bilingual; plan / job / tech /
  ARCHITECTURE / principle docs are English.
- **tech/ & policy/** docs are added incrementally as units land — the as-built rules each one needs.
