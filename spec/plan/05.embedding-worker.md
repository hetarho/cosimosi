# 05. embedding-worker - extract/embed/consolidate queue (as-built)

> 현재 구현 기준 문서. worker는 단순 embedding consumer가 아니라 extract, embed, nightly consolidate job을 우선순위로 처리하는 비동기 파이프라인이다. | 범위: BE async worker/AI ports | 상태: 구현 완료

## 목적

RecordMemory는 원본 일기를 저장하고 job을 적재한다. Worker는 이 job을 소비해 일기를 조각 별로 fan-out하고, 각 조각을 임베딩해 semantic synapse를 만든다. 같은 worker loop는 야간 공고화 job도 처리한다. 모든 단계는 재시도/lease reclaim에 견딜 수 있도록 idempotent하게 설계한다.

## AI 포트

- `ai.Embedder`
  - `Embed(ctx, text) ([]float32, error)`
  - `Dim() int`
  - `Model() string`
- `ai.NewEmbedder(cfg)`
  - `AI_EMBEDDER=mock`: 키 없이 결정론적 1536차원 벡터.
  - `AI_EMBEDDER=openai`: `OPENAI_API_KEY` 필요, `text-embedding-3-small`, 1536차원.
- `OpenAIEmbedder`
  - response dim 검증.
  - 입력 rune cap 4000. `memory.MaxBodyRunes`와 맞춰야 한다.
  - 텍스트 해시 인메모리 cache.
  - embedding calls, approximate tokens, cache hits, token cap hits를 분리 계측한다.
- `ai.Extractor`
  - 현재 extraction은 env knob가 아니라 admin console의 active LLM selection을 따른다.
  - active 설정이 없으면 keyless mock extractor로 degrade한다.
  - `cmd/api/main.go`가 `segmenterAdapter`로 memory service의 Extractor port와 연결한다.

## Queue 모델

- `jobs.kind`:
  - `extract`: record를 조각 별로 fan-out.
  - `embed`: fragment star를 임베딩하고 semantic links 생성.
  - `consolidate`: 사용자 그래프의 야간 공고화.
- `jobs.status`:
  - `pending`, `running`, `done`, `failed`.
- keying:
  - extract: `record_id`, `user_id`.
  - embed: `memory_id`.
  - consolidate: `user_id`.
- `ClaimJob`은 due pending job 또는 stale running job을 claim한다.
  - `FOR UPDATE SKIP LOCKED`로 concurrent worker 안전성을 얻는다.
  - running job이 lease보다 오래되면 reclaim 가능하다. 현재 lease는 120초다.
  - reclaim은 attempts를 증가시키지 않는다.
- 실패 시 `FailJob`이 attempts를 증가시키고 지수 backoff `next_run_at`을 기록한다.
- 최대 시도 후에는 `failed`로 보존한다. job을 삭제하지 않는다.
- worker는 주기적으로 queue summary를 로그에 남긴다.

## 처리 우선순위

`processOne`은 다음 순서로 claim한다.

1. `extract`
2. `embed`
3. `consolidate`

사용자가 막 작성한 일기가 먼저 별이 되고 연결되도록 interactive path를 야간 batch보다 우선한다.

## Extract Pipeline

흐름:

1. `Claim(KindExtract)`.
2. `FragmentIDs(record_id)`를 먼저 확인한다. 이미 fan-out되어 있으면 paid extraction을 다시 하지 않고 job을 complete한다.
3. `GetRecordForExtract(record_id)`로 immutable body, owner, entry_date, optional hints를 읽는다.
4. `Extractor.Extract(body)`로 segments를 얻는다.
5. `applyManualHint`가 degraded single neutral segment에만 whole-diary hints를 fallback으로 적용한다.
6. `FanOutFragments(record_id, user_id, segments)` 실행.
7. `CompleteJob`.

`FanOutFragments`는 하나의 트랜잭션에서 수행한다.

- N개의 `memories` insert.
- 각 fragment 별 `EnqueueEmbedJob`.
- 같은 record에서 태어난 모든 fragment pair에 intra-entry link 생성.
- intra-entry link는 `link_type='intra_entry'`, `weight=0.8`.
- `(record_id, fragment_index)` unique index와 existing fragment check가 retry/reclaim idempotency를 제공한다.

## Embed Pipeline

흐름:

1. `Claim(KindEmbed)`.
2. `GetMemoryForEmbed(memory_id)`로 owner, fragment text, entry_date를 읽는다. `fragment_text`가 없으면 whole record body를 fallback으로 쓴다.
3. `Embedder.Embed(text)`.
4. `len(vec) == Embedder.Dim()` 검증.
5. `UpsertEmbedding(memory_id, user_id, vec, Embedder.Model())`.
6. `KnnNearest(user_id, vec, self, candidateK)`로 같은 사용자 후보를 조회한다.
7. 후보 cluster의 흥분성 입력을 읽고 competitive allocation re-rank를 적용한다.
8. semantic links를 batch upsert한다.
9. `CompleteJob`.

연결 규칙:

- KNN SQL은 `user_id` 필터, self 제외, cosine similarity `>= 0.75`, `embedding <=> query` 오름차순 정렬을 사용한다.
- worker는 re-rank 여지를 위해 `candidateK = knnK * 2` 후보를 가져오고, 최종 `biasedK = 5`개를 선택한다.
- cluster excitability는 후보 별의 `last_recalled_at`과 후보 주변 synapse의 `last_activated_at`에서 파생한다. 별도 excitability column은 없다.
- selection score는 cosine similarity에 cluster excitability bias를 더한 값이다. **감정 유사도는 score에 안 들어간다**(후보 선택 불변, change 21).
- link weight 자체는 `initialWeight(cos_sim, temporal_bonus, emoSim)` 기반이다. **감정 유사도 항(change 21)**: `+emo_alpha·emoSim`(`emo_alpha=0.1`), `emoSim ∈ [0,1]` = 두 별 정동(valence×intensity-as-arousal) 원형 거리(`1 − 거리/√5`). 가까운 감정일수록 가산 — 단 아래 cap(0.79)·후보 게이트(`cos≥0.75`) 불변이라 감정은 *이미 의미로 후보가 된* 링크 weight만 미세 조정. `emo_alpha=0`이면 기존 동작.
- temporal bonus는 같은 날 `+0.3`, 7일에 걸쳐 선형으로 0까지 감소한다.
- cross-entry semantic weight는 `0.79`로 cap된다. intra-entry `0.8` 결속보다 강해지지 않게 하기 위해서다.
- DB upsert는 `LEAST/GREATEST`로 pair를 정규화한다. Go 문자열 byte order가 Postgres collation과 다를 수 있기 때문이다.
- conflict 시 semantic upsert는 기존 weight를 약화시키지 않고 `GREATEST`를 사용한다.

## Consolidate Pipeline

`KindConsolidate`는 spec 27의 야간 공고화 job이지만 같은 worker가 처리한다.

- `StartNightlyConsolidation` ticker가 active user별 consolidate job을 idempotently enqueue한다.
- worker claim priority에서는 extract/embed 뒤에 온다.
- `RunConsolidation`은 안정 좌표 cache, gist simplification, weak link pruning, job completion을 하나의 트랜잭션으로 묶는다.
- stable coords는 서버 재진입 cache이지 클라이언트 좌표 권위가 아니다.
- weak links는 삭제하지 않고 weight를 낮춘다.

## Runtime Wiring

- `cmd/api/main.go`가 `ai.NewEmbedder(cfg)`, `ai.NewExtractor(...)`, `job.NewRepository(db)`, `job.NewGraphStore(db)`를 조립한다.
- worker는 API 프로세스 안 goroutine으로 실행된다.
- signal context를 공유해 shutdown 시 worker가 멈춘다.
- shutdown 중 중단된 job은 failure attempt를 태우지 않고 running lease reclaim으로 다음 실행에서 회수된다.

## Flutter 동등성 기준

- Flutter는 embed/extract 작업을 직접 하지 않는다. 저장 후 `GetUniverse` refetch/polling으로 별/시냅스 도착을 반영한다.
- 저장 직후 `memory_ids[]`가 비어 있을 수 있다는 점을 UI 상태로 처리해야 한다.
- 좌표/밝기는 Flutter가 계산하지만, 서버가 주는 link weight, last timestamps, ambient, relevance, reshaping fields를 렌더 입력으로 써야 한다.

## 수용 기준

1. `AI_EMBEDDER=mock`이면 `OPENAI_API_KEY` 없이 extract/embed path가 돈다.
2. extract job은 record를 N fragment memories로 만들고, 각 fragment에 embed job을 적재한다.
3. 같은 record 재처리는 기존 fragment ids를 감지해 중복 fan-out하지 않는다.
4. embed job은 1536차원 embedding과 model name을 저장한다.
5. KNN은 같은 사용자 후보만 보고 self를 제외한다.
6. semantic link는 threshold, temporal bonus, excitability re-rank, `0.79` cap을 적용한다.
7. intra-entry link는 같은 record의 fragment pair를 `0.8`로 묶는다.
8. 실패 job은 backoff 후 retry되고, 최대 시도 후 `failed`로 보존된다.
9. stale running job은 lease 뒤 reclaim된다.
10. worker는 원본 record를 수정/삭제하지 않고, 좌표를 proto 권위로 저장하지 않는다.
