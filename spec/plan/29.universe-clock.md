## Purpose

cosimosi's clock is **diary-driven, not wall-clock** ([PRD ¬ß5.3.3](../PRD.md)): the universe sits frozen at "the last
time you advanced it," and it advances **only** when you launch a star or recall one ‚Äî just looking changes nothing
([T3]). This single property is what makes forgetting and consolidation *earned* rather than *elapsed*: a star dims
because you wrote past it, not because a week of real time passed while the app was closed.

Epic A shipped the launch/read loop but deliberately **derived** universe time as `max(created_universe_time)` over the
launched memories ‚Äî a stand-in that holds only until recall exists. That derivation cannot express the two things Epic B
needs: (1) a clock that sits at **today** after a recall-sync even though no memory was created today, and (2) the
resulting monotonic bar ‚Äî once the clock is at today, a diary dated *before* today can no longer raise a star ([T2]
case 2). Both require the clock to be **stored authoritative state**, not a read-time max.

This unit lays that foundation: the `universe_state` table (one row per user), and the pure domain model for the clock ‚Äî
how it advances (monotonically, never backward, [I10]), the diary-date constraint that decides whether a new diary
launches a star ([T1]), and universe time's role as the **read-time "now"** that every later read-time derivation
(forgetting [F], consolidation/semanticization [C], synapse decay [L5]) reads to compute elapsed universe-days. It is
headless: it owns the model and its persistence shape, not the orchestration of *when* to advance (plan 30) or any UI
(plan 31).

## Scope / Non-goals

**In scope**
- **The `universe_state` table** (goose migration + `memory/pg` repository): one row per user holding the authoritative
  `current_universe_time DATE` (day granularity ‚Äî universe time is measured in *diary days*, matching the DATE columns
  and per-day timers already in the schema). This is the one place the clock is stored ([T5]).
- **The pure clock domain model** in `internal/memory` (canonical names, no proto/sqlc/pgx ‚Äî ARCHITECTURE ¬ß2.4):
  - `AdvanceClock(current, target) ‚Üí DATE` ‚Äî the monotonic advance: `max(current, target)`; the clock **never moves
    backward** ([I10]). Deterministic, side-effect-free.
  - the **diary-date monotonic constraint** predicate ([T1]): given the current clock and a diary date, decide whether
    the diary **launches a star** (diary date **‚â•** the clock) or is **saved-without-a-star** (diary date **<** the
    clock). This is the rule; enforcing it at launch is the use-case's job (plan 30 / already seamed in plan 20).
  - the clock's **birth**: a user has **no** universe time until their first star-launch; the row is created lazily when
    the clock first advances. Before that the universe is empty and the clock is absent (nil) ‚Äî matching Epic A's
    current "no memories ‚Üí no universe_time" behavior.
- **Universe time as the read-time "now" signal** ([T5]): the model fixes that universe time is a **scalar date** that
  read-time derivations consume as "now" to compute elapsed universe-days (forgetting decay [F1], semanticize timer
  [C6a], synapse strength decay [L5]) and as the temporal-proximity reference for linking ([L4]). It is **never** a
  spatial coordinate and never enters layout ([I7], force-sim takes graph structure only ‚Äî [19.force-sim-layout](19.force-sim-layout.md)).
- **Replacing Epic A's derived universe time with the stored clock** as the model of record: this plan defines that
  `GetUniverse.universe_time` and the launch guard read the `universe_state` clock; the *wiring* of those call sites is
  the use-case's (plan 30). Epic A's `max(created_universe_time)` derivation is retired here as the model, kept working
  until plan 30 swaps the call sites.
- **Per-user isolation** on the clock row and every clock query (the [04.auth-session](04.auth-session.md)/
  [05.data-persistence](05.data-persistence.md) `UserScope` convention, enforced by `pnpm lint:persistence`).

**Non-goals**
- **No advance orchestration.** *When* the clock advances (launch ‚Üí diary date; recall ‚Üí today) and the read-time
  progression hook that fans the advance interval out to [F]/[C] are [30.time-advance-usecase](30.time-advance-usecase.md).
  This unit provides `AdvanceClock` + the constraint + the store; it does not decide when to call them.
- **No RPC / proto changes.** The `GetUniverse` contract already carries `universe_time` (plan 23); no message changes
  here. Any new advance-related RPC surface (if recall needs one) is plan 30 / Epic C.
- **No forgetting, consolidation, semanticization, or recall math.** [F]/[C]/[R] are Epics C‚ÄìE. This unit only fixes
  that universe time is the "now" those derivations read; it computes none of them.
- **No temporal-link math.** The temporal-proximity synapse bonus ([L4]) and its window/coefficients live in
  [18.synapse-plasticity](18.synapse-plasticity.md) (`synapse.temporal_window_days`, `synapse.temporal_bonus`) and
  [21.link-usecase](21.link-usecase.md). This unit affirms time-as-link-signal; it does not re-own that math.
- **No UI.** The "Ïö∞Ï£ºÏùò ÏãúÍ∞Ñ" HUD, the acceleration animation, and the sync-confirm modal are

- **No stored positions or other emergent state** ([I5]) ‚Äî the clock is authoritative *state*, not a derived/emergent

## Design

### Placement (domain-first Go)
The universe clock is **core-domain behavior of the memory context** ‚Äî launching a memory advances it, recalling one
advances it, and the universe read returns it ‚Äî so it lives in `internal/memory`, not a new context (ARCHITECTURE ¬ß2.2,
be-architecture: keep tightly-coupled behavior in one context). The pure model (types + `AdvanceClock` + the constraint
predicate) sits in the context package; the `universe_state` table and its row‚Üîdomain mapping live in `memory/pg`, the
one package that touches sqlc/pgx (¬ß2.6). The repository **port** for the clock is declared by its consumer ‚Äî the
launch/recall use-cases in [30.time-advance-usecase](30.time-advance-usecase.md) (¬ß2.4 consumer-owned ports) ‚Äî so this
unit provides the concrete `memory/pg` implementation + queries, not a pre-declared implementation-side interface.

### The clock domain model (pure)
Universe time is a **DATE** ‚Äî a day on the diary timeline. Day granularity is deliberate and consistent with the whole
engine: memory dates (`created_universe_time`, `last_recalled_universe_time`), the semanticize timer ([C6a], "10 days"),
and synapse decay (`synapse.strength_decay_per_day`) are all counted in **universe-days**. A finer clock would carry no
signal the rest of the model can read.

- **`AdvanceClock(current, target) ‚Üí DATE`** = the monotonic advance. Returns `max(current, target)`; if `target` is
  not after `current`, the clock is unchanged. The clock **never decreases** ([I10]) ‚Äî there is no operation that moves
  it backward, by construction, so no caller can rewind time. Pure and deterministic.
- **The diary-date monotonic constraint ([T1]).** Given the current clock `C` and a diary date `D`:
  - `D ‚â• C` ‚Üí the diary **launches its star(s)**, and the clock advances to `AdvanceClock(C, D)` (= `D`, since `D ‚â• C`).
  - `D < C` ‚Üí the diary is **saved but launches no star** (past-dated). The `Diary` row always lands (readable in the
    archive / original reader, [I2]); no `EpisodicMemory` rises and the clock does not move.
  This is expressed as a pure predicate (`CanLaunchAt(diaryDate, clock) bool`) the use-case consults; the *actual*
  save-diary / skip-star / advance sequence is the use-case's transaction (plan 30), already seamed in plan 20.
- **Birth of the clock.** A user with no launches has **no** `universe_state` row and therefore **no** universe time
  (the universe is empty). The first star-launch advances the clock from "absent" to the first diary date ‚Äî the row is
  created lazily at that first advance (an upsert keyed by `user_id`). This keeps the empty-universe read returning a
  nil universe time exactly as Epic A does today, with no migration backfill needed.

The advance is **server-authoritative, single-writer state**, mutated only inside the launch/recall transactions
(plan 30). It is therefore **not** a golden-parity FE‚ÜîBE mirror: the frontend only *reads* the resulting DATE from
`GetUniverse` and feeds it as "now" to `packages/memory-logic`'s read-time fns ‚Äî it never advances the clock. (Contrast
[19.force-sim-layout](19.force-sim-layout.md), where the client re-derives positions and needs byte-parity; the clock
is a value the server hands down.)

### Universe time is a read-time "now", never a position axis ([T5][I7])
Fixing the clock as stored state makes universe time the **single read-time reference** ([T5], one clock) that all
elapsed-universe-day math reads:
- forgetting decay ([F1], Epic D): `now ‚àí last_recalled_universe_time` in universe-days ‚Üí effective brightness / decay
  stage;
- the semanticize timer ([C6a], Epic C/E): `now ‚àí semanticize_timer_reset_at` ‚Üí next gist stage;
- synapse effective strength ([L5], plan 18): `now ‚àí last_activated_universe_time` √ó `strength_decay_per_day`.

In every case universe time enters as a **scalar elapsed-days signal**, never as a coordinate. Layout (radius/angle/z)
comes from graph structure alone ([19.force-sim-layout](19.force-sim-layout.md), [I7][I5]); the clock has no path into
position. This is the [T5]/[I7] guardrail this unit makes structural: there is one clock, it is a date, and it is read,
not placed.

### The `universe_state` table (goose migration)
One forward-only-safe goose migration (with a matching down) under `apps/api/db/migrations/`, owned by this unit
(per [05.data-persistence](05.data-persistence.md): the unit adding a table owns its migration). It extends
[16.memory-aggregate-schema](16.memory-aggregate-schema.md) without reshaping any existing table.

```sql
-- universe_state ‚Äî the per-user, diary-driven universe clock ([T5]). Exactly one row per user; the clock is
-- authoritative stored state (NOT an emergent/derived value ‚Äî [I5] governs positions, not this). Advanced only by
-- star-launch (‚Üí diary date) and recall (‚Üí today) inside the memory use-case transactions ([T2]); monotonic ([I10]).
CREATE TABLE universe_state (
    user_id               TEXT PRIMARY KEY,        -- one clock per user [T5]
    current_universe_time DATE NOT NULL,           -- the last time the clock was advanced to [T6]
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Notes:
- **`user_id` is the primary key** ‚Äî one clock per user, so the advance is a single-row upsert with no race between
  concurrent launches (the row lock serializes them within the launch transaction).
- **`current_universe_time DATE`** ‚Äî day granularity, matching every other universe-time column.
- **No backfill migration.** Existing users (dev data only ‚Äî no production, per [00.overview](00.overview.md) ¬ß4) get a
  clock lazily on their next launch. **Decided (2026-07):** lazy birth + a **one-release read fallback** in plan 30
  (when the row is absent, the universe read falls back to `max(created_universe_time)` so no universe visibly resets)
  over a data-migration backfill ‚Äî simpler, and there is no production data to migrate.

### sqlc wiring & queries
Query files under `apps/api/db/queries/memory/**`, generated into `dbgen` via `pnpm gen:sql`; only `memory/pg` imports
`dbgen` (ARCHITECTURE ¬ß2.4 #4). The minimal surface the clock needs:
- **read the clock**: `SELECT current_universe_time FROM universe_state WHERE user_id = $1` (nil when absent).
- **advance the clock**: an idempotent upsert ‚Äî
  `INSERT INTO universe_state (user_id, current_universe_time) VALUES ($1, $2)
   ON CONFLICT (user_id) DO UPDATE SET current_universe_time = GREATEST(universe_state.current_universe_time, EXCLUDED.current_universe_time), updated_at = now()`
  ‚Äî the `GREATEST` mirrors `AdvanceClock` in SQL so a direct write can never rewind the clock either ([I10] enforced at
  two layers, like the synapse `CHECK` in plan 16). The domain still computes the advance; the `GREATEST` is
  defense-in-depth.

Every clock query takes and filters by the authenticated `user_id` (¬ß4, `lint:persistence`).

### Re-implementation note
The clock is fully specified by two rules ‚Äî *never move backward* (`AdvanceClock = max`) and *a diary before the clock
raises no star* (`CanLaunchAt = diaryDate ‚â• clock`) ‚Äî plus DATE (day) granularity. Any client/runtime reproduces it from
those rules; there is no tuning and no hidden state beyond the single per-user DATE.

## Acceptance Criteria
1. A goose migration creates `universe_state (user_id TEXT PRIMARY KEY, current_universe_time DATE NOT NULL, updated_at
   TIMESTAMPTZ NOT NULL DEFAULT now())`; `pnpm db:migrate` applies cleanly and `pnpm db:down` reverses cleanly.
2. `AdvanceClock(current, target)` returns `max(current, target)` and **never** returns a value earlier than `current`;
   there exists **no** operation (domain or SQL) that moves `current_universe_time` backward ([I10]).
3. The diary-date monotonic constraint is a pure predicate: a diary dated **‚â•** the current clock launches (and the
   clock advances to that date); a diary dated **<** the clock is **saved but launches no star** ([T1]). (Enforcing it
   at launch is plan 30; this unit provides the rule.)
4. Universe time is a **DATE** (day granularity) and is the model's single **read-time "now"** ‚Äî a scalar elapsed-days
   reference for [F]/[C]/[L] derivations and **never** a position/coordinate ([T5][I7][I5]).
5. A user with no launches has **no** `universe_state` row and reads a **nil** universe time; the row is created lazily
   on the first advance (upsert keyed by `user_id`), so the empty-universe read is unchanged from Epic A.
6. The clock advance is a **single-row upsert** with `GREATEST(...)` (SQL-level monotonic guard mirroring `AdvanceClock`),
   so concurrent launches cannot rewind the clock or create duplicate rows.
7. `pnpm gen:sql` generates the clock queries into `dbgen`; only `internal/memory/pg` imports `dbgen`, and no sqlc row
   type appears in the clock domain model or context behavior (ARCHITECTURE ¬ß2.4 #4).
8. Every clock read/write is **scoped to the authenticated user** and `pnpm lint:persistence` passes ([U1], ¬ß4).
9. Repository interfaces are **not** pre-declared here (they are consumer-owned by plan 30); `memory/pg` returns the
   concrete clock store + queries the use-case will bind to.
