# 41. 초대 코드 멤버십 게이트 — 사인인 비주얼 + 가입 게이트 + 발행 콘솔

> 현재 구현 기준 문서(as-built). 닫힌 베타 동안 "초대 코드를 한 번 redeem해야 우주에 들어온다"는 가입 게이트를, 실제 인증(01) 위에 **제거 가능한 한 겹**으로 얹는다. 동시에 빈약한 `/sign-in`을 랜딩(15)의 우주 비주얼·3D 별 로고로 다시 입히고(카드 폐지), `/admin`(34)을 탭으로 갈라 초대 코드 발행·관리를 추가한다. | 범위: FS | 상태: 구현 완료 | 값 SSOT: [policy/domain/access.md](../policy/domain/access.md)·values.yaml `invite`

## 목적

지금 `/sign-in`은 `GlassCard` 한 장에 "cosimosi" 텍스트와 입력 폼만 있는 빈약한 화면이다(랜딩 15의 우주 비주얼과 단절). 그리고 인증만 통과하면 누구나 개인 우주에 들어온다 — 닫힌 베타에 필요한 **가입 통제 수단이 없다**.

41은 두 가지를 푼다.

1. **첫인상.** `/sign-in`을 랜딩과 같은 세계로 보이게 한다 — 별이 부드럽게 반짝이는 배경 + "cosimosi" 워드마크. 로그인 자체는 기존(01) 그대로 이메일 OTP·Google OAuth를 쓴다.
2. **초대 게이트.** 새 사용자는 **최초 로그인 직후 초대 코드 입력 페이지(`/invite`)로 리다이렉트**되고, 유효한 코드를 redeem해야 비로소 멤버가 되어 우주에 입장한다. 게이트는 **서버가 강제**한다(멤버십 인터셉터) — 프런트 우회(Supabase 직접 호출)로 뚫리지 않는다. 코드 발행은 지금은 관리자만(`/admin` 새 탭), **나중에 일반 사용자도 발행**할 수 있도록 발행 권한과 발행 표면을 처음부터 분리해 둔다.

핵심 설계 원칙은 **제거 가능성**이다. 초대 게이트는 베타가 끝나면 사라질 수 있으므로, 도메인(`internal/invite`)·proto 서비스·DB 테이블·FE 페이지 슬라이스·서버 인터셉터를 한 덩어리로 격리하고, **하나의 env 플래그(`INVITE_GATE_ENABLED`)로 통째로 끌 수 있게** 한다. 끄면 코드를 지우기 전이라도 전원이 멤버로 통과한다.

> Q1 인터뷰 결정(중요): 초대 코드는 **사인인 화면 위가 아니라** 최초 로그인 후 별도 페이지로 입력받는다. 따라서 원래 구상한 "사인인 화면의 초대 버튼·비활성 로그인 버튼"은 폐기하고, "하나하나 입력 시 화려한 인터랙션 + confirm 후 환영"이라는 연출은 전용 `/invite` 페이지로 옮긴다. 사인인 화면의 로그인 수단(이메일 OTP·Google)은 전부 활성이다.

## 범위 / 비목표

**범위**
- `/sign-in`(`frontend/src/app/ui/SignInScreen.tsx`) 비주얼 리바인드: 랜딩(15) 별·반짝임 배경 + "cosimosi" 워드마크 재사용. 로그인 로직(01)은 불변.
- 새 보호 라우트 `/invite`: 인증됐으나 미멤버인 사용자가 초대 코드를 입력·redeem하는 전용 온보딩 페이지(글자 단위 입력 인터랙션 + redeem 성공 시 환영 연출).
- 멤버십 개념과 **서버 강제 게이트**: 코어 앱 RPC(Memory·Settings·Share·Gift)는 멤버만 호출 가능. 비멤버는 `/invite`로 보내는 신호만 받는다.
- 초대 코드 도메인(`backend/internal/invite`) + proto 2서비스(`InviteService`·`InviteAdminService`) + 테이블 2종(`invite_codes`·`invite_redemptions`).
- 초대 코드 발행 모델(직교): `max_uses`(1=1회용 / 무제한) × `expires_at`(지정 / 없음). UI 프리셋 = 1회용·시간지정·무제한.
- `/admin` 탭 분리: **LLM 관리 탭**(기존 34 콘솔 그대로) + **초대 코드 탭**(발행 폼 + 발행 목록 + revoke + 사용 현황 + 라벨/메모 + 발행인 표시 + 복사).
- `INVITE_GATE_ENABLED` env 플래그로 게이트 전체 on/off.

**비목표 (이번에 안 함)**
- **일반 사용자의 초대 코드 발행** — 발행 권한·표면은 분리해 두되, 실제 사용자 발행 RPC 노출은 후속(권한 정책·발행 한도 설계 필요). 지금 발행은 관리자 전용.
- **초대 코드로 추천인/리워드 연결**(레퍼럴 그래프) — redeem은 멤버십만 부여하고 발행인↔가입자 관계를 기능화하지 않는다(`invite_redemptions`가 누가 어떤 코드를 썼는지는 남기므로 후속 확장 여지만 둔다).
- **이메일 OTP·Google OAuth 자체 변경** — 01 그대로. 인증 실패/성공 흐름을 손대지 않는다.
- **초대 코드 키 회전·만료 스윕 잡** — 만료는 조회·redeem 시점 판정으로 충분(별도 정리 잡 없음).
- **사인인 화면에 무거운 R3F 별(ThemedStar) 필수화** — 성능·모바일 고려해 기본은 경량 `StarFieldCanvas`. ThemedStar는 추출 시 선택적 강조용(아래 설계 참고).

## 설계 요점

### 1) 전체 흐름

```
[비로그인]  /  ·  /gift/$token  →  SessionGate  →  /sign-in?redirect=<원래경로>
[사인인 화면]  이메일 OTP  ·  Google OAuth   (모두 활성, 01 그대로)
        │  인증 성공(JWT 수립)
        ▼
[멤버십 판정]  GetMembershipStatus()
        ├─ is_member=true  →  원래 경로(redirect) 또는 `/` 우주 입장
        └─ is_member=false →  /invite?redirect=<원래경로> 로 리다이렉트
                                   │  초대 코드 입력 → confirm
                                   ▼
                              RedeemInviteCode(code)   (인증 필요·원자적 소비+멤버십 부여)
                                   ├─ ok=true  →  환영 연출 → redirect(없으면 `/`)
                                   └─ ok=false →  사유별 에러 연출(흔들림 등), 코드 미소비
```

- 사인인 화면은 **초대 코드를 다루지 않는다**(인증 전이라 redeem할 JWT가 없다). 초대 코드는 인증 후 `/invite`에서만 입력한다 — confirm·redeem 모두 인증된 호출이라 단순하고, 게이트를 사인인에서 완전히 분리한다(제거성).
- 기존 멤버는 `/invite`를 거치지 않는다(멤버십 판정에서 바로 통과).
- **초대 URL 온보딩(change 05):** 관리자가 `/admin`에서 **초대 URL(`${origin}/invite?code=<code>`)을 복사/공유**(Web Share + 복사 폴백)한다. 받은 사람이 `/invite?code=`를 열면 미인증도 초대장 화면 + `회원가입하기`를 먼저 보고(코드 없는 미인증만 사인인으로), 가입 시 코드를 sessionStorage에 stash해 **OTP·Google OAuth 어느 경로로도 코드가 보존**된다. 인증 비멤버는 수동 입력 없이 **자동 redeem**(실패 시 수동 폴백), 이미 멤버면 무소비 통과. 라우트 분기는 `app/ui/InviteRoute`(세션). 서버 발송 RPC·DB·게이트 변경은 없다(기존 `RedeemInviteCode` 재사용). 상세 규칙은 [access.md](../policy/domain/access.md) §FE 표면.

### 2) 사인인·초대 비주얼 (`/sign-in`·`/invite`) — 카드 없이 배경 위에

- **백드롭:** `shared/ui/CosmosBackdrop` — `StarFieldCanvas`(shared) + CSS 방사 그라디언트 + 부드럽게 도는 보랏빛·청록빛 성운 글로우 + `GrainOverlay`. `GrainOverlay`는 랜딩에서 `shared/ui`로 승격하고 `.ld-grain` 베이스 opacity에 폴백을 더해 랜딩 테마 토큰 밖에서도 과하지 않게 한다. 상시 미세 모션, `prefers-reduced-motion`이면 멎는다([feedback_landing_visual_prefs] 준수, glow 비절단).
- **3D 별 로고:** 우리 별 오브제를 로고처럼 쓴다 — `widgets/star3d/BrandMark`(별 형태=`appearance.object`, 색=테마 accent) + "cosimosi" 워드마크. **카드(GlassCard) 없이** 백드롭 위에 로고·입력·버튼이 그대로 뜬다(랜딩 히어로와 같은 결). "내 마음 태양계" 부제는 제거.
- **레이어 이동(헌법4):** `SignInScreen`(app)·`InvitePage`(pages)가 같은 3D 로고를 공유하려면 `pages/landing` 내부를 끌어오면 안 되므로(app→pages는 되지만 pages→pages 금지), 랜딩의 `star3d`(ThemedStar·StarCanvas·Star3D·halo·star-canvas-context)를 **`widgets/star3d`로 이동**하고 그 안에 `BrandMark`를 둔다(widgets→entities 허용). 랜딩의 `use-in-view` 훅은 `widgets/star3d`로 동반 이동한다(React 훅이라 `shared/lib`엔 못 둔다 — 헌법4 순수 레이어). 랜딩 `HeroSection`은 import 경로만 갱신.
- 로그인 폼(이메일·코드·Google)은 **전부 활성**, 동작은 01 불변 — 비주얼만 백드롭 위에 올렸다.

### 3) 멤버십 게이트 — 서버 강제 (핵심)

게이트가 의미를 가지려면 FE UX가 아니라 **서버가 강제**해야 한다(Q2 결정). 35의 "노출 수위가 다른 service를 별도 인터셉터 체인으로 마운트" 패턴을 그대로 확장해, `server.go`에 체인을 셋으로 나눈다.

| 체인 | 인터셉터 | 마운트 service | 멤버십 필요 |
|---|---|---|---|
| 공개(무인증) | Logging (+Recover, readcap) | `VisitService`(35) | — |
| auth-only | Logging→Auth | **`InviteService`** | ✗ (게이트 통과 수단이므로 멤버십 불필요) |
| auth+membership | Logging→Auth→**Membership** | `MemoryService`·`SettingsService`·`ShareService`·`GiftService` | ✓ |
| auth+admin | Logging→Auth→AdminGate | `AdminService`(34)·**`InviteAdminService`** | ✗ (관리자는 부트스트랩상 멤버가 아닐 수 있음) |

- **멤버십 인터셉터** `backend/internal/platform/rpcserver/membership_gate.go`: `UserIDFromContext`로 `sub`를 얻어 `invite_redemptions`에 행이 있으면 통과, 없으면 `PermissionDenied`(불투명 메시지). 미인증은 그 앞 Auth 인터셉터가 `Unauthenticated`로 먼저 끊는다. **`ADMIN_USER_IDS` 관리자는 `IsAllowlistedAdmin`(admin_gate와 공유)으로 면제** — 초대 코드 없이 코어 우주에 통과하고 `GetMembershipStatus`도 관리자에겐 `is_member=true`를 돌려준다(게이트를 부트스트랩하는 주체라 자기 우주에서 잠기면 안 됨). **35의 `opts` 슬라이스를 앨리어싱하지 말고 복제 후 누적**(34 admin_gate 선례) — 멤버십이 실린 슬라이스가 다른 service로 새지 않게.
- **체인 누적 순서:** auth+membership = (공유 auth opts 복제) + `WithInterceptors(membership)`. auth+admin = (공유 auth opts 복제) + `WithInterceptors(adminGate)`. **admin 체인에는 멤버십을 얹지 않는다** — 첫 관리자가 코드를 발행하려면 멤버 아닌 상태로 `/admin`·`InviteAdminService`에 닿아야 한다.
- 멤버십 룩업은 인터셉터마다 1쿼리(`EXISTS`)다. 베타 규모에서 무시 가능. (원하면 요청 컨텍스트 캐시는 후속.)
- **제거성 플래그:** `INVITE_GATE_ENABLED=false`면 멤버십 인터셉터를 체인에 **추가하지 않고**(no-op), `GetMembershipStatus`는 인증된 전원에게 `is_member=true`를 돌려준다 → FE가 `/invite`로 보내지 않는다. 베타 종료 시 플래그만 끄면 게이트가 사실상 사라지고, 코드·테이블 삭제는 그 다음 단계.
- 회귀 가드: `server_test.go`에 `TestMembershipBoundary` — 멤버십 없는 토큰으로 Memory/Settings/Share/Gift는 `PermissionDenied`, `InviteService`(Validate/Redeem/GetMembershipStatus)는 핸들러에 도달. 35의 `TestUnauthBoundary`와 동거.

### 4) 초대 코드 데이터 모델 (직교 + 프리셋, Q3)

코드는 **두 독립 축**으로 저장한다: 사용 횟수 상한(`max_uses`)과 만료(`expires_at`). UI의 1회용/시간지정/무제한은 이 두 축의 프리셋일 뿐 — 나중에 "10회 + 7일" 같은 조합도 스키마 변경 없이 가능(확장성·관심사 분리).

| 프리셋 | max_uses | expires_at |
|---|---|---|
| 1회용 | `1` | NULL |
| 시간지정 | NULL(또는 지정값) | now + ttl |
| 무제한 | NULL | NULL |

`invite_redemptions`는 **사용자당 1행**으로 멤버십 마커이자 사용 내역을 겸한다(누가 어떤 코드를 언제 redeem했는지). `used_count`는 캡 판정용으로 `invite_codes`에 비정규화 저장하고, 사용 현황 목록은 `invite_redemptions`에서 코드별로 집계한다.

### 5) 마이그레이션 — `backend/internal/db/migrations/00009_invite_codes.sql`

```sql
-- +goose Up
CREATE TABLE invite_codes (
    id          TEXT PRIMARY KEY,              -- crypto/rand base64url 22자(내부 id, 35 newSlug 규약 재사용)
    code        TEXT NOT NULL UNIQUE,          -- 사람이 입력하는 코드(모호문자 제외 대문자 영숫자, 길이 = values invite.code_length)
    label       TEXT NOT NULL DEFAULT '',      -- 발행 메모('' = 없음)
    created_by  TEXT NOT NULL,                 -- 발행인 user_id(현재 관리자 sub; 후속 일반 사용자)
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at  TIMESTAMPTZ,                   -- NULL = 만료 없음
    max_uses    INTEGER,                       -- NULL = 무제한; CHECK(max_uses IS NULL OR max_uses > 0)
    used_count  INTEGER NOT NULL DEFAULT 0,
    revoked_at  TIMESTAMPTZ,                   -- NULL = 활성
    CONSTRAINT invite_codes_max_uses_pos CHECK (max_uses IS NULL OR max_uses > 0)
);
CREATE INDEX invite_codes_created_by_idx ON invite_codes (created_by);

CREATE TABLE invite_redemptions (
    user_id        TEXT PRIMARY KEY,           -- 사용자당 1행 = 멤버십
    invite_code_id TEXT NOT NULL REFERENCES invite_codes(id),
    redeemed_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX invite_redemptions_code_idx ON invite_redemptions (invite_code_id);

-- +goose Down
DROP TABLE invite_redemptions;
DROP TABLE invite_codes;
```

`schema.sql` 스냅샷에도 Up DDL을 평탄화 반영하고 `pnpm gen:sql`로 `db/gen/invite.sql.go`를 생성한다. 마이그레이션 번호는 `00009`(00008 star_gifts 다음). 원본 일기·별과 무관한 독립 테이블이라 헌법1/2와 충돌 없음.

### 6) 검증·소비 로직 (`backend/internal/invite`)

- **`RedeemInviteCode`(원자적·경쟁 안전):** 트랜잭션 안에서 `SELECT ... WHERE code=$norm FOR UPDATE`로 행 잠금 → 판정(`revoked_at IS NULL` ∧ (`expires_at IS NULL` ∨ `expires_at > now()`) ∧ (`max_uses IS NULL` ∨ `used_count < max_uses`)). **멱등:** 호출자가 이미 `invite_redemptions`에 있으면(빠른 사전 체크) 소비 없이 `ok=true`(이미 멤버). 통과 시 `INSERT invite_redemptions ON CONFLICT (user_id) DO NOTHING` + 영향 행 수로 `used_count` 증가를 가른다. `FOR UPDATE` 잠금이 같은 코드의 동시 redeem을 직렬화해 1회용 중복 소비를 막고, **서로 다른 코드를 같은 사용자가 동시에 redeem하는 경쟁**(코드 잠금이 직렬화 못 함)은 INSERT의 0행(=이미 멤버)으로 PK 위반(`CodeInternal`) 대신 멱등 `ok`가 되며 둘째 코드는 소비되지 않는다(code-review·codex 공동 지적 반영). 발행 시 `max_uses`·`ttl_seconds`는 양수여야 한다(0/음수 → `InvalidArgument`, 즉시 만료/무효 코드 방지).
- **`ValidateInviteCode`(비소비, NO_SIDE_EFFECTS):** 같은 판정을 소비 없이 수행해 `{valid, reason}`을 돌려준다. `/invite`의 인라인 사전 검사·confirm 버튼 활성화·환영 연출 사전 게이트용. **권위는 항상 Redeem**(Validate 통과 후 Redeem이 다시 원자 재검).
- **사유 enum** `InviteReason { OK, NOT_FOUND, EXPIRED, EXHAUSTED, REVOKED }` — FE가 사유별 카피를 띄운다(error-feedback 준수). 단, 존재 비노출이 필요하면 NOT_FOUND/EXPIRED/REVOKED를 동일 카피로 합칠 수 있다(베타에선 사유 노출이 친절 — 정책에서 결정).
- 코드 정규화: 입력은 대문자화·공백 제거 후 대조(`code`는 정규형으로 저장). 모호문자(0/O/1/I/L 등)를 제외한 알파벳은 **코드 콘텐츠**라 values가 아니라 `internal/invite`의 상수표에 둔다(테마 CSS·mood 색표가 코드에 남는 것과 같은 원칙).

### 7) proto 계약 — `proto/cosimosi/v1/invite.proto`

노출 수위가 다른 **2 service**로 분리(35 ShareService/VisitService 선례). DTO 전용·unary 전용.

- `InviteService` (auth-only 체인 — 멤버십 불필요):
  - `GetMembershipStatus() → {bool is_member}` (NO_SIDE_EFFECTS) — FE 게이트가 호출.
  - `ValidateInviteCode(code) → {bool valid, InviteReason reason}` (NO_SIDE_EFFECTS).
  - `RedeemInviteCode(code) → {bool ok, InviteReason reason}` — 원자 소비+멤버십.
- `InviteAdminService` (auth+admin 체인 — admin allowlist 뒤):
  - `IssueInviteCode({string label, optional int32 max_uses, optional int64 ttl_seconds}) → {InviteCode}` — `ttl_seconds` 있으면 `expires_at = now + ttl`(절대시각 대신 TTL로 클라 시계 의존 제거). `max_uses` 생략 = 무제한.
  - `ListInviteCodes({}) → {repeated InviteCode}` (NO_SIDE_EFFECTS) — 발행 목록(created_at desc). 현재 관리자는 전체.
  - `RevokeInviteCode({string id}) → {InviteCode}` — `revoked_at = now`. 갱신된 행 반환.
- 메시지 `InviteCode`: `id`·`code`·`label`·`created_by`(발행인)·`created_at`·`expires_at`·`max_uses`·`used_count`·`status`(서버 계산 enum `ACTIVE|EXPIRED|EXHAUSTED|REVOKED`). 발행인 표시(Q4)는 `created_by`로 충족.

> **후속 사용자 발행 경로(비목표지만 분리해 둠):** 일반 사용자 발행을 열 땐 `IssueInviteCode`/"내 발행 목록(created_by=self)"을 `InviteService`(auth) 쪽으로 옮기고 발행 한도 정책을 얹으면 된다 — 도메인 `invite.Service`는 그대로 두고 핸들러 표면만 바꾼다(handler 2개가 같은 service를 호출, 35 Handler가 Share/Visit 둘 다 구현하는 구조와 동일).

## 수용 기준 (인수 조건)

1. `/sign-in`은 랜딩(15)과 같은 별·반짝임 배경 + "cosimosi" 워드마크를 렌더하고, 이메일 OTP·Google 로그인 수단은 전부 활성이며 동작은 01과 동일하다(reduced-motion이면 모션 폴백).
2. 멤버가 아닌 사용자가 로그인에 성공하면 `/invite?redirect=<원래경로>`로 리다이렉트된다. 이미 멤버인 사용자는 `/invite`를 거치지 않고 `redirect`(없으면 `/`)로 입장한다.
3. `/invite`에서 유효한 코드를 confirm하면 `RedeemInviteCode`가 성공하고 환영 연출 후 `redirect`(없으면 `/`)로 입장한다. 무효 코드는 소비되지 않고 사유별 에러를 보인다.
4. **서버 강제:** 멤버십이 없는 인증 토큰으로 `MemoryService`·`SettingsService`·`ShareService`·`GiftService`의 RPC를 호출하면 `PermissionDenied`를 받는다. 같은 토큰으로 `InviteService`(GetMembershipStatus·ValidateInviteCode·RedeemInviteCode)는 핸들러에 도달한다. 미인증은 `Unauthenticated`다.
5. 1회용 코드(`max_uses=1`)는 첫 redeem 후 두 번째 redeem이 `EXHAUSTED`로 거부된다. 같은 코드를 동시에 redeem해도(경쟁) 정확히 1회만 소비된다(`FOR UPDATE` 직렬화).
6. 시간지정 코드는 `expires_at` 이후 redeem이 `EXPIRED`로 거부되고, 그 전에는 (무제한이면) 여러 번 redeem된다.
7. 무제한 코드(`max_uses` NULL·`expires_at` NULL)는 횟수·시간 제한 없이 redeem된다.
8. `RedeemInviteCode`는 멱등하다 — 이미 멤버인 사용자가 다시 호출하면 `used_count`를 늘리지 않고 `ok=true`를 반환한다.
9. `RevokeInviteCode` 후 그 코드의 redeem은 `REVOKED`로 거부된다.
10. 발행된 코드 목록(`ListInviteCodes`)은 각 코드의 코드값·상태·`used_count`/`max_uses`·만료·**발행인(`created_by`)**·라벨을 포함한다.
11. `InviteAdminService`의 모든 RPC는 `ADMIN_USER_IDS` allowlist 뒤에 있다 — 비관리자/미인증은 발행·목록·revoke에 접근할 수 없다(34와 동일 게이트, 멤버십 무관).
12. `/admin`은 "LLM 관리"·"초대 코드" 두 탭으로 나뉘고, LLM 관리 탭의 기능(키·모델·활성 선택·대시보드)은 34와 동일하게 동작한다(회귀 0). `?tab=`으로 딥링크된다.
13. `INVITE_GATE_ENABLED=false`이면 멤버십 게이트가 강제되지 않고(`GetMembershipStatus`가 인증 전원에 `is_member=true`), 코어 RPC가 멤버십 없이 통과하며 FE는 `/invite`로 보내지 않는다.
14. 관리자는 멤버십(redeem) 없이도 `/admin`과 `InviteAdminService`로 첫 코드를 발행할 수 있다(admin 체인에 멤버십 게이트 부재).
15. 도메인 타입(`internal/invite`)에는 db/json/proto 태그가 없다(헌법5). 좌표·원본 일기와 무관한 독립 테이블이라 헌법1/2를 건드리지 않는다.
