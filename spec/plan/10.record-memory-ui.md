# 10. record-memory-ui — 작성 폼 → 별 등장 (수직 슬라이스)

> 작성 폼 제출로 RecordMemory를 호출해 별이 즉시 떠오르고, 홈 진입 시 GetUniverse로 우주를 로드하는 첫 끝-끝 수직 슬라이스. | 범위: MVP | 선행: 02, 04, 08 | 차단: 11

## 목적 (Why)
지금까지의 백엔드(02 transport, 04 RPC)와 프론트 렌더(08 별)를 **하나의 사용자 흐름으로 잇는다.** 사용자가 일기를 쓰면 `RecordMemory`로 원본이 영구 저장되고, 화면에는 새 별이 **낙관적으로 즉시** 떠오른다. 홈에 진입하면 `GetUniverse`로 기존 우주를 받아 렌더한다. 이 스펙이 끝나면 "쓰면 별이 생기고 새로고침해도 남는다"는 코어 루프의 첫 절반이 동작한다.

## 범위
- **포함:**
  - `features/record-memory`: 작성 폼 UI(본문·감정·강도·날짜), `RecordMemory` 호출 훅, 제출→낙관적 `addStar`→실패 롤백(`removeStar`).
  - `entities/memory/api`: `GetUniverse` 응답의 `stars[]` → 도메인 `StarNode[]` 매퍼, `setStars`로 스토어 적재(`synapses`는 무시).
  - `pages/home`: 폼 + `universe-canvas` 조립, **마운트 시 `GetUniverse` 1회 로드**.
- **비목표(Out of scope):**
  - 회상·공동 회상 강화 → **11(recall-reinforce)**.
  - 감쇠·최소 밝기·잠든 별 → **12(decay-dormant)**.
  - 별 InstancedMesh·TSL 렌더 자체 → **08(star-rendering)**(여기선 스토어에 넣을 뿐).
  - 시냅스 선 렌더 → **09(synapse-rendering)**. 임베딩·연결 생성(워커) → **05**(여기선 호출 안 함, 잠시 후 워커가 채운 연결은 다음 `GetUniverse`에서 보임).
  - 인증 토큰 부착 → **02**의 인터셉터가 담당(여기선 생성된 Connect 클라를 가져다 씀).

## 설계 요점
**계약(02/04 산출 gen 타입 사용 — 여기서 proto를 새로 정의하지 않음).**
- `RecordMemoryRequest { body: string; mood?: Mood; intensity?: number; entryDate: string }` (entryDate = `YYYY-MM-DD`).
- `RecordMemoryResponse { memory_id: string }` — 04가 원본 저장 후 **별 객체가 아니라 `memory_id`만** 반환한다("별 생성 중", 연결은 아직 없음). body/mood는 클라가 폼 값으로 보유.
- `GetUniverseResponse { stars: Star[]; synapses: Synapse[] }` — `stars`만 매핑한다. `synapses`는 09/11 소관이라 **여기선 무시**한다. (`memories`/`links` 필드는 없음 — 02/04 실제 계약은 `stars`/`synapses`.)
  - `Star { memory_id: string; mood: Mood; intensity: number; last_recalled_at: string }` — 별 시각 입력. **body·entry_date·created_at 없음**(불변 원본은 records, RecallMemory(11)로 회상).

**도메인 타입(순수, three/React 미의존 — 헌법 4).** `entities/memory/model/types.ts`는 **08이 단일 권위**다. 본 스펙은 도메인 타입을 새로 정의하지 않고 08 정의를 그대로 따른다(08은 본 스펙의 선행).
```ts
// 08 정의(권위, 본 스펙에서 변경 금지)
export interface Memory {
  id: string; mood: Mood; intensity: number;
  lastRecalledAt: number /* epoch ms */; seed: number;
}
export interface StarNode { id: string; memory: Memory; index: number; }
```
> ⚠️ 08 `Memory`에는 **body·entryDate·createdAt가 없다**(헌법 1 + Architecture §2.7 — 불변 원본은 렌더 스토어에 넣지 않는다). 작성 중 본문/날짜는 `features/record-memory/model` 드래프트에만, 회상 시 원본은 RecallMemory(11) 응답 Record를 패널에 표시한다.

**매퍼(`entities/memory/api/map-star.ts`).** proto `Star` → 도메인 `StarNode`. **별 좌표는 매핑하지 않는다**(헌법 3: 좌표는 클라 force-sim에서 창발). 매퍼는 순수 함수(three 미의존). `seed`는 **`entities/memory`의 공유 헬퍼 `seedFromId(memory_id)`**(08 소유, 단일 출처)로 파생하고 — 인라인 해시 금지 — `lastRecalledAt`는 `last_recalled_at`(ISO)→epoch ms 변환, `mood`는 proto `Mood`→7종 매핑. `index`는 스토어 적재 순서 슬롯.

**낙관적 흐름(`use-record-memory.ts`).** Zustand 액션 사용(**StarNode 기준**):
1. 제출 → 임시 id(`temp-${crypto.randomUUID()}`)로 폼 값(mood·intensity)과 `seed=seedFromId(tempId)`로 도메인 `StarNode` 구성 → `addStar(optimistic)` → 별 즉시 등장.
2. `client.recordMemory(req)` 호출 → `res.memory_id` 수신.
3. 성공 → `res.memory_id`+폼 값으로 확정 `StarNode` 구성(`seed=seedFromId(memory_id)`로 재파생 — 별이 저장 시 한 번 "굳는" 효과, 허용) → `replaceStar(tempId, node)`(서버 id로 교체, force-sim은 기존 별 pin 유지 — 헌법 3).
4. 실패 → `removeStar(tempId)` 롤백 + 에러 카피 노출.
> body/entryDate는 응답에 없고 클라 폼 값으로만 보유한다(렌더 스토어 미적재). 신규 별의 시냅스 연결은 워커가 비동기로 채우므로 **다음 `GetUniverse` refetch에서 본다**(폴링/스트리밍 아님 — 헌법 6).

**스토어 계약(`entities/memory/model/store.ts`, 08이 권위 — 본 스펙은 액션만 확장).** 08이 만든 `stars: StarNode[]; selectedId; setStars; select`를 **재정의하지 않고**, 낙관적 흐름에 필요한 3개 액션만 추가한다.
```ts
// 08 정의(권위) + 10이 추가하는 액션
interface MemoryState {
  stars: StarNode[]            // 08: 별 목록(StarField가 의존)
  selectedId: string | null    // 08
  setStars: (s: StarNode[]) => void   // 08: GetUniverse 적재
  select: (id: string | null) => void // 08
  // ── 10 추가(StarNode 기준) ──
  addStar: (node: StarNode) => void           // 낙관적 등장
  replaceStar: (tempId: string, node: StarNode) => void
  removeStar: (tempId: string) => void
}
```
> ⚠️ `removeStar`는 **낙관적 임시 별만**(id가 `temp-` 접두인 경우) 제거한다. 서버에서 온 별은 절대 삭제하지 않는다(헌법 2 — 우주에서 별 삭제 금지). `memories[]`/`Memory[]` 스토어로 **재정의 금지**(08의 `stars: StarNode[]`가 단일 진실, StarField는 `store.stars`/`StarNode.index` 슬롯에 의존).

**조립.** `HomePage`가 `<UniverseCanvas/>`(별 렌더) + `<MemoryForm/>`(2D HUD, 캔버스 밖 DOM — Architecture §3.1 "씬 안 DOM 금지")를 함께 렌더. 마운트 시 `GetUniverse` 1회.

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 사용자가 본문이 비어있지 않은 폼을 제출 THE SYSTEM SHALL 임시 id(`temp-`)의 `StarNode`를 폼 값과 `seed=seedFromId(tempId)`로 구성해 `addStar`로 스토어에 즉시 추가하고 `RecordMemory`를 호출한다.
1.2 WHEN `RecordMemory`가 `memory_id`를 반환 THE SYSTEM SHALL `memory_id`+폼 값으로 확정 `StarNode`(`seed=seedFromId(memory_id)`)를 구성해 `replaceStar(tempId, node)`로 교체하고 기존 별 위치는 유지한다(force-sim pin — 헌법 3).
1.3 IF `RecordMemory`가 실패(네트워크/서버 오류) THEN THE SYSTEM SHALL `removeStar(tempId)`로 임시 별만 롤백 제거하고 한국어 에러 카피를 표시하며, 이미 존재하던 서버 별은 보존한다(헌법 2).
1.4 IF 본문이 공백뿐 THEN THE SYSTEM SHALL 제출을 막고 "일기 본문을 입력하세요" 안내를 표시한다(RPC 미호출).
1.5 WHEN 날짜 미선택 THE SYSTEM SHALL 기본값 오늘(`YYYY-MM-DD`, 로컬)로 `entryDate`를 채운다(concept §시점: 기본 오늘, 과거 지정 가능). `entryDate`/`body`는 `features/record-memory/model` 드래프트에만 보유하고 렌더 스토어(08 `StarNode`/`Memory`)에는 넣지 않는다.
2.1 GIVEN 홈 페이지 마운트 WHEN 컴포넌트가 렌더 THE SYSTEM SHALL `GetUniverse`를 1회 호출해 응답 `stars[]`를 `StarNode[]`로 매핑·`setStars`로 적재하고 별로 렌더한다(`synapses`는 무시).
2.2 GIVEN 별을 작성한 뒤 새로고침 WHEN 홈이 다시 마운트 THE SYSTEM SHALL `GetUniverse` 응답 `stars[]`에 그 별을 포함해 렌더한다(원본 영속 — 헌법 1).
2.3 WHEN 신규 별 저장 후 워커가 임베딩·연결을 채운 뒤 다음 `GetUniverse` refetch THE SYSTEM SHALL 그 별의 시냅스 연결을 `synapses`로 받는다(폴링/스트리밍 아님 — 헌법 6; 본 스펙은 `synapses`를 적재하지 않고 11이 소비).
2.4 IF `GetUniverse` 응답의 `stars[]`가 빈 배열(별 0개) THEN THE SYSTEM SHALL 빈 우주 빈 상태 카피(예: "아직 별이 없어요. 첫 일기를 적어 첫 별을 띄워보세요")를 표시하고 폼 작성을 유도하며, force-sim은 0노드에서 크래시 없이 동작한다.
3.1 THE SYSTEM SHALL 매퍼에서 별 좌표를 만들지 않는다(서버 응답에 좌표 없음, 좌표는 force-sim 창발 — 헌법 3).
3.2 THE SYSTEM SHALL `entities/memory/api`·`model`에서 `three`/React/DOM을 import하지 않는다(헌법 4).
