# 10. record-memory-ui — 작성 폼 → 별 등장 (수직 슬라이스)

> 작성 폼 제출로 RecordMemory를 호출해 별이 즉시 떠오르고, 홈 진입 시 GetUniverse로 우주를 로드하는 첫 끝-끝 수직 슬라이스. | 범위: MVP | 선행: 02, 04, 08 | 차단: 11

## 목적 (Why)
지금까지의 백엔드(02 transport, 04 RPC)와 프론트 렌더(08 별)를 **하나의 사용자 흐름으로 잇는다.** 사용자가 일기를 쓰면 `RecordMemory`로 원본이 영구 저장되고, 화면에는 새 별이 **낙관적으로 즉시** 떠오른다. 홈에 진입하면 `GetUniverse`로 기존 우주를 받아 렌더한다. 이 스펙이 끝나면 "쓰면 별이 생기고 새로고침해도 남는다"는 코어 루프의 첫 절반이 동작한다.

## 범위
- **포함:**
  - `features/record-memory`: 작성 폼 UI(본문·감정·강도·날짜), `RecordMemory` 호출 훅, 제출→낙관적 `addMemory`→실패 롤백.
  - `entities/memory/api`: `GetUniverse` 응답 → 도메인 `Memory[]` 매퍼, `setMemories`로 스토어 적재.
  - `pages/home`: 폼 + `universe-canvas` 조립, **마운트 시 `GetUniverse` 1회 로드**.
- **비목표(Out of scope):**
  - 회상·공동 회상 강화 → **11(recall-reinforce)**.
  - 감쇠·최소 밝기·잠든 별 → **12(decay-dormant)**.
  - 별 InstancedMesh·TSL 렌더 자체 → **08(star-rendering)**(여기선 스토어에 넣을 뿐).
  - 시냅스 선 렌더 → **09(synapse-rendering)**. 임베딩·연결 생성(워커) → **05**(여기선 호출 안 함, 잠시 후 워커가 채운 연결은 다음 `GetUniverse`에서 보임).
  - 인증 토큰 부착 → **02**의 인터셉터가 담당(여기선 생성된 Connect 클라를 가져다 씀).

## 설계 요점
**계약(02/04 산출 gen 타입 사용 — 여기서 proto를 새로 정의하지 않음).**
- `RecordMemoryRequest { body: string; mood?: string; intensity?: number; entryDate: string }` (entryDate = `YYYY-MM-DD`).
- `RecordMemoryResponse { memory: Memory }` — 04가 원본 저장 후 별 1개 즉시 반환("별 생성 중", 연결은 아직 없음).
- `GetUniverseResponse { memories: Memory[]; links: Synapse[] }` — links는 09/05 소관, 여기선 무시 가능.

**도메인 타입(순수, three/React 미의존 — 헌법 4).** `entities/memory/model/types.ts`(08 정의, 없으면 본 스펙에서 최소 정의):
```ts
export interface Memory {
  id: string; body: string; mood?: string; intensity?: number;
  entryDate: string; createdAt: string; lastRecalledAt?: string;
}
```

**매퍼(`entities/memory/api/map-memory.ts`).** proto 타입 → 도메인 `Memory`. **별 좌표는 매핑하지 않는다**(헌법 3: 좌표는 클라 force-sim에서 창발). 매퍼는 순수 함수(three 미의존).

**낙관적 흐름(`use-record-memory.ts`).** Zustand 액션 사용:
1. 제출 → 임시 id(`temp-${crypto.randomUUID()}`)로 도메인 `Memory` 생성 → `addMemory(optimistic)` → 별 즉시 등장.
2. `client.recordMemory(req)` 호출.
3. 성공 → `replaceMemory(tempId, res.memory)`(서버 id로 교체, force-sim은 기존 별 pin 유지 — 헌법 3).
4. 실패 → `removeOptimistic(tempId)` 롤백 + 에러 카피 노출.

**스토어 계약(`entities/memory/model/store.ts`, 08과 공유 — 부족하면 본 스펙에서 보강).**
```ts
interface MemoryStore {
  memories: Memory[]
  setMemories: (m: Memory[]) => void      // GetUniverse 적재
  addMemory: (m: Memory) => void           // 낙관적 등장
  replaceMemory: (tempId: string, m: Memory) => void
  removeOptimistic: (tempId: string) => void
}
```
> ⚠️ `removeOptimistic`는 **낙관적 임시 별만** 제거한다. 서버에서 온 별은 절대 삭제하지 않는다(헌법 2 — 우주에서 별 삭제 금지).

**조립.** `HomePage`가 `<UniverseCanvas/>`(별 렌더) + `<MemoryForm/>`(2D HUD, 캔버스 밖 DOM — Architecture §2.7 "씬 안 DOM 금지")를 함께 렌더. 마운트 시 `GetUniverse` 1회.

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 사용자가 본문이 비어있지 않은 폼을 제출 THE SYSTEM SHALL 임시 id의 별을 스토어에 즉시 추가하고 `RecordMemory`를 호출한다.
1.2 WHEN `RecordMemory`가 성공 응답 THE SYSTEM SHALL 임시 별을 서버 id의 별로 교체하고 기존 별 위치는 유지한다(force-sim pin — 헌법 3).
1.3 IF `RecordMemory`가 실패(네트워크/서버 오류) THEN THE SYSTEM SHALL 임시 별만 롤백 제거하고 한국어 에러 카피를 표시하며, 이미 존재하던 서버 별은 보존한다(헌법 2).
1.4 IF 본문이 공백뿐 THEN THE SYSTEM SHALL 제출을 막고 "일기 본문을 입력하세요" 안내를 표시한다(RPC 미호출).
1.5 WHEN 날짜 미선택 THE SYSTEM SHALL 기본값 오늘(`YYYY-MM-DD`, 로컬)로 `entryDate`를 채운다(concept §시점: 기본 오늘, 과거 지정 가능).
2.1 GIVEN 홈 페이지 마운트 WHEN 컴포넌트가 렌더 THE SYSTEM SHALL `GetUniverse`를 1회 호출해 `memories`를 `setMemories`로 적재하고 별로 렌더한다.
2.2 GIVEN 별을 작성한 뒤 새로고침 WHEN 홈이 다시 마운트 THE SYSTEM SHALL `GetUniverse` 응답에 그 별을 포함해 렌더한다(원본 영속 — 헌법 1).
3.1 THE SYSTEM SHALL 매퍼에서 별 좌표를 만들지 않는다(서버 응답에 좌표 없음, 좌표는 force-sim 창발 — 헌법 3).
3.2 THE SYSTEM SHALL `entities/memory/api`·`model`에서 `three`/React/DOM을 import하지 않는다(헌법 4).
