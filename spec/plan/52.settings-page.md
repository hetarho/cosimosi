# 52. settings page

> Settings page — the app's one settings surface: a NEW route (web `/settings`) + screen (mobile `SettingsScreen`) that
> hosts **account** (basic — sign out, the plan 04 session), the **emotion-palette change UI** (consuming plan 51's
> palette registry + stored preference), and a **reserved staging-layer customization slot** ([P4] placeholder —
> background / theme / effect / camera-mood, non-meaning layers only, not built here). A small composition unit — it
> composes existing features and reserves a slot; it owns no new engine.
> Scope: FE | Status: planning
>
> Layer P (presentation) — the settings unit of Epic I, paired with [51.palette-customization](51.palette-customization.md)
> (the palette registry + preference + guardrails it hosts) and [53.auth-universe-gate](53.auth-universe-gate.md) (the
> route/gate that makes `/settings` reachable behind auth). Home: a new `pages/settings` screen composing
> `features/change-palette` ([51]) + a small `features/account-settings` (sign-out over the plan 04 session), plus the
> `features/customize-staging` **placeholder** slot. It fills the "where does the user change their palette / sign out"
> seam and leaves the free-edit staging layer as a reserved-but-empty slot ([P4] SHOULD).

## Purpose

The product has a palette the user can swap ([51], [P1]) and an authenticated session ([04]), but no place to _do_
either from the running app — no screen to change the emotion→color mapping, and no screen to sign out. This unit is
that place: the single **settings** surface (PRD [§1.6](../PRD.md) — "설정 페이지 (별도) — 계정, 팔레트 변경, 연출
커스터마이징"). It exists so the three settings concerns have **one home** rather than being scattered as ad-hoc menus,
and so the [P4] "연출 레이어 커스터마이징" (staging-layer customization) has a **reserved slot** that later work slots
into without a redesign — the PRD's reserve-the-slot discipline ([V8] precedent).

It is deliberately a **small composition**: the palette-change behavior, registry, preference storage, and guardrails
are [51.palette-customization](51.palette-customization.md); the session and sign-out capability are
[04.auth-session](04.auth-session.md); the route/gate is [53.auth-universe-gate](53.auth-universe-gate.md). This unit
lays out the screen, composes those pieces, and books the staging slot — it introduces no new backend, no new engine,
and no free-edit customization surface.

## Scope / Non-goals

**In scope**

- **`pages/settings`** — a new URL-addressable screen: web route **`/settings`** (registered through the typed route
  registry [15.web-router-foundation](15.web-router-foundation.md), the `app/routes` segment) and mobile
  **`SettingsScreen`** (registered in the mobile `app/navigation` route registry, [13.mobile-app-shell](13.mobile-app-shell.md)).
  Reached from the universe page (a settings affordance) and gated behind auth by [53.auth-universe-gate](53.auth-universe-gate.md).
  The page **composes** the features below into sectioned groups (account · appearance · staging); it holds no domain
  logic of its own.
- **`features/account-settings`** — the **basic** account section: shows the signed-in identity (from the plan 04
  session snapshot — `userId` / the account's display facts already available client-side) and a **sign out** action
  that calls the plan 04 auth facade's sign-out. **Decided (2026-07):** account here is **basic only** — sign out plus
  read-only identity display; profile edit, email/password change, delete-account, and billing are **out** (delete of a
  _memory_ is Epic H [50.deletion-ui](50.deletion-ui.md); account deletion is a later plan).
- **`features/change-palette`** — the **palette change UI**, owned by [51.palette-customization](51.palette-customization.md)
  and _hosted_ here. It lists the palette **registry** (≥2 named palettes, [51] content), shows the current stored
  **preference**, and lets the user pick one; selection persists via [51]'s get/set preference path and calls
  `setMoodPalette(chosen)` behind the SAME single entry point ([M6][51]). This unit **mounts** that feature and lays it
  out; it does not re-own the registry, the preference storage, or the swap wiring.
- **`features/customize-staging`** — the **[P4] staging-layer customization slot**, **built as a placeholder**: a
  visible, disabled/"coming later" section that **reserves** the place for background / theme / effect / camera-mood
  choices (the non-meaning layers). It renders the reserved slot and states the boundary; it does **not** expose any
  free-edit control. **Decided (2026-07):** v1 ships the slot **empty (reserved)** — no background/theme/effect/camera
  editing is buildable here yet; the slot exists so later work fills it without re-plumbing the settings screen and so
  the guardrails ([P2][P4][I11]) are stated at the point they will apply.
- **Guardrail statement at the slot** — the staging section states, in copy and structure, that customization touches
  **non-meaning layers only**: it can never override "color = emotion", a star's emotion, or any position / strength
  ([P2][P4][I11]). The palette section carries [51]'s axis-consistency guardrail ([P3]) — this page surfaces it, [51]
  owns it.
- **UI copy via i18n** ([08.i18n-seam](08.i18n-seam.md) / `packages/i18n`) — no hardcoded strings; a **literary,
  restrained Korean** voice (section titles plain and honest, the reserved slot's "coming later" copy unadorned).
  **web + mobile parity** — the same page + features on both apps; `model`/`api` shared verbatim, `*.native.tsx`
  siblings **only** for the screen chrome / list-row / section primitives that genuinely differ ([ARCHITECTURE.md](../ARCHITECTURE.md) §3.5).

**Non-goals**

- **No palette engine, registry, preference storage, or swap wiring.** `moodColor` / `activePalette` / `setMoodPalette`,
  the palette registry, the stored user preference, the get/set RPC, and the axis-consistency guardrails are all
  [51.palette-customization](51.palette-customization.md). This unit only _hosts_ the palette-change feature.
- **No auth engine.** The session model, sign-in, refresh, token, and the auth facade are [04.auth-session](04.auth-session.md).
  This unit only reads the session snapshot and calls the existing sign-out action.
- **No route/gate ownership.** Whether `/settings` requires auth, and the login→universe direct flow, are
  [53.auth-universe-gate](53.auth-universe-gate.md). This unit _registers_ the route and _assumes_ the gate.
- **No free-edit staging customization.** Background / theme / effect / camera-mood editing is a **reserved placeholder**
  ([P4] SHOULD) — the slot is booked, not built. No sliders, no color pickers for the scene, no camera editor in v1.
- **No meaning-layer customization ever.** The user can never override "color = emotion", a star's emotion, or any
  position / strength — structurally absent from every control here ([P2][I3][I11]).
- **No profile / billing / account deletion / invite management.** Account here is sign-out + identity display only;
  invite-for-growth is Epic F earn ([G3], [44.earn-spend-usecase](44.earn-spend-usecase.md)); memory deletion is Epic H
  ([50.deletion-ui](50.deletion-ui.md)).
- **No new backend or proto.** The only backend touched is [51]'s preference get/set (owned there) and [04]'s existing
  session — this unit adds none.
- No new tuning scalars hardcoded in FE (this page is layout + composition; see §Policy).

## Design

### Placement (FSD, §3.1)

```
apps/web/src/pages/settings/
├── ui/            SettingsPage (sectioned layout: account · appearance · staging) + *.native.tsx sibling for screen chrome
├── model/         (thin) — section ordering / labels config; no domain logic
└── index.ts       public API (the page + the "open settings" affordance the universe page uses)

apps/web/src/features/
├── account-settings/   ui (identity display + sign-out button) · api (reads plan 04 session, calls sign-out)
└── customize-staging/  ui (the reserved [P4] slot — disabled "coming later" section stating the non-meaning boundary)
```

- `features/change-palette` is **owned by [51.palette-customization](51.palette-customization.md)** (it lives in [51]'s
  slice set); this page **imports and mounts** it. `pages/settings` composes three features (`account-settings`,
  `change-palette`, `customize-staging`) into sectioned groups (§3.1: a `page` composes `features`). It holds **no**
  domain logic — no palette state, no session state — reading each feature's public API only.
- The page imports **no** `three` and **no** visual entity (§3.4): changing a palette flows through [51]'s
  `setMoodPalette` single entry point; the running canvas re-reads the active palette (the swap is behind the same
  entry, so nebula/star rendering stays palette-agnostic, [M6][51]). Settings never writes a visual word up the
  projection.
- **web + mobile parity (§3.5):** the same page + features on both apps; `model`/`api` segments shared verbatim (pure,
  no DOM/native). Only `ui` forks where a primitive differs — the **screen chrome** (web page shell / DOM ↔ RN screen +
  scroll container), section rows, and the **sign-out** confirm primitive — via `*.native.tsx` siblings. web =
  open-beta scope, mobile = MVP; both draw from the same message keys.

### The route / screen ([15] / [13])

- **Web:** a new route **`/settings`** added to the typed route tree in `apps/web/src/app/routes/` ([15] — the only
  segment that imports `@tanstack/react-router`); its component imports `SettingsPage` from `pages/settings`. The route
  is registered in the type registry so navigation to it is typed; it inherits the provider stack (observability →
  error boundary → i18n → auth → cache → router) so it sees the session and i18n.
- **Mobile:** a new **`SettingsScreen`** registered in the mobile `app/navigation` typed route registry
  ([13.mobile-app-shell](13.mobile-app-shell.md)); it renders the same `pages/settings` composition.
- **Auth gate:** whether `/settings` requires an authenticated session is owned by
  [53.auth-universe-gate](53.auth-universe-gate.md) (authenticated → app; unauthenticated → login; no landing in v1,
  [U4]). This page **assumes** the gate: it renders for an authenticated user and does not implement its own redirect.
- **Entry affordance:** a settings affordance on the universe page ([23.universe-canvas](23.universe-canvas.md))
  navigates here via the typed navigation seam ([15]/[13]). The affordance is minimal (an icon/menu entry), consistent
  with the always-button-minimal voice.

### Account section — basic ([04])

`features/account-settings` reads the plan 04 **session snapshot** (the authenticated `userId` and whatever display
facts the session exposes; no new fetch) and renders a read-only identity line plus a **sign out** action wired to the
plan 04 auth facade's existing sign-out. Sign-out transitions the session machine ([04] `authenticated → signedOut`);
the gate ([53]) then routes to login. **Decided (2026-07):** sign-out uses a small **confirm** step (a plain "정말
로그아웃할까요?" i18n prompt) so it is not an accidental single-tap; no other account mutation exists here.

### Appearance section — the palette change UI ([51], [P1]–[P3])

This section **hosts** `features/change-palette` from [51.palette-customization](51.palette-customization.md):

- It lists the palette **registry** (the ≥2 named palettes [51] defines as content — mood→color tables are **not**
  values, per the config-exclusion rule) and marks the **current stored preference** ([51]'s get preference).
- Selecting a palette **persists** the choice through [51]'s set-preference path and applies it via `setMoodPalette` at
  the same single entry point ([M6][51]); the running universe re-reads the active palette so the change is felt
  immediately. This page owns the **layout and the mount point**, not the persistence or the swap.
- **Axis-consistency guardrail ([P3])** is [51]'s (valence=warm/arousal etc. — a palette cannot strip emotion of its
  color meaning); this page **surfaces** it (e.g. the registry only offers guardrail-respecting palettes; a free
  palette editor does not exist here) rather than re-implementing it.
- **Meaning-layer immutability ([I11][P2]):** the swap changes the emotion→color _mapping_ (presentation), never "color
  = emotion" nor a star's emotion. There is structurally no control to set an individual star's emotion/position/
  strength anywhere on this page ([I3]).

### Staging-layer customization — reserved placeholder ([P4], SHOULD)

`features/customize-staging` renders the **reserved slot** for the non-meaning staging layers (background / theme /
effect / camera-mood, [P4]):

- **What it renders now:** a visible, clearly **disabled / "이후 추가될 예정" (coming later)** section that names the
  reserved concern and states the boundary — customization touches **non-meaning layers only**, never "color =
  emotion", a star's emotion, or any position / strength ([P2][P4][I11]). No editable control ships.
- **Why book the slot:** so later staging work drops in **without re-plumbing** the settings screen, and so the
  guardrails are stated at the point they will apply — the PRD's reserve-the-slot discipline ([V8]/[P4] SHOULD). The
  active staging skin today is a **build-time** selection (`rendering.active_skin` in [values.yaml], owned by
  [14.rendering-foundation](14.rendering-foundation.md), which already reserved a `[P4]`-ready `useSkin` seam); making it
  _user-choosable_ is exactly the deferred [P4] work this slot reserves — it will consume plan 14's `useSkin` seam.
- **Boundary enforced by structure:** because no free-edit control exists, there is nothing for a user (or a
  prompt-injected model) to set that could touch the meaning layer — the [P4]/[I11] guarantee is structural, not just
  copy.

### Control-state (§3.2) — mostly declarative

The page is largely declarative (read the session, read the preference, render sections). The only transient control-
state is a **save-in-flight / signing-out** indication; per §3.2 that is a small XState machine under the relevant
feature's `model/` (or a simple loading flag if it stays trivial) with the preference value and session snapshot held
outside machine context. No per-frame canvas coupling exists here (settings does not drive the scene per frame).

### Copy (i18n) — literary, restrained Korean

All strings resolve through the i18n seam ([08.i18n-seam](08.i18n-seam.md)); none hardcoded (`lint:raw-strings`). The
section titles (계정 / 팔레트 / 연출) are plain; the sign-out confirm is honest and unadorned; the reserved-slot "coming
later" copy is quiet, not a marketing tease. Web + mobile draw from the **same** message keys (parity).

## Acceptance Criteria

1. A new **settings screen** exists as a URL-addressable route on web (**`/settings`**, registered in the typed route
   registry [15]) and a registered **`SettingsScreen`** on mobile ([13]); a settings affordance on the universe page
   navigates to it via the typed navigation seam (PRD [§1.6](../PRD.md)).
2. The settings screen is **reachable only behind auth** — it assumes the [53.auth-universe-gate](53.auth-universe-gate.md)
   gate and does not implement its own redirect ([U1][U4] owned by 53).
3. An **account** section shows the signed-in identity (read from the plan 04 session snapshot, no new fetch) and a
   **sign out** action that calls the [04.auth-session](04.auth-session.md) facade's sign-out (via a plain confirm step),
   after which the gate routes to login ([04]).
4. A **palette (appearance)** section **hosts** `features/change-palette` ([51]): it lists the palette registry, marks
   the current stored preference, and lets the user select a palette; selection **persists** via [51]'s set-preference
   path and applies through `setMoodPalette` at the same single entry point so the running universe reflects it ([P1][M6]).
5. The palette section carries the **axis-consistency guardrail** ([P3]) by surfacing only guardrail-respecting registry
   palettes (no free palette editor); [51] owns the guardrail, this page does not re-implement it.
6. A **staging-layer customization** section renders a **reserved placeholder** ([P4]): a visible, disabled "coming
   later" slot that names background / theme / effect / camera-mood and states the non-meaning boundary — **no** editable
   staging control ships in v1.
7. **No control anywhere on the page** can override "color = emotion", set a star's emotion, position, or strength — the
   meaning layer is structurally untouchable from settings ([I3][I11][P2]).
8. The page holds **no domain logic** — it composes `features/account-settings`, `features/change-palette` ([51]-owned),
   and `features/customize-staging`, reading each feature's public API; it owns no palette state, session state, or new
   backend / proto (§3.1).
9. Any transient **save-in-flight / signing-out** state is control-state (an XState machine under the feature's `model/`
   or a trivial loading flag), with the preference value and session snapshot held **outside** machine context (§3.2).
10. **All** user-visible copy resolves through the i18n seam (no raw strings; `lint:raw-strings` passes), in a literary /
    restrained Korean voice.
11. **web + mobile parity:** the same page + features run on both apps, sharing `model`/`api` verbatim; `*.native.tsx`
    siblings exist **only** for the screen chrome / section-row / sign-out primitives (§3.5).

## Policy / Values Impact

- **policy/ux** — the rules this screen makes true land in a new `policy/ux/settings.md` once implemented (written at
  `/implement-job`; **do not** pre-create policy files):
  - **One settings home** — account (basic), palette change, and staging customization live on a single settings screen
    reached from the universe page and gated behind auth (PRD [§1.6](../PRD.md)).
  - **Account is basic** — sign-out + read-only identity only; profile/billing/account-deletion are out of v1 settings.
  - **Palette change is hosted, not owned** — the swap goes through [51]'s single entry point and persisted preference;
    only guardrail-respecting palettes are offered ([P1][P3][M6]).
  - **Staging customization is non-meaning-only and reserved in v1** — the slot names background/theme/effect/camera-mood
    and ships disabled; customization can never touch "color = emotion", a star's emotion, or position/strength
    ([P4][P2][I11]).
- **tech/** — the as-built FE rules (the `pages/settings` composition over three features; the web `/settings` route +
  mobile `SettingsScreen` registration; the "hosts [51]'s `change-palette`, owns no domain logic" composition; the
  reserved staging slot seam) are recorded in a tech doc (e.g. [tech/web-routing.md](../tech/web-routing.md) for the
  route registration and a state-machine note if a save machine is added) at `/implement-job`, reinforcing
  [ARCHITECTURE.md](../ARCHITECTURE.md) §3.1/§3.2/§3.5 rather than introducing a new architecture concern.

  - `rendering.active_skin` (existing, owned by [14.rendering-foundation](14.rendering-foundation.md) — its `[P4]`-ready
    `useSkin` seam) — the **build-time** staging skin the reserved [P4] slot will one day make user-choosable; named here

  - Excluded per the config rule: message strings (i18n), the section labels/ordering (code), route paths (code), and
    any proto/DB schema (none introduced). If a future save-debounce or toast-duration scalar proves cross-surface
    tunable, a `settings.*` group is added via `pnpm gen:values` — **confirmed at implement**, not declared now.
- **Dependencies:** depends on [51.palette-customization](51.palette-customization.md) (the `change-palette` feature +
