# 20. encode usecase

> Encode use-case — the backend orchestration that turns one diary into episodic memories: event-boundary split,
> name suggestion, primary emotion, and typed neuron extract/normalize (dedup), with a schema-forced LLM output and a
> natural-language-edit defense. Also the `PersistEncoded` use-case invoked at launch.
> Scope: BE | Status: built (job 26)
>
> Layer U (use-cases). Home: `internal/memory` use-cases `Encode` / `PersistEncoded` + the consumer-owned `Extractor`
> port. The concrete extractor adapter + keyless mock live in [22.ai-worker-pipeline](22.ai-worker-pipeline.md); the
> aggregate/schema in [16.memory-aggregate-schema](16.memory-aggregate-schema.md); the emotion VO + arousal init in
> [17.emotion-model](17.emotion-model.md); synapse creation in [21.link-usecase](21.link-usecase.md); the write flow
> UI in [27.writing-flow-ui](27.writing-flow-ui.md).

## Purpose

Writing a diary is the one moment cosimosi turns experience into structure. The user writes freely; the product must
break that text into the units the whole memory engine is built on — episodic memories (stars) and their neurons
(shared elements) — _and it must do so in front of the user_, who names, edits, and confirms the result before any
star is launched ([W2][W2a][W4]). Everything downstream (linking, forgetting, semanticization, the emotion color of
the universe) inherits the boundaries this unit draws.

This unit owns that transformation on the backend: the `Encode` use-case that orchestrates the split, and the
`PersistEncoded` use-case that atomically commits the user-confirmed result at launch. It fixes two hard properties:
neuron **normalization/dedup** ([E10]) so the constellation does not fragment into "스벅" and "스타벅스"; and a
**schema-forced LLM output** ([W4a]) so that natural-language editing — a place users type arbitrary instructions —
cannot structurally produce an invariant-violating result. The LLM lives entirely behind a consumer-owned `Extractor`
port; this plan specifies the orchestration and the contract, not the model.

## Scope / Non-goals

**In scope**

- The `Encode` use-case in `internal/memory`: diary body → event-boundary split into 2–5 episodic memories ([E1][E2])
  → per-memory **name** suggestion ([W2a]) + **primary emotion (mood)** assignment ([M1]) + **typed neuron** extraction
  (semantic / spatial / entity, [E3]) with **≥1 semantic neuron per memory** ([E4]) and optional spatial/entity ([E5]),
  **time never a neuron** ([E6]).
- **Neuron normalization/dedup** ([E10]): a conservative, type-differentiated canonicalization against the user's
  existing neurons, assisted (narrowly) by fetching candidates by name/type + an embedding nearest-neighbour lookup.
- The consumer-owned **`Extractor` port** declared in `internal/memory`: `Split(ctx, body, date, existingNeurons)`
  plus a revise variant taking the prior result + a natural-language instruction. **Output schema-forced** ([W4a]).
- The two synchronous RPCs backed by `Encode`: **`SplitDiary`** (synchronous preview — the user sees it [W2], persists
  nothing) and **`ReviseSplit`** (natural-language edit re-run [W4a]), with retry / output-token caps.
- The **`PersistEncoded`** use-case invoked by `LaunchStars` (the `SplitDiary`/`ReviseSplit`/`LaunchStars` contract is owned **here** — see the RPC section — and consumed by [27.writing-flow-ui](27.writing-flow-ui.md)):
  atomically persists Diary + EpisodicMemory + Neuron (deduped) + NeuronActivation, sets `seed` + arousal-modulated
  `base_strength`, enqueues `embed` + `semanticize` jobs, then hands off to `Link`. Optimistic write (returns ids).
- Thin RPC handlers (`memory/rpc`): proto↔domain mapping + call only; all policy in the use-case / domain.
- Per-user scoping on every fetch, canonicalization, and write.

**Non-goals**

- **No concrete LLM/embedding adapter and no keyless mock** — those implement this plan's ports and live in
  [22.ai-worker-pipeline](22.ai-worker-pipeline.md), injected at the composition root.
- **No synapse creation** — `Encode`/`PersistEncoded` extract and persist neurons; edges are `Link`'s job
  ([21.link-usecase](21.link-usecase.md), [L1][L2][L4]). `PersistEncoded` _invokes_ `Link`; it does not implement it.
- **No embedding or gist generation inline** — `embed` (Embedder) and `semanticize` (Semanticizer, [C7]) are async jobs
  enqueued here and run by the worker; results fill on the next read (§2.8 optimistic write).
- **No universe-clock authority** — `universe_state` and clock advance ([T1][T2][I10]) are Epic B. Epic A only leaves
  the monotonic seam at launch (default: save Diary, launch no star for a past-dated diary; see below).
- **No schema/table definition** — tables are owned by [16.memory-aggregate-schema](16.memory-aggregate-schema.md);
  this unit consumes repositories against them.
- **No writing-flow UI, XState machine, or client optimistic rendering** — that is [27.writing-flow-ui](27.writing-flow-ui.md).
- **No emotion VO math or mood→color palette** — the `Emotion` value object, mood valence/arousal coordinates, and the
  arousal→initial-strength function are [17.emotion-model](17.emotion-model.md); `Encode` consumes them.
- No editing of already-launched stars ([W4]); no delete/merge of past memories from the encode surface.

## Design

### Placement (domain-first Go)

`Encode` and `PersistEncoded` are application use-cases in the core-domain `internal/memory` context. They start inside
the context package (ARCHITECTURE §2.3); the `{app,domain,pg,rpc}` split is a later refactor, not a starting tax. The
`Extractor` port is declared here because `Encode` consumes it (§2.4, consumer-owned ports); its concrete adapter is
injected at `cmd/api` from [22.ai-worker-pipeline](22.ai-worker-pipeline.md). Dependency direction:
`memory/rpc → Encode/PersistEncoded (use-case) → pure domain (Emotion VO, arousalToInitialStrength, dedup identity rules)`, with
`Extractor`, the repositories, and `JobQueue` reached only through consumer-owned interfaces.

### `Encode` orchestration (used by `SplitDiary` / `ReviseSplit`)

`Encode` produces a _proposed_ split — nothing is persisted. Given `(userScope, body, diaryDate)`:

1. **Fetch dedup candidates.** Load the user's existing neurons that could match this diary's content — by exact/near
   `name` + `type`, plus a **narrow embedding nearest-neighbour** assist (top-k over `embeddings`, within a similarity
   threshold). Per-user scoped. This is a _candidate set for the LLM to canonicalize against_, not a linking step.
2. **Call `Extractor.Split(ctx, body, diaryDate, existingNeurons)`.** The port returns the schema-forced result: a list
   of proposed memories, each `{name, mood, neurons:[{name, type}]}` — and _only_ those fields (see the schema below).
   The extractor performs the split, name suggestion, mood assignment, typed neuron extraction, and the conservative
   canonicalization against `existingNeurons` in one structured call. `diaryDate` is passed for the temporal _guideline_
   only; **it never becomes a neuron** ([E6]).
3. **Enforce domain invariants on the returned structure** (policy lives in the use-case/domain, not the handler):
   - memory count within `[encode.min_memories, encode.max_memories]` (2–5, [E2]); if the extractor returns out-of-range,
     it is a retry/repair condition, not a silent clamp — the LLM is re-prompted within the retry cap.
   - each memory has **≥1 `semantic` neuron** ([E4]); this is enforced as a _guideline the extractor is prompted to
     meet_ — the use-case treats a missing semantic neuron as a repair/retry signal, **not** by injecting a hollow
     placeholder neuron.
   - every neuron `type` ∈ {`semantic`, `spatial`, `entity`} ([E3]); spatial/entity are optional ([E5]).
   - no positional/color/strength/time field is present (structurally impossible given the schema — see [W4a]).
4. **Return the proposed split** to the caller for the user to review, rename, re-mood, and confirm ([W2][W2a][W4]).
   `Encode` computes no coordinate, no color, no strength, no seed here — those belong to persistence/rendering.

`ReviseSplit` is the same orchestration with `Extractor`'s **revise variant**: it takes the prior `SplitDiaryResponse`

- a natural-language `instruction` and re-runs against the same schema-forced output ("merge the meeting and lunch into
  one star", "treat these two neurons as the same"). Same invariant enforcement, same caps.

### Neuron normalization / dedup ([E10])

Dedup is an **identity judgement** — "is this the same neuron the user already has?" — performed conservatively by the
extractor, then honored by the domain when persisting. It is **distinct from [L3]** (no similarity _linking_): merging
decides _sameness_, it does not create an edge between different neurons. Over-merging collapses the constellation, so
the bias is conservative — merge only true synonyms / same-referent, keep related-but-distinct concepts separate.

The judgement is **type-differentiated**:

- **entity** — same _person/proper referent_ ("엄마" / "어머니" → one neuron).
- **spatial** — synonyms only, **granularity preserved**: "스타벅스" ≠ "스타벅스 강남점" (different places, not merged);
  only true aliases of the _same_ place merge.
- **semantic** — strictest: merge "성취감" → "성취" but keep "성취" ≠ "성공".

The embedding nearest-neighbour lookup (step 1) is a **narrow assist** to surface candidates the name/type match would
miss; the final merge decision is the extractor's conservative canonicalization, not a raw similarity cutoff. The
embedding-assist **similarity threshold is TUNABLE** — the key is `encode.dedup_similarity_threshold` (decided 2026-07;
see the values keys below), its value set at implement. When the extractor canonicalizes a proposed neuron onto an existing one, the persisted
memory references the existing neuron's id; a genuinely new neuron is created at persist time.

### `Extractor` port (consumer-owned, declared in `internal/memory`)

Declared where `Encode` consumes it (§2.4). Domain-shaped inputs/outputs only — no proto, no sqlc, no SDK types cross
this boundary:

- `Split(ctx, body, diaryDate, existingNeurons) → (ExtractResult, error)` — split + name + mood + typed neuron
  extract + conservative canonicalization against `existingNeurons`. **Output schema-forced** ([W4a]).
- a **revise variant** — `ReviseSplit(ctx, previous, instruction) → (ExtractResult, error)` — same schema-forced
  output, taking the prior proposed split + a natural-language instruction. (As-built: the prior result carries the
  split context, so body/diaryDate are not re-passed; the use-case's repair re-prompts run through this same variant.)

The concrete adapter (real SDK) and the keyless deterministic mock both implement this port and are provided by
[22.ai-worker-pipeline](22.ai-worker-pipeline.md), cost-metered, injected at the composition root. `existingNeurons` is
the per-user candidate set assembled in step 1.

### RPC contract (`memory.v1.MemoryService`, Connect, unary — §2.7)

proto is a DTO layer mapped at `memory/rpc`; handlers are thin (map proto↔domain + call the use-case, §2.9#7). The
critical property is that the encode output schema — the shape the LLM is forced to emit and the shape the RPC returns —
contains **only** `{memories:[{name, mood, neurons:[{name, type}]}]}`. There are **no** `position`, `color`, `strength`,
`seed`, `time`, or `delete` fields anywhere in it. This is the structural half of the [W4a] double defense: even if a
prompt injection defeats the prompt engineering, the model has no field in which to emit an invariant-violating value
(no coordinate → cannot set position [I3][I5]; no delete → cannot remove [I1]; no time → cannot rewrite the clock [I10]).

```proto
// preview split (synchronous — the user sees it [W2]); persists nothing.
rpc SplitDiary(SplitDiaryRequest) returns (SplitDiaryResponse);
// natural-language edit re-run [W4a]; same schema-forced output.
rpc ReviseSplit(ReviseSplitRequest) returns (SplitDiaryResponse);

message SplitDiaryRequest  { string body = 1; string diary_date = 2; }        // diary_date = user-entered [W5]
message ReviseSplitRequest { string body = 1; string diary_date = 2;
                             SplitDiaryResponse previous = 3; string instruction = 4; }

message SplitDiaryResponse { repeated ProposedMemory memories = 1; }
message ProposedMemory     { string name = 1; string mood = 2;                // mood = 13-enum [M1][M2]
                             repeated ProposedNeuron neurons = 3; }
message ProposedNeuron     { string name = 1; string type = 2; }              // type ∈ semantic|spatial|entity [E3]
// NOTE: no position/color/strength/seed/time/delete field exists anywhere in this schema — [W4a] structural defense.
```

`SplitDiary` and `ReviseSplit` are **synchronous preview** calls (the split is user-facing [W2], not a background job)
and are **not** GET/`NO_SIDE_EFFECTS` — they invoke the LLM and are cost-metered, but they persist nothing. Retry and
output-token caps (`encode.max_revise_retries`, `encode.max_output_tokens`) are enforced in the use-case; on exceeding
them the call returns a canonical error rather than an unbounded retry loop.

`LaunchStars` (the persist-and-launch RPC) is owned **here** (plan 20), alongside `SplitDiary`/`ReviseSplit` — this plan
defines the whole `MemoryService` write contract and implements its handlers. Its handler calls `PersistEncoded`; it is
**consumed by** [27.writing-flow-ui](27.writing-flow-ui.md)'s `features/launch-stars`:

```proto
rpc LaunchStars(LaunchStarsRequest) returns (LaunchStarsResponse);
message LaunchStarsRequest  { string body = 1; string diary_date = 2;
                              repeated ConfirmedMemory memories = 3; }         // user-edited split result
message ConfirmedMemory     { string name = 1; string mood = 2;
                              repeated ProposedNeuron neurons = 3; }
message LaunchStarsResponse { repeated string memory_ids = 1;                 // optimistic: ids only (§2.8)
                              repeated string new_neuron_ids = 2;             // neurons genuinely created (not deduped)
                                                                              // — drives the latent-star awaken ([25])
                              bool past_dated = 3; }                           // monotonic-guard outcome [I10][T1]
```

**As-built note (job 37):** `LaunchStarsResponse` carries `past_dated` — the monotonic launch guard's authoritative
outcome (invariant [I10], theme [T1]). The FE gates its optimistic insert on this flag (a past-dated launch inserts nothing) rather than
inferring it from an empty `memory_ids`; the pre-launch button confirmation still _predicts_ past-dating client-side
against the cached `universe_time`, which is a warning-before-the-call, not the authority.

**As-built notes (job 26):** `ReviseSplit` returning `SplitDiaryResponse` shares one response message across two RPCs,
so buf lint's `RPC_RESPONSE_STANDARD_NAME`/`RPC_REQUEST_RESPONSE_UNIQUE` are `ignore_only`-scoped to `memory.proto`.
The `memory.v1.MemoryService` proto also carries the **`GetUniverse`** read (contract shape owned by
[23.universe-canvas](23.universe-canvas.md)); its thin handler lives in `memory/rpc` over plan 16's universe queries,
deriving `universe_time` as the max `created_universe_time` within the same read snapshot (Epic A derivation — the
`universe_state` clock stays Epic B). The full as-built contract is recorded in
[tech/memory-encode.md](../tech/memory-encode.md).

### `PersistEncoded` use-case (invoked by `LaunchStars`)

Given `(userScope, body, diaryDate, confirmedMemories)`, `PersistEncoded` commits the user-confirmed split **atomically**
in one transaction (`platform.InTx`):

1. **Insert the Diary** — append-only, one INSERT, no UPDATE ([I2][W5]). The Diary is the objective record and is written
   here, at launch, not at split (split is a preview — cross-cutting decision, brief §7). `diary_date` is the user-entered
   date.
2. **Monotonic launch guard** ([I10][T1]) — compare `diary_date` against the latest launched memory's
   `created_universe_time`. If `diary_date` is **not** past-dated, proceed to raise stars. If it _is_ past-dated
   (`diary_date < latest launched`), the **Diary is still saved** (readable in the diary list / original reader) but
   **no EpisodicMemory is launched** — a past-dated diary leaves no star in the universe. _Universe-clock advance itself
   is Epic B (`universe_state`); Epic A only enforces this guard as the seam._ **Decided (2026-07):** Epic A surfaces a
   one-time confirmation notice in the writing flow before a past-dated launch (owned by
   [27.writing-flow-ui](27.writing-flow-ui.md)); the Diary is saved, no star launches; Epic B revisits the full
   past-date experience.
3. **For each confirmed memory** (when not past-dated): insert an `EpisodicMemory` (aggregate root) with `name`, the
   `Emotion` VO derived from `mood` ([M1], via [17.emotion-model](17.emotion-model.md)), a generated **`seed`** ([E7]
   — a visual-form/anchor hint, _not_ a stored coordinate [I5]), `created_universe_time = diary_date`, and an
   arousal-modulated **`base_strength`** ([E8a], from `arousalToInitialStrength(arousal)` in [17.emotion-model](17.emotion-model.md)).
4. **Persist neurons with dedup applied** — for each proposed neuron, reference the existing neuron id when the extractor
   canonicalized it onto one; otherwise create the `Neuron` (typed [E3]). Neurons are shared across memories by id
   (aggregate boundary, §2.5) and are **never deleted** by the system ([I1][F3]).
5. **Insert `NeuronActivation`** join rows (memory↔neuron membership + `weight`) — the co-membership that later drives
   the memory's centroid pull ([E8]). The schema-forced `Extractor` output carries **no** weight (by [W4a] design), so
   `PersistEncoded` sets `weight` at persist time: **Epic A writes uniform `1.0`** per membership (decided 2026-07);
   differentiated co-membership weighting ([E8]) is a later-epic refinement behind this same column. The column is
   `NOT NULL`, so a value is always written.
6. **Hand off to `Link`** ([21.link-usecase](21.link-usecase.md)) inside the same transaction — creates neuron↔neuron
   synapses from same-memory co-membership ([L1]) and shared neurons ([L2]); temporal-proximity strengthening ([L4]) is
   `Link`'s concern. No memory↔memory edge is ever created ([I4][I6]).
7. **Enqueue async jobs** ([22.ai-worker-pipeline](22.ai-worker-pipeline.md), §2.8): `embed` (Embedder → per-neuron
   vectors for dedup + latent-field assist) and `semanticize` ([C7], Semanticizer → the 4 gist-stage texts). These run
   on the worker; `semanticize` never mutates the Diary.
8. **Return the new `memory_ids` and `new_neuron_ids`** — optimistic write (§2.8): the client renders immediately.
   `new_neuron_ids` are the neurons genuinely _created_ in step 4 (not the ones deduped onto existing neurons) — newness
   is a server-only decision, so it is surfaced here for the latent-star **awaken** animation ([25.latent-star-field](25.latent-star-field.md),
   [E7a]). Embeddings and gist texts are `NULL` until the worker fills them and surface on the next read (`GetUniverse`),
   no polling.

The transaction makes Diary + EpisodicMemory + Neuron + NeuronActivation + synapses one atomic unit: either a launch
lands wholly or not at all. Per-user scoping applies to every insert and every dedup-candidate query.

### Thin handler / dependency rule

`memory/rpc` handlers only map proto DTOs to domain inputs and call `Encode` / `PersistEncoded`; the split-count check,
the ≥1-semantic-neuron guideline, the dedup identity rules, the retry/token caps, and the monotonic guard all live in
the use-case / domain (§2.9#7). No proto message or sqlc row is seen by the use-case or the pure domain (§2.9#4); the
`Extractor` and repositories are consumer-owned interfaces (§2.9#6).

## Acceptance Criteria

1. `Encode` splits a diary body into **2–5 episodic memories** on event boundaries (place/person/activity/topic shifts),
   **not** emotion shifts; a result outside `[encode.min_memories, encode.max_memories]` is a retry/repair condition, not
   a silent clamp. ([E1][E2])
2. Each proposed memory carries an LLM-suggested **name** ([W2a]) and one **primary mood** from the 13-mood enum ([M1]);
   both are returned for the user to edit ([W2][W4]).
3. Each memory has **≥1 semantic neuron**, achieved as an extraction guideline (retry/repair when missing) — **never** by
   inserting a hollow placeholder neuron. ([E4])
4. Neurons are typed as exactly one of **semantic / spatial / entity**; spatial and entity are optional. ([E3][E5])
5. **Time is never emitted as a neuron** — the diary date informs temporal handling only and appears in no neuron. ([E6])
6. Neuron dedup is **conservative and type-differentiated**: entity = same referent; spatial = synonyms only, preserving
   granularity ("스타벅스" ≠ "스타벅스 강남점"); semantic = strictest. Merging is an identity judgement, distinct from
   similarity linking ([L3]). The embedding-assist similarity threshold is tunable (value set at implement). ([E10])
7. The encode output schema contains **only** `{memories:[{name, mood, neurons:[{name, type}]}]}` — no position, color,
   strength, seed, time, or delete field exists — so a prompt injection has no field in which to emit an
   invariant-violating value ([I1][I3][I5][I10]); this backs the strong prompt engineering as the second defense. ([W4a])
8. `ReviseSplit` re-runs the split from a natural-language instruction + the prior result against the same schema-forced
   output, bounded by `encode.max_revise_retries` and `encode.max_output_tokens`; exceeding a cap returns a canonical
   error, not an unbounded loop. ([W4a])
9. `SplitDiary` (and `ReviseSplit`) are **synchronous previews** that **persist nothing** — no Diary, EpisodicMemory,
   Neuron, or job is written. ([W2])
10. `PersistEncoded` (via `LaunchStars`) **atomically** persists Diary + EpisodicMemory + Neuron (deduped) +
    NeuronActivation in one transaction, sets `seed` and arousal-modulated `base_strength`, hands off to `Link`, and
    **enqueues `embed` + `semanticize` jobs**; it returns `memory_ids` **and `new_neuron_ids`** (the genuinely-created
    neurons, for the awaken animation) optimistically (embeddings/gist fill on next read). ([E7][E7a][E8a][C7], §2.8)
11. The **Diary is immutable** — `PersistEncoded` inserts it once (append-only), and nothing in this unit updates it;
    `Encode`/`ReviseSplit` never write it. ([I2])
12. A **past-dated** diary (`diary_date < latest launched`) is **saved** but launches **no star**; the monotonic guard is
    enforced at launch. (Clock advance is Epic B; the writing flow surfaces a one-time confirmation notice — plan 27.)
    ([I10][T1])
13. RPC handlers are **thin** (proto↔domain map + call); split-count, semantic-neuron, dedup, cap, and monotonic policy
    live in the use-case/domain; no proto/sqlc type leaks into the use-case or pure domain; the `Extractor` and
    repositories are consumer-owned interfaces. (§2.4/§2.7/§2.9#4/#6/#7)
14. Every neuron fetch, dedup-candidate query, and write is **scoped to the authenticated user**. (§4)
