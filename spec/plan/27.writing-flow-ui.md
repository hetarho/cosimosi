# 27. writing flow ui

> Writing flow UI — the `widgets/writing-flow` panel launched from the universe page: *write a diary → 별 쪼개기
> (synchronous split preview) → edit (touch + natural language) → 별 띄우기 (optimistic launch, a star appears)*. One
> XState machine owns the flow's control-state; the split/edit data lives in Zustand/Query; the launch is optimistic.
> Scope: FE | Status: planning

## Purpose

Epic A's read side (`universe-canvas`, `star-neuron-rendering`, `latent-star-field`, `nebula-color-field`) can render a
memory universe, but nothing puts a star *into* it. This unit is the **write side** — the one flow through which a user
turns a diary into stars. It realizes the product's opening loop (PRD [§4](../PRD.md) step 2, [§5.2](../PRD.md) [W1]–[W5]):
the user writes an entry, sees it split into 2–5 episodic memories each with a suggested name and primary emotion, edits
that proposal by hand or in natural language, and confirms — at which point the diary is persisted and the new star(s)
appear on the canvas.

It exists so that the split preview, the natural-language revise, the session-only editing boundary, and the optimistic
launch are built **once** as a composed widget over four single-action features, rather than scattered across screens.
It consumes the RPC contract owned by [20.encode-usecase](20.encode-usecase.md) (`SplitDiary`, `ReviseSplit`,
`LaunchStars`) and the render substrate + read model owned by [14.rendering-foundation](14.rendering-foundation.md) /
[23.universe-canvas](23.universe-canvas.md); it triggers the star body ([24.star-neuron-rendering](24.star-neuron-rendering.md))
and the awaken animation ([25.latent-star-field](25.latent-star-field.md)) at launch. It builds the flow **UI and its
control-state only** — the backend behavior, the read model, and the visual bodies are their own plans.

## Scope / Non-goals

**In scope**
- **`widgets/writing-flow`** — a modal / bottom-sheet, launched from the universe page's **"일기 쓰기"** affordance
  ([W1]); the entry point is distinct from the final **"별 띄우기"** confirm ([W3]). The widget composes the four
  features below and mounts over the running `widgets/universe-canvas` scene (it does not remount the renderer, [14]).
- **`features/write-diary`** — the diary **body** input plus a **diary-date** picker defaulting to **today** ([W5]).
  Produces the draft `{ body, diary_date }` that seeds the split.
- **`features/split-diary`** — calls **`SplitDiary`** (the **synchronous** Encode preview, [W2] / brief §3). Renders the
  returned **2–5 proposed memories** ([E2]), each showing its suggested **name** ([W2a]) and primary **emotion**/mood
  ([W2]) — **editable**. No persistence happens here; the split is a preview.
- **`features/revise-split`** — natural-language re-run via **`ReviseSplit`** ([W4a]) ("회의랑 점심을 한 별로 합쳐줘",
  "이 뉴런과 저 뉴런은 같은 걸로"), sending the prior `SplitDiaryResponse` + the user's instruction; **plus** hand-edit
  of a memory's name and primary emotion, and merge/split of memories ([W4]). All edits are **session-only**.
- **`features/launch-stars`** — calls **`LaunchStars`** with the user-confirmed split; on success the star(s) **appear**
  immediately with a visual effect ([W3]). Optimistic: `LaunchStars` returns memory ids, the client renders the new
  star(s) at once, and embeddings / gist texts fill on the **next** `GetUniverse` read (§2.8, [23]).
- **The flow control-state as one XState machine** — `model/writing-flow.machine.ts` (ARCHITECTURE §3.2): the exclusive
  phases `idle → writing → splitting → reviewing → revising → launching → done` (+ error paths). The `splitting` /
  `revising` states are the **loading** states that cover the synchronous LLM latency. Draft body, diary date, the
  proposed memories, and the user's edits live in **Zustand/Query**, not in machine context (the machine holds ids /
  refs and the current phase only, §3.2).
- **Session-only editing boundary** ([W4]): inside the write session the user may merge/split memories, change a
  memory's primary emotion (simple selection), and fix normalization — via **touch AND natural language**. Already
  launched past stars are **not** directly editable from here; their representation changes only through natural
  processes (recall / forgetting / consolidation, Epics C–E). The widget states and enforces this boundary.
- **Schema-forced editable surface** — the proposed-split UI exposes **only** name / emotion / neuron membership; it has
  **no** position, color, strength, or time controls (mirrors the [W4a] output-schema defense — the user never sets
  position/strength/time [I3]).
- **UI copy via i18n** (`packages/i18n` / the app `shared/i18n` seam, [08.i18n-seam](08.i18n-seam.md)) — no hardcoded
  strings; a **literary, restrained Korean** voice (not AI-flavored). **web + mobile parity** — the same feature slices;
  a `*.native.tsx` sibling exists **only** for the input / sheet primitives that genuinely differ (ARCHITECTURE §3.5).

**Non-goals**
- **No backend behavior.** `SplitDiary` / `ReviseSplit` / `LaunchStars`, the `Extractor` port, dedup, `Link`, and the
  enqueued `embed` / `semanticize` jobs are owned by [20.encode-usecase](20.encode-usecase.md) /
  [21.link-usecase](21.link-usecase.md) / [22.ai-worker-pipeline](22.ai-worker-pipeline.md). This unit only *calls* the
  contract and renders the result.
- **No read model or star body.** The `GetUniverse` read + domain-mirror entities are [23.universe-canvas](23.universe-canvas.md);
  the star / cell-star / filament visual bodies are [24.star-neuron-rendering](24.star-neuron-rendering.md); the latent
  field + awaken animation are [25.latent-star-field](25.latent-star-field.md). This unit **triggers** the appearance;
  it does not implement the body or the animation.
- **No recall / detail panel** (Epic C) — reading or reconsolidating a past star is a different flow.
- **No forgetting visuals** (Epic D), **no consolidation replay** ([V8]), **no nebula color logic** ([26]).
- **No economy gating** ([G], Epic F) — whether a launch costs 별가루 is deferred; the flow assumes it may proceed.
- **No diary archive reader / export UI** ([D], [W6], Epic G) — the write flow persists the immutable diary; reading it
  back and CSV/MD export are their own plans.
- **No universe-clock advance.** Time monotonicity ([I10]/[T1]) is enforced server-side at launch (Epic B seam); this
  unit surfaces the **past-date** outcome to the user but does not own the clock.
- No new tuning scalars hardcoded in FE (they come from generated config, §Policy).

## Design

### Placement (FSD, §3.1)
```
apps/web/src/widgets/writing-flow/
├── ui/            WritingFlowSheet (modal/bottom-sheet host) + panels; *.native.tsx siblings for sheet/input primitives
├── model/         writing-flow.machine.ts (XState) · the draft/proposal Zustand store · pure edit helpers (merge/split)
└── index.ts       public API (the launch affordance + the sheet)

apps/web/src/features/
├── write-diary/   ui (body input + diary-date picker) · model (draft store slice)
├── split-diary/   ui (proposed-memory list, name+emotion editors) · api (SplitDiary call + proto↔domain map)
├── revise-split/  ui (NL instruction input + hand-edit controls) · api (ReviseSplit call)
└── launch-stars/  ui (별 띄우기 confirm) · api (LaunchStars call + optimistic cache write)
```
- `widgets/writing-flow` composes the four `features/*` (each = **one user action**, §3.1) and owns the **flow machine**
  and the draft/proposal store. It imports the domain-mirror entities (`entities/episodic-memory`, `entities/neuron`)
  from [23] to *shape* the proposal (the optimistic insert is **memory-level only** — see Launch); it never imports `three` or a visual entity
  (§3.4). The launch's *visual* consequence (star body, awaken) is the read model's projection, not this widget's code.
- **web + mobile parity (§3.5):** the same widget + four features on both apps; `model` and `api` segments are shared
  verbatim (pure, no DOM/native). Only `ui` forks where a primitive differs — the **sheet host** (web modal / DOM ↔ RN
  bottom-sheet) and the **text input** / date picker — via `*.native.tsx` siblings. web = open-beta scope, mobile = MVP.

### The three RPCs (contract owned by [20], consumed here — brief §3, §2.7 unary)
- **`SplitDiary(SplitDiaryRequest{ body, diary_date }) → SplitDiaryResponse{ repeated ProposedMemory memories }`** —
  the **synchronous** split preview; the user waits and sees the result ([W2], not background). Each
  `ProposedMemory{ name, mood, repeated ProposedNeuron{ name, type } }`. **No persistence.** The `splitting` machine
  state covers this call's latency.
- **`ReviseSplit(ReviseSplitRequest{ body, diary_date, SplitDiaryResponse previous, string instruction }) →
  SplitDiaryResponse`** — the natural-language re-run ([W4a]); same schema-forced output. Sent with the *current*
  (possibly hand-edited) proposal as `previous`. The `revising` state covers its latency.
- **`LaunchStars(LaunchStarsRequest{ body, diary_date, repeated ConfirmedMemory memories }) →
  LaunchStarsResponse{ repeated string memory_ids }`** — persists the diary + episodic memories + neurons + activations
  (dedup applied server-side), runs `Link`, enqueues `embed` + `semanticize` jobs, and **returns ids**.
  `ConfirmedMemory{ name, mood, repeated { name, type } neurons }` = the user-edited split result.
- **Schema-forced boundary ([W4a]/§3.3 brief):** the request/response shapes carry **only** `{ name, mood, neurons[] }`
  per memory — **no** position / color / strength / time / delete fields. The FE editable surface mirrors this exactly:
  there is structurally nothing for the user (or a prompt-injected model) to set that could violate [I3]. The
  proposed-split UI shows only name, emotion, and neuron membership.

### Flow machine (`model/writing-flow.machine.ts`, §3.2)
Control-state only (the "one of N phases" of the write session); **all data** (draft body, diary date, the proposed
memories, edits) lives in the Zustand/Query store, referenced by id from the machine.

| State | Meaning | Exits |
|---|---|---|
| `idle` | sheet closed | `OPEN` → `writing` |
| `writing` | body + diary-date being entered ([W5]) | `SPLIT` → `splitting`; `CLOSE` → `idle` |
| `splitting` | **loading** — `SplitDiary` in flight ([W2]) | `done` → `reviewing`; `error` → `writing` (retriable) |
| `reviewing` | editable proposal shown ([W2][W2a][W4]) | `REVISE` → `revising`; `EDIT` (self, hand-edit) ; `LAUNCH` → `launching`; `BACK` → `writing` |
| `revising` | **loading** — `ReviseSplit` in flight ([W4a]) | `done` → `reviewing`; `error` → `reviewing` (retriable) |
| `launching` | `LaunchStars` in flight; optimistic insert already applied | `done` → `done`; `error` → `reviewing` (rollback) |
| `done` | star(s) appeared; sheet closes to the universe | `RESET` → `idle` |

- `splitting` and `revising` are explicit loading states so the UI can show a restrained "쪼개는 중" affordance during
  the synchronous LLM call, and so a failed/timed-out call returns to a retriable state (retry/token caps are the
  backend's `encode.*`, [20] — the FE only surfaces the ceiling, it does not set it).
- Per §3.2, if any part of the widget drives the canvas per-frame it reads the machine via `getSnapshot()` in
  `useFrame`, never via React state. (In practice the launch visual is a discrete event handed to the read model.)

### Editing — session-only, meaning-inputs only ([W4], [I11])
- **Within the session**, on the `reviewing` state, the user may: rename a memory; change its **primary emotion** (a
  simple mood selection); **merge** two memories into one or **split** one into two; and correct **neuron normalization**
  (e.g. mark two neurons as the same, or keep a distinction) — the `[E10]` dedup correction. Both **touch** (hand-edit
  controls) and **natural language** (`ReviseSplit`) are supported paths to the same edits.
- Hand-edits are applied **locally** to the proposal store (pure merge/split helpers in `model`); NL edits round-trip
  through `ReviseSplit` and replace the proposal. Either way the *edited proposal* is what `LaunchStars` confirms.
- **Boundary:** editing is **session-only**. Once launched, a star is a persisted past memory and is **not** directly
  editable from this flow — its representation changes only through natural processes (recall / forgetting /
  consolidation, Epics C–E), and the objective `Diary` never changes at all ([I2]). The widget enforces this by scope:
  it edits the *proposal* (pre-launch), never a `GetUniverse` memory (post-launch).

### Launch — optimistic, and the visual consequence (§2.8)
- On **별 띄우기**: the machine enters `launching`, the client **optimistically inserts** the confirmed memories into the
  `entities/episodic-memory` store using the `memory_ids` returned by `LaunchStars`, and the canvas renders the new
  star(s) **immediately** with a launch visual effect ([W3]). The insert is **memory-level only**: the deduped neuron
  ids, synapse edges, and emergent positions are server-decided and **not** in the response — they, along with
  embeddings and the 4 gist texts ([C7]), surface on the **next** `GetUniverse` read (a refetch), per the optimistic
  write pattern (§2.8). No polling, no streaming (§2.7).
- **Awaken:** a new star appearing **awakens** a latent star — the entry animation owned by `features/awaken-neuron`
  ([25]). This flow forwards the **`new_neuron_ids`** returned by `LaunchStars` (the genuinely-created neurons — newness
  is a server decision); [25] chooses a nearby latent anchor and plays the awaken for each.
  The latent field is rendering-only (no DB, [E7a]/[V7]); the seed anchor is presentation, the final position emergent.
- **Failure / rollback:** if `LaunchStars` fails, the optimistic insert is rolled back and the machine returns to
  `reviewing` (the proposal is intact and retriable). The diary is only persisted on success ([I2] — persisted
  immutable, atomically with its memories, at launch; not at split — brief §7).

### Past-date handling ([W5]/[T1]/[I10]) — surfaced, not owned
- The diary date defaults to **today** ([W5]) and is user-editable. Time is monotonic ([I10]): a diary dated **before**
  the latest launched memory's universe-time saves the `Diary` (readable later) but launches **no** star ([T1]).
- The server enforces the monotonic guard at launch (the `universe_state` clock is Epic B; brief §7 places the guard as
  a seam here). This flow **surfaces the outcome** to the user in restrained copy ("이 날짜의 기록은 우주에 별을
  띄우지 않고 보관돼요" or similar i18n copy) rather than silently dropping the star. **Decided (2026-07): Epic A
  shows a one-time confirmation notice** — before a past-dated launch the flow tells the user the diary will be saved
  without a star and asks them to confirm; after that, save the diary, skip the star. Epic B (universe clock) revisits
  the full past-date experience.

### Copy (i18n) — literary, restrained Korean
- All user-visible strings resolve through the i18n seam ([08.i18n-seam](08.i18n-seam.md) / `packages/i18n`); none are
  hardcoded (enforced by `lint:raw-strings`). The visible layer uses the poetic UI vocabulary (별 쪼개기 / 별 띄우기 /
  기억의 별) from PRD [§3.1](../PRD.md), in a restrained literary voice — no decorative emoji, no translation-ese, no
  clerical AI phrasing. Web and mobile draw from the **same** message keys (parity).

## Acceptance Criteria
1. A **"일기 쓰기"** affordance on the universe page opens the `widgets/writing-flow` modal / bottom-sheet **over** the
   running universe canvas (the renderer is not remounted); the entry point is distinct from the final "별 띄우기" confirm
   ([W1][W3]).
2. `features/write-diary` provides a diary **body** input and a **diary-date** picker whose default value is **today**
   ([W5]); the date is user-editable.
3. **별 쪼개기** calls **`SplitDiary`** as a **synchronous** preview and renders **2–5** proposed memories ([E2]), each
   showing its suggested **name** ([W2a]) and primary **emotion/mood** ([W2]); the split performs **no** persistence.
4. Each proposed memory's **name** and **primary emotion** are **editable**, and memories can be **merged / split** and
   neuron **normalization** corrected — all within the session ([W2][W4][E10]).
5. A **natural-language** instruction re-runs the split via **`ReviseSplit`** (sending the current proposal + the
   instruction) and replaces the proposal with the same schema-forced output ([W4a]); hand-edit and NL edit reach the
   same result.
6. The editable surface exposes **only** name / emotion / neuron membership — there are **no** position, color,
   strength, or time controls anywhere in the flow ([W4a][I3]).
7. Editing is **session-only**: an already-launched (past) star is **not** directly editable from this flow; the widget
   edits the pre-launch proposal, never a `GetUniverse` memory ([W4]).
8. **별 띄우기** calls **`LaunchStars`**, which returns memory **ids**; the client **optimistically** inserts the new
   star(s) and the canvas renders them **immediately** with a visual effect ([W3]), while embeddings / gist texts fill on
   the **next** `GetUniverse` read (§2.8) — no polling, no streaming.
9. A newly launched star **awakens** a latent star (the `features/awaken-neuron` animation is triggered, [25]); this
   flow emits the launch event and does not itself implement the animation or the star body.
10. The `Diary` is persisted **immutable** and only at **launch** (atomically with its memories) — not at split; a failed
    launch rolls back the optimistic insert and returns to the editable proposal, leaving nothing persisted ([I2]).
11. A **past-dated** diary (before the latest launched universe-time) **saves the diary but launches no star**, and the
    flow surfaces that outcome as a **one-time confirmation notice** before launch, in i18n copy ([W5][T1][I10]).
12. The flow's **control-state** is one XState machine (`idle → writing → splitting → reviewing → revising → launching →
    done` + error paths) with `splitting`/`revising` as loading states; **draft body, diary date, proposed memories, and
    edits live in Zustand/Query**, not in machine context (§3.2).
13. **All** user-visible copy resolves through the i18n seam (no raw strings; `lint:raw-strings` passes), in a literary /
    restrained Korean voice.
14. **web + mobile parity:** the same widget + four features run on both apps, sharing `model` / `api` verbatim;
    `*.native.tsx` siblings exist **only** for the sheet / input / date-picker primitives (§3.5).
