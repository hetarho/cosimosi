# 56. 2D UI design language

> 2D UI design language — the product's visual identity for DOM surfaces (typography application, color-role
> application, spacing/density rhythm, component look, 2D motion, iconography, layout & information hierarchy). Scope:
> FE | Status: planning
>
> This is a **design plan** — a different kind than the prior build plans. Prior plans were _developer-verified_
> (tests/build/vet pass = done, one sequential pass). This one is _designer-reviewed_ and completed through
> **continuous iterative feedback**: its Definition of Done is a design-reviewer sign-off against a rubric across
> review rounds, not a single test pass (see Acceptance Criteria). Its durable output — the living guideline
> ([tech/design-language.md](../tech/design-language.md), created when the job completes) — is the SSOT every future
> presentation unit cites.

## Purpose

[09.design-system](09.design-system.md) built the design-system **mechanism** — the token machine (`tokens.ts` →
`theme.gen.css`), the accessible primitives (`packages/ui`), the a11y baseline, and the theme/background seam. It
**deliberately did not** define the product's visual identity: its non-goals exclude "mood-color product model" and
"final product screens", and it reserves "a safe token seam for later product colors", stating "Future product units
may map domain emotion to visual color inside the rendering projection". Plan 09 owns _how styling is expressed_, not
_what the product looks like_.

That reserved identity has never been authored. Epic A shipped functional DOM screens (the writing flow) against
placeholder styling, and every remaining presentation unit ([31](31.time-acceleration-ui.md),
[35](35.star-detail-panel.md), [36](36.recall-flow-ui.md), [45](45.stardust-ui.md), [47](47.diary-reader-page.md),
[50](50.deletion-ui.md), [52](52.settings-page.md), [53](53.auth-universe-gate.md)) will invent look-and-feel ad hoc
unless a coherent language exists first. Ad-hoc styling also erodes the property Epic A earned — **zero hardcoded
style escapes** in feature/widget/page slices (every value comes from a token). This plan fills the seam plan 09
reserved: it establishes the **2D visual identity** for DOM surfaces and captures it as a **living guideline** all
presentation units consume, so the language is designed once (against the full known surface) rather than re-derived
per screen.

Its sibling [57](57.3d-assets-and-background-design.md) owns the 3D side (star shape, universe visuals, background).
Together they make the product read as one coherent thing.

## Scope / Non-goals

**In scope** — the visual identity of DOM surfaces:

- **Typography** — application of the type scale: hierarchy, weights, line-height rhythm, measure, casing.
- **Color application** — how the semantic color roles (`bg`/`surface`/`text`/`primary`/`danger`/…) and the emotion
  palette are _applied_ in 2D chrome (surfaces, text, borders, states, accents). The palette _values_ are referenced
  from plan 57; this plan owns their _2D application rules_, not the values.
- **Spacing / density / layout** — spacing rhythm, density, grid, and information-hierarchy principles for panels,
  HUD, modals, list rows, and full pages.
- **Elevation & depth** — surface layering, shadow/overlay usage, focus/hover/pressed treatments.
- **Component visual style** — the _look_ of the `packages/ui` primitives (Button, IconButton, Dialog, Toast, Badge,
  TextField, …) and the composed chrome patterns (panel, HUD, modal), restyled within plan 09's primitive API.
- **2D motion** — transition and micro-interaction language built on the existing `duration`/`ease` tokens
  (enter/exit, state changes, feedback), honoring reduced-motion.
- **Iconography** — icon style/weight conventions (usage, not a full icon set unless a slice needs one).
- **State treatments** — empty / loading / error / disabled visual language.
- **The living guideline** — [tech/design-language.md](../tech/design-language.md) (2D section), authored at job
  completion, plus a **design showcase surface** (a route) that renders the language across a canonical reference set
  on web + mobile — the substrate the designer reviews each round.
- **Retrofit** — the shipped Epic A DOM chrome (the writing-flow screens) is re-skinned to the new language as the
  first application and proof.

**Non-goals**

- **All 3D / rendering visuals** — star shape/geometry/material, cell-star / synapse filament / latent-field / nebula
  aesthetics, universe background/skin/atmosphere, and 3D visual states (dimming/gist/awaken) & choreography →
  [57.3d-assets-and-background-design](57.3d-assets-and-background-design.md). (This is the same boundary
  [24.star-neuron-rendering](24.star-neuron-rendering.md) now disclaims — the star's _visual design_ is plan 57's.)
- **The emotion palette values** (the redesigned default mood-color aesthetic) — owned by plan 57; this plan
  _references_ them for emotion chips / state accents to stay consistent.
- **The `mood → color` mapping structure and swap mechanism** — domain, owned by [17.emotion-model](17.emotion-model.md)
  (default single-entry mapping) and [51.palette-customization](51.palette-customization.md) (user swap). This plan
  touches neither the meaning layer nor the mapping ([I11]).
- **The token _mechanism_, primitive _API/behavior_, a11y semantics, theme-seam plumbing** — owned by plan 09; this
  plan restyles _within_ that mechanism and must not regress it.
- **Product copy** — comes through [08.i18n-seam](08.i18n-seam.md); the language defines slots, never strings.
- **New product tuning numbers** — none expected (visual tokens are not `spec/values.yaml` config, per plan 09).

## Design

### What the language defines

The 2D design language is the set of decisions that make DOM surfaces read as one product. It is authored as
[tech/design-language.md](../tech/design-language.md) (2D section) and demonstrated on the showcase surface. It
covers: typographic system (roles + rhythm), color-role application (including how emotion accents appear in 2D),
spacing/density/layout & hierarchy, elevation/depth, per-primitive visual style + composed chrome patterns (panel /
HUD / modal / list), 2D motion & micro-interactions, iconography conventions, and empty/loading/error/disabled
treatments.

### Architecture placement — expressed through the system, not per-screen

All visual changes are expressed **inside `packages/ui`** (`ARCHITECTURE` §3.1) — updated token values in
`tokens.ts` (regenerated to `theme.gen.css` via `pnpm gen:tokens`), restyled primitives in
`packages/ui/src/primitives/*`, and `base.css`. App slices (`apps/{web,mobile}/src/{features,widgets,pages}`)
consume **only** tokens and primitives; they gain no hardcoded color/size escapes. This preserves the property Epic A
earned and is the mechanism that keeps future re-skins O(1): the more the language lives in the token/primitive
layer, the fewer per-screen edits any later change costs. Cross-platform parity follows plan 09 — one token source,
web (`*.tsx`) + native (`*.native.tsx`) siblings only where they genuinely differ (`ARCHITECTURE` §3.5).

### The review substrate — a design showcase surface

Because a design plan has no test to run, its proof is a **reviewable rendered surface**. This plan produces a design
showcase route that renders, on web + mobile: (a) the full primitive catalog in every state (default / hover /
focus / pressed / disabled / loading / error), and (b) a representative **composed-screen set** — enough real chrome
to judge the language in context (e.g. the writing flow, a star-detail panel mock, a HUD, a modal/dialog, a list
page, empty/error states). The showcase is the artifact the design reviewer inspects each round. It builds on plan
09's story/test-harness surface and the [12.test-harness](12.test-harness.md) `/test` page rather than inventing a
parallel one.

### The iterative designer-review process

Completion is **not** a single sequential pass. The work proceeds in **review rounds**:

1. Apply/adjust the language in `packages/ui` + the showcase.
2. The **design reviewer** inspects the showcase and scores each **rubric dimension** — _hierarchy & focus ·
   typographic rhythm · spacing/density · color-role application & contrast · component consistency · motion
   coherence · emotional/brand fit · web↔mobile parity_ — as **Meets / Needs-work / Blocking**, and leaves notes.
3. Notes go into a **feedback ledger** tracked to `resolved` / `deferred` (deferred items are listed explicitly,
   never silently dropped).
4. Repeat until **sign-off**: every rubric dimension **Meets** and **zero open Blocking** items.

The "design reviewer" is whoever is designated (the product owner acting as designer, or an external designer); the
plan is agnostic to who, explicit about the gate.

### Retrofit as first proof

The shipped Epic A DOM chrome (writing-flow screens) is re-skinned to the language as the first real application —
proving the language works on live screens, not just the catalog, and demonstrating the zero-escape discipline holds
through a real migration.

### Relationship to plan 57

56 (this) and 57 share **one** guideline doc so the product reads as a single identity; each owns its section (2D /
3D). The emotion palette **values** are authored in 57 (the universe is where emotion-color is most load-bearing);
this plan **references** them so 2D emotion accents match the 3D universe.

## Acceptance Criteria

> Two tiers. **Tier 1** is objective and checkable. **Tier 2** is the iterative designer-review gate that defines
> "done" for a design plan — sign-off against a rubric across rounds, not a single pass.

1. **(Tier 1 · objective)** A living design-language guideline exists ([tech/design-language.md](../tech/design-language.md), 2D section)
   documenting: typographic system, color-role application (incl. 2D emotion accents), spacing/density/layout &
   hierarchy, elevation/depth, per-primitive visual style + composed chrome patterns, 2D motion, iconography
   conventions, and state treatments.
2. A design showcase surface renders the language across the canonical reference set (full primitive catalog in all
   states + the representative composed-screen set) on **web and mobile**.
3. All visual changes are expressed through `packages/ui` (`tokens.ts` + primitives + `base.css`); feature/widget/page
   slices remain free of hardcoded color/size escapes — the Epic-A zero-escape property is preserved (same guard).
4. The shipped Epic A DOM chrome (writing-flow screens) is retrofitted to the new language.
5. The plan-09 a11y baseline is preserved: token color pairs pass documented WCAG contrast; focus rings visible;
   reduced-motion honored; primitive keyboard/aria behavior unchanged.
6. Theme/background state stays presentation-only — no domain, cache, emotion, strength, or layout mutation.

7. **(Tier 2 · iterative designer-review gate)** The 2D rubric (hierarchy & focus · typographic rhythm · spacing/density · color-role application & contrast ·
   component consistency · motion coherence · emotional/brand fit · web↔mobile parity) is scored by the design
   reviewer each round as Meets / Needs-work / Blocking.
8. A feedback ledger records every round's notes, each tracked to `resolved` or `deferred` (deferred items listed
   explicitly).
9. **Definition of Done:** the design reviewer signs off — every rubric dimension **Meets** and **zero open
   Blocking** — recorded as a dated sign-off. Completion is sign-off-gated and iterative; there is no single
   sequential pass.
