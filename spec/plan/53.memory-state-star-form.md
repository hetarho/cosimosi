# 53. 기억-상태 기반 별 형태 — 단일 축 룩 3종 + 추상화 단계별 지오메트리

> 별마다 고유한 모양을 갖고, 잊혀(추상화)갈수록 형태가 단계적으로 또렷이 변한다 | 범위: FE(+proto Star.abstraction_stage) | 상태: 구현됨(as-built, change 29)
> 사용자는 단일 축 "별 형태(룩)" 3종 중 하나를 고르고, 그 룩이 기억의 추상화 단계(`abstraction_stage`)에 따라 실제 지오메트리를 바꾸며, 같은 룩·단계라도 별마다 시드로 실루엣이 다르다.

## 목적

기억은 저마다 다른 모양이어야 하고, 잊혀갈수록 윤곽이 단순해져 "요지"만 남아야 한다(요지화). 형태 선택지를 **단일 축 룩 3종**으로 줄여 추상화가 또렷이 보이게 하고(plan 52의 별 form×surface 2축을 흡수), 단계 변형을 *부드러운 변위 약화*가 아니라 **단계별 실제 지오메트리 교체**로 만들어 육안으로 구분되게 한다.

## 범위 / 비목표

**포함:**
- **단일 축 룩 3종**(사용자 선택, change 29): `polyhedron`(다면체)·`liquid`(액체→구름)·`spiky`(고슴도치). 각 룩이 모양+질감+4단계 변형을 한 덩어리로 정의. 무료 = `polyhedron`. (별의 form×surface 2축·합성 id·STAR_PRESETS는 폐기 — plan 52에서 흡수.)
- **추상화 단계(`abstraction_stage` 0~`gist_stage_radii` 길이=4) → 단계별 실제 지오메트리**(change 20 야간 요지화가 승급):
  - `polyhedron`: 20면체(Icosa)→12면체(Dodeca)→8면체(Octa)→4면체(Tetra) 면 수 감소.
  - `spiky`: 작은 코어 + 솟은 가시 → 단계가 오르면 가시 개수·높이↓ → 가시 없는 매끈 다각형.
  - `liquid`: 불투명 출렁이는 구슬 → 점점 투명·뭉게 → 구름처럼 흩어짐(투명 단계, alpha 바닥 유지).
- 별마다 **고유 3축 시드**(탄생 시 결정)로 같은 룩·단계 안에서도 in-shader 정점 변위·비대칭 → 실루엣이 별마다 다르다(랜덤성).
- 재공고화(plan 23)의 `form_seed_delta`가 시드에 누적돼 회상 시 실루엣이 미세 변형.

**비목표:**
- 색(감정)·크기(intensity)·밝기(거리) 규칙 변경 — 불변(plan 44 색, change 18·19).
- 나·시냅스의 form×surface 2축(plan 52) — 그대로 유지. 이 plan은 **별** 형태만 단일 축으로.
- AI 기반 *내용* 변형 — 별도(plan 54). 여기선 *형태(지오메트리)* 만.
- 감정별 형태 선택 — 별도(change 30 — 이 단일 룩 모델 위에 얹힌다).

## 설계 요점

- **단일 축 룩 모델.** `StarLook = 'polyhedron'|'liquid'|'spiky'`([entities/star/model/forms.ts](../../frontend/src/entities/star/ui/star-body.ts) `STAR_LOOKS`). 선택 wire(`star_object`)는 룩 id 하나. 미지/레거시는 디폴트 룩 폴백(호환 테이블 없음).
- **단계별 실제 지오메트리(toolkit).** `shared/lib/r3f/shader-art/geometry.ts`의 `polyhedronForStage(stage)`·`spikyGeometry({spikes,spikeLen,sharpness,detail})`(결정론, Math.random 비사용). liquid는 고분할 구 + 단계별 투명/변위. `buildStarBody(look, stage)`가 룩×단계로 지오메트리 + emissive 셰이딩을 조립한다(`STAR_LOOK_BUILDERS`).
- **(룩×단계) 버킷 렌더(헌법8 개정).** 별을 abstraction_stage로 버킷 분류해 **단계마다 InstancedMesh 1개**로 그린다([StarField.tsx](../../frontend/src/entities/star/ui/StarField.tsx)). 메시 개수 = 단계 수(상수, 별 수 무관) — 인스턴싱 이득 보존. 별이 단계를 넘으면 해당 버킷으로 이동(stars 갱신 시 재분류). 좌표·attribute·focus·raycast·탄생/공명 빌보드는 글로벌↔버킷 슬롯 매핑으로 라우팅.
- **단계는 빌드타임(어느 지오메트리), in-shader 노드 아님.** per-star 랜덤성(시드 변위·비대칭)만 in-shader. 단계 simplify는 빌드타임 스칼라로 변위 진폭을 줄여 높은 단계를 더 깔끔하게.
- **결정론.** 같은 memory id·stage·seed → 항상 같은 모양(세션·기기 무관).

## 수용 기준 (인수 조건)
1. 별 형태 선택이 **단일 축 3룩**(`polyhedron`·`liquid`·`spiky`)이다(form×surface 2축·합성 id 없음).
2. `abstraction_stage`가 오르면 형태가 **육안으로 또렷이** 단계 변형된다(다면체 면 수↓·고슴도치 가시 소멸·액체→구름). 단조.
3. 같은 룩·단계라도 별마다 실루엣이 다르다(시드 변위 — 회전만이 아님).
4. 같은 memory id·stage·seed면 항상 같은 모양(결정론).
5. 색(mood)·크기(intensity)·밝기(거리)는 불변(회귀 경계).
6. 렌더가 (룩×단계) 버킷별 InstancedMesh로 분할되고 메시 개수가 별 수와 무관한 상수다.
