# 23. 재공고화 재성형 + 변천사 데이터 - PE 게이트 별 재성형 + append-only evolution_history (as-built)

> 현재 구현 기준 문서. 회상이 예측 오차(PE) 게이트로 말랑한 창을 열 때만 별을 양방향으로 제한적으로 다시 빚고, 모든 변화를 append-only 변천사에 쌓는 데이터·렌더 계약의 SSOT다. | 범위: FS | 상태: 구현 완료, 반영: 24 변천사 UI·27 야간 요지(`nightly_gist` append)가 이 테이블/필드를 재사용

## 목적

설계 결정 3("별은 살아있다")을 데이터·렌더로 구현한다. 회상은 단순 재점화(11의 `last_recalled_at=now`)에 그치지 않고, 그 회상이 새로운 맥락(예측 오차)을 담을 때만 별을 말랑하게 만들어 양방향으로 다시 빚는다 — 밝아지거나 흐려지고, 색조가 좁게 흔들리고, 형태 시드가 미세하게 jitter한다. 두 생물학적 경계를 지킨다: (1) 강하고 오래된 별일수록 덜 변한다(강도 의존), (2) 변화는 회상한 별 + 직접 이웃(1-홉)에만 한정된다(내용 한정적). 모든 변형은 append-only `evolution_history`에 쌓이고(24가 읽음), 불변 원본 `records`는 한 글자도 건드리지 않는다(헌법1). 공동 회상 강화엔 간격 효과를 얹어, 마지막 함께 본 이후 시간이 길수록 강화 부스트를 키운다.

## 현재 구현

재성형 상태는 가변 별(`memories`) 레이어에만 산다(헌법1: 불변 `records`엔 절대 두지 않음). 한 회상이 별을 재성형하는 경로는 BE `RecallMemory` 안에 있고, FE는 그 누적 상태를 `GetUniverse` refetch로 받아 셰이더에 합성한다. 회상 직후 낙관 갱신은 하지 않는다.

- **PE 게이트(`backend/internal/memory/service.go`).** `pe = clamp(1 - cos(recall_ctx_emb, last_consolidated_emb), 0, 1)`. `pe < peThreshold=0.15`면 재성형도 변천사 append도 없는 단순 재점화(11과 동치).
- **MVP 구조적 no-op seam.** 회상 맥락 임베딩(`recall_ctx_emb`)과 마지막 공고화 임베딩(`last_consolidated_emb`)을 둘 다 회상 별 자신의 임베딩으로 채운다(별도 공고화 스냅샷 컬럼이 없다). 그래서 cos=1, pe=0 → 운영에서는 게이트가 항상 막혀 재성형이 일어나지 않는다. `ReshapeContext`의 두 임베딩 필드는 의도된 seam이다 — 미래의 회상 맥락/야간 gist 임베딩이 게이트 코드를 바꾸지 않고 pe>0을 만든다. 단위 테스트는 다른 벡터를 주입해 게이트 통과 경로를 검증하고, 데모(체험)는 novelty를 시뮬레이션한다.
- **강도 의존·재성형 크기.** `strength = clamp(0.15·log2(1+co_recall_total) + 0.30·clamp(age/90d, 0, 1), 0, 1)`. `magnitude = baseStep(0.22)·pe·(1-strength)` → strength↑ ⇒ magnitude↓.
- **양방향 적용(경계 안).** 방향 ±1은 회상 별 id의 FNV 해시 + version에서 결정론적으로 정한다(`directionFor`). 한 step에서: `brightness_offset += dir·clamp(magnitude, 0.10, 0.22)`(누적은 ±1.0 saturate), `hue_shift`는 `dir·magnitude·60°`씩 ±28° 안에서 누적, `form_seed_delta`는 `dir·magnitude·0.5`씩 ±0.6 안에서 누적, `version++`.
- **내용 한정적 범위.** 회상 별 + 직접 이웃(`memory_links` 1-홉)만 재성형. 이웃은 `neighborFactor=0.4` 축소 step을 받고 방향은 회상 별과 같은 dir로 끌려간다. 간접 이웃·나머지 우주는 불변.
- **원자성.** 별 UPDATE(`ApplyReshape`)와 변천사 INSERT(`AppendEvolution`)는 `ReshapeStar`가 한 트랜잭션으로 묶는다 — crash로 version만 오르고 변천사 행이 없는 상태가 생기지 않게.
- **best-effort.** 재성형은 회상의 부가 작업이다. 재성형 읽기/쓰기 실패는 회상을 단순 재점화로 degrade시킬 뿐 불변 원본 Record 반환을 막지 않는다(`RecallMemory`가 `_ = s.reconsolidate(...)`로 에러를 삼킨다).
- **간격 효과(`frontend/src/features/recall/model/co-recall.ts`).** 페어별 마지막 강화 시각을 들고, `delta = CO_RECALL_DELTA(0.05)·spacingBoost(gapDays)`로 부스트한다. `spacingBoost = 1 + SPACING_GAIN(1.0)·clamp(gapDays/SPACING_REF_DAYS(1), 0, 1)` — 한 세션 몰아보기(gap 0)는 1×, 하루 이상 간격은 최대 2×. BE `ReinforceLinks`(11)는 클라 산출 delta를 그대로 멱등 업서트하고 합산 weight를 1.0 cap한다.

캐노니컬 튜닝 값(`peThreshold`/`baseStep`/brightness ±0.10~0.22/hue ±28°/form ±0.6/`neighborFactor`/강도 계수/간격 게인)은 모두 `spec/policy/domain/star.md`(재공고화 재성형 절)와 `spec/policy/domain/memory.md`(변천사 레이어)에 미러된다 — 값의 SSOT는 정책 문서다. service.go 상수와 정책이 동기화 대상이다.

## 도메인 모델 (`backend/internal/memory/memory.go`)

- `Memory`(별 렌더 입력)에 재성형 4필드: `BrightnessOffset`, `HueShift`(도), `FormSeedDelta`, `Version`. 좌표·body는 없다.
- `EvolutionSnapshot`: 한 번의 재성형 이벤트. `Version`, `Brightness`(그 시점 `brightness_offset` 스냅샷), `HueShift`, `FormSeedDelta`, `Trigger`(`'recall'|'new_neighbor'|'nightly_gist'`), `PE`, `Dir`(+1/-1), `CreatedAt`.
- `ReshapeState`: 한 재성형 step의 입출력 누적 상태(`BrightnessOffset`/`HueShift`/`FormSeedDelta`/`Version`).
- `ReshapeContext`: 게이트 입력 — `State`, `RecallEmbedding`, `ConsolidatedEmbedding`(MVP에선 = `RecallEmbedding`), `CoRecall`(incident 링크의 `co_activation_count` 합), `CreatedAt`(age 계산).
- 도메인 타입에는 db/json/proto 태그가 없다(헌법5).

## `RecallMemory` 재성형 경로 (`service.go`)

- `RecallMemory(userID, memoryID)`: `TouchRecall` → `reconsolidate`(best-effort) → `GetRecord` 반환. 별이 없으면 `TouchRecall`은 WHERE-guard로 no-op이고 `GetRecord`가 `ErrNotFound`(→ 핸들러 `NotFound`).
- `reconsolidate`:
  1. `GetReshapeContext`. 임베딩 없는 별(extract/embed 미완)이면 `ErrNotFound` → 재성형 skip(11 재점화만).
  2. `peGate(recall, consolidated)`. 임베딩이 비었거나 zero-norm이면 pe=0(novelty 측정 불가 = 무변; pe=1로 읽지 않음). `pe < 0.15`면 skip.
  3. `ageDays = time.Since(createdAt)`, `magnitude = reshapeStep(pe, strengthOf(coRecall, ageDays))`, `dir = directionFor(memoryID, version)`.
  4. 회상 별: `applyAndLog(..., factor=1.0)`.
  5. `ListDirectNeighbors` 각 이웃: `GetReshapeContext`(임베딩 없으면 continue) 후 `applyAndLog(..., factor=neighborFactor=0.4)`.
- `applyAndLog`: `reshapeState(prev, magnitude, dir, factor)`로 다음 상태를 만들고 `ReshapeStar`(UPDATE+INSERT 한 트랜잭션)에 `trigger='recall'`로 넘긴다.
- 순수 헬퍼(단위 테스트 대상): `cosineSim`, `peGate`, `usableEmbedding`, `strengthOf`, `reshapeStep`, `directionFor`, `reshapeState`, `clampf`.

## `GetEvolutionHistory` (데이터 계약, UI는 24)

- `GetEvolutionHistory(userID, memoryID)`: repo 위임, user_id 격리. 그 별의 변천사를 version 오름차순으로 반환. 빈 목록 유효(재성형 안 된 별).
- proto rpc는 `NO_SIDE_EFFECTS`(HTTP GET 노출).

## 현재 RPC 표면 변화

- `MemoryService`에 `GetEvolutionHistory(GetEvolutionHistoryRequest) returns (GetEvolutionHistoryResponse)` 추가(idempotent).
- `Star`에 재성형 4필드 추가: `brightness_offset=5`, `hue_shift=6`, `form_seed_delta=7`, `version=8`. (9~13은 후속 스펙: 28 wayfinding·36 resonant·21 valence·26 relevance. 좌표 필드는 영원히 없다 — 헌법3.)
- `GetUniverse`/`ListDormant`/`RecallMemory` 응답의 모든 `Star`는 핸들러 공유 매퍼 `toStar`로 재성형 4필드를 싣는다. `ListDormant`는 재성형 컬럼은 채우지만 `record_id`/`relevance`/`resonant`는 비운다(dormant 검색은 별 단위).

## DB (`backend/internal/db/migrations/00005_reconsolidation.sql`)

- `evolution_history` 테이블 신규(append-only — UPDATE/DELETE 쿼리 0건, 헌법1·2): `id`(PK), `memory_id`(FK→memories), `user_id`(격리), `version`, `brightness`, `hue_shift`, `form_seed_delta`, `trigger`, `pe`, `dir`, `created_at`(DEFAULT now()). 인덱스 `evolution_history_memory_idx (memory_id, version)`.
- `memories`에 재성형 4컬럼 ADD: `brightness_offset`/`hue_shift`/`form_seed_delta`(REAL DEFAULT 0), `version`(INT DEFAULT 0).
- `records`에 대한 UPDATE/DELETE DDL·쿼리는 없다(`grep "UPDATE records"`/`"DELETE records"` 0건 — 쿼리 디렉터리 전체 확인). `00001`은 수정하지 않는다.
- 27(야간 공고화, `00006`)은 이 테이블·필드를 재사용한다 — 야간 요지는 `trigger='nightly_gist'`로 INSERT만, `form_seed_delta`를 단조 증가시킨다.

## 주요 쿼리/저장소

`backend/internal/db/queries/memory.sql`(sqlc):

- `GetReshapeContext` (:one) — `version`/누적 상태 + `embedding`(`embeddings` JOIN) + `co_recall_total`(incident `memory_links`의 `co_activation_count` SUM) + `created_at`. PE/strength 입력.
- `ListDirectNeighbors` (:many) — `memory_links` 1-홉 이웃 id(`CASE WHEN a_id=@id THEN b_id ELSE a_id`).
- `ApplyReshape` (:exec) — `memories`만 UPDATE(`brightness_offset`/`hue_shift`/`form_seed_delta` set, `version=version+1`). `records`는 안 건드림.
- `AppendEvolution` (:exec) — `evolution_history` INSERT 전용.
- `GetEvolutionHistory` (:many) — version ASC, user_id 격리.
- `ListMemoriesByUser`/`ListDormant` — `m.brightness_offset, m.hue_shift, m.form_seed_delta, m.version`를 추가 select(`ListDormant`는 동일 컬럼 shape).

`backend/internal/memory/repository.go` 포트 + `repository_pg.go` 구현:

- `GetReshapeContext` — `pgx.ErrNoRows` → `ErrNotFound`. pgvector 임베딩을 `embeddingToDomain`으로 `[]float64`로 평탄화하고, MVP라 `RecallEmbedding`/`ConsolidatedEmbedding`에 둘 다 같은 벡터를 채운다.
- `ListDirectNeighbors`, `GetEvolutionHistory` — sqlc 래핑.
- `ReshapeStar` — `newID()`로 evolution id 생성(16바이트 crypto, base64url) 후 `pool.Begin` → `ApplyReshape` → `AppendEvolution` → `Commit`. float64→float32 변환.

## Flutter 동등성 기준

- 재성형은 서버 권위 상태다. Flutter는 `GetUniverse`/`RecallMemory`/`ListDormant`가 싣는 `Star.{brightness_offset, hue_shift, form_seed_delta, version}`를 그대로 받아 렌더에 합성해야 한다 — `aBrightness = clamp(starBrightness + brightness_offset, A_MIN, 1)`, `aSeed = seed + form_seed_delta`, mood 색을 `hue_shift`(도)만큼 휘도 보존 회전. 이 합성식(`reshapedBrightness`/`reshapedSeed`)은 순수 모듈이라 그대로 포팅된다.
- 회상 직후 별 변화는 낙관 갱신이 아니라 다음 `GetUniverse` refetch로 받는다.
- PE 게이트·강도 의존·이웃 한정·변천사 append는 전부 서버에서 일어나므로 클라이언트가 재현할 비즈니스 로직은 없다 — 운영에서는 MVP seam(pe=0)으로 재성형이 구조적 no-op임을 알아야 한다(데모만 novelty 시뮬레이션).
- 간격 효과 부스트는 클라이언트가 co-recall delta를 산출할 때 적용한다(`delta = 0.05·(1+gap_boost)`); BE `ReinforceLinks`는 받은 delta를 그대로 멱등 업서트하고 1.0 cap한다.
- `GetEvolutionHistory(memory_id)`는 version 오름차순 스냅샷을 반환한다(타임랩스 UI는 24).

## 수용 기준

1. 회상의 `pe < 0.15`(novelty 없음)이면 `last_recalled_at`만 갱신하고 재성형하지 않으며 `evolution_history`에 행을 추가하지 않는다.
2. 운영에서는 회상 맥락 임베딩과 공고화 임베딩이 같아 pe=0이므로 재성형이 구조적으로 일어나지 않는다(MVP no-op seam). 별도 공고화 스냅샷이 들어오면 같은 게이트가 pe>0을 연다.
3. 회상의 `pe ≥ 0.15`이면 회상 별의 `brightness_offset`을 `dir·clamp(magnitude, 0.10, 0.22)`만큼, `hue_shift`를 ±28° 이내로, `form_seed_delta`를 ±0.6 이내로 양방향 갱신하고 `version++` 한다.
4. 같은 pe에서 더 자주·오래 공고화된(강도↑) 별일수록 더 작은 magnitude를 적용한다(`magnitude = 0.22·pe·(1-strength)`, `strength↑ ⇒ magnitude↓`).
5. 한 별을 재성형할 때마다 `evolution_history`에 `{version, brightness, hue_shift, form_seed_delta, trigger='recall', pe, dir}` 1행을 INSERT만 하고 기존 행을 UPDATE/DELETE하지 않는다.
6. 재성형은 회상 별 + 직접 이웃(1-홉 `memory_links`)만 변경하고, 이웃은 `neighborFactor=0.4` 축소 step을 회상 별과 같은 방향으로 받는다. 간접 이웃·나머지 우주는 불변.
7. 별 UPDATE와 변천사 INSERT는 한 트랜잭션(`ReshapeStar`)으로 묶여 version과 변천사가 어긋나지 않는다.
8. 어떤 회상·재성형이 일어나도 원본 `records`(body·entry_date·mood·intensity)는 변경되지 않는다(`UPDATE/DELETE records` 쿼리 0건).
9. 재성형이 누적돼도 별·시냅스를 삭제하지 않고 유효 밝기를 `A_MIN=0.05` 바닥 위로 유지한다.
10. `GetUniverse`/`ListDormant`/`RecallMemory`가 반환하는 `Star`는 재성형 4필드를 포함하되 좌표 필드는 포함하지 않는다.
11. 새로고침 후 `GetUniverse`가 누적된 재성형 상태를 반환하고, `StarField`가 `aBrightness`/`aSeed`/`aHueShift`에 합성해 변화가 보인다.
12. 재성형 best-effort: 재성형 읽기/쓰기 실패는 회상을 단순 재점화로 degrade시킬 뿐 불변 원본 Record 반환을 막지 않는다.
13. 임베딩이 없거나(extract/embed 미완) zero-norm인 별은 novelty를 측정할 수 없으므로 pe=0으로 읽고 재성형하지 않는다(pe=1로 읽지 않는다).
14. 같은 페어를 짧은 간격(같은 세션)에 다시 본 co-recall delta는 간격을 두고 본 경우보다 작다(`spacingBoost` 1× → 최대 2×).
15. `GetEvolutionHistory(memory_id)`는 그 별의 스냅샷을 version 오름차순으로 반환한다(빈 목록은 재성형 안 된 별).
16. `entities/memory/model/reshape.ts`·`features/recall/model/co-recall.ts`는 three/React/DOM을 import하지 않는다(순수·모바일 재사용).
17. 셰이더 애니메이션은 공유 `uTime` manual uniform을 `useFrame`이 수동 갱신해 쓴다 — BloomPass가 frozen시키는 TSL 내장 `time` 노드는 쓰지 않는다.
