# 45. stardust ui

> Stardust UI — the presentation unit of Epic F: it makes 별가루 (Twinkle) **visible, priced, and refillable**. A
> persistent **balance HUD** ([G2]) on the universe page shows basic + additional Twinkle (reads `twinkle.v1 GetBalance`);
> a **cost display** shows the price a spend will cost *before* a recall or a gist-view proceeds ([G4]); and the **charge
> flows** ([G3]) let a user earn Twinkle by verified payment, invite, and writing. One XState machine owns the
> charge-sheet's control-state; the balance + cost figures are data.
> Scope: FE | Status: planning
>
> Layer P (presentation) — the third and final unit of Epic F (Sprint 4). Home: a NEW `entities/twinkle` mirror + a
> `widgets/stardust` overlay composing three `features/*`, over the running `widgets/universe-canvas`
> ([23.universe-canvas](23.universe-canvas.md)). It reads the ledger built by [43.stardust-ledger](43.stardust-ledger.md)
> and the earn/charge contract built by [44.earn-spend-usecase](44.earn-spend-usecase.md). The **cost display** is a
> reusable surface the recall flow ([36.recall-flow-ui](36.recall-flow-ui.md)) and the gist-view (in
> [35.star-detail-panel](35.star-detail-panel.md), over [34.view-semantic-usecase](34.view-semantic-usecase.md)) compose;
> it does not itself own recall or view.

## Purpose

Epic F built the economy's *machinery* — the two-tier ledger ([43.stardust-ledger](43.stardust-ledger.md)) and the
earn/spend use-cases with the real spend gate ([44.earn-spend-usecase](44.earn-spend-usecase.md)) — but a ledger the user
cannot see is a tax they cannot understand. This unit is where the economy becomes **legible and honest**: a diarist
should always know **how much Twinkle they have** (basic that resets daily, additional that carries over — [G2]), should
be told **what a recall or a gist-view will cost before it happens** ([G4]), and should have a **clear, un-nagging way to
get more** when they run low ([G3]).

Without this unit the spend gate ([44.earn-spend-usecase](44.earn-spend-usecase.md)) would deduct Twinkle silently and a
denied recall would fail with no explanation and no path forward. It builds three presentation surfaces over the running
universe canvas — the **balance HUD**, the **cost display** shown up front (composed by the recall/view flows), and the
**charge flows** — and realizes the core-loop-protection promise ([G5]): everyday remembering ([M5]) is always affordable
from the daily basic grant, and the cost gate — with the charge affordance behind it — appears only when a spend would
exceed it. It builds the **UI and its control-state only**; the balance derivation, cost-curve values, and earn/spend
transactions are Epic F's use-case plans.

## Scope / Non-goals

**In scope**
- **`entities/twinkle`** — a NEW FE domain-mirror (pure, **no three**; parity with plan 23's mirror entities and plan
  31's `entities/universe-clock`): `model` = the `TwinkleBalance` type (`basic: number`, `additional: number`, and the
  derived `total`) + a small Zustand store; `api` = reads `twinkle.v1 GetBalance` and maps the earn/charge responses back
  into the store. Shared verbatim web + mobile via `packages/`.
- **`features/twinkle-balance-hud`** — the persistent **balance HUD** ([G2]): a small, restrained overlay on the universe
  page showing **basic** and **additional** Twinkle distinctly (basic = daily-reset, additional = permanent carry-over),
  read from `entities/twinkle`. It reads a value the user always has ([G5] — a nonzero basic grant is refilled daily), so
  it renders a figure from first login, not an empty state.
- **`features/spend-cost-display`** — the reusable **cost display** ([G4]): given a pending spend (a recall of a specific
  star, or a gist-view of a specific stage) it shows **the price** that spend will cost, sourced from the ledger's cost
  quote (see Design — the price is *computed server-side* by [43.stardust-ledger](43.stardust-ledger.md)'s cost curve; the
  FE never prices), and whether the current balance covers it. It is **built here, composed by the spend flows**:
  [36.recall-flow-ui](36.recall-flow-ui.md) surfaces it before 회고하기, and the gist-view affordance in
  [35.star-detail-panel](35.star-detail-panel.md) surfaces it before 요지 보기 — the same way plan 27's launch composes
  plan 25's awaken. When the balance is insufficient it offers the charge affordance rather than dead-ending.
- **`features/charge-twinkle`** — the **charge flows** ([G3]): a charge sheet exposing the three earn paths —
  **(1) payment** (a purchase surface that shows the price tiers, collects the platform store receipt, and calls the
  verified `Charge` path); **(2) invite** (a friend-invite affordance that surfaces
  the inviter bonus + the both-sides grant on a valid signup, [G3]); **(3) write-earn feedback** (a restrained
  confirmation that writing a diary granted Twinkle — the write itself is [27.writing-flow-ui](27.writing-flow-ui.md), this
  unit only renders the *reward feedback*, [G3]). All three land credited Twinkle into `entities/twinkle` via the earn
  contract ([44.earn-spend-usecase](44.earn-spend-usecase.md)).
- **`widgets/stardust`** — composes the three features over the universe page and owns the charge sheet's **XState
  machine** (`idle → charging → (paying | inviting) → idle`; §3.2): control-state only, with the balance + cost + charge
  results held in Zustand/Query (figures in data, phase in the machine). It mounts over the running
  `widgets/universe-canvas` and does not remount the renderer ([14.rendering-foundation](14.rendering-foundation.md)).
- **The insufficient-balance path** ([G3][G5]): when a spend's cost exceeds the total balance, the cost display states the
  shortfall in restrained copy and offers to open the charge sheet; the spend does not silently fail. Everyday recall from
  the basic grant never reaches this path ([G5]).
- **UI copy via i18n** ([08.i18n-seam](08.i18n-seam.md)) — no hardcoded strings; a **literary, restrained Korean** voice
  (the cost/charge copy especially — plain and honest about price, no sales pressure, no decorative emoji). **web +
  mobile parity** — same widget / features / entity mirror; `*.native.tsx` siblings **only** for the overlay / sheet /
  HUD primitives that genuinely differ (§3.5).

**Non-goals**
- **No ledger, balance derivation, or cost-curve math.** The two-tier balance (basic daily-reset + additional
  permanent), spend order (basic → additional), and the cost curves (recall-by-decay-depth, gist-by-gist-depth) are
  [43.stardust-ledger](43.stardust-ledger.md). This unit **reads** the balance and the price quote; it never computes a
  price (CC3 — no price constant appears in the FE).
- **No earn / spend transactions.** Earn (write / invite / payment) + Spend (recall / view-gist gating) + the real spend
  gate + core-loop protection are [44.earn-spend-usecase](44.earn-spend-usecase.md). This unit **calls** the earn/charge
  RPCs and **renders** the cost the gate will charge; it does not deduct, credit, or gate.
- **No payment verification logic in FE.** The payment surface starts the platform store purchase and sends the resulting
  receipt to [44.earn-spend-usecase](44.earn-spend-usecase.md)'s verified `Charge`; it never credits Twinkle locally and
  never trusts an unverified receipt. Provider SDK details stay behind the platform UI/API seam, and production value is
  granted only by the backend after verification ([G3]).
- **No invite-abuse policy.** Valid-signup criteria and self-invite / farming prevention ([G6]) are a reserved slot
  owned by [44.earn-spend-usecase](44.earn-spend-usecase.md); this unit renders the invite affordance and the granted
  bonus, not the fraud gate.
- **No recall or view flow.** 회고하기, prediction-error, rewrite, reconsolidation result are
  [36.recall-flow-ui](36.recall-flow-ui.md); gist-star viewing is [34.view-semantic-usecase](34.view-semantic-usecase.md)
  surfaced in [35.star-detail-panel](35.star-detail-panel.md). This unit builds the **cost display** those flows compose
  before spending; it does not build recall or view.
- **No free surfaces priced.** Meta info (shape / emotion / 작성일 / 강도 / forgetting state) and the free forgotten
  current text ([R1][G1]), and reading the original diary ([D3] archive), are **free** — this unit shows **no** cost for
  them (the cost display appears only on the two paying actions: 회고 and 요지 보기).
- **No universe-clock or time overlay.** The sync-consent modal and acceleration are
  [31.time-acceleration-ui](31.time-acceleration-ui.md); this unit adds no time surface.
- No new tuning scalars hardcoded in FE — the price figures come from the server quote, the earn amounts / grant caps
  from generated config owned by Epic F (§Policy).

## Design

### Placement (FSD, §3.1)
```
apps/web/src/entities/twinkle/
├── model/     TwinkleBalance type + Zustand store          (pure, shared via packages/, no three)
├── api/       maps twinkle.v1 GetBalance + earn/charge responses → the store
└── index.ts

apps/web/src/widgets/stardust/
├── ui/        StardustOverlay (balance HUD + charge sheet host) + *.native.tsx sibling for overlay/sheet primitives
├── model/     stardust.machine.ts (XState) · the balance/cost/charge store · pure display helpers
└── index.ts   public API (the overlay + the reusable cost-display component the spend flows compose)

apps/web/src/features/
├── twinkle-balance-hud/  ui (basic + additional balance overlay) — reads entities/twinkle
├── spend-cost-display/   ui (the price + covers?/shortfall panel) · api (reads the ledger cost quote) · model (pending-spend + decision)
└── charge-twinkle/       ui (payment tiers · invite · write-earn feedback) · api (earn/charge calls)
```
The same slices exist on mobile; `model`/`api` are shared verbatim (pure, no DOM/native), and only the `ui` overlay /
sheet / HUD primitives fork via `*.native.tsx` (§3.5). The widget imports the `entities/twinkle` mirror to *read/write the
balance* and the domain-mirror entities from [23] only to *identify the star being spent on*; it never imports `three` or a
visual entity (§3.4).

### The Twinkle mirror + contracts (owned by Epic F, consumed here — §2.7 unary)
`internal/twinkle` is the new bounded context (CC8); its `twinkle.v1` service is the contract this unit calls. This plan
consumes, does not define:
- **`GetBalance(GetBalanceRequest{}) → GetBalanceResponse{ int64 basic, int64 additional }`** — the two-tier balance
  ([G2]). `entities/twinkle` maps it to `{ basic, additional, total: basic + additional }` (`total` is a **derived**
  display value, not a stored field — the spend order basic → additional is the ledger's, not the HUD's, [43]). The HUD
  refetches whenever a spend or earn resolves (no polling; §2.7).
- **`QuoteSpend(QuoteSpendRequest) → QuoteSpendResponse{ cost, covered, shortfall }`** — the price the spend will cost.
  **Decided (2026-07): the cost display reads this server-supplied quote, it never computes a price.** The recall
  cost-by-decay-depth, gist-cost-by-gist-depth, and diary-recall-batch sum live behind
  [44.earn-spend-usecase](44.earn-spend-usecase.md)'s read-only quote, which uses [43.stardust-ledger](43.stardust-ledger.md)'s
  curves (CC3 — "no price constant appears outside F"). What is fixed here: **the FE displays the quote it is given; it
  holds no cost curve.**
- **The earn/charge RPCs** ([44.earn-spend-usecase](44.earn-spend-usecase.md)): the verified payment path,
  the invite path (returns the granted bonus after a valid signup), and the write-earn (fired by the launch in
  [27.writing-flow-ui](27.writing-flow-ui.md); this unit only renders its reward feedback). Each returns the updated
  balance (or the mirror refetches `GetBalance`), so the HUD reflects the credit immediately.
- The **spend** itself is not called here — it is inside recall / view ([44.earn-spend-usecase](44.earn-spend-usecase.md)'s
  gate). This unit shows the cost **before** the spend and reflects the debited balance **after** it (via the recall/view
  flow's completion → `GetBalance` refetch). No price is ever computed or hardcoded in the FE (CC2/CC3).

### The balance HUD ([G2][G5])
`features/twinkle-balance-hud` renders the persistent overlay showing **basic** and **additional** Twinkle distinctly, so
the user reads at a glance both "today's allowance" (basic, resets daily) and "my reserve" (additional, permanent). Basic
is always nonzero at day start (the daily grant, [G5]), so the HUD is never an empty state — it renders a figure from
first login. It updates whenever `GetBalance` refetches (after any spend or earn). The HUD shows figures only — it exposes
no placement / meaning control ([I11]).

### The cost display ([G4]) — reusable, composed by the spend flows
`features/spend-cost-display` is the reusable price surface. Given a **pending spend** — described only by *what action on
which star* (recall of star X, or view of star X at gist-stage S) — it shows:
- **the price** that spend will cost (the server quote, above) — a recall priced by the star's decay depth (deeper decay
  → costlier, [F4][G4]), a gist-view priced by gist depth (deeper gist → cheaper, [G4]);
- **whether the balance covers it** (from `entities/twinkle`), and **which tier pays** is *not* the FE's concern (spend
  order is the ledger's, [43]) — the display only states covers / shortfall;
- on **shortfall**, the shortfall amount + a **charge affordance** (opens the charge sheet) rather than a dead end ([G3]).
It is **built here and composed by**: [36.recall-flow-ui](36.recall-flow-ui.md) (before 회고하기 — the cost is shown in
the recall flow's first step, and the flow proceeds only on confirm), and the gist-view affordance in
[35.star-detail-panel](35.star-detail-panel.md) (before 요지 보기, over [34.view-semantic-usecase](34.view-semantic-usecase.md)).
It **never itself calls spend** — it returns a "proceed" decision (the composing flow then triggers the spend-gated
recall/view). "이 일기로 태어난 별 보기" ([D3], [47.diary-reader-page](47.diary-reader-page.md)) shows the **sum** of that
diary's recall costs via the same display, over a batch quote (the sum is the ledger's, this unit renders it).

### The charge flows ([G3])
`features/charge-twinkle` is the charge sheet with the three earn paths:
- **Payment** — shows price tiers (e.g. the "$1 = 100 별가루" structure, [G3]; the tier figures are Epic F content/config,
  not this unit's), starts the platform store purchase, and sends `{pack_id, platform, receipt}` to [44]'s verified
  `Charge`. The machine's `paying` state covers the store round trip + backend verification; balance updates only after
  the backend returns the credited result.
- **Invite** — a friend-invite affordance ([G3]): surfaces the invite link/share and, on a valid signup, the inviter
  bonus + the both-sides grant (the grant is the earn use-case's; the abuse gate [G6] is reserved there). The machine's
  `inviting` state covers the share/confirm.
- **Write-earn feedback** — writing a diary earns Twinkle ([G3]); the write is [27.writing-flow-ui](27.writing-flow-ui.md).
  This unit renders only the **reward feedback** (a restrained "일기를 써서 별가루가 쌓였어요"-style confirmation) when the
  launch's earn resolves, crediting `entities/twinkle`. There is **no login bonus** ([G3] — the daily basic grant plays
  that role); the charge sheet never offers one.

### Charge-sheet control-state (XState, §3.2)
`model/stardust.machine.ts` holds control-state only:

| State | Meaning | Exits |
|---|---|---|
| `idle` | HUD showing; sheet closed | `OPEN_CHARGE` → `charging` |
| `charging` | charge sheet open (the three paths) | `PAY` → `paying`; `INVITE` → `inviting`; `CLOSE` → `idle` |
| `paying` | store payment + backend verification in flight | `done` → `idle` (balance refetched); `error` → `charging` (retriable) |
| `inviting` | invite share / grant in flight | `done` → `idle` (balance refetched); `error` → `charging` (retriable) |

The balance figures, the pending-spend cost, and the charge results live in Zustand/Query (data); the machine holds the
phase + refs only (§3.2). The **cost display** carries its own tiny control (idle / shown / insufficient) but is driven by
the composing flow's machine (recall/view) — it does not need the charge machine to be shown; opening the charge sheet
from a shortfall transitions this widget's machine to `charging`.

### Golden-parity stance
This is a **presentation** unit — it holds **no** cross-surface math and therefore **no golden-parity obligation**. The
balance derivation and the cost curves that DO carry golden-parity live in [43.stardust-ledger](43.stardust-ledger.md)
(ledger balance derivation is golden-parity per the brief). This unit renders server-supplied figures verbatim: it never
re-derives the balance and never re-prices a spend (CC3). Web and mobile render the **same** figures from the **same**
contract; the only fork is the `ui` primitive layer (§3.5).

### Copy (i18n) — literary, restrained Korean
All strings resolve through the i18n seam ([08.i18n-seam](08.i18n-seam.md)); none hardcoded (`lint:raw-strings`). The
balance labels (기본 별가루 / 추가 별가루), the cost copy (state the price plainly), and the charge copy use the restrained
literary voice (PRD [§3.1](../PRD.md)) — the cost/charge especially are honest and unpressured (state the price, offer the
path, no sales language, no decorative emoji). Web + mobile draw the **same** message keys.

## Acceptance Criteria
1. A persistent **balance HUD** on the universe page shows **basic** and **additional** Twinkle distinctly (basic =
   daily-reset allowance, additional = permanent carry-over), read from `twinkle.v1 GetBalance` via `entities/twinkle`;
   it renders a figure from first login (basic is always granted) and updates whenever `GetBalance` refetches after a
   spend or earn ([G2][G5]).
2. `entities/twinkle` is a **pure** mirror (no three) with a `model` (the `TwinkleBalance` type + store, `total` derived
   as `basic + additional`) and an `api` that maps `GetBalance` + the earn/charge responses; it is shared verbatim web +
   mobile ([G2]).
3. A reusable **cost display** shows, for a pending spend, **the price** the spend will cost — a **recall** priced by the
   star's decay depth (deeper decay → costlier), a **gist-view** priced by gist depth (deeper gist → cheaper) — sourced
   from the ledger's server quote; the FE **never computes a price** ([G4], CC3).
4. The cost display states whether the balance **covers** the spend and, on **shortfall**, shows the shortfall and offers
   a **charge affordance** rather than dead-ending; it returns a proceed/cancel decision only and **never itself calls
   spend** ([G3][G4]).
5. The cost display is **composed by** the spend flows: [36.recall-flow-ui](36.recall-flow-ui.md) surfaces it before
   회고하기 and [35.star-detail-panel](35.star-detail-panel.md) surfaces it before 요지 보기; "이 일기로 태어난 별 보기"
   ([D3]) shows the **sum** of that diary's recall costs via the same display ([G4]).
6. **Free** surfaces show **no** cost: meta info (shape / emotion / 작성일 / 강도 / forgetting state), the free forgotten
   current text, and reading the original diary are never priced — the cost display appears only on 회상 (recall) and 요지
   보기 (gist-view) ([G1][G4]).
7. A **charge sheet** exposes the three earn paths: **payment** (tiers shown, platform store receipt sent to verified
   `Charge`, balance updated only after backend verification), **invite** (inviter bonus + both-sides grant on valid
   signup), and **write-earn feedback** (a restrained reward confirmation when a diary launch earns Twinkle); each credited
   amount lands in `entities/twinkle` ([G3]).
8. There is **no login bonus** anywhere — the charge sheet never offers one; the daily basic grant plays that role ([G3]).
9. **Everyday remembering is never blocked** by the cost gate: a recall affordable from the daily basic grant reaches the
   proceed path without ever hitting the shortfall / charge path ([G5][M5]).
10. The charge sheet's **control-state** is one XState machine (`idle → charging → (paying | inviting) → idle` + error
    paths); the balance figures, pending-spend cost, and charge results live in **Zustand/Query**, not machine context
    (§3.2).
11. The balance HUD, cost display, and charge sheet expose **no** meaning-layer or placement control — they show and
    transact Twinkle only, and never mutate a `Diary` ([I2][I11]).
12. **All** user-visible copy resolves through the i18n seam (no raw strings; `lint:raw-strings` passes), in a literary /
    restrained Korean voice, honest and unpressured about price.
13. **web + mobile parity:** the same widget + three features + the entity mirror run on both apps, sharing `model`/`api`
    verbatim; `*.native.tsx` siblings exist **only** for the overlay / sheet / HUD primitives (§3.5).

## Policy / Values Impact
- **policy/ux** — the rules this UI makes true land in a new `policy/ux/stardust.md` once implemented (written at
  `/implement-job`; **do not** pre-create policy files):
  - **The balance is always shown** — basic (daily-reset) and additional (permanent) Twinkle are visible on the universe
    page ([G2]).
  - **A spend is always priced before it happens** — recall and gist-view show their cost before proceeding; free
    surfaces (meta, free forgotten text, diary read) are never priced ([G1][G4]).
  - **A shortfall offers a path, never a dead end** — insufficient balance opens the charge flow ([G3]); everyday
    remembering from the basic grant never reaches it ([G5]).
  - **No login bonus; earn is write / invite / payment** ([G3]).
  - **No meaning-layer or placement control** in any Twinkle surface ([I11]); the economy gates *access*, never *meaning*.
- **tech/** — the as-built FE rules (the `widgets/stardust` composition over three features + the `entities/twinkle`
  mirror; the `stardust.machine.ts` phase set with the balance/cost/charge in Zustand/Query; the cost display as a
  reusable surface the recall/view flows compose; the balance-refetch-on-spend/earn pattern) are recorded in

  [ARCHITECTURE.md](../ARCHITECTURE.md) §3.1/§3.2/§2.7 rather than adding a new architecture concern.
