# 36. 함께한 기억 — 공명 (shared-memory-resonance)

> 함께한 기억의 별을 **친구에게 보낸다** — 친구가 링크로 받아 **수락하며 자기 관점으로 재작성**하면, 친구의 우주에 *친구가 다시 쓴 그 사건의 별*이 태어나 친구의 우주에 쌓인다. 내가 보낸 별과 친구가 재작성한 별은 **공명(resonance)으로 이어진다** — 두 우주에 걸친 하나의 사건, 두 개의 기억. 같은 사건이 두 뇌에 각각의 엔그램을 남기는 방식 그대로다. | 범위: FS | 선행: 21 (보내는 단위 = 조각 별), 35 (선택 — 공명 상대 우주 방문) | 차단: 37

## 목적 (Why)

- **재구성성이 소셜 레이어의 개념적 심장이다.** concept §함께한 기억 — 같은 사건은 두 사람의 뇌에 각각의 엔그램을 남기고, 회상은 재생이 아니라 재구성이라 두 기억은 처음부터 다르게 쓰이고 시간이 가며 더 갈라진다. "별 복사·전달"이 아니라 **수신자의 재작성으로 새 별이 태어나는** 흐름이 이 비전의 구현이다(수락 = 재작성 완료).
- **공유 없이 잇는다.** 우주 전체를 공개(35)하지 않아도, 별 하나를 콕 집어 한 사람에게만 보낼 수 있다 — 보내는 행위 자체가 *그 조각 하나에 한한* 명시적 opt-in 공개다("함께한 기억은 양쪽 동의로 잇는다", concept §함의).
- **각자의 별은 각자의 변천사를 가진다.** 공명으로 이어진 두 별은 이후 각자의 우주에서 따로 회상되고 따로 변한다 — 같은 사건이 서로의 우주에서 *다르게 빛나고 다르게 변해 가는* 것이 37 겹침 뷰의 재료가 된다.

## 범위

- **포함:**
  - proto 신설 `gift.proto` — `GiftService`(보내기·조회·수락·거절·취소·목록, 전부 인증).
  - DB 마이그레이션 `star_gifts`(토큰 링크·상태 머신) + `resonances`(memory↔memory 쌍).
  - `internal/gift` 바운디드 컨텍스트 — 전송 토큰 발급·수락 트랜잭션(record+memory+resonance 원자 생성)·만료.
  - FE: 별 상세에서 "별 보내기"(`features/send-star`), 받은 링크 페이지 `/gift/$token`(보낸 별 보기 + 재작성 폼), 자기 우주의 **공명 별 마커** + 보낸/받은 목록.
  - `Star` proto에 공명 표시 필드(field `11` — 36 소유).
- **비목표(Out of scope):**
  - **친구 계정 그래프·검색·팔로우** — 수신자는 링크 전달로 특정된다(1:1 토큰). 친구 목록 시스템은 만들지 않는다.
  - **알림(푸시·이메일)** — 수락 사실은 보낸 목록의 상태로 확인. 알림 인프라는 별도 스펙.
  - **공명 별의 공개 노출** — 35의 공개 우주(`SharedStar`)에 공명 마커를 싣지 않는다(제3자에게 관계 비노출). 공명의 시각화는 각자 자기 우주 + 37 겹침 뷰에서만.
  - **그룹 전송(1:N)·재전송 체인** — 1:1 단건만. 같은 별을 여러 친구에게 보내는 건 gift 여러 건으로 자연 지원되나 UI는 단건 흐름만.
  - 수신자 재작성 일기의 **다조각 추출** — 재작성은 *한 사건의 한 기억*이므로 단일 별(조각 1개)로 저장한다(아래 설계).
- ⚠️ **양쪽이 cosimosi 사용자여야 한다** — 수신자는 사인인 후 수락(계정 없으면 가입 유도). MVP 가입 장벽을 받아들인다(비전이 1:1 친밀 공유이므로).

## 설계 요점

**흐름 — 보내기 → 열기 → 수락(재작성) → 공명.**

1. **보내기**: 내 별(조각) 상세에서 "이 별 보내기" → 한마디 메시지(선택)와 함께 `SendStarGift(memory_id, message)` → 토큰 URL(`/gift/:token`) 발급, 사용자가 직접 전달(카톡 등). UI에 **명시 고지**: "이 링크를 받은 사람은 이 별의 조각 글을 읽게 돼요" — 보내기 = 그 조각 텍스트의 1:1 opt-in 공개.
2. **열기**: 수신자가 링크 접속 → 사인인 게이트 → `GetStarGift(token)`이 보낸 사람 표시명·메시지·**그 조각의 텍스트**(전체 일기가 아니라 보낸 조각만)·감정 색을 보여준다.
3. **수락 = 재작성**: 수신자가 *자기 관점으로* 그 사건을 다시 쓴다(기록 폼 재사용, 13감정·강도 입력). `AcceptStarGift(token, text, mood, intensity)`가 한 트랜잭션으로: 수신자의 불변 `record` 생성 → **단일 `memory`(조각 1개, fragment_index=0, extract 생략)** 생성 → `resonances` 행 생성 → embed job 적재(시냅스는 기존 파이프라인이 수신자 우주 안에서 연결). 거절은 `DeclineStarGift`(사유 무— 상태만), 보낸 쪽 취소는 `CancelStarGift`(pending에 한함).
4. **공명**: 양쪽 `GetUniverse`의 해당 `Star`에 `resonant=true`(field 11)가 실리고, FE가 은은한 공명 마커(링/맥동 — 수동 uniform)를 그린다. 별 상세에 "○○의 우주와 공명 중"(상대 표시명) + 상대가 35 공유 중이면 방문 링크.

**왜 extract를 생략하나.** 재작성은 "그 사건 하나"에 대한 기억이다 — 일기 전체가 아니라 한 조각의 짝이므로 1:1 별이 자연스럽고, 공명 쌍(memory↔memory)이 단순해진다. 수신자가 그 날 전체를 일기로 쓰고 싶으면 평소처럼 따로 쓰면 된다(그건 21 경로).

**DB — 토큰 상태 머신 + 공명 쌍.**

```sql
CREATE TABLE star_gifts (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  token             TEXT NOT NULL UNIQUE,          -- base64url 22자(128bit)
  sender_user_id    UUID NOT NULL,
  sender_memory_id  UUID NOT NULL REFERENCES memories(id),
  message           TEXT NOT NULL DEFAULT '',
  status            TEXT NOT NULL DEFAULT 'pending', -- pending|accepted|declined|canceled|expired
  recipient_user_id UUID,                          -- 수락/거절 시 기록
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at        TIMESTAMPTZ NOT NULL,          -- 생성 +30일(미수락 링크가 영구히 떠돌지 않게)
  responded_at      TIMESTAMPTZ
);
CREATE TABLE resonances (
  id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  gift_id             UUID NOT NULL UNIQUE REFERENCES star_gifts(id),
  sender_memory_id    UUID NOT NULL REFERENCES memories(id),
  recipient_memory_id UUID NOT NULL REFERENCES memories(id),
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

- 자기 자신에게 수락 불가(`sender_user_id = caller` → `InvalidArgument`). 만료·취소·중복 수락은 상태 검사로 거부(idempotent 거절). 토큰 NotFound는 35처럼 균일.
- 공명은 삭제하지 않는다(헌법2의 정신) — 별이 잠들어도 공명은 남는다.
- ⚠️ 마이그레이션 번호는 착수 시점 다음 번호(잠정 `00008` — 00.overview §공유 설계 결정이 단일 기준).

**proto — `gift.proto`(전부 인증·unary).** `SendStarGift`/`GetStarGift`/`AcceptStarGift`/`DeclineStarGift`/`CancelStarGift`/`ListStarGifts`(보낸/받은 탭용, NO_SIDE_EFFECTS). `GetStarGift` 응답엔 보낸 조각 텍스트·mood·보낸이 표시명(35의 display_name 재사용, '' = "어느 우주")·status·expires_at. **`Star` field `11 = bool resonant`** — 필드 번호 할당표(00.overview §공유 설계 결정)에 36 소유로 등록. 상세 패널용 `GetResonanceInfo(memory_id)`(상대 표시명·상대 공유 slug 유무)는 `GiftService`에 포함.

**FE.**
- `features/send-star` — 별 상세(회상 패널)에서 "별 보내기": 고지 문구 → 메시지 입력 → URL 발급·복사. pending 목록에서 취소.
- `pages/gift`(`/gift/$token`, SessionGate **안** — 비로그인은 사인인으로) — 보낸 별 카드(조각 텍스트·감정 색·보낸이·메시지) + "내 기억으로 다시 쓰기" 폼(record-memory 폼 재사용·13감정) + 거절. 수락 성공 → 내 우주로 이동, 새 별 fly-to(탄생 연출은 10/21 기존 것).
- 공명 마커 — `entities/star`에 resonant 인스턴스 속성 추가, TSL 은은한 링/맥동(수동 uniform — time 노드 금지). 별 상세에 공명 정보(상대 표시명·방문 링크).
- 보낸/받은 목록 — 31 오버레이 셸 패턴(미완이면 임시 패널).

## 수용 기준 (EARS / Given-When-Then)

1.1 WHEN 별 보내기를 시작하면 THE SYSTEM SHALL "받는 사람이 이 조각 글을 읽게 된다"는 고지를 표시한 뒤에만 토큰 URL을 발급한다(opt-in 명시).
1.2 WHEN `GetStarGift(token)`을 호출하면 THE SYSTEM SHALL 보낸 **조각 하나의 텍스트**·감정·보낸이 표시명·메시지만 반환한다(보낸이의 다른 일기·다른 조각·원본 일기 전문은 포함하지 않는다).
1.3 WHEN 토큰이 무효·만료·취소·이미 응답됨이면 THE SYSTEM SHALL 상태에 맞게 거부하고, 존재하지 않는 토큰은 35와 같은 균일 `NotFound`를 반환한다.
1.4 IF 보낸이가 pending 상태에서 `CancelStarGift`를 호출하면 THEN 수신 링크는 즉시 무효가 된다.
2.1 WHEN 수신자가 `AcceptStarGift(token, text, mood, intensity)`를 호출하면 THE SYSTEM SHALL 한 트랜잭션으로 수신자의 불변 record + 단일 memory(조각 1개) + resonance를 생성하고 embed job을 적재한다 — 부분 실패 시 전부 롤백.
2.2 WHEN 수락이 완료되면 THE SYSTEM SHALL 양쪽 사용자의 `GetUniverse`에서 해당 두 별에 `resonant=true`를 반환한다.
2.3 WHEN 자기 자신이 보낸 토큰을 자기 계정으로 수락하려 하면 THE SYSTEM SHALL `InvalidArgument`로 거부한다.
2.4 WHEN 수락으로 태어난 별이 이후 회상·재성형되어도 THE SYSTEM SHALL 상대 우주의 별에 어떤 변경도 전파하지 않는다(각자의 별은 각자의 변천사 — 공명은 링크일 뿐).
3.1 WHEN 공명 별을 우주에서 보면 THE SYSTEM SHALL 은은한 공명 마커를 렌더하고(수동 uniform 애니메이션), 별 상세에 상대 표시명을 보여준다.
3.2 WHEN 제3자가 35의 공개 우주를 방문하면 THE SYSTEM SHALL 공명 마커·공명 존재를 일절 노출하지 않는다(`SharedStar`에 resonant 필드 없음).
3.3 WHEN `ListStarGifts`를 호출하면 THE SYSTEM SHALL 내가 보낸/받은 gift를 상태·시각과 함께 반환한다(상대의 다른 정보는 표시명뿐).
