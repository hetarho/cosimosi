# 03. rpc transport

> RPC transport - Protobuf and Connect unary transport shared by Go, web, and mobile.
> Scope: FS | Status: built

## Purpose

Every use-case needs one transport seam that works the same way for web and mobile. This unit activates `proto/` as
the single contract source, generates Go and TypeScript transport code, mounts Connect unary handlers in the API, and
provides a shared frontend client facade that later features can consume directly or through TanStack Query. It adds
transport plumbing only; no product use-case or domain model is introduced here.

## Scope / Non-goals

**In scope**

- Buf module/configuration for `proto/` and deterministic code generation via `pnpm gen:proto`.
- A minimal platform proto package with a unary `Ping` RPC used only to prove the transport path.
- Go generated protobuf and Connect code under `apps/api/internal/gen`.
- TypeScript generated protobuf/service descriptors in a pure shared package that both web and mobile can import.
- API composition root updates: Connect mux, h2c, CORS, request-id logging, panic recovery, and interceptor seams.
- Frontend transport factory: base URL config, Connect Web transport, connect-query integration, typed client exports,
  and test doubles.
- HTTP GET support for idempotent unary reads so later read RPCs can be CDN-cacheable.

**Non-goals**

- No auth enforcement or user isolation. Auth interceptors are only a placeholder seam until
  [04.auth-session](04.auth-session.md).
- No business RPCs, domain messages, database access, worker jobs, or AI ports.
- No server streaming, websockets, subscriptions, or polling framework. The architecture is unary-only.
- No polished UI. A test page or hello-world probe may call `Ping`, but transport is not a product screen.
- No production deployment or public API gateway.

## Design

### Contract layout

`proto/` becomes a Buf module with one platform package:

```text
proto/
├── buf.yaml
├── buf.gen.yaml
└── cosimosi/platform/v1/platform.proto
```

`platform.proto` defines only transport proofing:

- `service PlatformService`
- `rpc Ping(PingRequest) returns (PingResponse)`
- `PingResponse.message`, `server_time`, and `request_id`

This service has no domain meaning and must not become a product API. Later feature units add their own packages and
messages using the ubiquitous language.

### Generation outputs

Generation is deterministic and committed:

- Go messages and Connect handlers go under `apps/api/internal/gen/...`.
- TypeScript messages/service descriptors go under `packages/api-client/src/gen/...`.
- `packages/api-client` exports `createApiTransport`, `createPlatformClient`, `createPlatformMockTransport`, generated
  descriptors, and a pure `createPlatformPingQueryOptions` helper. The package has no DOM, Vite, or Metro dependency.

The implemented pins are:

- Buf remote plugins: `protocolbuffers/go v1.36.11`, `connectrpc/go v1.20.0`, `bufbuild/es v2.12.1`.
- Go runtime: `connectrpc.com/connect v1.20.0`, `connectrpc.com/cors v0.1.0`,
  `google.golang.org/protobuf v1.36.11`.
- TypeScript runtime: `@bufbuild/protobuf v2.12.1`, `@connectrpc/connect v2.1.2`,
  `@connectrpc/connect-web v2.1.2`, `@connectrpc/connect-query-core v2.2.0`,
  `@tanstack/query-core v5.101.1`.

### API server

`apps/api/cmd/api/main.go` stays tiny and delegates to `internal/platform.NewHTTPServer`, which wires:

- `net/http` mux;
- Connect handler paths;
- h2c support for local HTTP/2;
- CORS rules for local web/mobile development;
- request-id interceptor;
- panic recovery interceptor;
- logging interceptor with method, request id, status, and duration;
- `/health` and `/` as plain HTTP routes.

`internal/platform` is transport-only. Its `PlatformService.Ping` handler returns `pong`, server time, and the request
id. It has no domain, use-case, database, or product dependency.

### Frontend client

Web and mobile import the same transport facade from `@cosimosi/api-client`. App roots provide:

- API base URL from runtime environment;
- a Connect Web transport;
- a QueryClient integration point for connect-query;
- test transport overrides for unit tests and the temporary `/test` harness.

The client has no global singleton hidden in a feature slice. It is provided at app composition so auth, cache, flags,
and tests can replace it cleanly. The current unit proves the shared import path with a web Vitest probe and a mobile
TypeScript probe.

### Idempotent reads and CDN GET

The transport layer records the convention that read-only unary RPCs are declared with
`idempotency_level = NO_SIDE_EFFECTS` and exposed through GET where Connect supports it. Later use-cases opt into this
convention per method. Mutating calls remain POST. `PlatformService.Ping` carries the option and is covered by a Go
HTTP GET test.

## Acceptance Criteria

1. `pnpm gen:proto` generates Go and TypeScript outputs from `proto/` deterministically and skips cleanly only when no
   `.proto` files exist.
2. The API serves Connect unary RPCs and `/health` from the same `net/http` composition root.
3. `PlatformService.Ping` succeeds from a Go handler test and from a TypeScript client test or probe.
4. Generated TypeScript transport code is importable by both `apps/web` and `apps/mobile` without DOM, Vite, or Metro
   specific dependencies.
5. Request-id, logging, panic recovery, CORS, and auth-placeholder interceptors are wired without embedding business
   policy in handlers.
6. The transport has no server-streaming API and no websocket/subscription path.
7. A read-method convention for idempotent GET and cacheable responses is documented in code/tests.
8. No product domain message, database table, or feature use-case is introduced by this unit.

## Policy / Values Impact

- **policy / ux:** no policy files are needed for this platform-only unit. The unary-only transport rule is owned by
  [ARCHITECTURE.md](../ARCHITECTURE.md) §2.7 and is repeated in `apps/api/internal/platform/doc.go` for code readers.
- **spec/values.yaml:** no entries. Base URLs, CORS origins, and deployment endpoints are runtime environment config,
  not product tuning values. Local CORS origins default in `internal/platform` and can be overridden through
  `COSIMOSI_CORS_ORIGINS`.
- **Dependencies:** depends on [01.platform-foundation](01.platform-foundation.md). It should be available before
  [04.auth-session](04.auth-session.md), [06.client-data-cache](06.client-data-cache.md), and
  [12.test-harness](12.test-harness.md).
