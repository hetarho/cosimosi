# 11. 회상·공동 회상 강화 (recall-reinforce)

> 별을 열어 불변 원본 일기를 읽고 이웃 시냅스로 항해하며, 함께 본 두 별 사이 연결을 클라 로컬+unary 배치로 강화한다. | 범위: MVP | 선행: 04, 09, 10 | 차단: 12

## 목적 (Why)
탐험(회상)이 우주를 강화하는 핵심 루프를 닫는다. 별을 클릭하면 그날의 **불변 원본 일기**가 뜨고(수정 불가), 이웃 시냅스를 따라 다른 별로 항해한다. 두 별을 함께/오가며 회상할 때마다 그 사이 연결이 **헵 규칙**으로 강해지고(공동 회상), 회상한 별은 재점화되어(`last_recalled_at=now`) 다시 밝아질 채비를 한다. 강화는 RN 미지원 스트리밍 대신 **클라 로컬 누적 + unary 배치 업서트**로 영속화한다.

## 범위
- **포함:**
  - FE `features/recall`: 별 클릭→원본 일기 패널(읽기 전용), 이웃 시냅스 항해(클릭한 별의 연결 이웃을 강조·이동), 회상 강화 로컬 누적+디바운스→`ReinforceLinks` 배치 호출, 별 열람→`RecallMemory` 호출.
  - BE `link`: `ReinforceLinks` RPC — `UNNEST` 배치 + `ON CONFLICT` 원자적 업서트(`weight=LEAST(1.0, …+0.05)`, `co_activation_count++`, `last_activated_at=now`).
  - BE `memory`: `RecallMemory` RPC — `last_recalled_at=now` 갱신 + 원본 반환(불변, 수정 경로 없음).
  - BE `db/queries/link.sql`: 공동 회상 배치 업서트 쿼리.
- **비목표(Out of scope):**
  - 감쇠 시각·잠든 별 탐색·강화 후 "밝아짐" 렌더 검증 → **12(decay-dormant)**. (여기선 데이터 영속까지만; 밝아짐 확인은 12에서.)
  - 별 재성형(형태 변형)·변천사 로그(`evolution_history`) → **v1 #20/#21**. 여기서 갱신하는 건 `last_recalled_at`뿐.
  - proto 계약·서버 와이어링·인증 인터셉터 정의 → **02**. 본 스펙은 02 패턴에 RPC 2개를 더한다.
  - 시냅스 렌더·`weight·brightness` 셰이더 → **09**. 여기선 09의 엔티티 스토어를 소비.

## 설계 요점
**계약(proto, 02 패턴으로 추가).** 두 RPC 모두 unary(원칙 6).
```proto
// proto/cosimosi/v1/link.proto (ReinforceLinks)
message LinkPair { string a_id = 1; string b_id = 2; } // 클라가 정규화: a_id < b_id
message ReinforceLinksRequest { repeated LinkPair pairs = 1; } // 디바운스 윈도의 누적 페어
message ReinforceLinksResponse { int32 applied = 1; }
// proto/cosimosi/v1/memory.proto (RecallMemory)
message RecallMemoryRequest { string memory_id = 1; }
message RecallMemoryResponse { Memory memory = 1; } // 04의 Memory(원본 body 포함) 재사용
```

**BE 원자적 배치 업서트(`db/queries/link.sql`, sqlc).** `:copyfrom`은 ON CONFLICT 불가 → `UNNEST` 사용. `+0.05` 상한 1.0(§5).
```sql
-- name: ReinforceLinks :exec
INSERT INTO memory_links (a_id, b_id, weight, link_type, co_activation_count, last_activated_at, created_at)
SELECT a, b, 0.05, 'co_recall', 1, now(), now()
FROM unnest(@a_ids::text[], @b_ids::text[]) AS t(a, b)
ON CONFLICT (a_id, b_id) DO UPDATE
SET weight              = LEAST(1.0, memory_links.weight + 0.05),
    co_activation_count = memory_links.co_activation_count + 1,
    last_activated_at   = now();
-- name: RecallMemory :one  (memory.sql)
UPDATE memories SET last_recalled_at = now() WHERE id = @id RETURNING *;
```
> 같은 윈도에서 한 페어가 2번 와도 1행으로 정규화(`a_id<b_id`)·세트화하므로 +0.05는 회상 "이벤트"당 1회. 동시 호출 원자성은 행 단위 `ON CONFLICT`가 보장.

**FE 로컬 누적·디바운스(`features/recall/model`, 순수 — three/DOM 미의존, 원칙 4).** 별 열람 시퀀스에서 "함께/오가며 본" 페어를 세트에 누적, 유휴/언마운트/임계치에서 `ReinforceLinks` 1회 배치. 능동 인출(열람) 기준 — 단순 스크롤 제외(concept §회상).
```ts
// model/co-recall.ts (순수)
type Pair = `${string}|${string}`;            // 정규화된 a|b (a<b)
interface RecallSession { window: Set<Pair>; lastViewedId: string | null; }
function onView(s: RecallSession, id: string): void;   // 직전 본 별과 페어 추가
function drainPairs(s: RecallSession): LinkPair[];      // 배치 페이로드로 비우기(디바운스/언마운트 시)
```
**FE UI(`features/recall/ui`, 플랫폼 레이어).** 별 선택→`RecallMemory`로 원본 패널 표시(읽기 전용, 편집 컨트롤 없음 — 원칙 1) + `onView` 누적. 패널의 이웃 목록은 09의 synapse 스토어에서 선택 별의 엣지를 읽어 렌더, 클릭 시 카메라 포커스 이동.

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 사용자가 별을 클릭하면 THE SYSTEM SHALL 그 별의 **원본 일기 본문·작성일·감정**을 읽기 전용 패널로 표시하고, 편집·삭제 UI를 노출하지 않는다. _(원칙 1)_
1.2 WHEN 별이 열람되면 THE SYSTEM SHALL `RecallMemory`를 호출해 `last_recalled_at=now`로 갱신한다(본문·원본 필드는 불변). _(원칙 1, §5 재점화)_
1.3 GIVEN 한 회상 세션에서 두 별 A·B를 함께/오가며 열람했을 때 WHEN 디바운스 윈도가 닫히면(유휴/언마운트/임계) THE SYSTEM SHALL 정규화·중복 제거한 페어를 `ReinforceLinks`로 **단일 unary 배치** 호출한다. _(원칙 6)_
1.4 WHEN `ReinforceLinks`가 페어 (a,b)를 받으면 THE SYSTEM SHALL `a_id<b_id`로 정규화된 행에 `weight=LEAST(1.0, weight+0.05)`, `co_activation_count++`, `last_activated_at=now`를 원자적으로 적용하고, 행이 없으면 `weight=0.05, link_type='co_recall'`로 생성한다. _(§5, §3.5)_
1.5 WHEN 동일 페어로 `ReinforceLinks`가 동시에 다중 호출되면 THE SYSTEM SHALL 행 단위 `ON CONFLICT`로 갱신을 직렬화하여 손실 없이 누적한다(weight 상한 1.0 유지). _(§3.8 함정 5)_
1.6 WHEN 페이지를 새로고침하면 THE SYSTEM SHALL 강화된 `weight`가 영속되어 `GetUniverse`(04)가 굵어진 시냅스를 반환한다. _(영속)_
1.7 IF `ReinforceLinks` 배치가 네트워크 오류로 실패하면 THEN THE SYSTEM SHALL 로컬 누적 세트를 비우지 않고 다음 윈도에서 재시도한다(강화 이벤트 유실 방지).
1.8 IF `RecallMemory`의 `memory_id`가 존재하지 않으면 THEN THE SYSTEM SHALL `NotFound`를 반환하고 별을 생성·수정하지 않는다.
1.9 WHEN `features/recall/model`이 빌드되면 THE SYSTEM SHALL `three`/React/DOM을 import하지 않는다(순수·모바일 재사용). _(원칙 4)_

## 완료 정의 (DoD)
- 별 클릭 시 **불변 원본 일기**가 읽기 전용으로 뜬다(편집 경로 없음). `RecallMemory` 후 `last_recalled_at`가 갱신되되 `body`·`entry_date`·`mood`는 불변(원칙 1).
- 두 별을 함께/오가며 회상→디바운스 후 **단일 unary `ReinforceLinks` 배치** 1회 호출(네트워크 탭에서 streaming 없음 — 원칙 6 확인), 사이 `weight`가 `+0.05`(상한 1.0) 증가·`co_activation_count++`.
- **새로고침 후에도 강화가 유지**된다: `GetUniverse`(04) 응답의 해당 엣지 `weight`가 증가해 있다(영속).
- 동시 호출 원자성: 동일 페어로 `ReinforceLinks`를 동시에 2회 호출해도 weight가 누적·직렬화되고 1.0을 넘지 않는다.

- `pnpm --filter frontend build`·`pnpm lint` 통과. `features/recall/model`이 `three`/React/DOM을 import하지 않음(원칙 4, grep 확인).
- Docker/WSL: `docker run --rm -v ${PWD}/backend:/src -w /src golang:1.26 go build ./...` 성공(`.exe` 호스트 직접 실행 안 함).
