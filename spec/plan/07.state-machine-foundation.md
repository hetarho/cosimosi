# 07. state machine foundation

> State machine foundation - XState v5 conventions for control state, app mode, and render-loop reads.
> Scope: FE | Status: planning

## Purpose

cosimosi will have interaction-heavy flows: session boot, test panels, memory creation, recall, selection, camera
modes, and rendering lifecycle. Those flows need explicit control-state machines rather than scattered booleans. This
unit installs the XState v5 foundation and records how machines interact with server data, local stores, and R3F
render loops.

## Scope / Non-goals

**In scope**
- XState v5 and app-level conventions for machines in `model/<name>.machine.ts`.
- A small machine catalog for platform-level control flows: session, test harness panel lifecycle, and generic async
  command lifecycle.
- React hooks/providers for interpreting machines without leaking implementation details into feature UI.
- Test utilities for deterministic machine transitions.
- Rule/examples showing that machine context stores ids and control metadata, not large data collections.
- R3F pattern for reading machine snapshots inside `useFrame` via actor `getSnapshot()` without driving 60 fps through
  React state.

**Non-goals**
- No product feature machine such as Encode, Recall, camera navigation, or universe tour.
- No server state inside machine context.
- No global event bus.
- No animation timeline system or Figma motion implementation.
- No renderer, force simulation, or Three.js integration beyond the read pattern contract.

## Design

### Machine placement

Machines live where the control flow belongs:

- app-wide lifecycle machines: `apps/*/src/app/model`;
- feature action machines: `features/<verb>/model`;
- entity control machines: `entities/<noun>/model`;
- shared pure helpers only when multiple slices need the same generic pattern.

Machines export named factory functions and TypeScript types. UI components use hooks/adapters rather than importing
deep machine internals.

### Context rule

Machine context may contain:

- ids;
- current control mode;
- timestamps for control transitions;
- small error/status metadata;
- references to actor ids when needed.

Machine context must not contain:

- server data collections;
- QueryClient data;
- Zustand store snapshots;
- large graph buffers;
- `Float32Array` coordinate data;
- Supabase session objects or access tokens.

The data lives in Query, Zustand, or refs and is selected by id when the machine state needs it.

### Catalog

This unit creates reusable patterns, not product workflows:

- `session.machine.ts`: reconciles with [04.auth-session](04.auth-session.md) and becomes the reference for auth
  lifecycle.
- `async-command.machine.ts`: idle/submitting/succeeded/failed/cancelled for local commands.
- `panel.machine.ts`: closed/open/loading/ready/error for test harness panels.

The catalog is intentionally small; feature-specific states are authored by their feature plans.

### React and R3F usage

React components subscribe to actor state through the official XState React bindings. R3F `useFrame` reads an actor
snapshot directly through a stable actor reference:

- no `setState` per frame;
- no machine transition per frame for continuous values;
- no render-loop writes into Query cache.

This rule matters even before the renderer exists because later presentation slices will reuse it.

## Acceptance Criteria

1. XState v5 is installed and available to web and mobile code through the agreed app/package boundary.
2. Machine files follow the `model/<name>.machine.ts` convention and export named factories/types.
3. The platform catalog includes session, async-command, and panel lifecycle patterns or documented equivalents.
4. Unit tests cover deterministic transitions for each catalog machine.
5. Example/tests demonstrate that machine context stores ids/control metadata rather than large server or render data.
6. React binding utilities are in place for components to consume actors without deep imports.
7. A documented R3F pattern reads actor snapshots inside `useFrame` without causing React re-renders per frame.
8. No product feature workflow, renderer implementation, or server data cache is added by this unit.
