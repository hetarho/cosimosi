# 07. state machine foundation

> State machine foundation — XState v5 catalog, control-state placement, and the
> R3F per-frame read rule. As-built.
> Scope: FE | Status: built

## Purpose

cosimosi has interaction-heavy flows: session boot, test panels, memory creation,
recall, selection, camera modes, and rendering lifecycle. Those flows need
explicit control-state machines rather than scattered booleans. This unit
installs the XState v5 foundation and records how machines interact with server
data, local stores, and R3F render loops.

## Scope / Non-goals

**In scope**

- XState v5 and app-level conventions for machines in `model/<name>.machine.ts`.
- A small machine catalog for platform-level control flows: session, test harness
  panel lifecycle, and generic async command lifecycle.
- React hooks/providers for interpreting machines without leaking implementation
  details into feature UI.
- Test utilities for deterministic machine transitions.
- Rule/examples showing that machine context stores ids and control metadata,
  not large data collections.
- R3F pattern for reading machine snapshots inside `useFrame` via actor
  `getSnapshot()` without driving 60 fps through React state.

**Non-goals**

- No product feature machine such as Encode, Recall, camera navigation, or
  universe tour.
- No server state inside machine context.
- No global event bus.
- No animation timeline system or Figma motion implementation.
- No renderer, force simulation, or Three.js integration beyond the read pattern
  contract.

## Design

### Packages and the React seam

| Concern | Location | Depends on |
|---|---|---|
| Catalog machines (`asyncCommandMachine`, `panelMachine`) + types | `packages/state-machine` | `xstate` only |
| Auth/session machine (auth-domain lifecycle reference) | `packages/auth` (from plan/04) | `xstate`, `@supabase/supabase-js`, `@cosimosi/api-client` |
| React binding hooks (`createActorContext`, `useActorRef`, `useSelector`, `useMachine`, `shallowEqual`) | `@cosimosi/state-machine/react`; apps re-export it from `shared/model/xstate-react.ts` | `@xstate/react`, `react` |
| Per-feature machines (when they arrive) | `apps/{web,mobile}/src/<slice>/model/<name>.machine.ts` | imported from `@cosimosi/state-machine` or feature-local |

Both apps depend on `@cosimosi/state-machine` and `@xstate/react`. The package
root stays React-free so the same catalog works in tests, Web Workers, and R3F
frame-loops. The optional `/react` export owns the shared binding seam once, and
each app's `shared/model` re-exports it so feature call sites stay stable.

### Machine placement

Machines live where the control flow belongs:

- app-wide lifecycle machines: `apps/{web,mobile}/src/app/model`;
- feature action machines: `features/<verb>/model`;
- entity control machines: `entities/<noun>/model`;
- shared pure helpers only when multiple slices need the same generic pattern
  (the catalog — `packages/state-machine`).

Files are named `<name>.machine.ts` and export named factory functions and/or
the machine constant plus its types (`<Name>Event`, `<Name>Snapshot`, …). No
default exports; no wildcard barrels. Deep machine internals are not part of the
public surface.

### Context rule

Machine context may contain:

- ids (`userId`, `commandId`, `panelId`, `resultId`, …);
- current control mode/status;
- timestamps for control transitions or diagnostics;
- small error/status metadata;
- a monotonic epoch/attempt counter for staleness guards.

Machine context must not contain:

- server data collections;
- QueryClient data;
- Zustand store snapshots;
- large graph buffers;
- `Float32Array` coordinate data;
- Supabase session objects or access tokens;
- functions, callbacks, or React refs.

The data lives in Query / Zustand / refs and is selected by id when the machine
needs it. The catalog tests in
`packages/state-machine/src/context-rule.test.ts` hold this contract for every
catalog machine: snapshots must be JSON-serializable and must expose only the
documented control fields.

### Catalog

Three platform-level patterns ship from this unit. Product workflows are
authored by their feature plans.

- `sessionMachine` (from `@cosimosi/auth`, plan/04) — reconciles with the auth
  adapter; the auth lifecycle reference. States: bootstrapping · signedOut ·
  signingIn · authenticated · refreshing · expired · failed.
- `asyncCommandMachine` (`packages/state-machine`) — generic local-command
  lifecycle. States: idle · submitting · succeeded · failed · cancelled. Keeps
  a monotonic `attempt` counter; `RESOLVE`/`REJECT` echo the `attempt` they
  observed so a stale resolution from a superseded SUBMIT is guarded out at the
  machine (not just by caller discipline). The counter never resets.
- `panelMachine` (`packages/state-machine`) — overlay / test-harness panel
  lifecycle. States: closed · open · loading · ready · error. `OPEN` carries
  `openedAt` (caller-supplied) so the machine never calls `Date.now()` and stays
  a pure `(state, event) → state`.

### React and R3F usage

React components subscribe to actor state through the official XState React
bindings (re-exported via `shared/model`). R3F `useFrame` reads an actor
snapshot directly through a stable actor reference:

- `useActorRef(machine)` → stable ref; no rerender per snapshot change;
- inside `useFrame`, call `actor.getSnapshot()` synchronously — no React state;
- continuous values are never in machine context; the machine only gates
  *discrete* modes (frozen, dragging, focused, …);
- transitions are sent from React handlers or the worker boundary, never from
  inside `useFrame`.

This rule matters even before the renderer exists because Phase 4 presentation
slices will reuse it. Full pattern with pseudocode:
[tech/state-machine.md](../tech/state-machine.md) §5.2.

## Acceptance Criteria

1. ✅ XState v5 is installed and available to web and mobile code through the
   agreed app/package boundary (`@cosimosi/state-machine` + `@xstate/react` in
   both apps).
2. ✅ Machine files follow the `model/<name>.machine.ts` convention and export
   named factories/types (catalog: `async-command.machine.ts`, `panel.machine.ts`;
   session reference: `packages/auth/src/session-machine.ts`).
3. ✅ The platform catalog includes session, async-command, and panel lifecycle
   patterns.
4. ✅ Unit tests cover deterministic transitions for each catalog machine
   (`<name>.machine.test.ts`).
5. ✅ Example/tests demonstrate that machine context stores ids/control metadata
   rather than large server or render data (`context-rule.test.ts`).
6. ✅ React binding utilities are in place (`apps/{web,mobile}/src/shared/model/`)
   for components to consume actors without deep imports.
7. ✅ A documented R3F pattern reads actor snapshots inside `useFrame` without
   causing React re-renders per frame (tech/state-machine.md §5.2).
8. ✅ No product feature workflow, renderer implementation, or server data cache
   is added by this unit.
