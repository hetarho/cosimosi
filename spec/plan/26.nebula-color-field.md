# 26. nebula color field

> Nebula color field — the ambient `entities/nebula` layer: a TSL color field that blends **many** per-star emotion
> colors at once — each star's local region reads its own mood color densely, **stronger stars bleed wider** (weighted
> by `EffectiveStrength`), and the universe's global tone **emerges** from those local colors, never a single average.
> Plus the honest **"mirror of the emotions you re-read"** notice, shipped as i18n copy — not hardcoded.
> Scope: FE | Status: planning
>
> Layer P (presentation). A **visual (rendering) entity** that projects the domain `Emotion` (via the plan-17 palette
> seam) to an ambient color field, rendered inside `widgets/universe-canvas` behind the stars/neurons and latent field.
> It reads emotion and **only** color; it never models the global tone and never touches position/connection.

## Purpose

The product's honest promise is that **the universe's color is the emotions you re-read — an honest mirror, not the
average of your feelings** ([PRD.md](../PRD.md) §1.4). The color layer is where that promise becomes visible: not one
averaged wash, but **many mood colors coexisting and bleeding**, in proportion to how strongly each memory pulls
([M4]). Because re-reading a memory grows its strength (recall accumulation, Epic C [R3]), a color you revisit spreads;
a color you never open recedes — so the field mirrors *what you often revisit*, which is a different, more honest thing
than "your emotional average" ([M5]).

Every part of that meaning is fragile if it is smeared across the codebase: color must come from **one** place
(emotion), strength-weighting must be a *visualization output* that never bends the graph, and the "mirror" claim must
be *told to the user* so the field does not read as a lie ("but I felt sad this month — why is my universe warm?"). This
unit builds the ambient color field as a single rendering entity that consumes the plan-17 palette seam, weights the
blend by each star's strength as an **output-only** step, keeps the global tone **emergent** (like positions emerge from
synapses — rendered, never modeled, [§3.4]), and ships the honest notice through i18n. It coexists with plan 14's
skinnable *background* without letting that presentation-ambiance layer ever read emotion.

## Scope / Non-goals

**In scope**
- **`entities/nebula` — a screen-space / volumetric color field (TSL, WebGPU with WebGL2 fallback)** that blends the
  per-star emotion colors of the currently rendered memories. Not a single average color: **many colors coexist and
  bleed**, each star's local region saturated with *its own* mood color, colors mixing where regions overlap ([M4],
  [V4]). Rendered as the ambient layer of the universe.
- **Strength-weighted bleed** ([M4]). Each memory contributes its mood color with a **bleed radius / falloff weighted by
  its `EffectiveStrength`**: a star's local region reads its own color densely, and **stronger stars bleed wider** into
  the surrounding field. `EffectiveStrength` is the read-time-derived size (base strength + recall accumulation),
  computed by the shared pure fns (`packages/memory-logic`, golden-parity) — the nebula *reads* it, it does not define
  it.
- **Emergent global tone.** The universe's overall color is **not modeled or stored** — it is what the eye reads off the
  sum of local, strength-weighted contributions. Like positions emerge from synapses ([I5]), the global tone emerges
  from the local colors and stays **rendered, never a domain value** ([M4], [§3.4]).
- **Color from the single palette entry-point only** ([M3], [M6], plan 17). Every color the nebula draws comes from
  `moodColor(memory.mood)` via the plan-17 palette seam — the nebula computes **no** color of its own. Emotion drives
  **only** color, never position or connection ([I3], [M7]).
- **The honest notice** ([M5]). Short UI copy — *"the universe's color is the emotions you re-read, not the average of
  your feelings"* — surfaced near the field (e.g. an info affordance on the universe HUD / a first-encounter hint;
  exact placement is UI). It is **i18n message content** via `@cosimosi/i18n` (Paraglide `m.*`, `packages/i18n/messages`)
  — **never a hardcoded string** in a component.
- **The strength-weighting seam, built now.** In Epic A there is no recall yet, so a star's `EffectiveStrength` is its
  arousal-set `base_strength` ([E8a], plan 17) — the field still blends by that base strength today. The **weighting
  input** is `EffectiveStrength` (not `base_strength`), so when recall accumulation lands (Epic C [R3]) the mirror
  behavior emerges with **no rewiring** of this unit.
- **Presentation-only guard.** The nebula is a *projection* (§3.4), one-way: it reads domain emotion + derived strength
  and outputs pixels. It writes **nothing** back — no color it draws re-enters the domain-mirror slices or the API
  mapper, and it never feeds position, connection, or strength.
- **Mobile MVP fidelity.** Mobile may render a **reduced-fidelity** field — fewer sampled contributors and/or a
  lower-resolution field / cheaper falloff — sharing the same TSL source and the same meaning (web = open beta,
  mobile = MVP).
- **`/test`-hostable demo** — the field driven by placeholder memories with assorted moods + strengths (via
  [12.test-harness](12.test-harness.md)), exercising the WebGL2 fallback, so the blend/bleed/emergence can be verified
  without live domain data.

**Non-goals**
- **No emotion model, no palette definition.** The `Emotion` VO, the 13-mood table, and the single swappable
  `mood → color` palette are [17.emotion-model](17.emotion-model.md); the nebula only *consumes* `moodColor` through
  that seam and never defines a color.
- **No `EffectiveStrength` math.** Base strength ([E8a]) is owned by [17.emotion-model](17.emotion-model.md); the
  read-time **memory-level `EffectiveStrength`** (and `EffectiveBrightness`) are the shared `packages/memory-logic` pure
  fns owned by [18.synapse-plasticity](18.synapse-plasticity.md) (their full recall/forgetting dynamics land in Epic
  C/D). The nebula **reads** the value; it does not compute or store it.
- **No star / neuron / synapse bodies.** Coloring the star body itself ([V4]) and the latent gray field are
  [24.star-neuron-rendering](24.star-neuron-rendering.md) and [25.latent-star-field](25.latent-star-field.md); the
  nebula is the *ambient field around and between* them.
- **No canvas host, camera, or force-sim.** The full-screen scene, camera/navigation rig, and coordinate feed are
  [23.universe-canvas](23.universe-canvas.md); the nebula mounts as one layer inside it and reads coordinates it is given.
- **No skinnable background.** The plan-14 skin *background/environment* (non-domain ambiance) coexists with the nebula
  but is a separate layer; this unit does not build or modify the skin. It only defines the domain-color field that sits
  *in front of / alongside* the skin.
- **No palette-editing / customization UI.** Swapping the palette ([P1], Epic I) happens through the plan-17 seam; this
  unit is palette-agnostic and builds no editor.
- **No recall/strength dynamics.** Recall growing a star's strength ([R3]) is Epic C; this unit builds the weighting
  *seam* that consumes the resulting `EffectiveStrength`, not the dynamics.
- No hardcoded tuning scalars in FE code (bleed/field scalars live in `values.yaml`); no hardcoded copy (the honest
  notice is i18n).

## Design

### What the nebula is (and is not)

`entities/nebula` is a **visual (rendering) entity** (§3.1/§3.4): it imports the domain-mirror `entities/episodic-memory`
(via `@x`) and the plan-17 palette seam, and projects them to an **ambient color field** rendered as one layer of the
universe scene. It is the emotional *atmosphere* of the universe — the "성운" of the ubiquitous language (PRD §3.4), which
has **no type in any code layer** because it is derived/rendered, not a domain concept. The nebula is **domain color**
(it reads emotion); it is deliberately distinct from plan 14's **skin background**, which is presentation ambiance and
must never read emotion (see *Layer coexistence* below).

### Color source — the single palette entry-point ([M3], [M6])

- Every color the field draws is `moodColor(memory.mood)` obtained through the **plan-17 palette seam**
  (`resolvePalette()` / `usePalette()`-style, mirroring the plan-14 skin seam). The nebula computes **no** color of its
  own — no per-mood color literal, no valence→hue math, no blend of "raw emotion axes" into color. This makes
  "color always comes from emotion" a **structural** fact, not a convention: there is exactly one code path to a color.
- The nebula is **palette-agnostic**. When the Epic I customization unit ([P1]) swaps the palette through the plan-17
  seam, the whole field re-colors with **no edit** to this unit. What a swap **cannot** do ([I11], [P2]): decouple color
  from emotion, override a memory's stored mood, or feed position/connection/strength — the seam consumes a `Mood` and
  returns a `Color`, nothing more.
- **Emotion drives only color** ([I3], [M7]). The nebula reads `memory.mood` (→ color) and `memory.EffectiveStrength`
  (→ bleed weight, an already-derived size); it reads **no** valence/arousal into anything that isn't color, and it
  emits **nothing** into layout or synapses.

### The blend — many colors, not one average ([M4])

- The field is a **spatial mix of per-star contributions**, not a global average. Each rendered memory contributes its
  `moodColor` centered at its universe position (the coordinate the canvas feeds this layer, from `packages/force-sim`
  — the nebula never computes position). Contributions are **summed/composited in the field**, so where two memories'
  regions overlap their colors **coexist and bleed** rather than collapsing to a mean.
- **A star's local region reads its own color densely**: the contribution is strongest at the star's position and falls
  off with distance (a falloff kernel — a formula, in code). Away from any star the field is faint; near a star it is
  saturated with that star's mood color.
- **Emergent global tone** ([M4], [I5], [§3.4]). There is no "universe color" value anywhere — not stored, not modeled,
  not computed as a domain fact. The overall tone is simply what the composited local contributions look like. This is
  the exact analogue of positions: as positions emerge from synapses, the global tone emerges from the local mood
  colors, and stays **rendered**. If a plan or store ever needs a single "average universe color", that is a violation
  of this unit's intent (left open only for a possible *diagnostic readout*, see root DECISIONS.md — and even then it is
  a read-only derivation, never a stored/authoritative value).

### Strength-weighting — stronger stars bleed wider (output-only) ([M4], [M7])

- Each contribution's **bleed radius / falloff extent is weighted by the memory's `EffectiveStrength`**: a stronger star
  spreads its color **wider** into the surrounding field; a weak star stays a tight local glow. (Optionally the weight
  also scales the contribution's *intensity*; the exact coupling of radius vs. amplitude is a formula in code, the
  scalar coefficients in `values.yaml`.) A bleed-radius coefficient maps `EffectiveStrength` → radius.
- **`EffectiveStrength` is read-time-derived, read here, not defined here.** It is `base_strength` ([E8a], arousal-set at
  encode) plus recall accumulation ([R3]), computed by the shared golden-parity fns (`packages/memory-logic` ↔ Go). The
  nebula consumes the number the canvas / mirror provides.
- **This weighting is a visualization-output step only** ([M7]). It affects **pixels**, not the graph: it never feeds the
  force-sim, never changes clustering, never touches synapse strength or position. Clustering is driven entirely by
  shared neurons + synapses ([L2]/[L4], [I3]/[I4]); the color field is downstream of everything and outputs nothing back
  into it. This is *why* the strength-weighting is compatible with [I3] — it is a rendering weight, not an emotion-driven
  layout force.

### The honest mirror — "the emotions you re-read" + the notice ([M5])

- **The mechanism is the message.** Because recall grows a star's strength ([R3]) and strength widens its bleed, colors
  you **re-read** spread through the field over time, and colors you never open stay small. So the field mirrors *"the
  emotions you often revisit"*, **not** *"the average of what you felt"* — an honest, and deliberately different, thing
  ([PRD §1.4], [M5]).
- **In Epic A** there is no recall yet, so `EffectiveStrength` = `base_strength` and the field blends by that arousal-set
  base ([E8a]). The **mirror behavior is not yet observable**, but the **weighting seam** (input = `EffectiveStrength`,
  not `base_strength`) is built now, so Epic C's recall accumulation activates the mirror with no rewiring.
- **The honest notice.** To prevent the confusion the mirror invites ("why is my universe warm when I felt down?"), a
  short piece of copy explains the definition near the field — e.g. an info affordance / first-encounter hint on the
  universe HUD (exact placement is UI, left to [23.universe-canvas](23.universe-canvas.md) / the HUD). Constraints:
  - **i18n, not hardcoded.** The copy is a **message** in `@cosimosi/i18n` (Paraglide `m.*`, source in
    `packages/i18n/messages/{en,ko}.json`), consumed by the component — **no literal string in a slice** ([§3.1],
    ARCHITECTURE §5: UI copy is i18n'd). Both `en` and `ko` messages ship.
  - **Voice.** Literary and restrained, per the project copy voice; the message *names the honest definition* (color =
    emotions you re-read, not your average) without over-explaining. The exact wording is copy content owned by the
    message catalogue, reviewed on implement (left open — see root DECISIONS.md — for final wording).

### Layer coexistence — nebula (domain) vs. skin background (ambiance) ([I3], [I11])

The universe scene stacks two presentation layers that must not be confused:

| Layer | Owner | Reads emotion? | Role |
|---|---|---|---|
| **Skin background/environment** | [14.rendering-foundation](14.rendering-foundation.md) | **No** — forbidden ([I3]/[I11]) | Non-domain *ambiance*: background/starfield/fog/exposure/camera mood, a swappable preset. Pure 연출. |
| **Nebula color field** | this unit | **Yes** — color only | Domain-driven **emotion color** field: the strength-weighted mood blend. |

- Both are **presentation-layer** and both mount inside `widgets/universe-canvas`, but they are **distinct layers with
  distinct rules**: the skin is *ambiance that must not read emotion*; the nebula is *emotion color that must not model
  the global tone or touch layout*. The nebula composites **in front of / over** the skin background (it is the emotional
  atmosphere within the ambient environment); exact compositing order and blend mode are a render detail (left open —
  see root DECISIONS.md — default: nebula additively/screen-composited over the skin background, behind the star bodies).
- The invariant guard is symmetric: the **skin cannot read emotion** ([I3]/[I11], plan 14's guard), and the **nebula
  cannot set ambiance or override emotion** — it only *reads* mood (→ color via the palette) and derived strength
  (→ bleed). Neither layer writes to the domain.

### Render approach (TSL, WebGPU→WebGL2, per-frame)

- **TSL, one source.** The field's shader graph is authored in **TSL** (the plan-14 `@cosimosi/3d-renderer` toolkit) so
  it compiles to WGSL (WebGPU) and GLSL (WebGL2) from one source — no raw-GLSL `onBeforeCompile` fork. The shader graph
  is **code** (like a skin's node graph, plan 14); only *scalars* are `values.yaml`.
- **Contributor inputs as uniforms/buffers.** The rendered memories' `(position, moodColor, EffectiveStrength)` are fed
  to the field as a compact per-contributor buffer (e.g. a data texture / uniform array), updated as the graph changes.
  Per-frame reads follow §3.3 — coordinates come from the force-sim worker / the canvas's coordinate buffers, read in
  `useFrame`, **never** from React state (no 60fps re-renders).
- **Screen-space or volumetric.** The field may be realized as a screen-space color layer (a full-screen TSL pass reading
  the contributor buffer) or a coarse volumetric field; the choice is a render-fidelity decision (left open — see root
  DECISIONS.md — default: screen-space on web for open-beta fidelity, reduced-sample screen-space on mobile). Either way
  the *meaning* (many colors, strength-weighted bleed, emergent tone) is identical and the TSL source is shared.
- **Mobile MVP.** Mobile reduces fidelity — a **sample/contributor budget** and/or a lower-resolution field / cheaper
  falloff — from the same TSL source (§3.5: web = open beta, mobile = MVP). The budget scalars live in `values.yaml`.
- **Data flow.**
  ```
  [read]    GetUniverse (plan 23) → EpisodicMemory facts (mood, base_strength, recall_count, …)
            packages/memory-logic → EffectiveStrength(memory)         (base + recall accumulation; golden-parity)
            packages/force-sim (worker) → position(memory)            (coordinate the canvas feeds this layer)
  [project] moodColor(memory.mood)  → color                           (plan-17 palette seam, the ONLY color source)
  [blend]   per contributor: (position, color, EffectiveStrength) → local contribution
            composite all contributions → the color field             (many colors, strength-weighted bleed)
  [emerge]  global tone = what the composite looks like               (rendered, never stored/modeled — [I5]/§3.4)
  [notice]  m.* honest-mirror copy (i18n) surfaced near the field     ([M5])
  ```

## Acceptance Criteria

1. `entities/nebula` renders an ambient color field (TSL, WebGPU with automatic WebGL2 fallback) that **blends many
   per-star emotion colors at once** — colors coexist and bleed rather than collapsing to a single average color ([M4],
   [V4]).
2. Each memory's local region reads **its own** mood color densely (strongest at the star, falling off with distance);
   away from any star the field is faint ([M4]).
3. A memory's contribution **bleed radius / falloff extent is weighted by its `EffectiveStrength`** — a stronger star
   bleeds **wider** into the field; the mapping uses a `values.yaml` bleed-radius coefficient ([M4]).
4. Every color the nebula draws comes **solely** from `moodColor(memory.mood)` via the single plan-17 palette seam; the
   nebula defines no color of its own, and there is no code path coloring the field from anything other than emotion
   ([M3], [M6]).
5. The nebula reads emotion for **color only**; it reads no emotion axis into layout/connection and emits nothing into
   position, synapses, or strength ([I3], [M7]).
6. The strength-weighting is a **visualization-output step**: it changes only rendered pixels and **never** affects the
   force-sim, clustering, synapse strength, or position ([M7]); disabling/altering it changes the picture, not the graph.
7. The universe's global tone is **emergent** — there is no stored/modeled "universe color" value anywhere; the overall
   tone is only what the composited local contributions render as (mirroring how positions emerge from synapses) ([M4],
   [I5], §3.4).
8. The weighting input is **`EffectiveStrength`** (base strength + recall accumulation), not raw `base_strength`, so the
   "emotions you re-read" mirror ([M5]) activates once recall accumulation (Epic C [R3]) lands **without rewiring** this
   unit; in Epic A the field blends by the arousal-set `base_strength` ([E8a]).
9. An **honest notice** is shipped explaining that the universe color is *the emotions you re-read, not your emotional
   average* ([M5], [PRD §1.4]); its copy is an `@cosimosi/i18n` message (`m.*`, `packages/i18n/messages`, `en` + `ko`) —
   **not** a hardcoded string in any component ([§3.1], ARCHITECTURE §5).
10. Swapping the plan-17 palette re-colors the whole field with **no edit** to this unit, and a swapped palette cannot
    decouple color from emotion, override a memory's stored mood, or feed position/connection/strength ([I11], [P2]).
11. The nebula (domain **emotion** color) coexists with the plan-14 skin **background** (non-domain ambiance) as a
    separate layer inside `widgets/universe-canvas`; the **skin never reads emotion** and the nebula never sets ambiance
    or overrides emotion — neither layer writes to the domain ([I3], [I11]).
12. Mobile renders a **reduced-fidelity** field (fewer sampled contributors and/or lower-resolution field / cheaper
    falloff) from the **same** TSL source, preserving the meaning (many colors, strength-weighted bleed, emergent tone)
    (web = open beta · mobile = MVP).
13. The field's shader graph is authored in **TSL** (no raw-GLSL fork) and mounts on the `@cosimosi/3d-renderer`
    substrate; per-frame contributor reads use `getSnapshot()`/coordinate buffers in `useFrame`, never React state
    (§3.3).
