# 12. test harness

> Test harness - temporary `/test` route for headless verification panels and later golden-parity probes.
> Scope: FE | Status: planning

## Purpose

Phases 2 and 3 are intentionally headless: they build domain logic and use-cases before real presentation. The team
still needs a way to exercise those units in the browser. This unit creates a temporary `/test` route in `apps/web`
where later plans can register verification panels, call RPCs, inspect cache state, run pure functions, and show
structured results without pretending to be final product UI.

## Scope / Non-goals

**In scope**
- A `/test` route in `apps/web` that is clearly marked as temporary and excluded from product navigation.
- A panel registry where later units can add verification panels without editing the route shell each time.
- App providers for auth, transport, QueryClient, i18n, design-system shell, and state-machine actors as they become
  available.
- Initial platform panels: transport ping, auth/session status, query cache probe, values display, i18n switch, and
  design-system primitive gallery where dependencies exist.
- Test helpers for fake transport/auth adapters and deterministic QueryClient state.
- A convention for headless domain/use-case units to expose browser-verifiable fixtures and golden-parity results.

**Non-goals**
- No production navigation entry, marketing page, onboarding, universe renderer, or final UI flow.
- No Playwright visual regression suite unless a later presentation unit owns it.
- No domain feature implementation.
- No server-streaming monitor, realtime dashboard, or admin console.
- No mobile test harness route; mobile can get its own dev surface later if needed.

## Design

### Route shell

`/test` is a development verification surface. It may exist in production builds if harmless, but it must not be
linked from product navigation and must be easy to disable by environment if deployment later requires that.

The route shell owns:

- provider setup;
- panel list/sidebar;
- selected panel state;
- common result formatting;
- error boundary;
- copy through i18n messages.

It does not own product domain logic.

### Panel registry

Panels register through a small typed registry:

- `id`
- `title` message key
- `description` message key when helpful
- `requiredCapabilities`
- `render`

Later domain/use-case units can add a panel next to their feature/domain code and export it through a registry entry.
The route shell imports a composed registry, not many feature internals.

### Initial panels

The first implementation should add only panels whose dependencies exist:

- Transport: call `PlatformService.Ping` and show response/request id.
- Auth: show session machine state and user id when authenticated.
- Query: inspect selected query keys and demonstrate invalidation with fake data.
- Values: show generated values metadata and selected Phase 1 entries.
- i18n: switch locale and verify messages update.
- Design system: render primitive states for quick manual inspection.

If a dependency has not landed yet, the panel is omitted or shows an explicit unavailable state without failing the
route.

### Headless unit convention

Later Phase 2-3 units expose verification in two forms:

- automated tests in their own package/app;
- optional `/test` panel for manual browser inspection.

Panels can call RPCs, run pure fixtures, compare FE/BE golden outputs, and show JSON/table results. They must not
become final product UI and must not encode product navigation.

## Acceptance Criteria

1. `apps/web` has a `/test` route that renders inside the app provider stack and is excluded from product navigation.
2. A typed panel registry supports adding future verification panels without rewriting the route shell.
3. The route can render available Phase 1 panels and gracefully omit or mark unavailable panels whose dependencies are
   not implemented yet.
4. Transport, QueryClient, auth, values, i18n, and design-system seams can each be inspected through a panel once their
   units are implemented.
5. Fake auth/transport/cache helpers allow panels and tests to run without production secrets.
6. Panel copy goes through the i18n seam and panel UI uses shared design-system primitives when available.
7. `/test` introduces no product domain behavior, final presentation slice, universe renderer, or production nav item.
8. The convention for later headless units explains how to add automated tests plus optional browser panels.
