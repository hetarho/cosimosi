# 13. mobile app shell

> Mobile app shell - React Native navigation skeleton and provider composition for the Phase 1 frontend seams.
> Scope: FE | Status: built

## Purpose

The foundation created a runnable React Native hello-world app, and Phase 1 now makes the shared frontend seams real:
session, transport/cache, state machines, i18n, and design-system primitives. This unit composes those seams into the
mobile app root so `apps/mobile` is a first-class peer of `apps/web` from the start. It proves the mobile baseline
without building mobile product screens or the universe renderer.

## Scope / Non-goals

**In scope**

- Replace the hello-world `App.tsx` with an app-shell composition root.
- Add a React Native navigation skeleton with stable route names and typed route params.
- Compose Phase 1 providers in the mobile root: session/auth, transport, QueryClient, state-machine actors, i18n,
  theme/design tokens, error boundary, observability facade, and safe-area/status-bar handling when available.
- Add a minimal shell home/diagnostics surface that confirms providers are mounted and can show platform status.
- Add mobile-specific adapters where the shared seams require native behavior: token storage, device locale,
  navigation primitives, status bar/safe area, and link handling.
- Keep shared model/api/config/i18n/state code in `packages/` or platform-pure modules so web and mobile consume the
  same contracts.
- Add mobile host checks to the quality gate if they are not already covered: TypeScript, lint, and a smoke test that
  renders the app shell with fake adapters.

**Non-goals**

- No mobile product feature UI, onboarding flow, diary/memory screens, or universe navigation.
- No RN WebGPU, three.js canvas host, force simulation, shaders, or 3D renderer.
- No mobile `/test` route equivalent. The Phase 1 browser harness remains [12.test-harness](12.test-harness.md).
- No new domain model, RPC method, database table, or product use-case.
- No production app-store setup, push notifications, deep-link campaign flows, biometric auth, or offline sync.
- No separate mobile state, i18n, or data library that forks from the web architecture.

## Design

### Mobile app root

`apps/mobile/src/app/App.tsx` becomes a thin composition root:

```text
App
└── MobileAppProviders
    ├── ErrorBoundary
    ├── ObservabilityProvider
    ├── I18nProvider
    ├── ThemeProvider
    ├── SessionProvider
    ├── TransportProvider
    ├── QueryClientProvider
    ├── MachineActorsProvider
    └── NavigationRoot
```

Provider order is explicit and tested. Feature slices do not instantiate their own long-lived providers. Each provider
accepts fake/test adapters so the shell can render in host tests without Supabase, a real API, or native device
features.

### Navigation skeleton

The shell owns navigation infrastructure, not product IA. The initial route set is deliberately small:

- `Boot`: transient state while session/i18n/theme providers initialize.
- `ShellHome`: a neutral placeholder confirming the mobile shell is ready.
- `Diagnostics`: development-only provider status surface for auth/cache/i18n/theme/transport checks.

Route names and params are typed. The route registry lives under `apps/mobile/src/app/navigation` and is the only
place that knows the navigation library. Feature screens are added later by presentation plans and register through the
same typed boundary.

The implementation should verify current official React Native and navigation-library docs before adding dependencies
or native setup. The plan requires a conventional RN navigation container/stack; it does not pin package versions.

### Native adapters

Mobile-specific behavior stays at the app/shared boundary:

- **Token storage:** the auth facade receives a secure native token storage adapter; feature code never touches it.
- **Locale:** the i18n provider reads the device locale through an RN adapter and falls back to the default locale.
- **Theme:** the mobile app consumes the same token source as web through a DOM-free token map and NativeWind/RN
  styling primitives.
- **Transport:** the Connect client uses the same generated API-client facade and resolves the base URL through mobile
  runtime environment/dev configuration.
- **Safe area/status bar:** shell layout handles device chrome without leaking those concerns into feature slices.
- **Link handling:** the app root owns inbound link parsing only to the extent needed for auth callbacks or future
  typed routes; product deep links are later feature work.

### Diagnostics surface

The diagnostics surface is not a product screen. It may show:

- app version/build label;
- current locale;
- theme name;
- session state without secrets;
- API base URL and transport ping result when [03.rpc-transport](03.rpc-transport.md) exists;
- QueryClient status summary;
- feature flag defaults when [10.observability-and-flags](10.observability-and-flags.md) exists.

It must not show tokens, diary text, generated memory content, embeddings, or product data. It can be disabled for
production builds through runtime/dev configuration when deployment policy requires it.

### FSD and package boundaries

The mobile app mirrors the web FSD layout:

```text
apps/mobile/src/
├── app/
│   ├── App.tsx
│   ├── providers/
│   └── navigation/
├── pages/
├── widgets/
├── features/
├── entities/
└── shared/
    └── ui/
```

Shared model/api/config/i18n/state code lives in `packages/` or platform-pure modules. Platform-aware UI and native
adapters live in app/shared UI or app provider boundaries. A `*.native.tsx` sibling is allowed only where DOM and RN
actually differ.

### As-built

- **Navigation library:** React Navigation v7 — `@react-navigation/native` + `@react-navigation/native-stack`
  (peers `react-native-screens`, `react-native-safe-area-context`). The library is confined to
  `app/navigation/NavigationRoot.tsx` (the navigator + screen registration + typed deep-link config) and
  `app/navigation/routes.ts` (the typed `RootStackParamList`, `ROUTES`, and the `ReactNavigation.RootParamList`
  augmentation). Shell screens receive their `navigation` prop typed via `RootStackScreenProps<…>` and never import the
  library themselves.
- **Screens:** `Boot` and `ShellHome` under `app/navigation/screens/`; `Diagnostics` under `app/diagnostics/`. These
  are app-shell infrastructure, not product features, so they live under `app/` rather than `pages/` (which stays for
  later product IA). `Boot` drives the shell-lifecycle actor and `navigation.reset`s to `ShellHome`.
- **Provider order** (`app/providers/MobileAppProviders.tsx`, outer→inner): observability (outermost, so the error
  boundary reports to it) → error boundary → i18n → theme → session(auth) → transport+QueryClient (one provider) →
  machine actors → children (`NavigationRoot`). The native shell (`SafeAreaProvider` + `StatusBar`) wraps the stack in
  `App.tsx`.
- **State machine:** `app/model/app-shell.machine.ts` (booting → ready) is the first app-wide lifecycle actor, hosted by
  `MachineActorsProvider` via `createActorContext` (added to the `shared/model` xstate-react seam). Placement follows
  ARCHITECTURE §3.2 (app-wide lifecycle → `app/model`).
- **Native adapters** live in `shared/native/` (device locale, secure token storage seam + in-memory impl, inbound-link
  seam, safe-area metrics); runtime config (API base URL, app-version label, diagnostics flag key) in `shared/config/`.
  No native-secure-storage/keychain dependency is added — the seam takes a host-supplied store at deployment.
- **Diagnostics gating:** reachable only when the `platform.diagnosticsSurface` flag (plan/10) is on; default off keeps
  it out of production builds. The surface shows locale, theme, session status, API base URL, a live transport ping,
  query-cache size, and flag defaults — never tokens or product/private data.
- **i18n:** `MobileI18nProvider` negotiates the locale in a mount effect (no external-store writes during render, so the
  provider stays pure under concurrent rendering); product screens render only after the boot gate, so they paint in the
  right locale. New `mobile_shell_*` / `mobile_diagnostics_*` message keys in `packages/i18n/messages`.
- **Host test gate:** Jest with `@react-native/jest-preset` + `@testing-library/react-native` + `react-test-renderer`
  (no emulator). The shell smoke test (`app/App.test.tsx`) renders the full stack with `createMobileShellFakes`
  (`shared/testing/`). Wired into the repo gate as `pnpm test:mobile`; `lint`/`typecheck` already covered mobile.

## Acceptance Criteria

1. `apps/mobile/src/app/App.tsx` is a provider composition root rather than a hello-world component.
2. The mobile app has a typed navigation skeleton with `Boot`, `ShellHome`, and `Diagnostics` or equivalent
   non-product routes.
3. Session, transport, QueryClient, state-machine, i18n, theme/design-system, observability, error-boundary, and native
   shell providers are composed in a documented order.
4. The shell can render in host tests with fake auth, transport, storage, locale, and observability adapters.
5. Mobile consumes the same generated/shared API, values, i18n, state-machine, and token contracts as web; no parallel
   mobile-only architecture is introduced.
6. Native-only concerns such as secure token storage, device locale, safe area/status bar, and link handling stay at
   app/shared boundaries and do not leak into feature/domain slices.
7. The diagnostics surface exposes provider health without secrets or product/private memory data.
8. Quality gates include mobile TypeScript/lint and a shell smoke test that does not require an emulator.
9. No product feature screen, universe renderer, RN WebGPU integration, new RPC, DB schema, or domain model is added.
