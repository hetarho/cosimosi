# 30. 개인 설정 — 서버 per-user 시각 설정 (personalization)

> 지금 클라 localStorage에만 사는 시각 설정(우주 배경색·오브젝트 모양)을 **서버 계정 종속 per-user 설정**으로 올리고, **감정색**을 새 축으로 추가한다. "본인만의 우주" 토대. 지금은 고정 기본값을 서버가 내려주고, 온보딩 피커·"xx%가 골랐어요" 집계는 미래에 이 데이터 모델 위에 붙는다. | 범위: FS | 선행: 29, 16 | 차단: —

## 목적 (Why)
시각 설정이 두 가지 문제를 갖는다: (1) 배경색(테마)·오브젝트 모양은 `entities/appearance` zustand store의 **localStorage**에만 있어 기기-로컬이고 계정을 따라오지 않는다. (2) 감정색(`MOOD_PALETTE`)은 코드 고정이라 바꿀 수 없다. "일기를 쓰면 별이 되는 본인만의 우주"라는 제품 핵심에서, 감정의 색 언어와 우주의 분위기는 **사용자의 것**이어야 하고 계정을 따라 모든 기기에 와야 한다. 또 "이 감정엔 어떤 색이 떠오르나요? xx%는 이 색을 골랐어요" 같은 추천은 **여러 사용자의 선택을 서버에서 집계**해야 가능하다.

그래서 세 축(우주 배경색·감정색·오브젝트 모양)을 **서버 per-user 설정**으로 통일한다. 이 스펙은 그 **데이터 모델·RPC·서버화**까지다. **온보딩 피커 UI와 추천/집계는 비목표**(미래)이며, 지금은 고정 기본값(29의 추천 팔레트 + 기존 테마/오브젝트 기본)을 서버가 제공한다 — 즉 스키마·기본값·읽기/쓰기 경로를 깔되, per-user 덮어쓰기는 미래 온보딩이 채운다.

## 범위
- **포함:**
  - **DB**(신규 마이그레이션 `00002_user_settings.sql`): `user_settings`(user_id PK, 단일값 = 배경 테마·오브젝트) + `user_emotion_colors`(user_id, mood, color — per-emotion, "xx% 집계"를 위해 정규화 행).
  - **proto/RPC**: `Settings` 메시지(theme·star_object·repeated EmotionColor{mood,color}) + `GetSettings`/`UpdateSettings` RPC(`cosimosi.v1`).
  - **BE**: 별도 `SettingsService` + `internal/settings`(memory 패턴 미러) service/handler/repository + sqlc 쿼리. 서버는 **사용자 오버라이드만 저장·반환**한다 — 기본값은 클라가 소유하고(미인증·오프라인 경로에서도 필요하므로) 그 위에 머지한다. 팔레트를 BE/FE 이중 출처로 두지 않아 드리프트를 막는다.
  - **FE**: `entities/appearance` store를 localStorage → **서버 백킹**(Connect-Query)으로 전환(미인증·오프라인은 기본값 폴백, localStorage는 캐시/레거시 승계로 강등). `AppearanceSwitcher`가 `UpdateSettings` 호출(낙관적). 감정색은 서버 설정이 `MOOD_PALETTE` 기본을 덮어쓰는 형태로 렌더에 주입.
  - **기본값 상수**: 배경 테마 = `vast`, 오브젝트 = `deepfield`, 감정색 = 29 추천 팔레트(hex).
- **비목표(Out of scope):**
  - **온보딩 색 피커 UI**("이 감정엔 어떤 색?" 단계별 선택) → 미래 스펙. 30은 설정을 읽고/쓰는 경로와 기본값까지.
  - **"추천 3색 + xx%가 골랐어요" 집계·추천** → 미래. 30은 `user_emotion_colors` 정규화로 `GROUP BY (mood,color)` 집계가 가능하게 **자리만** 만든다.
  - **감정 13종 분류 자체** → **29**. 30은 그 13 mood 키에 색을 매핑.
  - **자유 배경색(임의 hex) 피커** → 미래. 지금 배경은 기존 테마 프리셋(vast/lively/calm) 키를 저장(테마의 bg+accent+swatch 번들 보존). 자유 hex는 이 모델의 확장으로 둔다.
  - ⚠️ **25(ambient-mood)와의 레이어 구분:** `theme`(여기, 개인 설정)는 우주의 *deep base 색*이고, 25의 요즘-감정 nebula 풀은 그 위에 번지는 *overlay*다 — 25는 base를 대체하지 않는다(충돌 아님, 별개 레이어).
  - **별 색 연속 보간 렌더** → 미래(29 비목표와 동일).

## 설계 요점

**두 테이블 — 단일값 vs per-emotion.** 배경·오브젝트는 유저당 1값, 감정색은 13값이므로 분리한다. 감정색을 정규화 행으로 두는 핵심 이유는 미래의 `SELECT mood, color, count(*) FROM user_emotion_colors GROUP BY mood, color` 집계("xx%가 골랐어요")다.
```sql
-- 00002_user_settings.sql
CREATE TABLE user_settings (
    user_id      TEXT PRIMARY KEY,
    theme        TEXT,                        -- 배경 테마 키(vast/lively/calm); NULL → 서버 기본
    star_object  TEXT,                        -- StarObject id(deepfield/aurora/liquid/ember); NULL → 기본
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE TABLE user_emotion_colors (
    user_id  TEXT NOT NULL,
    mood     TEXT NOT NULL,                   -- 29의 13 mood 문자열 중 하나
    color    TEXT NOT NULL,                   -- hex #RRGGBB
    PRIMARY KEY (user_id, mood)
);
```

**오버라이드만 저장 — 기본값은 클라가 머지.** 기본 팔레트/테마/오브젝트는 **클라가 소유**한다(미인증·오프라인 렌더에 어차피 필요 — AC 1.5). 서버는 사용자가 *바꾼 값만* 저장하고 `GetSettings`는 그 오버라이드만 돌려준다(행 없으면 빈 `Settings`); 클라가 자기 기본값 위에 머지한다. 이렇게 해야 팔레트가 BE/FE 이중 출처로 갈라져 드리프트하지 않는다(서버에 13색 기본 상수를 두지 않는다). per-user 행은 **사용자가 바꿀 때만** 생성된다(온보딩 미구현이라 지금은 비어 있음 → 전원 기본값 렌더). "지금 DB에 반영"이란 *스키마·읽기/쓰기 RPC·서버화 경로*를 확립하는 것이고, 덮어쓰기 값은 미래 온보딩이 `UpdateSettings`로 채운다. (사용자 행을 미리 시드하지 않는다 — `users`는 Supabase auth 소관.)

**proto.** `cosimosi.v1`:
```proto
message EmotionColor { Mood mood = 1; string color = 2; }   // color = "#RRGGBB"
message Settings {                  // 저장된 오버라이드(완전한 config 아님 — 클라가 기본값에 머지)
  string theme = 1;                 // "" = 클라 기본(vast)
  string star_object = 2;           // "" = 클라 기본(deepfield)
  repeated EmotionColor emotion_colors = 3;   // 사용자가 바꾼 mood만(0~13)
}
message UpdateSettingsRequest {     // 부분 갱신: 보낸 필드만 upsert(생략=보존)
  optional string theme = 1;
  optional string star_object = 2;
  repeated EmotionColor emotion_colors = 3;
}
// 별도 service SettingsService { GetSettings(NO_SIDE_EFFECTS) · UpdateSettings }
```
`UpdateSettings`는 부분 갱신(보낸 필드만 upsert) — 배경만, 오브젝트만, 감정색 일부만 바꿀 수 있게.

**FE — store 서버화.** `useAppearance`(현 localStorage zustand)를 Connect-Query `GetSettings`로 시드하고, `setTheme`/`setObject`/`setEmotionColor`가 `UpdateSettings`(낙관적)로 쓰게 한다. **미인증·오프라인·로딩 중**엔 기본값(현 `DEFAULT_THEME`/`DEFAULT_OBJECT` + `MOOD_PALETTE`)으로 렌더 — 한 번도 깨지지 않는다(헌법 견고성). localStorage는 *오프라인 캐시 + 레거시 1회 승계*로 강등(서버가 단일 진실). 감정색은 `moodRgb`가 서버 오버라이드를 우선 참조하도록 주입(없으면 `MOOD_PALETTE` 폴백).

**검증.** `color`는 `#RRGGBB` 정규식, `mood`는 29의 13 화이트리스트, `theme`/`star_object`는 알려진 id만(밖이면 InvalidArgument — 17의 검증 패턴). 색 형식만 보고 미적 판단은 안 한다.

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 인증 사용자가 `GetSettings`를 호출하고 per-user 오버라이드가 없으면 THE SYSTEM SHALL 빈 `Settings`(theme=""·star_object=""·emotion_colors 없음)를 반환하고, **클라가 자기 기본값(vast·deepfield·`MOOD_PALETTE`) 위에 머지**해 완전한 설정을 만든다. _(오버라이드만 저장)_
1.2 WHEN 사용자가 일부 설정만 `UpdateSettings`로 보내면 THE SYSTEM SHALL 해당 필드만 upsert하고 나머지는 보존하며, 이후 `GetSettings`가 (덮어쓴 값 ⊕ 기본값)을 반환한다. _(부분 갱신)_
1.3 WHEN `UpdateSettings`의 `color`가 `#RRGGBB`가 아니거나 `mood`가 13 화이트리스트 밖이거나 `theme`/`star_object`가 미지값이면 THE SYSTEM SHALL `InvalidArgument`로 거부한다(부분 적용 금지). _(검증, 17)_
1.4 WHEN 감정색이 `user_emotion_colors`에 저장되면 THE SYSTEM SHALL `(user_id, mood)` 유일·`GROUP BY (mood,color)` 집계 가능한 정규화 행으로 둔다(미래 "xx% 집계"). _(데이터 모델)_
1.5 WHEN FE가 미인증·오프라인·로딩 중이면 THE SYSTEM SHALL 기본 테마·오브젝트·`MOOD_PALETTE`로 렌더하고 깨지지 않는다(설정은 점진적 향상). _(견고성)_
1.6 WHEN 인증 사용자가 `AppearanceSwitcher`로 배경/오브젝트/감정색을 바꾸면 THE SYSTEM SHALL `UpdateSettings`로 영속하고 새로고침·타 기기에서 같은 설정이 로드된다(계정 종속). _(서버화)_
1.7 WHEN 사용자가 감정색을 바꾸면 THE SYSTEM SHALL 별 렌더가 그 오버라이드 색을 쓰고(없는 mood는 기본 팔레트), 29의 13 mood 전부를 덮어쓸 수 있다. _(감정색 소비)_
1.8 WHEN RPC가 미인증이면 THE SYSTEM SHALL `Unauthenticated`로 거부한다(02 인터셉터). _(인증)_

## 완료 정의 (DoD)
- `GetSettings`가 오버라이드만 반환하고(행 없으면 빈 Settings), 클라가 기본값(vast·deepfield·`MOOD_PALETTE`) 위에 머지해 완전한 설정을 만든다. `UpdateSettings` 부분 갱신이 보낸 필드만 upsert하고 나머지를 보존한다.
- 잘못된 hex·mood·테마/오브젝트 id는 `InvalidArgument`로 거부되고 부분 적용되지 않는다. 미인증은 `Unauthenticated`.
- 감정색이 `(user_id,mood)` 정규화 행으로 저장돼 `GROUP BY (mood,color)` 집계가 가능하다.
