# 42. 별 시각 프리미티브 (star-visual-primitive)

> 별의 시각 정체성(4형태의 geometry + TSL 셰이딩)을 *어떻게 소비하느냐와 분리된* 한 벌의 WebGPU 소비 가능 프리미티브로 통합한다. 어떤 씬이든(우주 InstancedMesh·랜딩·사인인·초대의 단일 메시) 위치·움직임은 자기가 주입하고 별 본체는 가져다 쓴다. | 범위: FE | 상태: 구현됨(job 04)

> **구현된 형태(as-built) — Part A만 적용:** 프리미티브 = `entities/star/ui/star-body.ts`의 `buildStarBody(object, inputs)`(순수, geometry+material) + `STAR_FORM_SPIN`(form별 mesh-spin 메타). 소비: 우주 `StarField`(attribute 바인딩)·`widgets/star3d/Star3D`(uniform 바인딩) — 둘 다 time/brightness uniform을 직접 소유. 옛 `forms.ts`·`single.ts`는 삭제. **알려진 정규화:** 자가발광 통일로 단일(showcase) 별이 우주(forms) 튜닝으로 수렴 — 특히 aurora가 투명→불투명이 되며, bloom이 없는 `StarCanvas`에선 halo가 불투명 몸체에 가려 글로우가 약해진다(아래 참조).
>
> **구현 보강(job 28):** 별·배경·시냅스 TSL node conversion은 `shared/lib/r3f/tsl.ts` helper(`asFloatNode`, `asVec3Node`, `uniformColorNode`, `attribute*Node`)로 표현한다. Three TSL 타입 오버로드와 런타임 DSL의 차이는 helper 내부 경계에만 남기며, 프리미티브의 시각 수치·소비 계약은 바꾸지 않는다.
>
> **Part B(랜딩 히어로 한-캔버스 합성)는 되돌림(revert).** 박스(320px) 한정 합성이라 fluid 배경이 별 둘레에 **동그란 패치**로 떠 페이지에 안 녹았고, `StarCanvas`에 bloom이 없어 별이 안 빛났다. 즉 한-캔버스 합성은 **페이지/섹션 전체 + bloom** 수준에서 해야 옳다는 게 확인됐다 → 후속 plan(랜딩·사인인 배경 합성 + bloom)으로 분리. `HeroScene.tsx`·`fluid-material.ts`는 제거, `HeroSection`은 `ThemedStar`로 복귀.

## 목적
"기억마다 하나뿐인 형태가 의미에서 창발한다"는 별의 시각 정체성은 지금 **두 벌로 중복**돼 있다. `entities/star/ui/forms.ts`(우주 InstancedMesh용 — 색을 per-instance attribute로, ambient-only 자가발광)와 `entities/star/ui/single.ts`(랜딩·사인인·초대의 단일 별용 — 색을 uniform으로, directional 조명 튜닝)가 같은 4형태(deepfield/aurora/liquid/ember)를 ~90% 같은 셰이더 로직으로 두 번 구현한다.

중복의 근본 원인은 **시각 정의가 "소비 방식"에 결합**돼 있기 때문이다 — 인스턴스(attribute) vs 단일(uniform), ambient vs directional 조명. 한 형태의 셰이딩을 손보면 두 파일을 똑같이 고쳐야 하고, 새 페이지에서 별을 쓰려면 컴포넌트(`<ThemedStar>`)를 통째로 가져오거나 셰이더를 또 베껴야 한다.

목표는 별 본체를 **입력 바인딩과 분리된 단일 시각 프리미티브**로 만들어, 소비처가 자기 씬·렌더러·카메라·위치·움직임을 그대로 소유한 채 "별 모양"만 WebGPU에 꽂아 쓰게 하는 것이다. 이는 장기적으로 별 프리미티브를 라이브러리로 떼낼 수 있는 토대이기도 하다(이번 범위는 앱 내부 모듈).

## 범위 / 비목표
이 plan은 **두 부분**으로 구성된다 — (A) 별 프리미티브 통합(토대), (B) 그 프리미티브로 랜딩 히어로를 한 캔버스에 재합성해 "한 씬에서 별+배경"의 *성능·깊이 레이어링 가능성을 검증*(feasibility).

**A. 별 프리미티브 통합 (토대)**
- `entities/star`에 **통합 별-바디 빌더**: form별 geometry + TSL 셰이딩 노드를 만든다. 셰이더 입력(mood색·밝기·seed·hueShift·time)을 **노드 접근자**로 받아, 인스턴스 소비처는 `attribute()`로·단일 소비처는 `uniform()`로 같은 셰이더를 바인딩한다.
- 조명 가정을 **자가발광(emissive)으로 통일** — emissive는 씬의 광원과 무관하므로 ambient-only(우주)든 directional(쇼케이스)이든 동일하게 보인다.
- `forms.ts` + `single.ts` 를 통합 빌더로 대체한다.
  - 우주 `StarField`는 **attribute 바인딩**으로 통합 빌더를 소비하고, 별 그림결과가 이전과 동일하다(출력 회귀 없음 보장).
  - 단일 별(`Star3D`)은 **uniform 바인딩**으로 통합 빌더를 소비한다.
- 사인인·초대·랜딩은 현재 모습을 유지한 채 통합 빌더를 (기존 합성 레이어 `ThemedStar`/`BrandMark`/`StarCanvas`/halo를 통해) 소비한다.

**B. 랜딩 히어로 한-캔버스 합성 (feasibility — 경계 좁힘)**
- 랜딩 **히어로 영역만** 대상으로, 별 프리미티브와 배경 레이어(들)를 **하나의 R3F 씬/캔버스**에 함께 렌더한다. 오늘 히어로는 별 캔버스(`StarCanvas`, WebGPU 1개)와 배경(`StarFieldCanvas` 등)이 분리돼 있는데, 이를 **WebGPU 컨텍스트 1개**로 합친다 → 질문 1(성능)의 실증.
- 씬 안에서 배경 레이어를 **별 뒤(renderOrder 낮게)와 별 앞(투명/additive·depthWrite off)으로 나눠** 배치해, "배경 일부가 별 앞으로·일부가 뒤로 흐르는" 깊이 레이어링을 보인다 → 질문 2의 실증. 우주 셸의 씬 정렬(AmbientNebula renderOrder −10 → 별 → 시냅스) 방식을 차용한다.
- 카메라는 **단순 고정/fit 프리셋**(FSM 없음) — 라이브러리화 시 "기본 카메라 프리셋" 경로의 첫 사례.
- 히어로의 룩은 현재 인상을 유지하되, 레이어링 실증을 위한 합성 변화는 허용한다.

**비목표**
- 카메라·배경·bloom·grain을 묶는 "우주 엔진" 컴포넌트는 만들지 않는다. 이 plan은 **별 프리미티브 + 랜딩 히어로 합성**까지만 다룬다(범용 엔진은 후속이며, 이 프리미티브 위에 얹는다).
- **위치·움직임은 프리미티브 밖**이다 — instance matrix, force-sim 좌표, 표상 부동(38), wobble, mesh spin은 전부 소비처가 계산·주입한다. 빌더는 만들지 않는다.
- **사인인·초대는 한-캔버스로 재합성하지 않는다** — 이번엔 프리미티브만 소비(현재 분리 캔버스·현재 모습 유지). 한-캔버스 합성 실증은 랜딩 히어로 한 곳으로 좁힌다.
- **랜딩 배경·스크롤 시스템 전체 재작성은 비목표** — `LandingBackground`/`Calm`/`Lively`/`Vast`와 섹션 스크롤 구동은 그대로 둔다. 합성은 **히어로 영역**에 한정한다. (구현 중 이 경계를 넘어 배경 시스템 전체를 건드려야 하면 → 별도 plan 43으로 분리한다.)
- SVG 2D 프리미티브(`VizStar`·`ui/svg/*`)는 별개 렌더 경로라 통합 대상이 아니다.
- npm 패키지로의 실제 추출·배포는 비목표다. 디커플링이 그것을 *가능하게* 하되, 이번엔 앱 내부 모듈로 둔다.
- 우주 셸의 카메라·레이아웃·렌더 파이프라인 재구축은 비목표다(StarField의 별-바디 셰이더만 교체, 출력 동일).
- 시각 리디자인이 아니다. 통합에 따른 미세 정규화(단일 별의 directional 하이라이트가 자가발광으로 수렴)는 허용하되, 의도적 룩 변경은 하지 않는다.

## 설계 요점

### 입력 바인딩 추상화 (핵심 계약)
통합 빌더는 셰이더가 필요로 하는 입력을 **TSL 노드로** 받는다. 소비처가 그 노드를 attribute로 만들지 uniform으로 만들지 결정한다.

```ts
// 별 셰이더의 입력 계약 — 전부 TSL 노드(소비처가 attribute() 또는 uniform()로 공급)
interface StarShadeInputs {
  mood: Node          // 의미색(linear RGB, vec3). 우주=aMood attribute, 단일=Color uniform
  brightness: Node    // float. 우주=aBrightness attribute, 단일=bright uniform
  seed: Node          // float. 우주=aSeed attribute, 단일=상수/uniform
  hueShift: Node      // float(rad), 기본 0. 재공고화 색조(23). 단일은 보통 0
  time: Node          // float. 공유 uTime — 소비처가 매 프레임 .value를 올린다
}

// object(형태) + 입력 → geometry + material. 순수: 노드 그래프만 만든다(uniform .value를 돌리지 않음).
function buildStarBody(object: StarObject, inputs: StarShadeInputs): {
  geometry: THREE.BufferGeometry
  material: THREE.Material
}
```

- **time uniform 소유는 소비처.** 현재 `forms.ts`/`single.ts`는 빌더가 `update(time)` 클로저를 반환해 .value를 돌렸다. 통합판은 time을 `inputs`로 받으므로, uniform 생성·`useFrame`에서의 .value 갱신은 **소비처**(StarField·Star3D)가 가진다. 빌더는 노드 그래프만 만드는 순수 함수가 된다 — "위치·움직임 외부 주입" 원칙과 일관(빌더는 시간조차 돌리지 않는다). frozen-time 관용구(BloomPass가 TSL `time` 노드를 멈추므로 수동 uTime)는 그대로 유지하되, 그 uTime을 소비처가 소유한다.
- **인스턴스 소비처(StarField):** `mood=attribute('aMood','vec3')`, `brightness=attribute('aBrightness','float')`, `seed=attribute('aSeed','float')`, `hueShift=attribute('aHueShift','float')`, `time=공유 uTime uniform`. attribute 명세·값 계산(modulated/reshaped brightness 등 08의 로직)은 StarField가 그대로 소유한다. 통합으로 바뀌는 건 "머티리얼을 어디서 만드나"뿐 — 결과 동일.
- **단일 소비처(Star3D):** `mood=uniform(Color(hex))`, `brightness=uniform(b)`, `seed=상수`, `hueShift=uniform(0)`, `time=uniform(0)`. Star3D가 `useFrame`에서 time·brightness uniform을 갱신한다(현재 `buildSingleStar(...).update(t,b)` 역할을 소비처로 이관).

### 조명 통일 (자가발광)

> **⚠ spec 03 (changes/03, job 08)로 재정의됨 — "씬 독립"은 더 이상 전제가 아니다.** 아래의 "모든 형태 자가발광, 광원 무관"은 self-light 도입 전 결정이었다. 현재: **self-glow(자가발광) 베이스는 씬 독립으로 보존**하되, **반사(reflection) 항이 가산**된다 — 우주는 중앙 자아-별 점광, 배경(사인인/랜딩/초대)은 우상단 평행광. `buildStarBody`는 `StarShadeInputs`에 `selfLightPos`/`lightPositional`/`litMix`/`recency`/`focus`를 받고, 반사를 emissiveNode TSL 그래프 안에서 계산한다(진짜 PointLight 객체 없음 → 단일 InstancedMesh·헌법8 유지). crystal 면 음영은 ndv↔N·L 크로스페이드, liquid/aurora는 반사 제외(법선 부정확). 또 "완전 emissive-only"는 사실 **emissive-dominant**다(ambient_fill이 albedo를 약하게 lit). 상세 모델은 [policy/domain/star.md](../policy/domain/star.md) §밝기·감쇠 3채널.

- 4형태 모두 **forms.ts의 ambient-safe(자가발광/fresnel/변위) 셰이딩을 정전으로 채택**한다. `single.ts`가 directional 조명을 전제로 갈라뒀던 튜닝(예: crystal의 면 음영)은 자가발광 fresnel(ndv) 방식으로 수렴한다.
- 결과: 별 본체는 씬에 directional 광이 있든 없든 동일하게 보인다 → 어느 페이지에 꽂아도 일관. 단일 별은 directional 하이라이트가 사라지는 미세 정규화가 있을 수 있다(비목표에서 허용). *(↑ spec 03 이전 결정 — 위 ⚠ 참조.)*

### FSD 배치 & 라이브러리화 전제
- 프리미티브는 `three`/`three/tsl`/`three/webgpu`에만 의존하므로 `entities/star/ui`에 둔다(§4: `model`/`shared/lib`은 three 미의존이라 거기엔 못 둠).
- 프리미티브는 앱 store·테마(`appearance`)·force-sim·navigation/focus FSM·Supabase를 **import하지 않는다.** 그 결합은 모두 소비처(StarField는 memory/synapse store를, widgets/star3d는 appearance를) 쪽에 남는다.
- 이 import 청정성이 곧 라이브러리화 가능성이다 — 후속에 `three`를 peer dep으로 두고 그대로 lift할 수 있다(이번 범위 아님).

### 소비처별 합성(A에서 변경 없음)
- `widgets/star3d`(`Star3D`·`ThemedStar`·`BrandMark`·`StarCanvas`·`halo`)는 프리미티브를 소비하는 **합성 레이어**로 유지된다 — halo·캔버스·DOM 배치·테마 색 해석은 여기 책임. `Star3D`만 `buildSingleStar` → `buildStarBody` 소비로 바뀐다.
- 사인인·초대 페이지 코드는 변경 없음(BrandMark API 유지). 랜딩 히어로만 B에서 재합성된다.

### 랜딩 히어로 한-캔버스 합성 (B)
- 새 히어로 씬 컴포넌트 한 R3F `<Canvas>`(`createRenderer` 경로 — `StarCanvas`/`FluidGradient`와 동일)가 다음을 한 씬에 담는다:
  - **배경(뒤) 레이어** — fluid/nebula 류 절차적 배경을 `renderOrder` 낮게·`depthTest=false`로 별 뒤에. (기존 `FluidGradient`의 절차적 머티리얼을 재사용/이식하거나 단순 nebula 빌보드로.)
  - **별 본체** — A의 `buildStarBody`(uniform 바인딩)를 메시로. 위치·자전은 이 컴포넌트가 주입(고정 위치 + 셰이더 자전).
  - **배경(앞) 레이어** — 투명/additive·`depthWrite=false`·`renderOrder` 높게로 별 앞을 천천히 흐르는 한 겹. 별이 그 사이로 비친다.
- 카메라는 단순 fit/고정 ortho 프리셋(FSM·컨트롤러 없음). `manual=true`로 R3F 리사이즈 frustum 덮어쓰기를 막는다(FluidGradient에서 확인된 이슈).
- `HeroSection`이 `<ThemedStar>`(분리 캔버스) 대신 이 히어로 씬을 쓴다 — 결과적으로 히어로의 WebGPU 컨텍스트가 1개로 합쳐지고(질문 1), 앞/뒤 레이어 흐름이 가능해진다(질문 2).
- reduced-motion·dpr·프레임 스로틀 등 성능 노브는 기존 `FluidGradient`/`StarCanvas` 관용구를 따른다.

## 수용 기준 (인수 조건)
**A. 별 프리미티브 통합**
1. 별 4형태(deepfield/aurora/liquid/ember)의 geometry와 TSL 셰이딩이 `entities/star`의 **단일 통합 빌더 한 곳에서만** 정의된다. `forms.ts`/`single.ts`로 갈렸던 form 셰이더 중복이 제거된다.
2. 통합 빌더는 입력을 노드 접근자(`StarShadeInputs`)로 받아, **attribute 바인딩(인스턴스)·uniform 바인딩(단일)** 양쪽에서 같은 셰이더를 생성한다.
3. 우주 `StarField`는 통합 빌더(attribute 바인딩)를 쓰고, 별 그림결과(형태·mood색·밝기·자전·birth/focus/resonance)가 이전과 동일하다 — 코어 루프 시각 회귀가 없다.
4. 사인인·초대·랜딩의 별(`BrandMark`/`ThemedStar`/`Star3D`)이 통합 빌더(uniform 바인딩)를 쓰고, 현재 룩을 유지한다(자가발광 통일에 따른 directional 하이라이트 미세 정규화만 허용).
5. 통합 빌더는 위치·움직임을 만들지 않는다 — instance matrix, force-sim 좌표, 표상 부동(38), wobble, mesh spin은 전부 소비처가 주입한다.
6. time(및 단일의 brightness) uniform의 소유·매 프레임 갱신이 소비처(StarField/Star3D)에 있고, 빌더는 노드 그래프만 반환하는 순수 함수다(uniform .value를 돌리지 않는다).
7. 통합 빌더 모듈은 앱 store·테마·force-sim·FSM·전송 계층을 import하지 않는다(`three`/`three/tsl`/`three/webgpu`만 의존). FSD 단방향·라이브러리화 전제를 충족한다.
8. 셰이더는 TSL, 우주 별은 InstancedMesh를 유지한다(헌법8). build·lint·typecheck·steiger(FSD) 게이트를 통과하고, 우주 코어 루프(작성→별 등장·회상·focus·항행)에 회귀가 없다.

**B. 랜딩 히어로 한-캔버스 합성 (feasibility)**
9. 랜딩 히어로가 별 본체와 배경을 **하나의 R3F 씬/WebGPU 컨텍스트**에 렌더한다 — 히어로 영역에서 동시에 떠 있는 WebGPU 컨텍스트가 1개다(이전: 별 캔버스 + 배경 분리).
10. 그 씬에서 배경 레이어가 **별 뒤와 별 앞 양쪽**에 존재하고(깊이/`renderOrder`로 정렬), 별이 앞 레이어 사이로 비친다 — "배경 일부는 별 앞으로·일부는 뒤로 흐른다"가 실제로 보인다.
11. 히어로 별은 A의 `buildStarBody`를 소비하며(별도 셰이더를 새로 만들지 않음), 위치·자전은 히어로 씬이 주입한다(프리미티브가 만들지 않는다 — 기준 5와 일관).
12. 랜딩의 나머지(배경·스크롤 시스템, 히어로 외 섹션)는 변경되지 않는다. 합성이 히어로 경계를 넘어 배경 시스템 전체로 번지면 plan 43으로 분리한다(이번 인수에서 제외).

## 정책 영향 (policy / values)
- **values.yaml:** 새 항목 없음. form별 셰이더 계수(noise 옥타브·roughness·flow 속도·fresnel 지수 등)는 overview §values 규칙의 **제외 범주(공식/배열 콘텐츠 성격)**라 values.yaml 비대상이다. 다만 통합의 본질이 곧 "forms vs single로 갈렸던 계수를 하나로 못 박기"이며, 이때 **통합 빌더 자체가 그 SSOT**가 된다(코드가 단일 출처).
- **policy:** 도메인 규칙 변화 없음. 별 의미색·modulated/reshaped brightness·decay 바닥(`A_MIN`)·표상 부동은 08·26·38·`policy/`가 그대로 권위다. 이 plan은 *시각 정의의 중복을 없애는 구조 변경*이지 규칙·수치 변경이 아니다.
- **tech/architecture(구현 완료 시 교차 반영):** §3.1(렌더링)에 다음이 참이 된다 — "별 바디는 입력 바인딩(attribute/uniform)이 추상화된 단일 TSL 프리미티브(`entities/star/ui`)이며, 위치·움직임·time 소유는 소비처가 가진다. 프리미티브는 `three`만 의존해 라이브러리 추출이 가능하다." (구현 전까지는 본 plan의 이 절이 그 규칙의 출처다.)
