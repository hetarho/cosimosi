# 50. deletion ui

> Deletion + letting-go UI — the presentation unit of Epic H: the two user flows through which a diarist removes a
> memory trace. **Full delete** ([X1][X2]) removes a diary's derived episodic-memory stars with a reassuring
> **30-day restore** promise;
> **letting-go** ([X4][X6]) is the symbolic partial delete — *say the words to release* → AI suggests this-memory-only
> semantic neurons → the user approves → seal (permanent, no timer, [X5]). Both flows carry **export reassurance** copy
> ([W6], "언제든 CSV·MD로 내보낼 수 있어요") and a **professional-resource notice** ([X7]: letting-go is not therapy).
> Scope: FE | Status: planning
>
> Layer P (presentation) — the third and final unit of Epic H, built after [48.deletion-rules](48.deletion-rules.md)
> (D) and [49.release-usecase](49.release-usecase.md) (U). Home: `widgets/deletion-flow` (composing three
> `features/*`) opened from the diary reader ([47.diary-reader-page](47.diary-reader-page.md)) or from the star-detail
> panel's source-diary action ([35.star-detail-panel](35.star-detail-panel.md)), over the running
> `widgets/universe-canvas` ([23.universe-canvas](23.universe-canvas.md)). It consumes the `Release` /
> `SuggestLetGo` contract owned by [49.release-usecase](49.release-usecase.md). It builds the flows' **UI and
> control-state only** — the sealing, soft-delete, restore timer, and AI suggestion are the backend's.

## Purpose

Epic H's domain ([48.deletion-rules](48.deletion-rules.md)) and use-case ([49.release-usecase](49.release-usecase.md))
give the user the *right to remove* — the one exception to "the system never originates deletion" ([I1]). But a right the user
cannot reach is no right at all, and deletion is the **most emotionally charged** surface in the product. This unit is
where that capability becomes something a person can do **without fear** — because both paths are honest about what
survives (the original diary as an objective record during the restore/export window, [I2][D4]) and reassure that the
words can be exported at any time ([W6]).

It realizes the two removal flows from PRD [§5.5](../PRD.md):

- **Full delete** ([X1][X2]) — the diarist removes one diary's derived episodic-memory stars entirely. The UI makes the
  **soft-delete + 30-day restore** promise concrete ("30일 안에는 되돌릴 수 있어요") so the act never feels irreversible or
  punitive.
- **Letting-go** ([X4][X6]) — the *symbolic* partial delete: the diarist **says which words / meanings they want to
  release**, the AI identifies the semantic neurons **only this memory uses** and *proposes* them as seal candidates,
  the user **approves**, and the domain seals them (permanent, no restore timer, [X5]). Emotion, time, space, entity,
  color, and the star itself are untouched — what remains is a *silent engram*, a trace with no content ([X4]).

Because letting-go trades on the language of release and closure, it is bound by [X7]: it frames letting-go as
**symbolic**, claims **no therapeutic efficacy**, and — when a **heavy state** is detected — surfaces
**professional-resource guidance**, never implying the app substitutes for care. It builds the **flows and their
control-state only**; the sealing, shared-vs-orphan logic, the 30-day sweep, and the AI suggestion are owned by
[48.deletion-rules](48.deletion-rules.md) / [49.release-usecase](49.release-usecase.md).

## Scope / Non-goals

**In scope**
- **`widgets/deletion-flow`** — a modal / bottom-sheet opened primarily from the **diary reader**
  ([47.diary-reader-page](47.diary-reader-page.md)) for a diary, and secondarily from the **star-detail panel**
  ([35.star-detail-panel](35.star-detail-panel.md)) as "delete this star's source diary." It opens over the running
  `widgets/universe-canvas` when invoked in-universe (it does not remount the renderer, [23]). It composes the three
  features below and owns the flow's **XState machine** + the (small) selection store.
- **`features/delete-memory`** — the **full-delete** flow ([X1][X2]): a confirm step that states plainly that the source
  diary and **all stars born from it** are being released, lists/previews the affected stars, and carries the **30-day
  restore reassurance** ("30일 안에는 되돌릴 수 있어요") + the **export reassurance** ([W6]) before calling **`Release(diary_id)`**
  (full delete). On success all affected stars are **optimistically removed** from the canvas ([X3]); embeddings /
  centrality settle on the next `GetUniverse` read.
- **`features/restore-memory`** — the **restore** affordance ([X2]) for a soft-deleted diary release group still inside
  its 30-day window: surfaces the remaining window and calls **`Restore(diary_id)`**, re-inserting all affected stars.
  (Where the soft-deleted list lives — a section of the diary-reader [47] or a settings surface [52] — is **Decided**
  below.)
- **`features/let-go`** — the **letting-go** flow ([X4][X6]) in four steps: **(1) say the words** — a free-text input
  where the diarist names the words / meanings they wish to release; **(2) suggest** — call
  **`SuggestLetGo`** which returns the AI-identified *this-memory-only* semantic neurons as **seal candidates**;
  **(3) approve** — the user reviews the candidates and **selects which to seal** (AI suggests, the user decides,
  [X6]); **(4) seal** — call **`LetGo(approved_neuron_ids)`**, permanent with **no restore timer** ([X5]).
  The UI states throughout that emotion / time / space / entity / color / the star are kept ([X4]).
- **The professional-resource notice ([X7])** — letting-go copy frames the act as **symbolic release / blurring**,
  claims **no efficacy**, and never implies the app replaces care. When the release-usecase's **heavy-state signal**
  ([49]) is set on the `SuggestLetGo` response, the flow surfaces a **gentle, non-blocking professional-resource
  guidance** panel (region-appropriate help resources) before the approve step. The **detection** is the backend's;
  this unit only **renders** the notice honestly.
- **Export reassurance ([W6])** — both flows link [46.provenance-export](46.provenance-export.md), but the promise is
  flow-specific. Letting-go can say the diary remains exportable because it does not delete the original. Full-delete copy
  says the diary can be exported before deletion or during the 30-day restore window, and that after the window the
  original diary text is permanently deleted by the backend sweep ([I2][D4][X2]).
- **The flow control-state as one XState machine** — `model/deletion-flow.machine.ts` (§3.2): control-state only
  (which step of which flow), with the selected candidate ids + the typed release phrase held in Zustand/Query
  (ids/phase in the machine, data in the store).
- **UI copy via i18n** ([08.i18n-seam](08.i18n-seam.md) / the app `shared/i18n` seam) — **no hardcoded strings**; the
  **most restrained, careful Korean voice** in the product (this is the emotionally heaviest flow): plain, honest,
  unadorned, no decorative emoji, no translation-ese, no clerical AI phrasing, and **no claim of therapeutic effect**.
  **web + mobile parity** — the same widget + three features; `*.native.tsx` siblings **only** for the sheet / modal /
  input primitives that genuinely differ (§3.5).

**Non-goals**
- **No deletion / sealing domain logic.** Soft-delete, orphan-vs-shared neuron sealing, LTD on the deleted memory's
  synaptic contribution, exclusion from all dynamics ([X3]), and letting-go's semantic-neuron-only rule are owned by
  [48.deletion-rules](48.deletion-rules.md). This unit only *calls* the contract and renders the result.
- **No release use-case / RPC.** `Release` (full delete = 30-day soft-delete + restore; letting-go = permanent seal),
  `SuggestLetGo` (AI suggests this-memory-only semantic neurons), the 30-day hard-delete sweep, and the heavy-state
  detection are owned by [49.release-usecase](49.release-usecase.md). This unit adds **no** proto.
- **No export implementation.** The CSV / MD export itself + the append-only provenance it exports are
  [46.provenance-export](46.provenance-export.md). This unit only **links** it as reassurance.
- **No hard delete, ever, from the UI.** The 30-day sweep is a backend, user-initiated-in-origin operation ([49]); the
  UI never offers a "delete permanently now" button (letting-go is permanent [X5], but it *seals* — it never
  hard-deletes the diary [I2]).
- **No star-detail panel** ([35.star-detail-panel](35.star-detail-panel.md), the host of the affordances — this unit
  opens *from* it) and **no settings / reader page** — where soft-deleted memories are listed for restore is
  [47.diary-reader-page](47.diary-reader-page.md) / [52.settings-page](52.settings-page.md); this unit provides the
  restore *feature*, not its host page.
- **No new tuning scalars hardcoded in FE** — the 30-day window and any candidate cap come from generated config
  (§Policy), owned by [49.release-usecase](49.release-usecase.md).

## Grounding

## Design

### Placement (FSD, §3.1)
```
apps/web/src/widgets/deletion-flow/
├── ui/            DeletionFlowSheet (modal/bottom-sheet host) + step panels + professional-resource notice;
│                  *.native.tsx siblings for sheet/modal/input primitives
├── model/         deletion-flow.machine.ts (XState) · the selection/phrase store · pure selection helpers
└── index.ts       public API (the sheet + the "open diary full-delete" / "open letting-go" affordances hosts compose)

apps/web/src/features/
├── delete-memory/    ui (full-delete confirm + 30-day + export reassurance) · api (Release + optimistic remove)
├── restore-memory/   ui (restore affordance + remaining window) · api (Restore + optimistic re-insert)
└── let-go/           ui (say-the-words → candidate review/approve → seal + professional-resource notice)
                      · api (SuggestLetGo + LetGo)
```
- `widgets/deletion-flow` composes the three `features/*` (each = **one user action**, §3.1) and owns the **flow
  machine** and the selection store. It imports the domain-mirror entities (`entities/episodic-memory`,
  `entities/neuron`) from [23] only to *read/optimistically-update*; it never imports `three` or a visual entity (§3.4).
  The delete/seal's *visual* consequence (the star vanishing / the star persisting as a silent engram) is the read
  model's projection, not this widget's code.
- **web + mobile parity (§3.5):** the same widget + three features on both apps; `model` and `api` segments are shared
  verbatim (pure, no DOM/native). Only `ui` forks where a primitive differs — the **sheet / modal host** (web modal /
  DOM ↔ RN bottom-sheet) and the **text input** — via `*.native.tsx` siblings.

### The contract (owned by [49], consumed here — §2.7 unary)
The proto lives in `memory.v1.MemoryService` ([49.release-usecase](49.release-usecase.md) owns the shapes; this unit
only calls them). Epic H exposes **four distinct RPCs** — there is **no** single `Release(kind=…)` verb. This plan
states the **consumed shapes** (proto is excluded from values, per the config rule):

- **`Release(ReleaseRequest{ diary_id }) → ReleaseResponse{ diary_id, episodic_memory_ids, deleted_at }`** — **full
  delete** ([X1][X2]): soft-deletes every episodic memory born from the diary (30-day window) + seals orphan neurons +
  weakens the shared contribution (backend, [48]/[49]). The returned `deleted_at` (real-clock UTC) is what the restore
  surface reads to show the remaining window; `episodic_memory_ids` is the authoritative optimistic-removal set. The
  `deleting` machine state covers it.
- **`Restore(RestoreRequest{ diary_id }) → RestoreResponse{ diary_id, episodic_memory_ids }`** — reverses a diary full
  delete still **inside its window** ([X2]); re-inserts the affected stars.
- **`SuggestLetGo(SuggestLetGoRequest{ episodic_memory_id, words }) → SuggestLetGoResponse{ repeated SealCandidate
  candidates, HeavyState heavy_state }`** — **letting-go step 1**: the diarist's typed **words** are sent; the response
  returns the AI-identified **this-memory-only semantic neurons** as candidates (each `SealCandidate{ neuron_id, name,
  reason }` — the `reason` string is shown in the review step) plus the **heavy-state signal** ([X7]/[49]). **AI suggests
  only** — nothing is sealed by this call. The `suggesting` machine state covers its LLM latency.
- **`LetGo(LetGoRequest{ episodic_memory_id, approved_neuron_ids }) → LetGoResponse{ sealed_neuron_ids }`** —
  **letting-go step 2**: seals the **user-approved** subset, permanent, **no timer** ([X4][X5][X6]). The domain
  re-validates each id is a this-memory-only `semantic` neuron ([48][49]); the FE cannot seal a non-candidate.
- **Schema-forced boundary ([I3][I11]):** the letting-go request carries **only** `{ episodic_memory_id,
  approved_neuron_ids }` (full-delete / restore carry only `{ diary_id }`) — no emotion / position / color /
  strength / time field, and no way to command a hard delete or seal a shared/foreign neuron. There is structurally
  nothing for the user (or a prompt-injected model) to set that could touch the meaning layer. The FE editable surface
  mirrors this: the diarist selects *which suggested semantic neurons to seal* and nothing else.

### Full-delete flow ([X1][X2]) — reassurance-first
`features/delete-memory` is a **single confirm step** (no multi-screen ceremony — deletion should be clear, not
punitive). The confirm states plainly *what is removed*: the source diary's derived stars / episodic memories, including
a preview count/list of every affected star; their private/orphan meaning; and the shared meaning that is **kept** ([X1]).
If opened from a star-detail panel, the copy explicitly says this action affects **all stars born from the same diary**,
not only the currently selected star. It carries **two reassurances** before the action:
- **30-day restore** ([X2]): "30일 안에는 되돌릴 수 있어요." — the act is reversible for a month.
- **export** ([W6]): "삭제 전이나 30일 복구 기간 안에는 CSV·MD로 내보낼 수 있어요." — with a link to
  [46.provenance-export](46.provenance-export.md), and clear wording that the original text is permanently deleted after
  the window.

On confirm, `Release(diary_id)` is called and all affected stars are **optimistically removed** from the
`entities/episodic-memory` store; the canvas stops rendering them ([X3] — sealed/deleted things are excluded from *all*
dynamics, so there is no ghost pull [I5]). Orphan-neuron sealing, shared-neuron preservation, and the LTD on the removed
memories' synaptic contributions are the backend's ([48]); their visual settle arrives on the next `GetUniverse` read.
**Failure / rollback:** if `Release` fails, the optimistic removal is rolled back and the stars reappear; nothing was
deleted (§2.8).

**Decided (2026-07):** full-delete uses a **single, plainly-worded confirm with the two reassurances inline** — not a
"type the diary title to confirm" friction gate. The 30-day restore ([X2]) already makes the act safely reversible, so
heavy friction would read as punitive rather than protective; the honest reassurance carries the caution.

### Letting-go flow ([X4][X6]) — say the words → suggest → approve → seal
`features/let-go` is a **four-step** flow, each step a machine state:
1. **Say the words** (`phrasing`) — a restrained free-text input: the diarist names the words / meanings they wish to
   release. The framing is **symbolic** from the first screen ([X7]) — "놓아주고 싶은 말이나 의미를 적어 주세요" — with
   the honest note that this **blurs a trace, it is not treatment**.
2. **Suggest** (`suggesting`, loading) — `SuggestLetGo(words)` returns the AI-identified **this-memory-only
   semantic neurons** as candidates + the heavy-state signal. This is the LLM-latency loading state.
3. **Approve** (`approving`) — the candidates are shown as a reviewable list; the diarist **selects which to seal**
   (default selection is a decision left to [49]'s suggestion confidence — the UI presents, the user decides, [X6]).
   The UI states, persistently, that **emotion / time / space / entity / color / the star are kept** ([X4]) — what is
   sealed is only the *content words*, leaving a **silent engram**. If `heavy_state` is set, the
   **professional-resource notice** is surfaced **before** this step's action (see below).
4. **Seal** (`sealing`, loading) — `LetGo(approved neuron_ids)` seals the approved neurons, **permanently,
   with no restore timer** ([X5]). On success the star **remains** (color/shape/position live on, [X4]); the seal
   excludes those neurons from dynamics ([X3]) so the layout re-settles ([I5]). The export reassurance ([W6]) is shown
   here too — the released words still survive in the export.

**Decided (2026-07):** letting-go is **irreversible and the UI says so plainly at the seal step** ([X5] — no 30-day
timer, unlike full-delete). The flow does **not** offer an "undo" after sealing; instead it states before sealing that
this is permanent and reminds that the original words remain exportable ([W6]) — closure without destruction of the
record ([I2][D4]).

### Professional-resource notice ([X7]) — symbolic, no efficacy, non-substitution
- **Framing everywhere:** letting-go copy calls the act **symbolic release / blurring** and **claims no therapeutic
  efficacy**. No screen implies the app treats, heals, or substitutes for professional care.
- **Heavy-state surfacing:** the **detection is the backend's** ([49] sets `heavy_state` on the `SuggestLetGo`
  response — e.g. from the phrase + the memory's emotional signal). When set, the flow surfaces a **gentle,
  non-blocking professional-resource guidance** panel (region-appropriate help resources) **before** the approve step.
  It never blocks the user from proceeding (autonomy preserved) and never phrases the app as an alternative to that help.
- **Decided (2026-07):** the notice is **advisory, not gating**. The specific resources are **i18n content, region-aware**
  (a Korean-locale default set), authored at implement time; this unit reserves the notice's **placement and honesty
  rules**, not the resource list's contents. The no-clinical-claim rule is enforced by **copy review**, not a runtime check.

### Restore ([X2]) — the reversible half of full-delete
`features/restore-memory` surfaces a soft-deleted diary release group still inside its **30-day window** — the remaining
window is derived from the `deleted_at` returned by `Release` (real-clock UTC, [49]) plus
`release.soft_delete_retention_days` (config, owned by [49]) — and offers **되돌리기**: `Restore(diary_id)` re-inserts all
affected stars optimistically. The **host** of the soft-deleted *list* is not this unit's page.
**Decided (2026-07):** the restore *feature* lives here (Epic H owns its own affordance), but the **soft-deleted diary
release list** is hosted by [47.diary-reader-page](47.diary-reader-page.md) (a "지운 일기" / "지운 기억" section of the
archive) so restore sits beside the immutable diary that survives through the restore window — reinforcing [W6]/[D4].
[52.settings-page](52.settings-page.md) may also link it. This unit exports `features/restore-memory` for those hosts to
mount.

### Flow control-state (XState, §3.2)
`model/deletion-flow.machine.ts` holds control-state only; **all data** (the typed phrase, the candidate list, the
selected ids) lives in the Zustand/Query store, referenced by id from the machine.

| State | Meaning | Exits |
|---|---|---|
| `idle` | sheet closed | `OPEN_DELETE` → `confirmingDelete`; `OPEN_LETGO` → `phrasing` |
| `confirmingDelete` | diary full-delete confirm shown (affected stars + 30-day + export reassurance) | `CONFIRM` → `deleting`; `CANCEL` → `idle` |
| `deleting` | `Release(diary_id)` in flight; optimistic remove applied to affected stars | `done` → `done`; `error` → `confirmingDelete` (rollback) |
| `phrasing` | letting-go "say the words" input ([X6] step 1) | `SUGGEST` → `suggesting`; `CANCEL` → `idle` |
| `suggesting` | **loading** — `SuggestLetGo` in flight | `done` → `approving`; `error` → `phrasing` (retriable) |
| `approving` | candidate review + select-to-seal ([X6] step 3); heavy-state notice if flagged | `SEAL` → `sealing`; `BACK` → `phrasing`; `CANCEL` → `idle` |
| `sealing` | `LetGo` in flight | `done` → `done`; `error` → `approving` (retriable, nothing sealed) |
| `done` | the star vanished (delete) or persists as a silent engram (let-go); sheet closes | `RESET` → `idle` |

- `suggesting` and `sealing` are explicit loading states so the UI shows a restrained affordance during the LLM /
  release latency and so a failed call returns to a retriable state. `restore` is a separate one-shot feature (its host
  page drives it), not a state of this machine.
- Per §3.2, any per-frame scene effect (the star's fade-out on delete) reads the machine via `getSnapshot()` in
  `useFrame`, never React state; in practice the remove/settle is a discrete event handed to the read model.

### Optimistic update + the visual consequence (§2.8)
- **Full delete:** on success every affected memory in the released diary is **removed** from
  `entities/episodic-memory`; the canvas stops rendering those stars at once ([X3]). Neuron sealing / synapse weakening
  ([48]) settle on the next `GetUniverse`. A failed `Release` rolls the removal back.
- **Letting-go:** the star is **not** removed — it **persists** as a silent engram ([X4], color/shape/position kept).
  The optimistic update marks the approved neurons sealed in the `entities/neuron` mirror so the detail panel shows the
  thinned content at once; the emergent re-settle ([I5]) and any current-text word-loss arrive on the next read. A
  failed `Release` rolls the mark back (nothing sealed).
- No polling, no streaming (§2.7); slow consequences surface on the next read (§2.8), matching plan 27's launch pattern.

### Golden-parity stance
This is a **presentation** unit — it holds **no** cross-platform domain math (unlike [48.deletion-rules](48.deletion-rules.md),
whose sealing/exclusion logic and [18.synapse-plasticity](18.synapse-plasticity.md)-seam LTD are golden-parity domain).
Web and mobile share this unit's `model` (the XState machine, pure selection helpers) and `api` (the RPC calls)
**verbatim**; only `ui` primitives fork (§3.5). No `packages/memory-logic` function is added here.

### Copy (i18n) — the most careful voice in the product
All user-visible strings resolve through the i18n seam ([08.i18n-seam](08.i18n-seam.md) / `packages/i18n`); none are
hardcoded (enforced by `lint:raw-strings`). Because this is the emotionally heaviest flow, the voice is the most
**restrained and honest**: plain sentences, no decorative emoji, no translation-ese, no clerical AI phrasing. The
letting-go copy **must not** claim therapeutic effect ([X7]); the professional-resource notice is gentle and
non-substituting. Web and mobile draw from the **same** message keys (parity).

   the **30-day restore reassurance** and the **export-before/restore-window ([W6]) reassurance** before the action
   ([X1][X2][W6]).
2. Confirming full delete calls **`Release(diary_id)`**; on success all returned `episodic_memory_ids` are
   **optimistically removed** and the canvas stops rendering them with **no residual pull** ([X3][I5]); a failed call
   rolls back and the stars reappear (§2.8).
3. A soft-deleted diary release group inside its **30-day window** can be **restored** via `features/restore-memory`
   calling **`Restore(diary_id)`**, re-inserting the affected stars; the window length is read from generated config, not
   hardcoded ([X2]).
4. The **letting-go** flow runs in four steps — **say the words** → **`SuggestLetGo`** (this-memory-only semantic
   neuron candidates) → **user approves which to seal** → **`LetGo(approved_neuron_ids)`** — and **AI only
   suggests; the user decides which are sealed** ([X4][X6]).
5. Letting-go seals **semantic neurons only**; the flow states and the contract enforces that **emotion / time / space /
   entity / color / the star are kept** — the sealed memory **persists as a silent engram** (color/shape/position live
   on), and its content reads as thinned on the next read ([X4]).
6. Letting-go is **permanent with no restore timer** and the UI says so plainly at the seal step; it offers **no undo**
   after sealing, and reminds that the original words remain exportable ([X5][W6][I2]).
7. Letting-go copy frames the act as **symbolic release / blurring** and **claims no therapeutic efficacy**; **no screen
   implies the app substitutes for care** ([X7]).
8. When the backend sets the **heavy-state signal** on the `SuggestLetGo` response, the flow surfaces a **gentle,
   non-blocking professional-resource guidance** panel before the approve step (advisory, not gating); the **detection
   is the backend's**, this unit only renders the notice ([X7]).
9. The flow's request surface carries **only** `{ diary_id }` (full-delete / restore) and
   `{ episodic_memory_id, approved_neuron_ids }` (letting-go) — there is **no** `kind`, emotion, position, color,
   strength, or time field anywhere, and no way to command a hard delete or seal a shared/foreign neuron; the meaning
   layer cannot be overwritten ([I3][I11]).
10. Neither full delete nor letting-go mutates the `Diary` in place; the **export reassurance ([W6])** is shown in both
    flows and links [46.provenance-export](46.provenance-export.md). Full-delete copy says the objective record is
    preserved/exportable through the restore window and permanently removed after the sweep; letting-go copy says the
    original remains exportable because the diary itself is not deleted ([I2][D4][X2]).
11. The flow **never** triggers a system-driven delete and offers **no** "delete permanently now" for full-delete —
    removal is the user's explicit act, and the only hard delete (the 30-day sweep) is the backend's ([I1][X2]).
12. The flow's **control-state** is one XState machine (`idle → confirmingDelete → deleting → done` and
    `idle → phrasing → suggesting → approving → sealing → done` + error paths) with `suggesting`/`sealing` as loading
    states; **the typed phrase, candidate list, and selected ids live in Zustand/Query**, not machine context (§3.2).
13. **All** user-visible copy resolves through the i18n seam (no raw strings; `lint:raw-strings` passes), in the most
    restrained / honest Korean voice, with **no therapeutic-efficacy claim** ([X7]).
14. **web + mobile parity:** the same widget + three features run on both apps, sharing `model` / `api` verbatim;
    `*.native.tsx` siblings exist **only** for the sheet / modal / input primitives (§3.5).
