# 18. synapse plasticity

> Synapse plasticity — the pure domain math that grows, weakens, seeds, and reads synapse strength: saturating LTP,
> associative LTD, tiered initial strength, and read-time time-decay. IO-free Go domain functions, mirrored in TS for
> golden-parity so the client simulation and the server run the *same* math.
> Scope: BE | Status: built

## Purpose

Synapses are the load-bearing quantity of the universe: a synapse's strength drives the filament width/brightness a
user sees ([V6]) and, together with shared neurons, is what makes constellations *emerge* ([L2][L7]). The rules that
change that strength are neuroscience, not UI — they must be the *same* everywhere they run, and they must never violate
the model. This unit owns those rules as **pure, IO-free domain functions** over the `Synapse` value object:

- **Potentiation (LTP)** with a *saturating* curve so repeated co-firing yields diminishing returns and asymptotes to a
  single cap, never a runaway edge ([L8][L9]).
- **Depression (LTD)** — associative, local weakening, defined here as the primitive that later epics call.
- **Initial strength** — a low start, tiered by *why* the two neurons became linked ([L10]).
- **Effective (read-time) strength** — the stored base decayed by elapsed universe time ([L5]).

Because the client renders these values every frame and the server needs the same math for later cost-gating, the
functions live once in Go (`internal/memory`) and once in TypeScript (`packages/memory-logic`), pinned equal by
golden-parity fixtures. The tuning coefficients are the only part that leaves code, into `spec/values.yaml`.

## Scope / Non-goals

**In scope**
- Pure domain functions over the `Synapse` VO (`strength` in `0..1`):
  - `Potentiate(strength, rate) → strength'` — saturating LTP.
  - `Depress(strength, amount) → strength'` — associative LTD (the primitive; consumers wire it in later epics).
  - `InitialStrength(signalKind) → strength` — tiered initial strength by link signal.
  - An `EffectiveSynapseStrength`-style read-time function `EffectiveSynapseStrength(base, elapsedUniverseDays) → strength` — stored
    base decayed by elapsed universe time.
- **Memory-level read-time effective values (Epic-A stubs), seated here** because all read-time effective-value pure
  math shares one `packages/memory-logic` (TS) ↔ `internal/memory` (Go) home and one golden-parity harness — distinct
  from the synapse function above (PRD §3.2/§3.3 name these at the *memory* level): `EffectiveStrength(base_strength,
  recall_count) → strength` (the star's size [V3]) and `EffectiveBrightness(…) → brightness` (activation/recency [V2]).
  In Epic A they are **trivial** (`recall_count = 0` → `EffectiveStrength = base_strength`; brightness full); recall
  accumulation ([R3]) is Epic C and forgetting decay ([F1]) is Epic D.
- The **saturating strength curve** and the **single `1.0` cap** as the invariant shape; strength stays in `0..1` and
  approaches `1.0` asymptotically, never exceeding it.
- **Signal-tiered initials** ordered same-memory > shared-neuron > temporal-proximity.
- The TS ↔ Go **golden-parity** contract: identical inputs produce identical outputs within a fixed tolerance, pinned by
  shared fixtures.
- Enumerating the exact `synapse.*` coefficient/cap/initial keys these functions read.

**Non-goals**
- **No IO, no persistence, no transport.** This unit defines math, not tables, queries, RPCs, or use-cases. The
  `synapses` table and its stored facts are owned by [16.memory-aggregate-schema](16.memory-aggregate-schema.md); the
  `Link` use-case that *calls* these functions is [21.link-usecase](21.link-usecase.md).
- **No downscaling.** `Downscale` (SHY, the homeostatic global sleep-time renormalization) is a *different mechanism*
  from LTD and is owned by Epic E ([C4][I9]). This plan defines `Depress` (LTD) and names the seam only.
- **No decision about *which* signal a given link is** — choosing the tier (same-memory vs shared-neuron vs temporal)
  is the `Link` use-case's job ([21.link-usecase](21.link-usecase.md)); this unit only maps a chosen `signalKind` to a
  number.
- **No decision about *when* a synapse is created or upserted**, no `co_activation_count` / `last_activated_universe_time`
  bookkeeping (stored facts, owned by 16/21) — this unit is given a base strength and an elapsed time and returns a
  number.
- **No rendering.** Mapping effective strength to filament width/brightness is [24.star-neuron-rendering](24.star-neuron-rendering.md) [V6].
- **No emotion coupling.** Strength math never reads valence/arousal/mood — emotion never drives connection ([I3]).

## Design

The `Synapse` value object (owned by [16.memory-aggregate-schema](16.memory-aggregate-schema.md)) carries a `strength`
in `0..1` as its stored *base*, plus the cumulative facts `co_activation_count` and `last_activated_universe_time`. This
unit adds no fields; it adds the four pure functions that read and transform strength. All four are total, deterministic,
side-effect-free, and depend only on their arguments and the generated `synapse.*` constants — no clock, no repository,
no logger, no randomness. They mirror byte-for-byte (within tolerance) between Go and TS.

### The synapse VO surface (given, not defined here)

- `strength REAL` in `[0, 1]` — the stored *base* strength ([L9]).
- `co_activation_count INT` — how many times the pair co-fired (a stored fact; used by the `Link`/`Reinforce`
  consumers to decide *whether* to potentiate, not by the pure math).
- `last_activated_universe_time DATE` — the universe-time anchor for read-time decay ([L5]); the caller computes
  `elapsedUniverseDays` from this against the current universe time and passes it in.

The pure functions never mutate the VO; they take scalars and return a scalar new strength, so the same functions serve
a Go domain object and a TS mirror type without either owning the other's struct shape.

### `Potentiate` — saturating LTP ([L8][L9])

Signature (canonical; representation casing may differ per language):

```
Potentiate(strength float, rate float) → float
```

The increment is proportional to the **remaining headroom** `(1 − strength)`, scaled by the potentiation rate:

```
strength' = strength + rate · (1 − strength)
          = strength · (1 − rate) + rate            // equivalent lerp toward the cap
```

with `rate = synapse.potentiation_rate` (a coefficient in `0..1`) and the result clamped to the cap
`synapse.strength_cap` (= `1.0`).

Consequences that the acceptance criteria pin:

- **Monotone non-decreasing.** For `rate ≥ 0` and `strength ∈ [0, 1]`, `strength' ≥ strength`.
- **Diminishing returns.** The step size shrinks as `strength` grows (headroom shrinks), so the first co-firings matter
  most and frequent pairs stop gaining meaningfully — no runaway edge ([L8]).
- **Asymptotic to `1.0`, never over.** Repeated application approaches `synapse.strength_cap` (= `1.0`) from below and
  never exceeds it; a final clamp guards against floating-point overshoot ([L9]).
- **Single cap.** Per-signal differences are expressed only through the *initial* value and (if a consumer varies it)
  the `rate`, never through a different cap ([L9]).

`rate` is a parameter so a consumer can, if it ever needs to, potentiate more or less aggressively per signal without
changing the cap. **Decided (2026-07):** the signature keeps `rate` as an explicit argument (testable, reusable); every
Epic A caller passes the single generated `synapse.potentiation_rate` coefficient — varying the rate per signal is a
caller decision, not a formula change.

### `Depress` — associative LTD ([I9], PRD §3.3)

Signature:

```
Depress(strength float, amount float) → float
```

Associative, *local* weakening: it lowers a single synapse's strength, clamped at the floor `0.0` (strength never goes
negative). This is the LTD primitive; two later epics consume it (this plan defines the function, they define the policy
of *when* and *by how much*):

- **Recall neighbour-inhibition** — retrieval-induced weakening of strong competitor neighbours during recall ([R5],
  Epic C). Note the seam; the coefficient and the "count shared semantic neurons" gate are Epic C's.
- **Deletion contribution-weakening** — when a memory is deleted, shared neurons are preserved but *that memory's
  synaptic contribution is weakened* ([X1][X4], Epic H). Note the seam.

`Depress` is deliberately **not** `Downscale`: LTD is associative and local (one edge, driven by a specific
co-activation event), whereas `Downscale` (SHY) is a global, homeostatic renormalization applied across many synapses
during sleep/consolidation ([C4][I9], Epic E). They are different mechanisms and must never share an implementation.
This plan owns `Depress`; `Downscale` is Epic E — note the seam. **Decided (2026-07):** `Depress` takes a
caller-supplied `amount` (the per-signal weakening step is a consumer concern), so LTD tuning stays with the consuming
epic (C for neighbour-inhibition, H for deletion contribution-weakening).

### `InitialStrength` — signal-tiered initials ([L10])

Signature:

```
InitialStrength(signalKind SignalKind) → float
```

New synapses start *weak*, so meaning has to accrue through repetition and proximity ([L10]). The initial value is
tiered by the signal that produced the link, strictly ordered:

| `signalKind` | meaning | value |
|---|---|---|
| same-memory | both neurons co-fired inside one episodic memory ([L1]) | `synapse.initial_same_memory` |
| shared-neuron | the link arises from two memories sharing a neuron ([L2]) | `synapse.initial_shared_neuron` |
| temporal | the pair is bonded by temporal-proximity only ([L4]) | `synapse.initial_temporal` |

with the ordering invariant `initial_same_memory > initial_shared_neuron > initial_temporal`. All three are low
(well below the cap) so that a fresh link is faint and *becomes* strong only by being potentiated. `SignalKind` is a
small closed enum owned by this unit's domain (mirrored in TS); mapping a real link event to a `SignalKind` is the
`Link` use-case's job ([21.link-usecase](21.link-usecase.md)).

### `EffectiveSynapseStrength` — read-time time-decay ([L5])

Signature:

```
EffectiveSynapseStrength(base float, elapsedUniverseDays float) → float
```

[L5] splits strength into a **stored base** (cumulative facts — `co_activation_count`,
`last_activated_universe_time` — which are history and cannot be recomputed) and a **read-time decayed effective
value** (a pure function of the base and elapsed *universe* time). This function is that read-time computation: given
the stored base and how many universe-days have elapsed since `last_activated_universe_time`, it returns the currently
effective strength. It decays toward — but not below — `0.0`, so a synapse fades in appearance without being deleted
([I1] spirit; the base is never mutated by reading).

The decay is governed by one coefficient, `synapse.strength_decay_per_day` (a read-time decay rate). The exact decay
*shape* stays in code (a formula, excluded from `values.yaml`); **decided (2026-07):** the initial shape is
exponential, `effective = base · decayFactor^elapsedUniverseDays` where
`decayFactor = 1 − synapse.strength_decay_per_day` — replaceable by another monotone-decreasing curve as a formula
change if tuning demands. The invariant the acceptance criteria pin is the *behaviour*, not the shape:
`EffectiveSynapseStrength(base, 0) = base`, the result is non-increasing in `elapsedUniverseDays`, and it stays within
`[0, base]`.

**Where it runs.** `EffectiveSynapseStrength` is computed **client-side** every read for rendering (the
`packages/memory-logic` TS mirror feeds filament width/brightness, [V6], [24.star-neuron-rendering](24.star-neuron-rendering.md)),
and **server-side** for any later cost-gating — the same function, two call sites. Elapsed universe time is measured in
*universe* days (universe time only advances on launch/recall, [T]); Epic A derives the current universe time from the
latest launched memory's `created_universe_time` (the `universe_state` clock is Epic B — note the seam).

### Memory-level read-time values — `EffectiveStrength` / `EffectiveBrightness` (Epic-A stubs)

`EffectiveStrength` and `EffectiveBrightness` are the **memory-level** read-time derivations named in PRD §3.2/§3.3
(별 크기 / 밝기) — **distinct** functions from the synapse `EffectiveSynapseStrength` above (different entity, different
inputs). They are seated in this unit because all read-time effective-value pure math shares one `packages/memory-logic`
(TS) ↔ `internal/memory` (Go) home and the one golden-parity harness below, giving
[24.star-neuron-rendering](24.star-neuron-rendering.md) a defined owner to consume rather than a dangling reference.

- `EffectiveStrength(base_strength, recall_count) → strength` — the star's **size** ([V3]), grown by accumulated recall.
  **Epic A:** `recall_count = 0` at launch (recall is Epic C [R3]), so `EffectiveStrength = base_strength`; the
  recall-accumulation term is a seam.
- `EffectiveBrightness(…) → brightness` — the star's **activation/recency** ([V2]). **Epic A:** resolves to **full**
  brightness; the read-time forgetting decay that dims it is Epic D ([F1]). The signature reserves the decay input,
  stubbed in Epic A.

Both are pure, run client-side per read (rendering) and server-side for later gating, and are covered by the same golden
fixtures. Their full dynamics land in Epic C (strength) / Epic D (brightness); Epic A ships the signatures + trivial
behavior. No new `values.yaml` key is introduced by the Epic-A stubs (the recall-accumulation coefficient is Epic C, the
decay coefficient Epic D).

### Golden-parity (TS ↔ Go)

Per ARCHITECTURE §2.5, purity exists so the two runtimes agree. The Go implementation in `internal/memory` and the TS
mirror in `packages/memory-logic` are pinned equal by **shared golden fixtures**:
`apps/api/internal/memory/testdata/synapse-plasticity-golden.json` stores `(function, inputs) → expected output` rows
and a fixed `1e-9` tolerance. The fixture set covers the saturating curve at low/mid/high strength, repeated
potentiation converging to the cap, a potentiation attempt at `strength = 1.0` (stays at `1.0`), `Depress` at the floor,
`InitialStrength` for all three tiers (asserting the ordering), `EffectiveSynapseStrength` at `elapsed = 0` and across a
decay range, and the memory-level Epic-A stubs. Both consumers read the *same* generated `synapse.*` constants
(`pnpm gen:values` emits Go and TS), so a tuning change cannot desync the two sides.

### Purity & placement (ARCHITECTURE §2.4, §2.5)

- The functions live in the pure domain of `internal/memory` (Go) and `packages/memory-logic` (TS) — no `pg`, `rpc`,
  SDK, clock, or framework import. They are the innermost layer; nothing depends inward on them except by calling.
- They take and return domain scalars/VO fields, not proto DTOs or sqlc rows (§2.4 anti-corruption boundary).
- No formula is a value: only the coefficients/caps/initials go to `spec/values.yaml`; the shapes (the LTP lerp, the
  decay curve, the clamps) are code.

## Acceptance Criteria

1. `Potentiate(strength, rate)` is **monotone non-decreasing** in `strength` for `rate ∈ [0, 1]`, `strength ∈ [0, 1]`:
   `Potentiate(s, r) ≥ s`.
2. `Potentiate` exhibits **diminishing returns**: for a fixed `rate`, the increment `Potentiate(s, r) − s` is
   non-increasing in `s` (the step shrinks as strength grows), and repeated application **asymptotes to** `synapse.strength_cap`.
3. `Potentiate` **never exceeds the cap**: `Potentiate(s, r) ≤ synapse.strength_cap` for all valid inputs, and
   `synapse.strength_cap = 1.0`; potentiating a synapse already at `1.0` returns `1.0`.
4. The strength cap is a **single value** (`synapse.strength_cap = 1.0`) shared by all signal kinds; no per-signal cap
   exists — per-signal differences appear only in initials (and, if used, the rate).
5. `InitialStrength` returns a value for each of the three `SignalKind`s, strictly ordered
   `InitialStrength(same-memory) > InitialStrength(shared-neuron) > InitialStrength(temporal)`, and all three are
   `> 0` and `< synapse.strength_cap` (a low start).
6. `Depress(strength, amount)` weakens a single synapse (result `≤ strength`), **clamped at the floor `0.0`** (never
   negative), and is defined and named **distinctly** from `Downscale` — this plan implements no `Downscale`; `Downscale`
   (SHY, global/sleep) is Epic E ([I9]). The `Depress` seams for recall neighbour-inhibition ([R5]) and deletion
   contribution-weakening ([X]) are documented but their policy is not implemented here.
7. `EffectiveSynapseStrength(base, elapsedUniverseDays)` satisfies `EffectiveSynapseStrength(base, 0) = base`, is **non-increasing**
   in `elapsedUniverseDays`, and stays within `[0, base]` (decays toward but not below `0`, never mutating the stored
   base). It is governed by `synapse.strength_decay_per_day`.
8. The Go functions (`internal/memory`) and the TS mirror (`packages/memory-logic`) produce **identical outputs** for a
   shared golden-fixture table within a fixed tolerance (TS ↔ Go golden-parity), and both read the same generated
   `synapse.*` constants.
9. All four functions are **pure**: deterministic, IO-free (no clock, DB, transport, SDK, or randomness), and operate
   only on their arguments and generated constants — verified by their placement in the pure domain layer and by the
   parity tests being runnable with no infrastructure.
10. No formula, curve shape, or clamp value is stored in `spec/values.yaml`; only the coefficients, cap, and initials
    listed below are values (formulas stay in code).
11. The **memory-level** `EffectiveStrength(base_strength, recall_count)` and `EffectiveBrightness(…)` stubs are defined
    here, **distinct** from the synapse `EffectiveSynapseStrength`; in Epic A `EffectiveStrength(base, 0) = base` and
    `EffectiveBrightness` is full, with recall-accumulation (Epic C) and forgetting-decay (Epic D) reserved as seams and
    introducing no new `values.yaml` key.
