# 33. recall usecase

> Recall use-case — the backend orchestration for remembering a star. `Recall` resets the memory's recall anchors
> (last-recalled, count, gist timer) + `Reinforce`s its co-activated synapses (idempotent batch LTP) + bumps the
> star's `EffectiveStrength` size ([R3]); `Reconsolidate` rewrites `current_text`, changes the `seed`, and regenerates
> only the _remaining_ stage texts — **only on prediction error** ([R6][C7]) — while never touching the `Diary` ([I2]).
> Scope: BE | Status: planning
>
> Layer U (use-case) — the first behavioral unit of Epic C. Home: `internal/memory` use-cases (`Recall`, `Reinforce`,
> `Reconsolidate`) + new RPCs on `memory.v1.MemoryService`. It composes [30.time-advance-usecase](30.time-advance-usecase.md)'s
> `SyncToToday` capability, calls a consumer-owned `SpendGate` port ([bound no-op until Epic F](44.earn-spend-usecase.md))
> and the `PredictionError` port declared by [32](32.reconsolidation-rules.md) ([reuses the AI seam](28.ai-provider-abstraction.md)),
> and appends `memory_provenance` rows introduced by [32.reconsolidation-rules](32.reconsolidation-rules.md). It writes
> **anchors**, not pixels: the _visible_ brightness/decay/gist recovery is produced by the read-time functions Epics D/E
> define reading those anchors (see the seam note below).

## Purpose

Recall is the beating heart of cosimosi's memory model. The product's entire claim — that remembering is not replay but
reconstruction ([R]) — turns on this one use-case. Clicking a star to _see_ it is free ([R1][G1]); pressing **회고하기**
is the act of _remembering_: the star brightens, its trace strengthens, and — if what you wrote back differs in meaning
from what the star currently holds — it is **reconsolidated** into a new shape ([R6][V5]). "Show me the original" recall
does not exist; the original lives only in the `Diary` ([R7][I2]).

This unit owns that act on the backend. It fixes the load-bearing distinction the PRD draws ([R4][R6]): recalling a
memory _as it is_ (no meaning change) reinforces it — brighter, slightly larger, gist timer reset — but leaves its text
and shape untouched; recalling it _differently_ (a genuine prediction error) reconsolidates it — the current narrative
is rewritten, the `seed` (visual form) changes, and the not-yet-created stage texts are regenerated on the new basis,
while already-risen gist stars stay ([C7], z-axis is one-way). The prediction-error judgement, the Twinkle cost, and the
clock sync are each held behind a seam (a consumer-owned port / [30]'s capability); this plan specifies the orchestration
and those seams — not the model, not the price, not the clock.

**The Epic-C-before-D/E seam (Decided 2026-07, per the cross-cutting brief §CC1).** Recall runs in Sprint 2, _before_
forgetting-decay (D) and semanticization-rising (E). So `Recall` writes **anchors + counters** — `last_recalled_universe_time`,
`recall_count`, `semanticize_timer_reset_at`, the LTP on synapses, the `EffectiveStrength` bump, the provenance row —
and the **visible** brightness/decay/gist recovery ([F5][R2]) is produced by the read-time functions Epics D/E define,
which read exactly these anchors. Epic C is fully buildable and testable now at the domain/use-case level; the visible
effect lands when D/E ship against the anchors this unit sets.

## Scope / Non-goals

**In scope**

- **`Recall`** — the use-case behind the 회고하기 RPC. Given a user-authored rewrite of a _hippocampal episodic_ star,
  in one transaction it:
  - **composes `SyncToToday`** ([30.time-advance-usecase](30.time-advance-usecase.md)) so recall always happens on a
    clock synced to today ([R1a][T2] case 2) — the sync **consent gate is the UI's** ([36.recall-flow-ui](36.recall-flow-ui.md));
    this use-case exposes the capability and its ordering (sync, then recall);
  - **spends Twinkle** through the consumer-owned `SpendGate` port ([G1]) — allow-all no-op until Epic F binds the real
    gate;
  - **resets the recall anchors** ([R2][R3][C6a] reset): `last_recalled_universe_time = universe_time`,
    `recall_count += 1`, `semanticize_timer_reset_at = universe_time`;
  - **`Reinforce`s** the co-activated synapses — an _idempotent batch LTP_ use-case (distinct from the `Potentiate`
    primitive, [18.synapse-plasticity](18.synapse-plasticity.md)) — strengthening every synapse among the recalled
    star's neurons ([R3]);
  - **bumps `EffectiveStrength`** — the star's size grows with accumulated recall via the memory-level golden-parity
    function whose recall-accumulation term this unit activates ([R3][V3]);
  - **applies the neighbor forgetting ±** ([R5], neighbors only) — a recall effect, not reconsolidation-only (see
    `Reinforce` below);
  - **branches on prediction error** ([R6]): no error → `Reinforce` only, text/shape unchanged ([R4]); error →
    `Reconsolidate` ([32.reconsolidation-rules](32.reconsolidation-rules.md)).
- **`Reinforce`** — the idempotent recall-effects bundle applied on **every** recall (anchor resets + batch LTP +
  `EffectiveStrength` bump + neighbor forgetting ± [R5]), reused by [47.diary-reader-page](47.diary-reader-page.md)'s
  whole-diary jump ([D3], no rewrite). Idempotent per recall event.
- **`RecallDiaryStars`** — the no-rewrite whole-diary recall entry consumed by [47.diary-reader-page](47.diary-reader-page.md):
  after UI consent, run `SyncToToday`, spend the diary-recall quote, and apply `Reinforce` to each still-live memory born
  from the diary in **one transaction**. It returns the affected memory ids + sync interval so the UI can animate after
  the authoritative write. It never calls `Reconsolidate`.
- **`Reconsolidate`** — invoked only on prediction error ([R6]): update `current_text`, change `seed` ([V5]), regenerate
  the **remaining** (not-yet-created) semantic/decay stage texts on the new basis ([C7]) while keeping already-created
  gist stars/stages, apply the neighbor forgetting ± and the `EffectiveStrength`/gist-timer effects
  ([32.reconsolidation-rules](32.reconsolidation-rules.md) owns the rules/predicate/±math), and **append a
  `reconsolidated/user` `memory_provenance` row**. The `Diary` is never written ([R7][I2]).
- The consumer-owned **`SpendGate` port** (check-and-spend, shared with [34.view-semantic-usecase](34.view-semantic-usecase.md))
  and the **`MemoryProvenanceStore`** append port, declared in `internal/memory` and bound at the composition root; plus
  the **`PredictionError` port** declared by [32.reconsolidation-rules](32.reconsolidation-rules.md), which this unit
  wires (concrete AI adapter) and calls.
- The new RPCs on `memory.v1.MemoryService`: **`Recall`** (the rewrite → reinforce/reconsolidate result) and
  **`RecallDiaryStars`** (no-rewrite whole-diary recall). Thin handlers (proto↔domain map + call); all policy in the
  use-case/domain.
- Per-user scoping on every read, spend, compare, write, and provenance append (§4).

**Non-goals**

- **No prediction-error math, no Reshape/seed algorithm, no neighbor forgetting ± math, no `EffectiveStrength` growth
  formula.** The reconsolidation _rules_ — the semantic-compare predicate, the `seed` reshape, the `EffectiveStrength`
  size-bump curve, the neighbor forgetting-offset ± by shared-semantic-neuron count ([R5]), and the `memory_provenance`

  _orchestrates_ them; it does not define them.

- **No sync-consent flow / modal.** The consent gate ("우주 시간을 오늘로 맞출까요?") is the UI's

## Design

### Placement (domain-first Go)

`Recall`, `Reinforce`, and `Reconsolidate` are application use-cases in the core-domain `internal/memory` context,
alongside `PersistEncoded`/`Link`/`AdvanceOnLaunch`. They consume the reconsolidation rules
([32.reconsolidation-rules](32.reconsolidation-rules.md)) and the `Potentiate`/`EffectiveStrength` domain functions
([18.synapse-plasticity](18.synapse-plasticity.md)) as pure domain; the LLM compare, the Twinkle spend, the clock sync,
and provenance are reached only through consumer-owned ports declared here (§2.4). Dependency direction:
`memory/rpc → Recall/Reinforce/Reconsolidate (use-case) → pure domain`, with `SpendGate`, `PredictionError`,
`MemoryProvenanceStore`, `UniverseClockStore` (via `SyncToToday`), and the repositories reached only through interfaces.
Handlers stay thin (§2.7/§2.9#7).

### `Recall` orchestration (behind the 회고하기 RPC)

Given `(userScope, memoryId, rewriteText)` — the star the user is remembering and what they wrote back — `Recall` runs
in one `InTx`:

1. **Sync the clock.** Compose `SyncToToday(scope)` ([30]) so recall lands on a clock at today ([R1a][T2] case 2). The
   **consent** for this sync is the UI's ([36.recall-flow-ui](36.recall-flow-ui.md)); by the time `Recall` is called the
   user has already agreed. `universe_time` used below is the post-sync clock.
2. **Spend Twinkle.** Call `SpendGate.CheckAndSpend(ctx, scope, RecallIntent{memoryId})` — a `SpendIntent` naming the
   recall action + its target ([G1]). The gate is told _what_
   is being spent on (recall of this memory, whose accessibility/decay-depth signal Epic D supplies); it decides the
   price and whether the balance allows it. **Epic C binds an allow-all no-op** (the loop works without cost); **Epic F
   binds the real gate** ([44.earn-spend-usecase](44.earn-spend-usecase.md)). A denied spend aborts the transaction with
   a canonical `insufficient-twinkle` error — nothing is reset (§CC2).
3. **Load the star + its co-activated synapses** — the `EpisodicMemory` and the synapses among its member neurons
   (per-user scoped). Reject if it is soft-deleted/sealed or not an episodic star ([R1]).
4. **Prediction-error judgement.** Call `PredictionError.Differs(ctx, current_text, rewriteText) → (bool, error)` — an LLM
   _semantic_ compare (content change, not wording/spacing/order; [R6]). Held behind the consumer-owned port; concrete
   is an AI adapter, keyless mock deterministic ([28.ai-provider-abstraction](28.ai-provider-abstraction.md)).
5. **`Reinforce` the star — the recall-effects bundle (§below), always, on either branch** ([R2][R3][R5][C6a][F5]):
   - reset the anchors: `last_recalled_universe_time = universe_time`, `recall_count += 1`,
     `semanticize_timer_reset_at = universe_time` (delays the _next_ gist stage; already-risen gist stars stay [C7]);
   - **batch-LTP** the co-activated synapses ([R3]);
   - **bump `EffectiveStrength`** (size grows with the new `recall_count`, [R3][V3]);
   - **apply the neighbor forgetting ±** ([R5]) — nudge _neighbor_ memories' `forgetting_offset_days` by shared
     **semantic**-neuron count (rule owned by [32.reconsolidation-rules](32.reconsolidation-rules.md)); the recalled
     star itself recovers wholly, the ± is neighbors-only.
     This whole bundle is the recall-without-rewrite recovery ([F5]); [47.diary-reader-page](47.diary-reader-page.md)'s
     whole-diary jump batches it per memory ([D3]).
6. **Branch on the step-4 result:**
   - **No error → `Reinforce` only** ([R4]): `current_text` and `seed` are **unchanged**; step 5 stands; no provenance
     rewrite row.
   - **Error → `Reconsolidate`** (§below): rewrite `current_text = rewriteText`, change `seed` ([V5]), regenerate the
     remaining stage texts ([C7]), and append the provenance row. (Anchors, LTP, strength bump, and the neighbor ±
     already ran in step 5.)
7. **Return** the recall result — the branch taken, the new `current_text`/`seed`/`recall_count`/`EffectiveStrength`
   (so the UI re-renders the star), and the advance interval from step 1 (so the UI can play the sync). Optimistic
   write: regenerated stage texts that require an async LLM pass fill on the next read (see `Reconsolidate`).

The whole sequence is one transaction: sync + spend + anchors + reinforce + (reconsolidate) + provenance land wholly or
not at all. The `Diary` is **never** in this transaction as a write ([I2][R7]).

### `Reinforce` — the idempotent recall-effects bundle ([R3][R5][F5])

`Reinforce(scope, memoryId)` is the **recall-without-rewrite** effect applied on **every** recall (both branches): it
(1) resets the anchors (`last_recalled_universe_time`, `recall_count += 1`, `semanticize_timer_reset_at`), (2) strengthens
**every synapse among the recalled star's member neurons** by applying the `Potentiate` primitive
([18.synapse-plasticity](18.synapse-plasticity.md)) once per edge and increments each edge's `co_activation_count` /
`last_activated_universe_time`, (3) bumps `EffectiveStrength` (size, via the recall-accumulation term), and (4) applies
the neighbor forgetting ± ([R5], rule owned by [32.reconsolidation-rules](32.reconsolidation-rules.md), neighbors only).
The synapse strengthening is a **batch use-case, distinct from the `Potentiate` formula** it calls (per plan 18's
explicit seam), **idempotent per recall event** (saturating toward `synapse.strength_cap`; re-running the same recall
does not double-count). It is reused verbatim by [47.diary-reader-page](47.diary-reader-page.md)'s whole-diary jump ([D3],
reinforce without rewrite — the jump thus gets [F5] recovery + [R5] neighbor gain with **no** reconsolidation). No
memory↔memory edge is ever touched ([I4][I6]) — only neuron↔neuron synapses.

### `Reconsolidate` — only on prediction error ([R6][C7])

Invoked from `Recall` step 6 (error branch), executing the rules owned by
[32.reconsolidation-rules](32.reconsolidation-rules.md):

1. **Rewrite `current_text = rewriteText`** — the new reconstructed narrative on the star ([R6][R8a]). The `Diary` is
   untouched ([R7][I2]).
2. **Change `seed`** — `Reshape` picks a new visual-form seed so the star's _shape_ changes ([V5]); the visual signal
   of reconstruction. (Plain recall left `seed` alone, [R4].)
3. **Regenerate the _remaining_ stage texts** ([C7]) — only the **not-yet-created** semantic/decay stages are
   regenerated on the new `current_text`; **already-created gist stars/stages are kept** (they are things that actually
   happened; z-axis is one-way, [C6a]). The regeneration itself is an async LLM pass ([22.ai-worker-pipeline](22.ai-worker-pipeline.md)) —
   enqueued here, filling `semantic_stages`/`decay_stages` on the next read (optimistic write, §2.8).
4. **Append a `memory_provenance` row** — `kind = reconsolidated`, `source = user`, `text = rewriteText`,
   `universe_time`, per-user scoped ([R8a], the append-only 변천사 history). The table is introduced by
   [32.reconsolidation-rules](32.reconsolidation-rules.md); the "created/original" baseline entry is synthesized at read
   (§CC5), not backfilled here.

**Decided (2026-07):** the anchor resets, the `EffectiveStrength` bump, the gist-timer reset, **and the neighbor
forgetting ± ([R5])** are all recall effects (not reconsolidation-only), so `Reinforce` applies them on **both** branches
(step 5) _before_ the branch; `Reconsolidate` adds only the text/`seed`/remaining-stage/provenance deltas. This keeps
[R4] ("same recall → strengthen only") exactly true — a no-error recall still brightens, grows, resets the timer, and
nudges neighbors, it simply does not reshape or rewrite — and it matches the whole-diary jump ([D3]/[47]), which
`Reinforce`s (getting [F5] recovery + [R5] neighbor gain) with no reconsolidation.

### The consumer-owned ports (declared in `internal/memory`, §2.4)

Domain-shaped inputs/outputs only — no proto, no sqlc, no SDK types cross these boundaries:

- **`SpendGate.CheckAndSpend(ctx, scope, spend SpendIntent) → error`** — check-and-spend for a recall/view action.
  `SpendIntent` names the _action (recall / gist-view) + its target memory (+ stage for gist-view)_ — the **depth
  signal, never a price**. **Declared once** in `internal/memory` (this unit is the primary consumer;
  [34.view-semantic-usecase](34.view-semantic-usecase.md) references the _same_ port). **Epic C default: allow-all
  no-op.** Epic F concrete ([44.earn-spend-usecase](44.earn-spend-usecase.md)): balance check + deduct
  (basic→additional order), pricing by the cost curve (§CC2/§CC3). Denial returns the canonical `insufficient-twinkle`
  error.
- **`PredictionError.Differs(ctx, currentText, rewriteText) → (bool, error)`** — the prediction-error LLM semantic
  compare ([R6]). **Declared by [32.reconsolidation-rules](32.reconsolidation-rules.md)** (canonical name/contract);
  this unit wires the concrete AI adapter (keyless mock deterministic) at `cmd/api` and calls it. Bounded by the AI
  cost caps ([22.ai-worker-pipeline](22.ai-worker-pipeline.md)).
- **`MemoryProvenanceStore.Append(ctx, scope, entry) → error`** — append-only 변천사 write ([R8a]); the table is
  [32.reconsolidation-rules](32.reconsolidation-rules.md)'s.

### RPC contract (`memory.v1.MemoryService`, Connect, unary — §2.7)

proto is a DTO layer mapped at `memory/rpc`; handlers are thin (map proto↔domain + call the use-case, §2.9#7). Two new
RPCs:

```proto
// 회고하기 — remember an episodic star; reinforce, and reconsolidate on prediction error [R1][R6].
// NOT NO_SIDE_EFFECTS: it spends Twinkle, advances the clock (sync), and may rewrite the star.
rpc Recall(RecallRequest) returns (RecallResponse);
// 이 일기로 태어난 별 보기 — no-rewrite whole-diary recall [D3].
// NOT NO_SIDE_EFFECTS: it spends Twinkle, advances the clock (sync), and reinforces every affected star.
rpc RecallDiaryStars(RecallDiaryStarsRequest) returns (RecallDiaryStarsResponse);

message RecallRequest  { string memory_id = 1; string rewrite_text = 2; }   // the star + what the user wrote back
message RecallResponse {
  bool reconsolidated = 1;                    // false = reinforce-only [R4]; true = prediction error → reshape [R6]
  string current_text = 2;                    // unchanged on reinforce-only; new on reconsolidation
  int64  seed = 3;                            // unchanged on reinforce-only; new on reconsolidation [V5]
  int32  recall_count = 4;                    // incremented [R3]
  float  effective_strength = 5;              // bumped size [R3][V3]
  string previous_universe_time = 6;          // clock before the sync (for the acceleration replay [30/31])
  string universe_time = 7;                   // clock after the sync (= today)
}

message RecallDiaryStarsRequest  { string diary_id = 1; }
message RecallDiaryStarsResponse {
  string diary_id = 1;
  repeated string episodic_memory_ids = 2;    // still-live memories reinforced by this whole-diary recall
  string previous_universe_time = 3;          // clock before the sync (for the acceleration replay [30/31])
  string universe_time = 4;                   // clock after the sync (= today)
}
```

The `rewrite_text` is the _only_ content the client sends; there is **no** `seed`, `brightness`, `strength`, `decay`,
`price`, or `diary` field the client can set — the reshape/bump/anchors/price are all server-derived (§2.9#8, and the
same structural-defense discipline as [20.encode-usecase](20.encode-usecase.md)'s schema-forced encode). `Recall` is
**not** GET/`NO_SIDE_EFFECTS`: it spends, syncs, and may write.

### Golden-parity stance

`Recall` itself is **not** golden-parity — it is a server-authoritative orchestration (it spends Twinkle, calls an LLM
compare, advances the clock, and writes rows; none of that runs on the client). Its **read-time consequences are**: the
`EffectiveStrength(base_strength, recall_count)` size ([V3]) is the memory-level golden-parity function shared by
`internal/memory` (Go) ↔ `packages/memory-logic` (TS) ([18.synapse-plasticity](18.synapse-plasticity.md)); this unit
merely activates its recall-accumulation term by incrementing `recall_count`. So the _number that grows_ is pinned by
plan 18's golden-parity harness; the _act that increments it_ is server-only. Matching plan 18's precedent, this plan
states that stance explicitly: recall **writes anchors**, and the anchors feed the golden-parity read-time math.

### Thin handler / dependency rule

The `memory/rpc` handlers map request DTOs ↔ domain and call the use-cases; the sync ordering, the spend, the
prediction-error branch, the anchor reset, the reinforce/reconsolidate policy, the whole-diary affected-memory
selection, and the provenance append all live in the use-case/domain (§2.9#7). No proto/sqlc type reaches the use-case
or the pure domain (§2.9#4); `SpendGate`, `PredictionError`, `MemoryProvenanceStore`, and the repositories are
consumer-owned interfaces (§2.9#6).

## Acceptance Criteria

1. `Recall` and `RecallDiaryStars` **compose `SyncToToday`** ([30]) so recall happens on a clock synced to today, and
   return the advance interval `{previous_universe_time, universe_time}`; the **sync-consent gate is the UI's** ([36]/[47])
   — this use-case exposes the capability, not the modal. ([R1a][T2] case 2)
2. `Recall` and `RecallDiaryStars` **spend Twinkle** through the consumer-owned `SpendGate` port and **never hardcode a
   price**; Epic C binds an **allow-all no-op** (the loop works without cost), Epic F binds the real gate; a denied spend
   aborts with a canonical error and resets nothing. ([G1][D3], §CC2)
3. On **every** recall (either branch), `Reinforce` resets the anchors — `last_recalled_universe_time = universe_time`,
   `recall_count += 1`, `semanticize_timer_reset_at = universe_time` — batch-LTPs the co-activated synapses,
   **bumps `EffectiveStrength`** (size), and applies the neighbor forgetting ± ([R5], neighbors only). ([R2][R3][R5][C6a] reset)
4. The **visible** brightness/decay/gist recovery is **not** computed here — `Recall` writes anchors, and Epics D/E's
   read-time functions render the effect ([F5][R2]) against those anchors. ([R2][F5], §CC1)
5. `Reinforce` is an **idempotent batch LTP** over the memory's co-activated synapses (calls `Potentiate` once per edge,
   saturating toward the cap), **distinct** from the `Potentiate` primitive, and touches **no** memory↔memory edge.
   ([R3][I4][I6])
6. **Prediction error is an LLM semantic compare** of the rewrite vs. `current_text` (content change, not wording/order),
   held behind the consumer-owned `PredictionError` port. ([R6])
7. **No prediction error → reinforce only:** `current_text` and `seed` are **unchanged**; brightness/strength/timer still
   recover (via step 3). ([R4][I8])
8. **Prediction error → `Reconsolidate`:** rewrite `current_text`, **change `seed`** (new shape [V5]), regenerate only
   the **remaining, not-yet-created** stage texts on the new basis while **keeping** already-risen gist stars/stages,
   and append a `reconsolidated/user` `memory_provenance` row (the neighbor ± + anchors + strength already ran in
   `Reinforce`, step 5). ([R6][C7][R8a])
9. The **`Diary` is never written** by `Recall`/`Reinforce`/`Reconsolidate`; only representation (`current_text`, `seed`,
   stage texts, anchors, synapse strengths, provenance) changes. ([I2][R7])
10. The neighbor forgetting ± is applied on **every** recall (both branches, via `Reinforce`) to **neighbor** episodic
    memories only (by shared _semantic_-neuron count), never to the recalled star (which recovers wholly [F5]); the ±
    math/threshold/column are [32.reconsolidation-rules](32.reconsolidation-rules.md)'s, invoked here — so the
    whole-diary jump ([D3]/[47]) gets [R5] without reconsolidation. ([R5])
11. `RecallDiaryStars` applies `Reinforce` to every still-live memory born from the diary in one transaction with sync +
    spend; it does **not** call `PredictionError` or `Reconsolidate`, writes no `current_text`, and changes no `seed`.
    ([D3][R4][R6][I8])
12. Reconsolidation is on **episodic** stars only; gist-star viewing ([R8]) is [34.view-semantic-usecase](34.view-semantic-usecase.md)
    and is not a recall path. ([R1])
13. The `Recall` / `RecallDiaryStars` RPCs are **not** `NO_SIDE_EFFECTS`; `Recall` sends only `memory_id` + `rewrite_text`
    (no seed, strength, decay, price, or diary field), `RecallDiaryStars` sends only `diary_id`, so reshape/bump/anchors/
    price/affected memory ids are all server-derived. (§2.7/§2.9#8)
14. Handlers are **thin** (proto↔domain map + call); sync/spend/compare/branch/anchor/reinforce/reconsolidate/provenance
    policy lives in the use-case/domain; no proto/sqlc leaks into the use-case or pure domain; `SpendGate`,
    `PredictionError`, `MemoryProvenanceStore`, and the repositories are consumer-owned interfaces.
    (§2.4/§2.7/§2.9#4/#6/#7)
15. Every read, spend, compare, write, and provenance append is **scoped to the authenticated user** and
    `pnpm lint:persistence` passes (§4).
