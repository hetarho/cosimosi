# 03. 데이터 스키마 — pgvector + goose 마이그레이션 + sqlc 준비

> 엔그램 우주의 영속 계층(3겹: 불변 원본·가변 별·가중치 시냅스)을 Postgres+pgvector 스키마에 박는다. **이 파일이 04/05/11/12가 따르는 단일 권위 스키마다.** | 범위: MVP | 선행: — | 차단: 04, 05, 11, 12

## 목적 (Why)
이 스펙은 엔그램 우주의 영속 계층 — **원본 일기 불변(records) / 별 가변(memories) / 임베딩(embeddings) / 시냅스 가중치 그래프(memory_links) / 비동기 큐(jobs) / 회상 강화 멱등(processed_batches)** 를 pgvector 포함 Postgres + goose 마이그레이션으로 정의하고, sqlc 코드젠이 돌 수 있는 상태(스키마 입력)를 만든다. **이 DDL이 단일 권위**이며 04(memory-api)·05(worker)·11(recall)·12(decay)는 모두 이 스키마에 맞춰 쿼리한다(mood/intensity/entry_date/body가 필요하면 records JOIN).

## 범위
- **포함:**
  - `docker-compose.yml`의 postgres 이미지를 `pgvector/pgvector:pg16`으로 교체(로컬 docker).
  - **goose** 도입(컨테이너 initdb 마운트 방식 → 버전 관리형 up/down 마이그레이션으로 전환).
  - 마이그레이션: `CREATE EXTENSION vector`; **records**(불변 원본) / **memories**(별) / **embeddings**(`vector(1536)` + HNSW, user_id) / **memory_links**(시냅스, user_id) / **jobs**(큐, next_run_at) / **processed_batches**(회상 강화 멱등) — 6개 신규 테이블 + 인덱스.
  - **goose 실행 수단**: 로컬 바이너리 없이 Docker(`ghcr.io/kukymbr/goose-docker:3.27.1`)로 up/down. `docker compose up`만으로는 스키마 자동 생성 안 됨 → goose up이 별도 필수 단계.
  - `sqlc.yaml`(pgx/v5) `schema`를 마이그레이션 디렉터리가 아닌 순수 DDL 스냅샷 `internal/db/schema.sql`로 분리(goose Down DROP 충돌 회피). `sqlc generate`가 빈 쿼리로도 성공하는지 확인.
  - pgvector 타입 등록: `platform/postgres/postgres.go` after-connect 훅에 `pgxvec.RegisterTypes`.
- **비목표(Out of scope):**
  - `evolution_history` 테이블 — v1 #20 reconsolidation 담당.
  - 쿼리(`internal/db/queries/*.sql`)·리포지토리 구현·도메인 타입 — **04(memory-api)+** 담당. 이 스펙은 스키마/마이그레이션/타입 등록까지만.
  - Supabase 클라우드 적용·리전 코로케이션 — Architecture §7 인프라 단계(로컬 docker만).
  - `users` 테이블 실체화 — MVP 단일 사용자라 `user_id`는 컬럼만 두고 FK 없는 `TEXT`로 보류(인증은 01/v1).

## 설계 요점
> **이 파일이 단일 권위 스키마다.** 04(memory-api)·05(worker)·11(recall)·12(decay)는 모두 이 DDL에 맞춰 쿼리를 작성한다. `mood`/`intensity`/`entry_date`/`body`는 **records에만** 존재하므로, 별 행에서 이 값이 필요한 쿼리(GetUniverse/ListDormant/Recall)는 `memories m JOIN records r ON r.id = m.record_id` 경로로 읽는다(memories에 중복 컬럼을 두지 않는다 — 헌법1). 참조 방향은 **memories.record_id → records.id 단방향**이며 records에는 memory_id가 없다(별이 원본을 가리킴, 그 반대 아님).

**3겹 분리(원칙 1·2)**: 사용자가 쓴 텍스트(`records`)는 별/연결과 **물리적으로 다른 테이블**이며 수정·삭제 쿼리를 만들지 않는다. 별(`memories`)은 가변(`visual_spec`, `last_recalled_at` 진화)이고 `record_id`로 원본을 참조하되 원본을 건드리지 않는다.

**user_id 격리**: `embeddings`·`memory_links`에도 `user_id`를 두어 KNN/그래프/universe/recall/dormant 쿼리가 `WHERE user_id = $ctx`로 스코프된다(MVP는 WHERE + 인터셉터 user_id, RLS는 v1).

```sql
CREATE EXTENSION IF NOT EXISTS vector;

-- 불변 원본(원칙 1): UPDATE/DELETE 쿼리 절대 생성 금지
CREATE TABLE records (
    id              TEXT PRIMARY KEY,            -- 서버 생성 nanoid/uuid
    user_id         TEXT NOT NULL,               -- = Supabase auth uid (단일 사용자 MVP: FK 없음)
    body            TEXT NOT NULL,               -- 일기 원본(불변)
    entry_date      DATE NOT NULL,
    mood            TEXT,                        -- nullable; enum 이름 7종 중 하나 또는 null
    intensity       REAL,                        -- nullable; 0~1 비주얼용
    idempotency_key TEXT,                        -- nullable; RecordMemory 멱등키
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
-- 멱등성: (user_id, idempotency_key) 부분 UNIQUE — key가 NULL이면 제약 없음
CREATE UNIQUE INDEX records_idem_idx ON records (user_id, idempotency_key)
    WHERE idempotency_key IS NOT NULL;

-- 별(가변): 원본과 분리, record_id로 참조. mood/intensity/entry_date는 records에만(JOIN으로 읽음)
CREATE TABLE memories (
    id              TEXT PRIMARY KEY,
    user_id         TEXT NOT NULL,
    record_id       TEXT NOT NULL REFERENCES records(id),
    visual_spec     JSONB,                                -- nullable; MVP 미사용(FE가 memory_id로 seed 결정론 파생)
    last_recalled_at TIMESTAMPTZ NOT NULL DEFAULT now(),  -- 활성도 감쇠 기준(§6)
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX memories_user_idx ON memories (user_id);

-- 임베딩: ⚠️ vector(1536) 비수식 선언(public.vector 금지 — sqlc #3548)
CREATE TABLE embeddings (
    memory_id TEXT PRIMARY KEY REFERENCES memories(id),
    user_id   TEXT NOT NULL,            -- 격리: 모든 KNN 쿼리에 WHERE user_id 강제
    embedding vector(1536) NOT NULL,
    model     TEXT NOT NULL              -- 어댑터/차원 기록
);
CREATE INDEX embeddings_hnsw_idx ON embeddings USING hnsw (embedding vector_cosine_ops);
CREATE INDEX embeddings_user_idx ON embeddings (user_id);

-- 시냅스(가중치 그래프): 무방향 1행 정규화 a_id < b_id, 삭제 안 함(원칙 2)
CREATE TABLE memory_links (
    a_id                TEXT NOT NULL REFERENCES memories(id),
    b_id                TEXT NOT NULL REFERENCES memories(id),
    user_id             TEXT NOT NULL,           -- 격리: a_id/b_id 둘 다 같은 user 소속
    weight              REAL NOT NULL,           -- 0~1
    link_type           TEXT NOT NULL,           -- semantic/temporal/entity/co_recall
    co_activation_count INT NOT NULL DEFAULT 0,
    last_activated_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (a_id, b_id),
    CONSTRAINT memory_links_order_chk CHECK (a_id < b_id)
);
CREATE INDEX memory_links_b_idx ON memory_links (b_id);  -- 역방향 이웃 조회
CREATE INDEX memory_links_user_idx ON memory_links (user_id);

-- 비동기 큐(§4.6): FOR UPDATE SKIP LOCKED claim용
CREATE TABLE jobs (
    id          TEXT PRIMARY KEY,
    memory_id   TEXT NOT NULL REFERENCES memories(id),
    kind        TEXT NOT NULL,                    -- embed/extract
    status      TEXT NOT NULL DEFAULT 'pending',  -- pending/running/done/failed
    attempts    INT NOT NULL DEFAULT 0,
    error       TEXT NOT NULL DEFAULT '',
    next_run_at TIMESTAMPTZ NOT NULL DEFAULT now(), -- 지수 백오프 예약 시각(05: now + base*2^attempts)
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX jobs_claim_idx ON jobs (status, next_run_at);  -- claim: status + next_run_at<=now()

-- 회상 강화 배치 멱등성: 이미 본 batch_id면 ReinforceLinks skip(재전송 이중 가산 방지)
CREATE TABLE processed_batches (
    batch_id   TEXT PRIMARY KEY,
    user_id    TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**goose 명명**: `NNNNN_name.sql` 단일 파일에 `-- +goose Up` / `-- +goose Down` 섹션(또는 timestamp 형식). HNSW 인덱스 생성문은 `-- +goose StatementBegin/End`로 감싸 안전하게. Down은 역순 `DROP TABLE`.

**goose 로컬 실행**: 로컬 goose 바이너리 없이 **Docker 일회성 컨테이너**로 마이그레이션을 적용한다(이미지 `ghcr.io/kukymbr/goose-docker:3.27.1` — §goose 이미지 정정 참고). compose에 initdb 마운트가 없으므로 `docker compose up`만으로는 스키마가 자동 생성되지 않는다 → **goose up이 별도 필수 단계**다. 표준 호출(리포 루트, postgres가 compose 네트워크에서 실행 중):
```sh
docker run --rm --network cosimosi_default \
  -v "$PWD/backend/internal/db/migrations:/migrations" \
  -e GOOSE_DRIVER=postgres \
  -e GOOSE_DBSTRING="postgres://cosimosi:cosimosi@postgres:5432/cosimosi?sslmode=disable" \
  -e GOOSE_COMMAND=up \
  ghcr.io/kukymbr/goose-docker:3.27.1   # down 시 GOOSE_COMMAND=down
```
(또는 docker-compose의 일회성 `goose` 서비스로 동일 명령을 정의 — 14.deploy-cicd와 동일 이미지/패턴.)

**sqlc ↔ goose 주석 충돌 회피**: sqlc.yaml의 `schema`가 마이그레이션 디렉터리를 가리키면, 같은 파일의 `-- +goose Down` 섹션 `DROP TABLE`을 sqlc가 누적 DDL로 해석해 객체가 드롭된 것으로 오인할 수 있다(`-- +goose Up/Down` 라인 주석 자체는 `--`라 무해하나 Down의 DROP문이 문제). **결정: sqlc의 `schema` 입력을 goose 마이그레이션과 분리한다.** Up 섹션의 순수 DDL만 담은 `backend/internal/db/schema.sql` 스냅샷 파일을 두고 `sqlc.yaml`의 `schema`를 이 파일로 가리킨다(goose 마이그레이션은 적용 전용, sqlc는 schema.sql만 파싱). 마이그레이션 추가 시 schema.sql도 동기화한다(단일 출처는 마이그레이션, schema.sql은 그 Up DDL의 평탄화 스냅샷).

**타입 등록**(Architecture §4.5): pgx 풀 생성 시 after-connect 훅에서 `pgxvec.RegisterTypes(conn)` 호출 → `vector` 컬럼이 `pgvector.Vector`로 디코드. sqlc는 v1.23+가 `vector(1536)`을 `pgvector.Vector`로 자동 생성(오버라이드 불필요).

**⚠️ Windows**: go/sqlc/goose는 호스트 .exe 직접 실행 차단 → Docker/WSL에서. goose는 마이그레이션 도구라 docker compose의 일회성 컨테이너 또는 `docker run`으로 실행.

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN `docker compose up -d postgres` 후 `SELECT extversion FROM pg_extension WHERE extname='vector'` 실행 THE SYSTEM SHALL 1행(vector 확장 설치됨) 반환.
1.2 WHEN goose up 실행 THE SYSTEM SHALL `records, memories, embeddings, memory_links, jobs, processed_batches` 6개 테이블을 생성한다.
1.3 GIVEN goose up 완료 WHEN `\d embeddings` 확인 THE SYSTEM SHALL `embedding` 컬럼 타입이 `vector(1536)`이고 `embeddings_hnsw_idx`가 `hnsw ... vector_cosine_ops`임을 보인다.
1.4 WHEN `memory_links`에 `a_id >= b_id`인 행 INSERT 시도 THE SYSTEM SHALL `memory_links_order_chk` 위반으로 거부한다.
1.5 WHEN 같은 `(user_id, idempotency_key)`(둘 다 비-NULL)로 `records`에 2번째 INSERT 시도 THE SYSTEM SHALL `records_idem_idx` UNIQUE 위반으로 거부한다(멱등). 단 `idempotency_key IS NULL`인 행은 여러 개 허용된다.
1.6 GIVEN goose up 완료 WHEN `\d embeddings`/`\d memory_links` 확인 THE SYSTEM SHALL `user_id TEXT NOT NULL` 컬럼과 `embeddings_user_idx`/`memory_links_user_idx` 인덱스가 존재함을 보인다(격리).
1.7 GIVEN goose up 완료 WHEN `\d jobs` 확인 THE SYSTEM SHALL `next_run_at TIMESTAMPTZ NOT NULL DEFAULT now()` 컬럼과 `jobs_claim_idx (status, next_run_at)` 인덱스가 존재함을 보인다(05 백오프/claim 의존).
2.1 WHEN goose down(1단계) 실행 THE SYSTEM SHALL 6개 테이블과 `vector` 확장을 역순으로 제거해 클린 상태로 되돌린다(up→down→up 반복 일관).
2.2 IF 마이그레이션 중 임의 단계 실패 THEN THE SYSTEM SHALL goose 트랜잭션 경계로 부분 적용 없이 롤백한다(HNSW문은 StatementBegin/End로 보호).
3.1 WHEN Docker/WSL에서 `sqlc generate` 실행(쿼리 0개 상태, schema 입력=`internal/db/schema.sql`) THE SYSTEM SHALL goose 주석/Down DROP문 충돌 없이 스키마를 파싱하고 에러 없이 종료한다(스키마 DDL이 유효함의 증명).
3.2 THE SYSTEM SHALL `records`에 대한 UPDATE/DELETE 마이그레이션·제약을 두지 않는다(원칙 1 — 불변).
3.3 THE SYSTEM SHALL `vector(1536)`을 비수식으로 선언한다(`public.vector(...)` 미사용 — sqlc #3548 회피).
4.1 WHEN 앱이 pgx 풀로 연결 THE SYSTEM SHALL after-connect 훅에서 pgvector 타입을 등록해 `vector` 컬럼 read/write가 가능하다.

## 완료 정의 (DoD)
- `docker compose up -d postgres` 후 `docker compose exec postgres psql -U cosimosi -d cosimosi -c "SELECT extname FROM pg_extension WHERE extname='vector';"` → `vector` 1행.
- goose up은 `docker compose up` 후 **별도 필수 단계**(initdb 자동 적용 없음). 표준 명령: `docker run --rm --network cosimosi_default -v "$PWD/backend/internal/db/migrations:/migrations" -e GOOSE_DRIVER=postgres -e GOOSE_DBSTRING="postgres://cosimosi:cosimosi@postgres:5432/cosimosi?sslmode=disable" -e GOOSE_COMMAND=up ghcr.io/kukymbr/goose-docker:3.27.1`.
- goose up 후 `psql -c "\d"`에 `records, memories, embeddings, memory_links, jobs, processed_batches` 6개 테이블이 존재한다.
- `psql -c "\d embeddings"` → `embedding | vector(1536)` + `embeddings_hnsw_idx` (`hnsw, vector_cosine_ops`) + `user_id` 컬럼·`embeddings_user_idx`.
- `psql -c "\d jobs"` → `next_run_at` 컬럼 존재 + `jobs_claim_idx (status, next_run_at)`.
- `psql -c "INSERT INTO memory_links (a_id,b_id,user_id,weight,link_type) VALUES ('z','a','u1',0.5,'semantic');"` → CHECK 위반 에러(a_id<b_id 보장).
- 같은 `(user_id, idempotency_key)`(비-NULL)로 `records` 2회 INSERT 시 2번째가 `records_idem_idx` UNIQUE 위반(멱등). `idempotency_key IS NULL` 다중 행은 허용.
- goose down 후 위 6테이블·확장 모두 사라지고, 재 up 시 동일 스키마 재현(멱등).
- `sqlc/sqlc:latest generate`(Docker, schema=`internal/db/schema.sql`) 종료 코드 0(goose 주석/Down DROP 충돌 없이 스키마 파싱 성공).
- `records`에 UPDATE/DELETE를 수행하는 마이그레이션·제약이 존재하지 않음(원칙 1 검증, 코드 리뷰).

## 구현 노트 (03 진행 중 발견 — 스펙 가정 보정)

- **goose 이미지 정정**: `ghcr.io/pressly/goose:3`은 **존재하지 않는다**(pressly는 공식 Docker 이미지를 배포하지 않고 바이너리만 배포 — pull 시 `denied`). 유지되는 래퍼 이미지 **`ghcr.io/kukymbr/goose-docker:3.27.1`** 로 교체했다. 이 이미지는 `/migrations` 마운트 + `GOOSE_DRIVER`/`GOOSE_DBSTRING`/`GOOSE_COMMAND`(기본 `up`) env 방식이라 `scripts/db.mjs`의 goose 헬퍼를 그에 맞게 수정함(`-dir /m <cmd>` CLI → env 방식). `pnpm db:migrate`/`db:status`/`db:reset` 모두 동작 확인. **14.deploy-cicd도 동일 이미지로 맞출 것**(이전엔 "pressly:3 동일 이미지/패턴" 명시).
- **sqlc 빈 쿼리 종료코드**: 현재 sqlc(v1.30대)는 쿼리 0개일 때 `error parsing queries: no queries contained in paths`로 **종료 코드 1**을 낸다(`queries` 키를 제거해도 동일). 즉 3.1의 "빈 쿼리로도 exit 0" 가정은 이 버전에서 성립하지 않는다. 단, 이는 스키마 파싱 실패가 아니라(goose 주석/Down DROP 충돌은 없음 — schema 단계는 통과) **쿼리 부재** 때문이며, schema.sql의 유효성은 ① 동일 DDL의 goose up이 실DB에 6테이블+확장 생성 성공 ② sqlc가 schema 단계를 통과(에러가 "no queries")로 이중 검증됨. `sqlc generate` exit 0은 **04(memory-api)가 첫 쿼리를 추가하면** 자동 해소된다. 그때까지 `pnpm gen:sql`은 "no queries" 에러를 낸다(스키마는 정상).
