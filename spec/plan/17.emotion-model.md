# 17. emotion model

> Emotion model — the `Emotion` value object on `EpisodicMemory`, the 13-mood (valence×arousal) table, the single
> swappable `mood → color` palette seam, and the arousal-only initial-strength modulation. **Backend owns the domain
> VO + mood table + arousal math (Go); a pure `packages/emotion` package owns the FE mirror + the palette function.**
> Scope: FS | Status: planning
>
> Layer D (domain). This unit is *headless*: it defines the emotion domain and its projection seam, not any screen.
> Rendering entities (`star`, `nebula`) consume the palette seam later; the writing flow assigns/edits emotion later.

## Purpose

Every episodic memory carries exactly one primary emotion, and that emotion is the **only** source of a star's color
([M1]–[M3]). The product's honest promise — "the universe's color is the emotion you re-read, not your average
emotion" — rests on emotion being a single, well-typed domain fact that renders through **one** color seam and never
leaks into position or connection ([I3], [M7]). Before the LLM can assign emotion at encode ([E]/plan 20), before the
writing flow can let a user edit it ([W2]/plan 27), and before stars and the nebula can be colored (plans 24, 26),
there must be a canonical `Emotion` value object, a fixed 13-mood vocabulary laid out on the (valence × arousal)
plane, a **single swappable** `mood → color` palette function, and a pure arousal → initial-strength modulation.

This unit defines all four. It is a domain + shared-package unit: the Go domain owns the `Emotion` VO, the mood ENUM,
the quadrant structure, and the arousal math; `packages/emotion` owns the FE mirror of the VO/enum and the palette
function; the arousal → strength math is mirrored TS↔Go under golden-parity fixtures. It ships no screen and no LLM
call — it provides the seams those units plug into.

## Scope / Non-goals

**In scope**
- **`Emotion` value object** on `EpisodicMemory` — `{ mood, valence, arousal, intensity }`, one primary emotion per
  memory ([M1]). Lives in the Go core domain context `internal/memory` (a VO, not an aggregate root; it is a field of
  the `EpisodicMemory` root). A pure FE mirror lives in `packages/emotion`.
- **The 13-mood vocabulary** as a domain ENUM: `JOY · CALM · SAD · ANGER · FEAR · LOVE · NEUTRAL · EXCITEMENT ·
  GRATITUDE · RELIEF · STRESS · TIRED · EMPTINESS` ([M2]).
- **The quadrant structure** (four V±×A± quadrants + neutral, Russell 1980) as domain **code** — the mapping of each
  mood to its quadrant, plus the helper that reads a mood's `(valence, arousal)` coordinate.
- **Per-mood scalars in `values.yaml`** — the concrete `(valence, arousal)` numbers for each of the 13 moods live in
  `emotion.mood_valence` and `emotion.mood_arousal` (two one-level number-maps; this is a shape `gen:values` supports).
- **The `mood → color` palette** — a **single entry-point pure function** in `packages/emotion` that maps a mood to a
  color. The color table is **code content** (like theme CSS / a shader graph), not `values.yaml`. It is a *swappable*
  palette: exactly one default palette ships, behind one seam that a later customization unit ([P1]/palette-customization)
  can swap ([M6]). Rendering code (`star`, `nebula`) is palette-agnostic and imports only this seam.
- **A presentation-only guard** — the palette may re-map which color a mood shows, but it **cannot** override a
  memory's stored emotion, cannot decouple "color = emotion", and cannot touch position/strength ([I11], [P2]).
- **Arousal → initial strength modulation** ([E8a]) — a pure function `arousal (0..1) → base_strength` over the range
  `emotion.arousal_strength_min .. emotion.arousal_strength_max`. Arousal is read from the mood's arousal coordinate
  (no extra data). **Only arousal feeds strength; valence stays color-only** ([I3], [M7]). The result seeds
  `EpisodicMemory.base_strength` at creation.
- **Golden-parity** — the arousal → strength math is a pure function mirrored TS↔Go and pinned by shared fixtures.

**Non-goals**
- **No LLM emotion assignment.** `Extractor` assigning a mood at encode is plan 20 (`encode-usecase`); this unit only
  defines the VO/enum it produces.
- **No writing-flow UI.** The user editing a memory's mood in the split-preview is plan 27 (`writing-flow-ui`); this
  unit only defines the editable value.
- **No star/nebula rendering.** Coloring a star ([V4], plan 24) and blending the strength-weighted nebula ([M4], plan
  26) consume this unit's palette seam; they are not built here.
- **No forgetting-speed arousal modulation ([F6]).** That is Epic D. This unit only notes the seam: the *same* arousal
  coordinate that feeds initial strength will later feed decay-speed modulation.
- **No universe-color aggregation ([M4]/[M5]).** The strength-weighted blend across many emotions is the nebula unit's
  job (plan 26); this unit provides only the per-mood color and the honest single-source-of-color rule.
- **No palette-editing UI, no runtime palette switching.** The swap *seam* exists; the customization UI ([P1], Epic I)
  is a separate unit. Only one default palette ships now.
- **No DB schema.** The `episodic_memories` columns that store the VO (`mood`, `valence`, `arousal`, `intensity`,
  `base_strength`) are owned by plan 16 (`memory-aggregate-schema`); this unit specifies their *meaning*, not the DDL.
- No hardcoded tuning scalars in FE/BE code (mood coordinates and strength bounds come from `values.yaml`).

## Design

### The `Emotion` value object (Go domain, FE mirror)

`Emotion` is a **value object** in the `internal/memory` core domain context — a field of the `EpisodicMemory`
aggregate root, not a root itself and not persisted independently. Its shape:

| Field | Type | Meaning |
|---|---|---|
| `mood` | `Mood` (13-enum) | the primary mood label ([M1], [M2]) — the categorical emotion of this memory |
| `valence` | real, −1..1 | the mood's valence coordinate (V axis; negative = unpleasant, positive = pleasant) |
| `arousal` | real, 0..1 | the mood's arousal coordinate (A axis; low = calm, high = activated) |
| `intensity` | real, 0..1 | how strongly this memory is felt (a per-memory scalar, default `emotion.default_intensity`) |

- **One primary emotion per memory** ([M1]). A diary that splits into several episodic memories yields several
  `Emotion` values — one per memory — each independent.
- `valence`/`arousal` are **the mood's coordinates**, resolved from the mood via the coordinate helper (below). They
  are stored as columns on `episodic_memories` (plan 16) so a read does not need to re-resolve the table, and so a
  future palette/coordinate revision does not retroactively move already-launched stars. Storing them as denormalized
  copies of the mood coordinate is a deliberate write-time snapshot.
- `intensity` is a separate per-memory scalar (distinct from arousal): it defaults to `emotion.default_intensity` at
  creation and is reserved for a **later, not-yet-assigned** weighting use. (In Epic A the nebula weights its color
  bleed by memory-level `EffectiveStrength`, **not** `intensity` — see [26.nebula-color-field](26.nebula-color-field.md).)
  It is **not** an emotion-axis input to position/connection.
- **FE mirror** — `packages/emotion` exports a pure TS `Emotion` type + `Mood` union that mirror the Go types
  (canonical names, no `three`, no DOM). The FE domain-mirror entity (`entities/episodic-memory`, plan 23) carries the
  `Emotion` on each memory; `packages/emotion` owns the emotion-specific type + palette so both apps and the mirror
  share one source.

### The 13-mood vocabulary + quadrant structure (domain code)

The 13 moods are a fixed domain **ENUM** (Go `Mood` constants; TS string-union mirror). Their layout on the
(valence × arousal) plane is the four Russell-1980 quadrants plus neutral — this **structure** is code (a table
mapping mood → quadrant), while the concrete `(valence, arousal)` **scalars** live in `values.yaml` (next subsection):

| Quadrant | Sign | Moods |
|---|---|---|
| High-arousal · positive | V+ A+ | `JOY` · `EXCITEMENT` · `LOVE` |
| Low-arousal · positive | V+ A− | `CALM` · `GRATITUDE` · `RELIEF` |
| High-arousal · negative | V− A+ | `ANGER` · `FEAR` · `STRESS` |
| Low-arousal · negative | V− A− | `SAD` · `TIRED` · `EMPTINESS` |
| Neutral | — | `NEUTRAL` |

- The mood ENUM and this quadrant map are the single naming authority for emotion labels across Go, DB (the `mood`
  TEXT column's allowed values), proto (the mood field), and the FE mirror — one concept, one name in every layer (§1).
- A pure coordinate helper `moodCoordinate(mood) → (valence, arousal)` reads the two number-maps from generated config
  and returns the mood's coordinate. It is the *only* way emotion axes enter downstream math, so the axes cannot be
  smuggled in from anywhere else.
- The enum's cardinality (13) is derived from its content; it is not a declared count. Adding/removing a mood is a
  domain + `values.yaml` change (both the enum and the two number-maps must stay in sync — validated at `gen:values`).

### The `mood → color` palette (single entry-point, swappable — `packages/emotion`)

- **Single entry point.** Exactly one pure function — `moodColor(mood) → Color` — is the sole source of a mood's
  color ([M3], [M6]). No rendering slice, no shader binding, and no other package computes a mood's color; they all
  call this one function. This makes "color always comes from emotion" a structural fact, not a convention.
- **Color table = code content.** The palette (the 13 mood → color entries) is an **object/table in code**, exactly
  like a theme CSS table or a shader node graph — it is *array/table content*, so per plan 14's exclusion rule it
  stays in `packages/emotion`, **not** in `values.yaml`. (Only *scalars* go in `values.yaml`; a color table is content.)
- **Swappable, one seam.** The palette is a *named default* behind one seam (`resolvePalette()` / `usePalette()`-style,
  mirroring the skin seam of plan 14). Exactly one default palette ships now. The seam is shaped so the Epic I
  `palette-customization` unit ([P1]) can supply a different palette table **without changing any consumer** — the
  rendering entities import `moodColor` through the seam and never bind a concrete palette.
- **Palette-agnostic rendering.** `entities/star` (star color, [V4]/plan 24) and `entities/nebula` (per-mood color for
  the strength-weighted blend, [M4]/plan 26) import only `moodColor`; swapping the palette re-colors the whole universe
  through this one function with no rendering-code edits.
- **Presentation-only guard ([I11], [P2]).** The palette is a *projection* (§3.4), one-way. It may change which color a
  mood displays; it **cannot** (a) override a memory's stored `mood`/`valence`/`arousal`, (b) decouple color from
  emotion (a memory's color is always `moodColor(memory.mood)`), or (c) feed position, connection (synapse), or
  strength. The type shape enforces this: the seam consumes a `Mood` and returns a `Color`; it takes no memory
  identity and returns nothing a layout/strength function reads. Axis-consistency (V+ warm / V− cool, [P3]) is a
  recommendation the default palette follows; enforcing it on custom palettes is left open (see root DECISIONS.md).

### Arousal → initial strength modulation (pure fn, arousal-only, [E8a])

- **Function.** A pure `arousalToInitialStrength(arousal) → base_strength` maps `arousal ∈ 0..1` to a strength in
  `[emotion.arousal_strength_min, emotion.arousal_strength_max]`. Higher arousal → larger initial star ([E8a],
  amygdala consolidation enhancement). The mapping shape (e.g. linear interpolation across the bounds) is a formula and
  stays in code; only the two bound scalars live in `values.yaml`. Exact non-linearity is left open (see root
  DECISIONS.md) — the default is a simple linear map between the bounds.
- **Arousal-only, no extra data.** The `arousal` fed in is the mood's arousal coordinate from `moodCoordinate` — the
  13-mood table already carries it, so no additional per-memory arousal field is needed ([E8a]).
- **Valence never touches strength.** Only `arousal` is an input. `valence` is *never* passed to this function or to
  any layout/connection function — it is color-only ([I3], [M7]). This is enforced by the function signature (it takes
  a single arousal scalar, not an `Emotion`).
- **Where it runs.** `Encode` (plan 20) calls this at memory creation to seed `EpisodicMemory.base_strength`
  (persisted by plan 16). `base_strength` is the arousal-set floor; later recall accumulation ([R3]) and read-time
  `EffectiveStrength` (Epic D) build on it — those are other units' concerns, noted as seams only.
- **Forgetting-speed seam ([F6]).** Epic D will modulate decay speed by the *same* arousal coordinate (high arousal →
  slower forgetting). This unit does not implement it; it records that the arousal coordinate is the shared input, so
  Epic D plugs in without re-deriving arousal.

### Golden-parity (TS ↔ Go)

- `arousalToInitialStrength` is a pure function with a Go implementation (`internal/memory`) and a TS mirror. The TS
  mirror lives with the shared domain math — **left open (see root DECISIONS.md)** whether it sits in `packages/emotion`
  or `packages/memory-logic`; the default is `packages/emotion` (co-located with the `Emotion` type), promoted to
  `packages/memory-logic` only if the shared-math surface consolidates there.
- A single shared **fixture** (arousal inputs → expected `base_strength`, computed from the same `values.yaml` bounds)
  pins both implementations: the Go test and the TS test read the same fixture and must agree. This is determinism
  parity (identical math on both platforms), consistent with the project's read-time-derived-state parity rule.
- The `mood → color` palette is **not** golden-parity-pinned across TS↔Go: color is a presentation concern that lives
  only on the frontend (`packages/emotion`). The Go side owns emotion *facts* (mood, coordinates, strength math), not
  colors. This keeps the meaning layer (facts) server-authoritative and the projection layer (color) frontend-only.

### Data flow (who assigns, who consumes)

```
[assign]  Extractor (plan 20) → Mood on ProposedMemory
          user edits Mood in split preview (plan 27, [W2])
          Encode (plan 20): moodCoordinate(mood) → (valence, arousal)
                            arousalToInitialStrength(arousal) → base_strength
                            → Emotion VO + base_strength stored on EpisodicMemory (plan 16)
[consume] GetUniverse (plan 23) → Emotion facts on each memory (mood, valence, arousal, intensity)
          star (plan 24):  moodColor(memory.mood) → star color        (palette seam)
          nebula (plan 26): moodColor per memory → strength-weighted blend
```

Emotion is a **write-once-at-encode** fact for the color meaning; only the writing session may edit the mood ([W4]:
already-launched stars are not directly editable — their emotion changes only through natural processes, out of Epic A
scope). The palette re-maps *how* a mood looks at read time without ever changing *which* mood a memory has.

## Acceptance Criteria

1. `Emotion` is a value object on `EpisodicMemory` with fields `{ mood, valence, arousal, intensity }`, defined in the
   Go core domain context `internal/memory` and mirrored as a pure TS type in `packages/emotion`; it is a field of the
   aggregate root, never an independent aggregate ([M1]).
2. Each memory has exactly one primary `Emotion`; a diary splitting into N memories yields N independent emotions ([M1]).
3. The mood vocabulary is a fixed 13-value ENUM — `JOY, CALM, SAD, ANGER, FEAR, LOVE, NEUTRAL, EXCITEMENT, GRATITUDE,
   RELIEF, STRESS, TIRED, EMPTINESS` — used as the single naming authority across Go, DB, proto, and the FE mirror ([M2]).
4. The four V±×A± quadrants + neutral are represented as domain code (mood → quadrant map), matching the [M2] table;
   the per-mood `(valence, arousal)` scalars are read from `values.yaml`, not hardcoded.
5. A single pure function `moodColor(mood) → Color` in `packages/emotion` is the **only** source of a mood's color;
   no other slice, package, or shader binding computes mood color ([M3], [M6]).
6. The color table is code content in `packages/emotion` and is **not** present in `values.yaml`.
7. The palette is a named default behind one seam; a different palette can be supplied through the seam without editing
   any rendering consumer (`star`/`nebula` import only `moodColor`) ([M6]).
8. Star and nebula color derive solely from a memory's stored `mood` via `moodColor`; there is no code path that colors
   a body from anything other than its emotion ([M3], [V4], [I3]).
9. `arousalToInitialStrength(arousal) → base_strength` is a pure function bounded by `emotion.arousal_strength_min` and
   `emotion.arousal_strength_max`; it takes **only** arousal (the mood's arousal coordinate — no extra data) ([E8a]).
10. `valence` is never passed to any strength, position, or connection function; only color reads valence. No function
    signature accepts valence for a layout/strength/synapse computation ([I3], [M7]).
11. Swapping the palette cannot override a memory's stored `mood`/`valence`/`arousal`, cannot decouple "color =
    emotion", and cannot feed position/strength/connection; the seam consumes a `Mood` and returns a `Color` only
    ([I11], [P2]).
12. The arousal → strength math is mirrored TS↔Go and pinned by a shared golden-parity fixture; the Go and TS
    implementations produce identical `base_strength` for the same arousal inputs and the same `values.yaml` bounds.
13. Adding/removing a mood requires a synchronized change to the ENUM and both `values.yaml` number-maps
    (`emotion.mood_valence`, `emotion.mood_arousal`); a mismatch fails at `gen:values` / build.
