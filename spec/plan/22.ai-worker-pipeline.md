# 22. ai worker pipeline

> AI worker pipeline — the `internal/ai` external-service wrapper (concrete adapters + keyless mock) behind the
> memory-owned `Extractor`/`Embedder`/`Semanticizer` ports, plus the `jobs` queue and `cmd/worker` that drains it.
> Scope: BE | Status: planning

## Purpose

Epic A's launch loop triggers slow, third-party work: the LLM split/extract that turns a `Diary` into proposed
memories, the pre-generation of the four semanticization stage texts at creation ([C7]), the embedding of neurons for
dedup and latent-field assist, and the link-refinement pass. Some of it is user-facing and synchronous (the split
preview [W2]); most of it is expensive and belongs in the background.

This unit builds the two seams that let the memory context consume that work without ever importing an SDK or a queue
library: (1) the **`internal/ai` supporting external-service context** — concrete adapters that implement the
memory-owned `Extractor`/`Embedder`/`Semanticizer` ports, each with a real LLM/embedding SDK adapter **and** a keyless
deterministic mock fallback for dev/CI/tests; and (2) the **async worker** — the generic `jobs` queue claim/backoff
mechanism plus `cmd/worker`, which drains the queue and dispatches each job kind to a memory use-case. It is the
"behind consumer-owned ports + an async worker" half of the backend's AI stance (ARCHITECTURE §2.1, §2.8).

It exists so that `LaunchStars` returns an id immediately, the client renders optimistically, and the slow results
(embeddings, gist texts, refined synapses) surface on the **next read** — no polling, no streaming. It defines the
*shape* — adapters, ports, job kinds, the claim/backoff loop, cost metering — not the domain logic those jobs run;
`Encode`/`PersistEncoded`/`Link` behavior is owned by [20.encode-usecase](20.encode-usecase.md) and
[21.link-usecase](21.link-usecase.md), and the tables by [16.memory-aggregate-schema](16.memory-aggregate-schema.md).

## Scope / Non-goals

**In scope**
- The `internal/ai` context: a SUPPORTING external-service wrapper (ARCHITECTURE §2.2) holding **concrete adapters**
  that implement the memory-owned ports `Extractor`, `Embedder`, `Semanticizer`. No domain logic lives here.
- **Two adapters per port**: (1) a real LLM / embedding SDK adapter, (2) a **keyless deterministic mock** fallback that
  runs offline with no API key (dev, CI, tests). Selection at the composition root (`cmd/api` / `cmd/worker`) by config
  — presence of an API key picks the real adapter, absence picks the mock.
- Cost metering on every real call: per-call token cap (`ai.per_call_token_cap`), daily call cap (`ai.daily_call_cap`),
  and caching where sensible. The `Extractor` output is **schema-forced** ([W4a]) so a prompt-injected reply cannot
  produce an invariant-violating shape.
- The generic **`jobs` queue mechanism** (table owned by memory, plan 16): claim with `SELECT … FOR UPDATE SKIP
  LOCKED`, statuses `pending`/`running`/`done`/`failed`, an `attempts` counter, exponential backoff via `next_run_at`
  (ARCHITECTURE §2.8). The generic claim/backoff helper MAY live in `internal/platform`; the job *kinds* are memory
  use-cases the worker dispatches.
- Epic A job kinds: `embed` (embed neurons after launch), `semanticize` (pre-generate the 4 gist stage texts at
  creation [C7]), `link` (optional cross-memory linking refinement — see [21.link-usecase](21.link-usecase.md)).
- `cmd/worker` binary (Foundation stub, wired here): drains `jobs`, dispatches each kind to the memory use-case,
  completes on success or retries with backoff on failure up to a cap. May run as a goroutine in dev.
- The **optimistic write** contract: the write returns only ids; worker results surface on the next read/refetch.
- The `spec/values.yaml` keys this unit introduces (`ai.per_call_token_cap`, `ai.daily_call_cap`, `ai.job_max_attempts`,
  `ai.job_backoff_base_ms`; `ai.embedding_dim` is introduced by plan 16 and consumed here).

**Non-goals**
- No domain use-case logic. `Encode` / `PersistEncoded` (plan 20), `Link` (plan 21), the dedup canonicalization policy
  [E10], and the `SplitDiary`/`LaunchStars`/`ReviseSplit`/`GetUniverse` RPC contracts are owned elsewhere; this unit
  only provides the adapters those use-cases inject and the worker that runs their jobs.
- No new tables or migrations — the `jobs` table (and every other memory table) is created by
  [16.memory-aggregate-schema](16.memory-aggregate-schema.md); this unit consumes it.
- No `extract` or `consolidate` job kinds built here. `extract` is **synchronous** in Epic A (run inside `SplitDiary`
  via `Extractor`, so the user sees the result [W2] — not a background job); `consolidate` is Epic E. Both kinds are
  **reserved seams** (declared as valid `jobs.kind` values, plan 16) but no dispatch branch is built for them now.
- No prompt engineering, model selection, or embedding-model choice hard-coded here — the concrete LLM/embedding
  provider and model are **left open (see root DECISIONS.md)**; the port surface and mock are provider-agnostic.
- No server-streaming or progress push — forbidden by ARCHITECTURE §2.9 #9 (RN has no server-streaming) and the
  optimistic-write pattern (§2.8).
- No production autoscaling, external queue broker (SQS/Redis/etc.), or cron scheduler — the queue is the Postgres
  `jobs` table; the worker is a plain process/goroutine.
- No `Diary` mutation. Neither the `Extractor` nor the `Semanticizer` ever writes or reshapes a `Diary` ([I2]).

## Design

### The `internal/ai` external-service context

`internal/ai` is a **SUPPORTING external-service wrapper** (ARCHITECTURE §2.2): a thin ring around a 3rd-party
dependency, sitting at the outermost edge with the persistence adapters. It holds the **concrete adapters** that
implement the memory-owned ports and **nothing else** — no `Diary`/`EpisodicMemory`/`Neuron` types, no split rules, no
dedup policy, no synapse math. Those are domain content and live in `internal/memory` (plans 16–21). If a call site
inside `internal/ai` reaches for a memory domain type, that is the signal the logic is in the wrong package.

Per ARCHITECTURE §2.3 the wrapper may stay in its smaller shape forever; it does not earn the `{domain,app,pg,rpc}`
split. Its files are the adapter structs and their SDK/HTTP plumbing, keyed by port.

### Ports are consumer-owned in `internal/memory` (§2.4)

The three AI ports are **declared where they are consumed** — in `internal/memory`, next to the use-cases that call
them — not in `internal/ai` (ARCHITECTURE §2.4, forbidden pattern #6: no interfaces declared at the implementation).
`internal/ai` returns concrete structs that satisfy those interfaces *implicitly*. The port surface (canonical, from
[PRD.md](../PRD.md) §3.3 and the brief §4):

- `Extractor.Split(ctx, body, date, existingNeurons) → (memories, error)` — LLM split + name + emotion + neuron
  extract/normalize. Output is **schema-forced** ([W4a]): the adapter returns only the structured shape
  `{memories:[{name, mood, neurons:[{name, type}]}]}` — **no position/color/strength/time/delete fields exist in the
  output schema**, so a prompt-injected reply cannot structurally produce an invariant violation. A `ReviseSplit`
  variant takes the prior result + a natural-language instruction and returns the same schema-forced shape.
- `Embedder.Embed(ctx, texts) → ([]vector, error)` — element → vector. Vectors are dimension `ai.embedding_dim`,
  matching `embeddings.vector(N)` (plan 16).
- `Semanticizer.GenerateSemanticStages(ctx, memory) → ([4]string, error)` — the four gist stage texts at creation
  ([C7]). It reads memory facts and returns text; it **never** writes or reshapes a `Diary` ([I2]).

The domain terms crossing the port (`memory`, `neuron`, `mood`, `vector`) are the canonical domain types (or small
DTOs the memory context defines) — the adapter maps its SDK's request/response shape to and from them at its own
boundary, the same anti-corruption role `memory/pg` plays for sqlc rows and `memory/rpc` plays for proto (§2.4). No SDK
type leaks into memory behavior.

### Two adapters per port: real SDK + keyless mock

Each port has two concrete implementations in `internal/ai`:

1. **A real adapter** — calls the LLM / embedding SDK. The provider and model are **left open (see root
   DECISIONS.md)**: the default is the latest Claude models via the Anthropic SDK for the `Extractor` and
   `Semanticizer` (extraction and semanticization are language tasks), and an embedding model for the `Embedder`;
   the exact model ids and the embedding provider are confirmed at implement. API keys and model ids are **runtime
   env/secrets**, not `spec/values.yaml` (see Policy / Values Impact).
2. **A keyless deterministic mock** — runs fully **offline with no API key**, for dev, CI, and tests. It is
   *deterministic*: given the same input it returns the same output (e.g. split by a stable rule, a name derived from
   the body, a fixed neuron set; an embedding derived by a stable hash of the text into a `ai.embedding_dim`-length
   vector; gist stage texts derived from the memory's fields). Determinism is what lets tests assert exact results and
   lets golden-parity fixtures exist without a network. The mock honors the **same schema-forced output shape** and the
   same invariants (never mutates `Diary`) as the real adapter, so swapping adapters changes cost and quality, never
   contract.

**Selection at the composition root.** `cmd/api` and `cmd/worker` choose the adapter by config: if the API key is
present, inject the real adapter; if absent, inject the keyless mock (ARCHITECTURE §2.4 — the composition root sees
everything and injects concrete dependencies). No feature flag and no `spec/values.yaml` entry gates this — it is a
runtime wiring decision driven by secret presence. New providers/capabilities slot in behind the same consumed port,
and nothing in `internal/memory` changes (§2.8).

### Cost metering (real adapter only)

Every real call is cost-metered (ARCHITECTURE §2.8: "Cost-metered — per-call caps, caching, daily/monthly limits"):

- **Per-call token cap** — `ai.per_call_token_cap` bounds the max output tokens of any single LLM call (the concrete
  `max_tokens`-style ceiling; the `Extractor`'s own retry/token caps for revise are `encode.*`, owned by plan 20 —
  this unit's `ai.*` caps are the generic per-call and per-day guards).
- **Daily call cap** — `ai.daily_call_cap` bounds the number of billable LLM/embedding calls per user per universe-day
  (or per calendar day; the exact accounting window is a tuning detail confirmed at implement). Over the cap, the
  adapter returns a typed cost-limit error; the caller surfaces it (a job retries later per backoff; a synchronous
  split returns a limit error to the RPC).
- **Caching where sensible** — identical embedding inputs and identical extraction inputs are cache-eligible so a
  retried job or a re-run split does not re-bill. The cache is an adapter-internal concern (keyed by a hash of the
  request), not a domain concept.

The mock adapter is free and unmetered — caps do not apply to it.

### The `jobs` queue mechanism

The queue is the Postgres `jobs` table (owned by [16.memory-aggregate-schema](16.memory-aggregate-schema.md)):
`id`, `user_id`, `kind`, `payload JSONB`, `status`, `attempts`, `next_run_at`, `created_at`. The mechanism
(ARCHITECTURE §2.8):

- **Enqueue** — a use-case that triggers slow work inserts a `pending` job with a `payload` (e.g. the memory ids to
  embed, the memory id to semanticize) and `next_run_at = now()`. `LaunchStars` enqueues `embed` + `semanticize`
  (plan 20) and optionally `link` (plan 21) inside the same transaction that persists the aggregate, so a launch that
  commits always has its follow-up work queued (transactional outbox shape — no lost jobs).
- **Claim** — the worker claims due work with `SELECT … FOR UPDATE SKIP LOCKED`: pick the oldest `pending` row whose
  `next_run_at <= now()`, lock it, mark it `running`, all in one transaction. `SKIP LOCKED` lets multiple worker
  goroutines/processes claim disjoint rows without blocking each other.
- **Complete** — on success the row is marked `done`.
- **Retry with backoff** — on failure `attempts` is incremented and, if `attempts < ai.job_max_attempts`, the row goes
  back to `pending` with `next_run_at = now() + backoff(attempts)`, where `backoff` is exponential from a base of
  `ai.job_backoff_base_ms` (e.g. `base * 2^attempts`, optionally jittered — the exact curve is a tuning detail
  confirmed at implement). At `attempts >= ai.job_max_attempts` the row is marked `failed` and left for inspection — it
  is **not** deleted and it never deletes any domain row ([I1]).

The **generic claim/backoff helper MAY live in `internal/platform`** (the brief's decision) — it is mechanism, not
domain: a small helper that, given a `kind→handler` dispatch map, runs the claim/run/complete/retry loop. The `jobs`
table and the *job kinds* remain memory-owned; the platform helper knows only about generic job rows, not about
neurons or gist texts.

### `cmd/worker` — the async worker binary

`cmd/worker` is the Foundation-stubbed worker binary, wired here (ARCHITECTURE §2.3 layout: `cmd/worker/main.go`, same
composition style as `cmd/api`). It:

1. Composes concrete dependencies at its root: the memory repositories/`JobQueue` (`memory/pg`), the AI adapters
   (`internal/ai`, real or keyless mock by config), and the memory use-cases those adapters feed.
2. Runs the claim/backoff loop (the platform helper), polling for due `pending` jobs.
3. **Dispatches each claimed job by `kind` to the owning memory use-case**:
   - `embed` → embed the payload's neurons via `Embedder`, upsert `embeddings` rows (the use-case + query owned by
     plan 20 / the aggregate baseline).
   - `semanticize` → `Semanticizer.GenerateSemanticStages`, write the 4 gist texts into `episodic_memories.semantic_stages`
     (JSONB) for that memory — **never touching its `Diary`** ([I2], [C7]).
   - `link` → the optional cross-memory link-refinement pass (plan 21).
   - `extract` / `consolidate` → **no branch built** in Epic A (reserved seams — see Non-goals). An unknown/unhandled
     kind is a defensive `failed`, not a panic.
4. Completes or retries per the mechanism above.

The worker **may run as a goroutine inside `cmd/api` in dev** (ARCHITECTURE §2.8) so a single `docker compose up` gives
the full loop; in a real deployment it is a separate process. Both wire the same code.

### Optimistic write — how slow results surface

`LaunchStars` (plan 20) persists the aggregate, enqueues the jobs, and returns only the new memory ids. The client
renders the universe **optimistically** from the ids and the facts it already has. When the worker later fills the
embeddings (used server-side for dedup/latent assist) and the `semantic_stages` JSONB (the pre-generated gist texts),
those values appear on the **next `GetUniverse` read / refetch** (plan 23) — there is **no polling loop and no
server-streaming push** (ARCHITECTURE §2.8, §2.9 #9). A memory whose `semantic_stages` is still `NULL` simply reads as
"not yet semanticized"; the read query returns whatever facts exist at read time. This is the same derived-vs-stored
discipline the rest of the system uses: the write records discrete facts, the reader observes the latest state.

### Reserved seams

- **`extract` kind** — reserved but unbuilt: Epic A runs extraction synchronously inside `SplitDiary` (user-facing
  preview [W2]) via the `Extractor` port, not as a background job. The `jobs.kind` value exists so a later epic can
  move extraction to the background without a schema change.
- **`consolidate` kind** — reserved for Epic E (the "universe's sleep": replay/reconsolidation/`Semanticize`/
  `Downscale`). Declared as a valid kind now so Epic E slots a dispatch branch in without reshaping the queue.
- **Monthly cost caps / per-provider budgets** — ARCHITECTURE §2.8 allows "daily/monthly limits"; Epic A ships the
  per-call and daily caps and leaves a monthly ceiling as a later tightening (left open — see root DECISIONS.md).

## Acceptance Criteria
1. The `Extractor`, `Embedder`, and `Semanticizer` **ports are declared in `internal/memory`** (consumer-owned), and
   their **concrete adapters live in `internal/ai`**; `internal/ai` imports no memory domain behavior and contains no
   split/dedup/synapse/gist logic [ARCHITECTURE §2.2/§2.4].
2. Each port has two adapters — a real LLM/embedding SDK adapter and a **keyless deterministic mock** — and the
   composition root (`cmd/api` / `cmd/worker`) selects between them by config (API-key presence), with no change to
   `internal/memory` when switching [§2.4/§2.8].
3. The keyless mock runs with **no API key and no network**, and is **deterministic**: the same input yields the same
   split/name/neurons, the same `ai.embedding_dim`-length embedding vector, and the same 4 gist stage texts — so tests
   and golden-parity fixtures pass offline.
4. Both the real and mock `Extractor` return only the **schema-forced** shape `{memories:[{name, mood, neurons:[{name,
   type}]}]}` — no position/color/strength/time/delete field is present, so a prompt-injected reply cannot structurally
   yield an invariant violation [W4a].
5. The worker claims jobs with `SELECT … FOR UPDATE SKIP LOCKED`, moves them `pending→running`, and on failure retries
   with **exponential backoff via `next_run_at`** (base `ai.job_backoff_base_ms`) up to `ai.job_max_attempts`, after
   which the job is marked `failed` — and no failed/exhausted job deletes any neuron, memory, or synapse [I1][§2.8].
6. Concurrent worker goroutines/processes claim **disjoint** jobs (no two claim the same row) via `SKIP LOCKED`.
7. `cmd/worker` drains the `jobs` table and **dispatches each kind to the owning memory use-case**: `embed` (embed
   neurons → `embeddings`), `semanticize` (write the 4 gist texts to `episodic_memories.semantic_stages` [C7]), and
   `link` (link refinement, plan 21). The worker runs as a standalone process **and** can run as a goroutine in dev.
8. `extract` and `consolidate` are valid `jobs.kind` values (plan 16) but have **no dispatch branch** in Epic A
   (`extract` is synchronous via `SplitDiary`; `consolidate` is Epic E); an unhandled kind is marked `failed`, never
   panics.
9. The real adapters enforce cost metering: a call whose output would exceed `ai.per_call_token_cap` is bounded, and a
   user over `ai.daily_call_cap` gets a typed cost-limit error (job retries later, sync split returns a limit error);
   identical inputs are cache-eligible so retries/re-runs do not re-bill. The mock is unmetered.
10. Neither the `Extractor` nor the `Semanticizer` (real or mock) writes or reshapes a `Diary`; the `semanticize` job
    writes only `episodic_memories.semantic_stages` [I2][C7].
11. Slow results are **optimistic**: after `LaunchStars` returns ids, embeddings and `semantic_stages` appear on the
    **next `GetUniverse` read/refetch** with **no polling and no server-streaming**; a not-yet-filled `semantic_stages`
    reads as `NULL` [§2.8][§2.9 #9].
12. The `jobs`-queue enqueue happens in the **same transaction** as the launch persist (plan 20), so a committed launch
    always has its `embed`/`semanticize`(/`link`) jobs queued — no lost work.
