# 13. MVP 통합 검증 (E2E·정적 점검·한 바퀴 완성)

> 01–12 산출물을 한데 묶어 코어 루프 한 바퀴를 수동 E2E + 정적 검증으로 확인. | 범위: MVP | 선행: 01–12 | 차단: —

## 목적 (Why)
개별 스펙은 자기 영역만 검증한다. 이 스펙은 **사인인→작성→별 등장→연결→회상 강화→감쇠→잠든 별 재점화**가 한 시스템에서 실제로 이어지는지, 그리고 8개 헌법 원칙이 통합 상태에서도 깨지지 않는지를 최종 확인한다. 빌드·lint·마이그레이션·임포트 방향까지 정적으로 점검해 "한 바퀴 완성"을 선언 가능하게 만든다.

## 범위
- **포함:** `pnpm dev`가 pgvector compose + 프론트를 동시 기동하는지 확인, `.env.example` 정리(필요 키만 남김), 수동 E2E 시나리오 체크리스트, `check-errors`(프론트 build + ESLint + goose 마이그레이션 dry) 통과, FSD 임포트 방향 정적 점검(원칙4), `@react-three/postprocessing` 제거 확인, 불변 원본·별 비삭제(원칙1·2) 통합 점검.
- **비목표(Out of scope):** 배포(Supabase 클라우드/Hetzner/Cloudflare 실제 구성·CI 파이프라인) — 별도 v1+ 단계. 자동화 E2E(Playwright 등) 작성 — 본 스펙은 수동 체크리스트만. 각 기능의 단위 검증은 해당 스펙(04·05·11·12)이 담당하며 여기서는 통합 흐름만 본다.

## 설계 요점
**(1) `pnpm dev` 동시 기동.** `package.json`의 `dev`는 `concurrently`로 `frontend dev`(Vite :1214) + `dev:api`(`docker compose --profile dev up backend`)를 띄운다. backend는 `depends_on: postgres(healthy)`. 따라서 03에서 postgres 이미지가 `pgvector/pgvector:pg16`으로 전환되면 `pnpm dev` 한 번으로 pgvector + API + 프론트가 함께 뜬다. 본 스펙은 그 상태를 **확인**만 한다(코드 변경 최소).

**(2) `.env.example` 최종 형태(검증 기준).**
```dotenv
# Backend
PORT=8080
DATABASE_URL=postgres://cosimosi:cosimosi@localhost:5432/cosimosi?sslmode=disable
# AI (공급자 추상화 — 원칙7)
AI_EMBEDDER=openai          # openai | mock
OPENAI_API_KEY=sk-...       # AI_EMBEDDER=openai일 때만 필요
# Frontend (Supabase Auth — 원칙: 인증은 FE에서)
VITE_SUPABASE_URL=https://<project>.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=<publishable-key>
VITE_API_URL=http://localhost:8080
```
CORS_ORIGIN은 compose env로만 둔다(S3 키는 쓰지 않음 — 이미 부재).

**(3) 정적 점검 항목(통합 게이트).**
- 빌드: `pnpm --filter ./frontend build` 성공(TS·Vite 오류 0).
- Lint: `pnpm --filter ./frontend lint` 통과.
- 마이그레이션: goose가 깨끗한 pgvector DB에 `up` 적용 성공(03 산출).
- 임포트 방향(원칙4): `entities/*/model/**`·`shared/lib/**`에서 `three`·`react`·`@react-three/*`·`document`·`window` import 0건.
- postprocessing 제거(원칙8 Bloom은 TSL 경로로): 의존성·소스에서 `@react-three/postprocessing` 0건.
- 불변·비삭제(원칙1·2): sqlc 쿼리·마이그레이션 **Up 본문**에 원본 본문 `UPDATE records`/`DELETE FROM records`, 별·시냅스 row `DELETE FROM (memories|memory_links)` 0건(감쇠는 클라 밝기 계산만). goose `Down`의 `DROP TABLE`과 `memories`/`memory_links`의 `last_recalled_at`/`last_activated_at`/`weight` 갱신 `UPDATE`(11·12)는 제외.

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 개발자가 `pnpm dev`를 실행하면 THE SYSTEM SHALL pgvector postgres·Connect API(:8080)·Vite 프론트(:1214)를 단일 명령으로 모두 기동한다.
1.2 IF postgres가 healthy가 되기 전이라면 THEN THE SYSTEM SHALL backend 컨테이너를 대기시켜 연결 실패로 죽지 않게 한다.
2.1 WHEN `.env.example`을 열면 THE SYSTEM SHALL `VITE_SUPABASE_URL`·`VITE_SUPABASE_PUBLISHABLE_KEY`·`OPENAI_API_KEY`·`AI_EMBEDDER`·`DATABASE_URL`을 포함하고 `S3_*` 키를 포함하지 않는다.
3.1 GIVEN 빈 우주에 사인인한 사용자가 WHEN 일기를 작성하면 THEN THE SYSTEM SHALL 새 별 1개를 캔버스에 등장시킨다(10).
3.2 GIVEN 의미 유사한 두 기억이 존재할 때 WHEN 워커가 임베딩·연결을 처리하면 THEN THE SYSTEM SHALL cos_sim ≥ 0.75인 쌍에 시냅스(가중치)를 생성한다(05).
3.3 GIVEN 두 별이 한 화면에서 함께 회상될 때 WHEN 회상 배치가 전송되면 THEN THE SYSTEM SHALL 해당 시냅스 weight를 +0.05(상한 1.0) 강화하고 last_activated_at을 갱신한다(11).
3.4 GIVEN 오래 안 본 별이 WHEN 시간이 지나면 THEN THE SYSTEM SHALL activation=exp(-λ·Δt)로 어두워지되 a_min=0.05 미만으로 떨어뜨리지 않고 별을 잔존시킨다(12, 원칙2).
3.5 GIVEN 잠든 별이 존재할 때 WHEN dormant 페이지에서 탐색·재점화하면 THEN THE SYSTEM SHALL 그 별을 다시 밝히고 회상으로 강화한다(12).
4.1 WHEN `pnpm --filter ./frontend build`·`lint`를 실행하면 THE SYSTEM SHALL 오류 0으로 통과한다.
4.2 WHEN goose가 깨끗한 pgvector DB에 마이그레이션을 적용하면 THE SYSTEM SHALL 모든 up 마이그레이션을 오류 없이 적용한다.
5.1 THE SYSTEM SHALL `entities/*/model`·`shared/lib` 하위 어떤 파일도 `three`·`react`·`@react-three/*`·DOM(`document`/`window`)을 import 하지 않는다(원칙4).
5.2 THE SYSTEM SHALL 프론트 의존성·소스 어디에도 `@react-three/postprocessing`을 포함하지 않는다(원칙8).
6.1 THE SYSTEM SHALL 원본 본문 테이블(`records`)에 대한 row `UPDATE`/`DELETE`, 별·시냅스(`memories`/`memory_links`)에 대한 row `DELETE` 쿼리를 포함하지 않는다(원칙1·2). 단 goose `Down` 마이그레이션의 `DROP TABLE`(스키마 드롭)과 `memories`/`memory_links`의 `last_recalled_at`/`last_activated_at`/`weight` 갱신 `UPDATE`(11·12의 정당한 회상·강화)는 위반이 아니다.
