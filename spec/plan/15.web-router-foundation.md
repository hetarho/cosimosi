# 15. web router foundation

> Web router foundation — a **typed client-side router** for `apps/web` (TanStack Router), confined to the
> `app/routes` segment, mirroring the mobile app-shell's `app/navigation` discipline. Presentation plans add feature
> routes through one typed registry instead of extending an ad-hoc switch.
> Scope: FE | Status: built
>
> The web sibling of [13.mobile-app-shell](13.mobile-app-shell.md)'s navigation seam. Mobile already routes through
> a typed `react-navigation` registry; web still hand-rolls a two-way path switch. This unit closes that gap — the
> **last Foundation unit before Epic A's presentation units**, which land as routable screens.

## As-built status

Built by [job 19](../jobs/archive/19.web-router-foundation.md); as-built rules: [tech/web-routing.md](../tech/web-routing.md).

- **TanStack Router v1**, code-based route tree, lives in `apps/web/src/app/routes/` — the only segment that imports
  `@tanstack/react-router` (`route-tree.tsx` · `router.ts` · `WebRouterProvider.tsx` · `not-found.tsx` · the
  `navigation.ts` seam). `createAppRouter({ diagnosticsEnabled, initialEntries? })` builds the router; the type registry
  (`declare module … Register`) makes `Link`/`navigate` compile-time path-checked.
- **Routes:** `/` → `UniverseHomePage`, `/test` → `TestPage` gated by a `beforeLoad` — always reachable under the Vite
  dev server (`import.meta.env.DEV`), and in a production build only when the `platform.diagnosticsSurface` flag is on
  (unreachable → not-found otherwise) — and a localized `notFoundComponent`.
- **Composition:** `WebRouterProvider` reads the flag from the observability facade and mounts `<RouterProvider>` as the
  innermost child of the provider stack (observability → error boundary → i18n → auth → cache → router).
- **Testing:** SSR tests build a router at a route, `await router.load()`, then inject it into `<App router>` — no DOM.
- **Match semantics:** query strings are ignored for matching (preserved from the retired normalizer). Trailing-slash
  normalization (`/test/`) is **not** handled — out of scope for the two-route set; a later presentation plan owns the
  trailing-slash policy when a real IA exists.

## Purpose

`apps/web` has no router. [App.tsx](../../apps/web/src/app/App.tsx) resolves the screen with a hand-rolled
`window.location.pathname` check — `/test` → `TestPage`, everything else → `UniverseHomePage`. That was right when only
two screens existed, but every presentation unit from Epic A onward (`writing-flow-ui`, `recall-flow-ui`,
`star-detail-panel`, `diary-reader-page`, `settings-page`, `auth-universe-gate`) is a URL-addressable screen that needs
real navigation, params, and a not-found path. Extending the ad-hoc switch would scatter routing logic and lose the
type-safety the mobile shell already has.

Mobile solved this once: [routes.ts](../../apps/mobile/src/app/navigation/routes.ts) is a **typed route registry**, and
the navigation library lives in exactly two files — "feature screens are added later by presentation plans through this
same typed boundary." This unit gives web the equivalent substrate so Epic A consumes a ready, typed routing seam
(web↔mobile parity, ARCHITECTURE §3.1).

## Scope / Non-goals

**In scope**
- **TanStack Router as the web router**, confined to a single **`apps/web/src/app/routes/`** segment (ARCHITECTURE §3.1
  names this segment). That segment is the *only* place that imports `@tanstack/react-router` — the same
  "library-knowing files stay confined" rule mobile's `routes.ts` / `NavigationRoot` follow.
- **Code-based route tree** (`createRootRoute` / `createRoute` / `createRouter`), not file-based. File-based routing
  scatters route files across the tree and fights FSD; a code-based tree keeps routing in `app/routes/` while route
  components import screens from the `pages/` layer.
- **A typed route registry** — `declare module '@tanstack/react-router' { interface Register { router } }` — so every
  `Link` / `useNavigate` is compile-time path-checked (parity with mobile's `RootStackParamList`).
- **Migrate the two existing screens** off the hand-rolled switch: `/` → `UniverseHomePage`, `/test` → `TestPage`. The
  `route === 'test' ? … : …` branch in `App.tsx` is deleted.
- **`RouterProvider` composed inside the existing provider stack** so all routes see observability / i18n / auth /
  cache (providers stay outer, router inner; the root route renders the shared chrome + `<Outlet/>`).
- **A navigation seam** re-exported from the `app/routes` public API, so `pages` / `features` navigate with a typed
  `Link` / `navigate` without importing the router library name directly (mirrors the mobile seam).
- **Route-level not-found + error integration** — an unknown path renders a localized not-found screen (i18n `m.*`);
  route render errors surface through the existing `ObservedErrorBoundary`.
- **Testability preserved** — the app is mountable at an arbitrary route via injected **memory history**, replacing
  the current `routePath` prop, so tests/storybook render a route without `window.location` coupling.

**Non-goals**
- **No product IA / feature routes.** `writing-flow-ui`, `recall-flow-ui`, `star-detail-panel`, `diary-reader-page`,
  `settings-page`, `auth-universe-gate` are their own presentation plans; this unit ships only the substrate + the two
  screens that exist today (same posture as the mobile registry's deliberately small, non-product route set).
- **No SSR / framework mode.** Pure client SPA (Vite) — TanStack Start, server routes, and streaming are out.
- **No route-level code-splitting / lazy routes** now. The tree is structured so lazy loading can be added later
  without moving screens (the >500 kB single-chunk build warning is a *future* route-split win, not this unit's job).
- **No route loaders / data prefetching.** TanStack Router↔Query loader wiring is per-feature and deferred.
- **No product auth-gated redirects.** `auth-universe-gate` (Epic I) owns that rule; this unit provides only the
  route/guard seam, not the product redirect.
- **No mobile changes and no shared route-contract package.** Parity is *disciplinary*, not shared code — the two apps
  use different libraries and param shapes, and FSD puts navigation in each app's `app` layer.
- No deep-link / URL-scheme unification with mobile. No new tuning numbers hardcoded in FE.

## Design

### Library & placement
- **TanStack Router**, chosen for design-first type-safety (matches the mobile `RootStackParamList` discipline),
  first-class search-param APIs, and native integration with the `@tanstack/react-query` the web app already uses. It
  runs as a pure client-side router — no SSR baggage.
- All routing lives in **`apps/web/src/app/routes/`**. This segment (a `routes.ts`-style registry + the route tree +
  `RouterProvider` wiring) is the **only** code that imports `@tanstack/react-router`. Lower FSD layers never see the
  library — exactly the confinement `apps/mobile/src/app/navigation` enforces.

### Route tree (code-based, typed)
- `createRootRoute` defines the root: its component renders the shared chrome and a `<Outlet/>`. Child routes:
  - **`/`** → `UniverseHomePage` (the `pages/universe` screen — plan 14's universe).
  - **`/test`** → `TestPage` (the `pages/test` harness — plan 12), registered **only** when the dev/diagnostics flag is
    on (parity with mobile's flag-gated `Diagnostics`), so production does not expose it.
  - **catch-all not-found** → a localized `notFoundComponent` (i18n `m.*`).
- `createRouter({ routeTree })` builds the router; a typed registry is published:
  ```ts
  declare module '@tanstack/react-router' {
    interface Register { router: typeof router }
  }
  ```
  From here every `Link to=…` / `navigate({ to })` is compile-time path-checked — an unknown path fails `typecheck`,
  the web mirror of mobile's `RootStackParamList` guarantee.

### Composition in the provider stack
- `App.tsx` keeps its provider nesting (observability → `ObservedErrorBoundary` → i18n → auth → cache) and mounts
  **`<RouterProvider router={router} />` as the innermost child**, replacing the hand-rolled
  `route === 'test' ? <TestPage/> : <UniverseHomePage/>` switch. Providers stay **outer** so every route sees them;
  the router is **inner** and its root route renders the page via `<Outlet/>`. `WebObservabilitySessionBridge` and the
  error fallback stay where they are, wrapping the router.

### Testability & dev injection
- `App`'s `routePath` prop is replaced by an injectable **router / initial history** built on `createMemoryHistory`,
  so tests and storybook mount at any route without touching `window.location`. Production uses browser history by
  default. The other injectable facades (auth / query / transport / observability / locale) are unchanged.

### Navigation seam for consumers
- The `app/routes` public `index.ts` re-exports a typed `Link` and a `useNavigate`-style helper, so `pages` / `features`
  navigate through the app-layer seam rather than importing `@tanstack/react-router` directly — the same seam idea
  mobile uses to keep the navigation library confined.

### Deferred-but-compatible
- The tree is shaped so **route-level lazy loading** and **Query-integrated loaders** can be added later per feature
  without relocating screens. Neither is built here.

## Acceptance Criteria
1. `apps/web` navigates via TanStack Router; `@tanstack/react-router` is imported **only** within
   `apps/web/src/app/routes/`, and `pnpm lint:fsd:layout` sees a segmented `app/routes` (no flat `app/` files, web↔mobile
   parity).
2. Routes form a **code-based, typed** route tree; the typed registry (`declare module … Register`) makes `Link` /
   `navigate` compile-time path-checked — an invalid `to=` **fails `pnpm typecheck`** (parity with mobile's
   `RootStackParamList`).
3. The hand-rolled path switch in `App.tsx` is removed; `/` renders `UniverseHomePage` and `/test` renders `TestPage`
   through the router, with `RouterProvider` composed **inside** the existing provider stack (providers outer, router
   inner) so every route sees observability / i18n / auth / cache.
4. `/test` is registered only under the dev/diagnostics flag (parity with mobile's `Diagnostics`) and is **not** exposed
   in a production build.
5. An unknown path renders a **localized** not-found screen (i18n `m.*`, no raw strings), and route render errors surface
   through `ObservedErrorBoundary`.
6. The app is mountable at an arbitrary route via injected **memory history** (no `window.location` coupling); existing
   `App` tests / storybook keep passing.
7. `pages` / `features` navigate through a typed seam re-exported from `app/routes` — they do **not** import the router
   library name directly.
8. `pnpm typecheck`, `pnpm lint` (incl. `lint:fsd:layout`, web boundary probe, `lint:raw-strings`), `pnpm test:web`, and
   `pnpm build:web` all pass; no new mobile changes.
9. No SSR / framework mode, no route loaders, and no product feature routes are added — those belong to their own
   presentation plans.
