## Purpose

cosimosi ships real product surfaces (auth, the memory engine, the Twinkle economy, multiple AI providers), but the
operator has **no place to run the service**. Three concrete gaps, from the request:

1. **AI provider is unmanageable at runtime.** [28.ai-provider-abstraction](28.ai-provider-abstraction.md) made the
   provider/model/API-key an env secret chosen once at process start — switching providers, rotating a key, or trying a
   different model is a redeploy/ops action with no in-product control. An operator needs to see the effective AI
   configuration and change it live.
2. **Users are invisible.** There is no way to see who has signed up, what their stardust balance is, or to help them —
   e.g. by **granting stardust** (별가루 증정) for support, promotion, or make-goods.
3. **Admin identity is unbootstrapped.** `ADMIN_USER_IDS` exists in the environment but nothing consumes it, and there
   is no way to appoint additional admins without editing env and redeploying.

Beyond the three, an operator running an AI-backed service needs **operational visibility**: how much AI is being spent
against the plan-22/28 caps, and whether the async worker pipeline is healthy. Those two dashboards are in v1.

The load-bearing constraint that shapes every decision here: cosimosi stores **private diary/memory data** ([I2]). The
admin console is therefore built to expose **account metadata only** — identities, balances, counts, health — and to
be **structurally incapable** of reading a user's diary text, emotions, or memory content. Admin power is broad across
_accounts_; it is zero across _memory content_.

## Scope / Non-goals

**In scope**

- **A NEW `internal/admin` bounded context** (CC8, domain-first Go) exposing **`admin.v1.AdminService`** (Connect,
  unary), reached by no product code — only the admin web app. It **orchestrates** other contexts through
  consumer-owned ports at `cmd/api`; it **never imports** `memory`, `twinkle`, `account`, or `ai` internals.
- **An admin-authorization interceptor.** Layered on plan 04's auth interceptor: every `admin.v1` method requires the
  authenticated caller to be an **admin**, else the canonical **permission-denied** RPC error. "Admin" = the caller's
  user id (or verified email) is listed in `ADMIN_USER_IDS` **or** has a row in the `admin_users` table.
- **Admin identity & management** ([the request]):
  - `ADMIN_USER_IDS` (env, comma-separated Supabase UUIDs and/or verified emails) is the **seed/root** admin set,
    parsed at the composition root. Root admins **cannot be demoted** through the console (they are the trust anchor).
  - An `admin_users` table holds **DB-promoted** admins (who was promoted, by whom, when). `GrantAdmin` / `RevokeAdmin`
    promote/demote from the user list. Effective admin = env-seed ∪ DB rows.
  - `ListAdmins` shows the current admin set (seed vs promoted), and `GetAdminSelf` tells the _caller_ whether they are
    an admin (so the FE can gate `/admin` and show its affordance).
- **AI provider config (runtime, DB-override-over-env)** — a **change to [28](28.ai-provider-abstraction.md)**:
  - A new `ai_provider_config` table holding, per capability (LLM, embedding): provider name, model, base URL, and an
    **encrypted** API key (+ a `key_set` flag), with `updated_by`/`updated_at`.
  - `GetAIConfig` returns the effective config **with the key masked/never in plaintext** (`keySet: true` + a short
    hint only). `SetAIConfig` validates the provider is a **known, implemented** slot (else reject, exactly as plan 28's
    unknown/not-implemented startup errors do) and, for embeddings, that the model can honor `ai.embedding_dim`; it
    encrypts and stores the key.
  - The AI factory seam is changed from **build-once-at-startup-from-env** to **resolve-effective-config**: DB row if
    present, else env, else the keyless deterministic mock — and it **rebuilds the vendor client when config changes**
    so a `SetAIConfig` takes effect **without redeploy**.
- **User list** (account metadata only, [I2]):
  - `ListUsers` — paginated, searchable, sourced from an `AccountDirectory` port (Supabase Auth Admin API) for identity
    (id, email, signup date) **joined** with per-user domain facts read through ports: stardust balance (`twinkle`),
    non-content counts (# diaries / # stars from `memory`), and admin status. **No diary text, emotion, position, or
    memory content is ever read or returned.**
- **Grant stardust** (별가루 증정):
  - `GrantStardust(targetUser, amount, note)` credits the target's **additional** (permanent) balance via `twinkle`'s
    existing `Earn` path with a **new `admin_grant` reason**, append-only and idempotent (dedup by grant id), **capped**
    by `twinkle.admin_grant_max`. It writes an `admin_stardust_grants` audit row (who granted, to whom, how much, note,
    when) and a `ListStardustGrants` reads it back for accountability.
- **AI usage / cost dashboard** — `GetAIUsage`: today's call counts per capability against `ai.daily_call_cap` /
  `ai.per_call_token_cap`, from the plan-28 `Meter` snapshot (read-only). (Meter is in-process/in-memory today — see
  Design for the stated v1 limitation.)
- **Worker / job-queue health** — `GetJobHealth`: aggregate counts over the `jobs` table by status
  (pending / running / done / failed), backoff/attempts and dead-lettered counts (read-only, non-user-scoped
  operational read).
- **An append-only `admin_audit_log`** for sensitive actions (promote/demote, AI-config change, stardust grant) —
  accountability for the powers this console grants ([I1] spirit).
- **A web-only `pages/admin`** (route `/admin`, plan-15 typed registry) behind an **admin gate** (authenticated **and**
  admin), composing `features/admin-ai-config`, `features/admin-users` (list + grant + promote), `features/admin-usage`,
  `features/admin-jobs`. All copy through the i18n seam ([08](08.i18n-seam.md)).

**Non-goals**

- **No mobile admin surface** — admin is an internal operational console; the web+mobile parity rule
  ([ARCHITECTURE.md](../ARCHITECTURE.md) §3.5) is **explicitly waived** here. Web only.
- **No access to private memory content** ([I2]) — the console reads account **metadata** only; it never reads or
  renders a user's diary text, emotions, star meaning, positions, or connections. There is structurally **no** RPC that
  returns memory content to an admin.
- **No new AI provider adapters, no change to the capability interfaces, prompts, port adapters, error taxonomy, or the
  keyless mock** — [28](28.ai-provider-abstraction.md)/[22](22.ai-worker-pipeline.md) own those. This unit only makes
  the _selection_ runtime-editable and adds the config store; the set of supported provider slots is unchanged.
- **No change to the Twinkle ledger schema, balance derivation, spend order, or cost curves** — [43](43.stardust-ledger.md)/
  [44](44.earn-spend-usecase.md) own those. This unit adds one **earn reason** (`admin_grant`) and calls the existing
  `Earn`; it prices nothing and spends nothing.
- **No memory-content moderation, impersonation, "log in as user", or content takedown** — out of v1 (and content
  access is a non-goal on principle, [I2]).
- **No billing/refund/payment administration** — the payment path is [44](44.earn-spend-usecase.md)'s `Charge`; admin
  make-goods use the stardust grant, not payment reversal.
- **No durable cross-process AI usage warehouse** — the usage dashboard reads the existing in-memory meter snapshot;
  persistent aggregated analytics is a later plan (see Design → limitation).
- **No values.yaml keys for provider identity/keys** — those remain runtime config/secrets (now DB+env), never tuning.

## Design

### Placement — a NEW `internal/admin` context (domain-first Go, CC8)

Administration is not any existing context's behavior: it has its own authorization concern, its own tables
(`admin_users`, `admin_stardust_grants`, `ai_provider_config`, `admin_audit_log`), and it deliberately reads **across**
every other context. Per be-architecture / ARCHITECTURE §2.2 that makes it a **distinct core context**, `internal/admin`,
exposing `admin.v1.AdminService`. Dependency direction is **one-way and port-only** (CC8): `admin` calls into the other
contexts through **consumer-owned ports** wired at `cmd/api`, and **imports none of their internals** (the same rule that
keeps `twinkle` from importing `memory`). The ports `admin` declares:

- **`AccountDirectory`** — enumerate/lookup accounts (id, email, signup date, paginated + searchable). Concrete: a
  Supabase Auth **Admin API** adapter (service-role key, server-only) at `cmd/api`. A keyless fake serves tests/dev.
- **`StardustAdminPort`** — read a user's balance and `Earn(admin_grant)`. Concrete: bound over `twinkle`'s existing
  `GetBalance` + `Earn` (the `admin_grant` reason is added to `twinkle`; `admin` passes a scalar amount + target scope,
  never touches ledger internals).
- **`MemoryStatsPort`** — non-content per-user counts (# diaries, # launched stars). Concrete: a small aggregate read
  published by `memory` (counts only — **no** text, emotion, or position crosses the port; the port's return type has
  no content field, enforcing [I2] at the type level).
- **`AIConfigStore`** — get/set the `ai_provider_config` rows (the write side of the config seam, below).
- **`AIUsageReader`** — the plan-28 `Meter` snapshot (read-only).
- **`JobHealthReader`** — aggregate `jobs`-table status counts (read-only).

Only `admin/pg` imports `dbgen`; the pure domain (authorization rule, grant cap check, config validation) is
proto/sqlc-free (§2.4). Start in the context package; split `{app,pg,rpc}` only when earned (§2.3).

### Admin authorization ([the request])

Two sources, unioned:

- **Seed (root) admins** — `ADMIN_USER_IDS`, parsed once at the composition root into a set of Supabase UUIDs and/or
  verified emails. Root admins are the trust anchor: **they cannot be demoted through the console** (a `RevokeAdmin`
  against a seed admin is refused). This guarantees the service can never be locked out of its own admin surface.
- **Promoted admins** — rows in `admin_users` (`user_id` PK, `granted_by`, `granted_at`), created by `GrantAdmin` and
  removed by `RevokeAdmin` from the user list.

`IsAdmin(identity)` = identity ∈ seed **or** a row exists. The **`AdminAuthorizationInterceptor`** runs after plan 04's
auth interceptor (so the caller is already authenticated with a canonical user id in context); it resolves `IsAdmin` for
that caller and, on false, returns the canonical **permission-denied** RPC error (distinct from plan 04's
_unauthenticated_). Every `admin.v1` method except none is exempt — the whole service is admin-only.
`GetAdminSelf` returns `{ isAdmin }` for the caller so the FE can gate the route and render its affordance without
guessing.

### AI provider config — runtime DB override over env (the plan-28 change)

Today ([28](28.ai-provider-abstraction.md)) `NewAdaptersFromEnv` reads `COSIMOSI_LLM_*` / `COSIMOSI_EMBEDDING_*` **once
at startup** and builds the vendor clients. This unit keeps env as the **fallback/default** and adds a DB override that
takes effect **without redeploy**:

- **`ai_provider_config` table** — one row per capability (`capability` ∈ `{llm, embedding}` PK): `provider`, `model`,
  `base_url`, `api_key_encrypted` (nullable), `key_set` (bool), `updated_by`, `updated_at`. Not user-scoped (it is
  service-global operational state, like `universe`/config, not product data).
- **Effective-config resolution** — a `ProviderConfigSource` resolves each capability as: **DB row if present → else
  env → else the keyless deterministic mock** (unchanged plan-22/28 mock fallback). The factory moves from
  build-once to a **resolver that rebuilds the vendor client when the effective config's fingerprint changes** (cached
  by fingerprint; a `SetAIConfig` bumps it). The metered wrapper, error taxonomy, and port adapters are unchanged — the
  swap happens strictly _below_ the metering seam, so caps/caching/error-normalization behave identically.
- **`SetAIConfig(capability, provider, model, baseURL, apiKey?)`** — validates `provider` is a **known + implemented**
  slot via the existing registry (reusing plan 28's `ErrUnknownProvider` / `ErrProviderNotImplemented` semantics — an
  unknown or unimplemented name is **rejected**, never silently applied); for embeddings, validates the model can honor
  `ai.embedding_dim`. When `apiKey` is provided it is **encrypted at rest** (AES-GCM, key from a server-only env secret,
  e.g. `ADMIN_CONFIG_ENC_KEY`) and stored; `key_set` becomes true. Omitting `apiKey` on a later edit keeps the stored
  key.
- **`GetAIConfig`** — returns provider/model/baseURL and `keySet` + a short non-reversible hint (e.g. last-4) only; the
  **plaintext key is never returned** to the client (write-only secret). If no DB row exists it reports the env-derived
  effective config as the current state (marked as "from env").
- **Security posture** — the encryption key is env-only (never in DB/UI); keys are write-only across the RPC boundary;
  every `SetAIConfig` writes an `admin_audit_log` row (who/when/what capability + provider, **never** the key).

This is the deliberate supersession of [28](28.ai-provider-abstraction.md)'s "provider is an env-only ops choice, not
runtime-switchable." Env remains valid and is the fallback; the DB simply wins when set. Recorded in Policy/Values →
_Plan-28 delta_ and reconciled into [28](28.ai-provider-abstraction.md) + its tech doc at `/implement-job`.

### User list — account metadata only ([I2], the authorized cross-user read)

`ListUsers(page, query)` returns, per user: `id`, `email`, `signupAt`, `isAdmin` (+ seed/promoted), balance
`{ basic, additional, total }`, and non-content counts (`diaryCount`, `starCount`). It joins:

1. the `AccountDirectory` (Supabase Auth Admin API) for identity + signup date + the enumerable user set,
2. `StardustAdminPort.Balance` for the balance,
3. `MemoryStatsPort.Counts` for the counts (**counts only**, no content field exists on the port),
4. the admin resolver for status.

This is the codebase's **one authorized cross-user surface**: §4 per-user isolation scopes _product_ reads to the
caller; the admin reads are cross-user **by design and gated by admin authorization**. The privacy line is held not by
scoping but by **shape** — no admin RPC returns memory content, and the stats port cannot carry any ([I2]). Pagination
default `admin.user_list_page_size`; `query` matches email/id prefix.

### Grant stardust (별가루 증정) — a new `admin_grant` earn

`GrantStardust(targetUserId, amount, note)`:

1. authorization interceptor already proved the caller is an admin;
2. validate `0 < amount ≤ twinkle.admin_grant_max` (fat-finger guard);
3. call `twinkle.Earn(scope=targetUser, reason=admin_grant, amount)` — credits **additional** (permanent) balance,
   append-only, **idempotent** via a dedup key derived from a client-supplied grant id ([43]/[44] semantics);
4. write an `admin_stardust_grants` audit row (`id`, `granted_by`, `target_user`, `amount`, `note`, `created_at`) and an
   `admin_audit_log` entry, in the same transaction as the earn.

`ListStardustGrants(page)` reads `admin_stardust_grants` (who granted, to whom, how much, note, when) for
accountability — the lightweight grant history folded into the grant feature. The grant **spends nothing, prices
nothing, deletes nothing** ([I1][I3]) — it only credits a scalar balance through the sanctioned earn path.
`admin_grant` is added to the twinkle earn-reason enum (code content, not a value); `twinkle`'s domain is otherwise
untouched.

### Operational dashboards (read-only)

- **`GetAIUsage`** — returns today's per-capability call count vs `ai.daily_call_cap` and the `ai.per_call_token_cap`
  in effect, from the plan-28 `Meter` snapshot via `AIUsageReader`. **Stated v1 limitation:** the `Meter` is
  in-process/in-memory and per-user for cap enforcement; the dashboard therefore reflects the **current API process's**
  window, not a durable cross-process aggregate. A persistent usage warehouse is a later plan (Non-goals). The RPC
  surfaces exactly what the meter knows, honestly labeled.
- **`GetJobHealth`** — aggregate counts over the `jobs` table by `status` (pending/running/done/failed), plus
  backoff/attempts summary and dead-lettered (`job_max_claims`) count, via `JobHealthReader`. A global operational read
  (not user-scoped), read-only, no mutation of the queue.

### RPC contract (`admin.v1.AdminService`, Connect, unary — §2.7)

Thin handlers map proto↔domain and call the admin use-cases; all policy (authorization, grant cap, config validation,
seed-admin protection) lives in the use-case/domain (§2.9 #7). Sketch (exact proto confirmed at implement):

```proto
service AdminService {
  rpc GetAdminSelf(GetAdminSelfRequest) returns (GetAdminSelfResponse);       // { isAdmin }
  // admin management
  rpc ListAdmins(ListAdminsRequest) returns (ListAdminsResponse);
  rpc GrantAdmin(GrantAdminRequest) returns (GrantAdminResponse);             // promote a user
  rpc RevokeAdmin(RevokeAdminRequest) returns (RevokeAdminResponse);          // refused for seed admins
  // users + grant
  rpc ListUsers(ListUsersRequest) returns (ListUsersResponse);               // metadata only, paginated, searchable
  rpc GrantStardust(GrantStardustRequest) returns (GrantStardustResponse);   // admin_grant earn, capped, idempotent
  rpc ListStardustGrants(ListStardustGrantsRequest) returns (ListStardustGrantsResponse);
  // AI config (runtime override over env)
  rpc GetAIConfig(GetAIConfigRequest) returns (GetAIConfigResponse);         // key masked; never plaintext
  rpc SetAIConfig(SetAIConfigRequest) returns (SetAIConfigResponse);         // validates slot + embedding dim; encrypts key
  // dashboards (read-only)
  rpc GetAIUsage(GetAIUsageRequest) returns (GetAIUsageResponse);
  rpc GetJobHealth(GetJobHealthRequest) returns (GetJobHealthResponse);
}
```

Read RPCs (`GetAdminSelf`, `List*`, `Get*Usage/Health`, `GetAIConfig`) are `NO_SIDE_EFFECTS`; the mutations
(`GrantAdmin`/`RevokeAdmin`/`GrantStardust`/`SetAIConfig`) are idempotent per their key and each write an
`admin_audit_log` row. No proto/sqlc type reaches the use-case or pure domain (§2.4).

### FE — web-only `pages/admin` (FSD §3.1, parity waived §3.5)

- **Route** — web `/admin`, registered in the plan-15 typed route registry (`apps/web/src/app/routes`).
- **Admin gate** — extends the [53](53.auth-universe-gate.md) auth gate: authenticated **and** `GetAdminSelf().isAdmin`;
  a non-admin (or anonymous) is redirected away / shown not-found. The gate is the FE mirror of the BE interceptor (BE
  is authoritative; FE gate is UX). A `/admin` affordance appears only for admins.
- **Features** — `features/admin-ai-config` (get/set provider config; key input write-only, masked display),
  `features/admin-users` (paginated searchable list + `GrantStardust` + `GrantAdmin`/`RevokeAdmin` inline +
  `ListStardustGrants`), `features/admin-usage` (`GetAIUsage`), `features/admin-jobs` (`GetJobHealth`). The page
  **composes** features and holds no domain logic (§3.1). Transient submit/in-flight state is control-state (a small
  XState machine or trivial flag under the feature's `model/`, §3.2).
- **Copy** — all strings via the i18n seam ([08](08.i18n-seam.md)); admin console copy may be plain/utilitarian (an
  operator tool), not the product's literary Korean voice.
- **Mobile** — none. The parity rule is waived and this is stated as a non-goal.

### Re-implementation note

The console is fully specified by: an authorization predicate (`env-seed ∪ admin_users` rows, seed undemotable) gating
an `admin.v1` service; a per-capability `ai_provider_config` row (encrypted key) that overrides env in the factory's
effective-config resolver (rebuild-on-change), else env, else mock; a metadata-only cross-user user list joining
Supabase Auth identity with balance + non-content counts; a capped, idempotent `admin_grant` earn crediting additional
balance with an audit row; and two read-only snapshots (meter usage, jobs-table health). Any client reproduces the UI
from those RPCs; the BE interceptor + earn path are always authoritative.

   imports `dbgen` (§2.2/§2.4, CC8).
2. An **admin-authorization interceptor** runs after plan 04's auth interceptor and rejects any non-admin caller of any
   `admin.v1` method with the canonical **permission-denied** error (distinct from _unauthenticated_); an anonymous
   caller is rejected as unauthenticated by plan 04.
3. `IsAdmin` = the caller's user id/verified email ∈ `ADMIN_USER_IDS` (env seed, parsed at the composition root)
   **or** a row in `admin_users`. `GrantAdmin`/`RevokeAdmin` add/remove `admin_users` rows; a `RevokeAdmin` against a
   **seed** admin is **refused** (seed admins are undemotable). `GetAdminSelf` returns the caller's admin status.
4. `SetAIConfig` writes an `ai_provider_config` row (per capability) with the API key **encrypted at rest**; it
   **rejects** an unknown or unimplemented provider slot (reusing plan-28 registry semantics) and an embedding model
   that cannot honor `ai.embedding_dim`. `GetAIConfig` returns provider/model/baseURL + `keySet` (+ masked hint) and
   **never** returns a plaintext key.
5. The AI factory resolves **effective config = DB row → env → keyless mock**, and a `SetAIConfig` change takes effect
   **without a redeploy** (the vendor client rebuilds on config change); the metering, error taxonomy, port adapters,
   and mock fallback from [28](28.ai-provider-abstraction.md)/[22](22.ai-worker-pipeline.md) are unchanged.
6. `ListUsers` returns **account metadata only** — id, email, signup date, admin status, stardust balance, and
   non-content counts (# diaries/# stars) — paginated (`admin.user_list_page_size`) and searchable; **no** admin RPC
   returns any diary text, emotion, position, connection, or memory content, and the `MemoryStatsPort` return type has
   no content field ([I2]).
7. `GrantStardust(target, amount, note)` credits the target's **additional** balance via `twinkle.Earn` with the new
   `admin_grant` reason, is **capped** by `twinkle.admin_grant_max`, is **idempotent** (dedup by grant id), writes an
   `admin_stardust_grants` audit row + an `admin_audit_log` entry in the same transaction, and **deletes/prices/spends
   nothing** ([I1][I3]). `ListStardustGrants` reads the grants back.
8. `GetAIUsage` returns today's per-capability call counts against `ai.daily_call_cap`/`ai.per_call_token_cap` from the
   plan-28 meter snapshot (read-only), with its in-process/in-memory scope honestly reflected; `GetJobHealth` returns
   aggregate `jobs`-table status/backoff/dead-letter counts (read-only, non-mutating).
9. Every admin mutation (`GrantAdmin`/`RevokeAdmin`/`SetAIConfig`/`GrantStardust`) writes an append-only
   `admin_audit_log` row; the log never records a plaintext API key, and is never `UPDATE`d/`DELETE`d by the system
   ([I1]).
10. A **web-only** `/admin` route exists (plan-15 registry) behind an **admin gate** (authenticated **and** admin);
    a non-admin cannot reach it; the affordance shows only for admins. **No mobile admin surface exists** and the
    parity rule is documented as waived.
11. The `pages/admin` screen **composes** `features/admin-ai-config`/`admin-users`/`admin-usage`/`admin-jobs`, holds no
    domain logic, and all user-visible copy resolves through the i18n seam (`lint:raw-strings` passes) (§3.1).
12. Goose migrations create `admin_users`, `ai_provider_config`, `admin_stardust_grants`, and `admin_audit_log`;
    `pnpm db:migrate` applies cleanly and `pnpm db:down` reverses cleanly; the service-global tables
    (`ai_provider_config`) are correctly excluded from the per-user `lint:persistence` gate (or scoped where they carry
    a user id), and it passes.

## Policy / Values Impact

- **policy/** — a new `policy/ops/admin.md` lands at `/implement-job` (do **not** pre-create) with the rules this unit
  makes true:
  - **Authorization** — admin = `ADMIN_USER_IDS` (env seed, undemotable trust anchor) ∪ `admin_users` (DB-promoted);

    logged.
