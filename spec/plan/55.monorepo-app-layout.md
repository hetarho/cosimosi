# 55. monorepo app layout

> 제품 앱을 `apps/web`·`apps/api`·`apps/blog`로 승격하고, 미래 공유 코드가 들어갈 `packages/*` 경계를 마련한다. `proto/`는 루트 유지. | 범위: Infra/FS | 상태: 구현 완료

## 목적

리포는 모노레포다. 제품 앱은 `apps/`(web·api·blog, 미래 mobile) 아래에, 공유 패키지 경계는 `packages/`에 있어 "무엇이 앱이고 무엇이 공유 패키지인지"가 구조에서 먼저 보인다 — 모바일 앱 추가와 WebGL/TSL/R3F 렌더링 헬퍼·디자인 시스템 공유를 대비한 형태다.

이 구조는 제품 동작을 바꾸지 않는다 — 저장소 형태만 정리한 무동작 리팩터다. 핵심 결정은 API 서버를 `apps/web` 하위에 넣지 않는 것이다. Go 백엔드는 웹 전용이 아니라 웹·미래 모바일이 함께 쓰는 cosimosi 제품 API이므로 `apps/api`라는 독립 앱으로 둔다. `proto/`는 웹·API·미래 모바일이 공유하는 계약 원천이므로 루트에 둔다(`buf.gen.yaml` 포함).

## 범위 / 비목표

범위:

- 루트 앱 디렉터리를 다음 구조로 재배치한다.

  ```text
  apps/
    web/        # 현재 frontend, Vite/React/R3F 앱
    api/        # 현재 backend, Go Connect API + worker
    blog/       # 현재 blog, Astro 정적 블로그
    mobile/     # 미래 모바일 자리. 이번 계획에서는 README/placeholder만 둔다.
  packages/
    README.md   # 공유 패키지 승격 기준. 실제 패키지는 재사용 소비자가 생길 때 만든다.
  proto/         # 루트 유지. Connect/Protobuf 계약의 단일 원천.
  ```

- `pnpm-workspace.yaml`, 루트 `package.json` scripts, codegen, DB migration runner, Docker/Compose, GitHub Actions, 배포 워크플로의 경로를 새 구조에 맞춘다.
- `frontend/package.json`은 `apps/web/package.json`, `blog/package.json`은 `apps/blog/package.json`으로 이동하고 package name은 각각 `@cosimosi/web`, `@cosimosi/blog`처럼 앱 정체성이 보이게 바꾼다.
- `backend/go.mod`는 `apps/api/go.mod`로 이동하지만 Go module path(`github.com/cosimosi/backend`)는 이번 범위에서 유지한다. import path 대량 변경은 구조개편의 본질이 아니며 별도 변화로 분리할 수 있다.
- `proto/`는 루트에 그대로 둔다. 다만 `buf.gen.yaml`은 현재 `backend/` 소유처럼 보이지 않도록 `proto/buf.gen.yaml` 또는 동등한 루트 계약 위치로 옮긴다.
- `packages/`는 이번 계획에서 실제 공유 패키지를 만들지 않는다. WebGL/TSL/R3F 헬퍼, 디자인 시스템, 순수 memory-physics 같은 후보와 승격 기준만 문서화한다.

비목표:

- 모바일 앱을 구현하거나 React Native/Flutter 프로젝트를 스캐폴드하지 않는다.
- WebGL 관련 라이브러리, 디자인 시스템, `shared/lib` 코드를 바로 `packages/*`로 추출하지 않는다.
- RPC/proto 필드, DB schema/migration, API route, 인증/세션 동작, 제품 UX를 바꾸지 않는다.
- API Go module path를 rename하지 않는다.
- 배포 도메인, Cloudflare/VPS/Supabase 실환경 값을 바꾸지 않는다.

## 설계 요점

### 앱 경계

- `apps/web`은 브라우저 제품 앱이다. 기존 FSD 경계(`app/pages/widgets/features/entities/shared`)와 `@/` alias는 앱 내부 규칙으로 유지한다.
- `apps/api`는 웹 하위 서버가 아니라 제품 API 앱이다. 웹과 미래 모바일 모두 이 앱의 Connect RPC를 호출한다. `cmd/api`, `cmd/worker`, `internal/*`, `sqlc.yaml`, Dockerfile들은 이 앱 아래에 산다.
- `apps/blog`는 Astro 정적 콘텐츠 앱이다. 현재처럼 `spec/blog.md`를 콘텐츠 SSOT로 읽되, 이동 후 상대 경로만 새 위치에 맞춘다.
- `apps/mobile`은 이번 작업에서 placeholder만 둔다. 실제 앱은 별도 plan에서 기술한다.

### 공유 패키지 경계

- `packages/`는 만들어 두되 빈 패키지를 미리 만들지 않는다. 공유 패키지는 두 번째 실제 소비자가 생기는 순간 promote-on-reuse로 만든다.
- 후보:
  - `packages/rendering` 또는 `packages/webgl` 후보: TSL 노드 조립, WebGPU renderer 포트, R3F 헬퍼, shader-art toolkit. 단, 모바일 렌더러가 확정되기 전에는 app-local 유지가 기본이다.
  - `packages/ui` 또는 `packages/design-system` 후보: 웹과 블로그가 같은 토큰/기초 컴포넌트를 실제로 공유하게 될 때 승격한다.
  - `packages/core` 후보: 앱 독립 순수 로직(memory physics, values 해석, mood utilities 등)이 web 밖에서 필요해질 때 승격한다.
- 앱끼리 서로의 `src`를 relative import하지 않는다. 공유가 필요하면 `packages/*` 또는 루트 `proto/` 계약을 통해서만 공유한다.

### 루트 계약과 codegen

- `proto/`는 루트에 유지한다. 이는 API 앱 소유물이 아니라 모든 클라이언트가 공유하는 transport 계약이다.
- `buf.gen.yaml`은 `proto/buf.gen.yaml` 같은 계약 위치로 이동하고, 출력 경로만 새 앱 구조로 바꾼다.
  - Go: `apps/api/internal/gen/...`
  - TS: `apps/web/src/shared/api/gen/...`
- `scripts/gen.mjs`, `scripts/lib.mjs`, `scripts/gen-values.mjs`는 `backend`/`frontend` 하드코딩을 `apps/api`/`apps/web`으로 갱신한다.
- `spec/values.yaml`은 루트 유지. 생성 산출물만 `apps/api/internal/values`와 `apps/web/src/shared/config`로 간다.

### 로컬 개발/빌드/배포 경로

- `pnpm-workspace.yaml`은 `apps/web`, `apps/blog`, `packages/*`를 포함한다. `apps/api`는 Go module이라 pnpm workspace 패키지로 취급하지 않는다.
- 루트 scripts는 새 경로를 단일 진입점으로 유지한다.
  - `dev`: `apps/web` dev server + Docker compose API
  - `dev:web`: `pnpm --filter @cosimosi/web dev`
  - `dev:blog`: `pnpm --filter @cosimosi/blog dev`
  - `build:web`, `build:blog`: 새 package name 또는 path selector 사용
  - `build:api`: `cd apps/api && go build ...`
- `docker-compose.yml`은 API build context, source volume, tmp/go-mod volume 이름을 새 위치에 맞춘다. 외부 포트와 서비스 의미는 유지한다.
- `.github/workflows/ci.yml`은 web, blog, api를 앱 단위로 검증한다. 기존 web lint/build와 api vet/build를 유지하고, blog build도 앱으로 포함한다.
- `.github/workflows/deploy-backend.yml`은 path filter, Docker build context, migration sync source를 `apps/api/**` 기준으로 갱신한다.
- Cloudflare Workers 정적 자산 설정(`wrangler.jsonc`가 있으면)은 `apps/web/dist`를 가리키게 한다.

### 구현 순서

1. `apps/`와 `packages/`를 만들고 tracked app 파일을 `git mv`로 이동한다. `node_modules`, `dist`, tmp, 로컬 로그는 이동 대상이 아니다.
2. workspace/scripts/codegen/compose/CI/deploy 경로를 모두 새 구조로 바꾼다.
3. app 내부 상대 경로를 수정한다. 특히 `apps/blog/src/pages/index.astro`의 `spec/blog.md` import와 `apps/web/vite.config.ts`의 repo-root env 탐색 경로를 확인한다.
4. `pnpm install --lockfile-only` 또는 동등한 lockfile 갱신으로 workspace path/name 변화를 반영한다.
5. `pnpm gen`, web/blog build, FE lint/FSD, API Go vet/build를 통과시킨다.
6. `spec/tech/architecture.md`, `spec/tech/quality-gates.md`, `spec/ops/deploy-cicd.md`의 as-built 경로 설명을 새 구조로 반영한다.

## 수용 기준 (인수 조건)
1. tracked 제품 앱 코드는 루트 `frontend/`, `backend/`, `blog/`가 아니라 `apps/web`, `apps/api`, `apps/blog` 아래에 있다.
2. `apps/api`는 웹 하위 디렉터리가 아니며, 제품 API 앱으로 독립해 있다.
3. 루트 `proto/`는 유지되고, `pnpm gen`이 루트 proto 입력에서 `apps/api/internal/gen`과 `apps/web/src/shared/api/gen`을 생성한다.
4. `packages/README.md`가 존재하고 WebGL/TSL/R3F 헬퍼·디자인 시스템·순수 core 로직의 승격 기준을 설명하지만, 실제 공유 패키지는 이번 작업에서 만들지 않는다.
5. `pnpm dev`, `pnpm dev:web`, `pnpm dev:blog`, `pnpm build:web`, `pnpm build:blog`, `pnpm build:api`가 새 경로에서 동작한다.
6. `docker-compose.yml`의 API build context와 source volume은 `apps/api`를 가리키며, postgres/API/web 포트 의미는 기존과 같다.
7. CI는 web lint/build, blog build, api vet/build를 새 경로 기준으로 실행한다.
8. backend deploy workflow는 `apps/api/**` 변경에 반응하고, Docker build context와 migration sync source도 `apps/api` 기준이다.
9. `apps/blog`는 기존처럼 `spec/blog.md`를 콘텐츠 원천으로 빌드한다.
10. 앱 간 직접 relative import가 없다. 공유 코드가 필요하면 `packages/*` 또는 루트 `proto/` 계약으로만 연결한다.
11. API Go module path는 이번 작업 후에도 빌드 가능하고, import rewrite 없이 `go vet ./...`와 `go build ./...`가 `apps/api`에서 통과한다.
12. 제품 동작, RPC/proto 필드, DB schema/migration, 인증/세션 정책은 이 구조개편으로 바뀌지 않는다.

## 영향 파일

- `frontend/**` → `apps/web/**`
- `backend/**` → `apps/api/**`
- `blog/**` → `apps/blog/**`
- 신규 `apps/mobile/README.md`
- 신규 `packages/README.md`
- `pnpm-workspace.yaml`
- `package.json`, `pnpm-lock.yaml`
- `scripts/gen.mjs`, `scripts/gen-values.mjs`, `scripts/lib.mjs`, `scripts/db.mjs`, `scripts/setup.mjs`
- `proto/**`, `backend/buf.gen.yaml` → `proto/buf.gen.yaml`
- `docker-compose.yml`, `docker-compose.prod.yml`(존재 시)
- `.github/workflows/ci.yml`, `.github/workflows/deploy-backend.yml`
- `wrangler.jsonc`(존재 시)
- `deploy/**` 중 API 경로를 참조하는 파일
- `spec/tech/architecture.md`, `spec/tech/quality-gates.md`, `spec/ops/deploy-cicd.md`, `spec/plan/00.overview.md`

## 정책 영향 (policy / values)

- `policy/**`: 제품 도메인/UX 규칙 변화 없음.
- `spec/values.yaml`: 신규 튜닝 수치 없음.
- 구현 완료 시 `tech/architecture.md`는 새 모노레포 구조를 현재 truth로 반영해야 한다.
- 구현 완료 시 `tech/quality-gates.md`와 `ops/deploy-cicd.md`의 모든 `frontend`/`backend`/`blog` 경로는 `apps/web`/`apps/api`/`apps/blog` 기준으로 갱신해야 한다.
