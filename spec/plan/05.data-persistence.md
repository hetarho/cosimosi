# 05. data persistence

> Data persistence - Postgres, pgvector, Goose migrations, sqlc generation, and persistence port/adapter conventions.
> Scope: BE | Status: built

## Purpose

The backend needs a persistence seam before domain units start adding tables. This unit installs the database stack,
establishes migration and sqlc conventions, and proves that the API can connect to Postgres with pgvector available.
It does not define product data yet; it makes later domain tables safe to add.

## Scope / Non-goals

**In scope**
- Local Postgres with pgvector through Docker Compose.
- Database configuration loading in `apps/api` with a clear split between runtime environment and generated product
  values.
- Goose migration layout and root scripts for up/status/down/reset.
- Initial platform migration for required extensions and database sanity only.
- sqlc configuration, query layout, generated-code location, and `pnpm gen:sql` integration.
- Backend DB pool helper under the platform layer.
- Persistence port/adapter conventions: consuming behavior owns interfaces when needed; `<context>/pg` implements them;
  sqlc rows never leave generated/pg infrastructure.
- Per-user persistence convention: future product tables carry `user_id` and future queries scope by it.

**Non-goals**
- No product tables such as records, engrams, cells, synapses, jobs, embeddings, or flags.
- No repository implementation for a product aggregate.
- No Supabase Auth setup beyond consuming the user-id contract from [04.auth-session](04.auth-session.md).
- No production database provisioning, backups, RLS, connection pooling service, or cloud deployment.
- No ORM and no generic repository wrapper over sqlc.

## Design

### Database runtime

Local development uses Docker Compose for Postgres with pgvector installed:

- `docker-compose.yml` defines a `postgres` service using `pgvector/pgvector:0.8.3-pg17`.
- The service exposes `localhost:5432`, stores data in the Phase 1 `api_postgres_data` volume, and has a
  `pg_isready` healthcheck.
- The API container receives `DATABASE_URL=postgres://cosimosi:cosimosi@postgres:5432/cosimosi?sslmode=disable`.
- Host-side tools and tests use `postgres://cosimosi:cosimosi@localhost:5432/cosimosi?sslmode=disable`.
- `scripts/db.mjs` uses `COSIMOSI_MIGRATION_DATABASE_URL` when set, otherwise the Compose-network `postgres` URL.

These settings are not `spec/values.yaml` entries because they vary by deployment environment and are not product
tuning constants.

### Migration layout

The backend owns migrations in the active API root:

```text
apps/api/db/
├── migrations/
│   └── 00001_platform_extensions.sql
├── queries/
│   └── platform/persistence_probe.sql
└── gen/                  sqlc output, package `dbgen`
```

The first migration enables only the `vector` extension and creates no product tables. Goose is the migration runner
and root scripts delegate to `scripts/db.mjs`:

- `pnpm db:migrate` applies all migrations.
- `pnpm db:status` reports applied/pending migrations.
- `pnpm db:down` rolls back one migration.
- `pnpm db:reset` resets to zero and reapplies the clean schema.

Migration files must be forward-only safe and include down migrations where Goose format supports them. The job that
implements a product unit owns that unit's schema migration; this platform unit owns only the empty database seam.

### sqlc layout

`apps/api/sqlc.yaml` points at:

- schema input from `apps/api/db/migrations/*.sql`;
- query files grouped under `apps/api/db/queries/**/*.sql`;
- generated Go under `apps/api/db/gen` as package `dbgen`;
- `sql_package: pgx/v5`.

`pnpm gen:sql` runs `sqlc/sqlc:1.31.1` when `apps/api/sqlc.yaml`, migration SQL, and query SQL all exist. Before all
three inputs exist, the script skips cleanly. The initial generated query is a platform persistence probe (`SELECT 1`);
it exists to prove the codegen path before product tables exist.

Generated sqlc packages are infrastructure. Domain model and context behavior packages cannot import sqlc rows.
Context-specific `<context>/pg` packages map rows to domain objects and implement consumer-owned interfaces declared
where they are used (the context package while small, or `<context>/app` after a split).

### DB pool and health

`apps/api/internal/platform/db` owns:

- `ConfigFromEnv`, which reads `DATABASE_URL`;
- `Open`, which constructs a `pgxpool.Pool`;
- `Ping` and `Close` for lifecycle checks;
- `PgxPool` for infrastructure adapters that need the raw pool;
- `InTx`, which opens a `pgx.Tx` and commits or rolls back around repository work.

The API runtime is not forced to open the database during boot, so ordinary `/health` remains usable for local
bootstrapping. DB readiness is proven by Docker-backed Go tests rather than coupled to the public health route.

### Per-user convention

The platform documents and tests a future-table convention:

- product tables have a `user_id` column unless explicitly platform-owned;
- persistence methods require `platform.UserScope` from the auth context added in [04.auth-session](04.auth-session.md);
- cross-user queries are not allowed by default;
- RLS is deferred, but schema and query design must not block adding it later.
- `pnpm lint:persistence` scans product migrations and sqlc query files so new product tables/queries cannot skip the
  `user_id` convention without an explicit platform-owned exemption.

The non-product `apps/api/internal/persistenceprobe` package pins this boundary without creating domain tables:
`persistenceprobe.Store` is declared by the consuming package, while `persistenceprobe/pg.Store` implements it and
requires a `platform.UserScope` before touching the database pinger. Future product contexts follow the same shape
with their own domain mapping and sqlc-backed `<context>/pg` adapters.

## Acceptance Criteria

1. `docker compose up` starts a local Postgres service with pgvector available.
2. `pnpm db:migrate`, `pnpm db:status`, `pnpm db:down`, and `pnpm db:reset` target the clean `apps/api` database path.
3. The initial migration enables required platform extensions and creates no product domain tables.
4. `pnpm gen:sql` runs sqlc when schema/query inputs exist and skips cleanly before they do.
5. Generated sqlc code is placed under an infra-only package path and is not imported by domain model/context behavior.
6. The API platform DB helper can open a pool, ping the database, and close cleanly in tests.
7. Persistence port/adapter examples or tests show consumer-owned interfaces declared where used and implemented by
   `<context>/pg` packages when an interface is actually needed.
8. The persistence convention requires future product tables and queries to be scoped by authenticated user id, and the
   root lint gate enforces the convention before product tables arrive.
9. No ORM, server-authoritative client-derived layout state, or direct cross-context table access is introduced.
