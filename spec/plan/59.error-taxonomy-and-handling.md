# 59. error taxonomy and handling

> A service-wide **error contract**: a stable, machine-readable domain error-code taxonomy carried BE→FE over
> Connect **error details**, an explicit **expected (domain) vs unexpected (internal)** split, environment-gated
> internal-error detail, a client-facing **correlation id** on every failure, and a shared FE error seam
> (parse → i18n message → toast/boundary). Replaces today's "everything unmapped becomes `{code:"internal",
message:"internal server error"}`" opacity — where the FE can neither branch on _which_ domain rule failed nor
> report an id an operator can trace. Scope: FE|BE|FS | Status: ✅ built (job 78)
>
> Home (BE): the error contract lives in `internal/platform` (the interceptor chain + a new `platform/apperr`
> helper) and `proto/cosimosi/platform/v1` (a shared `ErrorInfo` detail message); each bounded context's
> `rpc/*.go` adapter keeps owning its own error→code mapping (CC8 — the taxonomy is shared, the mappings are
> per-context). Home (FE): a shared `packages/errors` seam (Connect error → typed `AppError`), reason-code→copy
> keys in `packages/i18n`, and an app-layer error boundary + toast hook wired in `apps/web` with `apps/mobile`
> parity.

## Purpose

Right now **every unexpected failure reaches the client as `{code:"internal", message:"internal server error"}`**
(observed in prod: `GetBalance`, `SyncStatus`, `GetUniverse`, `GetPalettePreference` all return it). Two distinct
problems hide behind that one string:

1. **Domain errors are indistinguishable.** Each context maps its errors onto coarse Connect codes
   ([twinkle rpc `domainError`](../../apps/api/internal/twinkle/rpc/server.go#L118), [memory rpc
   `domainError`](../../apps/api/internal/memory/rpc/server.go#L293)), but many distinct domain errors collapse onto
   the **same** Connect code — e.g. memory maps `ErrRecallMemoryUnavailable`, `ErrAlreadyReleased`,
   `ErrSyncConsentRequired`, and four others all to `FailedPrecondition`. The FE receives one code and **cannot tell
   which precondition failed**, so it cannot show the right message or take the right recovery action (open the
   consent modal vs. show "already released"). This is why the app feels blind even when the backend behaved
   correctly.

2. **Internal errors are untraceable from the client.** Masking the raw error is **correct** (leaking
   `pq: column … does not exist`, SQL, or stack traces is an information-disclosure risk), but today the client gets
   **no correlation id**, so a user-reported failure cannot be matched to the log line ([the raw error _is_ logged
   server-side](../../apps/api/internal/platform/interceptors.go#L56)) or the Sentry event. Support and debugging
   have nothing to join on.

The goal is **not** "expose everything" — it is: **domain errors carry a precise, stable code to the client;
internal errors carry a correlation id (and, off-prod, the real detail).**

## Scope / Non-goals

**In scope**

- A shared **`platform.v1.ErrorInfo`** proto message, attached as a Connect **error detail** on every RPC failure
  (both domain and internal), carrying: stable `reason` code, `domain` (owning context), `request_id`
  (correlation), safe `metadata`, and an off-prod-only `debug_detail`.
- A **domain error-code taxonomy**: one `SCREAMING_SNAKE` reason per distinct domain error, namespaced by context
  (`TWINKLE_INSUFFICIENT`, `MEMORY_SYNC_CONSENT_REQUIRED`, …). The stable string is the contract; the Connect code
  stays the coarse transport signal.
- A **`platform/apperr`** BE helper that builds a Connect error with the `ErrorInfo` detail attached and applies the
  expected/internal masking rule uniformly, replacing ad-hoc `connect.NewError` at the `default` branches.
- Refactor each context's `rpc/*.go` `domainError` to emit `(connectCode, reason)` via `apperr` — **no more
  raw-error `default`** that silently becomes `internal`.
- Interceptor changes ([interceptors.go](../../apps/api/internal/platform/interceptors.go)): `StructuredErrorInterceptor`
  attaches `request_id` to `ErrorInfo` on **every** error, masks message + omits `debug_detail` **only** for
  Internal/Unknown, and gates `debug_detail` on environment. `PanicRecoveryInterceptor` likewise emits an
  `ErrorInfo{reason:"INTERNAL", request_id}`.
- A shared FE **`packages/errors`** seam: `toAppError(unknown) → AppError { code, reason, domain, requestId,
metadata, debugDetail, retriable }`, decoding the `ErrorInfo` detail.
- **reason-code → user copy** keys in `packages/i18n` (localized per-reason message + a generic INTERNAL fallback
  that shows the correlation id), and a shared **toast + error-boundary** policy wired at the app layer (web +
  mobile parity), plus a **correlation-id display** for internal errors.

**Non-goals**

- **No change to _which_ situations are errors**, to the domain logic, or to the Connect **code** semantics already
  in use — this plan adds a parallel stable-reason channel and standardizes transport, it does not re-litigate
  business rules.
- **Not** a logging/metrics overhaul: Sentry capture + structured container logs already exist and are kept as-is
  (only enriched with the reason/correlation fields).
- **No new retry semantics**; the existing client-cache retry policy
  ([http-policy.ts](../../packages/client-cache/src/http-policy.ts)) is reused — this plan only exposes a
  `retriable` classification for FE display decisions.
- **No i18n copy for every one of the 100+ backend `errors.New` values** — only for the reason codes a FE surface
  actually branches on; the rest fall back to their coarse-code generic message.

## Design

### D1 — Two error classes (the load-bearing decision)

Every failure is exactly one of:

| Class                     | Examples                                                       | Connect code                 | Message to client                                         | `debug_detail`                                  | Reason code                          |
| ------------------------- | -------------------------------------------------------------- | ---------------------------- | --------------------------------------------------------- | ----------------------------------------------- | ------------------------------------ |
| **Expected (domain)**     | insufficient twinkle, not found, consent required, invalid arg | the mapped non-Internal code | the domain message (safe, English; FE overrides via i18n) | —                                               | precise, e.g. `TWINKLE_INSUFFICIENT` |
| **Unexpected (internal)** | DB down, schema drift, nil panic, any unmapped `default`       | `Internal`                   | generic `"internal server error"`                         | **prod: empty · dev/staging: raw error string** | `INTERNAL`                           |

Both classes **always** carry `request_id`. The distinction is decided by the Connect code after the context's
mapping: non-Internal/Unknown ⇒ domain; Internal/Unknown ⇒ internal. **The current bug is that too much lands in
the internal bucket** because every `domainError` has a raw-error `default`; D3 removes those.

### D2 — Proto: the shared `ErrorInfo` detail

Add to [`proto/cosimosi/platform/v1/platform.proto`](../../proto/cosimosi/platform/v1/platform.proto):

```proto
// ErrorInfo is attached as a Connect error *detail* on every failed RPC. The Connect code stays the coarse
// transport signal; `reason` is the stable, machine-readable domain contract the FE branches on.
message ErrorInfo {
  string reason = 1;                 // stable SCREAMING_SNAKE code, e.g. "TWINKLE_INSUFFICIENT" | "INTERNAL"
  string domain = 2;                 // owning context: "twinkle" | "memory" | "account" | "admin" | "platform"
  string request_id = 3;             // correlation id (mirrors X-Request-Id) — present on EVERY error
  map<string, string> metadata = 4;  // safe structured context only (never diary/memory content, never secrets)
  string debug_detail = 5;           // raw error string; populated OFF-PROD only, always empty in production
}
```

Carried via Connect's error-details mechanism (`*connect.Error.AddDetail` with the marshaled `ErrorInfo`), so it
rides the standard Connect envelope and is readable by any Connect client (web, mobile, future Flutter).

### D3 — BE: `platform/apperr` + per-context mappings

- New `internal/platform/apperr` package:
  - `apperr.Domain(code connect.Code, reason string, err error, meta map[string]string) error` — builds a
    `*connect.Error` with the `ErrorInfo` detail (reason/domain/metadata) attached. `domain` is inferred from the
    reason prefix or passed explicitly.
  - `apperr.Internal(err error) error` — the single constructor for the internal bucket (used at every former
    raw-`default`).
- Each context's `rpc/*.go` `domainError` is rewritten to return `apperr.Domain(...)` with a **named reason
  constant per branch**; the reason constants live beside the mapping (e.g. `twinkle/rpc/reasons.go`). **The
  `default` branch calls `apperr.Internal`** — an unmapped domain error is a mapping gap to fix, never a silent
  passthrough.
- Reason namespacing: `<CONTEXT>_<ERROR>` — `TWINKLE_INSUFFICIENT`, `TWINKLE_QUOTE_TARGET_NOT_FOUND`,
  `MEMORY_SYNC_CONSENT_REQUIRED`, `MEMORY_OPERATION_CONFLICT`, `ADMIN_FORBIDDEN`, etc. The full list is enumerated
  from the existing `errors.New` inventory during implementation and recorded in `policy/`.

### D4 — BE: interceptor changes

In [`interceptors.go`](../../apps/api/internal/platform/interceptors.go):

- **`StructuredErrorInterceptor`** — on any error:
  1. Ensure an `ErrorInfo` detail exists; **set/overwrite `request_id`** from context on it (so it is present even
     when the handler built the error without one).
  2. If code is **not** Internal/Unknown (domain): pass through unchanged (detail + message preserved).
  3. If code **is** Internal/Unknown: capture to Sentry as today, then rebuild the error as
     `Internal` + generic message + `ErrorInfo{reason:"INTERNAL", request_id, debug_detail: envGatedRawError}`.
     `debug_detail` is the raw `err.Error()` **iff** `apperr.ExposeDetail()` (env gate, see Values), else empty.
- **`PanicRecoveryInterceptor`** — attach `ErrorInfo{reason:"INTERNAL", request_id, debug_detail}` to its
  `Internal` error, same env gate.
- **Correlation id on errors**: today `X-Request-Id` is set on `resp.Header()` but an errored unary call may have a
  nil response; carrying `request_id` **inside `ErrorInfo`** guarantees the client always receives it.
- `LoggingInterceptor` is unchanged (still logs the raw pre-mask error + request_id) — it already does the right
  thing.

### D5 — FE: `packages/errors` seam

New `packages/errors`:

```ts
export interface AppError {
  connectCode: Code // coarse Connect code
  reason: string // stable ErrorInfo.reason, or 'INTERNAL' / 'UNKNOWN'
  domain: string
  requestId: string // '' only if the server truly sent none
  metadata: Record<string, string>
  debugDetail: string // off-prod only
  retriable: boolean // derived from connectCode (Unavailable, DeadlineExceeded, …)
}
export function toAppError(err: unknown): AppError // decodes the ErrorInfo detail off a ConnectError
export function isReason(err: unknown, reason: string): boolean
```

`toAppError` reads the `ErrorInfo` detail via `ConnectError.findDetails(ErrorInfoSchema)`; falls back to the coarse
code when no detail is present (older server / non-RPC error). This is the single decoder every FE surface uses —
no more per-widget `catch` guessing (`WritingFlowSheet`, `DiaryReaderBlock`, `GistViewSheet`, `DeletionFlowSheet`
all currently hand-roll error handling).

### D6 — FE: copy + surfacing

- **`packages/i18n`**: a `reason → message` table (`error_TWINKLE_INSUFFICIENT`, …) plus a generic
  `error_internal` that **embeds the correlation id** ("문제가 발생했어요. 오류 코드: {requestId}"). Unmapped reasons
  fall back to a per-coarse-code generic string.
- **App-layer wiring** (`apps/web/src/app`, mirrored in `apps/mobile/src/app`):
  - Each app's `shared/model` owns the React context + **error toast hook** (`useErrorToast`) so lower FSD layers can
    consume it without importing upward from `app`. The matching `app/providers/error-provider.tsx` owns the rendered
    Toast, delegates `unknown` to `@cosimosi/errors`, and uses the generated auto-dismiss duration.
  - A top-level **error boundary** for fatal render/query failures, showing the generic screen **with the
    correlation id**.
  - The headless logic (map AppError → {severity, message, showId}) lives in `packages/errors` so web and mobile
    share it (parity rule); only the toast/boundary _rendering_ is per-app.
  - Settled `GetBalance` and `SyncStatus` query failures are also forwarded through this toast seam, so the
    production failures that motivated the plan expose their server correlation id instead of failing silently.
- **Correlation id** is shown to the user **only** for the internal bucket (domain errors show their meaningful
  message, no id noise).

### D7 — Placement summary

- BE: `proto/cosimosi/platform/v1/platform.proto` (ErrorInfo) · `internal/platform/apperr/*` (helper) ·
  `internal/platform/interceptors.go` (chain) · `internal/<ctx>/rpc/reasons.go` + `server.go` (per-context maps).
- FE: `packages/errors/*` (decode + taxonomy + severity logic) · `packages/i18n` (copy) ·
  `apps/{web,mobile}/src/shared/model/error-toast.ts` (context/hook) ·
  `apps/{web,mobile}/src/app/providers/error-provider.tsx` (platform Toast rendering) · app roots (boundary
  presentation).

### D8 — As-built runtime choices

- `COSIMOSI_ERROR_DETAIL=verbose` is the only value that enables `debug_detail`; production examples keep it empty.
- `errors.toast_auto_dismiss_ms` is `6000` in `spec/values.yaml` and is consumed only through generated config.
- `packages/observability` imports the decoder-only `@cosimosi/errors/core` subpath, keeping request-id extraction
  independent of localized presentation.
- Existing TanStack Query retry counts and RPC cache policy are unchanged. Error classification is for presentation
  and reason-specific recovery only.

## Acceptance Criteria

1. **Domain errors carry a precise reason to the client.** A denied spend returns Connect
   `ResourceExhausted` **and** an `ErrorInfo` detail with `reason == "TWINKLE_INSUFFICIENT"`; a recall needing
   consent returns `FailedPrecondition` with `reason == "MEMORY_SYNC_CONSENT_REQUIRED"` — distinguishable from the
   other `FailedPrecondition` reasons.
2. **Every failed RPC — domain or internal — includes a non-empty `request_id`** in its `ErrorInfo`, equal to the
   value the server logged for that call.
3. **Internal errors are masked in production**: an unexpected error (e.g. a DB failure) returns
   `{code:"internal"}`, `reason == "INTERNAL"`, **empty `debug_detail`**, and generic message — the raw error
   string never appears in the client-visible payload.
4. **Off-prod exposes the detail**: with the error-detail env gate on (dev/staging), the same internal error's
   `ErrorInfo.debug_detail` contains the raw error string, so a developer diagnoses from the browser without SSHing
   to the container.
5. **No raw-error `default` remains**: every context `rpc` `domainError` routes unmapped errors through
   `apperr.Internal` (verified by test), and every enumerated domain error has a mapped reason.
6. **FE decodes uniformly**: `toAppError` returns the correct `reason`/`requestId`/`retriable` for a domain error,
   an internal error, and a non-Connect error; a settled non-retriable failure surfaces the localized message, and
   an internal failure surfaces the generic message **with the correlation id**.
7. **The current prod symptom is explainable end-to-end**: once shipped, the `GetBalance`/`SyncStatus` failure
   shows the operator a correlation id in the UI that resolves to the exact root cause in logs/Sentry (this plan
   makes the failure _diagnosable_; the underlying infra/schema fix is separate).
8. No regression: existing green paths (successful RPCs, existing Connect-code behavior, retry policy) are
   unchanged; `pnpm gen`, build, lint, vet pass.

## Policy / Values Impact

**policy/** (lands on implementation, per /implement-job Step 6):

- New `policy/platform/errors.md` (or a section in the platform tech doc): the **error contract** — the two
  classes, the `ErrorInfo` fields, the masking + env-gate rule, and the **authoritative reason-code registry**
  (`<CONTEXT>_<ERROR>` → Connect code → whether the FE branches on it). This registry is the SSOT the FE i18n keys
  and BE mappings are checked against.
- Note the rule: **raw internal error strings never reach a production client** (information-disclosure guard).

**spec/values.yaml** — this feature is mostly **contract + env**, so it introduces little tuning config:

- The **debug-detail exposure is an ENV gate, not values.yaml** (runtime env/secret territory, explicitly excluded
  from values.yaml). Proposed: reuse an existing environment signal (e.g. `COSIMOSI_DEV_AUTH`/an `APP_ENV`) or a
  dedicated `COSIMOSI_ERROR_DETAIL=verbose`. Record the chosen env var in the platform tech/env doc, **not** in
  values.yaml.
- Any FE display tuning that _is_ config (e.g. `errors.toast_auto_dismiss_ms` if a new value is needed) goes in
  `spec/values.yaml` under an `errors:` group and is generated via `pnpm gen:values` — enumerate at implementation
  only if a real number is introduced (retry counts already live in `client_cache.*`).
- **Proto (`ErrorInfo`) and the reason-code strings are contract, not values** — they stay in proto/code, never in
  values.yaml.
