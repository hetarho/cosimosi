## Design

### Domain-mirror entities (the read model, pure â€” no three)
Three domain-mirror slices land here, each with a `model` and an `api` segment (ARCHITECTURE Â§3.1/Â§3.4). The pure FE
domain types + protoâ†’domain mappers are placed in `packages/` (shared verbatim by web + mobile, ARCHITECTURE Â§3.5), and
the app-layer `entities/*` slices expose the store + the slice public API. Domain-mirror entities carry the **canonical
domain names** (episodic memory, neuron, synapse) and know nothing about three or visual vocabulary â€” visuals are the
rendering entities' job (plan 24), and words never travel back up.

- **`entities/episodic-memory`** â€” `model`: the FE `EpisodicMemory` type (id, name, `Emotion` VO = mood + valence +
  arousal + intensity, `base_strength`, `recall_count`, `created_universe_time`, `last_recalled_universe_time`, `seed`)
  + a Zustand store holding the memory collection keyed by id. `api`: maps `EpisodicMemoryDto` â†’ the FE type.
- **`entities/neuron`** â€” `model`: the FE `Neuron` type (id, name, `neuron_type` = semantic|spatial|entity, and the
  per-neuron **connectivity/degree** the radius reads [V1]) + a store. `api`: maps `NeuronDto` â†’ the FE type.
- **`entities/synapse`** â€” `model`: the FE `Synapse` type (id, `neuron_a_id`, `neuron_b_id` (undirected, canonical
  a<b), `strength`, `co_activation_count`, `last_activated_universe_time`) + a store. `api`: maps `SynapseDto` â†’ the FE
  type.
- The activation edges (episodic-memory â†” neuron membership with weight) are carried on the read and stored so the
  graph builder can wire memory nodes to their neurons. **Decided (2026-07):** they live as a **membership list on the
  `episodic-memory` mirror** (mirroring `neuron_activations`), not as a standalone mirror slice.

These stores are **data**, populated once per `GetUniverse` fetch (Query cache â†’ store), and read by the graph builder
and the force-sim feeder. They are **not** read per-frame â€” per-frame the scene reads the coordinate buffer and the
selection snapshot (see State below).

### The `GetUniverse` RPC contract (owned here, consumed here)
This unit fixes the contract shape; the server implementation is the memory context's
([16.memory-aggregate-schema](16.memory-aggregate-schema.md)). Package `memory.v1`, service `MemoryService`
(ARCHITECTURE Â§2.7). **Unary, HTTP GET-eligible, proto-marked `NO_SIDE_EFFECTS`, per-user scoped** (never configured as
shared CDN cache â€” the data is user-scoped, ARCHITECTURE Â§2.7/Â§4).

```
GetUniverse(GetUniverseRequest{}) â†’ GetUniverseResponse {
  repeated EpisodicMemoryDto memories
  repeated NeuronDto         neurons
  repeated SynapseDto        synapses
  universe_time                          // current universe time (DATE), for read-time derivation
}
```

The DTOs carry the **stored facts** (never derived, never positions [I5]):
- `EpisodicMemoryDto` â€” id, name, emotion (mood + valence + arousal + intensity), `base_strength`, `recall_count`,
  `created_universe_time`, `last_recalled_universe_time`, `seed`, and its neuron activation edges (per-neuron
  membership + weight).
- `NeuronDto` â€” id, name (nullable), `neuron_type`, and per-neuron **connectivity/degree** for the radius [V1].
- `SynapseDto` â€” id, `neuron_a_id`, `neuron_b_id`, `strength`, `co_activation_count`, `last_activated_universe_time`.

Read-time derived values (effective brightness/strength/decay) are **not** on the wire; they are computed client-side
by `packages/memory-logic` (golden-parity with the Go domain, owned by plan 18 â€” the synapse `EffectiveSynapseStrength`
and the memory-level `EffectiveStrength`/`EffectiveBrightness`) from these stored facts + `universe_time`,
and server-side later for cost gating. Optimistic writes (embeddings, gist texts) surface on the *next* `GetUniverse`
read (ARCHITECTURE Â§2.8) â€” this unit only reads.

### Graph build + force-sim worker wiring (ARCHITECTURE Â§3.2/Â§3.3, plan 19)
1. **Fetch** `GetUniverse` via the generated Connect client (Query, GET). Map DTOs â†’ domain via each entity's `api`;
   populate the entity stores.
2. **Build the graph** the force-sim consumes: **nodes** = neurons (+ episodic-memory nodes anchored to their neuron
   membership centroid â€” the memory is a body positioned by the neurons it activates, honoring [I4]: no memoryâ†”memory
   force); **edges** = synapses (neuronâ†”neuron only, [I6]). Whether memory nodes are simulated bodies or derived from
   neuron centroids at read-back is a layout detail owned by [19.force-sim-layout](19.force-sim-layout.md); this unit
   supplies the graph and consumes the coordinate output.
3. **Run the sim in a Web Worker.** The node/edge arrays are handed to `packages/force-sim`'s `tick(dt)` in a worker
   (Barnes-Hut, deterministic seeded PRNG â€” plan 19). Center pull = self/identity, radius modulated by connectivity
   [V1][I7]; z constrained to the hippocampus band [V9]. The sim writes a `Float32Array` of coordinates.
4. **Read coords into the scene, not into React.** In `useFrame`, the renderer reads the latest coordinate buffer and
   writes `InstancedMesh.matrix` (positions) + uniforms; edges update their `Line2` endpoints. **No 60 fps React
   state** and **no per-frame store reads** (ARCHITECTURE Â§3.2/Â§3.3). The workerâ†”main coordinate handoff is a shared/
   transferable buffer (plan 19's contract); this unit is a reader of it.
