# 23. universe canvas

> Universe canvas — the first Epic-A **presentation** unit: it MOUNTS the `@cosimosi/3d-renderer` boundary
> (plan 14), reads the memory graph from `GetUniverse`, feeds it to the `packages/force-sim` worker, and renders it as
> generic instanced nodes + line-segment edges under a product camera/navigation rig (zoom · rotate · pan · focus ·
> fly). It also owns the domain-mirror **entities** (`episodic-memory` · `neuron` · `synapse`) and the `GetUniverse`
> RPC contract — the read model every P unit builds on.
> Scope: FE | Status: built ([job 28](../jobs/archive/28.universe-canvas.md))
>
> The scene _scaffold_ for Epic A. Plan 14 built the renderer substrate (skins · post · asset-source port · canvas
> host); this unit consumes it — it does **not** re-bootstrap the renderer. The specific star/neuron/synapse shader
> bodies are [24.star-neuron-rendering](24.star-neuron-rendering.md); this unit provides the generic instanced rendering
> those refine. The universe is the app's main page: `/` on web (plan 15 router) and `UniverseScreen` on mobile
> (plan 13).

## Purpose

cosimosi's core premise is a **navigable 3D memory universe** ([U3][V0], PRD §1.6: the universe page is the login
landing and the product's main entry). Everything before Epic A built substrate — the renderer
([14.rendering-foundation](14.rendering-foundation.md)), the web router ([15.web-router-foundation](15.web-router-foundation.md)),
the mobile shell ([13.mobile-app-shell](13.mobile-app-shell.md)), the memory schema
([16.memory-aggregate-schema](16.memory-aggregate-schema.md)), the plasticity math
([18.synapse-plasticity](18.synapse-plasticity.md)), the force-sim module
([19.force-sim-layout](19.force-sim-layout.md)) — but **no real memory has ever been drawn**. Plan 14's home page shows
a skinned background with placeholder floating stars and a demo orbit camera; no domain data reaches the scene.

This unit closes that gap. It is the first unit where the **read data model** lands (the domain-mirror entities and the
`GetUniverse` contract), where a real graph is projected into 3D, where the force-sim runs against real neurons and
synapses, and where the user actually **navigates** the space. It is deliberately the _scaffold_: generic instanced
nodes and generic 1px line-segment edges, a real camera rig, a real graph read — enough that the universe is a working,
explorable 3D space. The visual identity (star form, neuron cell, synapse filament, nebula, latent field) is layered on
top by later P units through the seams this unit provides. Because it owns the read model and the scene composition,
every other Epic-A presentation unit depends on it.

## Scope / Non-goals

**In scope**

- **Mount the renderer boundary, do not re-bootstrap it.** `widgets/universe-canvas` mounts `@cosimosi/3d-renderer`'s
  canvas host + skin/post pipeline (plan 14) and composes an R3F scene inside it. This unit adds **no** renderer
  lifecycle (`init`/`present`/resize/color/pixel-ratio), no skin system, no post pipeline, and imports `three` only
  through the package's seams — never directly (ARCHITECTURE §3.3, plan 14 AC 1).
- **The domain-mirror entities** (pure, **NO three**; ARCHITECTURE §3.1/§3.4): `entities/episodic-memory`,
  `entities/neuron`, `entities/synapse`. Each carries a `model` segment (the FE domain type + a Zustand store) and an
  `api` segment (the `GetUniverse` proto→domain mapper — the FE anti-corruption boundary mirroring the backend's).
  This is where the read data model for **all** Epic-A P units lands.
- **The `GetUniverse` RPC contract** this unit consumes (owned here; brief §3): the per-user read for rendering,
  returning `EpisodicMemoryDto` / `NeuronDto` / `SynapseDto` carrying **stored facts** + the current `universe_time`.
  HTTP GET-eligible / `NO_SIDE_EFFECTS` (ARCHITECTURE §2.7).
- **Graph → instanced nodes + line edges.** The projected graph feeds an `InstancedMesh` per node kind and plain
  GPU line segments (as-built; ARCHITECTURE §3.3's `Line2` fat-line + `instance_bucket_size` bucketing are deferred to
  plan 24 — see _Render approach_). Nodes are **generic** here (placeholder body, no star/neuron shader); edges are
  generic 1px filaments.
- **Force-sim in a Web Worker.** The projected node/edge arrays feed `packages/force-sim`
  ([19.force-sim-layout](19.force-sim-layout.md)) running in a worker; the renderer **reads back** the coordinate
  `Float32Array` into `InstancedMesh.matrix` / uniforms every frame — never driving 60 fps from React state
  (ARCHITECTURE §3.2/§3.3).
- **A product camera/navigation rig** [U3][V0]: zoom · rotate · pan · focus (frame a node) · fly (glide the camera to a
  target). This **replaces** plan 14's demo orbit camera for the real universe. Camera state and selection state live
  in an XState machine (shared via `@cosimosi/universe`, ARCHITECTURE §3.2).
- **z-axis 2-layer depth as 3D depth** [V9]: the scene renders the z-layer band as real 3D depth. Epic A occupies the
  **hippocampus** layer (`force_sim.hippocampus_z_min`/`_max` from plan 19); the neocortex band is reserved but
  unpopulated here (semanticize/z-lift is Epic C/E).
- **Center = self, radius = connectivity** [V1][I7]: the layout centers on self/identity and pulls better-connected
  memories toward the center; the `GetUniverse` DTOs carry the per-neuron connectivity/degree the radius reads, and
  positions are **emergent, never stored** [I5].
- **Web + mobile parity.** The same slices on both apps; `*.native.tsx` siblings only where the canvas host or the
  gesture/pointer input genuinely differs. Mobile MVP may use lower instance counts / fewer post-FX (ARCHITECTURE §3.5).

**Non-goals**

- **The star / neuron / synapse shader bodies** — the specific `entities/{star,cell-star,filament}` visual projections
  are [24.star-neuron-rendering](24.star-neuron-rendering.md). This unit ships generic instanced nodes + generic 1px
  `THREE.LineSegments` edges (the `Line2` fat-line refinement is deferred to plan 24, through the asset-source port).
- **The latent (gray) star field** ([25.latent-star-field](25.latent-star-field.md)) and the **emotion nebula color
  field** ([26.nebula-color-field](26.nebula-color-field.md)).
- **The writing flow** (`widgets/writing-flow`, `features/{write-diary,split-diary,revise-split,launch-stars}`) —
  [27.writing-flow-ui](27.writing-flow-ui.md). No `SplitDiary`/`ReviseSplit`/`LaunchStars` here.
- **Recall / star-detail panels, diary reader, letting-go** — Epic C and later. Clicking a node may **select** it (feed
  the selection state), but no detail panel, recall economy, or reconstruction is built here.
- **The universe clock.** `universe_time` is read from `GetUniverse` and rendered; the `universe_state` clock and its
  advance are Epic B. Epic A derives universe time from the latest launched memory (brief §2 seam) — the clock UI is
  deferred.
- **Read-time derived visual values.** Effective brightness/strength/decay are computed by shared pure fns in
  `packages/memory-logic` (golden-parity with the Go domain), all owned by [18.synapse-plasticity](18.synapse-plasticity.md)
  (the synapse `EffectiveSynapseStrength` **and** the memory-level `EffectiveStrength`/`EffectiveBrightness` stubs;
  forgetting decay is Epic D). **Positions** are the separate concern of [19.force-sim-layout](19.force-sim-layout.md).
  This unit reads the **stored facts** and hands them to those fns; it does not re-derive the math.
- **The runtime skin/theme switcher** ([P4]) and any new tuning scalars beyond scene-scaffold references — those stay
  in plan 14 / plan 19's values.
- **Any backend behavior beyond the `GetUniverse` read contract.** The query implementation, per-user scoping, and

## Design

### Domain-mirror entities (the read model, pure — no three)

Three domain-mirror slices land here, each with a `model` and an `api` segment (ARCHITECTURE §3.1/§3.4). The pure FE
domain types + proto→domain mappers are placed in `packages/` (shared verbatim by web + mobile, ARCHITECTURE §3.5), and
the app-layer `entities/*` slices expose the store + the slice public API. Domain-mirror entities carry the **canonical
domain names** (episodic memory, neuron, synapse) and know nothing about three or visual vocabulary — visuals are the
rendering entities' job (plan 24), and words never travel back up.

- **`entities/episodic-memory`** — `model`: the FE `EpisodicMemory` type (id, name, `Emotion` VO = mood + valence +
  arousal + intensity, `base_strength`, `recall_count`, `created_universe_time`, `last_recalled_universe_time`, `seed`)
  - a Zustand store holding the memory collection keyed by id. `api`: maps `EpisodicMemoryDto` → the FE type.
- **`entities/neuron`** — `model`: the FE `Neuron` type (id, name, `neuron_type` = semantic|spatial|entity, and the
  per-neuron **connectivity/degree** the radius reads [V1]) + a store. `api`: maps `NeuronDto` → the FE type.
- **`entities/synapse`** — `model`: the FE `Synapse` type (id, `neuron_a_id`, `neuron_b_id` (undirected, canonical
  a<b), `strength`, `co_activation_count`, `last_activated_universe_time`) + a store. `api`: maps `SynapseDto` → the FE
  type.
- The activation edges (episodic-memory ↔ neuron membership with weight) are carried on the read and stored so the
  graph builder can wire memory nodes to their neurons. **Decided (2026-07):** they live as a **membership list on the
  `episodic-memory` mirror** (mirroring `neuron_activations`), not as a standalone mirror slice.

These stores are **data**, populated once per `GetUniverse` fetch (Query cache → store), and read by the graph builder
and the force-sim feeder. They are **not** read per-frame — per-frame the scene reads the coordinate buffer and the
selection snapshot (see State below).

### The `GetUniverse` RPC contract (owned here, consumed here)

This unit fixes the contract shape; the server implementation is the memory context's
([16.memory-aggregate-schema](16.memory-aggregate-schema.md)). Package `memory.v1`, service `MemoryService`
(ARCHITECTURE §2.7). **Unary, HTTP GET-eligible, proto-marked `NO_SIDE_EFFECTS`, per-user scoped** (never configured as
shared CDN cache — the data is user-scoped, ARCHITECTURE §2.7/§4).

```
GetUniverse(GetUniverseRequest{}) → GetUniverseResponse {
  repeated EpisodicMemoryDto memories
  repeated NeuronDto         neurons
  repeated SynapseDto        synapses
  universe_time                          // current universe time (DATE), for read-time derivation
}
```

The DTOs carry the **stored facts** (never derived, never positions [I5]):

- `EpisodicMemoryDto` — id, name, emotion (mood + valence + arousal + intensity), `base_strength`, `recall_count`,
  `created_universe_time`, `last_recalled_universe_time`, `seed`, and its neuron activation edges (per-neuron
  membership + weight).
- `NeuronDto` — id, name (nullable), `neuron_type`, and per-neuron **connectivity/degree** for the radius [V1].
- `SynapseDto` — id, `neuron_a_id`, `neuron_b_id`, `strength`, `co_activation_count`, `last_activated_universe_time`.

Read-time derived values (effective brightness/strength/decay) are **not** on the wire; they are computed client-side
by `packages/memory-logic` (golden-parity with the Go domain, owned by plan 18 — the synapse `EffectiveSynapseStrength`
and the memory-level `EffectiveStrength`/`EffectiveBrightness`) from these stored facts + `universe_time`,
and server-side later for cost gating. Optimistic writes (embeddings, gist texts) surface on the _next_ `GetUniverse`
read (ARCHITECTURE §2.8) — this unit only reads.

### Graph build + force-sim worker wiring (ARCHITECTURE §3.2/§3.3, plan 19)

1. **Fetch** `GetUniverse` via the generated Connect client (Query, GET). Map DTOs → domain via each entity's `api`;
   populate the entity stores.
2. **Build the graph** the force-sim consumes: **nodes** = neurons (+ episodic-memory nodes anchored to their neuron
   membership centroid — the memory is a body positioned by the neurons it activates, honoring [I4]: no memory↔memory
   force); **edges** = synapses (neuron↔neuron only, [I6]). Whether memory nodes are simulated bodies or derived from
   neuron centroids at read-back is a layout detail owned by [19.force-sim-layout](19.force-sim-layout.md); this unit
   supplies the graph and consumes the coordinate output.
3. **Run the sim in a Web Worker.** The node/edge arrays are handed to `packages/force-sim`'s `tick(dt)` in a worker
   (Barnes-Hut, deterministic seeded PRNG — plan 19). Center pull = self/identity, radius modulated by connectivity
   [V1][I7]; z constrained to the hippocampus band [V9]. The sim writes a `Float32Array` of coordinates. As built
   (job 28): the worker↔main handoff is a two-buffer transferable ping-pong behind a `UniverseSimBridge` seam; on
   React Native (no standard Worker primitive) the same bridge runs the sim inline on the JS thread behind a per-app
   spawner, and a future RN worker primitive slots in behind that seam. A worker/sim error terminates the bridge and
   reads as an **empty** universe (never a zero-stacked one); the graph builder clamps out-of-range stored magnitudes
   (strength 0..1, non-negative weights/connectivity) so a skewed row cannot kill the scene. A `GetUniverse` refetch
   carries surviving nodes' coordinates onto the new graph by **stable id (`kind:id`), never by array slot** (the
   backend orders by random id, so a launch can reorder the arrays); this is shared by the worker and inline modes, so
   a launch/reorder keeps each node's position and web/mobile stay in refetch parity.
4. **Read coords into the scene, not into React.** In `useFrame`, the renderer reads the latest coordinate buffer and
   writes `InstancedMesh.matrix` (positions) + uniforms; edges update their line-segment endpoints. **No 60 fps React
   state** and **no per-frame store reads** (ARCHITECTURE §3.2/§3.3). The worker↔main coordinate handoff is a shared/
   transferable buffer (plan 19's contract); this unit is a reader of it.

### Render approach — generic instanced nodes + line edges

- **Nodes** = a single `InstancedMesh` sized to the active node count (generic geometry/material — a placeholder body
  the asset-source port later swaps for the real star/cell shader, [24.star-neuron-rendering](24.star-neuron-rendering.md)).
  As-built: one mesh per node kind (neuron/memory), not `rendering.instance_bucket_size` buckets — a fixed-capacity
  InstancedMesh started at count 0 makes the WebGPU backend build an invalid bind group, so the scaffold sizes to the
  real count and keeps the mesh `visible = false` until its matrices are written; bucketing returns in plan 24 when a
  graph exceeds one mesh.
- **Edges** = one synapse per line segment, generic color here (plan 24/§V6 refines width/brightness = synapse
  strength). As-built they draw as a plain `THREE.LineSegments` + `LineBasicNodeMaterial` over a `position`
  BufferGeometry (2 verts per edge) — **not** the `Line2` fat-line, whose transparent path reads the opaque viewport
  texture that the custom PostFX pipeline doesn't expose (invalid bind group). Lines are 1px; fat-line width is a
  plan-24 refinement. The layer is `visible = false` until it has ≥1 segment (same 0-count WebGPU guard as nodes).
  Filaments connect **neurons only** ([I4]/[I6]) — there is structurally no memory↔memory line.
- **Bodies arrive through the asset-source port** (plan 14 §3.4): this unit binds a _generic_ body source; plan 24
  binds the real shader/glTF bodies without this unit changing. The domain-mirror entities never import a concrete
  renderer.
- **Skin / post / background** come from the mounted `@cosimosi/3d-renderer` boundary unchanged — this unit adds none.

### Camera / navigation rig + selection (XState, ARCHITECTURE §3.2) [U3][V0]

Camera control and selection are **control-state** → an XState machine in the widget's `model/` segment; the graph,
coords, and caches are **data** in Zustand/Query (id references only in machine context, ARCHITECTURE §3.2).

- **Navigation modes** [V0]: `idle` (free navigation — zoom / rotate / pan), `focusing` (framing a chosen node),
  `flying` (an animated camera glide to a target), returning to `idle`. This **replaces** plan 14's demo orbit camera
  for the product universe.
- **Selection** [U3]: a node can be selected (hover/click), stored as an id in machine context; the selected id is read
  in `useFrame` via `getSnapshot()` to drive any per-frame highlight — no React re-render per frame. A selection _panel_
  is out of scope (Epic C); selection here only feeds focus/fly and later panels.
- **z-depth navigation** [V9]: the camera can traverse the z-band; Epic A shows only the hippocampus layer populated,
  with the neocortex band reserved above it. Fly easing/durations, zoom clamps, and focus framing are code-level
  presentation constants in the widget's `config/` segment (set at implement, job 28 — none proved values-tunable).
  The rig's free-navigation input is DOM pointer-driven and stays inert on native for the MVP; RN gesture input is a
  later input sibling behind the same rig seam.

### Placement (FSD) + web/mobile parity

- **`widgets/universe-canvas`** — the full-screen R3F scene block (ARCHITECTURE §3.1: `widgets` = a big self-contained
  block). Composes: the mounted `@cosimosi/3d-renderer` host, the graph builder, the force-sim worker wiring, the
  instanced-node + line-segment render, and the camera/selection machine.
- **Pages / screens** compose the widget: web `/` → `UniverseHomePage` (the `pages/universe` screen, through plan 15's
  typed route registry — plan 14's placeholder home is replaced by the real graph); mobile → `UniverseScreen` (the
  route reserved by plan 13, through its typed navigation registry).
- **Parity**: the widget's platform-agnostic core — the navigation machine, the graph builder, the sim bridge, and the
  camera-rig scalars — is shared verbatim through the **`@cosimosi/universe`** package (with the entity `model`/`api`
  over `@cosimosi/memory`); the app widget slices keep only the fetch→store wiring, scene composition, and the per-app
  sim-worker spawner. Per-app / `*.native.tsx` siblings exist **only** for the canvas host wiring and the sim-worker
  spawner (web module Worker vs RN inline), per ARCHITECTURE §3.5. Sharing the core through a package — not
  copy-mirroring it into both apps — is what makes "verbatim" hold (copies drift on formatting alone). Mobile MVP may
  reduce instance counts / post-FX (brief §5).

## Acceptance Criteria

1. `widgets/universe-canvas` **mounts** `@cosimosi/3d-renderer`'s canvas host + skin/post pipeline (plan 14) and
   composes its scene inside it; it does **not** re-bootstrap the renderer (no `init`/`present`/resize/color/pixel-ratio,
   no skin system, no post pipeline) and imports `three` only through the package's seams — never directly.
2. The three domain-mirror entities `entities/{episodic-memory,neuron,synapse}` exist, are **pure (no three import)**,
   each with a `model` (FE domain type + Zustand store) and an `api` segment that maps the `GetUniverse` proto DTOs
   (`EpisodicMemoryDto`/`NeuronDto`/`SynapseDto`) → FE domain types.
3. `GetUniverse` is a **unary, HTTP GET-eligible, `NO_SIDE_EFFECTS`, per-user-scoped** read returning the stored facts
   (emotion, `base_strength`, `recall_count`, created/last-recalled universe time, seed, `neuron_type`, per-neuron
   connectivity/degree, synapse strength + co-activation, activation edges) plus the current `universe_time`; it carries
   **no** positions/coordinates [I5] and no read-time-derived values.
4. The projected graph renders as **`InstancedMesh` nodes + GPU line-segment edges** (as-built: plain `LineSegments`;
   `Line2` fat-line + bucketing deferred to plan 24); edges connect **neurons only** (never memory↔memory) [I4][I6].
5. `packages/force-sim` runs in a **Web Worker**; the renderer reads the coordinate `Float32Array` into
   `InstancedMesh.matrix` / uniforms in `useFrame`, and the scene is **not** driven by 60 fps React state (no per-frame
   React re-render, no per-frame store read; per-frame control reads use `getSnapshot()`).
6. A product **camera/navigation rig** provides zoom · rotate · pan · focus · fly and **replaces** plan 14's demo orbit
   camera for the real universe; camera + selection modes are an XState machine (shared via `@cosimosi/universe`, consumed by the widget).
7. Layout **centers on self/identity** with **radius = connectivity** [V1][I7]; positions are **emergent and never
   stored/persisted** by this unit [I5]; the z-layer 2-tier band is shown as **3D depth**, with Epic A populating the
   hippocampus layer [V9].
8. The universe is the app's **main page**: web `/` → `UniverseHomePage` (through plan 15's typed route registry, replacing
   plan 14's placeholder home) and mobile `UniverseScreen` (through plan 13's typed navigation registry), rendering the
   real graph.
9. Web + mobile share the widget, machine, graph builder, and entity `model`/`api` **verbatim**; `*.native.tsx` siblings
   exist **only** for the canvas host + gesture/pointer input; mobile MVP may run lower instance counts / fewer post-FX.
10. The star/neuron/synapse shader bodies, latent field, nebula, writing flow, and recall/detail panels are **not**
    built here (nodes are generic instances, edges generic filaments); the visual bodies arrive through plan 14's
    asset-source port so plan 24 refines them without changing this widget.
