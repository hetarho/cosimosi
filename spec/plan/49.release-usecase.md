# 49. release usecase

> Release use-case — the backend orchestration for the two user-initiated ways to let a memory go: **full delete**
> (diary-scoped 30-day soft-delete with a restore window) and **letting-go** (a permanent, timer-free semantic-neuron sealing where
> the AI _suggests_ candidates and the user _approves_ before the domain seals). System dynamics never delete; the user's
> full-delete path is the explicit [I1]/[X] exception.
> Scope: BE | Status: planning
>
> Layer U (use-cases) — the use-case unit of Epic H. Home: `internal/memory` use-cases `Release` / `Restore` / `LetGo`
>
> - a consumer-owned `SealSuggester` port (this-memory-only semantic-neuron suggestions, reusing the AI seam of
>   [22.ai-worker-pipeline](22.ai-worker-pipeline.md) / [28.ai-provider-abstraction](28.ai-provider-abstraction.md)) and a
>   `RetentionSweeper` capability. It executes the sealing _rules_ owned by [48.deletion-rules](48.deletion-rules.md)
>   (orphan/shared neuron sealing, LTD on contribution, exclusion), over [16.memory-aggregate-schema](16.memory-aggregate-schema.md)'s
>   reserved `deleted_at` / `sealed_at` columns plus a retention-scoped release-effect ledger, and is driven by the
>   delete/letting-go UI in [50.deletion-ui](50.deletion-ui.md).

## Purpose

Forgetting in cosimosi is dimming, never erasure ([I1][F1]) — the whole engine is built on the silent engram (Roy 2017).
But the user is the one exception: a person has the right to delete their own record, and sometimes the point of a memory
product is to _let something go on purpose_. This unit owns the two ways that happens on the backend, and it fixes the
two rules that keep them honest.

First, **full delete is recoverable**. A person who deletes a diary in a hard moment must be able to change their mind:
every full delete is a **30-day soft-delete with a restore window** ([X2]), and the only hard delete that ever runs is a
30-day sweep of rows _the user themselves_ soft-deleted — never a system-driven purge of anything the user did not choose
to remove ([I1]). A diary can create several episodic-memory stars, so full delete operates at the **diary boundary**:
all stars born from that diary are released together ([X1]). Second, **letting-go is deliberate and permanent**. It is
not a delete of the star — the star, its
emotion, its color, its spatial/entity/time neurons all stay; only the **semantic** neurons this one memory uses are
sealed ([X4]), leaving a content-less trace (a silent engram). Because it is an act of intentional closure, it carries
**no 30-day timer** — it is permanent ([X5]). And because sealing the wrong meaning would be a quiet betrayal, the flow
is **AI-suggests / user-approves / domain-executes** ([X6]): the AI reads the user's words and proposes _this-memory-only_
semantic neurons as seal candidates, the user approves, and only then does the domain seal. The AI never deletes anything.

This unit is the use-case orchestration for `Release`, `Restore`, and `LetGo` and their RPCs; the _rules_ they execute —
orphan vs. shared classification, the LTD on a shared neuron's synaptic contribution, and the exclusion of sealed things
from all dynamics — belong to [48.deletion-rules](48.deletion-rules.md), which this unit calls.

## Scope / Non-goals

**In scope**

- The **`Release`** use-case in `internal/memory` — full delete of one `Diary` ([X1]): soft-delete every linked
  `EpisodicMemory` (`deleted_at = now()`), seal their activations, apply [48.deletion-rules](48.deletion-rules.md)'s
  removal-set neuron rules (orphan neurons → `sealed_at`; shared neurons → kept + LTD on each removed memory's synaptic
  contribution), all in one transaction and per-user scoped. Soft-delete only — no hard delete during `Release` ([X2]).
- The **retention-scoped release-effect ledger** — enough flow-owned records to make `Restore` exact within 30 days:
  release id/diary id, affected episodic memory ids, orphan neurons sealed by this release, and synapse strengths/deltas
  changed by this release. These rows are not long-term provenance; they are swept with the release after the retention
  window.
- The **`Restore`** use-case — undo a diary full delete **within the 30-day window** ([X2]): clear `deleted_at` for the
  released memories, unseal the orphan neurons this Release sealed, and reverse the shared-neuron LTD from the ledger,
  returning all released memories to full participation in all dynamics. Refuses once the release has been swept.
- The **`LetGo`** use-case — letting-go ([X4][X5][X6]): a **two-step** flow. Step 1 `SuggestLetGo` — take the user's words/
  meaning, call the consumer-owned `SealSuggester` port to identify **this-memory-only `semantic` neurons** as seal
  candidates, and return them for the user to review (**AI suggests only** — nothing is written). Step 2 `LetGo` — take the
  user-**approved** neuron ids and seal them ([48.deletion-rules](48.deletion-rules.md) applies orphan-seal / shared-keep+LTD
  per neuron). **Permanent, no timer** ([X5]); the star, emotion, color, and spatial/entity/time neurons are untouched.
- The consumer-owned **`SealSuggester`** port declared here (the `LetGo` use-case consumes it), reusing the AI seam
  ([22.ai-worker-pipeline](22.ai-worker-pipeline.md) / [28.ai-provider-abstraction](28.ai-provider-abstraction.md)):
  domain-shaped in/out, schema-forced output (it can only emit _candidate neuron references_, never a delete/seal command).
- The **[X7] heavy-state signal** — `SuggestLetGo` returns an optional `heavy_state` hint (derived server-side from the
  user's words + the memory's emotional signal) that [50.deletion-ui](50.deletion-ui.md) surfaces as a gentle,
  non-blocking professional-resource notice. **Decided (2026-07):** this unit owns the _signal_ (a field on the
  `SuggestLetGo` response); the _detection heuristic_ is a **reserved slot** (v1 conservative — e.g. emotion/keyword
  cue, refined later); the _notice UI_ is [50]'s. Letting-go is framed as symbolic and claims **no** efficacy ([X7]).
- The **`RetentionSweeper`** capability — the sole hard delete: a per-user sweep that hard-deletes diary release groups
  whose `deleted_at` is older than the retention window (`release.soft_delete_retention_days`), including the original
  `Diary` row/body and retained provenance for those memories, honoring [I1] (it deletes only _user-soft-deleted_ rows; it
  never touches un-deleted data). Not a cron — see the trigger decision below.
- The RPCs on `memory.v1.MemoryService`: **`Release`**, **`Restore`**, **`SuggestLetGo`**, **`LetGo`** — thin handlers
  (proto↔domain map + call), all per-user scoped; the retention window value in `spec/values.yaml`.
- Per-user scoping on every read, seal, unseal, delete, and suggestion (§4).

**Non-goals**

- **No sealing rules.** _Which_ neurons are orphan vs. shared, the LTD-on-contribution math (`Depress`, plan 18 seam),
  the exclusion-from-all-dynamics guarantee, and the "keep spatial/entity/emotion/color/star" law are owned by
  [48.deletion-rules](48.deletion-rules.md). This unit orchestrates and calls them; it does not redefine them.
- **No concrete AI adapter.** The real LLM adapter + keyless mock implementing `SealSuggester` live behind the AI seam
  ([22.ai-worker-pipeline](22.ai-worker-pipeline.md) / [28.ai-provider-abstraction](28.ai-provider-abstraction.md)),
  injected at the composition root. This unit declares the consumer-owned port only.
- **No broad aggregate schema rewrite.** `deleted_at` (on `episodic_memories`) and `sealed_at` (on `neurons`) are already
  reserved by [16.memory-aggregate-schema](16.memory-aggregate-schema.md); this unit activates them and adds only the
  retention-scoped release-effect ledger required for exact restore.
- **No exclusion read change.** `GetUniverse` already filters `deleted_at IS NULL` (memories) and `sealed_at IS NULL`
  (neurons) on every sub-query ([16.memory-aggregate-schema](16.memory-aggregate-schema.md) / brief §as-built). Epic H
  activates the _write_ side; confirming the read exclusion holds for sealed neurons is [48.deletion-rules](48.deletion-rules.md)'s.
- **No delete / letting-go / restore UI, confirmation copy, or professional-resource notice.** The [X7] "this is not
  therapy" framing, the notice **UI**, and the export-reassurance flow are [50.deletion-ui](50.deletion-ui.md); this unit
  owns only the backend **`heavy_state` signal** that triggers that notice (above), not its presentation.
- **No per-star full-delete API.** Full delete acts at the diary boundary ([X1]); [50.deletion-ui](50.deletion-ui.md)
  owns how clearly the UI previews the affected stars. Letting-go remains the per-star partial-release flow.
- **No twinkle cost.** Release / letting-go / restore spend nothing; the economy ([G], Epic F) does not gate deletion.
- **No export.** Provenance/export ([R8a][W6][D4]) is [46.provenance-export](46.provenance-export.md); the delete UI's
  reassurance ("your original is preserved") reads it, but this unit does not implement export.

## Design

### Placement (domain-first Go)

`Release`, `Restore`, and `LetGo`/`SuggestLetGo` are application use-cases in the core-domain `internal/memory` context,
alongside `Encode`/`PersistEncoded` and the recall/forgetting use-cases. They start inside the context package
(ARCHITECTURE §2.3). The **sealing rules** they invoke are [48.deletion-rules](48.deletion-rules.md)'s domain functions
(orphan classification, `Depress`-based contribution LTD, exclusion), reached as pure domain calls — this unit is the
_use-case_ seam over those _domain_ rules (the epic build order D → U → P). The `SealSuggester` port is declared here
because `LetGo` consumes it (§2.4); its concrete adapter is injected at `cmd/api` from the AI seam. Dependency direction:
`memory/rpc → Release/Restore/LetGo (use-case) → pure domain (48's sealing rules, plan 18's Depress)`, with the
`SealSuggester`, the repositories, and the retention query reached only through consumer-owned interfaces.

### `Release` — full delete = diary soft-delete + restore window ([X1][X2])

Given `(userScope, diaryId)`, `Release` runs one transaction (`platform.InTx`):

1. **Load the diary release set** (per-user): the `Diary` must exist and have at least one live linked `EpisodicMemory`.
   A second Release for an already-soft-deleted diary release group is a no-op / canonical error, not a double-seal.
2. **Create a release record** with `diary_id`, `deleted_at = now()` (real-clock UTC), and the affected
   `episodic_memory_ids`. The 30-day window is real-world time, not universe-time — a user must be able to restore over a
   real month.
3. **Soft-delete the memories** — set `episodic_memories.deleted_at = deleted_at` for every linked memory in the release
   set. This is one diary-scoped release, never a per-star partial full-delete.
4. **Seal their activations** and apply [48.deletion-rules](48.deletion-rules.md)'s removal-set neuron rules: neurons
   activated only by the released set → seal (`sealed_at = now()`); neurons still used by other live memories → kept,
   with each removed memory's **synaptic contribution weakened (LTD via `Depress`)** — shared neurons are never deleted
   ([X1][I1]).
5. **Record release effects** needed for exact restore: orphan neurons sealed by this release and every synapse
   strength/delta changed by this release.
6. **Return** the released `diary_id`, affected memory ids, and `deleted_at`. All affected stars vanish from the next
   `GetUniverse` read (exclusion already active) and leave all dynamics ([X3]); nothing is hard-deleted yet.

Because the release-effect ledger records which neurons this Release sealed and what LTD it applied, `Restore` can
reverse the diary release exactly. **Decided (2026-07):** the `deleted_at` timestamp is stored in **real (wall-clock) UTC
time**, not universe-time — the restore window is a human-time promise ("30 days to change your mind"), independent of
whether the user launches or recalls (which is what moves universe-time [T2]). The retention window is
`release.soft_delete_retention_days`.

### `Restore` — undo within the window ([X2])

Given `(userScope, diaryId)`, `Restore` refuses if the release group is absent (already swept), outside the retention
window, or not soft-deleted, else, in one transaction: clear `deleted_at` for the released memories; unseal the
**orphan neurons this Release sealed**; reverse the shared-neuron LTD using the release-effect ledger. All released
memories return to full participation — they reappear in `GetUniverse`, force-sim, decay, and consolidation. Restore is
only reachable while `now() − deleted_at < release.soft_delete_retention_days`; after the sweep there is nothing to
restore ([X2]). **Decided (2026-07):** letting-go has **no `Restore`** — [X5] makes it permanent; only `Release` (full
delete) is restorable.

### `LetGo` — letting-go = AI-suggest → user-approve → domain-seal, permanent ([X4][X5][X6])

Letting-go is a **two-RPC** flow so the "AI suggests, user approves, domain executes" contract is structural, not a
convention:

- **`SuggestLetGo(userScope, episodicMemoryId, words)`** — the user says the words/meaning they want to release. The
  use-case calls `SealSuggester.Suggest(ctx, memory, words, thisMemorySemanticNeurons)`, which returns _candidate_
  `semantic` neurons **used only by this memory** — never a shared semantic neuron, never a spatial/entity/emotion/color
  neuron or the star (those are unconditionally kept [X4]). The candidate set is pre-filtered to this-memory-only semantic
  neurons **in the use-case** before the AI sees them, so the AI can only rank/select within a set that is already safe;
  the AI's schema-forced output can only _reference candidate neuron ids_, never emit a seal/delete command ([X6] — AI
  suggests only). **Nothing is written.** Returns the candidates for the user to review.
- **`LetGo(userScope, episodicMemoryId, approvedNeuronIds)`** — takes the user-**approved** subset and, in one
  transaction, seals exactly those neurons via [48.deletion-rules](48.deletion-rules.md) (each approved neuron: orphan →
  seal; shared → keep + LTD on this memory's contribution — but note letting-go only ever _offers_ this-memory-only
  neurons, so approved neurons are orphan-to-this-memory by construction). The use-case **re-validates server-side** that
  every approved id is a this-memory-only `semantic` neuron (§2.9#8 — never trust the client's list); a non-semantic,
  shared, or foreign id is rejected. Emotion, color, star, and spatial/entity/time neurons are untouched ([X4][I11]).
  **Permanent — no `deleted_at`, no timer, no restore** ([X5]).

The star survives as a **silent engram**: its position/connections naturally recalculate as the sealed semantic neurons
drop out of the force-sim ([I5], via 48's exclusion), and its color/emotion/size remain — a content-less trace ([X4]).

### `SealSuggester` port (consumer-owned, declared in `internal/memory`)

Declared where `LetGo` consumes it (§2.4). Domain-shaped in/out only — no proto, no sqlc, no SDK types cross this
boundary; the concrete adapter + keyless mock live behind the AI seam ([22.ai-worker-pipeline](22.ai-worker-pipeline.md) /
[28.ai-provider-abstraction](28.ai-provider-abstraction.md)) and are cost-metered (§2.8):

- `Suggest(ctx, memory, words, candidates) → (SealSuggestion, error)` — given the memory, the user's words, and the
  pre-filtered this-memory-only semantic-neuron candidate set, return a schema-forced ranking/selection of candidate
  neuron references (with a short human reason for the UI). **Output schema-forced**: the model has no field in which to
  emit a delete command, a shared/foreign neuron, or a star/emotion reference — the structural half of "AI suggests only"
  ([X6]), mirroring the [W4a] defense in [20.encode-usecase](20.encode-usecase.md).

### `RetentionSweeper` — the only hard delete ([X2][I1])

A per-user capability: `Sweep(ctx, scope, now) → (deletedCount, error)` hard-deletes release groups whose `deleted_at`
is older than `release.soft_delete_retention_days`: retained `memory_provenance` rows for the released memories, the
soft-deleted `episodic_memories`, their now-unreferenced sealed activations/exclusive dependents, the release-effect
ledger rows that existed only to support restore, and the original `Diary` row/body. It **only** removes rows the user
soft-deleted (user-origin), and it never touches `deleted_at IS NULL` data — this is the single
point where [I1]'s "system never originates deletion" is honored by construction (the system executes the _user's_ deletion after the
grace period; it originates no deletion of its own). **Decided (2026-07):** the sweep is **not a cron** — consistent with
[T4]/CC7's no-cron discipline, it runs as a `RetentionSweeper` capability invoked from an existing operational trigger
(a maintenance job kind on the `jobs` queue, or opportunistically at a per-user read), the concrete trigger chosen at
implement. What is fixed is the _rule_ — soft-deleted release groups older than the window are hard-deleted, nothing
else, and no timer runs against letting-go. Sealed **shared** neurons are never swept — only the user-soft-deleted
release group's diary row/body, memory rows, retained provenance, exclusive dependents, and release-effect rows.

### RPC contract (`memory.v1.MemoryService`, Connect, unary — §2.7)

proto is a DTO layer mapped at `memory/rpc`; handlers are thin (map proto↔domain + call the use-case, §2.9#7). These are
mutations — **not** GET / `NO_SIDE_EFFECTS`. `SuggestLetGo` invokes the LLM and is cost-metered but **persists nothing**.

```proto
// full delete — diary-scoped soft-delete + 30-day restore window ([X1][X2]); persists a soft-delete, no hard delete.
rpc Release(ReleaseRequest) returns (ReleaseResponse);
// undo a full delete within the retention window ([X2]).
rpc Restore(RestoreRequest) returns (RestoreResponse);
// letting-go step 1 — AI suggests this-memory-only semantic neurons; persists nothing ([X6]).
rpc SuggestLetGo(SuggestLetGoRequest) returns (SuggestLetGoResponse);
// letting-go step 2 — seal the user-approved neurons; permanent, no timer ([X4][X5][X6]).
rpc LetGo(LetGoRequest) returns (LetGoResponse);

message ReleaseRequest      { string diary_id = 1; }
message ReleaseResponse     { string diary_id = 1;
                              repeated string episodic_memory_ids = 2;
                              string deleted_at = 3; } // deleted_at = real-clock UTC
message RestoreRequest      { string diary_id = 1; }
message RestoreResponse     { string diary_id = 1;
                              repeated string episodic_memory_ids = 2; }

message SuggestLetGoRequest { string episodic_memory_id = 1; string words = 2; }       // the user's words/meaning
message SuggestLetGoResponse { repeated SealCandidate candidates = 1;
                              HeavyState heavy_state = 2; }               // [X7] reserved backend heavy-state hint
message SealCandidate       { string neuron_id = 1; string name = 2; string reason = 3; } // this-memory-only semantic
message HeavyState         { bool detected = 1; string severity = 2; }    // [X7] → [50] surfaces a professional-resource notice
message LetGoRequest        { string episodic_memory_id = 1;
                              repeated string approved_neuron_ids = 2; }                 // user-approved subset
message LetGoResponse       { repeated string sealed_neuron_ids = 1; }
// NOTE: no field lets the client (or the AI) command a hard delete, seal a shared/foreign neuron, or touch the star/
// emotion — [X6] structural defense; the server re-validates approved_neuron_ids against this-memory-only semantic set.
```

### Thin handler / dependency rule

`memory/rpc` handlers only map proto DTOs to domain inputs and call the use-cases; the soft-delete/seal orchestration, the
restore-window guard, the this-memory-only-semantic re-validation, and the retention rule live in the use-case / domain
(§2.9#7). No proto message or sqlc row is seen by the use-case or pure domain (§2.9#4); the `SealSuggester` and the
repositories are consumer-owned interfaces (§2.9#6). Orphan-ness, sealed-ness, and the approved-id validity are
**server-side decisions** — never taken from client-supplied state (§2.9#8).

### Golden-parity stance

This unit shares **no** FE↔BE math. Release/Restore/LetGo are transactional writes; the sealing/LTD math is
[48.deletion-rules](48.deletion-rules.md)'s (which states its own parity stance — the `Depress` LTD is plan 18's
golden-parity primitive), and the suggestion is an LLM call, not a deterministic function. So there is no golden-parity
obligation on this plan; correctness is by transaction + re-validation, not by a mirrored pure function.

## Acceptance Criteria

1. `Release` **soft-deletes** every `EpisodicMemory` linked to one `Diary` (`deleted_at = now()`, real-clock UTC) in one
   transaction, seals their activations, and applies [48.deletion-rules](48.deletion-rules.md)'s removal-set neuron
   rules (orphan → `sealed_at`; shared → kept + LTD on each removed memory's synaptic contribution); it performs **no
   hard delete** and never updates the `Diary` body in place. ([X1][X2][I1][I2])
2. A `Release`d diary's memories are **excluded from all dynamics** — absent from `GetUniverse`, force-sim, decay, and
   consolidation — and their sealed shared neurons are preserved, not deleted. ([X3][I1], via [48.deletion-rules](48.deletion-rules.md))
3. `Release` persists a retention-scoped release-effect ledger sufficient for exact restore: affected memory ids, orphan
   neurons sealed by the release, and synapse strengths/deltas changed by the release; ledger rows are swept with the
   release after the retention window. ([X2])
4. `Restore` reverses a diary `Release` **within `release.soft_delete_retention_days`**: it clears `deleted_at` for all
   released memories, unseals the orphan neurons that Release sealed, and reverses the shared-neuron LTD from the ledger,
   returning the memories to full participation; it **refuses** once the release has been swept or if the diary release
   was not soft-deleted. ([X2])
5. `SuggestLetGo` calls the `SealSuggester` over a **pre-filtered this-memory-only `semantic` neuron** candidate set and
   returns candidates for review; it **persists nothing** and can never surface a shared, spatial, entity, emotion, color,
   or star reference. ([X4][X6])
6. `LetGo` seals **only the user-approved** neurons, after **server-side re-validation** that each approved id is a
   this-memory-only semantic neuron (a shared/foreign/non-semantic id is rejected); the star, emotion, color, and
   spatial/entity/time neurons are untouched. ([X4][X6][I11])
7. Letting-go is **permanent** — `LetGo` writes no `deleted_at`, starts no timer, and has **no `Restore` path**; only full
   delete (`Release`) is restorable. ([X5])
8. After `LetGo`, the star survives as a **silent engram**: its position/connections recalculate as the sealed semantic
   neurons drop out of the force-sim, while its color/emotion/size remain. ([X4][I5])
9. The **only** hard delete is `RetentionSweeper`, which removes solely soft-deleted diary release groups (their original
   `Diary` row/body, `episodic_memories`, retained `memory_provenance`, exclusive sealed dependents, and release-effect
   rows) whose `deleted_at` is older than `release.soft_delete_retention_days`; it never deletes `deleted_at IS NULL`
   data, never deletes shared neurons, and runs with **no cron** (an operational trigger chosen at implement). ([X2][I1][T4])
10. `SuggestLetGo` (and `Release`/`Restore`/`LetGo`) are mutations — **not** `NO_SIDE_EFFECTS`; `SuggestLetGo` invokes the
    cost-metered LLM but writes nothing. ([X6], §2.8)
