# 45. 감정색 온보딩 (mood-color-onboarding)

> 로그인한 사용자가 개인 우주에 들어가기 전 13개 감정별 색을 자기 설정으로 확정해 DB에 저장하고, 별·시냅스·감정 UI가 그 색을 우선 사용하게 하는 필수 설정 흐름. 기본 추천색은 보편적 감정-색 연상을 따르되, 사용자는 모든 감정에 대해 완전히 자유로운 `#RRGGBB` 색을 고를 수 있다. | 범위: FE

## 목적

cosimosi에서 별의 색은 단순 장식이 아니라 "이 기억이 어떤 감정으로 남았는가"를 읽는 핵심 언어다. 그런데 지금 인증 사용자는 아무 감정색도 확정하지 않은 채 우주에 들어오고, 클라이언트 기본 팔레트로만 렌더된다. plan 30은 `user_emotion_colors` DB와 `UpdateSettings` 경로를 이미 준비했지만, 실제로 사용자가 감정별 색을 정하고 저장하는 표면은 아직 없다.

목표는 최초 로그인 플래그가 아니라 **설정 존재 여부**를 기준으로 한다. 로그인 후 개인 우주를 그리려면 감정별 색 정보가 필요하므로, 서버에 13개 감정색이 모두 저장돼 있지 않으면 우주를 렌더하기 전에 감정색 설정 페이지로 보낸다. 사용자는 널리 통용되는 추천색을 그대로 받아도 되고, 각 감정마다 완전히 다른 색을 직접 골라도 된다. 저장된 색은 DB의 사용자 설정이 되며, 해당 감정의 별은 그 색으로 나온다.

## 범위 / 비목표

**범위**
- 인증·멤버 사용자에게 개인 우주 진입 전 감정색 완료 여부를 확인하는 FE 게이트.
- 전용 라우트 `/emotion-colors`: 13개 mood 색을 추천값으로 시작해 한 번에 저장하는 필수 설정 페이지이자, 이후 재방문해 수정할 수 있는 편집 페이지.
- 감정색 완료 기준: `GetSettings` 응답의 `emotion_colors`가 13개 mood 전부에 대해 유효한 `#RRGGBB` 값을 가진 상태.
- 저장 경로: 기존 `SettingsService.UpdateSettings`의 `emotion_colors` repeated field로 13개 색을 한 번에 upsert. 별도 DB 마이그레이션이나 신규 RPC 없이 plan 30의 `user_emotion_colors`를 사용한다.
- 추천색: 기존 `MOOD_PALETTE`의 13색을 "보편적 추천 팔레트"로 노출하고, 저장 시 추천색도 사용자 선택으로 DB에 기록한다.
- 컬러 피커 UX: 투박한 브라우저 기본 `<input type="color">`를 그대로 노출하지 않고, Apple/Figma 계열의 현대적인 스펙트럼 피커(큰 saturation/value 면, hue rail, 실시간 별 미리보기, 추천색 복귀, hex 입력, 지원 브라우저의 EyeDropper)를 앱 UI로 제공한다.
- 저장 직후 settings query cache와 appearance store를 즉시 갱신해 게이트 루프를 막고, 우주 캔버스가 refetch 없이 새 색으로 재베이킹되게 한다.

**비목표**
- 감정 taxonomy(13종), mood enum, AI 추출 규칙은 바꾸지 않는다(29 유지).
- `user_emotion_colors` 스키마, proto 메시지, BE `SettingsService` 계약은 바꾸지 않는다. 현재 계약으로 충분하다.
- 별의 색 규칙 자체를 바꾸지 않는다. 이미 구현된 `resolveMoodRgb(mood, emotionColors)` 경로를 이용한다.
- 별 모양·배경·나·시냅스 스타일 소유권/별가루(44)는 건드리지 않는다. 이 plan은 감정색을 판매하거나 잠그지 않는다.
- 미인증 랜딩, 체험 모드, 공개 방문 우주(`/u/$slug`)에 필수 설정을 요구하지 않는다. 이들은 기존처럼 기본값 또는 공유된 owner appearance로 렌더한다.
- 색 선택을 심리검사나 정답 판정으로 만들지 않는다. 추천은 출발점일 뿐, 모든 색 선택은 사용자의 의미 부여를 우선한다.

## 설계 요점

### A. 완료 판정: "처음 로그인"이 아니라 "13색이 서버에 있나"

`EmotionColorGate`는 인증·멤버십 통과 뒤, 개인 우주를 렌더하기 전에 `settingsQueryOptions()`로 `GetSettings`를 읽는다.

완료 조건:
- 13개 canonical mood(`Mood` 유니온: `joy`...`emptiness`)가 모두 존재한다.
- 각 mood의 color가 `#RRGGBB` 정규식에 맞는다. 대소문자 hex는 허용하되 클라이언트 저장 상태는 `#RRGGBB` 대문자로 정규화한다.
- `MOOD_UNSPECIFIED`, unknown mood, malformed color는 완료 판정에서 무시한다.

하나라도 빠지거나 깨져 있으면 `/emotion-colors?redirect=<현재경로>`로 보낸다. 이 기준은 날짜·가입 시점·첫 로그인 여부를 보지 않는다. 나중에 DB에서 일부 색이 지워져도 다음 우주 진입 때 다시 설정 페이지로 간다.

### B. 라우트 배치

현재 루트 흐름:

```tsx
<SessionGate>
  <MembershipGate>
    <HomePage />
  </MembershipGate>
</SessionGate>
```

이 plan 이후 개인 우주 흐름:

```tsx
<SessionGate>
  <MembershipGate>
    <EmotionColorGate>
      <HomePage />
    </EmotionColorGate>
  </MembershipGate>
</SessionGate>
```

- `/`와 `/gift/$token`처럼 개인 우주나 내 별 생성으로 이어지는 보호 라우트가 `EmotionColorGate` 대상이다.
- `/emotion-colors`는 `SessionGate`와 `MembershipGate` 안이지만 `EmotionColorGate` 밖이다. 그래야 미완료 사용자가 설정을 끝낼 수 있다.
- `/invite`, `/admin`, `/landing`, `/sign-in`, `/u/$slug`는 감정색 게이트 밖이다.
- `safeRedirect`는 `/emotion-colors`, `/invite`, `/sign-in` 자기 자신/게이트 라우트를 redirect 대상으로 받지 않아 루프를 막는다.
- 데모 모드는 감정색 게이트를 통과한다. 서버 DB가 없고, 체험 우주는 기존 기본 팔레트를 쓴다.

### C. 설정 페이지 UX

`/emotion-colors`는 마케팅이나 설명 페이지가 아니라 바로 쓸 수 있는 설정 도구다.

데스크톱:
- 왼쪽 또는 상단: 13개 감정 swatch 목록. 사분면 순서(HAP → LAP → HAN → LAN → neutral)를 따른다.
- 선택된 감정: 큰 별 미리보기와 mood label, 현재 hex, 추천색 복귀 버튼.
- 오른쪽 또는 중앙: 현대적인 color picker.

모바일:
- 감정 목록은 가로 스크롤 또는 compact grid.
- picker는 화면 폭에 맞는 정사각형 saturation/value 면 + hue rail, 저장 버튼은 safe-area를 피한 하단 고정 command bar.

피커 계약:
- visible UI는 커스텀 피커다. 네이티브 `<input type="color">`는 접근성/폴백용으로 숨겨서 사용할 수 있지만 주 UI로 노출하지 않는다.
- 색 모델은 UI 내부에서 HSV/HSB로 조작하고, 저장은 `#RRGGBB` sRGB hex 하나로 한다.
- 큰 saturation/value 면, hue rail, 현재/추천 preview, hex 입력, 복사/붙여넣기 정규화, "추천으로 되돌리기", "전체 추천으로 초기화"를 제공한다.
- `window.EyeDropper` 지원 브라우저에는 eyedropper 버튼을 표시한다. 미지원이면 버튼을 숨기거나 disabled 처리한다.
- 모든 색은 허용한다. 너무 어두워 별이 잘 안 보이는 경우에는 저장을 막지 않고, 미리보기와 비차단 안내만 제공한다.
- pointer drag, touch drag, keyboard arrow adjustment, roving focus가 동작한다. reduced-motion에서는 선택 전환·preview glow 모션을 줄인다.

### D. 추천 팔레트

추천은 기존 29번 mood palette를 사용한다. 이 값들은 이미 별 렌더 기본값이자 "널리 알려진 감정색"의 앱 내 해석이다.

| mood | 라벨 | 추천 인상 |
|---|---|---|
| `joy` | 기쁨 | 골드 |
| `excitement` | 설렘 | 앰버 |
| `love` | 사랑 | 로즈 |
| `calm` | 평온 | 시안 |
| `gratitude` | 감사 | 민트 |
| `relief` | 안도 | 청록 |
| `anger` | 분노 | 레드 |
| `fear` | 두려움 | 바이올렛 |
| `stress` | 스트레스 | 마젠타 |
| `sad` | 슬픔 | 블루 |
| `tired` | 힘듦 | 슬레이트 |
| `emptiness` | 공허함 | 인디고 |
| `neutral` | 중립 | 그레이 |

구현은 `MOOD_PALETTE`를 `rgbToHex`로 변환해 추천 hex를 만든다. 추천 테이블을 별도 하드코딩하지 않는다. 사용자가 아무것도 바꾸지 않고 저장하면 이 13개 추천색이 그대로 `user_emotion_colors`에 저장된다.

### E. 저장과 캐시

저장은 full set 방식이다.

```ts
UpdateSettings({
  emotionColors: [
    { mood: Mood.JOY, color: '#FFD64D' },
    // ...
    { mood: Mood.EMPTINESS, color: '#525780' },
  ],
})
```

- 저장 버튼은 13개 mood가 모두 유효한 hex일 때만 활성화한다.
- 저장 전 클라이언트가 색을 대문자 `#RRGGBB`로 정규화한다.
- 서버는 plan 30의 기존 검증을 그대로 수행한다. 하나라도 invalid mood/color가 있으면 `InvalidArgument`이고 어떤 행도 쓰이지 않는다.
- 성공 응답의 `Settings`로 `applySettings`를 즉시 호출하고, React Query의 settings query cache도 성공 응답으로 갱신한다. 그렇지 않으면 redirect 후 `EmotionColorGate`가 stale empty settings를 보고 다시 `/emotion-colors`로 튕길 수 있다.
- 저장 후 `redirect`가 있으면 그 경로로, 없으면 `/`로 이동한다.
- 이미 완료된 사용자가 `/emotion-colors`에 직접 들어오면 현재 서버 색으로 초기화해 편집 페이지처럼 동작한다. "전체 추천으로 초기화"는 저장 전 로컬 draft만 바꾸고, 저장해야 DB에 반영된다.

### F. 렌더 반영

별·시냅스·작성 폼은 이미 `useAppearance.emotionColors`와 `resolveMoodRgb`를 통해 색 오버라이드를 소비한다.

- `StarField`는 `emotionColors` 변경을 dependency로 받아 per-instance `aMood`를 재베이킹한다.
- `UniverseCanvas`의 synapse endpoint 색도 `emotionColors` 변경 시 다시 계산된다.
- `MemoryForm`, `GiftPage`, `EvolutionPanel`, overlay 우주는 같은 `resolveMoodRgb` 경로를 쓴다.
- 별 좌표, 형태, 밝기, 연결, 원본 일기는 바뀌지 않는다. 색 attribute만 사용자 설정으로 바뀐다.

### G. FSD 배치

- `EmotionColorGate`는 app 라우트 게이트다. `app/ui/EmotionColorGate.tsx`에 두고 `entities/appearance`의 query/완료 판정 helper만 import한다.
- 완료 판정과 추천 팔레트 변환은 `entities/appearance/model/emotion-colors.ts` 같은 순수 모듈에 둔다. 이 모듈은 `shared/config/mood`만 import하고 React/DOM/three에 의존하지 않는다.
- **13감정 색 편집 로직은 재사용 가능한 `EmotionColorEditor`(`features/pick-emotion-colors`)로 추출한다**(change 09). 이 에디터는 두 곳이 공유한다: (1) `/emotion-colors` 최초 설정 게이트 라우트(`pages/emotion-colors`의 `EmotionColorPage` — 목적 무변: 필수 초기 설정 + `EmotionColorGate`), (2) 우주 셸 커스터마이즈 모달의 `감정 색` 탭(44). 풀세트 저장 규칙 + 드래프트/추천 시딩은 양쪽에서 동일하다.
- `/emotion-colors` 화면은 `pages/emotion-colors`에 두고 `EmotionColorEditor`를 호스팅한다. 에디터가 두 표면에서 재사용되므로 `features/pick-emotion-colors`에 둔다.
- `settings-query.ts`에는 `isEmotionColorComplete(settings)`와 `saveEmotionColors(colors)` 또는 이에 준하는 thin helper를 추가해, page/gate가 같은 기준을 쓰게 한다.

### H. 견고성

- `GetSettings` 로딩 중에는 우주를 그리지 않고 스플래시를 보여준다. 기본 팔레트로 먼저 우주를 띄웠다가 설정 페이지로 전환하는 깜빡임을 만들지 않는다.
- `GetSettings` 실패 시에는 우주로 들여보내지 않고 재시도 UI를 보여준다. 서버가 막힌 상태에서 색 설정 여부를 알 수 없기 때문이다.
- 로그아웃·계정 전환·체험 전환 시 `resetServerSettings()`가 기존처럼 감정색을 비운다. 다음 사용자는 자기 `GetSettings` 결과로 다시 판정된다.
- 저장 실패 시 draft는 유지하고 사유별 error-feedback을 보여준다. 서버 실패가 있어도 localStorage에 per-user 감정색을 영속하지 않는다.

## 수용 기준 (인수 조건)

1. 인증·멤버 사용자가 `/`에 진입할 때 `GetSettings` 응답에 13개 mood 전부의 유효한 `#RRGGBB` 감정색이 없으면 `/emotion-colors?redirect=<원래경로>`로 리다이렉트되고, `HomePage`와 `GetUniverse`는 아직 마운트되지 않는다.
2. 감정색 완료 여부는 최초 로그인 여부가 아니라 서버 설정 내용으로만 판정된다. 기존 사용자라도 13색 중 하나가 없거나 malformed이면 설정 페이지로 간다.
3. `/emotion-colors`는 인증·멤버 사용자만 접근 가능하고, `EmotionColorGate` 밖에 있어 미완료 사용자가 루프 없이 저장을 끝낼 수 있다.
4. 설정 페이지는 13개 mood를 모두 보여주며, 초기 draft는 서버에 저장된 색이 있으면 그 값, 없으면 `MOOD_PALETTE` 기반 추천색이다.
5. 사용자는 각 mood에 대해 추천색과 완전히 다른 임의의 `#RRGGBB` 색을 선택할 수 있다. 너무 어둡거나 특이한 색도 저장을 막지 않는다.
6. 컬러 피커는 커스텀 스펙트럼 UI(saturation/value 면, hue rail, preview, hex 입력, 추천 복귀, 지원 시 EyeDropper)를 제공하고, 브라우저 기본 color input만 덩그러니 보이는 UI가 아니다.
7. 저장은 `UpdateSettings.emotion_colors`에 13개 색을 한 번에 보내며, 성공 후 DB의 `user_emotion_colors`에는 호출자 user_id 기준 mood당 한 행이 저장된다.
8. 서버가 invalid mood/color를 거부하면 어떤 감정색도 부분 저장되지 않고, 페이지는 draft를 유지하며 오류를 보여준다.
9. 저장 성공 직후 `applySettings`와 settings query cache 갱신이 일어나, redirect 후 `EmotionColorGate`가 stale empty settings 때문에 다시 설정 페이지로 튕기지 않는다.
10. 저장된 색은 `useAppearance.emotionColors`에 반영되고, 별(`StarField`)과 시냅스(`UniverseCanvas`)가 refetch 없이 해당 mood 색으로 재베이킹된다.
11. 이미 완료된 사용자가 `/emotion-colors`에 직접 들어가면 현재 서버 색으로 편집할 수 있고, 저장 전에는 DB가 바뀌지 않는다.
12. 데모 모드, 미인증 랜딩, 공개 방문 우주(`/u/$slug`)는 이 게이트를 거치지 않고 기존 기본/공유 appearance로 렌더된다.
13. 로그아웃·계정 전환 후 이전 사용자의 감정색이 다음 사용자에게 보이지 않는다. 다음 사용자 판정은 새 `GetSettings` 응답으로만 한다.
14. FSD 단방향이 유지된다. 완료 판정/추천 변환은 React/DOM/three 미의존 순수 모듈이고, 라우트 게이트는 app, 화면은 pages, store/RPC helper는 entities/appearance에 둔다.
15. build·lint·typecheck·steiger를 통과하고, 기존 30번 테마/오브제 설정 저장과 41번 초대 게이트 동작에 회귀가 없다.

## 정책 영향 (policy / values)

> 아래 규칙은 구현 완료 시 policy에 반영한다. 미구현 상태에서는 이 plan이 출처다.

- **`policy/domain/star.md`** — 인증 개인 우주에서는 감정색이 단순 코드 기본값이 아니라 사용자별 `user_emotion_colors` 완료 세트로 확정된 뒤 렌더된다는 규칙을 추가. 별 색=mood 원칙은 유지한다.
- **`policy/domain/data-sync.md` 또는 신규 appearance 관련 policy** — per-user 감정색은 공용 PC에 영속하지 않고 서버에서 seed하며, 완료 판정은 `GetSettings` 결과의 13색 존재 여부라는 규칙을 추가.
- **`policy/ux/interaction.md` 또는 `motion-accessibility.md`** — 필수 설정 게이트, reduced-motion, keyboard/touch color picker 접근성 규칙을 추가.
- **`spec/values.yaml`** — 새 항목 없음. mood 목록 개수는 `MOOD_PALETTE`/`Mood` 키 집합에서 파생한다. 추천 팔레트의 색 배열은 values 제외 범주(배열 콘텐츠)이며 `shared/config/mood.ts`가 단일 출처다. picker의 시각 그라디언트/레이아웃 치수는 컴포넌트 디자인 세부로 두고 config 스칼라로 분리하지 않는다.
- **44 커스터마이즈와의 관계** — 44는 모양/스타일 판매와 소유권이고, 이 plan은 감정색 의미 설정이다. 감정색은 판매·잠금 대상이 아니며 모든 사용자에게 무료 필수 설정으로 남는다.

## Flutter 동등성 기준

- Flutter도 인증·멤버 사용자가 개인 우주를 열기 전 `GetSettings`로 13개 emotion color 완성 여부를 판정해야 한다.
- 완료 기준은 웹과 동일하다: 13 mood 전부의 유효 `#RRGGBB`. 최초 로그인 플래그나 로컬 캐시는 기준이 아니다.
- 저장은 `UpdateSettings.emotion_colors`에 13개 색을 한 번에 보내고, 성공 응답으로 로컬 appearance를 갱신한다.
- 추천색은 같은 `MOOD_PALETTE`를 hex로 변환한 값이어야 한다.
- 공개/체험/오프라인 폴백에서는 기존 기본 팔레트로 렌더가 깨지지 않아야 한다.
