# 21. 기억 분할 + AI 감정 (memory-fragmentation)

> 일기 한 편을 AI가 N개의 조각(엔그램)으로 나눠 각자의 감정을 가진 N개의 별로 띄우고, 같은 일기의 조각끼리는 강한 일내(intra-entry) 시냅스로 묶는다. 원본 일기는 여전히 1개의 불변 record. | 범위: FS | 선행: 20 | 차단: 22, 23, 25, 26

## 목적 (Why)
지금까지 우주는 **1 일기 = 1 record = 1 memory = 1 별**이다(`memory/repository_pg.go` `RecordMemory` 트랜잭션, `RecordMemoryResponse{memory_id}` 단일). 그러나 하루는 여러 감정을 품는다 — 아침의 평온, 낮의 분노, 밤의 안도. 사람의 기억도 사건 경계에서 **이산적 흔적(discrete trace)** 으로 끊겨 부호화된다(event segmentation). 이 스펙은 사용자의 #1 요구이자 시리즈의 **핵심 기능**을 구현한다: 일기 1편을 20번 스펙이 깐 `ai.Extractor`로 **N개의 조각**으로 나누고, 각 조각을 **자기 감정·강도를 가진 별**로 fan-out하며, 같은 일기에서 나온 조각끼리는 **강한 일내 시냅스(w=0.8)** 로 묶는다(within-event binding). 감정은 더 이상 사용자가 `<select>`로 고르지 않고 **AI가 조각마다 감지**한다(수동 입력은 선택적 fallback). 원본 일기는 한 글자도 바뀌지 않는 단 1개의 record로 남는다(헌법1).

## 범위
- **포함:**
  - BE fan-out 경로: `RecordMemory`는 record 1개 + **`extract` job 1개**만 enqueue(별은 아직 없거나 "조각내는 중" placeholder 1개). 새 워커 `extract` 단계가 `ai.Extractor.Segment`로 조각 N개를 만들어 **N개 `InsertMemory`**(공유 `record_id`·`fragment_index`·`fragment_text`·`mood`·`intensity`) + **조각당 `embed` job 1개**를 fan-out.
  - 마이그레이션: `memories`에 `mood`/`intensity`/`fragment_index`/`fragment_text` 컬럼 추가(가변 별 레이어에 조각별 감정을 둔다 — 헌법1이 불변 record에 조각 데이터를 두는 것을 금지).
  - 조각별 임베딩: `GetMemoryForEmbed`가 `fragment_text`(NULL이면 `r.body` fallback)를 반환해 조각마다 따로 임베딩.
  - 일내 시냅스: 같은 `record_id` 조각끼리 자동 연결(`link_type='intra_entry'`, `w_intra=0.8`). 일간(cross-entry)은 조각 임베딩 KNN을 유지하되 `w_intra` 아래로 캡.
  - proto: `RecordMemoryResponse{record_id, repeated memory_ids}`, `RecordMemoryRequest`의 `mood`/`intensity`는 선택적 힌트로 강등.
  - FE: `MemoryForm`에서 감정 `<select>`+강도 슬라이더 제거(본문+날짜만, 선택적 "수동 감정" 토글 fallback), `useRecordMemory`가 "조각내는 중" 표시 후 `GetUniverse` refetch로 N개 조각 별 수신, 스토어에 `addStars`/`replaceMany` 추가, 조각별 `seed=stableHash`.
  - concept.md에 "기억 분할(event segmentation)" 설계 결정 절 추가(T001).
  - 랜딩 신규 카드(`FragmentationCard`) + 체험(체험 모드에서 다감정 일기 → N개 별 분할).
- **비목표(Out of scope):**
  - 경쟁적 할당(흥분성 기반 배치)·라이브 force-sim 좌표 배선 → **22**.
  - 재공고화 양방향 재성형·`evolution_history` → **23**, 변천사 UI → **24**.
  - 감쇠 변조(λ_eff)·`co_activation_count` 노출 → **26**.
  - 요즘 상태(ambient mood) 집계·배경 → **25**.
  - 야간 공고화 배치 → **27**.
  - `ai.Extractor` 포트 배선·실제 LLM 어댑터·세그멘테이션 알고리즘 자체 → **20**(본 스펙은 20이 깐 `Extractor.Segment`를 소비만 한다).
  - 원본 일기로 별 찾기·엔그램/별 길찾기·조망 하이라이팅 → **28**(본 작업은 `record_id`·`fragment_text` 데이터를 만들고, 길찾기 UI·`Star.record_id` 노출은 28이 소비).

## 설계 요점

**proto 계약(누적 변경 — 21이 추가하는 부분, `cosimosi.v1`).** 일기 저장은 조각 fan-out이 비동기이므로 즉시 별을 못 돌려준다. record_id만 확정 반환하고 `memory_ids`는 비어 있을 수 있다(조각은 다음 `GetUniverse` refetch에서 도착 — 헌법6, "별은 비동기 도착"). `RecordMemoryRequest.mood`/`intensity`는 **선택적 힌트**(AI가 감정을 감지하므로 필수 아님; 비면 무시).
```proto
// RecordMemory: 원본 1 record + 비동기 extract → N 조각 별
message RecordMemoryRequest {
  string body = 1;
  Mood mood = 2;            // optional hint (AI가 조각 감정 감지; 비면 무시)
  double intensity = 3;     // optional hint
  string entry_date = 4;    // YYYY-MM-DD
  string idempotency_key = 5;
}
message RecordMemoryResponse {
  string record_id = 1;            // 불변 원본 id (확정)
  repeated string memory_ids = 2;  // 조각 별 id들 — 비어 있을 수 있음(refetch로 도착)
}
```
> 기존 `RecordMemoryResponse{memory_id=1}`는 **단일 필드 제거 + 신규 두 필드**로 교체된다. FE 낙관적 흐름이 단일 별 → "조각내는 중" placeholder로 바뀐다(아래).

**마이그레이션(신규 goose 파일 — 기존 00001은 단일 권위라 수정 금지, 헌법: append-only DDL).** 조각별 감정은 **가변 별(memories)** 레이어에 둔다(불변 record에 조각 데이터를 두면 헌법1 위반 — record는 사용자가 쓴 1개의 원본).
```sql
-- +goose Up
ALTER TABLE memories ADD COLUMN mood           TEXT;            -- 조각 감정(AI 감지, nullable)
ALTER TABLE memories ADD COLUMN intensity      REAL;            -- 조각 강도 0~1(nullable)
ALTER TABLE memories ADD COLUMN fragment_index INT NOT NULL DEFAULT 0;  -- 일기 내 조각 순서(0-based)
ALTER TABLE memories ADD COLUMN fragment_text  TEXT;            -- 임베딩용 조각 요약(record 본문 편집 아님), NULL이면 r.body fallback
-- +goose Down
ALTER TABLE memories DROP COLUMN fragment_text;
ALTER TABLE memories DROP COLUMN fragment_index;
ALTER TABLE memories DROP COLUMN intensity;
ALTER TABLE memories DROP COLUMN mood;
```
> `records.mood`/`records.intensity`는 **선택적 전체-일기 사용자 힌트(prior)** 로 남는다(요구 아님). `ListMemoriesByUser`/`ListDormant`는 이제 `m.mood`/`m.intensity`(memories)를 읽고, `GetRecordByMemory`는 여전히 records JOIN으로 불변 원본 body를 읽는다.

**extract 워커 fan-out(`job` 패키지 + `worker.go` 새 단계).** `RecordMemory`는 record 1개와 `extract` job 1개만 enqueue한다(별은 아직 없거나 "조각내는 중" placeholder 1개). 새 `extract` 단계가 조각을 만든다.
```
RecordMemory(tx): InsertRecord(record_id) → EnqueueJob(kind='extract', record_id 참조용 memory_id placeholder) → commit
worker.processOne: Claim(KindExtract) → handleExtract
handleExtract(j):
  rec ← GetRecordForExtract(j)                       // r.body, r.user_id, r.entry_date
  frags ← ai.Extractor.Segment(ctx, rec.Body)        // []Segment{Text, Mood, Intensity} (20 소관; N≤cap)
  for i, f := range frags:                            // fan-out (한 트랜잭션 권장)
      mid ← InsertMemory(record_id, fragment_index=i, fragment_text=f.Text, mood=f.Mood, intensity=f.Intensity)
      EnqueueJob(kind='embed', memory_id=mid)         // 조각당 embed job 1개
  BatchUpsertLinks(intra_entry: 모든 (mid_a, mid_b) 쌍, w=0.8)   // 일내 결속(within-event binding)
  Complete(j)
worker.handle(embed):  // 기존 단계 — fragment_text로 임베딩
  m ← GetMemoryForEmbed(memory_id)  // m.fragment_text(NULL이면 r.body)
  vec ← Embed(m.Text) → UpsertEmbedding → KnnNearest(조각 임베딩) → BatchUpsertLinks(semantic, w<w_intra) → Complete
```
- **placeholder 처리(택1, DoD 검증):** (A) `RecordMemory`가 placeholder memory를 만들지 않고 `extract` job만 record를 참조 → `jobs.memory_id`가 record를 가리키도록 `GetRecordForExtract`를 record 기준으로 조회, 또는 (B) "조각내는 중" placeholder memory 1개를 만들고 extract 후 첫 조각으로 `ReplaceMemory`. 본 스펙은 **(A)** 를 채택(별은 비동기 도착 — 헌법6, FE가 "조각내는 중" 카피 표시). `jobs.memory_id`는 NOT NULL FK→memories이므로 (A)에서는 record_id를 별도 컬럼/조인으로 넘긴다(아래 영향 파일).
- **조각 cap:** 비용은 일기당 `1 extract + N embed`(N×). `ai.Extractor.Segment`(20)가 조각 수에 상한을 둔다(예: N≤8); 본 스펙은 그 상한을 신뢰하고 fan-out 한다.

**일내 시냅스(`db/queries/link.sql` 확장).** 같은 `record_id`의 모든 조각 쌍을 `link_type='intra_entry'`, `weight=0.8`로 결속한다(05의 `BatchUpsertLinks`가 'semantic' 리터럴을 박으므로 별도 쿼리/파라미터화 필요). `a_id<b_id` 정규화는 05처럼 DB 콜레이션에서(LEAST/GREATEST).
```sql
-- name: BatchUpsertIntraEntryLinks :exec
-- 일내(within-event) 결속: 같은 일기 조각끼리 강한 고정 가중치(0.8). a<b는 DB 콜레이션에서 정규화.
INSERT INTO memory_links (a_id, b_id, weight, user_id, link_type)
SELECT LEAST(a, b), GREATEST(a, b), 0.8, u, 'intra_entry'
FROM ( SELECT unnest(@a_ids::text[]) AS a, unnest(@b_ids::text[]) AS b, unnest(@user_ids::text[]) AS u ) AS pairs
ON CONFLICT (a_id, b_id) DO UPDATE SET weight = GREATEST(memory_links.weight, EXCLUDED.weight);
```
> 일간(cross-entry) semantic 링크는 05의 `w0=clamp(α·cos+temporal,0,1)`를 그대로 쓰되, 일내보다 약하도록 **상한을 `w_intra` 미만으로 캡**(`min(w0, 0.79)`)해 같은-사건 결속이 항상 더 강하게 보이도록 한다.

**조각별 임베딩(`GetMemoryForEmbed` 확장).** 조각 텍스트로 따로 임베딩한다. `fragment_text`가 NULL(수동/단일 조각 경로)이면 `r.body`로 fallback.
```sql
-- name: GetMemoryForEmbed :one
SELECT m.user_id, COALESCE(m.fragment_text, r.body) AS text, r.entry_date
FROM memories m JOIN records r ON r.id = m.record_id
WHERE m.id = $1;
```

**FE 흐름 변화(`features/record-memory`).** `MemoryForm`은 본문 textarea + 날짜만(감정 `<select>`·강도 슬라이더 제거; 접힌 "수동 감정" 토글이 fallback 힌트). `useRecordMemory`는 단일 낙관적 별을 띄우는 대신:
1. 제출 → `RecordMemory(body, entry_date[, hint])` 호출 → `res.record_id` 수신, `memory_ids`는 보통 빈 배열.
2. "조각내는 중…" 상태 표시(즉시 별을 못 만든다 — 조각 수·감정을 클라가 모름).
3. 짧은 지연 후 `getUniverse()` refetch → N개 조각 별이 도착(`addStars`로 묶음 추가, 조각 id마다 `seed=seedFromId(memory_id)`).
4. (refetch가 조각을 아직 안 보여줄 수 있으므로) 가벼운 재시도/폴 1~2회 후 안내.
- 스토어는 단건 `addStar`/`replaceStar` 외에 **`addStars(nodes[])`**(묶음 등장)·**`replaceMany(ids[], nodes[])`** 를 추가(fan-out 묶음 교체). 헌법2: 서버 별은 절대 삭제하지 않으므로 placeholder는 `temp-` 접두 임시 별만 정리.

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 인증 사용자가 여러 장면·감정이 담긴 일기로 `RecordMemory`를 호출 THE SYSTEM SHALL 불변 `records` 1행만 쓰고 `extract` job 1개를 enqueue한 뒤, 워커가 `ai.Extractor.Segment`로 나눈 조각마다 `memories` 1행(공유 `record_id`, `fragment_index`, `fragment_text`, AI 감지 `mood`/`intensity`)을 생성한다. _(헌법1·7)_
1.2 WHEN 조각 fan-out이 끝나면 THE SYSTEM SHALL 조각마다 `kind='embed'` job을 enqueue하고, embed 워커는 `GetMemoryForEmbed`가 돌려준 `fragment_text`(NULL이면 `r.body`)로 **조각별로** 따로 임베딩한다. _(헌법7)_
1.3 WHEN 같은 일기에서 2개 이상 조각이 나오면 THE SYSTEM SHALL 그 조각 쌍을 `link_type='intra_entry'`, `weight=0.8`로 결속하고, 일간 semantic 링크는 항상 `w_intra` 미만으로 캡되어 같은-사건 결속이 더 강하다. _(within-event binding)_
1.4 WHEN 조각마다 감정이 다르면 THE SYSTEM SHALL 각 조각 별이 자기 `mood`/`intensity`(=색·강도)를 가지며, `GetUniverse`의 `Star.mood`/`intensity`는 `memories`에서(더 이상 records JOIN 아님) 읽힌다.
1.5 THE SYSTEM SHALL 한 일기에 대해 `records`를 정확히 1행만 쓰고 그 행을 절대 UPDATE/DELETE 하지 않는다(조각화는 가변 별 레이어에서만 일어난다; `rg "UPDATE records|DELETE FROM records"` 무결과). _(헌법1)_
1.6 WHEN 사용자가 감정을 직접 고르지 않아도 THE SYSTEM SHALL AI가 조각 감정을 감지해 채우며, `RecordMemoryRequest`의 mood/intensity는 선택적 힌트로만 작동한다(비어도 정상). 수동 토글을 켜면 그 힌트가 fallback으로 쓰인다.
1.7 WHEN `RecordMemory`가 응답하면 THE SYSTEM SHALL `RecordMemoryResponse{record_id, memory_ids}`를 돌려주고(`memory_ids`는 비어 있을 수 있음), 클라는 "조각내는 중"을 표시한 뒤 다음 `GetUniverse` refetch에서 N개 조각 별을 받는다(폴링/스트리밍 아님). _(헌법6)_
1.8 GIVEN 같은 `record_id`로 두 번 `GetUniverse`를 호출 WHEN 조각 별을 매핑 THE SYSTEM SHALL 각 조각 별의 `seed`가 그 `memory_id`로 재현 가능하게 파생되어(동일 id→동일 모양) 새로고침 후에도 같은 별 모양을 보인다. _(헌법3)_
1.9 THE SYSTEM SHALL 좌표를 서버에서 만들지 않으며 조각 별의 위치는 클라 force-sim에서 창발한다(서버는 가중치 그래프만 권위; 라이브 좌표 배선은 22). _(헌법3)_
1.10 THE SYSTEM SHALL 도메인/proto/sqlc 경계를 유지한다(proto=전송 DTO, 도메인 순수 무태그, sqlc=영속; `Segment`/extract 흐름이 도메인에 db/proto 태그를 새지 않게). _(헌법5)_
2.1 WHEN 체험 모드에서 다감정 일기를 작성 THE SYSTEM SHALL 그 일기를 색이 다른 N개 별로 분할하고 조각끼리 강한 intra-entry 선으로 묶어 보여준다(같은 record body 공유).
2.2 WHEN 랜딩의 `FragmentationCard`가 뷰포트/hover로 활성 THE SYSTEM SHALL 일기 1편이 색이 다른 3개 별로 갈라지며 강한 선으로 이어지는 모습을 VizStar/VizSynapse로 보여준다.
