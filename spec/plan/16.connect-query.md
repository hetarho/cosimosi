# 16. connect-query — 선언적 데이터 계층 + 캐싱 전략

> 죽어있는 connect-query/TanStack Query 레이어를 살리고, 프로덕트 데이터 특성에 맞춘 캐싱 전략(이벤트 기반 무효화·불변 캐시·병합 동기화)을 세운다. | 범위: FE (베타 하드닝) | 선행: — | 차단: 17

## 목적 (Why)

**전제 교정:** "Connect RPC가 안 쓰인다"가 아니다 — Connect RPC 자체는 02부터 제대로 사용 중이다(`shared/api/transport.ts`의 typed `memoryClient` + 인증 인터셉터 + keepalive fetch). 안 쓰이는 것은 그 위의 **선언적 쿼리 레이어**다:

- `@connectrpc/connect-query`가 의존성에 있고 `QueryClientProvider`가 앱을 감싸고 있지만(`app/query-client.ts` 주석도 "추후 connect-query가 이 위에 얹힌다(spec 02)"라고 예고), **`useQuery`/`useMutation` 사용처가 0개**다.
- `getUniverse()`는 `HomePage`의 맨 `useEffect`에서 명령형으로 1회 호출되고, 실패하면 **`console.error`만 남기고 침묵**한다(`HomePage.tsx` 147–153행). 로딩 인디케이터·에러 화면·재시도·캐시·refetch가 전부 없다.
- Architecture §4.6의 핵심 스토리 — *"별은 즉시, 연결은 다음 GetUniverse refetch에서"* — 를 실행할 **refetch 메커니즘 자체가 없다**. 일기를 쓴 뒤 워커가 만든 시냅스는 새로고침 전엔 영영 안 보인다.
- 같은 별을 다시 열 때마다 `RecallMemory`를 기다린다 — **원본 일기는 불변(헌법 §1)인데** 매번 네트워크를 왕복한다.

베타 사용자 50명에게는 "흰 우주에서 멈춘 로딩"과 "쓴 일기의 연결이 안 생기는 것처럼 보이는 경험"이 곧 이탈이다.

## 범위

**포함**
- `TransportProvider` 마운트 + 기존 transport 재사용 (인증 인터셉터·keepalive 유지)
- 읽기 RPC의 쿼리화: `GetUniverse`, `ListDormant` — 로딩/에러/재시도 UI 포함
- **캐싱 전략 구현(아래 §캐싱 전략이 단일 기준):** 쿼리별 staleTime/gcTime, 이벤트 기반 무효화 맵, Record 불변 캐시, **병합 동기화(replace→merge)**
- `RecordMemory` 성공 후 **지연 invalidate**(워커 임베딩 완료 추정 ~10s 후 1회)로 §4.6 refetch 스토리 구현
- (선택) `GetUniverse`/`ListDormant`의 HTTP GET 노출(proto `idempotency_level` + `useHttpGet`) — §4.4 정합
- 체험 모드(`isDemoMode`) 경로 보존

**비목표 (Out of scope)**
- **`ReinforceLinks` flush의 mutation화 금지.** beforeunload/visibilitychange + keepalive fetch로 페이지 teardown을 견디는 현재 설계(11, 헌법 §6)는 React 훅 생명주기에 묶이면 깨진다. 명령형 유지. flush에 따른 캐시 무효화도 하지 않는다(아래 §캐싱 전략 — max-merge가 수렴 보장).
- 낙관적 record 플로우(`use-record-memory.ts`의 addStar→replaceStar/rollback) 전면 재작성 — 기존 로직 유지, invalidate 연결만 추가.
- 60fps 데이터(좌표 버퍼)의 query화 — 렌더는 ref/스토어 경로 그대로(§3.2).
- 스트리밍/SSE/무한 폴링(헌법 §6 unary 전용 — 무효화는 횟수 제한 이벤트 기반만).
- **캐시 영속화(persistQueryClient/localStorage) 도입 안 함** — universe 페이로드는 body 없는 저민감 메타지만(proto `Star`에 body 없음) mood도 개인 데이터이고 공용 PC 시나리오가 있다. 오프라인/즉시 기동이 필요해지면 별도 스펙으로.
- CDN 응답 캐시 — `Authorization` 헤더 기반 인증 응답이라 Cloudflare 기본 캐시 대상이 아님. GET *노출*까지만(추후 옵션을 열어두는 것).
- 에러 바운더리·전역 에러 화면(17에서).

## 캐싱 전략 (이 스펙의 핵심 — 단일 기준)

### 왜 이 전략인가 — 데이터 변경 토폴로지

cosimosi의 데이터는 일반 CRUD 앱과 달리 **캐싱에 극단적으로 유리한 불변식**들을 갖는다. 전략은 전부 여기서 도출된다:

1. **단일 작성자 우주.** 한 사용자의 우주를 변경하는 주체는 ① 사용자 자신(record/recall/reinforce — 전부 *이 클라이언트가 시작*) ② 비동기 임베딩 워커(record 직후 시냅스 생성) 둘뿐이다. 외부 작성자가 없으므로 **시간 기반 만료(짧은 staleTime)는 낭비**고, **이벤트 기반 무효화**(내가 쓴 직후에만 refetch)가 정확하고 싸다. 유일한 외부 드리프트는 멀티 디바이스(폰에서 쓰고 데스크탑에서 봄) — focus refetch + 보수적 staleTime으로 커버한다.
2. **원본은 불변(헌법 §1).** `Record`(일기 본문)는 절대 변하지 않는다 → `staleTime: Infinity`로 **영구 캐시**. 같은 별을 다시 열면 스피너 없이 즉시 본문 표시.
3. **별·시냅스는 무삭제(헌법 §2) + MVP 변경은 단조.** 서버는 행을 지우지 않고, weight는 증가만(+0.05, cap 1.0), `last_recalled_at`/`last_activated_at`은 전진만 한다. 따라서 서버 응답과 로컬 상태의 충돌은 전부 **`max()` 병합으로 해소**된다(레이스 안전, 재시도 멱등).
4. **밝기·감쇠는 렌더 시 파생(§6).** 캐시된 타임스탬프에서 매 프레임 계산하므로 **캐시가 오래돼도 시각적으로 썩지 않는다** — staleTime을 길게 잡아도 우주는 계속 "살아서" 어두워진다.
5. **좌표는 캐시 밖.** 좌표는 클라 창발(§3)이고 스토어의 `StarNode.index`=인스턴스 슬롯이다. 쿼리 동기화가 기존 별 객체를 갈아끼우면 슬롯·좌표가 흔들린다 → **전체 교체(setStars) 금지, 병합 필수**.

### 쿼리별 정책표

| 캐시 | key | staleTime | gcTime | focus refetch | 무효화/갱신 트리거 |
|---|---|---|---|---|---|
| `GetUniverse` | connect-query 생성 키 | **5분** | **30분** (route 이동 후 복귀 시 즉시 표시) | **true** (stale일 때만 발화 — 멀티 디바이스 드리프트 커버) | RecordMemory 성공 **+10s 지연 invalidate 1회** (+30s 1회 선택 — 워커 지연 대비). 그 외 없음 |
| `ListDormant` | connect-query 생성 키 | **5분** (임계 ~100일짜리 초저속 데이터) | 10분 | 기본(false) | **RecallMemory 성공 시 invalidate** (회상된 별은 잠에서 깸 → 목록 이탈) |
| `Record` (원본) | `['record', memoryId]` 형태 | **Infinity** (불변 — 헌법 §1) | 30분 | — | 없음. RecallMemory 응답으로 `setQueryData` 시드 |

- **전역 기본값(`query-client.ts`)은 유지**(staleTime 60s·retry 1·focus refetch false) — 위 정책은 쿼리별 오버라이드로. 새 쿼리가 생기면 기본값이 보수적 안전망이 된다.
- **무한 폴링 금지.** record 후 갱신은 *횟수 제한* 지연 invalidate(+10s, 선택 +30s)뿐이다. 새 별이 연결을 못 받는 경우(τ=0.75 미달 이웃 없음)도 정상이므로 "링크가 생길 때까지" 류의 조건 폴링을 만들지 않는다.

### 변이별 캐시 상호작용 (무효화 맵)

| 변이 | 캐시 동작 |
|---|---|
| `RecordMemory` | 낙관적 별은 Zustand가 처리(기존 유지 — 쿼리 캐시에 낙관 쓰기 안 함). 성공 시 **+10s 후 `GetUniverse` invalidate 1회**(워커가 만든 시냅스 수신, §4.6). |
| `RecallMemory` | **touch는 매 열림마다 발사**(재점화 의미론 — 캐시 히트가 touch를 생략하면 감쇠 모델이 굶는다). 응답 `Record`를 `setQueryData(['record', id])`로 영구 시드. **재열람 = 캐시에서 즉시 본문 + touch는 백그라운드 발사**(stale-while-revalidate의 "revalidate"가 곧 재점화). touch 실패는 비차단(다음 열람에 재시도, Sentry 기록). 성공 시 `ListDormant` invalidate. |
| `ReinforceLinks` flush | **캐시 무효화 없음.** 서버는 batch_id 멱등 합산이고, 미flush 델타가 있는 상태에서 refetch가 와도 아래 병합 규칙(max)이 시각적 후퇴를 막는다. 다음 sync에서 자연 수렴. |

### 동기화 = 병합 (replace → merge)

쿼리 데이터 → Zustand 렌더 스토어 반영은 **전체 교체가 아니라 병합**이다. refetch가 도입되는 순간 `setStars(전체 교체)`는 ① 제출 중 temp 별 ② 기존 별의 슬롯/좌표 ③ 로컬이 앞서 있는 타임스탬프를 깨뜨릴 수 있다:

- **별:** `memory_id` 기준. **기존 `StarNode` 객체는 보존**(인스턴스 슬롯 `index`·seed·핀 좌표 유지 — 헌법 §3), 서버에만 있는 새 별은 끝에 append, `lastRecalledAt = max(서버, 로컬)`(내가 방금 회상한 게 서버 응답보다 앞설 수 있다).
- **시냅스:** `(a_id, b_id)` 키 병합. `weight = max(서버, 로컬)`, `lastActivatedAt = max` — 미flush 강화 델타가 있어도 **시각적 후퇴(굵었다 가늘어짐) 없음**. 단조 증가(MVP)라 max가 곧 정답이다. v1+에서 양방향 재공고화(#20·#23)가 들어오면 이 규칙을 재방문한다(스펙에 주석으로 남길 것).
- **temp 별:** 병합은 `temp-` 별을 건드리지 않는다. 서버 확정 별과의 교체는 기존 `replaceStar`가 소유(이미 mid-refetch 케이스를 방어함 — `store.ts` 33–37행 "If the temp star is gone (e.g. a GetUniverse refetch ran mid-submit), APPEND").
- 병합 함수는 **순수 함수로 `entities/memory/model`에**(react 미의존 — pure 레이어 lint) 두고 단위 테스트한다. 무삭제(§2) 전제라 병합은 추가-단조(append-monotone)로 단순하다.

### (선택) HTTP GET 노출 — §4.4 정합

- proto에 `option idempotency_level = NO_SIDE_EFFECTS`를 `GetUniverse`·`ListDormant`에 선언 + transport `useHttpGet: true` → 멱등 읽기가 HTTP GET으로 나간다. 지금 당장의 이득은 작지만(인증 응답이라 CDN 캐시 비목표) §4.4가 기록한 방향이고 비용이 거의 없다.
- proto 변경이므로 `pnpm gen` 재생성 필수(Windows: Docker/WSL). BE 핸들러는 connect-go가 GET을 자동 수용 — 코드 변경 없음 확인만.

## 설계 요점

```
app/App.tsx
  └─ TransportProvider(transport)  ← shared/api/transport.ts가 transport를 export
      └─ QueryClientProvider(queryClient)   (기존 — 전역 기본값 유지)

entities/memory/api/
  universe-query.ts   ← GetUniverse 쿼리 옵션(staleTime 5m·gcTime 30m·focus true) + demo 분기
entities/memory/model/
  merge.ts            ← 병합 순수 함수(별·시냅스, max 규칙) + 단위 테스트
pages/home/ui/HomePage.tsx
  useQuery(getUniverse) → 성공 시 merge → Zustand 반영(useEffect)
  isLoading → 로딩 인디케이터 / isError → 에러 카드 + 재시도 버튼
features/recall/
  recallMemory 호출부: setQueryData(['record', id]) 시드 + 재열람 캐시 읽기, ListDormant invalidate
features/record-memory/model/use-record-memory.ts
  성공 시: 기존 낙관적 replaceStar 유지 + ~10s 후 invalidateQueries(GetUniverse) 1회
pages/dormant/
  list-dormant 쿼리화(staleTime 5m) + 동일 로딩/에러 패턴
```

- **Zustand는 렌더 스토어로 유지.** 쿼리는 "서버 상태의 소유자", 스토어는 "렌더 트리가 60fps로 읽는 투영". 쿼리 success → **병합** → `setStars`/`setEdges`. (쿼리 데이터를 R3F가 직접 구독하지 않는다 — 리렌더 폭발 방지, §3.2)
- **순수성 주의:** 쿼리 *옵션·병합 정의*는 `entities/memory/{api,model}`(react import 금지 — ESLint pure 레이어), `useQuery` *훅 호출*은 ui 레이어에서. transport export는 기존 `shared/api` 공개 API에 추가.
- **demo 모드:** queryFn 안에서 `isDemoMode()` 분기(기존 `get-universe.ts` 패턴 이전). 데모도 같은 쿼리 캐시 경로를 타게 해 UI 분기를 없앤다.

## 수용 기준 (EARS/GWT)

- 1.1 WHEN `/universe` 진입 후 GetUniverse가 응답 전이면 THE SYSTEM SHALL 로딩 인디케이터를 표시한다(빈 우주를 "별이 없다"로 오인시키지 않는다).
- 1.2 WHEN GetUniverse가 실패하면 THE SYSTEM SHALL 에러 상태 UI와 재시도 버튼을 표시하고, 재시도 클릭 시 refetch한다. (silent console.error 금지)
- 1.3 WHEN RecordMemory가 성공하면 THE SYSTEM SHALL 낙관적 별을 즉시 표시하고(기존 동작 유지), 약 10초 후 GetUniverse를 1회 invalidate하여 워커가 생성한 시냅스가 새로고침 없이 나타난다.
- 1.4 WHEN GetUniverse refetch가 도착하면 THE SYSTEM SHALL 병합 규칙(기존 별 객체·슬롯 보존, 새 별 append, weight/timestamp는 max)으로 스토어에 반영한다 — 기존 별의 좌표가 흔들리거나 시냅스가 가늘어지는 시각적 후퇴가 없어야 한다.
- 1.5 WHEN 같은 별을 다시 열면 THE SYSTEM SHALL 본문을 캐시에서 즉시 표시하고(스피너 없음), 재점화 touch(RecallMemory)는 백그라운드로 매번 발사한다.
- 1.6 WHEN 잠든 별을 회상하면 THE SYSTEM SHALL ListDormant 캐시를 invalidate하여 다음 `/dormant` 진입 시 그 별이 목록에서 빠져 있다.
- 1.7 WHEN 체험 모드(`isDemoMode`)이면 THE SYSTEM SHALL 백엔드 호출 없이 더미 우주를 같은 쿼리 경로로 로드한다.
- 1.8 WHEN 탭이 hidden/beforeunload 되면 THE SYSTEM SHALL 기존과 동일하게 keepalive fetch로 reinforce 배치를 flush한다(회귀 없음). AND 미flush 델타가 남은 채 refetch가 와도 병합(max)이 시각적 후퇴를 만들지 않는다.
- 1.9 WHEN `/dormant` 진입 시 THE SYSTEM SHALL ListDormant를 쿼리로 호출하고 로딩/에러 상태를 표시한다.
