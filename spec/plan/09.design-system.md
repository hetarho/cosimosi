# 09. design system

> Design system - Tailwind tokens, accessible UI primitives, theme/background seam, and app-wide visual foundations.
> Scope: FE | Status: planning

## Purpose

Phase 4 will build visible product slices one at a time, but it should not invent buttons, panels, focus rings, color
tokens, or theme behavior inside each slice. This unit creates the frontend design-system foundation: tokens, global
styles, accessible primitives, and the seam where future themes can influence the background without becoming domain
logic.

## Scope / Non-goals

**In scope**
- Web styling (Tailwind + CSS-variable tokens) **and** mobile styling (NativeWind/RN), both set up in Phase 1 over one
  cross-platform token map (plain TypeScript, no DOM).
- Global reset/base styles, typography scale, spacing, radius, elevation, focus rings, and motion preference handling.
- Shared UI primitives: Button, IconButton, TextField, TextArea, Select/Menu, Tabs, SegmentedControl, Switch/Checkbox,
  Slider/Stepper, Dialog/Drawer, Tooltip, Toast, Badge, Skeleton, and VisuallyHidden. Primitives are cross-platform —
  shared/responsive by default, with a `*.native.tsx` sibling only where DOM and RN (`View`/`Text`/gesture) genuinely
  differ (ARCHITECTURE §3.5).
- Accessibility defaults: keyboard operation, focus trapping for modal surfaces, visible focus states, aria labeling
  patterns, reduced-motion support, and contrast checks for token pairs.
- Theme seam: app theme tokens can drive page/background styling and later universe background parameters without
  mixing theme choice into domain state.
- Story/test harness examples for primitives through [12.test-harness](12.test-harness.md) or the chosen component
  development surface.

**Non-goals**
- No final product screens, landing page, universe renderer, or 3D star/cell/filament visuals.
- No mood-color model, customization economy, shop, or user-created themes.
- No domain emotion palette decisions beyond reserving a safe token seam for later product colors.
- No dependency on MVP shadcn/radix code unless consciously reintroduced through this clean design-system unit.
- No hardcoded product copy; labels come through [08.i18n-seam](08.i18n-seam.md).

## Design

### Token layers

The design system separates three layers:

1. **Foundation tokens**: color roles, type scale, spacing, radius, z-index, focus ring, elevation, and motion timing.
2. **Component tokens**: primitive-specific slots such as button background, button foreground, field border, dialog
   backdrop, and tooltip surface.
3. **Theme/background seam**: app-level theme values that can influence non-domain backgrounds.

Foundation and component tokens are emitted from one source as **both** CSS variables (web) and a plain TypeScript
token map with no DOM dependency (mobile/NativeWind) — two outputs, one source, both in Phase 1.

### Primitive rules

Primitives are domain-agnostic and live under `shared/ui`. Most are responsive and shared verbatim across web and
mobile; a `*.native.tsx` sibling exists only where the DOM and RN primitives genuinely differ. They accept:

- controlled/uncontrolled values where conventional for the primitive;
- accessible names and descriptions;
- disabled/loading/error states;
- icon slots when an icon is appropriate;
- test ids only through a documented test prop or helper.

They do not import feature/domain slices. They do not contain product copy. They do not own data fetching.

### Accessibility baseline

The unit sets baseline requirements that future UI must inherit:

- all interactive primitives are keyboard reachable;
- modal surfaces trap focus and restore focus;
- focus rings are visible;
- disabled controls are conveyed visually and semantically;
- reduced motion is honored for transitions;
- token color pairs pass WCAG contrast for text-sized content.

### Theme and background seam

Theme is presentation state. It can select visual tokens and background styles, but it must not alter domain data such
as emotion, engram strength, recall state, or graph layout. Future product units may map domain emotion to visual color
inside the rendering projection, not inside the generic design system.

## Acceptance Criteria

1. Web Tailwind/global CSS tokens **and** a DOM-free cross-platform token map (mobile/NativeWind) exist from one token
   source, imported only at each app's composition boundary.
2. Shared UI primitives are implemented under `shared/ui` with named exports and no imports from feature/domain slices;
   they render on web and mobile, with `*.native.tsx` siblings only where DOM and RN genuinely differ.
3. Primitive examples/tests cover default, disabled, loading/error where applicable, focus, and keyboard behavior.
4. UI copy in primitives is provided through props or i18n message calls; primitives do not embed product strings.
5. Token color pairs used for text meet documented contrast expectations.
6. Reduced-motion preferences are respected by shared transition styles.
7. Theme/background state is presentation-only and cannot mutate domain or cache data.
8. No universe renderer, mood-color product model, customization catalog, or final product screen is introduced.

## Policy / Values Impact

- **policy / ux:** no current policy files are changed while this is only planned. When implemented, accessibility,
  primitive, and theme/background rules should land in UX/design-system policy docs if available.
- **spec/values.yaml:** no entries. Design tokens, CSS variables, theme CSS, and mood/color content tables are not
  generated numeric product tuning values.
- **Dependencies:** depends on [08.i18n-seam](08.i18n-seam.md) for copy boundaries and supports
  [12.test-harness](12.test-harness.md) and the mobile app shell (unit 13).
