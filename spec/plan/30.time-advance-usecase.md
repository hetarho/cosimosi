# 30. time advance usecase

> Time advance use-case — the backend orchestration that *moves* the universe clock. It advances the clock on the two
> triggers ([T2]): **star-launch → the diary date** (no warning) and **recall → today** (the sync capability recall
> composes). It refines Epic A's launch guard onto the real clock ([29.universe-clock](29.universe-clock.md)), surfaces
> the advance **interval** so the acceleration animation has something to play, and defines the **read-time progression
> hook** ([T4]) through which every advance feeds forgetting [F] / consolidation [C] — all without a cron ([T4]).
> Scope: BE | Status: planning
>
> Layer U (use-case) — the second unit of Epic B. Home: `internal/memory` use-cases (`AdvanceOnLaunch`, `SyncToToday`)
> + the consumer-owned `UniverseClockStore` port. It consumes [29.universe-clock](29.universe-clock.md)'s pure model +
> store, refines [20.encode-usecase](20.encode-usecase.md)'s launch, and is driven visually by
> [31.time-acceleration-ui](31.time-acceleration-ui.md). The recall side that calls `SyncToToday` is Epic C.

## Purpose

The universe clock ([29.universe-clock](29.universe-clock.md)) is inert on its own — a value and a `max`. This unit is
what actually **turns** it, and it fixes the product's most load-bearing rule: **time advances on exactly two triggers,
and nothing else** ([T2][T3]). Launch a star and the clock accelerates to that diary's date; recall a star and it syncs
to today. Just looking never moves it ([T3]) — which is what lets a memory sit untouched for real-world weeks and stay
exactly as bright as you left it.

Epic A shipped launch with a **stand-in guard** — it compared the diary date against `max(created_universe_time)` and,
for a past-dated diary, saved the diary but launched no star. That was explicitly Epic A's seam ([20.encode-usecase](20.encode-usecase.md)):
"Universe-clock advance itself is Epic B." This unit realizes it: it swaps the derived max for the authoritative stored
clock, advances that clock inside the launch transaction, and provides `SyncToToday` — the advance-to-today capability
that Epic C's recall will compose behind the sync-confirm gate ([R1a]).

It also lays the one seam Epics C–E depend on: the **read-time progression hook** ([T4]). Because forgetting and
consolidation are computed read-time (no batch job), advancing the clock does not *itself* forget anything — the next
read simply sees a later "now." But the writes an advance *implies* (semanticize a memory whose gist timer filled,
downscale weak synapses during the sleep) must be triggered **by the advance, not a scheduler**. This unit defines the
hook that carries the advance interval `[from, to]` to those handlers, and ships it wired with a no-op default so Epics
C/E slot their work in without re-plumbing when the clock moves.

## Scope / Non-goals

**In scope**
- **`AdvanceOnLaunch`** — inside the existing `LaunchStars` transaction ([20.encode-usecase](20.encode-usecase.md)):
  read the clock via the `UniverseClockStore` port, apply [29]'s `CanLaunchAt` constraint, and:
  - **launchable** (`diary_date ≥ clock`): proceed to raise the star(s) and `AdvanceClock(clock, diary_date)` — advance
    the clock to the diary date ([T2] case 1). **No warning** (moving forward loses nothing).
  - **past-dated** (`diary_date < clock`): save the `Diary`, launch **no** star, leave the clock unmoved ([T1][I10]).
  This **replaces** Epic A's `LatestLaunchedUniverseTime`-derived guard with the stored clock as the source of truth.
- **`SyncToToday`** — the advance-to-today capability ([T2] case 2): `AdvanceClock(clock, today)` where `today` is the
  UTC-normalized real date (reusing launch's existing `utcDate(now())` convention). Returns the advance **interval**
  `{ previous, current }` so the caller can animate/replay it. This is the building block **recall** ([R1a], Epic C)
  composes behind the sync-confirm modal; this unit provides and tests the capability, not the recall flow.
- **The advance interval on `LaunchStars`** — extend `LaunchStarsResponse` (contract owned by
  [20.encode-usecase](20.encode-usecase.md), extended here — backward-compatible field additions) with
  `previous_universe_time` and `universe_time` (the clock **before** and **after** the launch advance), so
  [31.time-acceleration-ui](31.time-acceleration-ui.md) can play the acceleration over `[previous, current]`. For a
  past-dated launch both equal the unmoved clock (no acceleration).
- **`GetUniverse.universe_time` from the stored clock** — rewire the universe read ([23.universe-canvas](23.universe-canvas.md)'s
  contract, server impl in the memory context) to return `universe_state.current_universe_time` instead of the
  `max(created_universe_time)` derivation. **One-release fallback**: when the clock row is absent (a pre-Epic-B user who
  has not launched since), fall back to `max(created_universe_time)` so no universe visibly resets (per [29]'s lazy-birth
  decision). The `GetUniverse` **contract shape is unchanged** — only the value's source.
- **The read-time progression hook** ([T4]) — a consumer-owned `AdvanceProgression` port called with `(scope, from, to)`
  on **every** clock advance (launch or sync), inside the same transaction. Epic B ships it wired with a **no-op
  default**: forgetting [F] is read-time (nothing to do at advance); the *writes* an advance implies —
  semanticize-timer crossings ([C6a]/[C6]) and sleep consolidation/SHY downscale ([C1][C2][C4]) — are Epics C/E, which
  bind a real handler behind this seam (enqueuing `semanticize`/`consolidate` jobs, kinds already reserved in
  [16.memory-aggregate-schema](16.memory-aggregate-schema.md)). No cron ([T4]).
- **The `UniverseClockStore` port** (consumer-owned, declared here, §2.4): `Read(ctx, scope) (*DATE, error)` +
  `Advance(ctx, scope, to DATE) (DATE, error)` (the `GREATEST` upsert from [29]). Bound to [29]'s `memory/pg` concrete
  at the composition root.
- **Per-user scoping** on every clock read, advance, and the progression call (§4).

**Non-goals**
- **No clock model or table.** `AdvanceClock`, the `CanLaunchAt` constraint, the `universe_state` table, and its
  `memory/pg` store are [29.universe-clock](29.universe-clock.md). This unit *calls* them.
- **No recall / reconsolidation.** `Recall`, `Reinforce`, `Reconsolidate`, the sync-confirm **flow**, prediction-error,
  neighbor forgetting — all Epic C. This unit provides `SyncToToday` as the capability recall composes; it does not
  build recall or its RPC.
- **No forgetting / consolidation / semanticization math or writes.** [F]/[C] are Epics D/E. This unit defines the
  progression hook and ships a **no-op** handler; it computes and persists none of that work.
- **No UI.** The acceleration animation, the "우주의 시간" HUD, and the sync-confirm modal are
  [31.time-acceleration-ui](31.time-acceleration-ui.md); this unit only returns the interval + the current clock they
  render.
- **No new `SplitDiary`/`ReviseSplit` behavior** — the split preview is unchanged ([20]); only the launch's
  clock/guard step and the response's interval fields are touched.
- **No new tuning scalars** — advancing a clock is `max` + a trigger; there is no coefficient. (Forgetting/semanticize
  rates belong to Epics D/E.)

## Design

### Placement (domain-first Go)
`AdvanceOnLaunch` and `SyncToToday` are application use-cases in the core-domain `internal/memory` context, alongside
`PersistEncoded`/`Link`. They consume the pure clock model + store from [29] through the consumer-owned
`UniverseClockStore` port, declared here (§2.4). The advance-triggered downstream work is reached through a second
consumer-owned port, `AdvanceProgression`, also declared here; its concrete is a no-op in Epic B and a real
enqueue-handler in Epics C/E — bound at `cmd/api`. Dependency direction:
`memory/rpc → AdvanceOnLaunch/SyncToToday (use-case) → pure clock domain ([29])`, with the clock store, the progression
hook, and the `jobs` queue reached only through consumer-owned interfaces. Handlers stay thin (§2.7/§2.9#7).

### `AdvanceOnLaunch` — the launch trigger ([T2] case 1)
Folded into the existing `LaunchStars` → `PersistEncoded` transaction ([20]), replacing the Epic A stand-in:

1. **Read the clock** for the user via `UniverseClockStore.Read` (nil = no clock yet, first launch).
2. **Insert the `Diary`** (append-only, always — [I2][W5]), exactly as today.
3. **Apply `CanLaunchAt(diary_date, clock)`** ([29]):
   - **clock is nil** (first ever launch) → launchable; there is no prior bar.
   - **`diary_date ≥ clock`** → launchable.
   - **`diary_date < clock`** → **past-dated**: set `PastDated`, launch no star, do **not** advance, return. The diary
     is saved and readable; the response interval is `{ clock, clock }` (no acceleration).
4. **On launchable**: raise the star(s) / neurons / activations / synapses (unchanged [20] steps), then
   **`Advance(clock, diary_date)`** — the `GREATEST` upsert moves the clock to the diary date (or leaves it if equal).
5. **Fire the progression hook** `AdvanceProgression.OnAdvance(scope, from=clock, to=new_clock)` in the same
   transaction (no-op in Epic B). `from == to` ⇒ the hook is a no-op regardless.
6. **Return** the launch result plus `previous_universe_time = from` and `universe_time = new_clock`.

The whole sequence stays in one `InTx` (Diary + memories + neurons + synapses + clock advance + progression enqueue are
atomic — a launch lands wholly or not at all). The pre-existing **future-date rejection** (`diary_date > today+1` →
error, from Epic A) is unchanged and runs before the transaction: the clock can never be advanced past real time by a
launch.

### `SyncToToday` — the recall trigger ([T2] case 2), the capability recall composes
`SyncToToday(scope) → { previous, current }`: read the clock, compute `today = utcDate(now())`, `Advance(clock, today)`,
fire the progression hook over `[previous, today]`, and return the interval. Idempotent — a second call the same day is a
no-op (`GREATEST` keeps `today`).

This is a **capability, not a user action**: there is no "sync the clock" button. Recall ([R1a], Epic C) is the only
trigger, and the **sync-consent gate is the recall flow's** — the modal ("회상하려면 우주 시간을 오늘로 맞춰야 해요… 진행할까요?",
[T2] case 2) is shown by [31.time-acceleration-ui](31.time-acceleration-ui.md) and the user's "예" is what leads Epic C's
`Recall` to call `SyncToToday` (then do the recall) while "아니오" cancels without turning the clock. Epic B builds and
unit-tests `SyncToToday`; it does **not** build recall or a standalone sync RPC. (Because `SyncToToday` moves the clock
forward to today, any not-yet-written past date is thereafter past-dated — this is the exact consequence the consent
modal warns about, and it falls out of the monotonic clock, no extra logic.)

### The read-time progression hook ([T4]) — reserve the slot, fire on every advance
[T4] requires forgetting/consolidation to run **read-time at the moment time advances, with no cron**. Two kinds of
consequence follow from an advance interval `[from, to]`:

- **Read-time-only (no work at advance).** Effective brightness/decay ([F1]) and effective synapse strength ([L5]) are
  pure functions of "now"; once the clock is at `to`, the next `GetUniverse` read simply computes them against the later
  now. Nothing is enqueued — the advance is *automatically* reflected. (This is why Epic B's default handler is a no-op
  and the universe still "forgets" the moment Epic D's read-time decay ships.)
- **Write-producing (must be triggered by the advance).** Semanticize-timer crossings ([C6a] → a new gist star, [C6],
  stage++, [C7] stage texts) and sleep consolidation (replay reorg [C2], SHY downscale [C4]) are *writes* the interval
  implies. They must fire **because** the clock moved — that is the "no cron" contract ([T4]).

This unit defines the seam for the second kind: `AdvanceProgression.OnAdvance(ctx, scope, from, to)`, called inside the
advance transaction on launch **and** sync. In Epic B it is a **no-op** default (documented, tested as a no-op). Epics
C/E bind a real handler that, given `[from, to]`, enqueues the `semanticize`/`consolidate` jobs the interval crosses
(the worker + kinds already exist — [22.ai-worker-pipeline](22.ai-worker-pipeline.md) / [16](16.memory-aggregate-schema.md)).
Reserving the slot now means Epics C–E hang their advance-triggered work on one tested seam without re-plumbing when the
clock moves ([T4]) — the same "reserve the slot" discipline the PRD uses for consolidation choreography ([V8][C8]).

### `GetUniverse.universe_time` — from the stored clock
The universe read's `universe_time` now returns `universe_state.current_universe_time`. The **contract shape is
unchanged** ([23]) — only the source moves from the `max(created_universe_time)` derivation to the authoritative clock.
The **one-release fallback**: if the clock row is absent (a user who launched pre-Epic-B and has not launched since),
return `max(created_universe_time)` as before (per [29]'s lazy-birth decision), so no one's universe time visibly resets;
the first post-Epic-B advance writes the real clock. The empty-universe read still returns nil universe time. This read
stays GET-eligible / `NO_SIDE_EFFECTS` / user-scoped ([23], §2.7) — reading never advances the clock ([T3]).

### Thin handler / dependency rule
The `memory/rpc` `LaunchStars` handler maps the extended response fields (proto↔domain) and calls the use-case; all
clock policy (read → constraint → advance → hook) lives in the use-case/domain (§2.9#7). No proto/sqlc type reaches the
use-case or the pure clock domain (§2.9#4); `UniverseClockStore` and `AdvanceProgression` are consumer-owned interfaces
(§2.9#6).

## Acceptance Criteria
1. On **launch of a launchable diary** (`diary_date ≥ clock`, or first-ever launch), the star(s) rise **and** the clock
   advances to the diary date via `AdvanceClock`/the `GREATEST` upsert; there is **no** warning ([T2] case 1).
2. On **launch of a past-dated diary** (`diary_date < clock`), the `Diary` is saved, **no** star launches, and the clock
   is **not** advanced ([T1][I10]); this replaces Epic A's `max(created_universe_time)` guard with the stored clock.
3. The whole launch — Diary + memories + neurons + synapses + **clock advance** + progression hook — is **one atomic
   transaction**; a failure rolls all of it back together.
4. `SyncToToday` advances the clock to `utcDate(now())`, is **idempotent** the same day, and returns the interval
   `{ previous, current }`; it is a **capability** with no standalone RPC — recall (Epic C) composes it behind the
   sync-consent gate ([T2] case 2 / [R1a]).
5. `LaunchStarsResponse` carries `previous_universe_time` and `universe_time` (clock before/after the advance) so the UI
   can play the acceleration; for a past-dated launch both equal the unmoved clock (no acceleration).
6. `GetUniverse.universe_time` returns the stored `universe_state.current_universe_time` (contract shape unchanged),
   with a **one-release fallback** to `max(created_universe_time)` when the clock row is absent, and **nil** for an empty
   universe; the read never advances the clock ([T3], `NO_SIDE_EFFECTS`).
7. Every clock advance (launch **and** sync) fires `AdvanceProgression.OnAdvance(scope, from, to)` inside the advance
   transaction; Epic B ships it as a **no-op** default (forgetting is read-time; [C]/[E] writes bind here later), with
   **no cron** anywhere ([T4]).
8. The pre-existing **future-date rejection** (`diary_date > today + 1 day`) still runs before the transaction, so a
   launch can never advance the clock past real time ([I10]).
9. Handlers are **thin** (proto↔domain map + call); the read→constraint→advance→hook policy lives in the use-case/domain;
   no proto/sqlc type leaks into the use-case or clock domain; `UniverseClockStore` and `AdvanceProgression` are
   consumer-owned interfaces (§2.4/§2.7/§2.9#4/#6/#7).
10. Every clock read, advance, and progression call is **scoped to the authenticated user** and `pnpm lint:persistence`
    passes (§4).
