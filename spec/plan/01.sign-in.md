# 01. 사인인 — Supabase Auth (프론트)

> Supabase Auth로 사인인/아웃하고 세션이 앱을 게이트하며, JWT를 이후 API 호출에 실어보낼 준비. | 범위: MVP(FE) | 선행: — | 차단: (API 인증 필요 지점 soft)

## 목적 (Why)
MVP는 단일 사용자라도 **실제 계정으로 데이터를 스코프**하고 v1+ 소셜(우주 공유·공명)을 대비해야 한다. 미인증 상태에선 우주 셸을 보이지 않게 게이트하고, 인증 세션에서 **JWT를 꺼내 02의 Connect transport가 Authorization 헤더로 실어보낼 수 있는 지점**을 마련한다. (서버 검증·다중 사용자 격리는 다른 스펙.)

## 범위
- **포함:** `@supabase/supabase-js` 클라이언트(`shared/api`), 사인인/아웃 UI(소셜 Google **또는** 이메일 매직링크 중 택1로 시작), 세션 게이트(미인증=사인인 화면, 인증=우주 셸 `HomePage`; **`/universe` 보호 라우트(또는 보호 레이아웃)에 적용** — `/` 랜딩은 게이트 없이 공개 유지), auth store(zustand, `app`), `onAuthStateChange` 구독, 매직링크 콜백(`emailRedirectTo=${origin}/universe`) 세션 복원, **세션 토큰 getter** 노출(02 transport가 사용), `.env.example`에 `VITE_SUPABASE_*` 추가.
- **비목표(Out of scope):**
  - 백엔드 JWT 검증·인증 인터셉터 → **02 (rpc-contract)** 담당.
  - 다중 사용자 데이터 격리(`user_id` 스코프 쿼리) → **04 (memory-api)**.
  - 프로필 편집·회원관리 고도화·역할/권한 → MVP 비목표(v1+).
  - Supabase 프로젝트 실제 프로비저닝(키 발급) → 인프라 작업(Architecture §7). 본 스펙은 env 슬롯과 코드만.

## 설계 요점
- **Supabase 클라이언트(공유 레이어, three/React/DOM 미의존):**
  ```ts
  // frontend/src/shared/api/supabase.ts
  import { createClient } from '@supabase/supabase-js'
  const url = import.meta.env.VITE_SUPABASE_URL
  const anon = import.meta.env.VITE_SUPABASE_ANON_KEY
  export const supabase = createClient(url, anon, {
    auth: { persistSession: true, autoRefreshToken: true, detectSessionInUrl: true },
  })
  // 02의 transport가 호출하는 동기 토큰 getter (인터셉터에서 await 가능)
  export async function getAccessToken(): Promise<string | null> {
    const { data } = await supabase.auth.getSession()
    return data.session?.access_token ?? null
  }
  ```
- **auth store(zustand, 진짜 글로벌 → `app`):** `session: Session | null`, `status: 'loading' | 'authed' | 'anon'`, `signInWithGoogle()` **또는** `signInWithOtp(email)`, `signOut()`. 부팅 시 `supabase.auth.getSession()` 1회 + `supabase.auth.onAuthStateChange((_e, s) => set({ session: s, status: ... }))` 구독해 store에 반영. **store는 `three`/React/DOM을 import하지 않는다**(상태·로직만; 구독 시작은 `useEffect`에서 호출).
- **세션 게이트(`/universe` 보호 라우트):** 게이트는 App.tsx/HomePage가 아니라 **`/universe` 라우트(또는 보호 레이아웃)**에 건다. `status==='loading'`→스플래시, `'anon'`→`<SignInScreen/>`, `'authed'`→우주 셸(`<HomePage/>`). 라우팅/진입 책임이므로 라우트 정의는 `app`(`router.tsx`)이 소유한다. `/` 랜딩(`LandingPage`)은 게이트 없이 공개로 유지한다.
- **택1 시작 방식 + 콜백:** 외부 OAuth 콜백 도메인 설정이 필요 없는 **이메일 매직링크(`signInWithOtp`)**를 MVP 기본으로 시작(소셜 Google은 같은 store 메서드로 슬롯인 가능하게 시그니처만 남김). `signInWithOtp(email, { emailRedirectTo: \`${window.location.origin}/universe\` })`로 발송하면 매직링크가 `/universe`로 복귀하고, `detectSessionInUrl: true`로 그 라우트에서 URL의 토큰을 자동 파싱해 세션을 수립한다. 복원 후 `/universe` 보호 라우트의 게이트가 `authed`로 전환되어 우주 셸을 렌더한다(별도 콜백 페이지 불필요).
- **토큰 주입 지점(soft 차단):** 02가 `getAccessToken`을 import해 Connect 인터셉터에서 `Authorization: Bearer <token>`을 붙인다. 본 스펙은 **getter가 동작하는 것까지만** 보장한다.

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 미인증 사용자가 `/universe` 보호 라우트에 진입 THE SYSTEM SHALL 우주 셸(`HomePage`) 대신 사인인 화면(`SignInScreen`)을 렌더한다. (`/` 랜딩은 공개이며 게이트하지 않는다.)
1.2 WHEN 사용자가 이메일 매직링크로 사인인에 성공(`emailRedirectTo=${origin}/universe`로 복귀) THE SYSTEM SHALL 세션을 store에 저장하고 `/universe`에서 우주 셸(`HomePage`)로 전환한다.
1.2b WHEN 매직링크가 토큰을 URL에 실어 `/universe`로 복귀 THE SYSTEM SHALL `detectSessionInUrl`로 세션을 자동 복원하고 해시/쿼리 토큰을 정리한 뒤 우주 셸을 렌더한다(별도 콜백 페이지 없이).
1.3 GIVEN 유효 세션 WHEN `getAccessToken()`을 호출 THEN THE SYSTEM SHALL 비어있지 않은 JWT 문자열을 반환한다(02 transport가 헤더에 첨부 가능).
1.4 WHEN 사용자가 사인아웃 THE SYSTEM SHALL 세션을 삭제(`supabase.auth.signOut()`)하고 사인인 화면으로 복귀한다.
1.5 GIVEN 인증된 세션 WHEN 페이지를 새로고침 THEN THE SYSTEM SHALL 저장된 세션을 복원해 우주 셸을 유지한다(`persistSession`).
1.6 IF `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY`가 없음 THEN THE SYSTEM SHALL 부팅 시 명확한 콘솔 오류로 실패하고 무한 로딩에 빠지지 않는다.
1.7 WHILE 초기 세션 조회 중 THE SYSTEM SHALL `status='loading'` 스플래시를 보이고 사인인 화면을 깜빡이지 않는다.
