# 01. 사인인 — Supabase Auth (프론트)

> Supabase Auth로 사인인/아웃하고 세션이 앱을 게이트하며, JWT를 이후 API 호출에 실어보낼 준비. | 범위: MVP(FE) | 선행: — | 차단: (API 인증 필요 지점 soft)

## 목적 (Why)
MVP는 단일 사용자라도 **실제 계정으로 데이터를 스코프**하고 v1+ 소셜(우주 공유·공명)을 대비해야 한다. 미인증 상태에선 우주 셸을 보이지 않게 게이트하고, 인증 세션에서 **JWT를 꺼내 02의 Connect transport가 Authorization 헤더로 실어보낼 수 있는 지점**을 마련한다. (서버 검증·다중 사용자 격리는 다른 스펙.)

## 범위
- **포함:** `@supabase/supabase-js` 클라이언트(`shared/api`), 사인인/아웃 UI(**두 방식 지원: ① 이메일 OTP 코드 ② Google OAuth** — 이메일은 OTP 전용, 매직링크 미사용), 세션 게이트(미인증=사인인 화면, 인증=우주 셸 `HomePage`; **`/universe` 보호 라우트(또는 보호 레이아웃)에 적용** — `/` 랜딩은 게이트 없이 공개 유지), auth store(zustand, `app`), `onAuthStateChange` 구독, **OTP 코드 검증(`verifyOtp`) / OAuth redirect 복귀로 세션 수립**, **세션 토큰 getter** 노출(02 transport가 사용), `.env.example`에 `VITE_SUPABASE_*` 추가.
- **비목표(Out of scope):**
  - 백엔드 JWT 검증·인증 인터셉터 → **02 (rpc-contract)** 담당.
  - 다중 사용자 데이터 격리(`user_id` 스코프 쿼리) → **04 (memory-api)**.
  - 프로필 편집·회원관리 고도화·역할/권한 → MVP 비목표(v1+).
  - Supabase 프로젝트 실제 프로비저닝(키 발급·Google provider 설정·custom SMTP) → 인프라/대시보드 작업(Architecture §7). 본 스펙은 env 슬롯과 코드만. (대시보드 전제는 DoD에 명시.)

## 설계 요점
- **Supabase 클라이언트(공유 레이어, three/React/DOM 미의존):**
  ```ts
  // frontend/src/shared/api/supabase.ts
  import { createClient } from '@supabase/supabase-js'
  const url = import.meta.env.VITE_SUPABASE_URL
  const publishableKey = import.meta.env.VITE_SUPABASE_PUBLISHABLE_KEY
  export const supabase = createClient(url, publishableKey, {
    auth: { persistSession: true, autoRefreshToken: true, detectSessionInUrl: true },
  })
  // 02의 transport가 호출하는 동기 토큰 getter (인터셉터에서 await 가능)
  export async function getAccessToken(): Promise<string | null> {
    const { data } = await supabase.auth.getSession()
    return data.session?.access_token ?? null
  }
  ```
  > `detectSessionInUrl: true`는 **Google OAuth redirect 복귀** 시 URL의 세션 토큰을 자동 파싱하는 데 쓰인다(OTP 코드 흐름엔 불필요하나 무해).
- **auth store(zustand, 진짜 글로벌 → `app`):** `session: Session | null`, `status: 'loading' | 'authed' | 'anon'`, `signInWithOtp(email)`[코드 발송], `verifyOtp(email, token)`[코드 검증→세션], `signInWithGoogle()`[OAuth redirect], `signOut()`. 부팅 시 `supabase.auth.getSession()` 1회 + `supabase.auth.onAuthStateChange((_e, s) => set({ session: s, status: ... }))` 구독해 store에 반영. **store는 `three`/React/DOM을 import하지 않는다**(상태·로직만; 구독 시작은 `useEffect`에서 호출).
- **세션 게이트(`/universe` 보호 라우트):** 게이트는 App.tsx/HomePage가 아니라 **`/universe` 라우트(또는 보호 레이아웃)**에 건다. `status==='loading'`→스플래시, `'anon'`→`<SignInScreen/>`, `'authed'`→우주 셸(`<HomePage/>`). 라우팅/진입 책임이므로 라우트 정의는 `app`(`router.tsx`)이 소유한다. `/` 랜딩(`LandingPage`)은 게이트 없이 공개로 유지한다.
- **방식 ① 이메일 OTP 코드(매직링크 아님):** 이메일은 **OTP 전용**이다. `signInWithOtp({ email })`로 OTP 코드를 발송하고(매직링크용 `emailRedirectTo` 불필요), 같은 탭에서 입력한 코드를 `verifyOtp({ email, token, type: 'email' })`가 **그 자리에서 세션으로 교환**한다 — 메일을 다른 브라우저/기기에서 열어야 하는 매직링크 마찰이 없다. ⚠️ **이메일 템플릿**: Supabase 기본 "Magic link or OTP" 템플릿은 `{{ .ConfirmationURL }}`(링크)만 노출하므로, 코드가 메일에 나오려면 대시보드 Authentication→Email Templates에 `{{ .Token }}`(OTP 코드, 자릿수는 Supabase 설정값)를 포함해야 한다. ⚠️ **발송 한도**: 빌트인 이메일은 시간당 2통(초과 시 429) — 활발한 개발·프로덕션은 **custom SMTP**(Resend 등)로 한도를 올린다.
- **방식 ② Google OAuth(1급):** `signInWithOAuth({ provider: 'google', options: { redirectTo: `${window.location.origin}/universe` } })`를 호출하면 supabase-js가 Google 동의 화면으로 redirect하고, 인증 후 `/universe`로 복귀할 때 `detectSessionInUrl: true`가 URL의 세션을 파싱해 수립한다(→ `onAuthStateChange` → `authed` → 우주 셸, 별도 콜백 페이지 없음). ⚠️ **대시보드 전제**: Google Cloud OAuth 클라이언트(Web) 생성 → **Authorized redirect URI = `https://<project-ref>.supabase.co/auth/v1/callback`**, **Authorized JS origin = `http://localhost:1214`** → Client ID/Secret를 Supabase Authentication→Providers→Google에 입력. 앱 `redirectTo`(`${origin}/universe`)는 Supabase Redirect URLs 허용목록에 있어야 한다.
- **토큰 주입 지점(soft 차단):** 02가 `getAccessToken`을 import해 Connect 인터셉터에서 `Authorization: Bearer <token>`을 붙인다. 본 스펙은 **getter가 동작하는 것까지만** 보장한다.

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 미인증 사용자가 `/universe` 보호 라우트에 진입 THE SYSTEM SHALL 우주 셸(`HomePage`) 대신 사인인 화면(`SignInScreen`)을 렌더하며, **이메일 OTP**와 **Google** 두 진입을 모두 제공한다. (`/` 랜딩은 공개이며 게이트하지 않는다.)
1.2 WHEN 사용자가 이메일로 받은 OTP 코드를 같은 탭에 입력해 검증(`verifyOtp`)에 성공 THE SYSTEM SHALL 세션을 store에 저장하고 `/universe`에서 우주 셸(`HomePage`)로 전환한다(리다이렉트·콜백 페이지 없이 같은 탭에서).
1.2b WHEN 사용자가 이메일을 제출 THE SYSTEM SHALL `signInWithOtp({ email })`로 OTP 코드를 발송하고 코드 입력 단계로 전환한다(매직링크 미사용).
1.8 WHEN 사용자가 "Google로 계속하기"를 클릭 THE SYSTEM SHALL `signInWithOAuth({ provider:'google' })`로 Google 동의 화면으로 이동하고, 인증 후 `/universe` 복귀 시 `detectSessionInUrl`로 세션을 수립해 우주 셸을 렌더한다(대시보드에 Google provider 설정 전제).
1.3 GIVEN 유효 세션 WHEN `getAccessToken()`을 호출 THEN THE SYSTEM SHALL 비어있지 않은 JWT 문자열을 반환한다(02 transport가 헤더에 첨부 가능).
1.4 WHEN 사용자가 사인아웃 THE SYSTEM SHALL 세션을 삭제(`supabase.auth.signOut()`)하고 사인인 화면으로 복귀한다.
1.5 GIVEN 인증된 세션 WHEN 페이지를 새로고침 THEN THE SYSTEM SHALL 저장된 세션을 복원해 우주 셸을 유지한다(`persistSession`).
1.6 IF `VITE_SUPABASE_URL`/`VITE_SUPABASE_PUBLISHABLE_KEY`가 없음 THEN THE SYSTEM SHALL 부팅 시 명확한 콘솔 오류로 실패하고 무한 로딩에 빠지지 않는다.
1.7 WHILE 초기 세션 조회 중 THE SYSTEM SHALL `status='loading'` 스플래시를 보이고 사인인 화면을 깜빡이지 않는다.
