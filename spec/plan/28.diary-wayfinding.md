# 28. 원본 일기·엔그램·별 길찾기 (diary-wayfinding)

> 원본 일기 ↔ 엔그램 조각 ↔ 현재 별의 3겹 연결을 길찾기로 잇는다 — 원본 일기로 그 일기의 **모든 별을 조망 위치에서 하이라이팅**(far), 엔그램(조각)으로 **단일 별을 근접 탐색**(near). | 범위: FS | 선행: 21 | 차단: —

## 목적 (Why)

21이 일기 한 편을 N개의 엔그램 별로 쪼갠 뒤, 사용자는 **"그날 내가 쓴 일기가 지금 우주의 어디에, 몇 개의 별로 흩어져 있나"** 를 찾고, 거꾸로 별에서 그 조각·원본 일기로 돌아올 수 있어야 한다(concept §일기 한 편, 별 여럿). 같은 일기의 조각은 태어날 땐 근처에 모이지만, 시간이 지나며 회상·재공고화(23)·재배치(22·27)로 **어디로 흩어졌을지 알 수 없다.** 그래서 원본 일기로 찾으면 그 일기의 **모든 별을 한눈에 담는 조망 위치를 매번 새로 계산**해 보여주고 별들을 강조한다. 가까이(회상 근접)에서는 한 기억의 한 조각 — 즉 **단일 별(엔그램)** 만 집어 든다. 원본 일기는 불변·영구이며(헌법1), 본 스펙은 그것을 **읽고 길찾기에 쓰기만** 한다.

## 범위

- **포함:**
  - BE: `Star`에 `record_id`(+ `fragment_index`) 노출 → 클라가 별을 **일기 단위로 그룹**. 원본 일기 목록 조회 `ListRecords`(원본 일기 id·작성일·본문 일부·별 개수). 별이 든 조각 텍스트 표시를 위해 `fragment_text`를 `RecallMemory` 응답(또는 `Star`)에 노출(21이 `memories.fragment_text` 컬럼을 만듦).
  - FE 원본 일기 보기/검색: 일기 목록·텍스트 검색을 **우주 셸 위 오버레이(바텀시트/패널 — 31)** 로 띄운다(별도 페이지로 우주를 떠나지 않음) → **"이 일기의 별 보기"** → 시트가 peek로 잦아들며 그 일기의 별들을 프레이밍하는 조망 위치로 fly-to + 하이라이팅.
  - FE **frame-all 카메라**(12 fly-to 확장): 선택 일기 별 좌표 집합의 중심(centroid)·반경(bounding radius)을 구해 **모두 화면에 들어오는 far 거리**로 카메라를 보간 이동(lookAt = centroid). 좌표는 클라 force-sim/정적 fibonacci에서 읽기만(헌법3).
  - FE 하이라이팅: 선택 일기의 별을 강조(밝기 부스트/링/펄스), 그 사이 `intra_entry` 시냅스 강조, 나머지 별·먼지는 dim(잠든 별 dust dimming 패턴 재사용). 강조 상태는 별·record를 수정하지 않는다.
  - FE 엔그램 단위 찾기: 조각(엔그램) 목록/검색 → 단일 별 **근접** fly-to(12 `focusStar` 재사용).
  - FE near/far 가드: 회상 근접(`recall` 모드)에서는 **단일 엔그램만** 포커스 가능; 일기 전체 조망은 far(`nebula`류)로 전환을 강제한다.
  - FE 별 → 조각 → 원본 연결: 별 클릭 패널(11 `MemoryPanel` 확장)에 그 **조각 텍스트(`fragment_text`)** + **"원본 일기 전체 보기"**(불변 `Record` body) 동선.
- **비목표(Out of scope):**
  - 조각 분할·조각별 감정·`intra_entry` 생성·`record_id`/`fragment_text` 컬럼 신설 → **21**(본 스펙은 21이 만든 데이터를 소비·노출만).
  - 라이브 force-sim 좌표 구동 → **22**(미배선이면 정적 fibonacci 좌표로 frame-all 동작; 22 후 라이브 좌표로 자동 정합 — 좌표 출처 단일).
  - 재공고화 재성형(별이 흩어지는 *원인*) → **23**. 본 스펙은 흩어진 별을 *찾기*만 한다.
  - 변천사(별의 변화 이력) 열람 → **24**.
  - AI 추출기 포트 → **20**.

## 설계 요점

**proto 계약(누적 — 28이 추가, `cosimosi.v1`).** 별을 일기 단위로 그룹·조망하려면 `Star`가 자기 `record_id`를 알아야 한다. 원본 일기 목록·조각 텍스트도 노출한다.

```proto
// Star: 별이 어느 원본 일기의 몇 번째 조각인지 + 그 조각 텍스트를 운반(그룹·하이라이팅·패널용)
message Star {
  string memory_id      = 1;
  Mood   mood           = 2;
  double intensity      = 3;
  string last_recalled_at = 4;
  string record_id      = 9;  // 28 신설 — 일기 단위 그룹/조망 키 (5–8은 23 재성형 필드 예약)
  int32  fragment_index = 10; // 28 신설 — 일기 내 조각 순서(21이 채움)
}
// 원본 일기 목록(원본 일기로 별 찾기 진입). body는 일부(excerpt)만.
message RecordSummary { string record_id = 1; string entry_date = 2; string body_excerpt = 3; int32 star_count = 4; }
message ListRecordsRequest {}
message ListRecordsResponse { repeated RecordSummary records = 1; }
// 조각 텍스트(별 → 조각): RecallMemory 응답에 더해 노출(원본 Record는 11이 이미 반환)
message RecallMemoryResponse { Record record = 1; string fragment_text = 2; } // fragment_text = 28 신설(NULL이면 빈 문자열)
```

> `Star.record_id`로 클라가 `GetUniverse`(04) 응답을 **일기 단위로 그룹**하므로 별-그룹핑엔 신규 쿼리가 필요 없다. 원본 일기 *목록*(찾기 진입 UI)만 `ListRecords`로 받는다. `Record`(불변 원본 전체)는 11 `RecallMemory`가 이미 반환하고, 28은 거기에 그 별의 **조각 텍스트**를 더한다.

**BE 쿼리(`db/queries/memory.sql`, sqlc).** 모두 `user_id` 격리. `records`는 읽기만(헌법1).

```sql
-- name: ListRecords :many  — 원본 일기 목록 + 일기별 별 개수(조각 수)
SELECT r.id AS record_id, r.entry_date, left(r.body, 80) AS body_excerpt, count(m.id)::int AS star_count
FROM records r JOIN memories m ON m.record_id = r.id
WHERE r.user_id = $1
GROUP BY r.id, r.entry_date, r.body
ORDER BY r.entry_date DESC;
-- GetUniverse(04)의 별 SELECT에 m.record_id, m.fragment_index 추가(그룹/조망 키).
-- RecallMemory(11)의 응답에 m.fragment_text 추가(별 → 조각 텍스트; records.body는 원본 전체로 유지).
```

**FE frame-all 카메라(`widgets/universe-canvas`, 12 fly-to 확장).** 12는 단일 별 `focusStar(id)`를 lerp/damp(`k=1−exp(−dt·3)`)로 따라간다. 28은 **별 집합**을 담는 조망을 더한다. 좌표는 클라가 산출한 것을 **읽기만** 한다(헌법3, navigation 정책).

```ts
// 선택 일기의 별 id 집합 S(=record_id로 그룹) → 모두 담는 카메라 위치 계산(순수)
function frameTarget(positions: Float32Array, ids: number[], fovRad: number) {
  // centroid c, 최대 반경 R = max|p_i − c|; distance d = (R / sin(fovRad/2)) · MARGIN(≈1.3)
  // 카메라 = c + viewDir·d, lookAt = c. 단일 별이면 R≈0 → 12 단일 focus 거동으로 수렴(1.8).
}
function frameStars(ids: string[]): void;   // far 모드로 전환 후 frameTarget으로 fly-to(+하이라이트 set)
```

**하이라이팅(`entities/memory`/`entities/synapse` 셀렉터 + 렌더).** `highlightedRecordId` 상태 → 그 record의 `memory_id` 집합. `StarField`(08)는 강조 별의 `aBrightness`/링을 부스트하고 비강조는 dim(잠든 별 dust dimming 재사용), `intra_entry` 엣지(21)는 강조. 강조는 **시각 상태일 뿐** record/별을 수정하지 않는다(헌법1·2). 갱신은 선택 변경 시에만(매 프레임 React state 금지, §3.2).

**near/far 가드(navigation, 12 카메라 모드 위).** 일기 전체 조망(`frameStars`)은 `nebula`(far)에서만 — `recall`(근접)에서 요청하면 먼저 far로 전환 후 프레이밍한다. 단일 `focusStar`(엔그램)는 어느 모드에서나 허용. *"근접에서는 무조건 엔그램 단위(단일 별)만."*

**별 → 조각 → 원본 패널(`features/recall`, 11 확장).** 별 클릭 시 패널에 (a) 그 **조각 텍스트**(`fragment_text`), (b) **"원본 일기 전체 보기"** → 불변 `Record` body(11 `RecallMemory`), (c) "이 일기의 다른 별들 보기" → `frameStars(record_id 그룹)`. 편집/삭제 컨트롤 없음(헌법1).

## 수용 기준 (EARS / Given-When-Then)

1.1 WHEN 사용자가 원본 일기를 골라 "별 보기"를 하면 THE SYSTEM SHALL 그 일기(`record_id`)에서 태어난 **모든 별**을 한 화면에 담는 조망 위치로 카메라를 fly-to하고 그 별들을 하이라이팅한다(나머지는 dim).
1.2 GIVEN 같은 일기의 별들이 회상·재공고화·재배치로 우주 곳곳에 흩어져 있을 때 WHEN "별 보기"를 하면 THE SYSTEM SHALL 그 순간의 별 좌표로 조망 위치(centroid·반경)를 **매번 새로 계산**해 모두 보이게 한다.
1.3 WHEN 사용자가 엔그램(조각) 하나를 고르면 THE SYSTEM SHALL 그 **단일 별**로 근접 fly-to한다(12 `focusStar` 재사용).
1.4 WHEN 카메라가 회상 근접(`recall`) 상태일 때 THE SYSTEM SHALL 단일 엔그램(별)만 포커스하게 하고, 일기 전체 조망 요청 시 먼저 far(`nebula`)로 전환한 뒤 프레이밍한다. _(근접=엔그램 단위만)_
1.5 WHEN 사용자가 별을 클릭하면 THE SYSTEM SHALL 그 별의 **조각 텍스트(`fragment_text`)** 와 **불변 원본 일기 전체(`Record`)** 로 가는 동선, 그리고 "이 일기의 다른 별들 보기"를 제공한다(원본·조각·별 3겹 연결). _(헌법1)_
1.6 THE SYSTEM SHALL 찾기·하이라이팅이 `records`/`memories`/`memory_links`를 수정·삭제하지 않는다(읽기·시각 전용; `rg "UPDATE records|DELETE FROM"` 무결과). _(헌법1·2)_
1.7 THE SYSTEM SHALL 좌표를 서버에서 만들지 않으며 frame-all은 클라 좌표를 **읽기만** 한다(좌표 출처는 force-sim/정적 fibonacci 단일 — 22 전후 동일). _(헌법3)_
1.8 WHEN 한 일기에 별이 1개뿐이면 THE SYSTEM SHALL frame-all이 그 단일 별 근접과 동일하게 수렴한다(degenerate, 반경≈0).
1.9 WHEN `ListRecords`를 호출하면 THE SYSTEM SHALL 호출 user의 원본 일기를 작성일 내림차순으로 `RecordSummary`(별 개수 포함)로 반환하고, body는 일부(excerpt)만 보낸다.
1.10 WHEN `features/wayfinding/model`이 빌드되면 THE SYSTEM SHALL `three`/React/DOM을 import하지 않는다(순수·모바일 재사용). _(헌법4)_

## 완료 정의 (DoD)

- Docker/WSL: `buf generate`(proto) + `sqlc generate` + `go build ./...` 무오류. `Star`에 `record_id`/`fragment_index`, `ListRecords` rpc, `RecallMemoryResponse.fragment_text` 생성 확인.
- 다조각 일기(21)를 만든 뒤 **원본 일기 목록에서 한 일기를 골라 "별 보기"** 하면, 그 일기의 모든 별이 한 화면에 담기는 위치로 카메라가 이동하고 그 별들이 강조된다(나머지 dim). 별 좌표를 흩어 놓아도(수동 이동/회상) 매번 다 보이는 위치로 프레이밍된다.
- **엔그램(조각)** 을 고르면 단일 별로 근접 이동한다. 회상 근접 상태에서 일기 전체 조망을 요청하면 먼저 far로 물러난 뒤 프레이밍한다(근접=단일만).
- 별 클릭 패널에서 그 **조각 텍스트**와 **불변 원본 일기 전체**, "이 일기의 다른 별들 보기"가 연결된다(원본·조각·별 3겹). 편집/삭제 컨트롤 없음.
- 찾기·하이라이팅 어디서도 `records`/`memories`/`memory_links`를 수정·삭제하지 않는다(`rg "UPDATE records|DELETE FROM" backend/internal` 무결과 — 헌법1·2). 좌표는 클라가 읽기만(헌법3).
- 한 일기에 별이 1개뿐이면 frame-all이 단일 근접과 동일하게 동작(degenerate)한다.
- `pnpm --filter frontend build`·`pnpm lint` 통과. `features/wayfinding/model`이 `three`/React/DOM 미import(헌법4, grep 확인).

## 과학적 근거

원본·조각·별의 연결과 "같은 일기 별은 근처에 모인다"는 **사건 분할(event segmentation)** + **사건 내 결속(within-event binding)** 에 근거하고, "별이 흩어질 수 있어 조망이 필요하다"는 **재공고화에 따른 표상 변화**에 근거한다(상세 인용은 20·21·23).

- 사건 분할(연속 경험을 이산 사건으로 부호화): Zacks JM et al., *Psychological Bulletin* (2007). [DOI](https://doi.org/10.1037/0033-2909.133.2.273)
- 사건 경계의 이산적 기억 흔적: Baldassano C et al., *Neuron* (2017). [DOI](https://doi.org/10.1016/j.neuron.2017.06.041)
- 사건 내 결속(같은 사건 항목이 더 강하게 묶임 → 근처 분포): DuBrow S, Davachi L, *Journal of Experimental Psychology: General* (2013). [DOI](https://doi.org/10.1037/a0034024)
- 재공고화에 따른 표상 변화(별이 흩어지는 근거): Lee JLC, Nader K, Schiller D, *Trends in Cognitive Sciences* (2017). [DOI](https://doi.org/10.1016/j.tics.2017.04.006)
