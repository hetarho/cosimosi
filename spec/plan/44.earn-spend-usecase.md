# 44. earn spend usecase

> Earn / Spend use-cases ‚Äî the backend orchestration of the Twinkle economy: it BINDS the real spend gate that recall
> and gist-view call (balance-check + deduct, spend order basic‚Üíadditional), and it grows every earn path (write, invite
> both-sides, payment) plus `GetBalance`. It turns plans 33/34's allow-all seam into a priced core loop that still lets
> everyday rumination through ([G5]).
> Scope: BE | Status: built ([job 55](../jobs/archive/55.earn-spend-usecase.md))
>
> Layer U (use-cases) ‚Äî the second unit of Epic F. Home: the NEW `internal/twinkle` context ([CC8]) ‚Äî use-cases `Earn`,
> `Spend`, `GetBalance` + the `twinkle.v1.TwinkleService` RPCs. It consumes [43.stardust-ledger](43.stardust-ledger.md)'s
> pure ledger model (balance derivation, spend order, cost curves) and **binds** the consumer-owned `SpendGate` port that
> [33.recall-usecase](33.recall-usecase.md) / [34.view-semantic-usecase](34.view-semantic-usecase.md) declared ‚Äî replacing
> their Epic-C no-op default. Memory's launch ([20.encode-usecase](20.encode-usecase.md)) reaches the write-earn path
> through a consumer-owned port; twinkle never imports memory internals. The balance HUD / charge UI is
> [45.stardust-ui](45.stardust-ui.md). Store payment is an external actor verified through a production receipt adapter
> ([G3]).

## Purpose

Recall costs Twinkle ([G1]) ‚Äî but until Epic F there is no Twinkle to spend. Epics C‚ÄìE deliberately shipped recall and
gist-view against a **no-op allow-all gate** ([CC2]) so the memory loop could be built and tested before the economy
existed. This unit is where the economy actually turns on: it binds the **real** spend gate ([43.stardust-ledger](43.stardust-ledger.md)'s
balance-check + deduct, in the basic‚Üíadditional order [G2]) so that from here on a recall or a gist-view first checks the
user's balance and deducts the cost the ledger's curves price ‚Äî and it grows the ways a user _acquires_ Twinkle so the
loop stays affordable.

The load-bearing product rule this unit protects is **[G5]**: the cost gate must never block everyday rumination. The
daily-reset **basic** balance ([G2], owned by [43.stardust-ledger](43.stardust-ledger.md)) is sized so that ruminating on
a good memory to color the universe ([M5]) is _always_ possible within it; the gate only bites _excess_ recall, which is
then covered by **additional** balance earned by writing, inviting friends, or paying. So this unit owns both halves of
the affordability contract: the spend that enforces cost, and the earn that keeps the core loop free at everyday volume.

`Earn` has three sources ([G3]): **writing a diary** (a per-launch grant fired through the consumer-owned port
[20.encode-usecase](20.encode-usecase.md) left open), **inviting a friend** (a both-sides bonus on a valid signup), and
**payment** (the charge RPC verifies a store receipt before crediting additional balance).
Anti-abuse for invites ([G6]) is a reserved `valid-signup` seam whose concrete criteria land later.

## Scope / Non-goals

**In scope**

- **Bind the real `SpendGate`** ([CC2]) ‚Äî provide the concrete `internal/twinkle` implementation of the consumer-owned
  `SpendGate` port that [33.recall-usecase](33.recall-usecase.md) and [34.view-semantic-usecase](34.view-semantic-usecase.md)
  declared, injected at `cmd/api` in place of the Epic-C no-op default. `CheckAndSpend(ctx, scope, spend SpendIntent)`
  **prices the intent** via [43.stardust-ledger](43.stardust-ledger.md)'s kind-split curves (`RecallCost` /
  `GistViewCost` over the intent's depth signal ‚Äî the caller never prices), **derives the balance**, **plans the draw**
  with [43]'s pure `PlanSpend` (basic‚Üíadditional [G2]), and ‚Äî on `ok` ‚Äî **writes the spend ledger rows** (the write is
  this unit's; [43] is pure); on `!ok` it returns the canonical `insufficient-twinkle` error and writes **nothing** (the
  recall/view is refused, not partially charged).
  Atomic with the caller's transaction so a spend and its recall commit or roll back together.
- **`Earn` use-case** ‚Äî one entry that appends a signed **earn** entry to the ledger for a reason, all reasons crediting
  **additional** (permanent) balance ([G2], basic is the daily reset, never earned):
  - **write** ([G3]) ‚Äî a fixed grant per launched diary. Memory's `LaunchStars`/`PersistEncoded` calls
    `EarnPort.OnDiaryLaunched(scope)` through a **consumer-owned port declared in `internal/memory`** ([CC8]) whose
    concrete is this unit's `Earn(write)`, injected at `cmd/api`. Fires inside the launch transaction so a launch and its
    grant are atomic; a past-dated diary that launches no star grants nothing (no star, no earn).
  - **invite** ([G3]) ‚Äî both sides credited on a **valid signup**: the inviter gets a bonus and the newly-signed-up
    friend gets a bonus (e.g. 500 each ‚Äî confirmed at implement). Idempotent per (inviter, invitee) pair so a signup
    cannot be double-claimed.
  - **payment** ([G3]) ‚Äî the earn path (`Earn(payment, amount)`) credits additional balance only after `Charge` verifies a
    store receipt through a `StorePaymentVerifier` external-service port. Test/dev may use a keyless fake, but production
    grants **no value without a verified store receipt**. No login/attendance bonus ([G3] ‚Äî the daily basic reset plays
    that role).
- **`GetBalance` RPC** ‚Äî returns the derived `{ basic, additional, total }` for the authenticated user (the read
  [45.stardust-ui](45.stardust-ui.md)'s HUD renders), `NO_SIDE_EFFECTS`, user-scoped.
- **`QuoteSpend` RPC** ‚Äî a read-only quote for the UI: given a spend kind + target id (recall, gist-view, or diary-recall
  batch), resolve the authoritative depth signal server-side, price with [43]'s curves, derive current balance, and return
  `{cost, covered, shortfall}` without writing.
- **`twinkle.v1.TwinkleService`** (NEW proto/service) ‚Äî `GetBalance`, `QuoteSpend`, verified `Charge`, and the invite-claim
  RPC; thin handlers mapping proto‚Üîdomain over the `Earn`/`Spend`/`GetBalance`/`QuoteSpend` use-cases.
- **The `valid-signup` seam** ([G6]) ‚Äî a consumer-owned predicate the invite earn consults before crediting; Epic F ships
  a **permissive default** (signup exists) and reserves the slot for the real anti-abuse criteria (min-activity, device/
  self-invite checks) with no re-plumbing later.
- Per-user scoping on every balance derivation, earn, and spend (¬ß4); anti-abuse idempotency keys on invite/payment.

**Non-goals**

- **No ledger table, balance-derivation math, spend-order rule, or cost curves.** The `twinkle_ledger` schema, the
  two-tier `basic`(daily-reset)+`additional`(permanent) balance derivation, the basic‚Üíadditional spend order, and the
  cost-curve _shapes_ (recall-by-decay-depth, gist-by-gist-depth) plus **all their VALUES** are
  [43.stardust-ledger](43.stardust-ledger.md) ([CC3][G1][G2][G4]). This unit _calls_ the ledger's pure functions ‚Äî
  `RecallCost` / `GistViewCost` (curves), `basicRemaining` (balance derivation), and `PlanSpend` (spend-order planner) ‚Äî
  and **writes** the resulting spend/earn `LedgerEntry` rows (the transaction is this unit's; [43] is pure and does not
  write). The **cost amount** a spend needs is produced by the ledger's curves from the depth signals D/E compute
  ([CC3]) ‚Äî recall/view never hardcode a price.
- **No recall / view-semantic / reconsolidation.** `Recall`, `Reinforce`, `Reconsolidate`, `ViewSemantic` are Epic C
  ([33.recall-usecase](33.recall-usecase.md) / [34.view-semantic-usecase](34.view-semantic-usecase.md)); they _call_ the
  gate this unit binds. This unit does not know what a memory is.
- **No store business logic inside the domain.** The store is an external actor reached through `StorePaymentVerifier`;
  this unit owns the receipt-verification seam and the verified credit, not provider SDK internals or merchandising copy.
  A keyless fake is allowed only for tests/dev; production credits require a verified receipt ([G3]).
- **No concrete anti-abuse criteria** ([G6]) ‚Äî the `valid-signup` predicate is a reserved seam with a permissive default;
  the real min-activity / self-invite rules land later. Reserving the slot now means no re-plumbing when they do.
- **No FE.** The balance HUD, cost display, charge flows (payment / invite / write) are [45.stardust-ui](45.stardust-ui.md);
  this unit returns the balance + the earn/spend results they render.
- **No emotion/rendering coupling** ‚Äî Twinkle is an economy scalar; it touches no position, color, size, or synapse ([I3]).

## Design

### Placement (domain-first Go, [CC8])

`Earn`, `Spend`, and `GetBalance` are application use-cases in the **NEW `internal/twinkle`** context ‚Äî a distinct
bounded context from `internal/memory` ([CC8]). They start inside the context package (ARCHITECTURE ¬ß2.3); the
`{app,pg,rpc}` split is a later refactor. Cross-context wiring goes **only through consumer-owned ports at the
composition root** (¬ß2.4):

- **`SpendGate`** is declared by the _consumers_ ‚Äî `internal/memory`'s recall/view use-cases ([33]/[34]) ‚Äî and
  **implemented here**; `cmd/api` injects this unit's concrete in place of the Epic-C no-op.
- **`EarnPort` (write)** is declared by `internal/memory` (the launch consumer) and **implemented here**; `cmd/api`
  injects `Earn(write)`.
- Twinkle **never imports memory**; the depth signals a cost needs arrive as scalar inputs to the gate, not as memory
  types. Dependency direction: `twinkle/rpc ‚Üí Earn/Spend/GetBalance (use-case) ‚Üí pure ledger domain ([43])`, with the
  ledger store and the store-payment adapter reached only through consumer-owned interfaces.

### `CheckAndSpend` ‚Äî binding the real gate ([CC2][G1][G2])

The recall/view path (Epic C) already looks like: _hand the depth signal (a `SpendIntent`) to `SpendGate.CheckAndSpend(ctx,
scope, spend)` ‚Üí on success do the recall/view_. Epic C bound a no-op that always allowed. This unit binds the concrete ‚Äî
**the gate prices the intent; the caller never computes a price** ([CC3]):

1. **Price the intent** with [43.stardust-ledger](43.stardust-ledger.md)'s **kind-split cost curves**: read the
   `SpendIntent`'s kind + depth signal and call `RecallCost(accessibilitySignal)` for a recall (D/38 supplies the
   accessibility/decay-depth signal) or `GistViewCost(semanticStage)` for a gist-view (E/40 supplies the stage). Both
   are pure golden-parity curves ([43]); the caller passed only the `SpendIntent`, not a number.
2. **Derive balance** ‚Äî `additional` = the stored running total; `basic` = [43]'s `basicRemaining(now)` (the daily-reset
   amount minus today's basic spend, derived read-time against the clock's "now", [43]/[I10]; `Balance = { basic,
additional }` is [43]'s _type_, not a reader ‚Äî the derivation reads the ledger repo).
3. **Plan the draw** with [43]'s pure `PlanSpend(basicRemaining, additional, cost) ‚Üí { fromBasic, fromAdditional, ok }`
   (basic‚Üíadditional order [G2]). If `!ok` ‚Üí return `insufficient-twinkle`, write nothing, the recall/view is refused
   (surfaced by [45.stardust-ui](45.stardust-ui.md) as "not enough Twinkle ‚Üí charge"). Nothing is deleted ([I1]).
4. **Write the ledger rows.** `PlanSpend` only _plans_ the split ([43] is pure ‚Äî it does not write); **this unit writes**
   the spend `LedgerEntry` rows (`fromBasic` + `fromAdditional`) via the twinkle ledger store ‚Äî [43] assigns the
   earn/spend transaction + its rows to plan 44.
5. **Atomic with the caller.** The write runs **inside the recall/view transaction** (`platform.InTx`, passed through
   the port), so the spend and its recall/reconsolidation commit or roll back as one ‚Äî no charge without the recall, no
   recall without the charge. **Decided (2026-07):** the `SpendGate.CheckAndSpend` signature carries the caller's tx
   handle (via the existing scope/tx convention) so cross-context spend is atomic with cross-context work; twinkle owns
   the ledger write inside memory's transaction, still through the port (no shared table access). **As built:** the
   handle is memory's opaque `EconomyTx` (`any`) ‚Äî recall/launch pass their repo tx surface, the tx-less gist view
   passes nil (the gate runs its own ledger transaction); the composition root extracts the pgx handle via the
   memory/pg store's `DB()` accessor and binds the twinkle store onto the same transaction. The recall use-case loads
   the memory first (a not-found target never charges) and derives the intent's accessibility signal from that loaded
   state at the post-sync clock; the whole-diary recall prices **all** members from one pre-action anchor snapshot
   (batch read), then spends, then reinforces ‚Äî so a reinforce nudging a sibling's forgetting offset can never
   reprice the batch mid-way. A raced tier exhaustion inside the guarded write also surfaces as the canonical
   insufficient-twinkle.

So the **price is derived by the gate** from [43]'s pure cost curves + `PlanSpend` over the intent's depth signal
([CC3]); the **caller never prices** ‚Äî it hands over the `SpendIntent`. `CheckAndSpend` prices, plans the split, writes
the rows, and enforces the balance.

### `Earn` ‚Äî the three acquisition paths ([G3])

One use-case, `Earn(scope, reason, amount?)`, appends an **earn** ledger entry crediting **additional** balance ([G2] ‚Äî
basic is never earned, it is the daily reset). Reasons:

- **write** ‚Äî `amount = twinkle.earn_write`, a fixed per-launched-diary grant. Fired by memory's launch through the
  consumer-owned `EarnPort.OnDiaryLaunched(scope)`, **inside the launch transaction** ([20]): a launch and its grant are
  atomic. **A past-dated diary that launches no star earns nothing** ([I10] ‚Äî the guard that suppresses the star also
  suppresses the write grant; no star, no earn). **Decided (2026-07):** the grant is _per launched diary_, not per star,
  so splitting one diary into more stars does not inflate the reward ([G3] "ÏùºÍ∏∞ Ïì∞Í∏∞" is one act).
- **invite** ‚Äî both sides credited on a **valid signup**: `Earn(inviter, invite)` grants `twinkle.earn_invite_inviter`
  and `Earn(invitee, invite)` grants `twinkle.earn_invite_invitee` (500 each, confirmed; two keys so an asymmetric
  inviter bonus is expressible). The claim is **idempotent per (inviter, invitee)** via an idempotency key so a single
  signup credits exactly once each side. Gated by the `valid-signup` predicate ([G6], below). **As built:** the invite
  code IS the inviter's user id (share links carry it; no code table ‚Äî job 54's two tables stay the epic's whole
  schema), and idempotency is two-layered: the invitee-side entry is keyed once-per-signup (`invite_signup:<invitee>`
  ‚Äî a second claim with ANY code is a replay that credits no one) and the inviter-side entry per pair
  (`invite:<inviter>:<invitee>`), both in one ledger transaction.
- **payment** ‚Äî `Charge(scope, packId, platform, receipt)` calls `StorePaymentVerifier.Verify(...)` and, only on a valid
  receipt for a known pack, calls `Earn(scope, payment, verifiedAmount)` to credit additional balance. The verifier returns
  the authoritative grant amount / idempotency key; replaying the same receipt is idempotent. A keyless fake verifier may
  exist in test/dev, but production grants no value without a verified store receipt ([G3]). Price structure (e.g. $1 =
  100 Twinkle) is product/config. **As built:** the shipped binding is the deterministic `KeylessPaymentVerifier`
  (accepts only the single v1 `DefaultChargePackID` with a non-empty receipt; dedup key = digest of the receipt
  content alone, so a respelled platform can never double-credit); the production store-SDK adapter is a deferred
  seam behind the same port and must be bound before real payments launch.

There is **no login/attendance earn** ([G3]) ‚Äî the daily basic reset ([43][G2]) is that role by design.

### Core-loop protection ([G5]) ‚Äî the affordability contract

[G5] is an _invariant on the numbers_, enforced across this unit and [43]: the daily **basic** reset must cover everyday
rumination ([M5]) so the gate only bites _excess_ recall. Concretely: `twinkle.basic_daily_amount` (owned by [43]) is set so
that a typical day's ruminative recalls at the _cheap end_ of the recall cost curve fit inside it, and `twinkle.earn_write`
tops up additional balance at roughly the rate writing generates recall-worthy stars. **Decided (2026-07):** [G5] is
verified as a **relationship test** at implement ‚Äî `basic_daily_amount ‚â• expected_daily_ruminations √ó cheap_recall_cost` ‚Äî not a
single magic constant; the plan asserts the property, [43] owns `basic_daily_amount` and the cost curve, this unit owns
`earn_write`, and the test lives across both. Rumination stays free at everyday volume; charge appears only past it.
**As built:** the test's documented assumption is 5 expected daily ruminations; with the shipped values the margin is
100 ‚â• 5 √ó 15.

### `valid-signup` seam ([G6]) ‚Äî reserve the slot

Invite earn consults a consumer-owned `ValidSignup(inviter, invitee) ‚Üí bool` predicate before crediting either side.
Epic F ships a **permissive default** (a real, distinct signup exists ‚Äî the pair is not self-invite by the same
account; as built: `twinkle.DistinctSignup`, bound at `cmd/api`). The real anti-abuse criteria (minimum activity after
signup, device/payment-instrument dedup, rate caps) are a reserved seam ‚Äî bound later behind the same predicate with
no change to `Earn(invite)`. This mirrors the PRD's "reserve the slot" discipline ([G6] placeholder): the structure
knows anti-abuse lands here without building it now.

### `QuoteSpend` ‚Äî the server quote the UI renders ([G4])

`QuoteSpend(scope, kind, target)` is the read-only twin of `CheckAndSpend`: it uses the same pricing and balance
derivation, but writes nothing. For a recall or gist-view, the request carries an `episodic_memory_id`; for "Ïù¥ ÏùºÍ∏∞Î°ú
ÌÉúÏñ¥ÎÇú Î≥Ñ Î≥¥Í∏∞" ([D3]) it carries a `diary_id`. The use-case resolves the authoritative depth signal server-side through a
consumer-owned `SpendSignalReader` port bound at `cmd/api` (memory facts by id only; no memory internals imported), then:

1. prices the intent with [43]'s `RecallCost` / `GistViewCost` curves;
2. derives the current `{basic, additional}` balance;
3. plans the draw with `PlanSpend`;
4. returns `{cost, covered, shortfall}`.

It is `NO_SIDE_EFFECTS`: no ledger row, no clock advance, no recall/view. `CheckAndSpend` remains the authoritative write
at action time, so a stale quote can be refused by the real spend if the balance or depth changed.

**As built:** the `SpendSignalReader` binds over memory's published reads (`RecallAccessibility`,
`DiaryRecallAccessibilities` ‚Äî one batch anchor read per diary, no text payload ‚Äî and `ViewableGistStage`); signals
derive at the clock a real recall would sync to (`GREATEST` of the guard baseline and real today, with the unborn
clock falling back to the latest launched memory exactly like the sync guard). The gist-view quote prices the
memory's **reached** stage ‚Äî the deepest (cheapest) viewable gist, the primary view flow; a shallower stage view can
cost more than that quote, which the authoritative spend enforces (a per-stage quote is a contract extension for the
UI unit if it needs one).

### RPC contract (`twinkle.v1.TwinkleService`, Connect, unary ‚Äî ¬ß2.7)

A NEW proto service and package (`internal/twinkle/rpc`), thin handlers mapping proto‚Üîdomain and calling the use-cases;
all policy (balance check, spend order, earn reasons, idempotency, valid-signup) lives in the use-case/domain (¬ß2.9#7):

```proto
// balance read ‚Äî the HUD source ([45]); no side effects, user-scoped.
rpc GetBalance(GetBalanceRequest) returns (GetBalanceResponse);
// spend quote ‚Äî server-priced, read-only; the UI displays this before recall/gist-view/diary-jump.
rpc QuoteSpend(QuoteSpendRequest) returns (QuoteSpendResponse);
// invite claim ‚Äî both-sides credit on a valid signup; idempotent per (inviter, invitee).
rpc ClaimInvite(ClaimInviteRequest) returns (ClaimInviteResponse);
// payment charge ‚Äî verifies a store receipt before crediting additional Twinkle ([G3]).
rpc Charge(ChargeRequest) returns (ChargeResponse);

message GetBalanceRequest  {}
message GetBalanceResponse { int64 basic = 1; int64 additional = 2; int64 total = 3; } // derived [G2]

enum SpendKind { SPEND_KIND_UNSPECIFIED = 0; SPEND_KIND_RECALL = 1; SPEND_KIND_GIST_VIEW = 2; SPEND_KIND_DIARY_RECALL = 3; }
message QuoteSpendRequest  { SpendKind kind = 1; string episodic_memory_id = 2; string diary_id = 3; }
message QuoteSpendResponse { int64 cost = 1; bool covered = 2; int64 shortfall = 3; }

message ClaimInviteRequest  { string invite_code = 1; }                 // invitee redeems the inviter's code
message ClaimInviteResponse { int64 balance_total = 1; }                // idempotent: replay returns same total

message ChargeRequest  { string pack_id = 1; string platform = 2; string receipt = 3; }
message ChargeResponse { int64 balance_total = 1; }
// NOTE: the write-earn has NO RPC ‚Äî it fires from memory's LaunchStars via the consumer-owned EarnPort, in-transaction.
// The spend has NO RPC ‚Äî it fires from recall/view via the consumer-owned SpendGate this unit binds.
```

`GetBalance` and `QuoteSpend` are GET-eligible / `NO_SIDE_EFFECTS` (reading the balance or a price never earns or spends).
`ClaimInvite`/`Charge` mutate the ledger and are idempotent per their key. Earn-on-write and spend have **no** RPC of their own ‚Äî they are
cross-context port calls fired from memory's launch and recall/view transactions, so the economy stays a _consequence_
of the memory actions, not a separate user step.

### Golden-parity stance

The **balance** the gate reads and the cost curves are [43]'s, where the golden-parity stance is declared (FE previews a
cost/balance; BE is authoritative). This unit's own logic is **not** golden-parity: earn reasons, idempotency, the
valid-signup gate, and the store verifier are BE-only orchestration with **no FE math to mirror** ‚Äî the client reads results
(`GetBalance`, `QuoteSpend`, the charge/earn responses), it never recomputes an earn or the actual debit (¬ß2.9#8).

### Thin handler / dependency rule

`twinkle/rpc` handlers map proto DTOs to domain inputs and call `Earn`/`Spend`/`GetBalance`; the balance check, spend
order, earn-reason routing, idempotency, and valid-signup policy live in the use-case/domain (¬ß2.9#7). No proto or sqlc

## Acceptance Criteria

1. The concrete `internal/twinkle` **`SpendGate`** replaces the Epic-C no-op at `cmd/api`; a recall/gist-view now
   **derives balance ‚Üí checks ‚Üí deducts** before proceeding ‚Äî the priced core loop is live. ([CC2][G1])
2. `CheckAndSpend(scope, spend)` **prices the `SpendIntent`** via [43]'s `RecallCost` / `GistViewCost`, plans the draw
   with [43]'s pure `PlanSpend` and, on `ok`, **writes the spend rows basic-first then additional** ([G2] order); on
   `!ok` it returns the
   canonical `insufficient-twinkle`, writes nothing, and the recall/view is **refused** (nothing deleted [I1]). ([G1][G2])
3. The spend is **atomic with its caller's transaction** ‚Äî the recall/reconsolidation and its debit commit or roll back
   together; there is no charge without the recall and no recall without the charge. (¬ß2.8, [CC2])
4. The **price is derived by the gate** from [43.stardust-ledger](43.stardust-ledger.md)'s cost curve over the intent's
   depth signal (recall-by-decay-depth, gist-by-gist-depth); the **caller passes only the `SpendIntent`** and **never
   hardcodes or computes** a price. ([CC3][G4])
5. `QuoteSpend` returns a server-derived `{cost, covered, shortfall}` for recall, gist-view, and diary-recall batch spends,
   is `NO_SIDE_EFFECTS`, and writes no ledger row; the real `CheckAndSpend` remains authoritative at action time. ([G4])
6. **Earn-on-write** grants `twinkle.earn_write` to **additional** balance, fired from memory's launch via the consumer-
   owned `EarnPort` **inside the launch transaction**, once **per launched diary** (not per star); a **past-dated diary
   that launches no star earns nothing**. ([G3][I10])
7. **Earn-on-invite** credits the inviter `twinkle.earn_invite_inviter` and the new friend `twinkle.earn_invite_invitee`
   on a **valid signup**, **idempotent per (inviter, invitee)** so a signup credits each side **exactly once**. ([G3])
8. **Earn-on-payment** credits additional balance only after `Charge` verifies a store receipt through `StorePaymentVerifier`;
   repeated receipts are idempotent, and production grants no value without verification. ([G3])
9. There is **no login/attendance earn** ‚Äî the daily-reset basic balance ([43][G2]) plays that role. ([G3])
10. **Core-loop protection** ([G5]) holds as a relationship: the daily **basic** reset covers expected everyday rumination
    ([M5]) at the cheap end of the recall curve ‚Äî everyday recall stays free within basic; the gate bites only _excess_.
    Verified as a `basic_daily_amount ‚â• expected_daily_ruminations √ó cheap_recall_cost` property, not a magic constant. ([G5])
11. **`GetBalance`** returns the derived `{ basic, additional, total }`, is `NO_SIDE_EFFECTS`, and never earns or spends
    on read. ([G2])
12. The **`valid-signup` seam** ([G6]) is a consumer-owned predicate with a permissive default (real distinct signup,
    not self-invite); the concrete anti-abuse criteria are a reserved slot bound later with no change to `Earn(invite)`. ([G6])
13. Handlers are **thin** (proto‚Üîdomain map + call); balance-check, spend-order, earn-reason, idempotency, and valid-
    signup policy live in the use-case/domain; no proto/sqlc type leaks inward; `SpendGate`/`EarnPort` (implemented here),
    the ledger store, `SpendSignalReader`, the store verifier, and `ValidSignup` are consumer-owned interfaces; **twinkle never imports
    memory**. (¬ß2.2/¬ß2.4/¬ß2.7/¬ß2.9#4/#6/#7, [CC8])
14. Every balance derivation, quote, earn, and spend is **scoped to the authenticated user**, and `pnpm lint:persistence`
    passes (¬ß4).
