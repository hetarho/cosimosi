# 37. 두 우주 겹쳐보기 - universe-overlay + ShareService.GetResonanceBridges (as-built)

> 현재 구현 기준 문서. 친구의 공개 우주(35) 위에 내 우주를 한 씬에 이격 배치하고, 공명(36)으로 이어진 별 쌍 사이에 두 우주를 가로지르는 빛의 다리를 놓는 소셜 캡스톤의 SSOT다. | 범위: FE(+BE read 1 RPC) | 상태: 구현 완료, 반영: XState 항행/포커스 머신 가산 확장(tech/state-machines.md)

## 목적

겹쳐보기는 공명의 보상 장면이다. 36까지 공명은 "내 별의 resonant 마커 + 상대 표시명"으로만 보인다. 두 우주를 실제로 겹쳐 다리가 놓인 장면이, 별을 보내고 재작성한 수고의 감정적 결실이다(concept §함께한 기억의 구현 형태 (b) "두 사람이 공유하는 겹침 공간").

차이를 보여주는 것이 핵심이다. 공명 쌍은 같은 사건의 두 기억이고, 색(감정)·밝기(회상 빈도)·형태(재성형)가 다르게 변해 왔다. 겹침 뷰는 그 갈라짐을 나란히 보여준다. 그래서 겹침은 서버 합성 상태가 아니라 순간의 뷰이며, 그동안 어떤 쓰기 RPC도 부르지 않는다.

## 현재 구현

- 겹침은 별도 위젯이 아니라 `widgets/universe-canvas/ui/overlay/*`에 산다. navigation 머신이 같은 위젯이라 FSD 동일 레이어 교차를 피하려고 여기에 둔다. barrel(`widgets/universe-canvas/index.ts`)이 `UniverseOverlay`/`OverlayComparePanel`/`Bridge`를 노출한다.
- 두 호스트가 같은 렌더·다리·비교 패널을 재사용한다:
  - 공개 방문 페이지 `pages/visit/ui/VisitPage.tsx` — 로그인 사용자 한정 "내 우주와 겹쳐보기" 토글.
  - 데모 home `pages/home/ui/HomePage.tsx` — `useDemoOverlay` 플래그가 켜지면 두 페르소나 우주를 띄우는 전용 뷰.
- 두 우주는 한 `<Canvas>`(`UniverseOverlay`) 안에 세로 축으로 이격 배치된다. 내 우주는 아래(`offset y = -66`), 친구 우주는 위(`offset y = +66`) — 상수 `OVERLAY_OFFSET = 66`(`UniverseOverlay.tsx`). 각 우주는 자기 force-sim으로 좌표가 창발한다(헌법3 — 서버 좌표 없음).
- 친구 별무리는 시각 구분된다: 친구의 30 설정 색을 유지하되, 별 레벨에서 공통 atmosphere 색 쪽으로 `TINT_STRENGTH = 0.16`만큼 끌어당기는 틴트 + 우주를 감싸는 faint sphere(opacity 0.06, additive). 내 우주는 따뜻한 `#3a2f5a`, 친구 우주는 차가운 `#23406a` atmosphere로 "남의 하늘" 느낌을 준다.

### `UniverseOverlay` (씬 조립)

- `widgets/universe-canvas/ui/overlay/UniverseOverlay.tsx`.
- 단일 우주 `UniverseCanvas`와 별개의 자족 `<Canvas>`다 — 두 우주는 단일 우주용 싱글턴(memory/synapse store)을 공유할 수 없으므로, 데이터를 **PROPS**(`mine`/`theirs`/`bridges`)로 받는다.
- `UniverseCanvas`의 렌더러 수명주기를 그대로 미러한다: async WebGPU factory + WebGL2 폴백(`createRenderer`), `requestAnimationFrame` 지연 마운트, `ResizeObserver`, init-error를 throw로 surfacing.
- 두 WebGPU 컨텍스트 공존을 막기 위해 호스트가 `overlayReady`일 때만 `UniverseCanvas` 대신 이것을 마운트한다.
- 구성: `OverlayDust`(독립적인 1400개 ambient 먼지) + `OverlayUniverse` 2개 + `ResonanceBridges` + `OverlayCamera` + `BloomPass`.

### `OverlayUniverse` (우주당 1개 — force-sim + 렌더)

- `widgets/universe-canvas/ui/overlay/OverlayUniverse.tsx`.
- 우주당 force-sim **1개**(`shared/lib/force-sim`의 `createSim`/`advance`/`isSettled`). 그래프당 1개 — 한 Worker가 두 그래프를 도는 구조가 아니다.
- sim 노드는 단일 우주와 같은 시드 규약으로 빚는다: 강함 셸(`radiusOf`, 38) × `fibonacciStarPosition`, 이웃이 없는 외톨이 별은 `scatterDirection`으로 흩는다(38). `RADIAL_SIM_PARAMS`(`radial-layout.ts`) 재사용.
- 싱글턴 store를 건드리지 않고 `StarField`/`SynapseFilaments`를 외부 소스 prop으로 구동한다(`StarField`의 spec-37 `stars`/`edges`/`tint`/`tintStrength` 입력, `SynapseFilaments`의 `positionsRef`/`idIndex` 입력).
- `positionsRef`는 sim 내부 버퍼(`sim.px`)를 가리킨다 — `advance`가 제자리 갱신하므로 매 프레임 복사 없이 별·다리가 라이브로 따라온다.
- effect 내 동기 setState를 피하려고 빌드 effect는 sim만 세팅하고 레이아웃 발행은 `useFrame`이 맡는다(`pendingRef` 'seed'/'empty'). 오버레이는 정적 뷰라 settle 후 재-kick 하지 않는다.
- `handleRef`(`OverlayHandle`)로 이 우주의 라이브 버퍼·id→row 인덱스·group offset을 부모에 노출한다 — 다리/카메라가 월드 좌표 = 로컬 + offset으로 환산한다.

### `ResonanceBridges` (우주를 가로지르는 유일한 선)

- `widgets/universe-canvas/ui/overlay/ResonanceBridges.tsx`.
- 각 다리는 내 별 ↔ 상대 별(36 공명)을 잇는다. `Bridge = { myId, theirId }` 순수 두-id 쌍 — 스냅샷 인덱스 결합은 호출부(VisitPage)에서 `shared-N`으로 환원해 넘긴다.
- WebGPU 경로: `LineSegments2` + `Line2NodeMaterial`(09의 밝은 시냅스/다리 렌더 관용구, 헌법8). 다리당 `SEG = 22` 분할로 매끈한 호를 만들고, additive 블렌딩 + Bloom으로 가느다란 밝은 호가 빛의 다리로 번진다.
- 빛 흐름 애니메이션은 수동 uniform이다 — BloomPass 아래에서 TSL `time` 노드가 동결되므로, 수동 시간 누적(`tRef`)으로 매 프레임 정점 색에 흐름 패킷(`flowPacket` smoothstep 봉우리)을 굽는다. `FLOW_SPEED = 0.16` packets/sec(느린 빛 흐름).
- 두 sim이 매 틱 움직이므로 다리는 매 프레임 양 우주 핸들에서 월드 끝점을 다시 읽는다. 끝점이 아직 준비 안 됐으면 그 다리 세그먼트를 0-length로 접어 보이지 않게 한다(떨어진 다리 없음, 3.3).
- 다리 중점에 클릭 가능한 glint 핸들(`HANDLE_RADIUS = 1.6` sphere, additive)을 둔다. 클릭 → focus 머신 `SELECT_PAIR` + navigation 머신 `FRAME_PAIR`. 선택된 쌍의 다리는 `FOCUS_BOOST = 1.7`로 더 밝게 탄다. 라인 자체는 raycast off — 클릭은 핸들/별로만.

### `OverlayCamera` (자유 궤도 + 쌍 프레이밍 비행)

- `widgets/universe-canvas/ui/overlay/OverlayCamera.tsx`.
- 두 하늘을 함께 도는 자유 궤도(drei `OrbitControls`, `overlay.viewing`)와 다리 클릭 시 두 별을 한 프레임에 담는 비행(`overlay.framingPair`)을 한다.
- `FRAME_PAIR`로 navigation이 `framingPair`에 들면 두 별의 월드 좌표(각 우주 라이브 버퍼 + offset)를 풀어 둘 다 화면에 담는 vantage로 lerp(`k = 1 − exp(−dt·4)`)하고, 도착 시 `ARRIVED`를 보낸다 — 단일 우주 `FrameAll` 비행 감각과 동형.
- 비행 중엔 거리 클램프를 푼다(`minDistance` 0.01 / `maxDistance` 100000) — 근접 쌍에서 클램프 재적용이 도착 판정을 흔드는 지터를 막는다. settled `viewing`에선 다시 조인다(min 40 / max 2000). `framingPair`를 떠나면 비행을 양보한다.

### `OverlayComparePanel` (비대칭 비교 — 상대는 시각 정보만)

- `widgets/universe-canvas/ui/overlay/OverlayComparePanel.tsx`.
- R3F 씬 밖 DOM HUD(헌법4)다. focus 머신의 `pairFocus`(`selectPairFocus`)를 읽어, 공명 쌍이 선택됐을 때만 뜬다.
- 설계상 비대칭(2.3): 내 별은 감정(색 swatch)·강도 미터 + 조각 텍스트를, 친구 별은 **감정·강도 시각 정보만** 보여준다 — 친구 쪽은 텍스트가 절대 렌더되지 않는다(콘텐츠 제로, 35).
- 밝기(회상 빈도) 차이는 3D 씬의 실제 별 밝기로 이미 드러나므로, 패널은 정적 데이터(감정·강도)만 보여 렌더 순수성을 지킨다(`now` 미사용).
- 내 별 텍스트는 호스트가 `resolveMyText`로 읽기 전용으로만 공급한다(쓰기 금지 3.1). 방문 호스트는 회상 fragment 텍스트 캐시 → 없으면 내 일기 목록(읽기 전용 `ListRecords`) 발췌로 폴백한다. 데모 호스트는 페르소나 일기에서 직접 읽는다(`demoFragmentText`/`demoRecall`). 둘 다 없으면 "내 우주에서 이 별을 회상하면 그 일기를 볼 수 있어요" 힌트.

## navigation 머신 — `overlay` 상태 (39 가산 확장)

- `widgets/universe-canvas/model/navigation.machine.ts`(순수 TS, 헌법4).
- 겹쳐보기 = `overlay` 상태로 진입(`nebula`/`recall`과 배타). `nebula`/`recall`/`flyingToStar`/`framingDiary`/`modeTransition` 어디서든 `ENTER_OVERLAY`로 진입 가능하고, 진입 직전 모드를 `overlayReturn`에 기록한다.
- `overlay`는 `viewing`/`framingPair` 자식 상태를 가진다:
  - `viewing`: 자유 궤도. `FRAME_PAIR` → `framingPair`(`setPair`로 `pairMyId`/`pairTheirId`/`pairSeq` 갱신).
  - `framingPair`(tag `transitioning`): 쌍 프레이밍 비행. `ARRIVED` → `viewing`, 같은/다른 다리 재클릭은 `pairSeq`로 구분(reenter). 타깃 미해결 시 `FLIGHT_TIMEOUT_MS = 10000ms` 안전망.
- `overlay` entry/exit가 쓰기 게이트를 켜고/끈다(`blockWrites`/`unblockWrites` → `setOverlayWriteBlocked`). boolean 플래그가 아니라 상태가 단일 출처다(3.1). `viewing↔framingPair` 내부 전이엔 entry/exit가 재실행되지 않는다.
- `EXIT_OVERLAY` → `overlayReturn`에 따라 `recall` 또는 `nebula`로 복귀(항해 모드 유실 방지) + `clearPair`.
- selector: `selectFramingPair`(`{ myId, theirId, seq }` | null). 오버레이 마운트/비행 중 여부는 navigation snapshot 상태/tag를 직접 읽어 판정한다.

## focus 머신 — `pair` 상태 (39 가산 확장)

- `entities/memory/model/focus.machine.ts`(순수 TS). 37 초안의 옛 universe model 위치가 아니라, focus 머신은 `entities/memory/model`에 산다(widgets와 features가 단방향으로 읽으려면 공통 하위가 entities뿐).
- `idle`/`star`/`diary`에 더해 `pair` 상태를 가진다. `SELECT_PAIR { myId, theirId }` → `pair`(단일 `starFocus`/`diaryFocus`와 배타 — 한 번에 하나). `DISMISS` → `idle`.
- selector: `selectPairFocus`(`{ myId, theirId }` | null) — 비교 패널·다리 강조의 단일 출처.

## 쓰기 게이트

- `shared/lib/write-gate.ts`(순수 모듈, 헌법4) — 모듈 스코프 boolean. navigation `overlay` 상태가 출처이고, FSD 하향 의존상 features(쓰기 경로)가 widgets의 머신을 직접 읽을 수 없어 이 shared 게이트로 파생 백스톱한다.

## 현재 RPC 표면 — `ShareService.GetResonanceBridges`

- 인증 RPC, `NO_SIDE_EFFECTS`. 무인증 `VisitService`가 아니라 인증 `ShareService`에 둔다 — 공명 매칭에 호출자 신원이 필요하고, 공개 표면은 콘텐츠 제로로 유지하기 위함이다.
- 친구 우주 데이터 자체는 35의 무인증 `VisitService.GetSharedUniverse`를 그대로 쓴다(겹침이 공개 표면을 넓히지 않는다).
- request: `slug`(겹칠 우주 = 주인). response: `repeated ResonanceBridge { my_memory_id, their_star_index }`.
- `my_memory_id`는 호출자 자기 별 id(이미 그 텍스트를 가진다). `their_star_index`는 그 slug의 `GetSharedUniverse` 응답 배열 인덱스 — 상대 memory id를 절대 노출하지 않고 다리 끝점을 둘 수 있게 한다(콘텐츠 제로 유지).

### BE 처리 (`Service.ResonanceBridges`)

- `backend/internal/share/service.go`. handler는 `backend/internal/share/handler.go`(`ShareServiceHandler`, JWT sub로 `userID`).
- slug → 주인 user_id(`UserBySlug`). 미존재/비활성/회전된 slug면 `ErrNotFound` → `CodeNotFound`로 균일 응답(주인이 공유를 끄는 순간 겹침도 막힘, 3.2 / 35의 균일 NotFound와 동일).
- `ResonancesBetween(callerUserID, ownerUserID)`로 두 사람 사이의 공명 쌍을 읽는다. 0개면 빈 배열(비당사자거나 아직 공명 없음 — 존재 비노출, 2.2).
- 상대 memory id → 스냅샷 인덱스: `ListStarIDs`가 공개 스냅샷 조립(`ListStars`)과 **같은 `ORDER BY m.id`**(쿼리 `ListSharedStars`/`ListSharedStarIDs`, `share.sql`)를 쓴다 — 두 RPC의 배열 인덱스가 정합해 인덱스 표류가 없다. 현재 스냅샷에 없는 주인 별은 드롭한다(stray index 미방출).
- 도메인 타입은 transport/db 태그 없는 순수 값(헌법5).

### BE 의존 배선 (컴포지션 루트)

- `share`는 `gift`(36 resonances)를 import하지 않는다(`gift`는 35 표시명 때문에 `share`에 의존 — 직접 import는 순환). `share`는 consumer 포트 `ResonanceReader`만 안다.
- `backend/cmd/api/main.go`의 `shareResonanceAdapter`(포인터, settable `inner`)가 `gift.Service`를 `share.ResonanceReader`로 어댑트한다. share를 먼저 만들고, gift를 만든 뒤 `resonanceAdapter.inner = giftSvc`로 순환을 닫는다(메서드는 요청 시점에만 호출되므로 그때 inner는 바인딩됨).
- `gift.Service.ResonancesBetween`는 `caller == owner`면 빈 배열(공명은 늘 두 사용자를 가른다 — self-accept 거부)이고, 그 외엔 repo 패스스루.
- SQL `ListResonanceBridges`(`backend/internal/db/queries/gift.sql`): `resonances`를 양쪽 끝점 `memories`에 조인하고, 양방향 user_id 가드(caller×owner OR owner×caller)로 정확히 그 두 사람의 별일 때만 매칭한다. `CASE`로 내 쪽/상대 쪽을 정규화(무방향). 삭제 쿼리 없음(헌법2).

## 데모 변형 (서버 없는 (b) 겹침 쇼케이스)

- 데모엔 서버가 없어 gift/accept 흐름이 없으므로 "이미 공명된 두 우주"를 코퍼스에서 파생한다.
- `shared/lib/demo/simulate.ts`의 `crossResonances(a, b, max=4)`: 두 페르소나 코퍼스의 조각을 펼쳐(`flattenFragments`), 페르소나 내부 의미 링크와 같은 `topicCosine`을 두 코퍼스 *사이*에 적용한다. 임계 `SIM_TAU = 0.4` 이상 후보를 강도로 살짝 우대(`cos + (intensity 합)*0.05`)해 점수 desc 정렬하고, 1:1 매칭(각 별은 한 다리에만)으로 최대 4쌍을 고른다. 결정론적(난수 없음).
- `shared/lib/demo/data.ts`의 `demoOverlayData()`: 활성 페르소나(`getDemoPersona`) = mine, 다른 페르소나 = theirs, 그 사이 `crossResonances`. 코퍼스가 1개뿐인 축소 빌드면 친구 쪽을 비우고 다리도 없다.
- `shared/lib/demo/overlay-mode.ts`의 `useDemoOverlay`(zustand): demo flow가 켜고 home이 읽는다(페이지와 universe-canvas widget을 직접 결합하지 않기 위한 가벼운 플래그).
- `pages/home/ui/HomePage.tsx`가 `demoOverlayData`를 proto→StarNode/SynapseEdge로 매핑해 T004/T005 렌더·다리·비교 패널을 재사용하는 전용 겹침 뷰를 띄운다(활성 페르소나·가상 시계 변경 시 재시뮬).

## Flutter 동등성 기준

- 렌더·다리·카메라·force-sim 2그래프·Bloom·수동 uniform 빛 흐름은 WebGPU/three 전용이라 그대로 포팅되지 않는다. Flutter는 자체 3D 렌더러로 두 별무리를 이격 배치하고 각 우주의 자체 force-sim으로 좌표를 창발해야 한다(헌법3 — 서버 좌표 없음).
- 포팅해야 하는 BE 계약:
  - 인증 `ShareService.GetResonanceBridges(slug)`를 호출해 `{ my_memory_id, their_star_index }` 쌍을 받고, `their_star_index`를 같은 시점 `GetSharedUniverse` 응답 배열의 별로 환원해 다리 끝점을 둔다.
  - 비당사자는 빈 배열, 공유 off는 균일 NotFound임을 같게 처리한다.
  - 두 RPC가 따로 페치되므로 친구 별 수 범위 밖 인덱스는 버려 떨어진 다리를 그리지 않는다.
- 포팅해야 하는 동작 계약:
  - 겹침은 순수 읽기 뷰 — 동안 어떤 쓰기 RPC도(회상 강화 포함) 부르지 않는다.
  - 비교 패널은 비대칭 — 내 쪽만 텍스트, 친구 쪽은 감정·강도 시각 정보만(절대 텍스트 없음).
  - 겹쳐보기 토글은 로그인 사용자에게만 노출한다.

## 수용 기준

1. 로그인 사용자가 친구의 `/u/$slug`에서 겹쳐보기를 켜면 내 우주와 친구 우주가 한 씬에 이격 배치되고, 각 우주 좌표는 각자의 force-sim에서 창발한다(서버 좌표 없음).
2. 비로그인 방문자에게는 겹쳐보기 토글을 노출하지 않는다(35 화면 그대로).
3. 호출자와 우주 주인 사이에 공명이 있으면 `GetResonanceBridges`가 그 쌍을 반환하고 FE가 두 별을 잇는 다리를 렌더한다.
4. 호출자가 공명 당사자가 아니면 빈 배열을 반환한다 — 다리·공명 존재가 제3자에게 노출되지 않는다.
5. 상대 memory id는 서버를 떠나지 않는다 — 응답은 내 memory id와 공개 스냅샷 인덱스만 싣는다.
6. `their_star_index`는 같은 `ORDER BY m.id`를 공유해 `GetSharedUniverse` 배열과 정합하고, 범위 밖 인덱스는 끝점 없는 다리로 그려지지 않게 버려진다.
7. 다리를 클릭/근접하면 두 별을 한 프레임에 담는 카메라로 이동하고, 비교 패널에 내 별의 텍스트·감정과 상대 별의 시각 정보만(텍스트 없음)을 표시한다.
8. 겹침 뷰가 떠 있는 동안 어떤 쓰기 RPC도 호출하지 않는다(회상 강화 포함 — navigation `overlay` 상태가 쓰기 게이트의 단일 출처).
9. 상대가 공유를 끄거나 슬러그를 회전하면 방문·겹침 모두 균일 NotFound로 막힌다.
10. 두 sim이 매 틱 움직여도 다리 끝점이 매 프레임 양쪽 별 월드 좌표(로컬 + group offset)를 따라간다.
11. 겹침은 뷰 상태일 뿐 — 서버에 합성 우주 상태를 만들지 않는다.
12. 데모에서 두 페르소나 우주가 이격 배치되고, 코퍼스에서 결정론적으로 파생한 공명 쌍(`crossResonances`) 사이에 다리가 놓인다.
