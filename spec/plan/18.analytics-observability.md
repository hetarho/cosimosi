# 18. analytics-observability - Sentry 강화 + PostHog 제품 지표 + 큐 가시성 (as-built)

> 현재 구현 기준 문서. 베타가 "무엇이 깨졌고(에러), 사용자가 어디서 막히는가(행동)"를 측정하기 위한 관측 배선의 SSOT다. 정전 규칙은 policy/ux/analytics-privacy.md. | 범위: FS (FE 분석·Sentry + BE 큐 로그) | 상태: 구현 완료(키 없으면 no-op), 반영: 39 계측을 XState 머신으로 이식, 21 record_memory에 fragment_count 추가. 실계정 연결은 33(미빌드)

## 목적

베타는 "출시"가 아니라 측정 장치다. 에러는 Sentry로, 행동 지표는 PostHog로 본다. 두 도구 모두 키가 없으면 초기화조차 하지 않아(no-op) 로컬 dev와 체험 모드는 아무것도 보내지 않는다 — 노이즈·쿼터 낭비가 없다. 핵심 제약은 프라이버시다: cosimosi의 정체성은 "일기는 비공개"이므로 측정이 본문을 만지는 일은 어떤 예외도 없다. 분석 래퍼는 일기 body·임베딩·기억 내용이 어떤 이벤트에도 실리지 못하도록 컴파일 단계에서 막는다(길이 버킷·mood까지만). 큐 가시성은 백로그가 침묵 속에 쌓이는 것을 `docker logs`만으로 보게 한다.

## 현재 구현 — FE 분석 래퍼

PostHog 위의 얇은 타입 계층이 `frontend/src/shared/lib/analytics/index.ts`에 있고, `frontend/src/shared/lib/index.ts` 배럴로 재export된다. 이 모듈은 pure 레이어다 — react/three를 import하지 않는다(헌법4·3.7). posthog-js는 DOM 전역만 쓰는 비 react 라이브러리라 허용이다.

- 이벤트 이름은 `EVENTS` 상수로 못 박혀 있다(오타 계측 방지). 계측 지점은 반드시 이 키로 capture한다.
- `EventProps`는 이벤트별 허용 속성의 닫힌 목록이다. 자유 텍스트 필드가 없어 본문이 끼어들 구조적 여지가 없다 — "본문이 슬쩍 실리는" 사고를 타입이 막는다.
- `capture<E>(event, props)`는 모든 이벤트에 공통 속성 `demo`(`isDemoMode()`)를 자동으로 붙인다. 체험 트래픽이 베타 퍼널을 오염시키지 않게 대시보드에서 거른다. `enabled`가 아니면 즉시 반환(no-op).
- `initAnalytics({ key, host })`는 부팅 1회(main.tsx)다. key가 없거나 이미 초기화됐으면 반환(영구 no-op·이중 init 무시). PostHog 설정은 보수적 기본:
  - `mask_all_text: true` + `mask_all_element_attributes: true` — autocapture가 어떤 요소 텍스트/속성도 줍지 않는다.
  - `disable_session_recording: true` — Session Replay 미도입(스펙의 선택 항목, 백로그).
  - `person_profiles: 'identified_only'` — 익명 방문자는 프로필을 만들지 않는다.
  - `api_host`는 opts.host 또는 `https://us.i.posthog.com` 기본, `defaults: '2026-01-30'`.
- `identifyUser(userId)`는 식별자를 Supabase uid만 보낸다(PII 최소화, 헌법4). 이메일·이름은 보내지 않는다.
- `resetAnalyticsIdentity()`는 `posthog.reset()`을 호출해 다음 방문자가 이전 uid로 이어지지 않게 끊는다.
- `bodyLengthBucket(codePoints)`는 본문 길이를 `short`(≤100) · `medium`(≤500) · `long`(>500)로 버킷한다. 원문·길이 원값은 보내지 않는다. 코드포인트 기준(서버 RuneCount 상한과 같은 단위). 경계 정전 값은 policy/ux/analytics-privacy.md.

### `universe_loaded` 합류

렌더러 백엔드(캔버스 onCreated)와 우주 데이터(GetUniverse 성공)는 서로 다른 시점에 비동기로 도착한다. 래퍼가 모듈 레벨로 둘을 모은다:

- `reportUniverseRenderer('webgpu' | 'webgl2')` — 첫 값만 기록.
- `reportUniverseData({ star_count, synapse_count })` — 첫 값만 기록.
- 둘이 다 모이면 `universe_loaded`를 페이지 로드당 1회만 보낸다. `load_ms`는 페이지 오픈(time origin)부터 우주가 처음 그려질 준비가 된 순간까지(`Math.round(performance.now())`).

## 현재 구현 — 이벤트 계측 지점

발화 로직은 대부분 XState 머신·액션에 이식돼 있다(과거 store에서 옮김). 정전 발화 규칙·귀속 가드는 policy/ux/analytics-privacy.md.

- `sign_in` — `frontend/src/app/model/session.machine.ts`의 `syncSession` 액션. uid가 새로 확인되는 순간 1회(신규 사인인·세션 복원·계정 전환). 같은 uid 이벤트(TOKEN_REFRESHED)는 통과. 계정 A→B 직접 전환 시 `identify` 전에 `resetAnalyticsIdentity()`로 먼저 끊는다.
- `universe_loaded` — 렌더러 쪽은 `frontend/src/widgets/universe-canvas/ui/UniverseCanvas.tsx`와 `widgets/universe-canvas/ui/overlay/UniverseOverlay.tsx`의 `onCreated`에서 `reportUniverseRenderer(rendererBackend(gl))`. 데이터 쪽은 `frontend/src/pages/home/ui/HomePage.tsx`에서 `applyUniverse` 직후 `reportUniverseData(...)`.
- `record_memory` — `frontend/src/features/record-memory/model/compose.machine.ts`의 `submitting` 상태. `onDone`은 `captureSuccess`(success=true), `onError`는 `captureFail`(success=false). 속성은 `recordMetric`이 만든다: `mood: 'auto'`(조각 단위 감정이 도입되며 상수로 고정), `fragment_count`(조각 수, 21), `body_length_bucket`, `success`. captureSuccess는 resetDraft보다 먼저 실행돼야 한다 — v5는 같은 전이의 assign 갱신이 이후 액션에 보이므로 순서를 바꾸면 fragment_count=0을 읽는다.
- `recall_open` — `frontend/src/features/recall/ui/MemoryPanel.tsx`. ≥2s active dwell 후 touch 직전. `is_dormant`는 발화 직전 별의 `lastRecalledAt`을 `isDormant(..., virtualNowMs())`로 판정.
- `reinforce_flush` — `frontend/src/features/recall/model/recall-flush.machine.ts`의 `captureFlush` 액션. 강화 배치 전송 성공 시 1회(`pair_count = inFlight.items.length`). 재시도는 같은 batchId로 합쳐져 성공 1회만 잡힌다. captureFlush는 rotateBatch 전에 호출해야 inFlight가 살아 있다.
- `dormant_visit` — 레거시 이벤트명. 잠든 별 탐색이 망원경 별 탭으로 흡수된 뒤 별도 시트 발화 지점은 제거됐다. 다시 계측할 경우 `StarExplorerList`의 dormant 필터 노출/선택 기준으로 재정의한다.
- `appearance_switch` — `frontend/src/features/switch-appearance/ui/AppearanceSwitcher.tsx`의 테마 SwatchRadioGroup `onChange`. **다른** 테마 칩 선택만(같은 테마 재클릭·오브제 전환은 세지 않는다). `theme = id`.

### autocapture 이중 가드 (`ph-no-capture`)

`mask_all_text`가 텍스트를 전면 마스킹하지만, 일기 본문이 보이는 요소는 클래스 `ph-no-capture`로 서브트리 자체를 autocapture에서 제외한다(이중 가드). 적용 지점: `features/record-memory/ui/MemoryForm.tsx`(본문 입력·조각 textarea), `features/recall/ui/MemoryPanel.tsx`(원문 article), `features/evolution/ui/EvolutionPanel.tsx`(원문 article), `features/diary-list/ui/DiaryCard.tsx`·`DiarySheet.tsx`(발췌), `features/send-star/ui/SendStarModal.tsx`·`StarGiftsButton.tsx`(메시지), `pages/gift/ui/GiftPage.tsx`(본문·메시지).

## 현재 구현 — FE Sentry 강화

`frontend/src/main.tsx`가 `Sentry.init`을 소유한다(원래는 DSN·environment만 넘기던 것을 18이 강화).

- `VITE_SENTRY_DSN`이 없으면 init을 스킵한다 — 로컬/dev·키리스 흐름 무영향.
- `environment` = `VITE_SENTRY_ENVIRONMENT` ?? `import.meta.env.MODE`.
- `release` = `VITE_APP_VERSION` 또는 undefined. `VITE_APP_VERSION`은 `frontend/vite.config.ts`가 빌드 시 주입한다: 명시 값이 없으면 CF Workers 빌드의 `process.env.WORKERS_CI_COMMIT_SHA`를 쓰고, 로컬 빌드는 빈 문자열 → release 태깅 생략. `define`으로 `import.meta.env.VITE_APP_VERSION`에 박는다.
- `integrations: [Sentry.browserTracingIntegration()]`, `tracesSampleRate: 0.1`(페이지 로드/네비게이션 성능만 낮게).
- `tracePropagationTargets`는 기본값(동일 출처) 유지 — 교차 출처 API(api.<도메인>)에 sentry-trace 헤더를 붙이면 백엔드 CORS allow-headers에 없어 프리플라이트가 깨진다.
- Session Replay는 도입하지 않았다(스펙의 선택 항목). 도입 시 텍스트·입력 마스킹 기본이 전제(3.5).

## 현재 구현 — BE Sentry 훅

BE Sentry 배선은 spec 14(DSN 운영)·17(recover 핸들러)이 소유하고, 18은 release·environment 태깅과 패닉 캡처 경로를 보강한다. 코드는 `backend/cmd/api/main.go`.

- `cfg.SentryDSN`이 비면 `sentry.Init`을 통째로 스킵한다(로컬/dev·테스트 무영향). 키는 `SENTRY_DSN`·`SENTRY_ENVIRONMENT`(VITE 접두사 없음, `backend/internal/platform/config/config.go`). 기본 environment는 `development`.
- `sentry.Init`: `Dsn=cfg.SentryDSN`, `Environment=cfg.SentryEnvironment`, `Release=version`. 여기서 `version`은 `cmd/api`의 하드코딩 상수 `"0.0.1"`다 — FE의 커밋 SHA release와 다르다(BE는 빌드 상수, FE는 `VITE_APP_VERSION`).
- 초기화 성공 시 `defer sentry.Flush(2s)` + `defer sentry.Recover()`.
- RPC 패닉은 connect 안에서 recover 핸들러(`backend/internal/platform/rpcserver/recover.go`, 17)가 잡으므로 sentryhttp 래프로 전파되지 않는다. 대신 `PanicCapture` 훅으로 Sentry에 닿는다: `main.go`가 `sentryEnabled`일 때 `func(ctx, _, p)`를 주입하고, 핸들러는 request-scoped hub(`sentry.GetHubFromContext`, 없으면 `CurrentHub`)에 `RecoverWithContext(ctx, p)`. 훅이 nil이면 log-only(rpcserver는 Sentry를 import하지 않는 infra-only 유지).
- `sentryhttp.New({ Repanic: true }).Handle`로 서버 핸들러를 감싼다 — request-scoped hub를 붙이고(캡처 훅이 읽음), `/health` 등 비 RPC mux 핸들러의 패닉을 잡는다. Repanic으로 net/http가 그쪽엔 500을 돌려준다.

## 현재 구현 — BE 큐 요약 로그

임베딩 워커가 큐 백로그를 주기 로그로 가시화한다. 코드: `backend/internal/job/worker.go`.

- `Worker.Run`은 매 루프 `time.Since(lastSummary) >= summaryInterval`이면 `logQueueSummary`를 호출한다. `lastSummary` 제로값으로 기동 직후 1회도 즉시 찍힌다 — 배포 후 첫 로그가 이전 실행이 남긴 백로그를 바로 보여준다. `summaryInterval` 기본은 5분(`defaultSummaryInterval = 5 * time.Minute`).
- `logQueueSummary`는 `w.jobs.Stats(ctx)`를 읽어 slog Info 한 줄: `queue summary pending=N due_pending=N running=N failed=N oldest_pending=<dur>`. `oldest_pending`은 초 단위로 절삭(`Truncate(time.Second)`).
- 통계 조회 실패는 ctx가 살아 있을 때만 Warn 1줄 남기고 반환한다 — 관측이 파이프라인을 막지 않는다.
- 카운트는 `JobQueueStats`(`backend/internal/db/queries/job.sql`)가 단일 쿼리로 집계한다: `pending`(status='pending' 전체) · `due_pending`(`status='pending' AND next_run_at <= now()`, ClaimJob과 같은 처리 가능 기준) · `running` · `failed` · `oldest_pending_seconds`(가장 오래된 pending의 `now() - created_at`, 없으면 0). pending 전체에는 backoff 대기 재시도가 섞여 있어 due_pending을 따로 센다.
- `repository_pg.Stats`가 `oldest_pending_seconds`(float)를 `time.Duration`으로 환산해 도메인 `QueueStats`(`backend/internal/job/repository.go`)로 매핑한다.

## Flutter 동등성 기준

- 분석은 제품 행동 지표라 웹에 한정되지 않는다. Flutter도 같은 7개 이벤트(`EVENTS`)와 닫힌 속성 모양·공통 `demo`를 재현해야 베타 퍼널을 합칠 수 있다.
- 프라이버시 헌법은 플랫폼 무관 불변이다: 일기 body·임베딩·내용을 어떤 이벤트에도 싣지 않는다(길이 버킷·mood까지만), 식별자는 Supabase uid만, 로그아웃·계정 전환 시 reset.
- `universe_loaded`의 `renderer`는 웹의 WebGPU/WebGL2 폴백 측정 신호다 — Flutter는 자체 렌더 백엔드 식별자로 대체하면 된다.
- `record_memory`는 서버 왕복 성공/실패를 잡고, 로컬 검증 차단은 세지 않는다.
- BE 큐 요약 로그·Sentry는 서버 측이라 클라이언트 종류와 무관하다(서버 한 곳이 모두 본다).

## 수용 기준

1. `VITE_POSTHOG_KEY`가 없으면 PostHog는 초기화조차 되지 않고 모든 capture/identify가 no-op이다(로컬·체험 무영향).
2. `VITE_SENTRY_DSN`이 없으면 FE Sentry init을 스킵한다.
3. 일기 작성 시 `record_memory`를 보내되 body 원문·일부를 포함하지 않는다 — `mood`(상수 'auto')·`fragment_count`·`body_length_bucket`·`success`만.
4. 우주 첫 렌더 시 `universe_loaded`는 실제 선택된 렌더러 백엔드(webgpu/webgl2)와 카운트·`load_ms`를 페이지 로드당 1회만 기록한다(렌더러·데이터 두 비동기 신호 합류).
5. `sign_in`은 uid가 새로 확인될 때만 발화하고, 같은 uid 토큰 갱신에는 발화하지 않는다.
6. `reinforce_flush`는 강화 배치 전송 성공 시 1회 발화한다(재시도 이중 가산 없음).
7. `dormant_visit`은 잠든 별 목록 첫 도착 시 방문당 1회 발화한다.
8. `appearance_switch`는 다른 테마 선택에만 발화하고 같은 테마 재클릭에는 발화하지 않는다.
9. 모든 capture는 공통 속성 `demo`(체험 모드 여부)를 갖는다.
10. 식별자는 Supabase uid만 보내고, 사인아웃·계정 전환 시 `posthog.reset()`으로 식별을 끊는다.
11. autocapture는 일기 입력·원문 영역을 줍지 않는다(`mask_all_text` + `ph-no-capture` 이중 가드).
12. FE 프로덕션 빌드에서 캡처된 에러는 release(`VITE_APP_VERSION`=커밋 SHA)·environment 태그를 갖는다.
13. BE Sentry가 켜지면 RPC 패닉이 `PanicCapture` 훅을 통해 release·environment 태그와 함께 도착한다.
14. 워커는 5분 주기(+기동 직후 1회)로 큐 상태(pending/due_pending/running/failed/oldest_pending) 요약을 구조화 로그 1줄로 남긴다.
