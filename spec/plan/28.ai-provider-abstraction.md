# 28. AI provider abstraction

> AI provider abstraction — the provider-agnostic LLM/embedding **client seam inside `internal/ai`**: capability
> interfaces (`LLMClient` with schema-forced structured output, `EmbeddingClient`) that the plan-22 port adapters
> consume, with one concrete provider adapter per vendor (Anthropic Claude, OpenAI, DeepSeek, Z.ai, Gemini; Voyage for
> embeddings). **Epic A implements exactly one provider per capability**; the rest are slots.
> Scope: BE | Status: built
>
> Layer: the outermost ring. This unit subdivides the [22.ai-worker-pipeline](22.ai-worker-pipeline.md)
> external-service wrapper into two insulation levels: _task_ knowledge (the port adapters — prompts, domain mapping,
> output schemas) and _vendor_ knowledge (the provider clients — SDK/HTTP, auth, native structured-output mechanics).

## Purpose

The product must not be coupled to a single AI vendor. Model quality, pricing, and availability shift fast, and the
concrete provider is an operational choice, not a product one — the user decision (2026-07) is that the system
**supports Anthropic Claude, OpenAI, DeepSeek, Z.ai, and Gemini interchangeably** (plus a dedicated embedding
provider such as Voyage AI), while **implementing only the one provider actually used today**.

Plan 22 already insulates the domain: `internal/memory` consumes the `Extractor`/`Embedder`/`Semanticizer` ports and
never sees an SDK. But without this unit, the _adapters_ behind those ports would each hardcode one vendor SDK, and
swapping providers would mean rewriting the prompts, the schema-forcing plumbing, and the domain mapping along with
the transport. This unit splits the wrapper so that the expensive, product-shaped knowledge (what to ask the model,
what shape it must return, how to map it to domain DTOs) is written **once**, provider-neutral, and only the thin
vendor client varies.

ARCHITECTURE §2.1 says abstractions are **earned**, not speculative. This one is earned by an explicit product
requirement — multi-provider support was decided by the user, not inferred — and it is deliberately _narrow_: two
small interfaces, one implemented adapter each, and named slots for the rest.

## Scope / Non-goals

**In scope**

- **Two capability interfaces inside `internal/ai`**, declared next to their consumers (the plan-22 port adapters —
  consumer-declared, ARCHITECTURE §2.4):
  - **`LLMClient`** — one completion call: request carries messages/instructions **plus an output JSON schema**;
    the response is the schema-conforming structured value (or a typed error). Structured output is **first-class in
    the contract**, because the Extractor's schema-forcing ([W4a]) must survive any provider swap.
  - **`EmbeddingClient`** — `texts → vectors` of dimension `ai.embedding_dim` (plan 16).
- **Per-provider client adapters** implementing those interfaces. Supported slots: **Anthropic Claude, OpenAI,
  DeepSeek, Z.ai, Gemini** for `LLMClient`; **Voyage AI, OpenAI, Gemini** for `EmbeddingClient`. Each adapter owns its
  SDK/HTTP plumbing, auth, and the mapping from the generic structured-output request to its **native mechanism**
  (Anthropic tool-use/structured output, OpenAI `response_format` JSON schema, Gemini `responseSchema`, …).
- **Epic A implements exactly one concrete adapter per capability.** Default: **Anthropic** (latest Claude) for
  `LLMClient`, **Voyage AI** (Anthropic's recommended embedding partner; Anthropic has no embedding API) for
  `EmbeddingClient` — exact model ids confirmed at implement. Every other provider is a **named slot**: adding it
  later touches only a new adapter + the factory, never a consumer.
- **Selection at the composition root by runtime env** (e.g. provider name + API key per capability): `cmd/api` /
  `cmd/worker` call an `internal/ai` factory; a missing key selects the plan-22 **keyless deterministic mock**; an
  unknown provider name is a **startup error**, never a silent default.
- **Error normalization**: vendor errors are mapped at the client adapter into a small set of typed errors —
  rate-limited · auth-failed · cost-capped · malformed-structured-output — so plan 22's metering, job backoff, and RPC
  error mapping are provider-independent.
- **Metering at the seam**: plan 22's cost metering (`ai.per_call_token_cap`, `ai.daily_call_cap`, caching) wraps the
  generic client interfaces, so the caps apply uniformly to every provider.

**Non-goals**

- **No change to the memory-owned ports.** `Extractor`/`Embedder`/`Semanticizer`, their prompts, the domain DTO
  mapping, retries-on-schema-mismatch, and the keyless mock are [22.ai-worker-pipeline](22.ai-worker-pipeline.md);
  `internal/memory` is untouched by anything here (two-level insulation: domain ↛ port adapter ↛ vendor).
- **No per-provider prompt forks.** Prompts and output schemas live once in the port adapters and are
  provider-neutral; a provider that cannot honor the structured-output contract is not a valid slot.
- **Not implementing every provider.** One concrete adapter per capability in Epic A; the other slots are compile-time
  seams, added when operationally needed.
- **No provider fallback chains, A/B routing, load balancing, or multi-provider budgets** — a later tightening if
  earned; Epic A picks one provider per capability at startup.
- **No `spec/values.yaml` keys.** Provider names, model ids, API keys, and base URLs are **runtime env/secrets**
  (plan 22's stance, same as plans 05/10); switching providers is an ops action, not a tuning change.
- **No FE surface** and no new tables/RPCs.

## Design

### Two insulation levels inside `internal/ai`

Plan 22's "real adapter" becomes two layers:

1. **Port adapters** (plan 22, unchanged in role) — implement the memory-owned `Extractor`/`Embedder`/`Semanticizer`.
   They own **task knowledge**: the prompts, the output JSON schemas ([W4a]), the retry-on-schema-mismatch policy,
   and the mapping to/from the memory context's DTOs. They call the capability interfaces below and contain **no**
   vendor SDK import.
2. **Provider clients** (this unit) — implement `LLMClient` / `EmbeddingClient` for one vendor each. They own
   **vendor knowledge**: SDK/HTTP transport, auth, model-id plumbing, the native structured-output mechanism, and
   error normalization. They contain **no** prompt text, no domain DTOs, no knowledge of what the call is _for_.

The test for a misplaced line: if a provider client mentions a neuron, a mood, or a prompt sentence, it belongs in a
port adapter; if a port adapter mentions a vendor SDK type, an HTTP header, or a model id, it belongs in a provider
client.

### The capability contracts

- **`LLMClient`** — a single completion operation. The request carries: the instruction/message content (built by the
  port adapter), an **output JSON schema** the response must conform to, and per-call bounds (max output tokens from
  `ai.per_call_token_cap`). The response is the parsed, schema-conforming value. Every provider adapter realizes the
  schema constraint through its **strongest native mechanism** (tool-use / `response_format` / `responseSchema`);
  a response that fails schema validation surfaces as the typed `malformed-structured-output` error — the _retry_
  decision belongs to the port adapter (plan 22 / plan 20 caps), not the client.
- **`EmbeddingClient`** — `Embed(ctx, texts) → vectors`, each of dimension **`ai.embedding_dim`** exactly. A
  provider/model combination that cannot produce that dimension (natively or via its output-dimension parameter)
  **fails at startup wiring**, not at row-insert time.

Both interfaces are deliberately minimal — no streaming (forbidden anyway, ARCHITECTURE §2.9 #9), no tool-calling
surface, no chat-session state. A capability the product later needs is added to the interface when it is earned.

### Provider matrix

| Capability        | Slots (supported by contract)                        | Implemented in Epic A                                                                                              |
| ----------------- | ---------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| `LLMClient`       | Anthropic Claude · OpenAI · DeepSeek · Z.ai · Gemini | **Anthropic** — `claude-opus-4-8` (official `anthropic-sdk-go`; structured output via `output_config` json_schema) |
| `EmbeddingClient` | Voyage AI · OpenAI · Gemini                          | **Voyage AI** — `voyage-3.5` (plain HTTP; no official Go SDK)                                                      |

"Slot" means: the contract, error taxonomy, and factory registry are designed so the adapter drops in without touching
any consumer. Only the implemented column has code in Epic A; a recognized-but-unimplemented slot name fails at startup,
never silently defaulting. Model ids are defaults, overridable per deployment via env.

### Selection & wiring

The composition roots (`cmd/api`, `cmd/worker`) read runtime env — a provider name and an API key **per capability**
(LLM and embedding are selected independently; e.g. Claude + Voyage). The env vars are `COSIMOSI_LLM_PROVIDER` /
`COSIMOSI_LLM_API_KEY` (+ optional `COSIMOSI_LLM_MODEL` / `COSIMOSI_LLM_BASE_URL`) and the `COSIMOSI_EMBEDDING_*`
equivalents. An `internal/ai` factory maps `(provider, key) → client`:

- key present + known provider → that provider's client, wrapped in the metering layer;
- key absent → the plan-22 **keyless deterministic mock** for that capability (dev/CI/tests);
- unknown or recognized-but-unimplemented provider name → **startup error** (fail fast; never a silent default).

The factory uses a **driver registry**: each provider subpackage registers its constructor from `init()`
(`ai.RegisterLLMProvider` / `ai.RegisterEmbeddingProvider`), and `cmd/*` blank-imports the subpackages it wants
available — so `internal/ai` never imports a vendor subpackage (no import cycle). Adding a provider is a new subpackage
plus one blank import, no consumer change.

Nothing in `internal/memory` or in the port adapters changes when the env changes — provider choice is invisible
above the seam.

### Error normalization & metering

Each provider client maps its vendor's failure modes into the shared typed errors: **rate-limited** (retryable —
feeds the jobs backoff), **auth-failed** (terminal — surfaces to ops), **cost-capped** (plan 22's typed cost-limit
error when metering trips), **malformed-structured-output** (schema violation — port adapter decides retry). The
metering wrapper (plan 22's caps + caching) decorates the generic interfaces, so per-call/daily caps and
identical-input caching behave identically for every provider. Caching is keyed by user + the port adapter's content
hash; a response the port adapter rejects (via an optional `Validate` hook it passes on the request) is not cached, so
an identical retry can re-sample rather than being served a poisoned entry.

### Placement (Go, ARCHITECTURE §2)

All of it stays inside the `internal/ai` supporting wrapper — this unit adds **no** new context. The capability
interfaces are declared in `internal/ai` beside the port adapters that consume them; each provider client lives in
its own subpackage (e.g. `internal/ai/anthropic`, `internal/ai/voyage`) so vendor SDK dependencies stay confined
per-package. The exact file split beyond that is earned at implement (§2.3). `cmd/*` remains the only place that
sees the factory.

## Acceptance Criteria

1. `LLMClient` and `EmbeddingClient` capability interfaces exist in `internal/ai`, **declared next to their consumers**
   (the plan-22 port adapters); the port adapters import **no vendor SDK type**, and no provider client imports a
   memory domain type, a prompt string, or a port DTO [§2.2/§2.4].
2. The `LLMClient` request carries an **output JSON schema** and every provider adapter enforces it via its native
   structured-output mechanism — the schema-forced Extractor shape ([W4a]) survives a provider swap with **zero**
   change to port adapters or `internal/memory`.
3. **Exactly one** concrete provider per capability is implemented in Epic A — Anthropic (`LLMClient`) and Voyage AI
   (`EmbeddingClient`), exact model ids confirmed at implement — and the remaining slots (OpenAI, DeepSeek, Z.ai,
   Gemini; OpenAI/Gemini embeddings) require only a new adapter + factory case, no consumer change.
4. Provider selection is composition-root wiring from runtime env, **independently per capability**: key present →
   that provider (metered); key absent → the keyless deterministic mock; unknown provider name → **startup error**,
   never a silent default.
5. Vendor errors normalize to the shared typed set — rate-limited · auth-failed · cost-capped ·
   malformed-structured-output — and plan 22's job backoff / RPC error mapping consume only the typed set (no vendor
   error type escapes `internal/ai`).
6. Cost metering (`ai.per_call_token_cap`, `ai.daily_call_cap`, identical-input caching) wraps the capability
   interfaces and applies **uniformly to every provider**; the mock is unmetered.
7. Every `EmbeddingClient` yields vectors of dimension `ai.embedding_dim`; a provider/model that cannot honor it
   fails at startup wiring, not at insert time.
8. No `spec/values.yaml` key is introduced: provider names, model ids, API keys, and base URLs are runtime
   env/secrets only.
