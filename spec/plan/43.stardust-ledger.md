## Purpose

Recall in cosimosi is **never free** — every 회고(재공고화) and every 요지 별 열람 spends Twinkle ([G1]). But the design
is deliberately _not_ a paywall on remembering: a daily-reset **basic** allowance carries everyday 곱씹기 ([M5][G5]), and
only _excess_ recall reaches for **additional** (charged, permanent) Twinkle. This unit owns the domain that makes that
economy honest and reproducible — **what a balance is, how it is spent, and what a given recall costs** — and it must
hold three things in one place:

1. **Two tiers, different lifetimes** ([G2]). A fixed daily **basic** grant that _resets_ (no carry-over); a permanent
   **additional** balance that _accumulates_ from charges ([G3]). A balance is a pair — and `basic` is a _derivation_ of
   the daily grant against "now," not a stored counter.
2. **A fixed spend order** ([G2]): draw **basic first, additional only for overflow** — so daily recall never touches
   the paid wallet ([G5]).
3. **Two cost curves** ([G4]). 회고 cost **rises** with decay-depth (a faded memory is harder to pull back — it prices
   the accessibility signal [38.forgetting-cost](38.forgetting-cost.md) computes from [F4]); 요지 별 열람 cost **falls**
   with gist-depth (deeper abstraction = cheaper skim [R8], pricing the `semantic_stage` signal
   [40.semanticization](40.semanticization.md) computes).

Balance is **server-authoritative**, like the universe clock ([29.universe-clock](29.universe-clock.md)): the server is
the single ledger writer; the FE only _reads_ and _displays_ it. The cost **curves** are golden-parity pure math so the
FE can price a recall pre-spend without a round-trip — the same function on both sides, pinned by fixtures (like
[18.synapse-plasticity](18.synapse-plasticity.md)).

## Scope / Non-goals

**In scope**

- **The `internal/twinkle` context package** (NEW — CC8, domain-first Go): the Twinkle aggregate types (`Balance`,
  `LedgerEntry`, tier enum), its pure domain functions, and — in [44.earn-spend-usecase](44.earn-spend-usecase.md) — its
  use-cases and RPC surface. `twinkle` **never imports** `memory`; memory calls twinkle's spend through a
  consumer-owned port at the composition root (CC2/CC8).
- **The `twinkle_ledger` table** (goose migration + `twinkle/pg` repository): one **balance row per user** holding the
  additional (permanent) balance and the basic-reset anchor, **plus** an append-only **earn/spend log** for auditability
  and idempotency. As built: a _balance row + event log_, not a pure event-sourced ledger — see Design.
- **The balance domain model + derivation** ([G2]): `Balance = { basic, additional }`, where `basic` is _derived_ from
  the daily grant and the last-reset anchor against "now," and `additional` is the stored accumulated permanent balance.
- **The spend-order rule** ([G2]): a pure `PlanSpend(balance, cost) → { fromBasic, fromAdditional, ok }` that draws
  basic first, additional for the overflow, and reports insufficiency (never goes negative).
- **The two cost curves** (pure, golden-parity):
  - `RecallCost(accessibilityCost) → twinkle` — 회고 cost **non-decreasing** in decay-depth ([F4][G4]), consuming the
    accessibility/cost-weight [38.forgetting-cost](38.forgetting-cost.md) exposes (CC3).
  - `GistViewCost(semanticStage) → twinkle` — 요지 별 열람 cost **non-increasing** in gist-depth ([R8][G4]), consuming
    the `semantic_stage` signal [40.semanticization](40.semanticization.md) computes.
- **The basic-reset cadence** ([G2] "매일"): **real calendar day (UTC), distinct from universe time** — see Design.
- **Golden-parity stance:** the cost curves + `PlanSpend` mirror TS↔Go so the FE prices pre-spend; the _balance itself_
  is server-authoritative (server-only writer), not a mirror.
- Enumerating every `twinkle.*` value key (basic daily amount, cost-curve coefficients, tier caps) — see Policy /
  Values Impact for the shipped numbers.

**Non-goals**

- **No earn/spend orchestration, no charge paths, no `SpendGate` binding.** _When_ Twinkle is earned (write / invite /

## Design

### Placement — a NEW `internal/twinkle` context (domain-first Go, CC8)

The Twinkle economy is **not** the memory aggregate's behavior: it has its own aggregate (`Balance`), lifecycle (daily
reset, charge, spend), and persistence (`twinkle_ledger`), and the PRD frames it as a separable concern (§5.9). Per the
be-architecture procedure and ARCHITECTURE §2.2, that makes it a **distinct core context**, `internal/twinkle`, exposing
a `twinkle.v1` service ([44.earn-spend-usecase](44.earn-spend-usecase.md) adds `GetBalance` + the earn/charge RPCs). The
dependency direction is **one-way**: `memory` (recall/gist-view) calls twinkle's spend through a **consumer-owned
`SpendGate` port** wired at the composition root; **`twinkle` never imports `memory`** (CC2, CC8). This unit ships the
pure domain (types + balance/spend/cost functions) in the context package and the `twinkle_ledger` table + row↔domain
mapping in `twinkle/pg` (§2.6, the only package touching sqlc/pgx). The **repository port** is consumer-owned (declared
by the plan-44 use-case, §2.4), so this unit provides the concrete `twinkle/pg` store + queries, not a pre-declared
interface (matching plans 16/29).

### The balance model — two tiers, one derived ([G2])

A balance is a pair, not a single number:

```
Balance = { basic int, additional int }          // both in Twinkle units (whole numbers)
```

- **`additional`** — the **permanent, carrying** balance ([G2]). Accumulated from charges (payment / invite / write
  [G3]) and decremented by spend overflow. A **stored counter** on the ledger row; the _earn_ side (plan 44) is the
  only writer that increases it.
- **`basic`** — the **daily-reset, non-carrying** allowance ([G2]). **Not a stored counter** — it is a **derivation**
  from three facts: the daily grant `twinkle.basic_daily_amount`, the amount already spent from basic _within the
  current reset window_, and whether the window rolled over since the last write:

  ```
  basicRemaining(now) =
      if resetWindowOf(now) > last_basic_reset_window:   basic_daily_amount            // fresh day → full grant, no carry
      else:                                              basic_daily_amount − basic_spent_this_window
  ```

  When the reset window advances, the previous window's unspent basic is **discarded** (does not carry — [G2]) and
  `basic_spent_this_window` conceptually resets to 0. This is realized by storing the window anchor + the in-window basic
  spend and computing `basicRemaining` at read (see the table), exactly as the clock stores an anchor and derives
  elapsed. Basic is a _lazy_ derivation (no scheduled reset job — matching [T4]'s no-cron
  discipline). The reset "happens" the first time the balance is read or spent in a new window; the row's window anchor
  is rolled forward on that first write. No background sweep.

### The reset cadence — real calendar day, NOT universe time ([G2])

[G2] says basic Twinkle is granted "매일" (daily). This is a **real calendar day (UTC), and it is
deliberately distinct from universe time.** The rationale, stated explicitly because the engine otherwise counts
everything in universe-days:

- The economy's job is to pace the _user's real-world habit_ of everyday recall ([M5][G5]) — a daily wallet that refills
  each real day. Universe time only advances when the user launches or recalls ([T2][T3]); tying the basic refill to it
  would mean a user who only _views_ (which spends gist-Twinkle [G1]) never refills, breaking [G5].
- So the reset window is the **UTC calendar day** (`resetWindowOf(now) = date(now, UTC)`), matching the `ai.daily_call_cap`
  UTC-calendar-day convention already in the codebase ([22.ai-worker-pipeline](22.ai-worker-pipeline.md)) — one
  established "real daily" window, reused for coherence. Universe time ([29.universe-clock](29.universe-clock.md)) stays
  the "now" for [F]/[C]/[L] read-time math; the Twinkle basic reset is the **one place** the economy reads real time.

This is the single crossing between real time and the otherwise diary-driven engine, and it is intentional and isolated
to `internal/twinkle`.

### The spend order — basic first, additional for overflow ([G2][G5])

A pure function decides how a cost is drawn, protecting the core loop:

```
PlanSpend(basicRemaining int, additional int, cost int)
    → { fromBasic int, fromAdditional int, ok bool }
```

- `fromBasic  = min(cost, basicRemaining)`
- `fromAdditional = cost − fromBasic`
- `ok = fromAdditional ≤ additional` // enough only if the overflow fits in the permanent balance

The rule is **basic is exhausted before additional is touched** ([G2]) — so everyday recall inside the daily grant never
spends the paid wallet ([G5]). Neither tier goes negative (`ok = false` → the use-case rejects/charges, plan 44; this
function only _plans_ the draw). `PlanSpend` is pure and golden-parity so the FE can show, pre-spend, exactly which tier
a recall will draw from ([45.stardust-ui](45.stardust-ui.md)).

### The two cost curves (pure, golden-parity) ([G4])

Both curves are **pure price functions** — deterministic, IO-free, mirrored TS↔Go, reading only their argument and the
generated `twinkle.*` coefficients (the [18.synapse-plasticity](18.synapse-plasticity.md) pattern). They return a
non-negative whole-number Twinkle price. The _shapes_ are formulas (code, excluded from `values.yaml`); only the
coefficients are values.

**회고 (재공고화) cost — rises with decay-depth ([F4][G4]):**

```
RecallCost(accessibilityCost float) → int
```

The `accessibilityCost` is the accessibility/cost-weight [38.forgetting-cost](38.forgetting-cost.md) exposes (CC3) — a
deeper-decayed memory yields a larger weight (harder to pull back [F4]). `RecallCost` maps it to Twinkle: a **base**
price plus a **decay-depth term**, `round(twinkle.recall_base_cost + twinkle.recall_depth_coefficient · accessibilityCost)`,
clamped to `twinkle.recall_max_cost`. **Non-decreasing** in decay-depth (the acceptance criterion pins the behavior, not
the shape). Recovery-on-recall ([F5]) is not priced here — it is the decay side lowering the _next_ recall's
accessibilityCost; this curve simply reads whatever accessibility the memory currently has. As built,
`RecallCost` takes the _accessibility signal_, not raw decay-stage, so D owns "how decayed → how inaccessible" and F owns
only "how inaccessible → how many Twinkle" — the CC3 split with no price constant leaking into D.

**요지 별 열람 cost — falls with gist-depth ([R8][G4]):**

```
GistViewCost(semanticStage int) → int   // semanticStage 0..4, from plan 40 / the schema
```

The deeper the gist (higher `semantic_stage`), the **cheaper** to skim ([R8] — "요지화 단계 깊을수록 저렴"). Maps stage to
Twinkle as a **non-increasing** function: `twinkle.gist_base_cost` at stage 1, decreasing by
`twinkle.gist_stage_discount` per stage, floored at `twinkle.gist_min_cost` (so it is cheap but never free — free is the
meta/forgotten-text surface [G1], not a gist read). Stage 0 (not yet gistified) has no gist star to view, so the curve
is defined over the gistified stages (1..4). **Non-increasing** in `semanticStage` is the pinned behavior.

**The diary-sum price ([D3][G4]).** "이 일기로 태어난 별 보기" costs the **sum of `RecallCost`** over the diary's episodic
stars ([47.diary-reader-page](47.diary-reader-page.md) consumes this — the whole-recall price is derived by summing this
curve per star, not a new curve). Reading the original diary is **free** (archive [G1]).

### The `twinkle_ledger` table (goose migration)

One forward-only-safe goose migration (with a matching down) under `apps/api/db/migrations/`, owned by this unit (per
[05.data-persistence](05.data-persistence.md): the unit adding a table owns its migration). It introduces the new
context's storage without touching any memory table.

**A balance row + an append-only event log**, not a pure event-sourced ledger. Rationale: the balance
read is on the hot path (every universe/HUD load), so a single authoritative row that answers "current balance" in one
lookup beats folding an event stream on every read; the append-only `twinkle_ledger_entries` log preserves auditability
([I1] spirit — nothing is destroyed), supports **idempotent** charges/spends (via a dedup key so a retried spend does
not double-charge), and lets the balance row be _reconstructed_ from the log if ever needed. The balance row is the
authoritative state; the log is history.

```sql
-- twinkle_balances — one authoritative balance row per user. `additional` is the permanent (carrying) balance [G2];
-- basic is DERIVED (not stored as a live counter) from the daily grant + the in-window basic spend + the reset anchor.
-- Server-authoritative single-writer state (like universe_state) — the FE reads it, never writes it ([I5] governs
-- positions, not this).
CREATE TABLE twinkle_balances (
    user_id                  TEXT PRIMARY KEY,            -- one balance per user
    additional               INT  NOT NULL DEFAULT 0,     -- permanent, carrying balance [G2]; CHECK (>= 0)
    basic_spent_this_window  INT  NOT NULL DEFAULT 0,     -- basic Twinkle spent within the current reset window
    basic_reset_window       DATE NOT NULL,               -- UTC calendar day of the current basic window (reset anchor)
    updated_at               TIMESTAMPTZ NOT NULL DEFAULT now(),
    CHECK (additional >= 0),
    CHECK (basic_spent_this_window >= 0)
);

-- twinkle_ledger_entries — append-only earn/spend log ([I1] spirit): auditability, idempotency, reconstruction.
-- Never UPDATEd or DELETEd by the system. `dedup_key` makes a retried earn/spend idempotent.
CREATE TABLE twinkle_ledger_entries (
    id             TEXT PRIMARY KEY,
    user_id        TEXT NOT NULL,
    kind           TEXT NOT NULL,        -- 'earn' | 'spend'
    reason         TEXT NOT NULL,        -- 'daily_reset' unused (derived) | 'payment' | 'invite' | 'write_diary'
                                         -- | 'recall' | 'gist_view' — the earn/spend source [G3][G1]
    amount         INT  NOT NULL,        -- always positive; kind gives the sign
    from_basic     INT  NOT NULL DEFAULT 0,  -- for spends: portion drawn from basic [G2]
    from_additional INT NOT NULL DEFAULT 0,  -- for spends: portion drawn from additional [G2]
    dedup_key      TEXT,                 -- idempotency key for retried earn/spend (NULL = not deduped)
    created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (user_id, dedup_key),
    CHECK (amount > 0),                  -- reconstruction invariants: the log stays foldable
    CHECK (from_basic >= 0),             -- back into the balance row
    CHECK (from_additional >= 0),
    CHECK (kind <> 'spend' OR amount = from_basic + from_additional)
);
```

Notes:

- **`user_id` PRIMARY KEY on `twinkle_balances`** — one row per user; the spend/charge is a single-row upsert serialized
  by the row lock inside the use-case transaction (plan 44), so concurrent spends can't oversell (the [29.universe-clock](29.universe-clock.md)
  single-writer pattern).
- **`basic` is not a column** — it is derived at read from `basic_reset_window`, `basic_spent_this_window`, and `now`
  (the derivation above). The reset is lazy: a read/spend in a new window recomputes with a full grant and rolls the
  anchor forward on the next write. No stored "basic balance" to drift.
- **Append-only log, no system delete** ([I1]): `twinkle_ledger_entries` is never `UPDATE`d or `DELETE`d by the system;
  it is the auditable history and the idempotency guard (`UNIQUE (user_id, dedup_key)`).
- **Per-user isolation**: both tables carry `user_id`, every query scopes by it ([04.auth-session](04.auth-session.md)/
  [05.data-persistence](05.data-persistence.md) `UserScope`, enforced by `pnpm lint:persistence`).
- The `basic_daily_amount`, the cost-curve coefficients, and any tier caps are `values.yaml` — **not** hardcoded in the
  migration or the domain (the grant literal lives in `twinkle.basic_daily_amount`, not the DDL default).

### sqlc wiring & queries

Query files under `apps/api/db/queries/twinkle/**`, generated into `dbgen` via `pnpm gen:sql`; only `twinkle/pg` imports
`dbgen` (ARCHITECTURE §2.4 #4, §2.6). The minimal surface this unit provides (the use-case in plan 44 adds the earn/spend
transaction on top):

- **`GetBalanceRecord`** — the balance-row read (returns nil when absent → the lazy-birth full-basic default):
  `SELECT ... FROM twinkle_balances WHERE user_id = $1`.
- **`ApplyBalanceDelta`** — one earn/spend against the single balance row, **update-first, then birth-insert, then
  retry-update** (a plain `INSERT … ON CONFLICT DO UPDATE` upsert cannot carry a negative delta: PG evaluates the
  proposed insert tuple's CHECK constraints even when it conflicts into the UPDATE arm). The UPDATE rolls a stale
  `basic_reset_window` forward (a fresh window's spend starts from just the new delta; the anchor never rolls
  backward), and its WHERE carries the **grant guard** — the new `basic_spent_this_window` must stay within
  `twinkle.basic_daily_amount` (passed as a query argument, never a DDL literal) — so a basic draw planned against a
  stale read is refused (`ErrBasicGrantExceeded`), and the `additional` CHECK refuses the paid tier symmetrically. The
  store rejects negative `basicSpentDelta` and any delta/amount outside int32 (`ErrDeltaOutOfRange`).
- **`AppendLedgerEntry`** — the log append with `ON CONFLICT (user_id, dedup_key) DO NOTHING`; returns whether the row
  was written (`false` = already-applied retry). The plan-44 transaction appends **first** and skips the delta on a
  retry — that pairing makes a retried earn/spend idempotent end to end.

Repository interfaces are **not** pre-declared here (consumer-owned by plan 44); `twinkle/pg` returns the concrete store

- queries the use-case binds to.

### Golden-parity stance (which parts mirror, which don't)

Matching the [18.synapse-plasticity](18.synapse-plasticity.md)/[29.universe-clock](29.universe-clock.md) precedents,
this unit is **split**:

- **Golden-parity (TS↔Go, pinned by fixtures):** the pure price/plan functions — `RecallCost`, `GistViewCost`,
  `PlanSpend`, and the `basicRemaining` derivation. They live once in Go (`internal/twinkle`) and once in TS
  (`packages/twinkle-logic`, a new mirror package alongside `packages/memory-logic`), read the same generated
  `twinkle.*` constants, and are pinned equal by `apps/api/internal/twinkle/testdata/stardust-ledger-golden.json` with a
  fixed tolerance. This lets [45.stardust-ui](45.stardust-ui.md) show a correct **pre-spend price** and **which tier
  will pay** without a round-trip — the FE prices, the server enforces.
- **Server-authoritative (NOT a mirror):** the **balance itself**. The ledger is single-writer server state
  (like `universe_state`) — the FE reads the current `Balance` from `GetBalance` (plan 44) and displays it; it never
  advances the balance. The actual spend is committed server-side inside the use-case transaction, so a stale/optimistic
  FE price can never oversell.

### Re-implementation note

The economy is fully specified by: a `{ basic, additional }` balance where `basic` is `basic_daily_amount` minus
in-UTC-day basic spend (no carry) and `additional` is a stored carrying counter; spend order basic→additional; two
monotone price curves (recall rises with accessibility, gist falls with stage); and a balance-row + append-only log with
UTC-day reset and dedup-keyed idempotency. Any client/runtime reproduces the _prices_ from the curves + coefficients; the
_balance_ is always read from the server.

   `UNIQUE (user_id, dedup_key)`; `pnpm db:migrate` applies cleanly and `pnpm db:down` reverses cleanly ([G2][I1]).
2. Both tables carry `user_id`, every query scopes by it, and `pnpm lint:persistence` passes ([U1], §4).
3. The balance is **two-tier**: `Balance = { basic, additional }`; `additional` is a **stored, carrying** counter,
   `basic` is a **derivation** that equals `twinkle.basic_daily_amount` at the start of a reset window and does **not**
   carry unspent amount into the next window ([G2]).
4. The basic reset cadence is a **real UTC calendar day**, explicitly distinct from universe time, and the reset is
   **lazy** (no scheduled/cron job — the window rolls forward on the first read/spend in a new day) ([G2][T4]).
5. `PlanSpend(basicRemaining, additional, cost)` draws **basic first, additional for the overflow**, returns the
   per-tier draw and an `ok` flag, and **never** produces a negative balance in either tier ([G2][G5]).
6. `RecallCost(accessibilityCost)` is **non-decreasing** in decay-depth (a more-decayed memory costs at least as much to
   recall), consumes the accessibility signal from [38.forgetting-cost](38.forgetting-cost.md) (no decay math here), and
   is bounded by `twinkle.recall_max_cost` ([F4][G4], CC3).
7. `GistViewCost(semanticStage)` is **non-increasing** in gist-depth (a deeper gist costs no more than a shallower one),
   consumes `semantic_stage` from [40.semanticization](40.semanticization.md), and is floored at `twinkle.gist_min_cost`
   (cheap but never free) ([R8][G4]).
8. The pure functions (`RecallCost`, `GistViewCost`, `PlanSpend`, `basicRemaining`) produce **identical outputs** Go
   (`internal/twinkle`) ↔ TS (`packages/twinkle-logic`) for a shared golden-fixture table within a fixed tolerance, and
   both read the same generated `twinkle.*` constants (golden-parity, per [18.synapse-plasticity](18.synapse-plasticity.md)).
9. The **balance is server-authoritative**: it is written only server-side, the FE reads and displays it (never advances
   it), and a spend is committed inside a single-writer transaction so concurrent spends cannot oversell ([I5]-consistent
   stored state, per [29.universe-clock](29.universe-clock.md)).
10. `twinkle_ledger_entries` is **append-only** (no system `UPDATE`/`DELETE`), and a retried earn/spend carrying the same
    `dedup_key` does **not** double-apply (idempotent) ([I1]).
11. `twinkle` is a **standalone context** that does **not** import `internal/memory`; the `SpendGate` and ledger
    repository ports are **consumer-owned** (declared by memory / the twinkle use-case, not pre-declared here), and only
    `twinkle/pg` imports `dbgen` (CC2/CC8, ARCHITECTURE §2.4).
12. No formula, curve shape, clamp, or the reset-window rule is stored in `spec/values.yaml`; only the coefficients,
    caps, and the daily grant listed below are values (formulas stay code).
