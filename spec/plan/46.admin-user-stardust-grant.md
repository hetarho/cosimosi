# 46. 관리자 사용자 목록 + 별가루 지급

> `/admin`에 **사용자** 탭을 추가해 전체 사용자 목록을 보고, `user_id`로 검색하고, 선택한 사용자에게 별가루(Stardust)를 관리자 보정 지급하는 운영 표면. | 범위: FS | 상태: 기획

## 목적

44 커스터마이즈 시스템은 별가루 지갑과 구매 흐름을 만들었지만, 운영자가 사용자별 잔액을 확인하거나 보정 지급할 수 있는 표면은 없다. 베타 운영 중에는 초대 사용자에게 보상성 별가루를 주거나, 테스트/CS 상황에서 특정 계정의 잔액을 보정해야 한다. 이 plan은 그 최소 운영 기능만 `/admin`에 얹는다: **사용자 전체 목록, `user_id` 검색, 별가루 지급**.

이 기능은 유료 충전/결제/환불 시스템이 아니다. 사용자가 직접 별가루를 늘리는 경로는 만들지 않고, 기존 `AdminService` allowlist 뒤의 관리자만 서버 RPC로 지급할 수 있다.

## 범위 / 비목표

**범위**
- `/admin` 셸에 `사용자` 탭을 추가한다. 기존 `LLM 관리`·`초대 코드` 탭과 같은 `?tab=` 딥링크 패턴을 따른다.
- 사용자 목록을 페이지네이션으로 조회한다. production Supabase DB에서는 `auth.users`를 기준으로 **계정이 있지만 아직 일기/지갑 행이 없는 사용자도** 보이고, 로컬처럼 `auth.users`가 없으면 앱 도메인 테이블의 `user_id` 합집합으로 fallback한다.
- 목록 행은 운영에 필요한 최소 정보만 보여준다: `user_id`, 현재 별가루의 **유효 잔액**, 지갑 행 시드 여부. 유효 잔액은 `user_wallet` 행이 있으면 그 값, 없으면 44의 `customization.starting_stardust`를 읽기 전용으로 표시한다.
- `user_id` 검색을 제공한다. 검색어는 `user_id` 부분 일치(대소문자 무시)로 서버에서 필터링한다.
- 관리자가 사용자 행에서 양의 정수 amount를 입력해 별가루를 지급한다. 지갑이 아직 없으면 먼저 `starting_stardust`로 멱등 시드한 뒤 amount를 더한다.
- 지급은 서버 트랜잭션 하나에서 처리하고, `admin_stardust_grants` 감사 행을 남긴다. UI는 지급 이력 조회를 제공하지 않는다.

**비목표**
- 사용자 삭제·정지·멤버십 변경·초대 재발행·프로필 편집·이메일 변경·별/일기 열람은 하지 않는다.
- 별가루 차감, 환불, 결제 PG, 충전 상품, 쿠폰, 자동 보상 규칙은 만들지 않는다.
- 커스터마이즈 아이템 가격/무료 여부를 admin에서 편집하지 않는다. 가격은 계속 `spec/values.yaml`이 단일 출처다.
- 사용자 검색은 `user_id`만 대상으로 한다. 이메일/이름/메타데이터 검색은 이번 범위가 아니다.
- 지급 감사 이력 화면은 만들지 않는다. 감사 테이블은 운영 추적과 사고 복구용 내부 기록이다.

## 설계 요점

### A. AdminService 확장

기존 `/admin` 운영 표면이 `AdminService` 뒤에 있으므로 새 사용자/지급 RPC도 `proto/cosimosi/v1/admin.proto`의 같은 서비스에 추가한다. 인증·권한은 34와 동일하다: 미인증은 `Unauthenticated`, allowlist 밖은 `PermissionDenied`이고 FE는 NotFound로 숨긴다.

신규 메시지:

```proto
message AdminUser {
  string user_id = 1;
  int64 stardust = 2;       // effective balance: wallet row value, or starting_stardust when unseeded
  bool wallet_seeded = 3;   // true if user_wallet row exists
}

message ListAdminUsersRequest {
  string user_id_query = 1; // optional, case-insensitive contains filter
  int32 page_size = 2;      // default/cap from values
  string page_token = 3;    // last user_id from previous page
}

message ListAdminUsersResponse {
  repeated AdminUser users = 1;
  string next_page_token = 2;
}

message GrantUserStardustRequest {
  string target_user_id = 1;
  int64 amount = 2; // positive integer
}

message GrantUserStardustResponse {
  AdminUser user = 1; // target's post-grant state
}
```

신규 RPC:
- `ListAdminUsers` — `NO_SIDE_EFFECTS`, HTTP GET 가능. 사용자 목록/검색.
- `GrantUserStardust` — mutation. target 지갑을 시드한 뒤 amount를 더하고 감사 행을 남김.

### B. 사용자 목록의 "모두" 기준

production Supabase DB에는 `auth.users`가 있으므로 그 테이블이 사용자 목록의 1차 출처다. 이미 34의 `AdminTotals`가 `to_regclass('auth.users')` 가드 패턴을 쓰므로 같은 원칙을 따른다.

- `auth.users`가 있으면 `SELECT id::text AS user_id FROM auth.users`.
- 없으면 fallback으로 앱 도메인 테이블의 user_id 합집합을 쓴다: `records`, `memories`, `user_settings`, `user_wallet`, `user_owned_items`, `user_emotion_colors`, `universe_shares`, `memberships` 등 현재 스키마에 존재하는 사용자 소유 테이블.
- 목록 조회는 `user_id` 오름차순 keyset pagination이다. `page_token`은 직전 페이지 마지막 `user_id`이고, 검색어가 있으면 같은 필터 안에서 `user_id > page_token`을 적용한다.
- 별가루 표시값은 `COALESCE(user_wallet.stardust, values.CustomizationStartingStardust)`로 계산하되, 목록 조회만으로 지갑 행을 만들지 않는다. `wallet_seeded=false`는 "아직 지갑 row는 없지만 첫 인벤토리/지급 시 시작 잔액으로 시드될 사용자"라는 뜻이다.

### C. 별가루 지급 규칙

별가루 지급은 기존 44의 구매 차감 모델에 **관리자 보정 증가 경로**를 하나 추가하는 것이다. 사용자에게 직접 노출되는 충전은 여전히 없다.

- `amount`는 양의 정수여야 한다. 0 이하, 정수 범위 초과, 지급 후 `user_wallet.stardust`가 DB `INTEGER` 상한을 넘는 요청은 `InvalidArgument` 또는 `FailedPrecondition`으로 거부한다.
- `target_user_id`는 실제 사용자여야 한다. `auth.users`가 있는 production에서는 존재하지 않는 id를 `NotFound`로 거부한다. 로컬 fallback DB에서는 앱 도메인 테이블 합집합에 있거나 지갑 row가 있는 id만 허용한다.
- 트랜잭션 순서:
  1. target 사용자 존재 확인.
  2. `SeedWallet(target, starting_stardust)`로 지갑을 멱등 시드한다. 이미 있으면 잔액 유지.
  3. `UPDATE user_wallet SET stardust = stardust + amount`를 overflow guard와 함께 수행한다.
  4. `admin_stardust_grants`에 감사 행을 insert한다.
  5. post-grant `AdminUser`를 반환한다.
- 감사 행에는 최소한 `id`, `admin_user_id`, `target_user_id`, `amount`, `balance_before`, `balance_after`, `created_at`을 기록한다. `admin_user_id`는 auth context의 관리자 `sub`다. UI에서 이 이력을 노출하지 않아도 DB에는 남긴다.

### D. FE `/admin` 사용자 탭

- `AdminPage.tsx`의 `TABS`에 `{ value: 'users', label: '사용자' }`를 추가하고 `active === 'users'`일 때 `UsersTab`을 렌더한다.
- `UsersTab` 구성:
  - 상단 `user_id` 검색 input + 검색 버튼. 검색은 서버 쿼리 파라미터로 보낸다.
  - 사용자 목록 행: `user_id`, 별가루 유효 잔액, 지갑 시드 여부, 지급 amount input, 지급 버튼.
  - 다음 페이지 버튼(`next_page_token`이 있을 때만).
  - 지급 성공 시 해당 목록 query를 invalidate하거나 응답의 target row를 캐시에 반영해 잔액이 즉시 갱신된다.
- 기존 admin 패턴과 동일하게 일반 실패는 `ErrorNotice`, 권한 거부는 `AdminPage` 셸의 NotFound 위장 렌더가 처리한다.

### E. FSD / 바운디드 컨텍스트

- 지급 RPC는 `admin` bounded context가 소유한다. `settings.Service`의 구매/소유권 로직을 UI에서 우회하지 않는다.
- DB query는 `backend/internal/db/queries/admin.sql`에 둔다. `admin.Repository`에 user listing/grant용 포트를 추가해 service가 정책을 소유한다.
- `settings.sql`의 `SeedWallet`과 같은 시드 규칙을 재사용하거나 같은 의미의 admin query를 만든다. 시작 잔액 수치는 반드시 `values.CustomizationStartingStardust`에서 온다.

## 수용 기준 (인수 조건)

1. `/admin?tab=users`에 진입하면 admin allowlist 사용자는 사용자 탭을 볼 수 있고, 비관리자는 기존 `/admin`과 동일하게 NotFound 위장 화면을 본다.
2. production DB에서 사용자 목록은 `auth.users`의 모든 사용자 id를 포함한다. 아직 일기·지갑·설정 행이 없는 사용자도 목록에 나타난다.
3. `auth.users`가 없는 로컬 DB에서도 목록 조회는 에러 없이 앱 도메인 테이블의 `user_id` 합집합으로 동작한다.
4. 목록 행은 `user_id`, 별가루 유효 잔액, 지갑 시드 여부를 보여준다. 지갑 행이 없는 사용자는 DB를 쓰지 않고 `customization.starting_stardust`를 유효 잔액으로 표시한다.
5. 검색어를 입력하면 서버가 `user_id` 부분 일치(대소문자 무시)로 필터링해 결과를 반환한다.
6. 목록은 안정적인 `user_id` 오름차순 keyset pagination을 사용하고, `next_page_token`으로 다음 페이지를 이어서 조회할 수 있다.
7. 관리자가 양의 정수 amount를 지급하면 target 지갑이 없을 경우 `starting_stardust`로 먼저 시드되고, 지급 후 잔액은 기존 유효 잔액 + amount가 된다.
8. 지급은 한 트랜잭션이다. 잔액 증가와 감사 행 insert 중 하나라도 실패하면 둘 다 적용되지 않는다.
9. 지급 성공 시 `admin_stardust_grants`에 관리자 id, 대상 user_id, amount, 이전/이후 잔액, 시각이 남는다.
10. 0 이하 amount, overflow를 일으키는 amount, production에서 존재하지 않는 target user_id는 거부되고 지갑/감사 행을 바꾸지 않는다.
11. 사용자 직접 충전·결제·환불·차감 경로는 생기지 않는다. 별가루 증가 경로는 시작 잔액 시드와 관리자 지급뿐이다.
12. 신규 RPC는 모두 `AdminService` admin gate 뒤에 있고 unary다. build·lint·vet·기존 admin/커스터마이즈 테스트가 회귀 없이 통과한다.

## 정책 영향 (policy / values)

> 아래는 **구현 완료 시** policy에 반영될 규칙이다. 현재는 plan이 출처이며, 미구현 규칙을 policy에 기성사실로 쓰지 않는다.

- **`policy/domain/admin.md`** — `/admin` 표면에 사용자 탭이 추가된다. AdminService는 LLM 운영/초대 코드/사용자 별가루 보정 지급을 모두 같은 admin allowlist 뒤에서 제공한다. 사용자 탭은 user_id 목록·검색·별가루 지급만 포함하고, 사용자 데이터 열람/삭제/정지 기능은 없다.
- **`policy/domain/customization.md`** — 별가루 잔액 증가 경로가 기존 "시작 잔액 시드"에 더해 "관리자 보정 지급"으로 확장된다. 사용자 직접 충전·결제·환불은 여전히 비목표다. 구매 차감과 관리자 지급은 둘 다 서버 권위 지갑을 통해서만 일어난다.
- **`spec/values.yaml` — 신규 admin pagination config**:
  - `admin.user_list_default_page_size: 50`
  - `admin.user_list_max_page_size: 100`
  - 지급 amount의 양수/overflow 검증은 도메인 불변식이므로 별도 튜닝 값으로 두지 않는다. 시작 잔액은 기존 `customization.starting_stardust`를 재사용한다.
