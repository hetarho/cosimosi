# 32. develop 환경 격리 — 스테이징 Supabase·백엔드·프론트 분기 (staging-isolation)

> `develop`(스테이징)이 프로덕션과 백엔드·DB를 공유하지 않도록 격리한다 — 스테이징 Supabase·스테이징 백엔드 스택 + 브랜치별 프론트 빌드(`develop`→스테이징 값, `main`→prod 값). | 범위: Infra/FS | 선행: 14 | 차단: —

## 목적 (Why)

지금은 Cloudflare Workers Builds가 **빌드 변수 목록을 하나만** 갖기에(대시보드는 Production/Preview 분리 불가 — 공식 문서 확인, API로만 트리거별 가능), `develop` 프리뷰와 `main` 프로덕션이 **같은 `VITE_API_URL`·같은 Supabase 값으로 빌드**된다. 즉 코드는 develop/main으로 갈리지만 **데이터(백엔드·DB)는 공유**한다. 실사용자가 없고 백엔드도 안 떠 있는 **출시 전엔 무해**하지만, **베타 사용자가 생긴 뒤엔 develop에서의 테스트 작성이 프로덕션 DB를 오염**시킨다. 본 스펙은 베타 출시 직전에 `develop`=스테이징을 프로덕션과 **데이터까지 격리**한다(브랜치 전략의 "develop→스테이징 / main→프로덕션" 전제를 실제로 충족). **지금 당장 구현하는 게 아니라**, 베타 사용자를 받기 직전에 수행하는 작업으로 등재한다.

## 범위

- **포함:**
  - **스테이징 Supabase 프로젝트**(prod와 별도, 서울 `ap-northeast-2`): staging 마이그레이션이 prod 데이터에 절대 닿지 않게. 연결 문자열 분리(앱 런타임=트랜잭션 풀러 6543 / 마이그레이션=직접 5432). (DEPLOY.md §2 수행.)
  - **스테이징 백엔드 스택**: Lightsail `/srv/cosimosi-staging` + `api.staging.<도메인>`(Caddy TLS), `CORS_ORIGIN`=스테이징 프론트. (DEPLOY.md §3 수행 — VPS 한 대가 prod·staging 두 스택 호스팅.)
  - **프론트 브랜치별 빌드(핵심 — 공용 변수 1세트 제약 우회)**: Cloudflare가 주입하는 `WORKERS_CI_BRANCH`로 build command에서 환경 분기 — `main`이면 prod `VITE_*`, 그 외(`develop` 등)면 staging `VITE_*`로 빌드한다. 값은 공용 변수 목록에 `VITE_*_PROD`/`VITE_*_STAGING` **두 세트**로 넣고 빌드 스크립트가 고른다.
  - **GitHub Actions 환경 분리**: `production`/`staging` Environments에 `DIRECT_DATABASE_URL`(각 Supabase 5432) 등 환경별 시크릿. (DEPLOY.md §4.)
  - **DEPLOY.md 정정**: (a) 프론트 호스팅을 "Cloudflare Pages"가 아니라 **Workers(정적 자산)** 로, (b) "트리거별 빌드 변수가 다르다"는 서술을 **"대시보드 변수는 공용 1세트 → 브랜치별 빌드(`WORKERS_CI_BRANCH`)로 분기"** 로 바로잡는다(현 서술이 실제 대시보드와 어긋남).
  - (선택) 스테이징 전용 도메인 `staging.cosimosi.haeram.me` — 안 하면 프리뷰 URL(`*-cosimosi.<sub>.workers.dev`)을 스테이징 URL로 쓴다.
- **비목표(Out of scope):**
  - 프로덕션 최초 배포·CI/CD 파이프라인 자체 → **14**(본 스펙은 14의 산출물 위에 스테이징 격리만 더한다).
  - Worker 2개로 가르는 대안(별도 `cosimosi-staging` Worker) — 설계 요점에 대안으로만 기술(기본은 브랜치별 빌드 1 Worker).
  - 신규 제품 기능.

## 설계 요점

**브랜치별 빌드(1 Worker, 공용 변수 + 분기 스크립트 — 기본 채택).** Cloudflare 빌드 변수(공용 목록)에 두 세트를 넣고, 빌드 직전 스크립트가 `WORKERS_CI_BRANCH`로 골라 **Vite가 읽는 `.env.production.local`을 써** 준다(자식 프로세스 env는 전파 안 되므로 *파일*로 넘긴다 — Vite가 production 모드에서 `.env.production.local`을 로드).

```
Cloudflare 빌드 변수(공용 목록):
  VITE_API_URL_PROD, VITE_API_URL_STAGING
  VITE_SUPABASE_URL_PROD, VITE_SUPABASE_URL_STAGING
  VITE_SUPABASE_PUBLISHABLE_KEY_PROD, VITE_SUPABASE_PUBLISHABLE_KEY_STAGING
  (VITE_SENTRY_DSN_*, 선택)
Build command(대시보드):  node scripts/ci-build-env.mjs && pnpm --filter ./frontend build
Deploy command:           npx wrangler deploy            (main = 프로덕션 승격)
Version command:          npx wrangler versions upload   (그 외 = 프리뷰)
```

```js
// scripts/ci-build-env.mjs (신규) — WORKERS_CI_BRANCH로 환경 골라 frontend/.env.production.local 작성
import { writeFileSync } from 'node:fs'
const prod = process.env.WORKERS_CI_BRANCH === 'main'          // main만 prod, 그 외(develop 등)=staging
const pick = (base) => process.env[`${base}_${prod ? 'PROD' : 'STAGING'}`] ?? ''
const env = {
  VITE_API_URL: pick('VITE_API_URL'),
  VITE_SUPABASE_URL: pick('VITE_SUPABASE_URL'),
  VITE_SUPABASE_PUBLISHABLE_KEY: pick('VITE_SUPABASE_PUBLISHABLE_KEY'),
  VITE_SENTRY_ENVIRONMENT: prod ? 'production' : 'staging',
}
writeFileSync('frontend/.env.production.local',
  Object.entries(env).map(([k, v]) => `${k}=${v}`).join('\n') + '\n')
console.log(`[ci-build-env] branch=${process.env.WORKERS_CI_BRANCH} → ${prod ? 'PROD' : 'STAGING'}`)
```

- `frontend/.env.production.local`은 **gitignore**(빌드 산출물). 로컬 개발은 기존 `.env`(현행) 유지.
- 누락 값은 빈 문자열 → 코드(`shared/config`)가 빈 값에 안전하게 폴백/경고하도록 한다(없는 변수에 크래시 금지).

**대안(기술만 — 비채택): Worker 2개.** `cosimosi`(main→prod, `cosimosi.haeram.me`) + `cosimosi-staging`(develop→staging, `staging.cosimosi.haeram.me`). 각 Worker가 자기 변수·도메인을 가져 격리가 가장 깔끔하나 설정이 2배다. wrangler `env`(`--env staging/production`)로도 가능(공식 "Advanced Setups"). 브랜치별 빌드로 충분하면 채택 안 함.

**백엔드·DB 격리(DEPLOY.md §2·§3 수행).** staging Supabase·`/srv/cosimosi-staging` 스택을 세우고, staging 프론트(`VITE_API_URL_STAGING=https://api.staging.<도메인>`)가 staging 백엔드만 호출하게 한다. 마이그레이션은 환경별 `DIRECT_DATABASE_URL`(5432)로 해당 DB에만 적용.

## 영향 파일

- 신규 `scripts/ci-build-env.mjs` — `WORKERS_CI_BRANCH`로 `frontend/.env.production.local` 작성(위 스크립트).
- 수정 `.gitignore` — `frontend/.env.production.local` 무시(없으면 추가).
- 수정 `frontend/src/shared/config/*`(또는 env 소비처) — 빈 `VITE_*`에 안전 폴백/경고(미설정 시 크래시 금지).
- 수정 `wrangler.jsonc` 주석 — 빌드 변수는 `*_PROD`/`*_STAGING` 두 세트 + 브랜치별 빌드 방식임을 명시(설정 자체는 대시보드).
- 수정 [DEPLOY.md](../../DEPLOY.md) — (a) §브랜치→환경 표의 "Cloudflare Pages"→**Workers**, (b) §1의 "트리거별 빌드 변수" 서술을 **"공용 변수 1세트 + `WORKERS_CI_BRANCH` 브랜치별 빌드"** 로 정정, (c) 빌드/배포/버전 command·`*_PROD`/`*_STAGING` 변수 명세 반영.
- 수정 `.env.production.example` — 백엔드 staging/prod 환경 변수 키 문서화(값 없이; `API_DOMAIN`·`DATABASE_URL`·`CORS_ORIGIN`·`SUPABASE_JWT_SECRET` 등 — DEPLOY.md §3과 정합).
- (수동·코드 외) Cloudflare 대시보드 빌드 변수 2세트 입력, Supabase staging 프로젝트, Lightsail `/srv/cosimosi-staging`, GitHub `staging`/`production` Environments 시크릿 — DEPLOY.md 체크리스트.

## 수용 기준 (EARS / Given-When-Then)

1.1 WHEN `develop`(또는 `main` 외 브랜치)이 push되면 THE SYSTEM SHALL 프리뷰 빌드가 **staging `VITE_*`** (스테이징 API·스테이징 Supabase)로 번들되어 프리뷰 URL에 뜬다.
1.2 WHEN `main`이 push되면 THE SYSTEM SHALL 프로덕션 빌드가 **prod `VITE_*`** 로 번들되어 `cosimosi.haeram.me`에 뜬다.
1.3 WHEN 사용자가 `develop` 프리뷰에서 일기를 작성하면 THE SYSTEM SHALL 그 데이터가 **스테이징 Supabase에만** 저장되고 프로덕션 DB에는 닿지 않는다(데이터 격리).
1.4 WHEN 스테이징 마이그레이션을 적용하면 THE SYSTEM SHALL 스테이징 DB(5432 직접 연결)에만 적용되고 프로덕션 DB에는 적용되지 않는다.
1.5 IF 빌드 변수(`*_STAGING`/`*_PROD`) 중 일부가 비어 있으면 THEN THE SYSTEM SHALL 빌드는 성공하되 앱이 빈 값에 안전하게 폴백/경고하고 크래시하지 않는다.
1.6 WHEN 빌드 로그를 보면 THE SYSTEM SHALL `[ci-build-env] branch=… → PROD|STAGING`으로 어느 환경으로 빌드됐는지 드러난다(오설정 조기 발견).
1.7 THE SYSTEM SHALL `frontend/.env.production.local`을 커밋하지 않는다(gitignore — 빌드 산출물·환경 의존).
1.8 THE SYSTEM SHALL 시크릿(서비스 롤 키·DB URL·JWT secret)을 번들/리포에 넣지 않는다 — 런타임 백엔드 시크릿은 VPS `.env`/GitHub Secrets에만, 프론트는 `VITE_*` 공개값만(헌법: 시크릿 비커밋).

## 정책 영향(policy)

- **제품 domain/ux 정책 변경 없음** — 배포/인프라 구성이다. "환경 분리(develop=스테이징/main=프로덕션)와 데이터 격리"의 단일 운영 출처는 [DEPLOY.md](../../DEPLOY.md)이며, 본 스펙 구현 시 그 문서를 위 정정대로 갱신한다.

## 완료 정의 (DoD)

- `develop` push → 프리뷰 URL의 앱이 **스테이징** API/Supabase를 호출한다(네트워크 탭에서 `api.staging.<도메인>`·스테이징 Supabase 호스트 확인). `main` push → `cosimosi.haeram.me`가 **prod**를 호출한다.
- `develop` 프리뷰에서 작성한 일기가 **스테이징 Supabase에만** 보이고 프로덕션 DB엔 없다(양쪽 DB 조회로 격리 확인).
- 빌드 로그에 `[ci-build-env] branch=… → STAGING|PROD`가 찍히고, 일부 변수 누락 시에도 빌드 성공 + 앱 비크래시(폴백).
- `frontend/.env.production.local`이 git에 추적되지 않는다(`git check-ignore`로 확인). 리포·번들에 시크릿 없음(grep).
- DEPLOY.md가 Workers(정적 자산) + 공용 변수 1세트 + `WORKERS_CI_BRANCH` 브랜치별 빌드로 정정돼, 다음 세션이 "트리거별 변수"로 오해하지 않는다.
- `pnpm --filter frontend build`가 `.env.production.local` 유무와 무관하게 로컬에서 성공한다.
