## Design

### The gate rule (one rule, two app expressions)

The gate is a pure function of the auth session snapshot ([04.auth-session](04.auth-session.md)) â†’ a routing decision:

| Session status ([04])              | Web `/` guard (`beforeLoad`)                                | Mobile nav root            |
| ---------------------------------- | ----------------------------------------------------------- | -------------------------- |
| `authenticated`                    | render the universe (`UniverseHomePage`)                    | the `UniverseScreen` stack |
| `signedOut` / `expired` / `failed` | `redirect` to the **login** entry (carry `redirect`/`from`) | the **login** stack        |
| `bootstrapping` / `refreshing`     | hold on a neutral loading state (no redirect)               | a neutral splash           |

- **Neutral hold, never a flash.** `bootstrapping`/`refreshing` must **not** redirect to login â€” [04](04.auth-session.md)
  preserves `userId` across an in-flight refresh precisely so the UI does not flash signed-out. The gate waits for the
  session to settle before deciding. **Decided (2026-07):** the gate treats `refreshing` as _provisionally
  authenticated_ (hold on the universe if already there; hold on the splash if entering cold) â€” it redirects to login
  **only** on a settled `signedOut`/`expired`/`failed`.
- **`failed` routes to login, not an error screen.** A failed auth bootstrap is a signed-out user from the product's
  point of view; [04]'s error surfacing (observability) still fires, but the _route_ is login.
- The rule is authored **once conceptually** and expressed through each app's app-layer nav seam (web `beforeLoad`,
  mobile initial-stack selection) â€” disciplinary parity, not a shared route package (matching [15]'s parity stance and
  [ARCHITECTURE Â§3.5](../ARCHITECTURE.md)). The tiny "status â†’ decision" mapping is the only shared logic and lives with
  the auth facade / a small `app`-layer helper.

  `login`), so v2 slots in without re-architecting the gate. v1 does not build it and does not route to it.

## Acceptance Criteria

1. On **web**, the `/` route carries a `beforeLoad` **auth guard**: an `authenticated` session renders the universe; a
   settled `signedOut`/`expired`/`failed` session **redirects to the login entry**; `bootstrapping`/`refreshing` **holds
   on a neutral loading state without redirecting** (no signed-out flash) ([U1][U4]).

   `UniverseScreen`; settled signed-out â†’ the login stack; `bootstrapping`/`refreshing` â†’ a neutral splash â€” webâ†”mobile
   parity ([U1][Â§3.5]).
3. There is **no landing/marketing page** in v1 on either app: the unauthenticated default is **login**, the
   authenticated default is the **universe**, with no intermediate route ([U4]).
4. A successful sign-in ([04]'s action reaches `authenticated`) routes to the universe â€” on web to the carried
   `redirect`/`from` (or `/`), on mobile by swapping to the `UniverseScreen` stack.
5. **Sign-out** routes back to login on both apps and unmounts the universe view; it **never deletes** the user's data â€”
   a subsequent sign-in for the same user re-reads the same universe ([U1][I1]).
6. A **first-run** user (no memories) lands on the universe rendered as the **gray latent field**

7. A directly-entered **web URL** while unauthenticated redirects to login and, after sign-in, returns to the originally
   requested route (`redirect`/`from` param); the router/nav library stays confined to `app/routes` / `app/navigation`,
   and the gate reads auth **through the [04] facade**, never Supabase directly ([Â§3.1]).
8. The gate never lets `GetUniverse` (or any product read) run without an authenticated session â€” the presentation-side
   complement of [04]'s server-side per-user isolation ([U1][Â§2.7]).
9. **All** user-visible copy (login entry, first-run welcome) resolves through the i18n seam (no raw strings;
   `lint:raw-strings` passes), in a literary / restrained Korean voice.
10. This unit adds **no** auth engine, renderer, read model, write flow, settings UI, RPC, or DB change â€” it composes
    [04]/[15]/[13]/[23]/[25]/[27]; `pnpm typecheck`, `pnpm lint` (incl. `lint:fsd:layout`, `lint:raw-strings`),
