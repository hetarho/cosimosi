# 16. memory aggregate schema

> Memory aggregate schema — the core `internal/memory` context's persistence and aggregate layout: goose migrations,
> sqlc + pgvector, and the product tables Epics B–H build on without reshaping.
> Scope: BE | Status: built

## Purpose

[05.data-persistence](05.data-persistence.md) installed the persistence stack (Postgres, pgvector, goose, sqlc) but
deliberately created **no** product tables. This unit lays the CORE domain's persistent shape: the `internal/memory`
context package and the full **memory aggregate** schema — `Diary`, `EpisodicMemory` (aggregate root), `Neuron`,
`NeuronActivation`, `Synapse`, `Embedding`, and the `jobs` queue table.

It exists so that Epic A's launch/read loop has real tables to write and read, and so that Epics B–H (universe clock,
semanticization, decay, twinkle economy, deletion) **extend** the schema rather than rebuild it. It defines the
_shape_ — tables, columns, keys, indexes, aggregate boundaries, and the baseline CRUD the aggregate needs — and
reserves the slots later epics fill. It does **not** implement use-case logic, LLM extraction, linking math, or
rendering; those are owned by later plans (20–27) against this schema.

## Scope / Non-goals

**In scope**

- The `internal/memory` context package layout: start as one package, split into `memory/{domain,app,pg,rpc}` only when
  it earns it (ARCHITECTURE §2.3). `memory/pg` is the only place that touches sqlc/pgx.
- One goose migration set creating all product tables: `diaries`, `episodic_memories`, `neurons`,
  `neuron_activations`, `synapses`, `embeddings`, `jobs` — with exact columns, types, primary/foreign keys, the
  undirected-synapse uniqueness/check constraint, and the pgvector `vector(1024)` column + HNSW index.
- Aggregate boundaries: `EpisodicMemory` as aggregate root referencing `Neuron` by id through `neuron_activations`;
  `Neuron` as its own shared root; `Synapse` as a neuron↔neuron edge only.
- Per-user isolation on every table (`user_id`) and per-user scoping on every query (the [04.auth-session](04.auth-session.md)/
  [05.data-persistence](05.data-persistence.md) `UserScope` convention).
- Immutability/delete stance: live `diaries` are append-only and never `UPDATE`d; system dynamics never hard-delete;
  Epic H may hard-delete user-deleted diary records after its restore window.
- `sqlc.yaml` wiring for query files under `db/queries/memory/**`, generated into the existing `dbgen` package, plus
  the baseline aggregate CRUD Epic A's launch and read need (insert diary/memory/neuron/activation/synapse; a
  universe read).
- Reserved seams for later epics (universe clock, twinkle ledger, decay/semantic stage population, deletion flows), so
  the schema has their slots without building them.
- One new `spec/values.yaml` key introduced by this unit: `ai.embedding_dim` (the embedding vector dimension `N`).

**Non-goals**

- No use-case behavior (`Encode`/`PersistEncoded`/`Link`), LLM ports, or dedup logic — those are plans 20–22.
- No domain pure functions (`Potentiate`/`Decay`/`EffectiveBrightness`/emotion arousal fns) — plans 17–18.
- No RPC handlers or proto contract (`SplitDiary`/`LaunchStars`/`GetUniverse`) beyond the query surface the read needs;
  the transport contract is owned by plans 20/23.
- No `universe_state` clock table (Epic B), `twinkle_ledger` (Epic F), or populated decay/semantic stages (Epics C/D/E).
- No stored positions, constellations, latent neurons, or nebula color — those are emergent and never persisted [I5].
- No force-sim, rendering, or frontend mirrors.
- No RLS hardening; schema and queries stay RLS-compatible for later.

## Design

### The `internal/memory` context package

`internal/memory` is the product's **core domain context** (ARCHITECTURE §2.2). It owns the memory aggregate types and
canonical names (`Diary`, `EpisodicMemory`, `Neuron`, `NeuronActivation`, `Synapse`, `Emotion`, `Embedding`, the `jobs`
concept), and — in later plans — its domain pure functions, use-cases, and consumer-owned ports.

Per ARCHITECTURE §2.3, it **starts as one package** and splits into `memory/{domain,app,pg,rpc}` only when the
one-package shape stops being readable (a refactor, not a starting tax). This unit introduces only the two pieces the
schema needs:

- the domain types for the aggregate (canonical names, free of proto/sqlc/pgx/JSON/DB tags — ARCHITECTURE §2.4);
- `internal/memory/pg` — the persistence adapter. **`memory/pg` is the only package in this context that touches
  sqlc/pgx** (ARCHITECTURE §2.6). It maps rows↔domain and implements repository interfaces declared by the consumer.
  **No sqlc row (from `dbgen`) ever escapes `memory/pg` into context behavior or the domain model** (ARCHITECTURE §2.4
  forbidden pattern #4). Repository ports themselves (`DiaryStore`, `EpisodicMemoryStore`, `NeuronStore`,
  `SynapseStore`, `EmbeddingStore`, `JobQueue`) are declared where they are consumed — by the use-cases in plans 20–22
  — so this unit does not pre-declare implementation-side interfaces (ARCHITECTURE §2.4 forbidden pattern #6). It
  provides the concrete `pg` struct and the queries those ports will bind to.

IDs: the backend mints `TEXT` primary keys (nanoid); clients never mint IDs (ARCHITECTURE §5). Times are stored UTC.

### Aggregate boundaries

Drawn per ARCHITECTURE §2.5 and the two-layer model in [PRD.md](../PRD.md) §2:

- **`EpisodicMemory` is the aggregate root** (the "star" / engram — a sparse ensemble of neurons). It has its own
  lifecycle (created at launch, later recalled/reconsolidated/decayed/semanticized/soft-deleted). It references
  `Neuron` **by id, through the `neuron_activations` membership join** — it never owns neurons and never reaches inside
  a neuron's internals.
- **`Neuron` is its own aggregate root**, **shared across many memories** (§2 "작은 별 = 뉴런 … 여러 기억에 공유"; [E3]).
  A neuron is referenced by id from activations and synapses; it is not a child of any single memory.
- **`Synapse` is a neuron↔neuron edge only** [I6][L6]. There is **no memory↔memory edge and no memory↔memory attraction**
  [I4][E9] — inter-memory proximity is _emergent_ from shared neurons + synapse strength, not a stored relationship.
- **Emergent things are not persisted** [I5] (ARCHITECTURE §2.5, forbidden pattern #3, #8): positions/coordinates
  (force-sim output), constellations, latent neurons, and nebula color have **no table and no domain type**. The server
  stores only the source facts — neurons, synapses, activations, and per-neuron connectivity that read-side queries
  derive centrality from. `synapses` is the one materialized edge table, and it is materialized only because atomic
  weighted upserts and graph traversal need it (ARCHITECTURE §2.5/§2.6) — it is rebuildable from co-activation facts,
  not a promoted memory↔memory relationship.

### Immutability & deletion stance

- **`diaries` is append-only while live** [I2]: the objective record is never `UPDATE`d by the system or by
  recall/reconsolidation. Representational change lives on `episodic_memories` (`current_text`, stage texts, `seed`),
  never on the diary. No `updated_at` on `diaries`; the user-delete path in Epic H is the only exception and may hard-delete
  the diary record after its restore window.
- **System dynamics never hard-delete** [I1][F3]: forgetting is dimming (a read-time derivation), not a row delete. There
  is no cascade delete driven by forgetting, recall, reconsolidation, consolidation, or visualization.
- **Soft-delete is user-only and reserved for Epic H.** `deleted_at timestamptz NULL` on `episodic_memories` and
  `sealed_at timestamptz NULL` on `neurons` are declared now (so the aggregate's shape is stable) but no delete/seal
  flow is built in Epic A. Their semantics (30-day soft-delete, shared-neuron preservation, synapse-contribution
  weakening) are Epic H ([X1]–[X6]).

### DB schema (goose migration, verbatim from the brief)

One forward-only-safe goose migration (with a matching down) under `apps/api/db/migrations/`, owned by this unit's job
(per [05.data-persistence](05.data-persistence.md): the unit that adds a product table owns its migration). Every table
carries `user_id TEXT NOT NULL`; every product query scopes by it; `id TEXT PRIMARY KEY` (nanoid) unless noted. The
`vector` extension is already enabled by the platform migration ([05.data-persistence](05.data-persistence.md)).

```sql
-- diaries — the live immutable, append-only objective record ([I2]). No UPDATE; post-window hard delete only via Epic H user full-delete.
CREATE TABLE diaries (
    id          TEXT PRIMARY KEY,
    user_id     TEXT NOT NULL,
    body        TEXT NOT NULL,
    diary_date  DATE NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- episodic_memories — the aggregate root (the "star"). Emotion is stored as columns on the root ([M1]: one primary
-- emotion per memory). seed nullable → set at creation. Stage/decay JSONB nullable until later epics fill them.
CREATE TABLE episodic_memories (
    id                          TEXT PRIMARY KEY,
    user_id                     TEXT NOT NULL,
    diary_id                    TEXT NOT NULL REFERENCES diaries(id),
    name                        TEXT NOT NULL,          -- LLM-recommended, user-editable [W2a]
    current_text                TEXT NOT NULL,          -- last reconstructed narrative; ≠ Diary [R8a]
    seed                        BIGINT,                 -- visual form seed [V5]; nullable → set at creation
    mood                        TEXT NOT NULL,          -- 13-enum [M2]
    valence                     REAL NOT NULL,          -- Emotion VO
    arousal                     REAL NOT NULL,          -- Emotion VO
    intensity                   REAL NOT NULL,          -- Emotion VO
    base_strength               REAL NOT NULL,          -- arousal-modulated initial size [E8a]
    recall_count                INT  NOT NULL DEFAULT 0,
    created_universe_time       DATE NOT NULL,          -- = diary_date at launch (universe-time [T])
    last_recalled_universe_time DATE,                   -- NULL until first recall
    semantic_stage              SMALLINT NOT NULL DEFAULT 0,  -- 0..4, populated Epic C/E
    semanticize_timer_reset_at  DATE,                   -- universe-time anchor, Epic C/E
    semantic_stages             JSONB,                  -- 4 gist texts, LLM-pregenerated [C7]; NULL until job fills it
    decay_stages                JSONB,                  -- forgetting texts, filled progressively Epic D
    deleted_at                  TIMESTAMPTZ             -- soft-delete, Epic H [X2]
);

-- neurons — a memory element, shared across memories (referenced by id). name NULL reserved, but latent neurons are
-- NOT persisted [E7a] — a row exists only once a neuron is actually activated.
CREATE TABLE neurons (
    id          TEXT PRIMARY KEY,
    user_id     TEXT NOT NULL,
    name        TEXT,                                   -- extracted name; NULL reserved
    neuron_type TEXT NOT NULL,                          -- 'semantic' | 'spatial' | 'entity' [E3]
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    sealed_at   TIMESTAMPTZ                             -- Epic H letting-go [X4]
);

-- neuron_activations — the EpisodicMemory↔Neuron membership join with co-membership weight [E8].
CREATE TABLE neuron_activations (
    episodic_memory_id TEXT NOT NULL REFERENCES episodic_memories(id),
    neuron_id          TEXT NOT NULL REFERENCES neurons(id),
    user_id            TEXT NOT NULL,
    weight             REAL NOT NULL,                   -- co-membership weight → centroid pull [E8]
    PRIMARY KEY (episodic_memory_id, neuron_id)
);

-- synapses — undirected neuron↔neuron edge [I6][L6]. Canonical storage: neuron_a_id < neuron_b_id.
CREATE TABLE synapses (
    id                           TEXT PRIMARY KEY,
    user_id                      TEXT NOT NULL,
    neuron_a_id                  TEXT NOT NULL REFERENCES neurons(id),
    neuron_b_id                  TEXT NOT NULL REFERENCES neurons(id),
    strength                     REAL NOT NULL,         -- stored base 0..1 [L9]
    co_activation_count          INT  NOT NULL,
    last_activated_universe_time DATE NOT NULL,
    created_at                   TIMESTAMPTZ NOT NULL DEFAULT now(),
    CHECK (neuron_a_id < neuron_b_id),
    UNIQUE (user_id, neuron_a_id, neuron_b_id)
);

-- embeddings — element→vector for semantic nearest-neighbour. Embed NEURONS (dedup + latent-field assist operate on
-- neurons). 1024 = ai.embedding_dim.
CREATE TABLE embeddings (
    neuron_id TEXT NOT NULL REFERENCES neurons(id),
    user_id   TEXT NOT NULL,
    vector    vector(1024) NOT NULL,                    -- 1024 = ai.embedding_dim
    PRIMARY KEY (neuron_id)
);
CREATE INDEX embeddings_vector_hnsw ON embeddings USING hnsw (vector vector_cosine_ops);

-- jobs — async work queue. Claimed FOR UPDATE SKIP LOCKED; exponential backoff via next_run_at (ARCHITECTURE §2.8).
CREATE TABLE jobs (
    id          TEXT PRIMARY KEY,
    user_id     TEXT NOT NULL,
    kind        TEXT NOT NULL,                          -- 'embed' | 'semanticize' | 'link' | 'extract' | 'consolidate'
                                                        -- Epic A dispatches embed/semanticize/link; extract (sync in A)
                                                        -- + consolidate (Epic E) are reserved seams. See plan 22.
    payload     JSONB NOT NULL,
    status      TEXT NOT NULL,                          -- 'pending' | 'running' | 'done' | 'failed'
    attempts    INT  NOT NULL DEFAULT 0,
    next_run_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Notes on the schema decisions (from the brief):

- **`vector(1024)` dimension.** `1024 = ai.embedding_dim`, generated from `spec/values.yaml`. It uses the default
  dimension supported by the Voyage embedding model family targeted by [28.ai-provider-abstraction](28.ai-provider-abstraction.md),
  keeping the migration literal and the future `Embedder` contract on one source.
- **Undirected synapse.** The `UNIQUE (user_id, neuron_a_id, neuron_b_id)` constraint plus the canonical
  `neuron_a_id < neuron_b_id` ordering (enforced by the domain/`pg` mapper before write, not by a SQL trigger in
  Epic A) guarantees one row per unordered neuron pair per user — the invariant behind idempotent LTP upserts ([L8]).
  The migration also carries a `CHECK (neuron_a_id < neuron_b_id)` guard so direct SQL cannot persist a reversed pair.
- **HNSW over cosine.** The `hnsw` index with `vector_cosine_ops` supports the narrow embedding nearest-neighbour that
  neuron dedup ([E10]) and latent-field assist use; no custom HNSW parameters are set yet.
- **Emotion on the root.** `mood`/`valence`/`arousal`/`intensity` are columns on `episodic_memories` ([M1]: one primary
  emotion per memory), not a separate table — the `Emotion` value object is owned by plan 17 and maps to these columns.
- **Indexes for scoping.** The shipped migration adds per-user read indexes for the settled baseline read shape:
  `diaries(user_id, diary_date, id)`, `episodic_memories(user_id, created_universe_time, id) WHERE deleted_at IS NULL`,
  `episodic_memories(user_id, diary_id)`, `neurons(user_id, neuron_type, id) WHERE sealed_at IS NULL`,
  `neuron_activations(user_id, episodic_memory_id)`, `neuron_activations(user_id, neuron_id)`,
  `synapses(user_id, neuron_a_id)`, `synapses(user_id, neuron_b_id)`,
  `synapses(user_id, last_activated_universe_time)`, `embeddings(user_id, neuron_id)`, and
  `jobs(user_id, status, next_run_at)`.

### sqlc wiring & baseline queries

`apps/api/sqlc.yaml` already points at `db/migrations/*.sql` for schema and `db/queries/**/*.sql` for queries, emitting
`dbgen` (pgx/v5) — [05.data-persistence](05.data-persistence.md). This unit adds query files grouped under
`apps/api/db/queries/memory/**` and runs through `pnpm gen:sql`. Generated code stays infra: only `memory/pg` imports
`dbgen`.

Queries are kept **minimal** — later plans' use-cases own their own queries (ARCHITECTURE §2.3: each context owns its
tables/queries, and a use-case adds the queries it consumes). This unit defines only the baseline the aggregate needs
for Epic A's launch and read:

- **Inserts for the launch path** (consumed by `PersistEncoded`, plan 20): insert a `Diary`; insert an
  `EpisodicMemory`; insert/upsert a `Neuron`; insert a `NeuronActivation`; upsert a `Synapse`
  (`INSERT … ON CONFLICT (user_id, neuron_a_id, neuron_b_id) DO UPDATE`, the atomic weighted upsert of ARCHITECTURE
  §2.6). Insert an embedding row and enqueue a `jobs` row are also part of the baseline surface.
- **The universe read** (consumed by `GetUniverse`, plan 23): read a user's `episodic_memories`, `neurons`, and
  `synapses` (plus activation edges and per-neuron connectivity/degree) scoped to `user_id`. This returns the **stored
  facts** only; read-time derived values (effective brightness/strength, decay) are computed by shared pure functions
  (plans 18 / `packages/memory-logic`), never stored [I5].

Every query takes the authenticated `user_id` and filters by it — the [04.auth-session](04.auth-session.md)/
[05.data-persistence](05.data-persistence.md) `UserScope` convention, enforced by `pnpm lint:persistence` [U1] (per-user isolation, [ARCHITECTURE.md](../ARCHITECTURE.md) §4).

### Reserved seams for later epics

The schema deliberately leaves slots so Epics B–H extend without reshaping:

- **Universe clock (Epic B).** There is **no `universe_state` table** here. Epic A derives universe time from the
  latest launched memory's `created_universe_time`; the per-user clock is added in Epic B. Seam only.
- **Twinkle economy (Epic F).** No `twinkle_ledger` — the recall currency is Epic F.
- **Decay stages (Epic D).** `decay_stages JSONB` exists but is **NULL** in Epic A; the progressive word-deletion texts
  are filled in Epic D.
- **Semantic stages (Epics C/E).** `semantic_stage`, `semanticize_timer_reset_at`, and `semantic_stages JSONB` exist;
  `semantic_stages` is pregenerated by the semanticize job ([C7]) but the fill/advance behavior is Epics C/E.
- **Deletion flows (Epic H).** `deleted_at` and `sealed_at` exist but no soft-delete/seal/restore flow is built here
  ([X1]–[X6]).
- **Past-dated launch guard.** The monotonic guard — a diary with `diary_date < latest launched` saves the `Diary` but
  launches no star [T1][I10] — is owned by the launch use-case (plan 20); this schema supports it (a `Diary` row can
  exist without any `episodic_memories` referencing it). **Decided (2026-07):** Epic A ships the guard _with_ a
  one-time confirmation notice in the writing flow (plan 27) — the diary is saved and readable in the archive, no star
  launches; Epic B (universe clock) revisits the full past-date experience.

## Acceptance Criteria

1. A goose migration creates `diaries`, `episodic_memories`, `neurons`, `neuron_activations`, `synapses`, `embeddings`,
   and `jobs` with the exact columns, types, and keys in the schema above; `pnpm db:migrate` applies cleanly and
   `pnpm db:down` reverses cleanly (up/down round-trip leaves no residue).
2. Every product table has a `user_id TEXT NOT NULL` column, and `pnpm lint:persistence` passes (no table or query skips
   the per-user convention) [U1].
3. `synapses` has `UNIQUE (user_id, neuron_a_id, neuron_b_id)`, and stored rows satisfy the canonical
   `neuron_a_id < neuron_b_id` ordering, so an unordered neuron pair has at most one row per user [L8][I6].
4. `embeddings.vector` is `vector(1024)` with `1024` sourced from `ai.embedding_dim`, and an HNSW index exists on that column.
5. `episodic_memories` references `neurons` only through `neuron_activations` (no neuron columns on the root); `synapses`
   references `neurons` on both endpoints and no table carries a memory↔memory edge [I4][I6].
6. `episodic_memories.deleted_at` and `neurons.sealed_at` exist and default to NULL, and no migration or Epic A query
   performs a system-driven hard delete of any diary, neuron, memory, or synapse [I1].
7. `diaries` has no `UPDATE` query and no `updated_at` column; representational change targets `episodic_memories`
   fields, never `diaries` [I2].
8. No table stores positions, coordinates, constellations, latent neurons, or nebula color [I5]; the read query returns
   only stored facts (emotion, base_strength, recall_count, universe times, seed, neuron_type, synapse strength +
   co-activation, activation edges, per-neuron connectivity).
9. `pnpm gen:sql` generates the memory queries into `dbgen`; only `internal/memory/pg` imports `dbgen`, and no sqlc row
   type appears in the domain model or context behavior [ARCHITECTURE §2.4 #4].
10. Repository interfaces are not declared in this unit; `memory/pg` returns concrete structs, and the queries needed by
    Epic A launch (insert diary/memory/neuron/activation/synapse upsert) and read (`GetUniverse` shape) exist and are
    each scoped by `user_id`.
