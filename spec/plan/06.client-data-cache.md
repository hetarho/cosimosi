# 06. client data cache

> Client data cache - TanStack Query, connect-query, invalidation rules, optimistic writes, and cacheable GET reads.
> Scope: FE | Status: built

## Purpose

Later use-cases need a single client data layer instead of ad-hoc fetch state scattered through features. This unit
sets up TanStack Query and connect-query as the cache boundary for RPC data, records how optimistic writes and
invalidations work, and preserves the architecture rule that control state is not data state.

## Scope / Non-goals

**In scope**
- QueryClient providers for web and mobile app roots.
- connect-query integration with the generated API-client package from [03.rpc-transport](03.rpc-transport.md).
- Cache key conventions for RPC reads and user-scoped data.
- Invalidation helpers for future use-cases.
- Optimistic update pattern for unary writes: snapshot, optimistic patch, rollback on error, invalidate/refetch on
  settle.
- CDN/cacheable GET convention for idempotent unary reads.
- Test helpers for QueryClient isolation, fake transports, cache inspection, and the temporary `/test` harness.
- Values entries for cache timing defaults that are genuinely product/runtime tuning values.

**Non-goals**
- No product queries, product mutations, domain stores, or persisted offline cache.
- No polling, server streaming, websocket updates, or background sync framework.
- No cache of continuously-derived render state such as layout coordinates or per-frame buffers.
- No auth UI. The cache consumes session/user identity from [04.auth-session](04.auth-session.md) when available.
- No production CDN configuration; this unit only makes read RPCs compatible with cacheable GET.

## Grounding

- [ARCHITECTURE.md](../ARCHITECTURE.md) §2.7 requires idempotent unary reads to be available as HTTP GET and forbids
  server streaming.
- [ARCHITECTURE.md](../ARCHITECTURE.md) §3.2 separates control state in XState from data in Zustand/TanStack
  Query/refs.
- [ARCHITECTURE.md](../ARCHITECTURE.md) §4 requires derived state to be computed rather than stored per tick.
- [00.overview](00.overview.md) Phase 1 defines this as the client cache and invalidation unit.

## Design

### Query provider

Each app root owns one QueryClient provider — **both web and mobile**. Feature slices do not instantiate their own
long-lived QueryClient. Tests use a factory that creates an isolated client per test.

The shared cache facade lives in `@cosimosi/client-cache`:

- `createClientCacheQueryClient()` creates the app/test QueryClient with project defaults.
- `createClientCacheDefaultOptions()` exposes the generated default options for tests and future provider composition.
- `createClientCacheTestContext()` creates an isolated QueryClient plus a fake Connect transport for package tests and
  future `/test` panels.
- `inspectClientCache()` exposes query hashes, keys, status, and update timestamps without leaking app-specific data.

Default options come from generated values in `@cosimosi/config`:

- `client_cache.default_stale_ms`
- `client_cache.default_gc_ms`
- `client_cache.default_retry_count`
- `client_cache.optimistic_rollback_ms`

The generated TypeScript constants live at `packages/config/src/values.gen.ts`. The generated Go constants live at
`apps/api/internal/platform/values/values_gen.go`.

Web composition:

- `apps/web/src/app/App.tsx` nests `WebClientCacheProvider` inside `WebAuthProvider`.
- `WebClientCacheProvider` creates the authenticated Connect transport through `createApiTransport()`, wraps
  descendants in Connect Query's `TransportProvider`, and wraps them in TanStack Query's `QueryClientProvider`.
- The web API base URL resolves from `VITE_API_BASE_URL`, then `VITE_API_URL`, then the local API origin.

Mobile composition:

- `apps/mobile/src/app/App.tsx` nests `MobileClientCacheProvider` inside `MobileAuthProvider`.
- `MobileClientCacheProvider` creates the same generated transport facade, wraps descendants in Connect Query's
  `TransportProvider`, wraps them in TanStack Query's `QueryClientProvider`, and also exposes the transport through a
  mobile app context for RN-specific code that needs the transport directly.
- The mobile API base URL can be injected explicitly; its local fallback uses `localhost:8080` on iOS and
  `10.0.2.2:8080` on Android emulator builds.
- The mobile root depends on the same `@connectrpc/connect-query` React package as web so generated connect-query hooks
  read the same transport context on both app surfaces. Mobile React is kept on the same 19.2 patch line as web to avoid
  duplicate React runtimes in the hoisted workspace.

### Cache keys

RPC cache keys are derived from connect-query service/method descriptors and normalized request inputs. User-scoped
queries include authenticated user identity implicitly through the auth/transport boundary, not by manually adding
tokens to feature keys.

Feature code must not hand-build stringly cache keys when a generated query key helper exists. If a feature needs a
composite key, it exports the helper from its `api` segment.

The current API facade exports descriptor-derived helpers for the platform proof service:

- `createPlatformServiceQueryKey(transport?)`
- `createPlatformPingQueryKey(transport?)`
- `createPlatformPingQueryOptions(transport)`

`@cosimosi/client-cache` re-exports the current helpers as `platformCacheKeys`. Future feature RPCs follow the same
pattern from their API segment.

### Optimistic writes

Future unary mutations follow one pattern:

1. cancel relevant queries;
2. snapshot current cache values;
3. apply a small optimistic patch;
4. call the RPC;
5. rollback from snapshot on error;
6. invalidate or refetch affected reads on settle.

The cache layer supplies helper utilities but does not encode product-specific patch logic. Feature units own their
own optimistic patch functions.

`beginOptimisticMutation()` implements the shared lifecycle: cancel affected queries, snapshot current query existence
and state, clone cached data for rollback safety, apply feature-supplied patches, roll back from snapshots, and
invalidate affected reads on settle. Rollback re-clones snapshot data on every restore so in-place mutations after a
rollback cannot corrupt the saved snapshot. A patch that returns `undefined` removes that exact query, and rollback
restores either the previous query state or the previous absence. The helper carries
`client_cache.optimistic_rollback_ms` as the generated default rollback window, but product features still own their own
patch functions and mutation calls.

`setClientCacheData()` applies the same server-cache guard used by optimistic patches before writing data into Query.
An `undefined` update is treated as explicit exact-query removal instead of relying on TanStack Query's
`setQueryData()` updater semantics.

### Cacheable reads

For read-only RPCs, connect-query and the transport use GET where the method is declared idempotent. API handlers set
cache headers only when the method is safe to share and user isolation is respected. Authenticated user data must not
be cached publicly.

`createApiTransport()` keeps `useHttpGet: true` by default, so methods declared `NO_SIDE_EFFECTS` can use GET. Web and
mobile providers attach `createClientCacheRpcPolicyInterceptor()` to the transport, so the app request path preflights
registered cache policy against proto idempotency before Connect performs its POST-to-GET transform. The cache package
records the app-side convention through `defineRpcCachePolicy()`:

- GET is valid only for idempotent unary reads.
- non-idempotent writes remain POST.
- user-scoped RPC data cannot be marked for shared/public CDN caching.
- GET-eligible requests must have an explicit RPC cache policy entry, and `NO_SIDE_EFFECTS` methods cannot be registered
  with POST policies while `createApiTransport()` keeps HTTP GET enabled.

### Relationship to Zustand and refs

TanStack Query owns server data. Zustand may own client-local data collections or UI data that are not server cache.
Refs own high-frequency render buffers. XState machines own control states and ids only. This unit adds test examples
that reject ArrayBuffer-backed render-loop buffers, including buffers nested inside `Map`/`Set`, before they enter Query
cache. The guard rejects circular object graphs with a bounded error rather than recursing indefinitely. Protobuf
`bytes` values represented as `Uint8Array` remain valid server data.

## Acceptance Criteria

1. Web and mobile app roots have QueryClient providers and can use the same generated/connect-query client facade.
2. QueryClient test helpers create isolated clients and fake transports for unit tests and `/test` panels.
3. Cache key helpers are generated or exported through API segments; feature code does not rely on ad-hoc string keys.
4. Optimistic write helpers support snapshot, rollback, and settle invalidation without embedding product-specific
   logic.
5. Idempotent unary reads can use HTTP GET, while writes remain POST.
6. User-scoped data is never configured for public/shared CDN caching.
7. The cache layer does not store ArrayBuffer-backed render-loop data.
8. Cache timing defaults are read from generated values, not hardcoded in feature code.

## Policy / Values Impact

- **policy / ux:** no `spec/policy/**` tree exists in this repo yet. The active cache, optimistic-update,
  idempotent-read, and render-loop-data rules are reflected in this plan and in `@cosimosi/client-cache` tests.
- **spec/values.yaml:** owns the `client_cache.default_stale_ms`, `client_cache.default_gc_ms`,
  `client_cache.default_retry_count`, and `client_cache.optimistic_rollback_ms` values. FE code imports the generated
  `@cosimosi/config` constants instead of hardcoding timing defaults.
- **Dependencies:** depends on [03.rpc-transport](03.rpc-transport.md) and integrates with
  [04.auth-session](04.auth-session.md) for authenticated/user-scoped transports. It creates the Phase 1
  `client_cache.*` values that [11.config-values](11.config-values.md) will later harden as the general values system.
  The mobile QueryClient provider is available for [13.mobile-app-shell](13.mobile-app-shell.md) to compose into the
  full shell.
