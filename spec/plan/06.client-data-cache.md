# 06. client data cache

> Client data cache - TanStack Query, connect-query, invalidation rules, optimistic writes, and cacheable GET reads.
> Scope: FE | Status: planning

## Purpose

Later use-cases need a single client data layer instead of ad-hoc fetch state scattered through features. This unit
sets up TanStack Query and connect-query as the cache boundary for RPC data, records how optimistic writes and
invalidations work, and preserves the architecture rule that control state is not data state.

## Scope / Non-goals

**In scope**
- QueryClient providers for web and mobile app roots.
- connect-query integration with the generated API-client package from [03.rpc-transport](03.rpc-transport.md).
- Cache key conventions for RPC reads and user-scoped data.
- Invalidation helpers for future use-cases.
- Optimistic update pattern for unary writes: snapshot, optimistic patch, rollback on error, invalidate/refetch on
  settle.
- CDN/cacheable GET convention for idempotent unary reads.
- Test helpers for QueryClient isolation, fake transports, cache inspection, and the temporary `/test` harness.
- Values entries for cache timing defaults that are genuinely product/runtime tuning values.

**Non-goals**
- No product queries, product mutations, domain stores, or persisted offline cache.
- No polling, server streaming, websocket updates, or background sync framework.
- No cache of continuously-derived render state such as layout coordinates or per-frame buffers.
- No auth UI. The cache consumes session/user identity from [04.auth-session](04.auth-session.md) when available.
- No production CDN configuration; this unit only makes read RPCs compatible with cacheable GET.

## Grounding

- [ARCHITECTURE.md](../ARCHITECTURE.md) §2.7 requires idempotent unary reads to be available as HTTP GET and forbids
  server streaming.
- [ARCHITECTURE.md](../ARCHITECTURE.md) §3.2 separates control state in XState from data in Zustand/TanStack
  Query/refs.
- [ARCHITECTURE.md](../ARCHITECTURE.md) §4 requires derived state to be computed rather than stored per tick.
- [00.overview](00.overview.md) Phase 1 defines this as the client cache and invalidation unit.

## Design

### Query provider

Each app root owns one QueryClient provider — **both web and mobile**, wired in Phase 1 (the mobile provider is
composed into the mobile app shell, unit 13). Feature slices do not instantiate their own long-lived QueryClient.
Tests use a factory that creates an isolated client per test.

Default options come from generated values once [11.config-values](11.config-values.md) exists:

- `client_cache.default_stale_ms`
- `client_cache.default_gc_ms`
- `client_cache.default_retry_count`
- `client_cache.optimistic_rollback_ms`

The exact generated constant names follow the values generator's casing rules.

### Cache keys

RPC cache keys are derived from connect-query service/method descriptors and normalized request inputs. User-scoped
queries include authenticated user identity implicitly through the auth/transport boundary, not by manually adding
tokens to feature keys.

Feature code must not hand-build stringly cache keys when a generated query key helper exists. If a feature needs a
composite key, it exports the helper from its `api` segment.

### Optimistic writes

Future unary mutations follow one pattern:

1. cancel relevant queries;
2. snapshot current cache values;
3. apply a small optimistic patch;
4. call the RPC;
5. rollback from snapshot on error;
6. invalidate or refetch affected reads on settle.

The cache layer supplies helper utilities but does not encode product-specific patch logic. Feature units own their
own optimistic patch functions.

### Cacheable reads

For read-only RPCs, connect-query and the transport use GET where the method is declared idempotent. API handlers set
cache headers only when the method is safe to share and user isolation is respected. Authenticated user data must not
be cached publicly.

### Relationship to Zustand and refs

TanStack Query owns server data. Zustand may own client-local data collections or UI data that are not server cache.
Refs own high-frequency render buffers. XState machines own control states and ids only. This unit adds test examples
that fail or document when large collections are placed in machine context.

## Acceptance Criteria

1. Web and mobile app roots have QueryClient providers and can use the same generated/connect-query client facade.
2. QueryClient test helpers create isolated clients and fake transports for unit tests and `/test` panels.
3. Cache key helpers are generated or exported through API segments; feature code does not rely on ad-hoc string keys.
4. Optimistic write helpers support snapshot, rollback, and settle invalidation without embedding product-specific
   logic.
5. Idempotent unary reads can use HTTP GET, while writes remain POST.
6. User-scoped data is never configured for public/shared CDN caching.
7. The cache layer does not store per-frame coordinates, continuously-derived brightness, or other render-loop data.
8. Cache timing defaults are read from generated values, not hardcoded in feature code.

## Policy / Values Impact

- **policy / ux:** no current policy files are changed while this is only planned. When implemented, cache and
  optimistic-update conventions should be reflected in platform/frontend policy docs if available.
- **spec/values.yaml:** add `client_cache.default_stale_ms`, `client_cache.default_gc_ms`,
  `client_cache.default_retry_count`, and `client_cache.optimistic_rollback_ms` through
  [11.config-values](11.config-values.md). The numbers must not be hardcoded in FE/BE.
- **Dependencies:** depends on [03.rpc-transport](03.rpc-transport.md) and [11.config-values](11.config-values.md).
  It integrates with [04.auth-session](04.auth-session.md) for authenticated/user-scoped cache behavior. The mobile
  QueryClient provider is composed by the mobile app shell (unit 13).
