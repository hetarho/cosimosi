# 04. auth session

> Auth session - Supabase Auth session seam, explicit session state, and per-user request isolation.
> Scope: FE | Status: built

## Purpose

cosimosi stores private memory data, so every later use-case must run inside a known user boundary. This unit adds the
auth/session seam before product features arrive: frontend apps know whether a user is anonymous or authenticated, the
API can verify a bearer token and attach a user id to request context, and all future persistence work has one
isolation rule to follow.

## Scope / Non-goals

**In scope**
- Supabase Auth client setup for web and mobile behind a small app-owned auth facade.
- A shared session model and XState v5 session machine: bootstrapping, signed-out, signing-in, authenticated,
  refreshing, expired, and failed states.
- Web **and** mobile app providers that expose session snapshot, user id, access token, and auth actions to lower
  layers; the mobile session provider is composed into the mobile app shell (Phase 1 · unit 13).
- Connect client auth interceptor that attaches the current access token to RPC calls.
- API auth interceptor/middleware that validates Supabase JWTs, extracts the canonical user id, and stores it in
  request context for application services and repositories.
- Per-user isolation contract: future repositories must receive user id from context/use-case inputs and scope every
  query to it.
- Development/test auth adapters so local tests can run without real Supabase secrets.

**Non-goals**
- No branded onboarding, profile management, invite system, password policy, admin user management, or billing.
- No product routes are built beyond minimal auth proofing/callback handling.
- No database domain tables except auth-related persistence that Supabase owns externally.
- No row-level security hardening. RLS can be added later; this unit establishes app-level isolation first.
- No social/multi-user collaboration.

## Design

### Session model

The built frontend session model lives in `@cosimosi/auth` and is intentionally small:

- `status`: `bootstrapping | signedOut | signingIn | authenticated | refreshing | expired | failed`
- `userId`: present while authenticated, and preserved during an in-flight refresh so UI does not flash signed-out
- `accessToken`: held by the auth adapter and exposed only through `getAccessToken()`
- `expiresAt`: adapter-owned metadata surfaced to the auth facade, not feature stores
- `error`: surfaced for diagnostics and future auth UI

`packages/auth/src/session-machine.ts` owns the XState v5 lifecycle. Its context contains only the session snapshot
fields above, so JSON serialization never includes Supabase session objects, access tokens, refresh tokens, profile
objects, SDK clients, or storage handles. `packages/auth/src/facade.ts` wires an `AuthAdapter` to the machine and
guards async bootstrap/sign-in/refresh results plus adapter-driven events so a stale operation cannot re-authenticate
after a local sign-out. Terminal adapter events (`signedOut`/`expired`) are honored even while initial bootstrap is in
flight, so a resolving bootstrap cannot restore a session that was already revoked. If `adapter.signOut()` rejects, the
facade restores the prior local flags and machine snapshot before surfacing the error. Facades expose `dispose()` so
React providers can stop XState actors and Supabase subscriptions on unmount.

### Web and mobile adapters

Both apps consume the same auth facade and keep platform storage behind app/provider boundaries:

- `apps/web/src/app/auth-provider.tsx` creates a Supabase-backed facade when `VITE_SUPABASE_URL` and
  `VITE_SUPABASE_PUBLISHABLE_KEY` are configured, using browser-compatible Supabase session storage. Without those
  env vars it falls back to `FakeAuthAdapter` so local tests and hello-world boot do not require secrets.
- `apps/mobile/src/app/auth-provider.tsx` composes the same facade and accepts `MobileSupabaseAuthOptions` with a
  caller-supplied `secureStorage: SupabaseAuthStorage`. It configures the Supabase client with
  `detectSessionInUrl: false` and `flowType: 'pkce'`; job 13 supplies the native secure-storage implementation when
  the mobile app shell is composed.
- `packages/auth/src/fake-adapter.ts` provides the in-memory adapter used by tests and offline diagnostics.

No feature slice imports Supabase types or storage primitives. App roots provide the facade through React context;
lower layers can consume the facade snapshot/actions or the token accessor without seeing SDK objects.
The Supabase adapter refreshes the cached session before returning an access token when the token is expired or within
the generated near-expiry skew from `spec/values.yaml`.

### RPC integration

`packages/api-client/src/platform.ts` accepts an `auth` token provider on `createApiTransport`. The built
`createApiAuthInterceptor` reads `getAccessToken()` at call time and adds `Authorization: Bearer <token>` when a token
is available. App transport providers pass the auth facade directly to `createApiTransport`, so auth transport wiring
stays in the generated API-client seam instead of adding a second pass-through helper.

The API auth interceptor:

1. allows explicitly public methods such as `PlatformService.Ping`;
2. validates bearer JWTs for protected methods;
3. extracts the stable Supabase user id;
4. stores it in request context;
5. rejects missing/invalid tokens with the canonical unauthenticated RPC error.

`apps/api/internal/platform/AuthInterceptor` implements the allowlist and request-context user id storage.
`PlatformService.Ping` is public by default; tests can remove the allowlist to prove protected behavior. The runtime
composition root wires `SupabaseJWTVerifier` when `SUPABASE_PROJECT_URL` (or the `SUPABASE_URL` alias) is present. If
the URL is absent, the API starts with auth verification disabled even when a legacy JWT secret is present, rather than
crashing on a partial configuration. The verifier checks Supabase access tokens locally against the project's
`/auth/v1/.well-known/jwks.json`, caches the JWKS for the generated `supabase_auth.jwks_cache_ttl_ms` window, coalesces
cold/expired JWKS fetches, validates issuer/audience/expiration/subject, and rejects anonymous or non-`authenticated`
role tokens. Unknown `kid` values can trigger a rate-limited JWKS miss refresh using the generated
`supabase_auth.jwks_miss_refresh_interval_ms` window so key rotation works without letting arbitrary unsigned tokens
hammer Supabase. Legacy HS256 verification is available only when the server-only `SUPABASE_JWT_SECRET` is configured.
JWKS/upstream failures map to RPC `Unavailable`; malformed, expired, missing, or policy-rejected tokens map to the
canonical `Unauthenticated` error. RPC handlers do not parse tokens directly.

### Isolation contract

All future persisted domain rows must carry `user_id`, and all future repository queries must be scoped by that id.
`apps/api/internal/platform/UserScope` and `UserScopeFromContext` provide the isolation helper so future use-cases and
repository adapters can require authenticated identity before querying. This unit does not create product tables.

## Acceptance Criteria

1. Web and mobile have an auth provider/facade that exposes session status, user id, token access for transport, and
   auth actions without leaking Supabase objects into feature slices.
2. The session lifecycle is represented by an XState v5 machine whose context contains only serializable control data
   and ids.
3. RPC clients attach bearer tokens for protected calls and can be tested with an in-memory auth adapter.
4. The API validates Supabase JWTs for protected RPC methods and places the canonical user id in request context.
5. Public RPC methods remain callable without a token only when explicitly allowlisted.
6. Missing, expired, or invalid tokens return the canonical unauthenticated error shape.
7. A repository/use-case isolation helper or interface exists so later data units cannot accidentally query without a
   user id.
8. No onboarding flow, profile domain, invite flow, or product feature UI is introduced.

## Policy / Values Impact

- **policy / ux:** no `spec/policy/**` tree exists in this repo yet. The active rules are reflected in this plan and
  are consistent with [ARCHITECTURE.md](../ARCHITECTURE.md) §3.2 and §4.
- **spec/values.yaml:** `auth_session.access_token_refresh_skew_ms` controls how early the Supabase adapter refreshes
  access tokens before transport calls. `SUPABASE_PROJECT_URL`/`SUPABASE_URL`, optional server-only
  `SUPABASE_JWT_SECRET`, frontend publishable keys, JWT audience/issuer, and token storage choices are environment or
  adapter configuration, not product tuning constants.
- **Dependencies:** depends on [03.rpc-transport](03.rpc-transport.md) for authenticated RPC calls. It introduces the
  session machine now and should be reconciled with [07.state-machine-foundation](07.state-machine-foundation.md)'s
  broader machine catalog when that unit lands. Job 13 supplies the concrete native secure-storage implementation for
  the mobile shell.
