# 19. force sim layout

> Force-sim layout — a **pure, cross-app force-directed layout module** (`packages/force-sim`): a deterministic
> `tick(dt)` that reads the memory graph (neurons + synapses + activations) and writes coordinates, run in a Web
> Worker, off the render thread. Positions are **emergent, client-derived, and never stored** ([I5]).
> Scope: FE | Status: built
>
> A Layer-D (domain) unit of Epic A. The renderer ([14.rendering-foundation](14.rendering-foundation.md)) _reads_ the
> coordinates this module produces; it never authors them. The backend
> ([16.memory-aggregate-schema](16.memory-aggregate-schema.md)) owns the graph facts, never the layout.

## Purpose

The universe places memories and neurons in 3D **space** because the brain represents concepts and relationships as a
map, not by date ([PRD §2](../PRD.md) #2 · [V1]). That spatial arrangement must **emerge from relational structure** —
who shares which neurons, how strong the synapses are, how connected each node is — rather than be dictated or stored.

Every render unit in Epic A ([universe-canvas](23.universe-canvas.md), [star-neuron-rendering](24.star-neuron-rendering.md),
[latent-star-field](25.latent-star-field.md), [nebula-color-field](26.nebula-color-field.md)) needs coordinates to draw
anything, and both apps must draw the **same** universe from the same data. If each unit ran its own layout, or if
positions were persisted server-side, we would violate [I5] (positions are emergent, not stored) and diverge web from
mobile. This unit is the one place layout is computed: a pure `tick(dt)` that turns the domain graph into a coordinate
buffer, seeded so web and mobile reach identical positions, and run in a worker so the 60 fps render loop is never
driven from React state ([ARCHITECTURE §3.3](../ARCHITECTURE.md#33-rendering-stack--r3f-9--webgpurenderer--tsl)).

## Scope / Non-goals

**In scope**

- **A pure `packages/force-sim` module** (no DOM, no `three`, no React, no native deps —
  [ARCHITECTURE §3.5](../ARCHITECTURE.md#35-shared-packages--the-mobile-app)), shared verbatim by `apps/web` and
  `apps/mobile`. Signature: `tick(dt)` over in/out node + edge arrays → a `Float32Array` of coordinates.
- **A layout model** that encodes the product's placement rules as forces: radius from center = connectivity /
  self-relevance; center = self/identity; angle/cluster = relational structure (shared neurons); each episodic-memory
  "star" sits at the **weighted centroid** of its neurons; seed placement clusters co-created neurons and reuses the
  existing position of reused neurons; pattern-separation repulsion keeps nodes from collapsing.
- **A z-layer coordinate model.** Epic A computes layout only in the **hippocampus band**
  (`force_sim.hippocampus_z_min`..`force_sim.hippocampus_z_max`, i.e. z 0–10). The **neocortex band** (z 15–25) is
  reserved here as bounds + the coordinate contract; nothing is placed there in Epic A ([C5], [C6], [V9]).
- **Seeded determinism ("golden-parity").** A seeded PRNG (`force_sim.seed`) makes a given input graph produce byte-
  identical coordinates on any client; golden fixtures pin that cross-platform reproducibility.
- **A worker-hosted execution contract.** The module is designed to run inside a Web Worker (Barnes-Hut, O(N log N));
  the renderer reads the resulting buffer into `InstancedMesh.matrix` / uniforms and never drives the simulation from
  React state.
- The `force_sim.*` tuning **scalars** (charge, link distance, center strength, repulsion, alpha decay, z bounds,
  seed) sourced from `values.yaml`. The **formulas** that consume them stay in code.

**Non-goals**

- **No stored positions, no server authority.** Coordinates are never persisted, never sent to the server, never a
  proto field. The server owns neurons / synapses / activations / centrality facts only ([I5],
  [ARCHITECTURE §2.9](../ARCHITECTURE.md#29-forbidden-patterns) #8, [ARCHITECTURE §4](../ARCHITECTURE.md#4-cross-cutting-patterns)).
  This is **not** a Go position authority.
- **No rendering.** This module emits numbers, not meshes. Wiring the worker + reading coordinates is
  [23.universe-canvas](23.universe-canvas.md); projecting nodes to bodies is
  [24.star-neuron-rendering](24.star-neuron-rendering.md).
- **No memory↔memory attraction or edge.** There is no force, edge, or spring between two episodic memories; their
  proximity is emergent from shared neurons + synapse strength ([I4], [I6], [L6], [L7]).
- **No emotion input.** The simulation consumes graph structure only. Valence / arousal / mood / intensity never enter
  the layout ([I3], [M7]).
- **No neocortex placement.** The upper band is a reserved coordinate contract in Epic A; the "copy x,y, raise z" gist
  ascent and its (force-sim-free) stability belong to Epic E ([C6], [V9]).
- **No data fetch, no cache, no clock.** The graph arrives as an argument; the universe read is
  [23.universe-canvas](23.universe-canvas.md); universe time is Epic B.
- **No active pattern-separation logic.** Only the baseline repulsion force; deliberately splitting similar
  constellations is a reserved slot ([E11]).

## Design

### The module shape (`packages/force-sim`)

A pure TypeScript module in `packages/force-sim` — no DOM, no `three`, no React, no native imports — so it compiles and
runs identically on web and mobile and is worker-safe. It exports a stateful simulation created from the graph plus the
generated `force_sim.*` constants, advanced by a pure `tick(dt)`:

- **Input** = the memory **graph**, mapped from the domain mirror (never proto directly): the set of **neurons** (with
  a per-neuron **connectivity / degree** fact), the set of **synapses** (each an undirected neuron↔neuron edge carrying
  a **strength** in 0..1), the set of **episodic memories**, and the **neuron activations** that link each memory to its
  neurons with a **co-membership weight** (the pull weight for the centroid). Optionally, a **seed hint** per node for
  placement (see _Seed placement_). No emotion, no color, no time-derived state enters.
- **Output** = a `Float32Array` of coordinates (a flat `[x, y, z, x, y, z, …]` buffer), indexed by a stable node-index
  map the caller holds, for **both** neuron nodes and episodic-memory star nodes. The default is a **single interleaved
  buffer with the node-index map owned and returned by this module**. As built, `FORCE_SIM_COORDINATE_STRIDE = 3`,
  neuron nodes are indexed first in input order, episodic-memory nodes follow in input order, and each index maps to
  offset `index * 3` in the interleaved `[x, y, z]` buffer. The contract is: the renderer copies straight into
  `InstancedMesh.matrix` / uniforms with no per-frame allocation.
- **Advance** = `tick(dt)`: one integration step given elapsed time; deterministic given the same input + accumulated
  ticks + seed. A `tick_alpha_decay` term cools the simulation toward a settled layout so it converges rather than
  jittering forever.

The module is a **domain service in the DDD sense — a pure function over the aggregates**
([ARCHITECTURE §2.5](../ARCHITECTURE.md#25-aggregate-boundaries-a-structural-decision)). Purity is the property that
lets a client compute layout the server never has to.

### Execution: a Web Worker, off the render thread

The simulation runs in a **Web Worker** (mobile: the RN-equivalent off-thread host), posting the coordinate buffer to
the main thread ([ARCHITECTURE §3.3](../ARCHITECTURE.md#33-rendering-stack--r3f-9--webgpurenderer--tsl)). The renderer's
`useFrame` **reads** the latest buffer into instances/uniforms; it **never** drives the tick from React state and never
re-renders React at 60 fps. Neighbor forces use a **Barnes-Hut** approximation so the step is O(N log N), not O(N²).
The worker plumbing itself (message protocol, transferable buffers, worker lifecycle) is owned by
[23.universe-canvas](23.universe-canvas.md); this unit owns the pure module the worker hosts and keeps its API
worker-friendly (structured-cloneable in, transferable buffer out). As built, `tick(dt)` returns the module-owned
snapshot buffer for direct read use, and `tick(dt, outputBuffer)` writes the same coordinates into a caller-owned
`Float32Array` for transfer. A worker host can therefore transfer caller-owned buffers without detaching the simulation's
internal snapshot.

### Layout rules (the forces)

Positions are the settled state of these forces — never assigned directly:

- **Radius from center = connectivity / self-relevance.** Well-connected nodes (high synapse-weighted degree) sit
  **near the center**; sparsely connected nodes drift outward. The **center is self / identity, not time** — time is a
  link signal (via synapse structure), never a position axis ([V1], [I7]). Realized as a `center_strength` pull toward
  the origin, modulated by connectivity.
- **Angle / cluster = relational structure.** Where a node lands _around_ the center comes from **shared neurons** —
  nodes that co-participate cluster together — via `link_distance` springs along synapse edges and a general `charge`
  interaction. Constellations are **emergent** from shared neurons + synapse strength, not authored ([V1], [E9]).
- **Episodic-memory star = the weighted centroid of its neurons.** A memory node is pulled toward the position of its
  neurons, each neuron weighted by its **activation co-membership weight** — **stronger neurons pull harder** ([E8]).
  A memory has no independent position force of its own; it follows its neurons. There is **no** force between two
  memory nodes ([I4], [L7]).
- **Seed placement.** New nodes start from a **seed**, not a stored absolute position ([E7]). Neurons **co-created** in
  the same episodic memory seed **near each other**; a **reused existing neuron** seeds **at / near its current
  position** so the universe stays stable across launches; when no cue exists, the seed is drawn from the seeded PRNG.
  The seed is only a starting point — the final position emerges from the forces ([I5]). Latent "gray star" positions
  are a **rendering-only** background ([E7a], [V7], owned by [25.latent-star-field](25.latent-star-field.md)); their
  "awaken near a recent neuron" is an entry animation, and if that supplies a seed anchor for a newly activated neuron,
  it is a **seed hint** only — force-sim treats it as one more starting point, not a fixed coordinate.
- **Pattern-separation repulsion.** A baseline `repulsion` force keeps nodes from collapsing into a single mass — the
  visual implementation of pattern separation ([E11]). Active separation of _similar-but-distinct_ constellations is a
  **reserved slot** — Epic A relies on baseline repulsion + conservative merging only; a later epic fills the slot if
  the universe proves visually confusable.

The exact force composition, integrator, and how connectivity maps to the center pull are **formulas → code**, not
config; only their coefficients are `values.yaml` scalars (below).

### Emotion is never an input

The simulation signature accepts graph structure only. Valence, arousal, mood, and intensity are **not** parameters and
have no path into position or connection ([I3], [M7]). Emotion drives color (and arousal drives size/forgetting) at the
_visual output_ stage, downstream of and disjoint from layout — so it cannot collapse the layout into a single cluster.

### z-layer coordinate model (Epic A = hippocampus only)

The universe is a **z-axis 2-store (CLS)** space ([C5], [V9]):

- **Hippocampus band** — z within `force_sim.hippocampus_z_min`..`force_sim.hippocampus_z_max` (0..10). Epic A computes
  the full force layout **here**: episodic memories are created in this band and latent stars are scattered in it. The
  simulation clamps / biases its z output into this band.
- **Neocortex band** — z within `force_sim.neocortex_z_min`..`force_sim.neocortex_z_max` (15..25). **Reserved now** as
  the coordinate contract + bounds; **no force-sim runs there**. When gist ascent ships (Epic E), a neocortex node
  **copies the x,y** its hippocampal memory settled at and **raises only z** — the neocortex layer is stable and has no
  simulation of its own ([C6], [V9], [I5]). Epic A places nothing in this band; it only fixes the bounds and the "x,y
  copied, z raised" model so Epic E slots in without reshaping the coordinate space.

**Decided (2026-07):** a node's z **is a settled force output** within the hippocampus band (clamped/biased into the
band's bounds) — the universe is genuinely 3D from Epic A, camera rotation reads real depth, and the Epic E neocortex
transition ("x,y copied, z raised") slots in without a layout-mode switch.

### Determinism — "golden-parity"

Layout is client-computed on two platforms, so it must be **reproducible**, not merely "close":

- A **seeded PRNG** keyed by `force_sim.seed` drives every stochastic choice (initial seed placement, tie-breaks).
  Given the same input graph + the same seed + the same accumulated tick sequence, the module produces **byte-identical**
  coordinates on web and mobile.
- **Golden fixtures** — canonical (graph, seed) → coordinate-buffer pairs — pin this cross-platform determinism as a
  test, the same way [packages/memory-logic](../ARCHITECTURE.md#35-shared-packages--the-mobile-app) golden fixtures pin
  the read-time-derived math. This is the precise meaning of "golden-parity" for layout: **web == mobile determinism via
  a seeded PRNG**, verified by fixtures. It is explicitly **not** a Go/server position authority — positions remain
  client-only and emergent ([I5], brief §7).

### Re-implementation note (e.g. Flutter)

Because the module is pure and the forces + seed are fully specified by code + the `force_sim.*` scalars, another client
can re-implement `tick(dt)` and, given the same graph + seed, reproduce the same golden fixtures — the layout is a
property of the input and the constants, not of any one runtime.

## Acceptance Criteria

1. `packages/force-sim` is a pure module with **no** DOM, `three`, React, or native imports, and is consumed
   identically by `apps/web` and `apps/mobile`.
2. It exposes a `tick(dt)` that takes node + edge arrays (neurons, synapses with strength, episodic memories, neuron
   activations with weights, per-neuron connectivity) and returns a `Float32Array` of coordinates; the renderer reads
   that buffer into `InstancedMesh.matrix` / uniforms and does not drive the tick from React state.
3. Positions are **never** persisted, never sent to the server, and never a proto field; the server owns
   neurons / synapses / activations / centrality facts only ([I5], [ARCHITECTURE §2.9](../ARCHITECTURE.md#29-forbidden-patterns) #8,
   [§4](../ARCHITECTURE.md#4-cross-cutting-patterns)).
4. A node's **radius from center** is a function of its connectivity / self-relevance (well-connected → near center);
   the **center is self/identity, not time** ([V1], [I7]).
5. An episodic-memory node settles at the **neuron-weighted centroid** of its neurons — stronger (higher activation
   weight) neurons pull harder ([E8]) — and has no independent position force.
6. There is **no** force, edge, or spring between two episodic-memory nodes; memory proximity is emergent from shared
   neurons + synapse strength ([I4], [I6], [L6], [L7]).
7. **Emotion never affects layout**: valence / arousal / mood / intensity are not accepted as inputs and have no path
   into position or connection ([I3], [M7]).
8. New nodes start from a **seed**: co-created neurons seed near each other, a reused existing neuron seeds near its
   existing position, and the final position emerges from the forces ([E7], [I5]); baseline **repulsion** prevents nodes
   from collapsing into one mass ([E11]).
9. All computed layout is confined to the **hippocampus z-band** (`force_sim.hippocampus_z_min`..`hippocampus_z_max` =
   0..10); the **neocortex z-band** (15..25) is reserved as bounds + the "copy x,y, raise z" coordinate contract with
   no force-sim, and Epic A places nothing there ([C5], [C6], [V9]).
10. Given the same input graph + `force_sim.seed` + tick sequence, the module produces **identical** coordinates on web
    and mobile, verified by **golden fixtures** — a client-only determinism guarantee, not a server position authority
    ([I5], brief §7).
11. The simulation is designed to run in a **Web Worker** off the render thread with Barnes-Hut O(N log N) neighbor
    forces (its worker-friendly API and buffer contract are honored by the worker host in
    [23.universe-canvas](23.universe-canvas.md)).

- **tech/** — the as-built layout rules ("positions are emergent, client-derived, and **never stored**; layout is a
  pure `tick(dt)` run off the render thread; cross-platform determinism via a seeded PRNG + golden fixtures") reinforce
  [I5] and [ARCHITECTURE §2.9](../ARCHITECTURE.md#29-forbidden-patterns) #8 / [§4](../ARCHITECTURE.md#4-cross-cutting-patterns).
  They landed in [tech/force-sim.md](../tech/force-sim.md).
- **policy / ux** — the rule "layout is emergent from relational structure only — emotion never touches position or
  connection" reinforces [I3] / [I7]; it landed in [policy/ux/layout.md](../policy/ux/layout.md).
- **spec/values.yaml** (enumerated now; generated at implement via `pnpm gen:values`; formulas stay code):
  - `force_sim.charge` (0.035) — inter-node charge / repulsion-attraction interaction coefficient.
  - `force_sim.link_distance` (12) — target rest length of a synapse-edge spring.
  - `force_sim.center_strength` (0.018) — strength of the center (self/identity) pull that turns connectivity into
    radius.
  - `force_sim.repulsion` (2.4) — baseline node-node repulsion (pattern-separation, prevents collapse) [E11].
  - `force_sim.tick_alpha_decay` (0.022) — per-tick cooling term so the layout converges to a settled state.
  - `force_sim.hippocampus_z_min` (0), `force_sim.hippocampus_z_max` (10) — hippocampus z-band bounds [C5][V9].
  - `force_sim.neocortex_z_min` (15), `force_sim.neocortex_z_max` (25) — neocortex z-band bounds, reserved for Epic E
    [C6][V9].
  - `force_sim.seed` (190019) — the deterministic PRNG seed for golden-parity (web == mobile).
  - _(Excluded from values — code:_ the force formulas, the integrator, the connectivity→radius and centroid math, and
    the buffer packing. Node **counts** are derived array lengths, not declared.)*
- **Dependencies:** depends on [16.memory-aggregate-schema](16.memory-aggregate-schema.md) alone (which owns the
  neuron / synapse / activation / connectivity **facts** this module consumes) — this is a pure package with no
  renderer dependency. It **pairs with** [14.rendering-foundation](14.rendering-foundation.md) (the renderer that
  _reads_ these coordinates into instances/uniforms — the consumer, not an upstream), [23.universe-canvas](23.universe-canvas.md)
  (wires the worker + reads the coordinate buffer) and [24.star-neuron-rendering](24.star-neuron-rendering.md)
  (positions bodies from it). Its output is consumed by, and
  its determinism verified alongside, every Epic-A render unit.
