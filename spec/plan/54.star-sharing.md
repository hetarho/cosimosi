## Purpose

The product wants, eventually, to let someone show a friend a single star — a specific episodic memory — the way you
might read one page of a diary aloud. [S1] names it: *친구에게 별 보내기, 일화기억 단위로 공유.* It is deliberately
**out of v1** ([PRD.md](../PRD.md) §8.3 — social is a recorded non-goal). But "out of v1" is not "unplanned": the PRD's
scope discipline ([PRD.md](../PRD.md) §8.2) is *후순위 = 자리 예약* — a deferred capability keeps its slot so the v1
structure knows what will arrive, later work slots in without a redesign, and no intervening decision quietly builds
something that collides with the empty slot.

This unit is that reserved slot. It exists so that the v1 architecture makes exactly the commitments sharing will need —
a **separate `internal/sharing` bounded context** ([ARCHITECTURE.md](../ARCHITECTURE.md) §2.2, per-context isolation),
an **async, one-way** share of an *episodic-memory unit*, and strict **per-user aggregate isolation** — and **no more**.
It builds nothing: no table, no RPC, no UI, no cross-user read. Its whole job is to name the seam precisely enough that
Epics A–I never assume single-owner data in a way sharing would later have to unwind, and to keep the reader from
confusing star-sharing with the v1 friend-**invite** growth path (a distinct feature — see Non-goals).

**Decided (2026-07):** star sharing is authored as a concise placeholder (per brief CC12): scope = the seam + why
deferred + what must not be precluded. It does not specify the share transport, the recipient's view, revocation, or
any tuning number. Those are settled when the feature is actually built (a future plan/change → `/implement-job`).

## Scope / Non-goals

**In scope** (reserving the slot only — nothing is built)
- **The `internal/sharing` bounded-context seam.** Reserve a NEW context ([ARCHITECTURE.md](../ARCHITECTURE.md) §2.2;
  brief CC8) that will own share **grants** — never a copy of the memory aggregate. It will reference an
  `episodic_memory_id` (+ owner `user_id`) by **id across a context boundary**, reached through a consumer-owned port at
  the composition root; `sharing` never imports `internal/memory`'s internals and `internal/memory` never imports
  `sharing` ([ARCHITECTURE.md](../ARCHITECTURE.md) §2.4).
- **The share unit = one episodic-memory star** ([S1]). The shareable thing is a single `EpisodicMemory` (the star and
  the trace it renders), **not** a neuron, a synapse, a constellation, a diary, or a whole universe. Public-universe
  *browsing* is the separate sibling [55.public-universe](55.public-universe.md) ([S2]).
- **The async, one-way model** ([PRD.md](../PRD.md) §5.8 — *비동기·일방·후순위*). A share is a **snapshot handed one
  direction**, sender → recipient. It is not a live subscription, not a two-way sync, not a collaboration session; the
  recipient never writes back into the sender's universe.
- **A reserved FE slot.** A `features/share-star` slice ([ARCHITECTURE.md](../ARCHITECTURE.md) §3, Feature-Sliced
  Design) + a `sharing` entity mirror are **named but not created**, so the "이 별을 공유" affordance later hangs off the
  star-detail panel ([35.star-detail-panel](35.star-detail-panel.md)) without reshaping it.
- **The non-preclusion contract** (see Design): the concrete list of v1 invariants and boundaries this slot must not
  break when it is eventually filled.

**Non-goals** (explicitly NOT in this plan, and mostly NOT in v1)
- **No implementation of any kind.** No migration, no `sharing` table, no proto service, no RPC, no handler, no FE
  slice, no UI, no worker job. Status is **planning**; this plan reserves, it does not build ([PRD.md](../PRD.md) §8.3).
- **Not the friend-invite growth path.** Friend **invite-for-growth** (초대자·가입자 양쪽에게 별가루 지급) is a **v1**
  feature and lives in **Epic F earn** — [44.earn-spend-usecase](44.earn-spend-usecase.md) ([G3]), with the abuse-guard
  slot [G6]. Inviting a friend to *join* is unrelated to *sharing a star* and is out of scope here
  ([PRD.md](../PRD.md) §8.3 explicitly separates them: *단 친구 초대(그로스)는 v1 [G3] — 별 공유와 별개*).
- **No public universe.** Browsing someone's public universe ([S2]) is the sibling [55.public-universe](55.public-universe.md).
- **No cross-user reads in v1.** Every v1 query stays scoped to the authenticated user (the [16.memory-aggregate-schema](16.memory-aggregate-schema.md)
  `UserScope` convention); no v1 read may return another user's rows. Sharing does **not** loosen this — it will add a
  *new, grant-mediated* read path when built, not widen the existing per-user ones.
- **No shared mutation, no reconsolidation across users.** A recipient viewing a shared star never triggers the
  sender's recall/reconsolidation ([I2][I8]); a shared snapshot is not a live handle into the sender's aggregate.
- **No transport / privacy / revocation spec.** Delivery channel (link vs. in-app), consent/visibility model, expiry,
  revocation, and any Twinkle cost are deferred to the build-time plan/change; this unit does not decide them.
- **No new tuning numbers.** A placeholder introduces no `values.yaml` keys (see Policy / Values Impact).

## Design

This is a **reservation**, so the Design fixes commitments and boundaries, not mechanisms. It deliberately under-specifies.

### The reserved bounded context (`internal/sharing`)
When built, star sharing will live in a **new `internal/sharing` bounded context** ([ARCHITECTURE.md](../ARCHITECTURE.md)
§2.2; brief CC8) — a peer of `internal/memory` and `internal/twinkle`, not a table inside `memory`. It will own the
**share grant** (who shared which star with whom, in one direction), and nothing of the memory aggregate itself.

- **Reference by id across the boundary.** A grant references an `episodic_memory_id` (+ its owner `user_id`); it never
  duplicates the memory's columns ([ARCHITECTURE.md](../ARCHITECTURE.md) §2.5). To render a shared star the sharing
  context reads memory facts through a **consumer-owned port** (`SharedMemoryReader`-style), bound at the composition
  root (`cmd/api`); `sharing` never imports `memory`'s internals and `memory` never imports `sharing`
  ([ARCHITECTURE.md](../ARCHITECTURE.md) §2.4; brief CC8).
- **Its own service surface.** A future `sharing.v1` service will carry the share/accept/revoke RPCs; it is **not**
  added to `memory.v1.MemoryService`. Reserving a separate service keeps the memory contract clean and lets social ship
  as an isolated context later.

**Decided (2026-07):** the seam is reserved as a *separate context with a separate service*, not as extra columns on
`episodic_memories` or extra RPCs on `MemoryService` — so v1 never bakes a single-owner assumption that a bolted-on
share column would later contradict, and social can be added (or dropped) without touching the memory aggregate.

### The share model — async, one-way, episodic unit
The shape v1 must leave room for, fixed now so nothing collides with it later:

- **Unit = one episodic-memory star** ([S1]). A share targets a single `EpisodicMemory`. Not a neuron, synapse,
  constellation, diary, or universe. (Whole-universe browsing is [55.public-universe](55.public-universe.md).)
- **Direction = one-way** ([PRD.md](../PRD.md) §5.8). Sender → recipient. No two-way sync, no shared editing session.
  The recipient receives a view, never a write handle.
- **Timing = async** ([PRD.md](../PRD.md) §5.8). A share is issued and consumed on the recipient's own schedule; it is
  not a live, real-time collaboration ([PRD.md](../PRD.md) §8.3 excludes real-time multi-user).
- **What the recipient sees = a read-only snapshot** of the sender's star sufficient to render it (the trace: shape,
  emotion/color, brightness, the current text as the sender chose to share it) — computed the same way the owner's read
  computes it, **re-laid-out on the recipient's client** by force-sim from the shared graph facts ([I5]), never shipped
  with stored coordinates. The precise snapshot fields, and whether provenance/original-diary travel with a share, are
  a build-time decision ([R8a][D4] interplay), not fixed here.

### The non-preclusion contract (what v1 must NOT break)
The point of reserving the slot is that v1 keeps these true, so filling it later needs no redesign:

1. **Per-user aggregate isolation stays intact.** Every v1 table has `user_id`; every v1 query is user-scoped
   ([16.memory-aggregate-schema](16.memory-aggregate-schema.md)). No v1 feature may return another user's rows. Sharing
   will add a **grant-mediated** cross-user read *later* — it must not be pre-empted by a v1 query that already leaks
   across users, and it must not require loosening any existing user-scope.
2. **Aggregate boundaries hold.** A shared star is referenced **by id**, never by copying the memory aggregate into
   another context or another user's rows ([ARCHITECTURE.md](../ARCHITECTURE.md) §2.5). No v1 code may assume the memory
   aggregate is single-context-only in a way that forbids an external reader port.
3. **The shared star is read-only downstream** ([I2][I8][I1]). A recipient viewing it triggers no recall,
   reconsolidation, decay write, deletion, or `Diary` mutation in the sender's universe; revoking a share removes the
   grant only, never the memory.
4. **No stored positions creep in** ([I5]). The temptation to persist coordinates "so the friend sees the same layout"
   is refused up front: a shared star is re-laid-out by the recipient's force-sim from graph facts.
5. **Meaning layer stays immutable across users** ([I11]). A recipient's palette/customization cannot overwrite the
   shared star's emotion or meaning.

### Golden-parity stance
None. This plan introduces **no** shared FE↔BE math and **no** cross-platform-deterministic computation — it builds
nothing. (When sharing is built, rendering a shared star reuses the *existing* golden-parity layout/read math of
[19.force-sim-layout](19.force-sim-layout.md) and the memory read; it adds none of its own.)

### FE placement (reserved)
Per Feature-Sliced Design ([ARCHITECTURE.md](../ARCHITECTURE.md) §3), star sharing will later be a
`features/share-star` slice consuming a `sharing` **entity** mirror, with its affordance surfaced from the star-detail
panel ([35.star-detail-panel](35.star-detail-panel.md)). Web and mobile mirror each other (§3.5). **None** of these are
created in this plan — they are named so the placement is unambiguous when the feature is built.

## Acceptance Criteria
> This is a placeholder plan; its criteria assert that the **reserved seam holds and nothing is built**, not that a
> feature works.
1. This plan is **planning/deferred**: it introduces **no** migration, `sharing` table, proto/RPC, handler, FE slice,
   UI, or worker job. Star sharing remains a recorded non-goal for v1 ([PRD.md](../PRD.md) §8.3).
2. The reserved home is a **new `internal/sharing` bounded context** (a peer of `memory`/`twinkle`), owning **share
   grants** only, referencing an `EpisodicMemory` **by id** through a consumer-owned port at the composition root —
   never copying the memory aggregate or importing `memory`'s internals, and never imported *by* `memory`
   ([ARCHITECTURE.md](../ARCHITECTURE.md) §2.2/§2.4/§2.5; brief CC8).
3. The reserved share model is **async, one-way, at the single-episodic-memory unit** ([S1],
   [PRD.md](../PRD.md) §5.8): sender → recipient, read-only snapshot, no two-way sync and no real-time collaboration.
4. The plan **distinguishes star sharing from the v1 friend-invite growth path**: invite-for-growth is [G3] in
   [44.earn-spend-usecase](44.earn-spend-usecase.md) (Epic F), not here ([PRD.md](../PRD.md) §8.3).
5. The **non-preclusion contract** is stated and honored by v1: per-user aggregate isolation and user-scoped queries
   stay intact, aggregate boundaries hold (reference by id), the shared star is read-only downstream (no recall /
   reconsolidation / mutation on the sender), no stored positions, and the meaning layer stays immutable across users
   ([I1][I2][I5][I8][I11]).
6. The FE placement is **named but not built**: a `features/share-star` slice + `sharing` entity mirror off the
   star-detail panel ([35.star-detail-panel](35.star-detail-panel.md)), mirrored web↔mobile
   ([ARCHITECTURE.md](../ARCHITECTURE.md) §3/§3.5).
7. The plan pairs with [55.public-universe](55.public-universe.md) as the two deferred Epic J placeholders, jointly
   reserving the `sharing` context ([S1][S2]).
