# 24. 별 변천사 (evolution-history-ui)

> 별 하나의 형태·색·밝기가 변해 온 길을 스크럽 타임랩스로 보고, 각 변화의 계기(회상/새 이웃/야간 요지)를 표시하며, 옆에는 늘 불변 원본 일기를 둔다. | 범위: FE(+소량 BE read) | 선행: 23 | 차단: —

## 목적 (Why)
concept §변천사의 약속 — "내 기억은 살아 움직이지만, 그날 내가 쓴 말은 거기 그대로 있다" — 을 화면으로 만든다. 23이 `evolution_history`에 **추가만 되는(append-only)** 스냅샷을 쌓아 왔다(회상·새 이웃·야간 요지로 별이 재성형될 때마다 한 행). 24는 그 로그를 한 별 단위로 **읽어**, 버전을 시간순으로 스크럽하는 타임랩스를 그린다. 각 버전은 그 시점의 `form_seed_delta`·`hue_shift`·`brightness`로 `VizStar`를 재현하고, 변화 지점마다 *무엇이 별을 바꿨는지*(회상/새 이웃/야간 요지)를 가볍게 라벨링한다. 그리고 화면 한편에는 `RecallMemory`가 반환하는 **불변 원본 `Record`**(records 출처)를 나란히 고정해, 변한 것(별)과 변하지 않는 것(그날의 내 말)을 동시에 보게 한다. 읽기 전용·추가 전용 — 수정 경로 없음(헌법1).

## 범위
- **포함:**
  - BE `memory`(단일 `MemoryService`): `GetEvolutionHistory(memory_id)` **읽기 전용** unary RPC — 23의 `evolution_history`를 `version` 오름차순으로 `repeated EvolutionSnapshot`로 반환. `WHERE user_id = $ctx`로 격리(헌법). INSERT/UPDATE 없음.
  - BE `db/queries/memory.sql`(수정): `ListEvolutionHistory :many` 셀렉트 추가(append-only 테이블에서 읽기만; `RETURNING *` 없음).
  - FE `features/evolution`(신규): `GetEvolutionHistory` unary 호출 + 스냅샷 뷰모델 매퍼(api), 스크럽 슬라이더 + 버전별 `VizStar` 재현 + 계기 라벨 + 옆에 고정한 불변 원본 패널(ui), 순수 변천사 셀렉터/스텝 누적(model — three/DOM 미의존, 헌법4).
  - FE `entities/star/ui/VizStar.tsx`(재사용): 버전 리플레이는 `seed = base_seed + form_seed_delta`, `brightness`, hue는 부모 래퍼의 `hue-rotate(hueShift)` — `VizStar` 자체는 변경 없이 재사용(시그니처 보존). 필요 시 형태 단순화(야간 요지) 표현만 prop 없이 brightness/hue로 근사.
  - FE `features/recall/ui/MemoryPanel.tsx`(수정): 회상 패널에 "변천사 보기" 진입점 추가(선택 별의 변천사 화면을 연다).
  - FE: 변천사를 **31의 오버레이 셸(시트/패널, `panel='evolution'`)** 로 띄운다 — 별도 페이지/라우트로 우주를 떠나지 않는다(우주 캔버스 영속, 뒤에서 그 별 보임). 진입점은 회상 패널의 "변천사 보기".
- **비목표(Out of scope):**
  - **재공고화 재성형 로직·`evolution_history` 적재 자체 → 23(reconsolidation-reshaping).** 24는 23이 쌓은 데이터를 **읽기만** 한다(쓰기 경로 신설 금지).
  - 야간 배치(전체 재안정화·요지·가지치기)·`trigger='nightly_gist'` 적재 → 27(nightly-consolidation). 24는 그 trigger 값을 **표시만** 한다.
  - 경쟁적 할당·관련성 감쇠·요즘 상태 → 22/26/25. 변천사는 그 결과로 쌓인 스냅샷을 보여줄 뿐 계산하지 않는다.
  - 원본 일기 본문 fetch 자체 → 11(`RecallMemory`)이 담당. 24는 그 `Record`를 **재사용**해 옆에 둘 뿐(새 RPC 신설 금지).

## 설계 요점

**계약(proto — 23 권위, 24가 확인/추가).** 23 SHARED 결정상 `Star`에 reshape 상태(`brightness_offset`/`hue_shift`/`form_seed_delta`/`version`)와 `EvolutionSnapshot`·`GetEvolutionHistory`가 "23 또는 24에서" 정의될 수 있다. **먼저 23이 이미 추가했는지 확인**하고, 없으면 24가 아래를 추가한다(중복 정의 금지). 단일 `MemoryService` 유지(별도 서비스 신설 금지), unary 전용(헌법6).
```proto
// proto/cosimosi/v1/memory.proto — service MemoryService (02 권위)
// 변천사 한 스냅샷 = evolution_history 한 행(append-only, 헌법1/2). 좌표 필드 없음(헌법: 위치는 클라).
message EvolutionSnapshot {
  int32  version          = 1;  // 0=최초, 이후 증가
  double brightness       = 2;  // 그 시점의 별 밝기(0.05 floor 이상)
  double hue_shift        = 3;  // 원본 감정색 기준 누적 색조(±28° 이내)
  double form_seed_delta  = 4;  // base_seed에 더해 형태를 미세 변형(같은 기억의 변주)
  string trigger          = 5;  // 'recall' | 'new_neighbor' | 'nightly_gist'
  double pe               = 6;  // 예측오차(reshape 크기 근거; 표시 보조)
  int32  dir              = 7;  // +1 강화 / -1 약화 / 0 중립
  string created_at       = 8;  // RFC3339
}
message GetEvolutionHistoryRequest  { string memory_id = 1; }
message GetEvolutionHistoryResponse { repeated EvolutionSnapshot snapshots = 1; } // version ASC
// service MemoryService { ... 기존 ...
//   rpc GetEvolutionHistory(GetEvolutionHistoryRequest) returns (GetEvolutionHistoryResponse); // read-only
// }
```

**BE 읽기 전용 셀렉트(`db/queries/memory.sql`, 수정).** `ListDormant`/`GetRecordByMemory`와 동일하게 `user_id` 격리 + `RETURNING *` 없음. append-only 테이블이므로 read만; 별이 존재하는지(권한)는 `memory_id`+`user_id` 매칭으로 보장.
```sql
-- name: ListEvolutionHistory :many
-- A star's append-only evolution log (spec 23 writes it; 24 only reads). user_id-scoped
-- (constitution). Ordered by version so the client scrubs oldest→newest. NO mutation here.
SELECT version, brightness, hue_shift, form_seed_delta, trigger, pe, dir, created_at
FROM evolution_history
WHERE memory_id = @memory_id AND user_id = @user_id
ORDER BY version ASC;
```
- 서비스 `GetEvolutionHistory(ctx, userID, memoryID)`는 repo 호출 후 도메인 슬라이스 반환(태그 없는 순수 도메인, 헌법5). 빈 결과는 정상(아직 한 번도 재성형 안 된 별 — 빈 목록 반환, 에러 아님).
- 핸들러는 11/12 패턴 그대로: `rpcserver.UserIDFromContext` → service → 도메인→`EvolutionSnapshot` proto 매핑(얇게). `memory_id`가 빈 문자열이면 `InvalidArgument`.

**FE 변천사 화면(`features/evolution`, 플랫폼 레이어 ui).** ReconsolidationCard의 변천사 strip을 청사진으로 재사용한다:
- **스크럽 슬라이더**: `snapshots[]`(version ASC) 위를 `0…N-1` 인덱스로 스크럽. 현재 인덱스의 스냅샷을 큰 `VizStar`로 재현.
- **버전 재현(`VizStar` 재사용, 시그니처 불변)**: `seed = baseSeed + Math.round(form_seed_delta)`(형태 변주), `brightness = snapshot.brightness`, hue는 ReconsolidationCard처럼 래퍼 div의 `style={{ filter: 'hue-rotate(${hueShift}deg)' }}`로 적용. `concept`(StarObject)·`color`(원본 감정색)는 별 전 생애 동안 **고정**(회상은 색을 통째로 바꾸지 않는다 — concept §주의).
- **계기 라벨**: 각 변화 지점에 trigger를 한국어로 매핑 — `recall`→"회상", `new_neighbor`→"새 이웃", `nightly_gist`→"야간 요지". `dir`로 ↑(강화)/↓(약화) 보조 표기.
- **불변 원본 나란히**: 같은 화면 한편에 11의 `RecallMemory` `Record`(body·entry_date·mood — records 출처)를 **읽기 전용**으로 고정. 슬라이더를 어디로 옮겨도 원본 텍스트는 그대로(변한 것 vs 변하지 않은 것).
```ts
// features/evolution/model/history.ts (순수 — three/DOM 미의존, 헌법4)
const TRIGGER_LABEL = { recall: '회상', new_neighbor: '새 이웃', nightly_gist: '야간 요지' } as const
export interface EvolutionStep {
  version: number; brightness: number; hueShift: number; formSeedDelta: number
  triggerLabel: string; dir: 1 | -1 | 0
}
/** snapshots(version ASC) → 스크럽 가능한 스텝 배열. 형태/색은 base에서 delta로만 변주. */
export function toEvolutionSteps(snapshots: EvolutionSnapshotVM[]): EvolutionStep[]
/** 인덱스 클램프(슬라이더 경계). */
export function clampIndex(i: number, len: number): number
```
```ts
// features/evolution/api/evolution.ts — unary (헌법6), 데모 분기는 recall.ts 본보기
import { memoryClient } from '@/shared/api'
import { isDemoMode, demoEvolution } from '@/shared/lib/demo'
export async function getEvolutionHistory(memoryId: string): Promise<EvolutionSnapshotVM[]> {
  if (isDemoMode()) return demoEvolution(memoryId) // 체험: 더미 변천사
  const res = await memoryClient.getEvolutionHistory({ memoryId })
  return res.snapshots.map(toSnapshotVM)
}
```

## 수용 기준 (EARS / Given-When-Then)
1.1 WHEN 사용자가 회상 패널(11)에서 "변천사 보기"를 누르면 THE SYSTEM SHALL 선택 별의 `GetEvolutionHistory(memory_id)`를 호출해 `version` 오름차순 스냅샷으로 스크럽 가능한 타임랩스를 연다. _(헌법6 — unary)_
1.2 WHEN 사용자가 스크럽 슬라이더를 움직이면 THE SYSTEM SHALL 해당 인덱스 스냅샷의 `form_seed_delta`(형태)·`hue_shift`(색조)·`brightness`(밝기)로 그 시점의 별을 `VizStar`로 재현한다(같은 기억의 변주 — `concept`·원본 감정색은 고정). _(concept §변천사)_
1.3 WHEN 각 변화 지점을 표시할 때 THE SYSTEM SHALL 그 변화의 계기를 한국어로 라벨링한다 — `recall`→"회상", `new_neighbor`→"새 이웃", `nightly_gist`→"야간 요지" — 및 `dir`로 강화↑/약화↓를 보조 표기한다. _(concept §변천사 — 변화 지점 라벨)_
1.4 WHEN 변천사 화면이 열려 있는 동안 THE SYSTEM SHALL 화면 한편에 11의 `RecallMemory`가 반환한 **불변 원본 `Record`**(body·entry_date·mood — records 출처)를 읽기 전용으로 고정 표시하고, 슬라이더를 어디로 옮겨도 원본 텍스트는 변하지 않는다. _(헌법1)_
1.5 WHEN `GetEvolutionHistory`를 호출할 때 THE SYSTEM SHALL `evolution_history`를 `WHERE memory_id=$1 AND user_id=$ctx`로만 읽고(다른 user의 변천사 비노출), `version ASC`로 정렬해 반환한다. _(헌법 — user 격리)_
1.6 WHEN `GetEvolutionHistory`가 실행될 때 THE SYSTEM SHALL `evolution_history`·`memories`·`records`를 **수정/삭제하지 않는다**(read-only RPC; `RETURNING *`·UPDATE·DELETE 없음). _(헌법1·2)_
1.7 IF 별이 아직 한 번도 재성형되지 않아 스냅샷이 0건이면 THEN THE SYSTEM SHALL 빈 목록을 정상 반환하고, 화면은 "아직 변천사가 없어요(최초 모습 그대로)" 안내를 보이며 빈 화면 오류를 내지 않는다.
1.8 IF `memory_id`가 빈 문자열이면 THEN THE SYSTEM SHALL `InvalidArgument`를 반환한다(별 생성·수정 없음).
1.9 WHEN `features/evolution/model`이 빌드될 때 THE SYSTEM SHALL `three`/React/DOM을 import하지 않는다(순수 model·모바일 재사용). _(헌법4)_
1.10 WHEN `VizStar`로 버전을 재현할 때 THE SYSTEM SHALL 기존 `StarVisualProps`(`seed`/`brightness`/`color`/`concept`)만 사용하고 좌표·신규 prop을 추가하지 않는다(시그니처 보존). _(헌법8 — 별 렌더 단일 출처)_
1.11 WHEN 데모(체험) 모드에서 별의 변천사를 열 때 THE SYSTEM SHALL `demoEvolution(memory_id)`(≥3 버전)로 분기해 백엔드 호출 없이 스크럽·재현·원본 나란히를 보인다. _(체험)_

## 완료 정의 (DoD)
- 회상 패널에서 "변천사 보기"로 선택 별의 변천사가 열리고, **스크럽 슬라이더**로 버전을 오가면 각 시점의 별이 `form_seed_delta`(형태)·`hue_shift`(색조)·`brightness`로 또렷이 다르게 재현된다. `VizStar` 시그니처는 변경 없음(grep: `EvolutionPanel`이 `seed`/`brightness`/`color`/`concept`만 전달, 좌표·신규 prop 없음).
- 화면 한편의 **불변 원본 `Record`**(11 `RecallMemory`)가 읽기 전용으로 고정되고, 슬라이더를 끝까지 옮겨도 원본 텍스트는 그대로다(편집 컨트롤 없음 — 헌법1).
- 변화 지점마다 계기 라벨(회상/새 이웃/야간 요지)과 ↑/↓가 표시된다.
- `GetEvolutionHistory`는 **읽기 전용**이다: `memory.sql`/repository에 `evolution_history`를 향한 INSERT/UPDATE/DELETE·`RETURNING *`가 없다(grep 확인). 다른 user_id로 호출하면 빈 목록(격리). 빈 변천사 별은 빈 목록 + 안내를 보이고 오류를 내지 않는다.
  ```bash
  # Docker/WSL — 읽기 전용 RPC, user_id는 인터셉터(여기선 토큰 헤더로 주입)
  docker run --rm fullstorydev/grpcurl -plaintext -d '{"memoryId":"m1"}' host.docker.internal:8080 cosimosi.v1.MemoryService/GetEvolutionHistory
  ```
- 데모(체험) 모드에서 `demoEvolution`이 ≥3 버전을 돌려주고, 스크럽 시 형태가 구별되며 원본이 나란히 보인다(백엔드 호출 없음).
- `pnpm --filter frontend build`·`pnpm lint` 통과. `features/evolution/model`이 `three`/React/DOM을 import하지 않음(헌법4, grep 확인).
- Docker/WSL: `docker run --rm -v "$PWD/backend:/src" -w /src golang:1.26 go build ./...` 성공 · `sqlc generate` · `buf generate --template backend/buf.gen.yaml proto` 무오류(`.exe` 호스트 직접 실행 안 함).

## 과학적 근거
변천사는 **재공고화(reconsolidation)** — 인출 시 기억이 일시적으로 불안정해져 강화·약화·갱신될 수 있는 "말랑한 창"이 열린다는 발견 — 의 산물이다(23이 실제 재성형을, 24가 그 누적 기록의 열람을 담당). 변형은 양방향(`dir` ±1)이며 직접 회상·직접 이웃에 한정되고, 원본 기록은 영향받지 않는다(concept §주의).

- Lee, J. L. C., Nader, K., Schiller, D. *Trends in Cognitive Sciences* (2017). [DOI](https://doi.org/10.1016/j.tics.2017.04.006)
- Debiec, J., LeDoux, J. E. *Proceedings of the National Academy of Sciences* (2006). [DOI](https://doi.org/10.1073/pnas.0507168103)

> 정직한 경계: 모든 인출이 기억을 바꾸는 것은 아니며(경계 조건이 있다 — 예측오차 게이트 `pe≥0.15`, 23), 변형은 좁은 범위에 그친다. 24는 그 변형을 **숨기지 않고 변천사로 드러내되**, 그날의 원본 텍스트는 **불변으로 함께** 둔다 — "기억은 변하지만, 그날 쓴 말은 그대로"라는 제품의 약속을 화면으로 지킨다.
