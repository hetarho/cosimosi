# 10. observability and flags

> Observability and flags - Sentry, PostHog, privacy consent, typed feature flags, and API error telemetry.
> Scope: FS | Status: built

## Purpose

When the product starts adding real use-cases, failures and experiments need to be visible without scattering vendor
calls through feature code. This unit adds a privacy-aware observability seam, a typed feature-flag registry, and
consistent error/limit/panic handling across web, mobile, and API. It does not turn on growth analytics as a product
feature; it creates the platform boundary.

## Scope / Non-goals

**In scope**

- Sentry setup for web, mobile, and API panic/error reporting behind platform wrappers.
- PostHog setup for product analytics and feature flags behind a consent-aware analytics facade.
- Consent gate: analytics events and user-identifying telemetry are disabled until the user has consented where
  required.
- Typed feature flag registry with default values, owner notes, rollout intent, and kill-switch behavior.
- API middleware/interceptors for request-id propagation, structured errors, panic recovery, and safe logging.
- Frontend error boundary and app-level error reporting.
- Test/local adapters that record telemetry in memory instead of calling vendors.

**Non-goals**

- No production dashboard design, alert routing/on-call process, billing/cost monitoring, or data warehouse.
- No A/B experiment design for product features.
- No remote-config system for product tuning values; tuning values stay in `spec/values.yaml`.
- No tracking before consent except strictly necessary operational error reporting that avoids personal content.
- No logging of diary text, generated memory content, tokens, secrets, or raw embeddings.

## Design

### Observability facade

App code imports the local `@cosimosi/observability` facade rather than vendor SDKs directly:

- `captureException(error, context)`
- `captureMessage(message, level, context)`
- `track(eventName, properties)`
- `identify(userId, traits)`
- `setConsent(consent)`
- `setRequestId(requestId)`
- `getFeatureFlag(key)`

The facade rejects sensitive fields at type-check time for literal property bags and at runtime for dynamic property
bags. Feature code cannot send diary text, raw generated content, auth tokens, or embeddings through typed event
properties.

`packages/observability` owns the facade, React provider/error boundary, request-id Connect interceptor, in-memory
adapter, and typed flag registry. Web and mobile app roots wire platform adapters:

- web: `apps/web/src/app/providers/observability-provider.tsx` initializes `@sentry/react` and `posthog-js` only when environment
  keys exist, attaching those adapters from effects rather than during render;
- mobile: `apps/mobile/src/app/providers/observability-provider.tsx` initializes `@sentry/react-native` when a DSN is passed and
  accepts an injected PostHog-compatible native client, also attaching vendor adapters from effects;
- API: `apps/api/internal/platform/observability` wraps `github.com/getsentry/sentry-go` behind a reporter interface.

`pnpm lint:observability` forbids direct Sentry/PostHog imports outside those app/platform boundaries, including
subpath and dynamic imports.

### Consent behavior

The consent state controls analytics and identification:

- before consent: no PostHog identify and no product analytics events;
- after consent: analytics events may be sent through the facade;
- withdrawal: future analytics stop and client identity is reset when the vendor supports it.

Operational error reporting may run before analytics consent because it is configured to avoid personal content and
does not identify the user beyond the request id. Web PostHog starts opted out, with autocapture, pageview capture, and
session recording disabled. React providers keep the facade stable across StrictMode effect replays and expose an
error-boundary reset render prop for platform-specific recovery UI.

### Feature flags

Flags live in `platformFeatureFlags`, a typed registry:

- stable flag key;
- default value;
- owner/plan reference;
- description;
- expiry or review note;
- local override for tests/dev;
- remote provider mapping if PostHog flags are enabled.

Feature code reads flags through the facade. Missing or disabled remote values fall back to committed defaults. Remote
values are consulted only after analytics consent is granted. The initial committed flag is
`platform.diagnosticsSurface` (default `false`), an operational development-surface flag, not a product experiment.
Flags are boolean-only; product tuning numbers are not flags and do not bypass `spec/values.yaml`.

### API error and panic handling

The API middleware/interceptor layer:

- attaches/propagates request id;
- recovers panics and converts them to canonical internal errors;
- maps unexpected internal/unknown errors to stable RPC codes;
- includes safe `error_type`/`panic_type` discriminators in telemetry reports without raw error messages or panic values;
- logs method/status/duration without payload bodies;
- reports unexpected errors through the observability facade.

This preserves the rule that handlers stay thin and policy remains in context behavior/domain.

The as-built interceptor flow is:

1. request-id context/header propagation;
2. structured internal/unknown error reporting and stable `CodeInternal` remapping;
3. method/request-id/status/duration logging;
4. auth/public-procedure enforcement;
5. outer panic recovery that reports safe panic telemetry and returns stable `CodeInternal`.

The API entrypoint handles SIGINT/SIGTERM through `http.Server.Shutdown` and flushes the reporter on graceful shutdown
or fatal listen errors.

## Acceptance Criteria

1. Web, mobile, and API code call local observability wrappers rather than vendor SDKs from feature/domain slices.
2. Local/test adapters capture telemetry in memory and require no Sentry/PostHog secrets.
3. Analytics and user identification are blocked until consent is granted; withdrawal stops future analytics.
4. Sensitive content fields such as diary text, tokens, raw embeddings, and generated memory content are not valid
   telemetry properties.
5. Feature flags are read through a typed registry with committed defaults and test/dev override support.
6. Product tuning values cannot be represented as feature flags.
7. API panic recovery and structured error mapping produce stable RPC errors and report unexpected failures.
8. Request-id context is visible in API logs, RPC responses/metadata where appropriate, and frontend error reports.
9. No production dashboard, deployment, alerting process, or product experiment is introduced.

## Policy / Values Impact

- **tech:** [tech/observability-and-flags.md](../tech/observability-and-flags.md) owns the as-built telemetry,
  consent, safe-property, request-id, API error, and feature-flag rules.
- **policy / ux:** no policy tree exists yet; no production dashboard, alerting, source-map upload, billing, or
  experiment process is introduced.
- **spec/values.yaml:** no entries. DSNs, sample rates, PostHog keys, consent defaults, and remote flag state are
  environment/ops configuration. Product tuning remains in `spec/values.yaml` and must not move to flags.
- **Dependencies:** implemented on top of [03.rpc-transport](03.rpc-transport.md). Current SDK dependencies are
  `@sentry/react`, `posthog-js`, `@sentry/react-native`, and `github.com/getsentry/sentry-go`.
