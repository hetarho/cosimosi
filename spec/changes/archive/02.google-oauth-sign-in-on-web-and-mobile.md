# 02. Change: Google OAuth sign-in on web and mobile

> A **change proposal** (a WHAT delta) that modifies the already-shipped behavior of [plan/04.auth-session](../plan/04.auth-session.md).
> Filled by /cosimosi:create-change through a user interview — a reviewable doc, not a one-line prompt.
> The STEPS live in the job that /cosimosi:create-change-job creates. Write it in English.

## As-is

The auth adapter exposes exactly one sign-in method, and both login pages surface only that method.

- `AuthAdapter.signIn(credentials)` (`packages/auth/src/auth-adapter.ts`) accepts the email+password
  `SignInCredentials` shape; the Supabase adapter (`packages/auth/src/supabase-adapter.ts`) implements it with
  `client.auth.signInWithPassword`. No OAuth path exists anywhere in `@cosimosi/auth`.
- The web `LoginPage` (`apps/web/src/pages/login/ui/LoginPage.tsx`, [plan 53](../plan/53.auth-universe-gate.md))
  renders an email+password form only — by design it "exposes just what the adapter exposes".
  The mobile `LoginPage` (`apps/mobile/src/pages/login/ui/LoginPage.tsx`) mirrors it.
- The pre-rebuild app (retired `frontend/`, commit `d123a6b`) had first-class Google OAuth:
  `signInWithOAuth({ provider: 'google', options: { redirectTo: <origin>/universe } })`, full-page redirect,
  session established on return by `detectSessionInUrl`. The rebuild dropped it; only email+password survived.
- Supabase-side infrastructure is already live (DEPLOY.md §5 step 10): Google provider configured (GCP client,
  callback URI), Site URL `https://cosimosi.haeram.me`, Redirect URLs covering `…/universe` for prod, localhost,
  and preview builds. Nothing app-side consumes it.
- Platform wiring today: the web Supabase client is created with `detectSessionInUrl: true`; the mobile client
  with `detectSessionInUrl: false, flowType: 'pkce'` and Keychain/Keystore token storage. The mobile shell owns a
  linking seam (`apps/mobile/src/shared/native/linking.ts`, prefix `cosimosi://`) that already exists for auth
  callbacks per ARCHITECTURE §3.5, but no auth callback route consumes it.
- The session machine (`packages/auth/src/session-machine.ts`) already accepts externally-driven sign-ins: the
  adapter's `onChange` snapshot (`source: 'signedIn'`) and bootstrap both drive `authenticated` without a
  `signIn()` round-trip — the exact shape an OAuth redirect return produces.

## To-be

Google OAuth becomes a second first-class sign-in method — **added alongside** the unchanged email+password path —
on both web and mobile.

- **Adapter contract.** `@cosimosi/auth` grows a Google sign-in capability next to `signIn`. The platform-pure
  contract stays token-free and Supabase-free at the seam (per plan 04's design); the Supabase adapter implements
  it, and `FakeAuthAdapter` implements it for tests/dev (immediate fake session, same as `signIn`).
- **Web flow (redirect).** `signInWithOAuth({ provider: 'google', redirectTo: <origin>/universe })` → full-page
  redirect to Google consent → return to `/universe`, where `detectSessionInUrl` establishes the session and the
  existing `onChange`/bootstrap path drives the machine to `authenticated`. The auth gate on `/` then routes as
  for any authenticated user. No new machine states.
- **Mobile flow (PKCE + deep link).** The adapter obtains the provider URL without navigating
  (`skipBrowserRedirect`), the login screen opens it in the system browser, Google redirects to a
  `cosimosi://` auth callback deep link (the existing linking seam), and the adapter exchanges the PKCE code for
  a session (`exchangeCodeForSession`). Tokens land in the existing Keychain/Keystore storage. A dismissed or
  failed browser round-trip returns the machine to a retryable state (`failed` with `error`, or `signedOut`) —
  never stuck in `signingIn`.
- **Login pages.** Web and mobile `LoginPage` each gain a "Continue with Google" action above the email+password
  form. Button copy follows the existing login keys’ restrained voice (no dedicated voice doc exists); failure surfaces through the
  existing snapshot `error` slot on the page.
- **Ops delta.** Supabase Redirect URLs gain the mobile `cosimosi://` callback; DEPLOY.md §5 step 10 records it.
  No backend change: the API's JWT verification is provider-agnostic, and the invite gate ([spec 41]) treats a
  first-time Google user like any other authenticated non-member (routed to `/invite`).

## Scope / Non-goals (regression boundary)

**Changes**

- `packages/auth` — adapter contract + Supabase adapter + fake adapter + facade: one new Google sign-in path.
- `apps/web/src/pages/login` and `apps/mobile/src/pages/login` — the Google button and its pending/error wiring.
- Mobile auth callback consumption on the existing `cosimosi://` linking seam.
- DEPLOY.md §5 step 10 (Supabase Redirect URLs) and plan 04/53 SSOT reflection.

**Must NOT change**

- Email+password sign-in: form, adapter `signIn`, and its tests stay working unchanged on both platforms.
- Session machine states/events: `bootstrapping | signedOut | signingIn | authenticated | refreshing | expired |
failed` — no new states; OAuth reuses existing events and the external-change path.
- Token handling invariants: tokens never enter machine context; mobile tokens stay in Keychain/Keystore; web
  storage unchanged.
- Dev sign-in bypass (`VITE_DEV_USER_ID`/`COSIMOSI_DEV_AUTH`): unchanged, still takes precedence, still
  build-fails in production bundles.
- Backend: no API, interceptor, proto, or migration changes. Invite gate behavior unchanged.
- No email OTP, no sign-up funnel, no password reset, no account linking/merge UI, no additional providers.

## Acceptance Criteria

1. Web: the login page shows a Google sign-in action alongside the unchanged email+password form; completing the
   Google consent flow lands the user authenticated on the universe without a second sign-in step.
2. Mobile: the login screen's Google action opens the system browser and, after consent, the `cosimosi://` deep
   link returns the app to an authenticated session (PKCE code exchange, tokens in secure storage).
3. Email+password sign-in still works unchanged on web and mobile (regression: existing auth tests stay green).
4. A Google-authenticated user passes the API auth interceptor (user id in request context) and is treated by the
   invite gate exactly like an email+password user.
5. Cancelling or failing the OAuth round-trip (web return without session, mobile browser dismissed) leaves the
   session machine in a retryable state with the error surfaced on the login page — never stuck in `signingIn`.
6. `FakeAuthAdapter` covers the new method so unit tests and the dev bypass run keyless; a production build with
   `VITE_DEV_USER_ID` set still fails at build time.
