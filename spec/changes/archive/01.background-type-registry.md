# 01. Change: background type registry

> A **change proposal** (a WHAT delta) that modifies the already-shipped behavior of [plan/14.rendering-foundation](../plan/14.rendering-foundation.md).
> Filled by /cosimosi:create-change through a user interview — a reviewable doc, not a one-line prompt.
> The STEPS live in the job that /cosimosi:create-change-job creates. Write it in English.

## As-is

The skin system in `@cosimosi/3d-renderer` conflates **which background look** with **the parameters for that look**,
and the generic renderer is welded to one concrete background.

- A `UniverseSkin` (`packages/3d-renderer/src/skin/presets.ts`) is a **flat** param bundle:
  `{ key, label, clear, palette[3], pattern{warp,freq,detail}, bloom{...}, camera{fov} }`. There are three presets —
  `aurora` · `ember` · `void` — that differ **only** in colors/tuning.
- The `Background` layer (`packages/3d-renderer/src/layers/Background.tsx`) **hardcodes**
  `scene.backgroundNode = nebulaBackgroundNode(skin)`. `nebulaBackgroundNode` (`skin/background-node.ts`) is the single
  background node-builder, reading `skin.palette` + `skin.pattern`.
- **Consequence:** all three "skins" render the _same_ `nebulaBackgroundNode`; there is no notion of a background
  **type**, so a skin cannot express a non-nebula look, and the generic renderer core owns a concrete background. This is
  the smell the change removes — a "skin" today is a palette variation of one hardwired node, not a choice of look.
- Field consumption today: `palette`/`pattern` → `nebulaBackgroundNode`; `bloom` → `PostFX` (`layers/PostFX.tsx`);
  `camera.fov` → `UniverseCanvas` (`canvas/UniverseCanvas.tsx`); **`clear` is unused** (declared on every skin, read by
  nothing).
- The skin **seam** (`skin-context.ts` · `SkinProvider.tsx`: `useSkin` / `resolveActiveSkin` / build-time
  `rendering.active_skin`) and the three consumers — web universe page (`apps/web/src/pages/universe`), the `/test`
  render panel (`apps/web/src/pages/test/lib/render-demo-panel.tsx`), mobile `UniverseScreen`
  (`apps/mobile/.../screens/UniverseScreen.tsx`) — all read the flat `skin` and pass it whole into
  `UniverseScene`/`Background`/`PostFX`.

## To-be

Separate **"which background" (type)** from **"params for that background" (props)**, and separate the **generic
renderer core** from the **concrete background assets** — inside the same `@cosimosi/3d-renderer` package.

### 1. Background is a _type_; each type owns its props + node-builder (registry)

- A **background type** is a key (`'nebula' | 'gradient'`), and each type owns **its own props shape** and **its own TSL
  node-builder**. The mapping `type → node-builder` lives in **one registry** in the assets module.
- A skin's background is a **discriminated union** — `BackgroundSpec = { type: 'nebula'; props: NebulaProps } |
{ type: 'gradient'; props: GradientProps }` — so each type is statically paired with the props it accepts.
  - `NebulaProps = { clear; palette: [number, number, number]; pattern: { warp; freq; detail } }`
  - `GradientProps = { top: number; bottom: number }` (a deliberately different shape — proves per-type props)
- The registry resolves a spec to a node: `resolveBackgroundNode(spec): TSLNode` (dispatch on `spec.type`). Adding a new
  background type = one new `{ props, node-builder }` module + one registry entry — **no change to any layer/host/seam**.

### 2. A skin is a _typed instance_ + scene-level ambiance

- `UniverseSkin = { key; label; background: BackgroundSpec; bloom; camera }`.
- **Type-specific** params live inside `background.props` (nebula: `palette`/`pattern`/`clear`).
- **Scene-level ambiance** — `bloom` (post) and `camera` (fov) — stays at the **skin top level**, because it applies to
  any background type. `PostFX`/`UniverseCanvas` read `skin.bloom`/`skin.camera` and **never** reach into a
  type-specific props bag.
- The three existing presets become **nebula-type** skins (`background: { type: 'nebula', props: {…} }`) with
  **identical visual output** to today (pure restructure — no re-skinning of aurora/ember/void).

### 3. Generic core vs concrete assets (in-package)

The package splits into a **generic core** (knows no concrete background type) and an **`assets/` module** (the concrete
looks that _use_ the core):

```
packages/3d-renderer/src/
  shader-art/                         GENERIC toolkit (noise · field · pattern · finish · sdf · geometry) — unchanged
  canvas/  UniverseCanvas             GENERIC host — unchanged
  skin-context · SkinProvider         GENERIC skin seam (useSkin / resolveActiveSkin) — unchanged API
  asset-source                        GENERIC §3.4 port — unchanged
  layers/  Background · PostFX · StarField   GENERIC scene layers (type-agnostic; see below)
  assets/                             CONCRETE — uses shader-art, provides looks
    backgrounds/
      nebula.ts     NebulaProps + nebulaBackgroundNode
      gradient.ts   GradientProps + gradientBackgroundNode   (the new 2nd type)
      registry.ts   BackgroundSpec union + resolveBackgroundNode(spec)
    skins/
      presets.ts    UNIVERSE_SKINS — aurora/ember/void (nebula) + dawn (gradient)
    UniverseScene   concrete composition: resolves the skin's background node, wires the generic layers
```

- **The generic `Background` layer names no concrete type**: it receives a ready background **node** (resolved by the
  assets module) and assigns it to `scene.backgroundNode`. The `type → node-builder` dispatch happens **only** in
  `assets/backgrounds/registry.ts`. No generic layer, the canvas host, the skin seam, or the asset-source port imports a
  concrete background type/node-builder.
- `UniverseScene` (the concrete universe composition) reads the active skin, calls
  `resolveBackgroundNode(skin.background)`, and hands the node to `<Background>` and `skin.bloom` to `<PostFX>` — so the
  type dispatch lives in `assets/`, not in a layer. (`StarField`, the foreground floating-stars layer, is untouched.)

### 4. A second background type exists (proves the seam)

- Add a `gradient` background type — a plain two-stop vertical sky (`mix(bottom, top, screenUV.y)`, no noise), with its
  own `GradientProps` shape — and **one skin of it** (`dawn`). Switching the active skin to `dawn` renders a **visibly
  non-nebula** background, demonstrating that the registry dispatches to a different node with different props.

### 5. Seam & config unchanged in shape

- `SkinProvider` / `useSkin` / `resolveActiveSkin` keep the same public API; the build-time `rendering.active_skin`
  still selects the active skin **key** (a skin implies its type). The `/test` switcher still lists `SKIN_KEYS` — now
  spanning both types. Consumers change only in _where they read a field_ (e.g. `skin.background`), not in the seam they
  use.

## Scope / Non-goals (regression boundary)

**In scope (what changes)**

- Replace the flat `UniverseSkin` with a **typed** model: `background: BackgroundSpec` (discriminated union) + top-level
  `bloom`/`camera`.
- Introduce background **types** (`nebula`, `gradient`), each with its own props + node-builder, and a single
  `resolveBackgroundNode` **registry**.
- Reorganize the package into **generic core** vs **`assets/`** (backgrounds + skins + `UniverseScene`); make
  `Background` type-agnostic (receives a resolved node).
- Move `palette`/`pattern`/`clear` into `NebulaProps`; keep `bloom`/`camera` at skin level.
- Add the `gradient` type + a `dawn` skin.
- Update the three consumers and the `/test` switcher for the new field paths; update the package public exports.
- Update the as-built rules in [tech/rendering.md](../tech/rendering.md) (Skins section, structure diagram, the
  "pass skin to Background/PostFX" note).

**Non-goals (what must NOT change — the regression boundary)**

- **No visual regression** for `aurora`/`ember`/`void` — they render exactly as today (nebula, same palettes/tuning).
- **No package extraction** — everything stays in `@cosimosi/3d-renderer`; a separate `@cosimosi/3d-assets` package is
  explicitly _not_ this change (a possible later step).
- **No change to the skin seam's public API** (`SkinProvider`/`useSkin`/`resolveActiveSkin`) or its build-time-select,
  runtime-ready ([P4]) design; **no runtime end-user skin switcher UI** is built here.
- **No change** to the `shader-art/` toolkit public API, the `asset-source` port (ARCHITECTURE §3.4), the `StarField`
  layer, the `UniverseCanvas` host signature, or the RN build setup (metro/babel/pod) — code stays **shared verbatim**
  web ↔ native.
- **No new `values.yaml` scalar**; `rendering.active_skin` semantics are unchanged (still a skin key; default `aurora`).
- **No domain semantics.** A background type is still **non-domain ambiance** — it cannot set or override per-memory
  emotion / position / strength. Invariants [I3] (color = emotion), [I5] (positions emergent), [I11] (customization
  cannot touch the meaning layer) are untouched.

## Acceptance Criteria

<!-- The testable criteria that mark this change done. Copied into the job's Acceptance Criteria section. -->

1. Background is modeled as a **discriminated union of types** (`BackgroundSpec`), where each type owns its own props
   shape **and** its own TSL node-builder, resolved through a **single registry** (`resolveBackgroundNode`); adding a
   type touches only the assets module (one new type module + one registry entry).
2. `UniverseSkin` is a **typed instance**: `{ key, label, background: { type, props }, bloom, camera }`.
   `palette`/`pattern`/`clear` live in `NebulaProps`; `bloom`/`camera` remain **skin-level** — `PostFX`/`UniverseCanvas`
   read `skin.bloom`/`skin.camera` and never index into a type-specific props bag.
3. The three existing presets (`aurora`/`ember`/`void`) are expressed as **nebula-type** skins and render with **no
   visual change** from today.
4. A **second background type** `gradient` exists with its own `GradientProps` + node-builder and **at least one skin**
   (`dawn`); selecting it renders a **visibly non-nebula** background — proving the registry dispatches to a different
   node with different props.
5. **Generic vs concrete is separated in-package**: the `shader-art/` toolkit, `canvas/` host, skin seam, `asset-source`
   port, and the scene **layers** reference **no** concrete background type; the `type → node-builder` dispatch and the
   concrete looks live in an `assets/` module that depends on the generic core (not vice-versa). The `Background` layer
   receives a resolved node and names no specific background type.
6. The skin **seam** keeps its public API (`SkinProvider`/`useSkin`/`resolveActiveSkin`); all three consumers (web
   universe page, `/test` panel, mobile `UniverseScreen`) work with the new skin shape, and the `/test` switcher lists
   skins across **both** types.
7. `rendering.active_skin` still selects the active skin (no new `values.yaml` scalar); the default remains a nebula skin
   (`aurora`); an unknown key still falls back via `resolveActiveSkin`.
8. The skin/background layer remains **presentation-only** — it cannot set or override per-memory emotion / position /
   strength ([I3][I11]); no domain semantics are added.
9. **No `three` import leaks** outside the package; **web** passes typecheck · lint · test · build and **mobile** passes
   its source gates (typecheck · lint · test), with the scene code still shared verbatim across web and native.
