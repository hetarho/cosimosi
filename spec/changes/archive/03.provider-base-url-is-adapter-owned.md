# 03. Change: provider base url is adapter owned

> A **change proposal** (a WHAT delta) that modifies the already-shipped behavior of
> [plan/28.ai-provider-abstraction](../plan/28.ai-provider-abstraction.md) (and its admin surface in
> [plan/58.admin-console](../plan/58.admin-console.md)). Filled by /cosimosi:create-change through a user interview —
> a reviewable doc, not a one-line prompt. The STEPS live in the job that /cosimosi:create-change-job creates.

## As-is

Providers already differ by transport and are already **per-provider code adapters**: `internal/ai` is a driver
registry where each vendor is its own subpackage (`internal/ai/anthropic` via the official SDK, `internal/ai/voyage`
via plain HTTP) implementing `LLMClient`/`EmbeddingClient`, registered from `init()` and blank-imported by `cmd/*`.
That part is correct and stays.

The wrong part is **base URL treated as generic, runtime-managed config**:

- `internal/ai` carries a base URL through the provider-config seam — `CapabilityConfig.BaseURL` / `ProviderConfig.BaseURL`
  — sourced from the `COSIMOSI_LLM_BASE_URL` / `COSIMOSI_EMBEDDING_BASE_URL` env vars (plan 28 §Selection & wiring), and
  the anthropic/voyage adapters accept it as an endpoint override.
- Plan 58 extended this into the admin console: a per-provider `ai_provider_keys.base_url` column, a `base_url` field on
  `SetProviderKeyRequest` / `ProviderKey` (proto), threaded through the admin service/pg and edited in the provider-keys
  UI, and included in the runtime config fingerprint.

So the base URL is editable in three places (env, DB, admin UI) as if it were a universal per-provider knob — but each
provider's endpoint is really an intrinsic, transport-specific detail of its own adapter, not something an operator sets
per provider at runtime. Plan 28 stating "base URLs are runtime env/secrets" is itself the source of this drift and is
now considered incorrect.

## To-be

**The provider endpoint is owned by its adapter. `base_url` is removed from the provider-config seam entirely** — no
`base_url` in env, in the DB, in the admin console, or in `ProviderConfig`/`CapabilityConfig`.

- Each implemented adapter owns its endpoint internally: `internal/ai/voyage` holds its HTTP endpoint as an adapter
  constant; `internal/ai/anthropic` uses the SDK default. A new adapter brings its own transport **and** endpoint.
- The AI provider-config seam carries only **provider + model + API key** per selection. The admin console manages only
  **per-provider API keys** (encrypted at rest, write-only, masked hint) and **per-capability provider + model
  selection**. No base-URL input anywhere.
- SSOT is kept consistent across **both** plans: plan 28 no longer says "base URLs are env/secrets" (it says the
  endpoint is adapter-owned), and plan 58's admin AI-config drops base URL from the table, proto, service, and UI.
- If a specific provider ever needs a self-hosted / proxy endpoint override, it is added as **that adapter's own
  env seam** at that time (an adapter-scoped, deliberate decision) — never a general runtime/admin knob. Not in this
  change.

Providers stay code adapters (unchanged); only the misplaced base-URL config is removed.

## Scope / Non-goals (regression boundary)

**In scope**

- Remove base URL from `internal/ai`: `CapabilityConfig.BaseURL`, `ProviderConfig.BaseURL`, the
  `COSIMOSI_LLM_BASE_URL` / `COSIMOSI_EMBEDDING_BASE_URL` env reads (`EnvCapabilityConfigs` / `NewAdaptersFromEnv`), and
  the base-URL term in the runtime config fingerprint.
- Make each implemented adapter own its endpoint: `internal/ai/voyage` (constant), `internal/ai/anthropic` (SDK
  default); drop the base-URL parameter they consume.
- Remove base URL from the admin surface (plan 58): drop the `base_url` column from `ai_provider_keys` (migration
  00015 is not yet released — edit it in place, or a follow-up migration if released), the `base_url` fields from
  `SetProviderKeyRequest` / `ProviderKey` (proto + regen), and from admin `types.go` / `ports.go` / `service.go` /
  `pg` / `rpc`, the `ReadProviderKey` base-URL return, and the provider-keys UI field + its i18n placeholder.
- Update SSOT: plan 28, plan 58, `tech/ai-worker.md`, `tech/admin-console.md`, `policy/ops/admin.md`, and
  `.env.example` (drop `COSIMOSI_*_BASE_URL`); `docker-compose` needs no base-URL var.

**Non-goals / regression boundary (must NOT change)**

- The per-provider **adapter registry** mechanism, the provider **slots** (openai/gemini/anthropic/deepseek/glm/kimi,
  voyage), **capability support** + implemented-status, **API-key** encryption/management, and **provider + model
  selection** are all unchanged.
- No new provider adapters; the set of implemented providers (anthropic LLM, voyage embedding) is unchanged.
- The env/DB fallback resolution for provider + model + key (DB → env → keyless mock) stays; only the base-URL
  dimension is removed.
- No change to metering, the error taxonomy, the keyless mock, or admin authorization.

## Acceptance Criteria

1. `base_url` / `BaseURL` / `BASE_URL` no longer appears in the AI provider-config seam: `ProviderConfig` and
   `CapabilityConfig` carry no base URL, and `COSIMOSI_LLM_BASE_URL` / `COSIMOSI_EMBEDDING_BASE_URL` are gone from the
   factory and `.env.example`.
2. Each implemented adapter owns its endpoint — a grep shows no base-URL value flowing from config/env into
   `internal/ai/anthropic` or `internal/ai/voyage`; both still build and call their vendor correctly.
3. `ai_provider_keys` has **no** `base_url` column; the admin proto (`ProviderKey`, `SetProviderKeyRequest`), admin
   `types`/`ports`/`service`/`pg`/`rpc`, and the provider-keys UI carry **no** base URL. `pnpm db:migrate` /
   `pnpm db:down` still apply/reverse cleanly.
4. The admin console manages exactly: per-provider API keys (encrypted at rest, write-only, masked hint) and
   per-capability provider + model selection — nothing else.
5. SSOT is consistent: plan 28 (no "base URL is env"), plan 58, `tech/ai-worker.md`, `tech/admin-console.md`, and
   `policy/ops/admin.md` all state the endpoint is **adapter-owned**.
6. **No regression:** anthropic LLM extraction/semanticize/prediction-error and voyage embedding still work; the
   keyless mock fallback, per-provider key management, and capability selection are unaffected; `pnpm test:api`,
   `pnpm build:api`, `pnpm typecheck`, and `pnpm build:web` pass.
